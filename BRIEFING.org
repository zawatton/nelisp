* 1. ミッション

純 Elisp でセルフホスティングする独自 Elisp VM を設計・実装する研究プロジェクト。
最終的には Emacs のコア部分を (C でなく) この Elisp VM で実装することを視野に入れる。

** 明示的な非目標

- Scheme 化 (Guile-Emacs 方向) — 拒否
- Rust 再実装 (Remacs 方向) — 拒否
- Emacs 代替 editor の開発 — out of scope (編集機能は後続 Phase)
- anvil.el への機能還元 — 任意、必須ではない

* 2. ユーザーの哲学

以下は user 本人の発言ベース、研究の方向性を決める核心:

#+begin_quote
Common Lisp、または Scheme は哲学的に優れていますが Emacs Lisp は
実用範囲、経験値で勝っていると思います。考えとしては Emacs のコア部分を
その独自 Elisp VM で実装する事です。Scheme ではなく純 Elisp として。
#+end_quote

含意:
- Elisp syntax / semantics を変えない (macro / dynamic binding / buffer-local 等の文化を保持)
- CL 並みの "通用範囲" (standalone 実行、threading、proper package、stdlib) を獲得
- Scheme 的 minimalism 志向ではなく、蓄積された Elisp 資産を尊重
- SBCL (CL in CL) と同じ構図を Elisp で実現

補足コンテキスト:
- user は道具で能力を拡張する人類史の延長線上に AI を置く思想を持つ
  (memory: =user_collaboration_philosophy.md=)
- 小さくても "前例のない" 研究に価値を見出す
- 完成より設計の筋の良さを優先する傾向

* 3. 先行研究ランドスケープ

| プロジェクト           | アプローチ                             | 本研究との差分                            |
|------------------------+----------------------------------------+-------------------------------------------|
| Remacs (2016-2020)     | C → Rust 書き直し、Elisp 保持          | 実装言語が Rust。Elisp self-host ではない |
| emacs-ng (active)      | Rust + Deno を統合                     | JS runtime 追加路線                       |
| Guile-Emacs (Wingo)    | Scheme で Elisp 再実装                 | Scheme 中心、Elisp は compat 層           |
| native-comp (Emacs 28+)| Elisp → libgccjit 経由 native          | JIT のみ、VM 再設計なし                   |
| SBCL                   | CL in CL self-host                     | CL 文脈。本研究の methodology 参考        |
| PyPy                   | RPython で Python VM を self-host      | Python 文脈。bootstrap 論の参考           |
| PicoLisp               | Tiny Lisp、VM を Lisp 自身で書く       | Lisp 方言、規模が違う                     |
| 本研究                 | Elisp で Elisp VM を self-host         | *前例なし*                                |

「前例なし」が研究価値の核。
ただし Remacs が discontinued した教訓 = scope 管理が最大の敵。

* 4. 核心課題 (Phase 0 で詰める設計論点)

** 4.1 Bootstrap 戦略

"Elisp で書いた VM をどう最初に動かすか" の鶏卵問題。

- (a) *Host-Emacs bootstrap* [推奨]: 現行 Emacs で VM.el を eval、
  その VM で自身を再 eval して self-host 到達。PyPy / SBCL 方式
- (b) Minimal Elisp→C 一回だけ: 初期 VM を C で、ASAP で Elisp に
  切替。保守負担が二重化しやすい
- (c) native-comp bootstrap: VM を native-comp 経由で native binary 化。
  Emacs 28+ 依存だが最短経路

** 4.2 Primitive 境界

VM 外 (C / asm / OS syscall) に残すライン:

- 確実に外: syscall、memory alloc、GC core loop、文字コード変換
- Elisp 化候補: gap buffer、redisplay scheduler、event loop dispatcher、regexp engine
- 議論: GC policy (generational?)、JIT compiler を Elisp で書くか

参考: SBCL の "tiny runtime + everything else in CL"。

** 4.3 Concurrency (novel contribution 候補)

本研究の最大の研究的価値領域。既存 Emacs cooperative 1-thread を引きずる必要なし:

- Thread 第一級化 (現状の =make-thread= は buffer-local state に unsafe)
- Actor / channel (Erlang mailbox or Go channel)
- STM: buffer 変更を transaction 化
- Structured concurrency (nursery / scope)
- Fiber (stackful coroutine、yield point 明示化)

設計 input: 親プロジェクト anvil.el の以下の memory が生きた痛みデータ:
- =feedback_emacs_server_concurrency_model.md=
- =feedback_anvil_accept_process_output_no_server_yield.md=
- =feedback_emacs_mutex_no_cross_session.md=
- =feedback_anvil_worker_phase1_startup_freeze.md=

** 4.4 Package / namespace

- 現行 =prefix-= convention と CL =defpackage= の共存
- =(in-package :foo)= を opt-in、既存コードは素通り
- ASDF / Quicklisp 相当を ELPA と統合 or fork

** 4.5 FFI / C module 互換性

- Emacs 25+ dynamic module ABI 保持 (生態系断絶回避)
- 加えて cleaner FFI レイヤ設計
- 既存 Emacs C 関数 ~3000 を段階的に Elisp 置換するロードマップ
