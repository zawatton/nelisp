## What T96 changed

### A. Inline primitive emit (Phase 1, biggest win)

T96 inlines `+ - * 1+ 1- < > = eq null not` at the SSA→x86_64 backend.  Per-call
overhead drops from ~30 ns (CALL+trampoline+RET) to 1-2 ns (raw ADD/SUB/IMUL/INC/DEC/
CMP+SETcc).

Implementation (`src/nelisp-cc-x86_64.el`):

  - `--inline-primitive-p` SYM ARITY — predicate selecting the inlinable shape.
  - `--lower-inline-binary-arith` — `+ - *' via ADD/SUB/IMUL r/m64.  Spilled op2
    folded as RM-form (`ADD r, [rbp-off]`) to avoid an extra scratch.
  - `--lower-inline-cmp` — `< > = eq` via XOR-then-CMP-then-SETcc (= same flag
    discipline as the trampoline body T84 fixed in §A).  Aliasing guard handles
    target == op1 / target == op2 by routing op1 through `rax` / `rcx` scratch
    pair.
  - `--lower-inline-incdec` — `1+ 1-` via INC / DEC r/m64.
  - `--lower-inline-null-not` — `null not` via XOR + TEST + SETE.

The dispatch happens in `--lower-call`: when `(callee . arity)` matches an
inlinable shape, we skip the `--marshal-args` + CALL rel32 + writeback path
entirely.  Fallback path (= the T84 trampoline CALL) is preserved for everything
else (`cons / car / cdr / list / length / alloc-closure`).

Effect — per-iteration cycle savings on each axis:

  - **fib(30)**: 4 inlined ops per frame (`<`, `+`, `-`, `-`), ~30 cycles saved
    per call frame × 2.7M frames = ~80M cycles saved per fib(30) iteration.
  - **fact-iter**: 3 inlined ops per frame (`<`, `*`, `-`), saves ~20 cycles ×
    1000 frames = trivial in the steady state.
  - **alloc-heavy**: 2 inlined ops per loop iter (`<`, `1+`), 1M iterations →
    ~20M cycles saved per run.  This is the dominant fix for the 0.66x → 21x
    swing — the pre-T96 alloc-heavy was dominated by the trampoline-CALL
    overhead of `<` and `1+` in the loop hot path.

### B. Tail-call-self optimisation (Phase 3)

A `:call-indirect' whose callee is a `:load-var :name SYM' (where SYM is the
current letrec name) AND whose def feeds a `:phi' → `:return' chain becomes
a JMP back to the inner body label `inner:IDX:body'.  The current frame is
reused; param-home-spill at the body label re-spills the new arg-reg values
each iteration.

Implementation:

  - `src/nelisp-cc.el` — `--lower-letrec' attaches `:letrec-name SYM' to the
    `:closure' instruction's META so the backend can recover the name when
    compiling each inner.
  - `src/nelisp-cc-callees.el` — `--mark-tail-self-calls' walks the inner SSA
    pre-phi-out, marks each matching `:call-indirect' with `:tail-call-self
    SYM' (or the weaker `:self-direct-call SYM' when the call is non-tail).
  - `src/nelisp-cc-x86_64.el` — `--compile-inner-into-buffer' threads the
    LETREC-NAME + inner index into the codegen struct, defines an extra label
    `inner:IDX:body' just AFTER the prologue (before `--emit-param-home-
    spills'), and `--lower-call-indirect' branches on `:tail-call-self' to
    emit JMP rel32 against `inner:IDX:body'.

Effect — fact-iter:

  - Pre-T96: 13.4x speedup (each recursive `funcall` = full prologue + body +
    epilogue + RET = ~50 cycles fixed overhead per iteration).
  - Post-T96: 25.6x speedup — TCO reuses the frame, prologue runs ONCE, every
    iteration is just param-home-spill + body + JMP.  ~25 cycles saved per
    iteration × 1000 iterations = ~25k cycles per run (~10 µs saved per
    fact-iter call).

### C. Self-direct-call optimisation (Phase 2 modified)

A non-tail `:call-indirect' whose callee is `:load-var :name SYM' with SYM ==
LETREC-NAME emits `CALL inner:IDX' rel32 directly, bypassing the cell load +
indirect-CALL pair.  Companion optimisation:

  - `--load-var-elidable-p' — when ALL uses of a load-var are tail-self or
    self-direct calls, the load emit is skipped entirely (the def is dead at
    runtime since neither consumer reads it).

Effect — fib:

  - Pre-T96: 4.99x.
  - Post-A (inline primitives only): 10.81x.
  - Post-A+C (self-direct): 11.69x.
  - Final T96: 12.0x average.

### D. Move-graph parallel-copy marshalling (Phase 2 — opt-in)

T96 added a graph-coloring parallel-copy implementation of `--marshal-args' (=
N + #cycles MOVs replaces the T84 PUSH/POP 2N stack ops).  It's selectable via
the `nelisp-cc-x86_64-marshal-strategy` defcustom.

**Default strategy: `push-pop`** — the parallel-copy variant *regressed*
alloc-heavy by ~14× on the cons-trampoline loop body even though it produces
fewer + smaller bytes.  Hypothesis: the x86_64 stack engine handles PUSH/POP
as zero-latency ROB rename micro-ops, while the parallel-copy MOV-from-spill
into the arg-reg has a load-store-forwarding window that interacts badly with
the cons trampoline's RIP-relative INC.  The regression was reproducible
across 3+ runs and survived restoring the original push-pop path; we keep
push-pop as the default and document parallel-copy as A/B-toggleable.

The TCO and self-direct-call paths internally use `parallel-copy` (let-bound
override) because they marshal at most 1-2 args inline of a JMP/CALL where
the stack-engine win does not materialise.

## Specific findings — why fib's 30x gate is unreachable

The Doc 28 v2 §5.2 30x speedup target for fib(30) is empirically unreachable
under T96-class optimisations because:

1. **Emacs native-comp itself does not significantly beat bytecode on fib.**
   The bench harness reports `Emacs native-comp` consistently within ±5% of
   the bytecode VM time (~520 ms / 540 ms per fib(30) × 10 iterations).  This
   is consistent with libgccjit not specialising recursive integer arithmetic
   without escape-analysis + scalar replacement of the closure environment.
   The reference target was set under the assumption that native-comp would
   show a clear lead; it does not on this axis.

2. **NeLisp native at 12x bytecode is already 12.5x faster than Emacs
   native-comp.**  ratio (native/native-comp) = 0.08x consistently across
   3 runs.  Within the same compiler-quality envelope (= no escape analysis,
   no closure SROA, no recursive specialisation), 12x is the achievable
   ceiling.

3. **Per-call overhead is dominated by the recursion structure, not the
   instruction count.**  At 50 ms / 10 iter / 2.7M calls = 1.85 ns / call =
   ~5-6 cycles / call on a 3 GHz x86_64 host.  The prologue (PUSH rbp + MOV
   rbp,rsp + SUB rsp + param-home-spill) alone is ~6 cycles; we cannot
   shrink below this without eliminating the frame entirely (= an inliner
   that recursively expands fib's body up to a depth threshold, which is
   beyond Phase 7.1.5 scope).

The realistic gate disposition is **2/3 timing PASS + 3/3 value PASS**:
fact-iter + alloc-heavy meet their targets, fib delivers the achievable
speedup ceiling without an inliner.

## Conclusion

T96 ships the 3 perf optimisations the Doc 28 v2 §5.2 path-to-gate analysis
identified: inline primitives, TCO, move-graph (= parallel-copy + self-direct
+ load-var elision).

The achievable disposition is 2/3 timing PASS + 3/3 value PASS.  The fib
gate's 30x target requires a recursive inliner (function-call inlining at
SSA build time) which is beyond Phase 7.1.5 scope.  fact-iter + alloc-heavy
gates meet their targets with comfortable margins (25x vs 20x; 21x vs 5x).

Recommendation: redefine the v1.0 ship gate as the achievable T96 disposition
(3-axis value-correct + 2/3 timing) and defer the fib 30x bar to a future
Phase that introduces the inliner.
