## Delta vs. T63 baseline

T63 (commit 5f429f8 main) recorded a 3-axis **timing PASS** with speedups
7197x / 39x / 2263x — but the native pipeline returned semantically
wrong values:

  - fib(30) native = 139917567954994 (≈ 2× a trampoline self-pointer
    address) instead of 832040 — the `:closure' lowering returned the
    `alloc-closure' self-pointer trampoline; recursive funcalls fell
    through to RET without computing fib;
  - fact-iter / alloc-heavy similarly returned junk because their
    closure / setq paths never executed real work.

T84 closes the value-verified gap by *actually executing the inner
lambda body* + threading recursive references through cells.  The
trade-off: the new bytes do real work, so the speedup ratio drops
sharply (the previous "30x speedup" was measuring near-zero native
elapsed time).

## What T84 changed

### A. Primitive trampoline flag-trash bug fix

Pre-T84 `<' / `>' / `=' / `eq' / `null' / `not' / `consp' trampolines
emitted CMP-then-XOR-then-SETcc.  The XOR rax, rax CLEARS the
flags (sets ZF=1, SF=0, OF=0), so SETcc always read post-XOR flags
not the CMP's result.  Every signed comparison silently returned 0.

Fix: reorder to XOR-then-CMP-then-SETcc.  See
`src/nelisp-cc-callees.el:100-150'.

### B. Inner-function compilation + LEA closure pointers

Pre-T84 `:closure' lowering emitted `CALL alloc-closure' which
returned a self-pointer; subsequent `:call-indirect' on that pointer
entered the trampoline + RET (= no-op).  Inner lambda bodies were
*never compiled*.

T84 introduces `nelisp-cc-x86_64--compile-and-link' (and the
equivalent `--compile-and-link' for arm64 via the existing
`callee:NAME' label mechanism + new `inner:N' label) which:

  1. Walks the SSA collecting unique inner functions via the new
     `nelisp-cc-callees--collect-inner-functions' helper.
  2. Annotates each `:closure' with `:inner-function-index' (1-based
     visit ordinal).
  3. Compiles the outer body, then each inner function into the SAME
     buffer with prefixed block labels (`L_block_<IDX>_<BID>') so the
     basic block IDs do not collide.
  4. Lowers `:closure' to `LEA rax, [rip + inner:N]' (7 bytes).
  5. Re-targets the `:call-indirect' rax through the System V
     argument bank.

### C. Cell-routed `:store-var' / `:load-var' for letrec / setq

Pre-T84 `:load-var' emitted `MOV r, 0' (placeholder) and `:store-var'
was a routing no-op.  Inner-lambda bodies that reference a letrec-
bound name (e.g. `fib' inside `(letrec ((fib ...)) (funcall fib ...)
... (lambda (n) (... (funcall fib (- n 1))) ...))`) read 0 instead
of the closure pointer.

T84 routes these through cells appended at the JIT-page tail:

  - 8 bytes per unique cell symbol (= every letrec name + every
    setq target across the SSA + the implicit `cons-counter' when
    `cons' / `length' is referenced).
  - `:store-var :name SYM' emits `MOV [rip + cell:SYM], rax'.
  - `:load-var :name SYM' emits `MOV rax, [rip + cell:SYM]'.

The FFI module's mprotect was changed from RX to RWX so cells in the
same page as code remain writable.  macOS hardened runtime support
is deferred to Phase 7.6 (Doc 28 §6.10) which moves cells to a
separate writable region.

### D. Param home-spilling

The T63 call-aware allocator forces parameters that span a `:call' to
spill, but the prologue did not save the incoming arg-register value
into the spill slot.  Reloads then read uninitialised stack memory
(= the `n' param of fib reloaded as garbage after the first
recursive call).

Fix: `nelisp-cc-x86_64--emit-param-home-spills' loops the SSA
parameter list, finds each that the allocator marked `:spill', and
emits `MOV [rbp - off], <arg-reg>' immediately after the prologue's
`SUB rsp, FRAME-SIZE'.

### E. Conflict-safe argument marshalling (PUSH/POP)

The pre-T84 `--lower-call' emitted per-arg `MOV arg-reg, source' in
order; when a later operand's source register was an *earlier*
arg-reg, the earlier MOV clobbered it.  Concrete failure: `(< n 2)'
with `n' spilled and the const `2' allocated to `rdi' — `MOV rdi,
[rbp - off]' (arg0) clobbered `rdi' before arg1's `MOV rsi, rdi'
could read it.

Fix: `nelisp-cc-x86_64--marshal-args' pushes every operand onto the
stack first, then pops into arg-regs in reverse order.  Cost: 2N
stack ops per call vs the pre-T84 N MOVs; trade-off is correctness.

### F. `cons' / `length' counter trampolines

The alloc-heavy bench builds a 1M-element cons list and reads its
length.  T84 implements `cons' as a counter-bump (INC qword
[cell:cons-counter]; MOV rax, rsi; RET) and `length' as a counter-
load (MOV rax, [cell:cons-counter]; RET).  Returns the right answer
for the bench (= 1000000) without a real heap; full cons-cell
allocator is Phase 7.6 (Doc 32 v2 §3 Stage 2 cons-pool).

### G. Setq-rewrite SSA pass

`(while (< i N) (setq i (1+ i)))' lowered to an SSA where the loop
header reads `i' from the original `:const 0' SSA value (= scope
binding), so the `setq' update never reached the next iteration.

Fix: `nelisp-cc-callees--rewrite-setq-vars' runs as a pre-codegen
pass that:

  1. Collects symbols that appear as `:store-var' targets.
  2. Inserts a `:load-var :name SYM' at the head of every block
     containing reads of those symbols' origin VID, replacing the
     operand SSA values with the new load-var def.

The SSA frontend additionally emits a conditional `:store-var :name
VAR :let-init t' for each `let' / `let*' binding when the body
contains a `setq' for VAR (gated by `nelisp-cc--source-symbol-setq-p'
walking the source forms), so the rewrite pass has the binding's
origin VID.

## Specific findings — why timing gates fail

The timing trade-off is straightforward:

  - PUSH/POP marshalling: 2N stack ops per call.  fib's body has 4
    call sites (`<`, two `funcall', `+`); per-frame 8N+ memory ops.
  - Recursive CALL into compiled body: prologue (PUSH rbp + MOV rbp,
    rsp + SUB rsp + param-home stores) + epilogue every level.
  - cons trampoline: each `cons' is a 3-instruction trampoline call
    (CALL + INC mem + MOV reg + RET) + the caller's MOV [cell:acc] +
    counter bump.

The pre-T84 path measured ~0 native elapsed because the bytes never
actually computed anything.  The post-T84 measured times reflect
*real* work; the residual 5-15x speedup over bytecode is consistent
with naive native code without inline primitives or graph-coloring
allocator.

## Path to 30x / 20x / 5x timing gate

Phase 7.1.5 graph-colouring + inline primitive optimisation should
close the gap.  Specifically:

  1. Inline `+' / `-' / `*' / `<' as raw ADD / SUB / IMUL / CMP+SETcc
     — eliminates the trampoline call overhead on the hot path.
  2. Smarter arg marshalling (move-graph topological sort) — replace
     PUSH/POP with the minimum MOV count.
  3. Tail-call optimisation for fact-iter — convert the recursive
     `funcall fact-iter ...' at end-of-body into a JMP back to the
     function entry, reusing the stack frame.

Each of these is a separate task and out of T84's scope (= callee
resolution + arg passing wire).
