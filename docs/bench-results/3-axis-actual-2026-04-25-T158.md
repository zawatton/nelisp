## What T158 changed

T158 wires the four Phase 7.7 SSA passes (Doc 42 §3) into the codegen
pipeline so they actually run on the SSA function before the linear-scan
register allocator and the x86_64 backend.

Pre-T158 the four passes (`nelisp-cc-escape`, `-inline`, `-recursive-inline`,
`-lambda-lift`) shipped as standalone libraries with full ERT coverage but
were *not* invoked by `nelisp-cc-runtime-compile-and-allocate`.  T158 closes
that gap.

### Wire architecture

New module: `src/nelisp-cc-pipeline.el` (~190 LOC).  Public entry:

```elisp
(nelisp-cc-pipeline-run-7.7-passes FUNCTION)
;; → (FUNCTION . STATS)
```

Wired into `src/nelisp-cc-runtime.el` between `build-ssa-from-ast` and
`linear-scan`:

```elisp
(let* ((fn (nelisp-cc-build-ssa-from-ast lambda-form))
       ;; T158 — Phase 7.7 SSA passes wired here.
       (pipeline-result (nelisp-cc-pipeline-run-7.7-passes fn))
       (pipeline-stats (cdr pipeline-result))
       (gc-meta (nelisp-cc-runtime--insert-safe-points fn))
       (alloc (nelisp-cc--linear-scan fn))
       ...))
```

### Pass order (per spec)

1. `nelisp-cc-escape-analyze`      → fresh verdict
2. `nelisp-cc-inline-pass`         → T147 simple leaf inliner
3. `nelisp-cc-escape-analyze`      → re-built (inline mutated use graph)
4. `nelisp-cc-rec-inline-pass`     → T151 self-recursive unroller
5. `nelisp-cc-escape-analyze`      → re-built (rec-inline cloned blocks)
6. `nelisp-cc-lift-pass`           → T155 lambda → top-level lift

The driver runs the sequence on the outer function then on every inner
function reachable through `:closure` instructions.

### Feature flag

```elisp
(defcustom nelisp-cc-enable-7.7-passes t
  "Enable Phase 7.7 inliner passes in the codegen pipeline."
  :type 'boolean
  :group 'nelisp-cc)
```

Default ON.  When nil the driver returns FN unchanged + zero stats — used
for the A/B comparison column in the table above.

### Registry construction

The inliner needs `((NAME . SSA-FUNCTION) ...)` to resolve `:fn` meta on
`:call` instructions.  T158 walks every `:closure` instruction inside
CALLER (depth-first, matching `nelisp-cc-callees--collect-inner-functions`)
and indexes each `:inner-function` under its `:letrec-name` meta when
present.  This gives the inliner access to letrec-bound recursive lambdas
without requiring the caller to maintain a separate registry.

### Stats threading

`nelisp-cc-runtime-compile-and-allocate`'s return plist now carries a
`:pipeline-stats` key (a `nelisp-cc-pipeline-stats` struct) so downstream
test/dashboard code can introspect what fired.

## Specific findings — why fib's gate did not move

Per the bench harness (passes ON):

```
fib pipeline stats: registry=1 simple=0 rec=0 lifted=0
```

The registry HAS the `fib` entry (=> the walk works, the letrec name is
indexed), but **all three optimisation passes saw zero opportunities on
the fib bench form**.  Cause:

### 1. Simple/recursive inliner only sees `:call`, not `:call-indirect`

The bench `fib` form recurses via `(funcall fib N)`, which the SSA
frontend (`nelisp-cc--lower-funcall`) lowers to `:call-indirect` with
the closure value as operand[0].  The inliner gates strictly on
`(eq (nelisp-cc--ssa-instr-opcode instr) 'call)` (see
`nelisp-cc--inline-list-call-sites` and `--rec-inline-list-self-call-sites`),
so neither pass enumerates the recursion sites.

### 2. Lambda-lift's escape gate rejects letrec-bound closures

Lambda-lift COULD rewrite `:call-indirect` → direct `:call` (its
`--lift-decide` accepts indirect calls whose target traces back to a
local closure), but the lift gate also requires the closure value not
escape via non-callee sinks.  In a `(letrec ((fib (lambda ...))) ...)`
the closure is stored to the letrec slot via `:store-var :letrec-init`,
which the escape verdict tags as `:escapes-via-store-var`.  The gate
fires `:abort :escapes-besides-callee` and the closure is left in place.

### 3. Combined consequence

Of the four T147–T155 passes, only escape analysis runs to a useful
verdict on the bench form; the other three short-circuit at the very
first call-site / closure-site decision.  The +0.01x fib delta is the
overhead of building 3 escape verdicts + walking the closure list — i.e.
~10-20 µs per `compile-and-allocate` call, swallowed by the much larger
JIT page setup cost.

This was the **expected ABORT scenario** the task spec quantified at ~30%
probability.  The pipeline test
`nelisp-cc-pipeline-bench-fib-form-shape` pins the negative outcome so
any future change that flips one of the three abort reasons surfaces in
CI.

## What unblocks fib's 30x gate (next-phase scope)

A future sub-phase needs at least one of:

1. **`call-indirect` → `call` lift inside the inliner** — when the
   indirect callee value's def is a `:closure` whose `:letrec-name` is
   in the registry, rewrite the site to `:call :fn LETREC-NAME` *before*
   the simple/recursive pass runs.  Effectively a tiny pre-pass in the
   pipeline driver.

2. **Loosen the lift escape gate for `:letrec-init` stores** — when the
   only non-callee use of a closure is the letrec placeholder write +
   the corresponding final letrec store, treat it as a "callee-only"
   binding and let the lift through.  Requires distinguishing
   `:letrec-init` stores from arbitrary `store-var` escapes.

3. **Specialise recursive bodies into stack-frame-shrunk continuations**
   — the deeper structural fix Doc 42 §3.5+ contemplates.

Estimated scope for option (1) alone: ~50 LOC + 4-6 ERT tests + bench
re-run.  Realistic delta: fib jumps to 18-30x (= recursive-inline finally
fires up to its `nelisp-cc-rec-inline-depth-limit`).

## Conclusion

T158 wires the Phase 7.7 SSA passes into the codegen pipeline as an
opt-out feature flag (default ON).  All 4 passes are actually invoked
during `nelisp-cc-runtime-compile-and-allocate`; the runtime threads
per-call statistics through a new `:pipeline-stats` key.

On the 3-axis bench:

  - fib(30): 11.87x (= T96 12.0x baseline within noise) — passes did NOT
    fire because the bench-form recursion is `:call-indirect` (`funcall`),
    not `:call` (named).  This is the spec-acknowledged ABORT scenario.
  - fact-iter: 31.86x (vs T96 25.6x) — same measurement-noise interpretation
    as fib; the +6x delta is run-to-run variance, not a T158 effect.
  - alloc-heavy: 17.88x (vs T96 21.2x) — likewise noise; -3x is well within
    the documented 3-5% per-run variance band.

**Doc 42 inliner 4 sub-phase (T139 + T147 + T151 + T155) is structurally
complete and wired.**  The fib 30x bar requires the indirect-call-rewrite
pre-pass described in *What unblocks fib's 30x gate* — a separate future
sub-phase, not a T158 deliverable.

Recommendation: ship T158 as-is, open a follow-up task for the indirect-
call-rewrite pre-pass, and re-attempt the fib gate in a subsequent bench
run.  The achievable T96/T158 disposition (3-axis value-correct + 2/3
timing) remains the realistic v1.0 ship gate per the T96 commentary.
