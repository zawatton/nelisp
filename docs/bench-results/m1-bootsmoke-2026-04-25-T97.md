## tools/call probe

```
> {"jsonrpc":"2.0","id":3,"method":"tools/call",
   "params":{"name":"defs-index-status","arguments":{}}}
< {"jsonrpc":"2.0","id":3,"error":{"code":-32602,
   "message":"Tool handler must return string or nil, got: cons"}}
```

This is **not** an M1-substrate gap.  `anvil-defs--tool-index-status`
returns `(anvil-defs-index-status)` directly — i.e. a plist — but
`anvil-server` enforces a "string or nil" return contract at
`anvil-server.el:1066-1078`.  The handler needs an explicit
`prin1-to-string` / JSON encode wrap before its plist hits the
dispatcher.  The bug exists on host Emacs the same way; documented
here as **T97-FOLLOWUP-1** for a separate fix.

## missing subsystem (= next dispatch input)

The fact that *all 58 anvil-* modules load on host Emacs after the
substrate install means **at the `require` granularity** there is
no missing subsystem visible from M1 boot.  The real gaps live one
level down — names anvil-* code calls at runtime that the bridge
does not yet map.  Spot probe (representative, non-exhaustive):

| anvil-used name                       | in 54-entry bridge | gap kind            |
| ------------------------------------- | ------------------ | ------------------- |
| `message`                             | NO                 | I/O primitive       |
| `format`                              | NO                 | core stringer       |
| `replace-regexp-in-string`            | NO                 | regex (string-side) |
| `buffer-substring-no-properties`      | NO                 | text extraction     |
| `string-match`                        | NO                 | regex (string-side) |
| `split-string`                        | NO                 | string surgery      |
| `mapconcat`                           | NO                 | string surgery      |
| `setq` / `let` / `let*` / `defun`    | NO (special-form)  | reader / evaluator  |

The reader / evaluator / special-form group is **the** big-rock
gap — those are not bridge-able by `defalias`, they need a real
NeLisp reader + evaluator (which is a separate roadmap branch =
NeLisp Phase 7+ pipeline already well underway, not a substrate
subsystem).

The string-surgery group (`format`, `mapconcat`, `split-string`,
`replace-regexp-in-string`, `string-match`, `buffer-substring-*`)
is the **immediate next-task target** — a "Phase 9d.A5"-shaped
extension that would push the bridge from 54 → ~60-65 entries and
let anvil-server's hot path not need any host Emacs binding.

## next-step suggestion (top 3)

1. **NeLisp Phase 9d.A5 — string surgery extension**.  Add
   `format` / `mapconcat` / `split-string` / `replace-regexp-in-string` /
   `string-match` / `buffer-substring-no-properties` /
   `message` to nelisp-emacs-compat (or a new `-strops` module) and
   bridge them.  Brings the anvil hot path to ≈ 75% functional.
   Sized: ~600-1000 LOC + 30+ ERT.

2. **T97-FOLLOWUP-1 — anvil-server return-shape contract**.
   Either widen `anvil-server-with-error-handling` to accept
   plists/conses (auto-`prin1-to-string`) or fix the four
   defs-/state-/buffer- return points that surface this.
   Out-of-scope for M1; in-scope for any tools/call smoke run.

3. **NeLisp Phase 7+ reader-evaluator gate**.  The real M1
   blocker.  Without `setq`/`let`/`let*`/`defun`/`defmacro` running
   on the NeLisp evaluator, the substrate cannot host **any**
   anvil-* module standalone — the `defalias` bridge approach only
   substitutes function bodies, not special-form readers.  Already
   on the master roadmap (Phase 7+ design pipeline LOCKED 2026-04-25);
   M1 is gated on Phase 7+ landing.
