## Gap analysis — why fib didn't reach 30x

T163 measured fib **14.38x** with depth=2 + lifted=1 + pipeline ON. The 30x target is **15.62x** away (= a further ~2.1x compile-time speedup is needed on top of the inliner work already done).

### What is *not* the bottleneck

- **rec-inline depth.** depth=2-3 is the local maximum; raising depth does not help (and depth>=4 actively hurts via I-cache pressure + fact-iter regression).
- **value-correctness.** All depths (0-6) produce 832040; the survival-arm fix (G1 / 7914ff2) covers depth>=1.
- **Pipeline OFF baseline.** Pipeline OFF gives ~12.45x; pipeline ON gives ~14.38x (= +15% lift). The inliner is contributing but is not the dominant cost.

### What is dominating

The remaining cost is per-call overhead in the **non-inlined recursive tail** of fib(30). After depth=2 unroll, every leaf-of-unroll still issues a real `call` to the trampoline (= ~5 cycles call + 5 cycles ret + register save/restore + arg marshalling). With ~2.1M dynamic calls in fib(30), even ~10ns/call is ~21ms = >40% of the native runtime budget.

### What would close the gap (= ranked likelihood)

1. **Lambda-lift backend handling of leaf self-calls (= eliminate trampoline thunk for self-recursion).** Currently `lifted=1` resolves the closure but the recursive call path still goes through the trampoline. Direct PC-relative `call` to the `callee:fib` alias label (= already bound by T162) would shave ~3-4 cycles/call ≈ ~6-8ms savings → projected fib ~18-22x.
2. **Simple-inline base case (= fold `(if (< n 2) n)` with constant n at unroll boundaries).** When n is determinable at unroll time (= depth=2 covers n=30, 28, 26, ...), the base case branch becomes constant-foldable. Cost: needs interval analysis hooked into the simple-inline pass. Projected: another ~5-7x speedup if all base cases at depth=2 fold → fib ~25-32x.
3. **Tail-call optimization for the non-recursive `+` accumulator.** fib's two `(funcall fib ...)` returns are summed; if they were tail positions, `call` could become `jmp`. Projected: ~1.2-1.5x further.
4. **Calling-convention specialization for monomorphic `funcall` (= bypass the generic dispatch when the function-pointer is statically known).** Already partially T161-rewritten to `:call :fn fib`, but the backend `:call` still emits the generic prologue. Projected: ~1.3x.

A reasonable Phase 7.1.1+ roadmap to hit 30x fib:
- Phase 7.1.1: lambda-lift direct-call elision → ~20x
- Phase 7.1.2: simple-inline interval-aware base-case fold → ~28-32x
- Phase 7.1.3: tail-call jmp + calling-convention specialization → ~35-40x (= comfortable margin)

The full pipeline (escape + simple + recursive + lambda-lift) is *engaged* and *value-correct*; the remaining work is structural backend tuning, not SSA correctness.
