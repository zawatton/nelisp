## What T161 changed

T161 fixes the two SSA-level blockers that T158 (Doc 42 §3.4 lift +
§3.3 rec-inline post-mortem) identified:

1. **`:call-indirect` → `:call` rewrite pre-pass** (new module
   `src/nelisp-cc-rewrite.el`).  Before the inliner suite runs, the
   pipeline now walks every `:call-indirect` instruction and rewrites
   eligible sites to direct `:call` with `(:fn NAME :unresolved t
   :rewritten-from-call-indirect t)` meta.  Eligibility:

   - operand[0] is the def of a `:load-var SYM` whose `SYM` is in the
     letrec-callee registry, OR
   - operand[0] is the def of a `:store-var :letrec-init t' instruction
     whose `:name` SYM is in the registry (covers the outer-lambda
     funcall path where the letrec scope binds NAME directly to the
     store-var def).

   Public API: `nelisp-cc-rewrite-call-indirect-to-call FN REGISTRY` →
   `(FN . COUNT)`.  ~150 LOC + 6 ERT.

2. **Lambda-lift letrec-init exclusion** (`src/nelisp-cc-lambda-lift.el`
   `nelisp-cc--lift-uses-only-as-callee-p` + `--lift-def-escapes-besides-callee-p`).
   The lift gate now treats a `:store-var :letrec-init t` use of the
   closure as an *allowed sink* (= dead-code-pending after T161
   rewrites every `:call-indirect` reading the slot).  The companion
   `nelisp-cc--lift-drop-letrec-init-store` helper deletes the store
   from its block when the lift fires.

3. **Pipeline driver order swap + depth-limit cap** (`src/nelisp-cc-pipeline.el`):
   inner-function passes now run *before* the outer's lift step (lift
   renames the inner SSA function to `<CALLER>$lift$N`, which would
   defeat the recursive inliner's `(eq callee-name self-name)`
   self-detection).  Added `nelisp-cc-pipeline-rec-inline-depth-limit`
   defcustom (default 2) bound around `nelisp-cc-rec-inline-pass`
   invocations to cap fib unroll at 7 inlines / ~700 cloned blocks
   (~10 ms latency).  The unbounded depth would expose a pre-existing
   exponential-growth bug in `nelisp-cc--rec-inline-clone-blocks`
   (each splice clones the *grown* function, not the pre-pass body —
   future Phase 7.7.5+ rec-inline rework).

4. **Pipeline default flip to nil** (`nelisp-cc-enable-7.7-passes`).
   The SSA passes are correct (full ERT) but the lambda-lift step
   synthesises `:call :fn <CALLER>$lift$N' targets that the current
   backend's `nelisp-cc--link-unresolved-calls` cannot resolve (the
   synthetic name has no registered trampoline → fixup left at 0 →
   segfault on exec).  T161 keeps the flag default-off until the
   Phase 7.7.5+ backend learns to embed + resolve synthetic callee
   bodies in the same translation unit.  Tests that need the SSA-level
   outcome bind `nelisp-cc-enable-7.7-passes t` locally.

## Specific findings

### Why fib stays at ~10x even with T161 SSA work correct

The bench gate measures *runtime* speedup vs the bytecode VM.  T161
mutates SSA only when `nelisp-cc-enable-7.7-passes t`, and that path
produces synthetic callee names (`anon$lift$0`, `fib`) that the
backend's link step cannot resolve to native addresses.  Forcing the
flag on segfaults the bench (verified: with pipeline ON, even fib(5)
crashes during exec because the rec-inline residual call to `:fn fib`
falls through to address 0).

T161 therefore ships the SSA work + the test coverage proving it
works at the IR level, and leaves the *runtime* enablement gated on
a follow-up Phase 7.7.5+ task that:

- registers each lifted/rewritten synthetic name as a callee whose
  body lives in the same emitted byte buffer (= self-reference rel32
  fixup), or
- threads a per-translation-unit symbol table through
  `nelisp-cc--link-unresolved-calls` so synthetic names patch to the
  correct in-buffer offset rather than fall through to 0.

### Pre-existing rec-inline exponential-growth bug

`nelisp-cc--rec-inline-clone-blocks` snapshots
`(nelisp-cc--ssa-function-blocks callee)` at clone time — but for
self-recursion `callee == caller`, and after the first splice the
caller has *grown* (the cloned blocks are now part of its block
list).  The next splice's snapshot picks up the previously-cloned
blocks and re-clones them.  Block count grows as 2 → 22 → 766 → ∞
across depths 1 / 2 / 3.  Doc 42 §3.3 acknowledges this as a Phase
7.7.5+ rework (snapshot the original callee body before any splice).

### Why fact-iter dropped from 31.86x → 27.0x

T161 default-off pipeline is identical to T158's "pipeline OFF"
runtime path (= verdict re-build cost was the only T158 overhead and
it was negligible).  The ~5x delta is run-to-run variance on a shared
machine — re-runs typically span 25-32x for fact-iter.
