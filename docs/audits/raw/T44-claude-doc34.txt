# Doc 34 review (Phase 9b extension substrate, DRAFT v0)

## verdict: MAJOR REVISIONS

DRAFT v0 が sibling LOCKED v2 docs (28-32) の format / depth に揃える努力は
みえるが、本 doc は Phase 9b の親仕様 Doc 33 §4.2b を *機能拡張* した提案
(window/frame/keymap/event-loop を §4.2b の bullet にない 4 sub-system として
追加) を行いながら、これを Doc 33 errata 不要と宣言している点が最大の構造
問題。これは codex review が parent doc 整合性 (Doc 28 v2 観点 1 系列) で
最も厳しく見る点。Phase 9a (Doc 33 §4.2a top 20-30 API) の boundary も未整理
で、API double-counting と category 矛盾が複数。

DRAFT v1 への revise を出した上で v2 LOCKED 化を推奨。

## 観点 2: Phase 9b scope 妥当性

### [MAJOR-2.1] Phase 9a (Doc 33 §4.2a top 20-30 API) との API boundary が未確定 = double-counting

Doc 33 §4.2a (line 547-569) で top 20-30 APIs として列挙されている API:

- search: =re-search-forward= simple subset 4 件

- *undo*: =buffer-undo-list= primitive のみ
- file: =with-temp-buffer= 等 3 件

= text-property と undo の primitive が *Phase 9a に既に含まれる*。

ところが Doc 34 §4.6 (line 670 の text-properties 12 件) と §4.8 (line 695
の undo 6 件) で同 API がそのまま再列挙され、§3.1.2 / §3.4 では Phase 9b で

=src/nelisp-emacs-compat-textprops.el=", line 545 "新規 module:
=src/nelisp-emacs-compat-undo.el=")。

reasoning: Phase 9a で =put-text-property= / =get-text-property= /

"Phase 9a SHIPPED" を prerequisite と明記)、なのに同 API を Phase 9b で
"新規実装" すると衝突。Phase 9a での *基本のみ* と Phase 9b の *拡張* の
boundary が API list level で見えない。

proposal:
1. §4 各 API table の前に "Phase 9a 既に SHIPPED / Phase 9b 拡張 / Phase
   9b 新規" の 3 状態 column を追加
2. §11 v0→v2 想定 refine (line 1128 "Phase 9a (Doc 34a) との API 境界の
   明示") で "76 API のうち N 件は Phase 9a SHIPPED、残 M 件が Phase 9b
   新規 / 拡張" の overlap 表を必須化
3. text-properties は "Phase 9a で primitive 2 件 (`put-/get-`)、Phase 9b で
   拡張 10 件 (interval tree + change hook + propertize 等)" のような明示
   分割

## 観点 3: display engine 依存境界

### [MAJOR-3.1] §1.3 依存境界 table の "△" が semantic 不明 + window の display 依存度が過小評価

§1.3 (line 107-116) の依存度 table:
- window = △ (logical buffer 紐付けのみ、render は frame 経由)
- event-loop = △ (event source が display backend で発生)
- *frame* = ○ (display backend が frame handle 提供)

window が "△" とされているが、§3.3.1 (line 481-509) で window split tree /
window-point / window-start / 各種 resize op が記述されており、これらは
display backend の *layout primitive* を呼び出さなければ実装不可能 (=
Doc 33 §6 line 871-876 の =nelisp-display-layout-box= 等)。本 doc Phase 11
依存度を "弱 (frame 経由 indirect)" と書くのは Phase 11 SHIPPED 前 stub
backend で window resize / split が "意味のある動作" を ERT で確認できる
保証がない。

reasoning: Doc 33 §6.4 layout primitive (line 871) は Phase 10 spec で確立、
Phase 11 backend が実装。window split は frame size を分割計算するため
*frame size を知る = stub backend が固定値返却* と組み合わさってのみ意味
あり。本 doc line 533 "stub backend で ERT 駆動" だけでは window resize の
correctness verify が困難。

proposal:
- §1.3 table の "△ / ○ / ×" を 3 段階 (= no dep / indirect dep via frame /
  direct dep on backend) で精緻化、各 sub-system に "stub backend 動作可否"
  + "Phase 11 SHIPPED 後 integration test 必須かどうか" の 2 列を追加
- window を "indirect via frame、ただし stub backend では size 固定 (= 80
  x 24) を仮定して ERT 駆動可" と明示
- §9.3 (line 1023 frame backend stub mode semantics) に window size
  invariant を明記、stub mode で `(frame-width)` = 80 / `(frame-height)` =
  24 fixed が window split-window 計算に伝播することを spec 化

### [MAJOR-3.2] event-loop の event source contract が under-specified、§2.4 LOCK item が schema sketch only

§2.4 (line 200-225) で event source contract を schema sketch のみ (= 3
function = register / unregister / pull-next) で提示。

Doc 28 v2 §2.9 (line 306-329) GC metadata contract のように "metadata 形式
固定 + version constant + Phase 7.3 改訂時 migration path" と detail level
で書かれた precedent あり。本 doc §2.4 は (a) event 構造体の field (= key
code / mouse coord / window resize delta 等) が未定義、(b) timeout 単位
(秒 / ms / μs)、(c) 並列 event source の優先順位、(d) backpressure model
(= queue 溢れ時の drop policy) すべて未指定。

reasoning: Phase 11 backend (TUI keyboard input + SIGWINCH) と mock backend
(ERT 駆動) を差し替え可能と書きながら、両 backend が同一 event 構造を
emit する保証が contract 明示なし。

proposal: §2.4 schema を以下まで拡張:
1. event 構造体 typed alist 例 (key event / mouse event / window event /
   timer event の 4 variant)
2. timeout 単位 = ms (= Emacs `read-event` precedent と integer 整合)
3. 並列 source 優先順位 = register 順 (FIFO) で fixed
4. queue capacity + overflow drop policy
5. event source の lifecycle (= register 失敗 / unregister 中の pull) error
   condition

### [MAJOR-3.3] frame stub backend 動作 invariant が §9.3 open question 止まり、§3.3.2 SHIPPED 判定基準が回避不能

§3.3.2 (line 511-541) で frame は "stub backend (= no-op or text-only) で
Phase 11 SHIPPED 前は動作確認可" と書きながら、stub mode の invariant 仕様

これだと §6.1 risk (line 773 "Phase 11 SHIPPED 遅延で frame sub-system の
SHIPPED 判定が incomplete") の mitigation が成立しない。stub mode は SHIPPED
判定基準の中核なので、§9.3 で open question にしたまま v0 を main merge
すると、stub mode 仕様が後付けで Phase 9b.3 SHIPPED 解釈を歪める risk。

reasoning: Doc 28 v2 §6.1 (register allocation 性能 risk) は mitigation で
"graph-coloring 後付け可能な構造を §3.1 で確保" + "scope 削減 emergency
procedure invoke 可能化" の二段構え。本 doc §6.1 は stub mode 後付け仕様化
だけで mitigation 完結せず。

proposal:
- §9.3 open question を §2.X (LOCKED items) に昇格、stub mode invariant を
  v0 で LOCK (= frame-width 80 fixed, frame-height 24 fixed, make-frame =
  unique frame-id 返却, delete-frame = id reuse 不可)
- §6.1 mitigation に "stub mode invariant が §2.X で LOCK 済 = Phase 11
  SHIPPED 前後で window 計算 invariant 維持" を 1 句追加
- §3.3.2 SHIPPED 判定 (line 533) に "stub mode invariant 全 PASS で 9b.3
  暫定 SHIPPED、Phase 11 SHIPPED 後 actual integration test 追加で full

### [POSITIVE-3.4] §1.3 で "残 6 sub-system は display 非依存で完結 = Phase 11 と独立に進められる" の主張は健全

§1.3 line 127-128 "window / keymap / buffer-local / text-properties /

## 観点 4: risks / gates

### [MAJOR-4.1] §6.6 buffer-local thread-safety risk が "scope 外 mitigation" だけで済まされている (Phase 7.5+ co-design)

§6.6 (line 850-861) buffer-local thread-safety risk の mitigation:

"Layer 0 = dialect-agnostic backend" と Layer 0/1 separation を design
level で意識する方針を確立、Phase 8+ (本 doc 含む) で "新 code は Layer 0/1

7.3 連動 (line 743-758) の versioned contract pattern と同様の hygiene が
buffer-local でも必要。

reasoning: codex review で Doc 28 §6.10 (Phase 7.3 ABI 連動) は MAJOR
revision で追加された前例あり。Doc 34 buffer-local も同 hygiene なしでは
"single-thread MVP" を Phase 7.5+ で改訂不能の roadblock 化。

proposal:
- §6.6 mitigation に "(a) buffer-local hash の access API を thread-id
  aware な abstraction (= `nelisp-buflocal-with-thread`) で wrap、Phase
  9b 単体では single-thread default 実装、(b) Phase 7.5+ で abstraction

  buffer-local API contract 整合性 (= breaking change なしの retrofit

### [MAJOR-4.2] §6.7 anvil leverage 過大評価 risk の M3/M6/M9 replan gate が Doc 28 §6.6 と非対称 (kill criteria が milestone defer のみで scope 縮小を許さない)
