** §2.1 native compiler (T49) — 4 critical

nelisp-cc は T50 letrec SIGSEGV の root cause を抱えており、bench gate を
通過した form は "tested form" のみで、real-world Elisp の majority は
miscompile する。

*** §2.1.1 [CRITICAL T49-C1] x86_64 if branch layout が壊れている

- *file:line*: src/nelisp-cc.el:627-662, src/nelisp-cc.el:1369-1402,
  src/nelisp-cc-x86_64.el:898-934, src/nelisp-cc-x86_64.el:1180-1189
- *summary*: backend は THEN fallthrough 前提で =JE else= を吐くが、
  current DFS RPO は通常 =entry, else, then, merge= 配置。x86_64 の
  ordinary =if= が常に逆動作する。
- *impact*: T50 letrec SIGSEGV 根本原因の 1 つ、bench gate value-verify
  fail の主因。
- *proposal*: lowering を layout-independent に変更 (両 edge を explicit
  emit) または THEN-next layout を強制。executable =if= diamond ERT 必須。

*** §2.1.2 [CRITICAL T49-C2] arm64 が =:jump= 命令を完全に無視

- *file:line*: src/nelisp-cc.el:1390-1396, src/nelisp-cc.el:1585-1607,
  src/nelisp-cc-arm64.el:1074-1077, src/nelisp-cc-arm64.el:1127-1133
- *summary*: frontend が =jump= terminator を出すが arm64 dispatcher は
  no-op 化。CFG edges (=if= merge / =while= back-edge) が emitted code
  に消滅。
- *impact*: arm64 で then-block が else-block にフォールスルー、loop が
  header に jump 戻れない = arm64 verify gate fail。
- *proposal*: =jump= を =B <label>= fixup に lower。executable =while=
  back-edge ERT 必須。

*** §2.1.3 [CRITICAL T49-C3] call-crossing 値が両 ISA で破壊される

- *file:line*: src/nelisp-cc.el:815-862, src/nelisp-cc-x86_64.el:101-106
  / 113-121 / 843-896 / 1013-1084, src/nelisp-cc-arm64.el:129-156 /
  878-920 / 980-1043
- *summary*: 両 backend pool が caller-saved register のみ
  (x86_64 rdi..r11、arm64 x0..x7、全 call-clobbered)。linear-scan は
  call-crossing interval を split/spill しない。
- *impact*: call を跨ぐ live value が silent destruction、bench gate で
  単純 form 通過 / 複合 form silent miscompile。
- *proposal*: call-aware allocation または call 周りに explicit
  save/restore。

*** §2.1.4 [CRITICAL T49-C4] GC metadata が moving collector で使えない

- *file:line*: docs/design/28-phase7.1-native-compiler.org:312-321,
  src/nelisp-cc-runtime.el:501-514 / 530-560 / 575-585 / 600-609
- *summary*: spec は physical register / spill slot の root location +
  frame layout descriptor を要求するが、実装は SSA value-id の bool-vector
  + =:frame-size 0=。
- *impact*: Phase 7.3 GC inner (Doc 30) で moving collector を有効化した
  瞬間に native code が live root を再構築できず crash。standalone v1.0
  ship 不能。
- *proposal*: post-allocation safe-point table を physical register +
  spill slot 基準で生成、backend-specific frame descriptor 追加。

*** §2.1.5 関連 MAJOR (T49-M1..M5)

- M1: SSA verifier が dominance / phi predecessor 未検査 (line 461-542)
- M2: phi lowering が parallel-copy 安全でない (line 949-969 / 1003-1057)
- M3: loop-carried liveness が back-edge を跨いで extend されない
  (line 571-576 / 680-681 / 1553-1581)
- M4: x86_64 tail-call rewrite が spill frame > 0 を check せず

- M5: safe-point :pc-offset / :frame-size が rewrite 後に stale

- M6: closure pointer が GC root として relocate されない

** §2.2 allocator + gc-inner (T51) — 4 critical + 4 major

Phase 7.2 / 7.3 schema bridge が "memory contract レベルで存在しない"
状態。Doc 29 §6.10 / Doc 30 §2.13 (object header v2.13) と実装が乖離。

*** §2.2.1 [CRITICAL T51-C1] heap-region schema bridge 未実装

- *file:line*: src/nelisp-allocator.el:258-270, src/nelisp-gc-inner.el:236-249,
  src/nelisp-gc-inner.el:649
- *summary*: allocator は =nelisp-heap-region= struct + bare symbol を返す。
  gc-inner は =:region-id/:generation/:family= plist keyword schema
  しか受理しない。bridge =nelisp-allocator-snapshot-regions= は spec で
  約束されているが実装が存在しない。
- *impact*: Doc 29 §6.10 heap-region registry v1 の provider-consumer
  接続が成立しておらず、allocator 実体を gc-inner が consume できない。
- *proposal*: bridge を 1 箇所に実装、version/schema/family keyword 化を
  そこで強制。allocator 実体経由 interop ERT 追加。

*** §2.2.2 [CRITICAL T51-C2] family-tag enum mismatch (1..5 vs 0..4)

- *file:line*: src/nelisp-allocator.el:97-105, src/nelisp-gc-inner.el:2073-2094
- *summary*: allocator family-tag は 1..5、gc-inner decode は 0..4。
  tag=1 cons は closure と誤読、tag=5 large-object は unknown フォールバック
  で =:cons-pool= 化。
- *impact*: promotion / header decode / type-family dispatch が silent
  に壊れる。Doc 30 v2.13 family-tag invariant 違反。fail-fast せず
  silent corruption。
- *proposal*: family enum を単一定義に統合、両 module で共有。unknown
  fallback は禁止して error に上げる (fail-fast)。

*** §2.2.3 [CRITICAL T51-C3] nursery object header 未書込

- *file:line*: src/nelisp-allocator.el:542-654, src/nelisp-allocator.el:339-365
- *summary*: nursery で header を実書込みするのは tenured path のみ。
  =alloc-closure= / =alloc-string-span= / =alloc-vector-span= /
  =alloc-large-object= は bytes 予約のみ、=alloc-cons= は明示的に
  headerless。
- *impact*: Doc 29 §2.5 / Doc 30 §2.13 前提の family/size/forwarding
  情報が nursery object に残らず、GC が allocator-produced object を
  walk/forward/promote できない。
- *proposal*: 全 heap object の header ABI を実装、または side-metadata
  ABI を正式 contract 化して両 module で共有。

*** §2.2.4 [CRITICAL T51-C4] semispace overflow 検査なし

- *file:line*: src/nelisp-gc-inner.el:1172-1188, src/nelisp-gc-inner.el:1337-1360
- *summary*: =init-semispace= は from/to の同容量を検証せず、
  =--minor-copy-one= は =free-pointer= を =to-space :end= と比較しない。
- *impact*: live set が to-space を超えると minor GC が黙って semispace 外
  へ進み、Cheney copy が silent corruption。24h soak で必ず発覚する class。
- *proposal*: semispace init 時に equal-size 必須化、copy 前に
  =free + size <= to-end= 検査で fail-fast。

*** §2.2.5 関連 MAJOR (T51-M1..M4)

- M1: =collect-roots= が region.start を synthetic root 化 = nursery /
  tenured / large-object 各 region 先頭 object が常時 live (line 408-436)
- M2: =run-minor-gc-with-promotion= の returned :forwarded-roots が
  promotion 後の rewrite を反映しない (line 2211-2279 / 2099-2195)
- M3: scheduler が allocator hot path に hook されておらず、allocation
  rate に基づく minor/major heuristic が成立しない (allocator.el:511-538 /
  863-895、gc-inner.el:2563-2582)
- M4: tenured exhaustion が =nelisp-allocator-tenured-overflow= signal
  のみで emergency GC retry path なし (allocator.el:770-895)

** §3.2 C bindings / Emacs module FFI (T58)

*** §3.2.1 [CRITICAL T58-C1] partial dlsym poison

- *file:line*: nelisp-runtime-module.c:127-217
- *summary*: =dlopen= 成功直後に =runtime_handle= を global 確定、その後
  の =dlsym= 失敗時に handle を戻さない。次回呼出は
  =runtime_handle != NULL= で即成功扱い、未初期化 sym_* で NULL function
  pointer crash。=dlclose= 不在で handle leak も発生。
- *impact*: 一度の partial load で module が永続 poison state、subsequent
  calls がプロセス致命。
- *proposal*: local handle で全 dlsym 成功後に global へ commit、失敗時
  =dlclose(local)= + 全 sym_* NULL 化。

*** §3.2.2 [CRITICAL T58-C2] non-local exit pending 中の Emacs API 続行

- *file:line*: nelisp-runtime-module.c:116-123, 252-275, 307-312, 356-359
- *summary*: =non_local_exit_signal= 後に =env->intern= /
  =env->make_integer= を継続呼出。Emacs module ABI 違反 (pending exit
  中は API 呼出禁止)。
- *impact*: error path 自体が ABI 違反、戻り値の生成や intern が未定義
  動作寄り。
- *proposal*: =signal_error= 後は即 return、返り値は事前用意の =nil=
  固定。

*** §3.2.3 [CRITICAL T58-C3] in-process JIT 実行に fault containment ゼロ

- *file:line*: nelisp-runtime-module.c:343-345, 401-442
- *summary*: 任意 bytes を Emacs プロセス内でそのまま実行、CPU fault /
  abort / Rust fatal failure を Emacs signal へ変換する経路なし。
  =release= profile に panic policy も明示なし。
- *impact*: bad code path で Lisp error ではなく Emacs プロセス全体が
  落ちる。エラー契約コメントが実装より強い。
- *proposal*: comment を現実 (= CPU fault は fatal) に弱める、または
  execute 部を subprocess 隔離。Rust panic policy 明示。

*** §3.2.4 関連 MAJOR

- [T58-M1] =bind_function= / =provide_feature= が funcall non-local
  exit を check せず、=emacs_module_init= が常に 0 success 返却
  (nelisp-runtime-module.c:453-515)。partial init success が module
  load 成功と誤判定。
- [T58-M2] BYTES の unibyte 性検証なし、multibyte 入力で JIT bytes
  silent 変質 (line 337-345 / 361-385)。
- [T58-M3] C 側 =long (*)(void)= vs Rust =extern "C" fn() -> i64= の

** §3.3 defs-index + org-index (T59)

*** §3.3.1 [CRITICAL T59-C1] rebuild non-atomic + silent error swallow

- *file:line*: src/nelisp-defs-index.el:605-617,
  src/nelisp-org-index.el:471-482
- *summary*: rebuild が最初に全削除 commit 後にファイル単位で再投入。
  途中失敗で live DB が empty/partial state、しかも
  =condition-case _ (error nil)= で握り潰し。破損 source / lock failure
  が silent data loss。
- *impact*: rebuild 中の reader が partial DB を見る、subsequent search
  結果が不正。silent data loss path。
- *proposal*: shadow DB or single-transaction snapshot swap で all-or-nothing
  化、=condition-case= の error 握り潰し撤去。

*** §3.3.2 [CRITICAL T59-C2] WAL のみ + busy_timeout / BEGIN IMMEDIATE 不在

- *file:line*: src/nelisp-defs-index.el:227-228 / 153-159,
  src/nelisp-org-index.el:211-212 / 140-146
- *summary*: =journal_mode=WAL= は入っているが、=busy_timeout= / retry
  / =BEGIN IMMEDIATE= がない。multi-process write 競合で =SQLITE_BUSY=
  即失敗、しかも rebuild loop 側が黙殺。
- *impact*: Emacs server / emacsclient / subprocess 並行運用で不安定。
- *proposal*: =busy_timeout 5000= 設定 + =BEGIN IMMEDIATE= 移行 +
  retry policy。error 黙殺撤去。

*** §3.3.3 関連 MAJOR

- [T59-M1] org-index 検索が =LIKE '%term%'= のみ、=idx_headline_title=
  B-tree が効かず CJK trigram 未対応 (Doc 26 §6.6.3 前倒し必要)。

- [T59-M3] hidden dir / symlink policy が =git ls-files= 成功時と

* §8 audit hygiene retrospective

** §8.1 review pattern 強み

- *Claude + codex 独立 review consensus*: Doc 34 (T44/T47) / Doc 35
  (T45/T55) で同一 doc を 2 reviewer 独立投入し、共通 finding を §4.2 /
  §4.3 で抽出可能。confirmation bias を避ける structural pattern。
- *4 観点 critique format* (整合 / 妥当 / 依存 / risk-gate) で finding
  比較が容易、cross-reviewer での finding mapping (= 共通 / 単独) が
  自然に整理できる。
- *file:line + verdict + impact + proposal の 4 つ組* で actionability
  が高い、subsequent fix task (T63-T72) の scope 定義に直接使える。

** §8.2 review pattern 弱み

- *codex sandbox の write 制約*: =/tmp/*.md= 直接 write 不能で stdout
  経由 capture が必要。本 task の deliverable 永続化動機。
- *DRAFT v0 が main 未 merge*: codex は path resolve に
  worktree branch 切り替えが必要、初回 dispatch (T48) で file-not-found
  発生。retry で resolve。
- *ERT fixture realism gap*: T53 C1 (orchestrator routing alist 化
  fixture) が "本番 shape を再現しない ERT" の典型例。fix だけでなく
  ERT realism も check する layer が必要。
- *audit-of-audit*: 本 task 自体は audit を summarize するだけで、
  個々 audit の精度 (false positive / negative) を再評価していない。
  high-stakes finding は 3rd reviewer 推奨。

** §8.3 improvement proposals

1. *codex sandbox を =workspace-write= 化*: worktree 内なら write 可能
   にする dispatch profile を anvil-orchestrator に追加。本 task で
   /tmp 経由を強いられた弱点を構造解消。
2. *audit doc の git-managed 永続化* (= 本 task の deliverable):
   docs/audits/YYYY-MM-DD-master-audit.org + docs/audits/raw/ で baseline
   を残す pattern を確立、future audit iteration の比較 baseline 化。
3. *ERT fixture realism check*: anvil-dev-release-audit に "ERT fixture
   が本番 data shape を再現するか" の grep scanner を追加 (= hash-table
   vs alist の混在検出など)。
4. *spec-vs-impl drift gate*: Doc 28-32 の =contract version constant=
   を一覧化、impl 側 grep で version mismatch を CI で検出する gate
   追加。
5. *3rd reviewer escalation*: CRITICAL verdict の audit は最低 2 reviewer
   独立 + user signoff、MAJOR REVISIONS は 1 reviewer + user で OK の
   2-tier policy 化。
