* AOT grammar ops (`lisp/nelisp-aot-compiler.el`)

AOT source は =(defun NAME (...) BODY)= で書かれ、emit する。
`BODY` 内で使える primitive 群:

| Op                       | Semantics                                          | 出処               |
|--------------------------+----------------------------------------------------+--------------------|
| =(sexp-tag PTR)=         | Sexp の tag byte (offset 0) を読む                 | 既存               |
| =(sexp-payload-ptr PTR)= | **Cons (tag=7) only**、offset 8 から box ptr 読む    | 既存 (Cons-only)   |
| =(sexp-payload-ptr-record PTR)= | Record (tag=12) only、offset 8 から box ptr     | R6g `56a7c575`     |

新 grammar op 追加コスト = elisp ~25-40 LOC (parser + dispatch + emit)、
Rust LOC 0、R6g pattern を流用。

* よく踏む罠 (= 過去 subagent の教訓)

** Relative path probe の `default-directory=/tmp/` 罠
`./target/release/nelisp --batch -l /tmp/probe.el` で probe 内

** `condition-case t` を escape する error
`nl-jit-call-out-1` 失敗時の inner =(error ...)= → 再 signal が
外側 =condition-case= を escape する secondary bug あり (= R6h subagent
確認、=nl-jit-call-out-1= + Record carrying error data + 再 signal の
三重組合せで発火)。回避は inner で signal せず boolean を返す pattern。

** AOT emit `nelisp_jit_record_*` の latent hang
~50 globals-table-insert cycle 超で deterministic hang (R6h stress
test = iter 14 で再現)。**caller を `nl_jit_record_*` (typo) のまま
にすると常に dlsym miss → wrapper fallback signal で hide される**
古典 masking pattern。R6h rename で露呈、R6h reverted。

** =[profile.release] strip = true=、gdb 未 install
RUST_BACKTRACE=1 で elisp signal は反応しない。core dump trace は
`pyelftools + addr2line + objdump` で復元可能 (= R6h subagent 実績)。
gdb 必要時は `Cargo.toml` で =strip = false; debug = 2= 一時切替。

** Rust LOC「never increase」 hard rule (2026-05-19 設定)
全 fix は Rust LOC 0 が基本。waiver は user 明示承認時のみ、最小 LOC
(= R6j で 3-10 LOC waiver 試行中)。

* Wave 9 perf baseline (= main ed6659b9 実測 2026-05-23、subagent verify 短縮用)

** Lambda cost mystery (= R7 後の追加調査結果)
Wave 9 profile が「1.14s/lambda × N の linear」と推定したが claude main 実測:

| Probe                                            | N=1   | N=5   | N=10  | N=20  |
|--------------------------------------------------+-------+-------+-------+-------|
| =loop-only=                                      | 1.25s | 1.25s | 1.25s | 1.25s |
| =(cons 1 2)=                                     | -     | 1.23s | -     | -     |
| =(lambda () 42)= (bare)                          | 2.13s | 21.0s | 20.9s | 21.0s |
| =(let nil (lambda () 42))= (wrapped)             | -     | 6.19s | -     | -     |
| =(function (lambda () 42))= (explicit fn)        | -     | 6.14s | -     | -     |
| =(let ((c i)) (lambda () c))= (capture)          | -     | 1.25s | -     | -     |
| =(quote (lambda () 42))= (no eval)               | -     | 1.24s | -     | -     |

**観測** = 真の cost は **bare =(lambda ...)= の N≥5 で 21s plateau**。
N>=5 saturates (= per-lambda growth ではない、特定 path の one-time cost)。
=(let ((c i)) (lambda () c))= で captured-var 形にすると 1.25s に落ちる
= 真因は **captured nothing の bare lambda evaluator path 専用 cost**。
profile 仮説「depth=0 が遅い」も実は **lambda evaluator の特定 branch
が一度走ると 20s** で saturate していたのを「lambda 当たり 1s × N」と
誤推定していた可能性大。

R7 が 14% speedup (99.9→85.9s for N=100) だったのは、N=100 で plateau
21s + per-iter cost を測っており、per-iter 部分の改善が部分的に効いた。
R7 を N=5 で probe するなら 21s baseline → 6s 等の改善幅で見えるはず。

* Bootstrap self-host workflow (R6L, 2026-05-23)

NeLisp の build 経路は GCC 自己ホスト bootstrap pattern と同形:

** 1st build (= default、emacs bootstrap)
=cargo build --release= はそのまま動く (= 回帰なし、casual dev 用)。
=build.rs= が =emacs --batch= で =compile-elisp-objects-emit-all= を
走らせて =elisp-objects/*.o= 生成、linker が nelisp binary v1 に link。

所要 = 既存通り (= 数秒〜30秒)。

** 2nd build (= 任意、self-host validation)
```
NELISP_BUILD_WITH=nelisp cargo build --release
```
=target/release/nelisp= (= v1) が =compile-elisp-objects-emit-all= を
self-host で走らせて .o 再生成 → linker が nelisp binary v2 に link。

所要 = **数時間レベル** (= R6k 確認、closure capture cost で super-linear
slow)。byte-identical .o 期待 (= deterministic emit)、CI / release で
1 回走らせて self-host 健全性確認の用途。

** auto-mode
```
NELISP_BUILD_WITH=auto cargo build --release
```
=target/release/nelisp= 存在 → nelisp self-host、無 → emacs bootstrap に
fallback。CI で previous build artifact を持ち越す flow 用。

** env-var 一覧
| =NELISP_BUILD_WITH= | 動作                                  | 所要        |
|---------------------+---------------------------------------+-------------|
| unset / =emacs=     | =emacs --batch= (default)             | 〜30s       |
| =nelisp=            | required =target/release/nelisp=、無で panic | 数時間 |
| =auto=              | nelisp 存在で nelisp、無で emacs       | 1st 〜30s、2nd 数時間 |

** Wave 10 結論 (= 「elisp-only 15.6% が実用最大値」確定)

R11 の 15.6% は **「Rust LOC 0 + AOT grammar 現状」 で elisp-only
達成可能な実用最大値**。それ以上の speedup には以下のどれか必要:

- **AOT grammar 拡張 wave** (~250-500 LOC elisp、1-2 wave):
  hash-table primitive / array index lookup / static table emit を
  grammar に追加 → W10.1d/e の re-dispatch unblock
- **Rust LOC waiver** (~5-30 LOC、user hard rule 緩和):
  capture_lexical / lookup_function 等に direct fast path、~2-5x speedup
- **architectural rework** (= daemon / persistent / shared image、multi-wave):
  startup cost 償却、self-host build 実用化に必要

= [[feedback_nelisp_rust_loc_never_increase]] rule の根本的見直しが
trade-off の核心。次 session 開始時に user と再 align。
