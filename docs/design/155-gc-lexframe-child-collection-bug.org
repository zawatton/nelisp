#+TITLE: Doc 155 - GC collects live lexframe children (audit/lock-check SIGSEGV)
#+DATE: 2026-06-22
#+STATUS: OPEN — root cause NARROWED (§8.6: a stray write zeroes a live cloned Sexp box between frame_push and find, ONLY after arena growth; allocator/read/clone all exonerated by exit-code probes); stray-writer pinpoint + fix pending
#+SEVERITY: latent (masked in production by Doc 154 churn-reduction work)

* 1. Purpose

Track the dedicated effort to fix a garbage-collector memory-corruption bug in
the standalone reader (=target/nelisp=): under allocation churn during a deep
eval, the GC reclaims objects that are still reachable from a live lexical
=frame= (its =hash-table= / =buckets= vector), producing a NULL dereference on
the next variable lookup.

This was found while investigating the standalone =nelix audit= / =nelix
lock-check= SIGSEGV.  The *symptom* is currently masked in production (see §5);
this doc captures the *root cause* and an investigation plan so the GC fix can
be picked up as a focused, instrumented task rather than blind surgery.

* 2. Symptom

- =nelix audit MANIFEST= and =nelix lock-check MANIFEST= under the nelisp
  runtime (nelix-native backend, e.g. the 3-package =m3= manifest) crashed with
  SIGSEGV (rc=139), deterministically without gdb (5/5).
- =nelix lock= (standalone) returns rc=1 with empty output — this is a SEPARATE
  env issue (registry recipe resolution needs network/cache), NOT this GC bug.
  Do not conflate them.

* 3. Root cause (gdb core analysis)

Faulting instruction (core dump): =nl_vector_slot_ptr=, =mov (%rdi,%rsi,1),%rax=
with =rdi=0x0= (NULL base), =rsi=0x8=.

Call path:
#+begin_example
#0 nl_vector_slot_ptr            ; NULL deref
#1 nelisp_frame_stack_find_in_frame
#2..#6 nelisp_frame_stack_find_descend   ; innermost-first frame walk
#7 nelisp_frame_stack_find
#8 nelisp_env_lookup_value       ; variable lookup
#10 nl_eval_inner / nl_eval_arg_list_walk
#+end_example

A live lexframe's CHILD object is reclaimed by the GC while the frame Record
itself survives:
- =find_in_frame= walks =frame.slot0 (hash-table) -> slot1 (buckets) -> cells=.
- Guard experiments (lisp/nelisp-cc-frame-stack-find.el):
  - Guarding "frame Record is Nil" did NOT stop the crash → the frame Record is
    valid (tag != 0).
  - Guarding "frame.slot0 (hash-table) is Nil" did NOT stop the crash → the
    hash-table is valid; the *buckets* vector (=ht.slot1=) reads back NULL.
  - The reclaimed object VARIES (hash-table in one repro, buckets in another).
- ⇒ The whole =frame → hash-table → buckets → cells= chain is not kept alive
  across a collection.  Single-point NULL guards are whack-a-mole and
  insufficient.

The GC mark code *structurally* recurses this chain
(=nl_gc_mark_slot= for tag-12 Record → slots; tag-8 Vector → buf + slots; tag-7
Cons), and the root marker =nl_gc_mark_published_frame= marks
=(nl_gc_mark_slot (+ env 32))= (the frame stack).  Yet live objects are
collected → the gap is in GC *root traversal* (the =env= published to the
collection during a deep eval likely does not reach the live frame stack's full
child chain) or in *sweep*.  Pinpointing it needs GC-mark instrumentation
(see §6).

* 4. Ruled out

- Per-eval arena reset: disabled; =nelisp_eval_call= only does recursion-depth
  bounding (rec_cur/rec_max), no arena reset.
- Stack overflow: bigger stack (=ulimit -s 524288=) still crashes 5/5.
- ASLR alone: =setarch -R= (ASLR off) still crashes 5/5 (gdb avoids it via fork
  detach / timing, not ASLR — =disable-randomization off= under gdb does not
  crash either).
- =find_descend= depth off-by-one: a Nil-frame skip guard did not prevent it.
- Hash-table-only corruption: buckets are also observed NULL.

* 5. Why it is masked in production

Two Doc-154 changes reduced per-eval allocation churn enough that the GC no
longer fires at the vulnerable moment for these paths:
- nelisp =3f318889= "alloc-free builtin dispatch" (removed per-comparison
  =nl_alloc_symbol= churn).
- nelix =c435946= "parse standalone lock with native reader" (removed lock-read
  text-scan churn).

On the current feat / =origin/main= binary, =nelix audit= and =nelix lock-check=
are segfault-free (rc=0, verified m1+m3, multiple runs).  The underlying GC bug
is therefore LATENT: any future high-churn workload that triggers a GC during a
deep frame-stack eval can resurface it.

* 8. Refined findings — 2026-06-22 instrumentation session

Reproduced =nelix audit m3.el= at =838605da= (5/5 SIGSEGV) and traced GC state.
This SHARPENS §3: the crash is *not* a live object freed by a running collector
— *reclamation is disabled by design*.

** 8.1 All reclamation paths are OFF during audit

- Boundary collect =nl_gc_collect= is a NO-OP: arena-init sets the "reclaimer
  gate" =base+160 = 1= (=scripts/nelisp-standalone-build.el= ~line 989), comment:
  "Keep collection disabled until the standalone root/mark gap is fixed".  The
  root/mark gap is a KNOWN, deliberately-worked-around issue.
- Per-eval / boundary reclaim is DISABLED (Doc 140: "escape-epoch reset is
  unsound", =arena-boundary-reclaim-enabled-offset = 0=).
- The precise mark+sweep =nl_gc_collect_published= (via =nl_gc_safepoint=, called
  from =nl_sf_while='s backedge, 16 MiB trigger) is OFF by default — ARMED only
  by the =(nelisp--gc-diag 5)= test hook (=bf_gc_diag= mode 5), disarmed by mode
  6.  Normal =nelix audit= never arms it.

⇒ During audit NO collector runs; the arena only GROWS.

** 8.2 CONFIRMED: arena chunk-growth corruption (frame-object-specific)

The Linux arena is a 256 MiB FIRST CHUNK (=nelisp-standalone--linux-arena-size=,
=#x10000000=) plus =nl_chunk_alloc_new= growth chunks.  With no reclamation,
audit's deep eval (~606 frames) + packaged-registry processing keeps allocating
and crosses the 256 MiB first chunk → chunk growth.

CONFIRMED by the arena-size experiment (rebuild with =NELISP_LINUX_ARENA_SIZE=,
forcing a rebuild via =touch scripts/nelisp-standalone-build.el= since a plain
=make standalone-reader= does NOT re-pick the env):
- *512 MiB first chunk → =nelix audit m3= is 5/5 rc=0* (audit fits in one chunk,
  no growth), output correct.  vs *256 MiB → 5/5 SIGSEGV*.
  ⇒ The crash IS arena chunk-growth corruption.
- =2 GiB= first chunk produces a non-working binary (empty output) — =0x80000000=
  hits the signed-imm32 address-materialisation boundary; keep test sizes
  < 2 GiB (512 MiB is a good probe).

The corruption is FRAME-OBJECT-SPECIFIC, not generic:
- A plain cons churn that DOES force growth (700k conses ≈ 22 MB with a 16 MiB
  first chunk) SURVIVES (rc=0).  So =nl_chunk_alloc_new= growth works for plain
  cons allocation.
- Yet audit's growth nulls a lexframe child (hash-table / buckets — a
  Record/Vector reached via =record-slot-ref-ptr=).  So the bug is in how
  growth (or a pointer computed relative to the first-chunk base) handles
  RECORD / VECTOR frame objects specifically, OR an object that straddles /
  is reached across the chunk boundary.

This explains the MASKING precisely: Doc 154 churn reduction (=3f318889=
alloc-free dispatch removed ~per-comparison symbol allocs across the 606-frame
eval; =c435946= native lock-read) lowers audit's allocation enough to stay
inside the 256 MiB first chunk → no growth → no corruption.

** 8.3 Reproducer notes (what does NOT trigger it)

- Plain deep =let= recursion (300+), =while= loops, explicit =(garbage-collect)=
  (mode 0 — does a conservative scan), and *cons churn that forces growth* all
  complete cleanly.
- The trigger needs BOTH (i) allocation crossing the first-chunk boundary AND
  (ii) live lexframe RECORD/VECTOR children reachable across the growth.  The
  audit/manifest-load + packaged-registry eval shape supplies both; a minimal
  repro should pair a deep frame stack with > first-chunk allocation (tune a
  small first chunk + a frame-heavy loop allocating records/vectors).

** 8.4 Next steps (supersede §6)

1. [DONE] Arena-growth confirmed (512 MiB avoids it; cons-growth survives →
   frame-object-specific).
2. Find the growth bug: audit =nl_chunk_alloc_new= / =nl_alloc_bytes= +
   =nl_gc_in_arena= / =nl_gc_chunk_contains_any= for RECORD/VECTOR objects on a
   growth chunk — esp. any pointer/membership test computed relative to the
   FIRST-chunk base (=record-slot-ref-ptr= / =vector-ref-ptr= reaching a child
   that landed on a growth chunk).  Compare the cons path (works) vs the
   record/vector path (breaks) for what differs across the chunk boundary.
3. Proper fix: land a SOUND reclaiming GC (fix the root/mark gap so
   =nl_gc_collect= can be re-enabled, keeping the arena bounded so growth is
   rarely/never needed).  Existing machinery to make sound:
   =nl_gc_mark_published_contexts= (precise roots, capacity 64; the frame stack
   is a shared backing, so one published env marks all frames) + the mode-0
   conservative scan (=nl_gc_conserv_maybe=).
4. Interim mitigation option: a larger first chunk reduces growth frequency but
   is not a fix (any unbounded no-reclaim workload eventually grows).

** 8.5 Instruction-level localisation (2026-06-22, second session)

- Faulting op =nl_vector_slot_ptr=, =rdi=0=: the NULL is the BUCKETS VECTOR's
  DATA pointer (=vec_box+8=).  I.e. a lexframe's =hash-table.slot1= holds a
  buckets vector whose data buffer pointer is 0.  cons has no separate data
  buffer (single box) → it survives growth; a vector/record has a separate
  data buffer allocated via =nl_alloc_bytes= → that is what goes NULL.
- Synthetic repros that DO cross the boundary but do NOT crash (so "frames +
  growth" is NOT sufficient): deep =let= recursion (500 × 50 KB), frame-heavy
  churn (400k =(let ((y (cons i i))) ...)=), vector churn (300k
  =(make-vector 8 i)= pairs).  All rc=0.  The trigger needs the audit /
  manifest-load + packaged-registry allocation interleaving specifically — the
  missing ingredient is still unidentified.
- Allocation primitives: =nl_alloc_bytes= (freelist reuse → arena bump →
  =nl_chunk_alloc_new= grow) is the bottom; =nl_chunk_alloc_new= returns 0 on
  =nl_os_alloc_chunk= mmap failure (bumps counter @268436208).  =nl_alloc_vector=
  / =nelisp_cons_construct= are grammar-op primitives (not elisp defuns) that
  bottom out in =nl_alloc_bytes=.  Leading micro-suspect: a vector DATA-BUFFER
  =nl_alloc_bytes= at/across the growth boundary returns 0 (or a base-relative
  pointer that is wrong on a growth chunk), and the caller stores it without a
  NULL check.
- Core-forensics caveat: the control region literals in the source (e.g.
  =268436160=, =268436208=) are NOT absolute addresses for gdb — the arena maps
  at a runtime (ASLR) base (=arena-stats[0]= ≈ =0x7f..=), so those reads are
  base-relative.  Literal-address gdb reads on the core fail
  ("Cannot access memory at 0x100002c0"), and =info proc mappings= did not
  surface the chunks.  To do core forensics, FIRST resolve the runtime arena
  base, then read control slots as =base + (literal − 0x10000000)=.
- Next sub-task: (a) build a repro that lands a live vector's DATA BUFFER on a
  growth chunk (vs the box on the first chunk) — the synthetic repros above
  keep box+buffer co-located; or (b) instrument =nl_alloc_bytes= to log a 0
  return / a cross-chunk vector-data allocation during audit (write a marker
  byte via =nl_os_write_stderr= so it survives the crash).

** 8.6 Exit-code dissection: allocator/read/clone all EXONERATED (2026-06-22 b)

Method: a worktree at =838605da= (the last commit that crashes 5/5 at the
256 MiB default; HEAD no longer grows that far because =3f318889= alloc-free
dispatch dropped audit's allocation pressure below the first-chunk size — the
crash is now LATENT, not fixed).  Probes use distinct =exit(N)= codes
(=syscall-direct 60 N=) instead of stderr — the process crashes, so the exit
code is the only reliable channel.  ((b)'s stderr "G" marker via
=syscall-direct 4 2= never surfaced even though growth ran 29× on a cons churn
— a separate puzzle; exit codes work.)  Each probe is byte-for-byte the
original on the valid path.

Findings, in order (all on the crashing 838605da/256 MiB binary):
1. =nl_alloc_bytes= growth branch: post-growth =nl_chunk_try_alloc= NEVER
   returns 0 (no exit 77) and the returned obj is ALWAYS in
   [chunk_base, chunk_base+size) (no exit 78).  A 1M-cons churn grew the arena
   29× (=arena-stats[9]= 1->30) with rc=0.  Freelist reuse is impossible
   (=nl_dealloc_bytes= is a no-op, sweep is OFF).  => the ALLOCATOR returns
   valid, in-range, fresh pointers across growth.  EXONERATED.
2. =nl_vector_slot_ptr= split into hops: the fault is hop-1, =sexp+8= (the
   NlVector box pointer) == 0, NOT hop-2 (=box+8= data_ptr).  Encoding the slot
   tag in the exit code gave =exit 90= => the slot reads tag 0 (Nil), box 0.
   The lexframe hash-table's slot 1 (which must hold the buckets vector)
   reads as Nil.
3. =nl_record_slot_ptr= split into hops: the ht record's box (=sexp+8=) and
   data_ptr (=box+40=) are BOTH valid (no exit 85 / 100+tag).  The record READ
   path is clean — it correctly addresses a slot that genuinely contains Nil.
   EXONERATED.
4. The write path: =record-slot-set= -> =nl_record_set_slot= -> =nl_val_clone_into=
   -> (boxed) =nl_vci_box= -> =nl_sexp_clone_into=.  A read-back guard in
   =nl_vci_box= (after the clone, assert =box.tag == src.tag=) NEVER fires
   (no exit 110+tag).  => at WRITE time the cloned buckets-vector Sexp box is
   correctly populated (tag 8, valid child box).  The clone is EXONERATED.

Conclusion (precise root cause):
  A live, correctly-created 32-byte Sexp box — the cloned buckets vector stored
  in =lexframe.ht.slot1= — is ZEROED by a STRAY WRITE sometime BETWEEN its
  creation (=frame_push= step 3) and its read (=find_in_frame=), and ONLY once
  the arena has grown past the first chunk.  It is not an allocation fault, not
  a read-side addressing fault, and not a clone fault — it is a post-write
  corruption.  This supersedes §8.4 #2 (the allocator-growth-bug hunt) and §8.5
  (the "alloc returns 0 / bad ptr" micro-suspect): both are now ruled out.

Leading remaining hypothesis: the bump allocator hands out an already-live
address a second time (a growth-induced cursor / chunk-bookkeeping desync), and
the reuse zeroes the old box (e.g. a later =nl_sexp_clone_into= of a Nil value,
or =nl_alloc_zero_fill=, writes over it).  mmap(NULL) chunks cannot overlap, so
the suspect is intra-chunk cursor reuse or the first-chunk fixed-cursor
(=nl_chunk_cursor_addr= -> =268435456=) desynchronising after a growth.

Next probe: an address-reuse / high-water-mark detector in =nl_alloc_bytes=
(assert each returned obj is disjoint from the last live allocation on its
chunk), or a per-chunk cursor-monotonicity assertion across the growth event.
