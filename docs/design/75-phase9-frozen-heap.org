* §1 背景

** §1.1 Phase 7 Stage 7.7.a/b の到達点 (= 起動時 Rust reader skip)

Doc 72 で SHIPPED した AOT bake によって、起動時の bootstrap path は以下:

#+begin_example
[Production binary 起動時]
  Env::new_global
    1. globals + frames を空で初期化
    2. install_builtins(&mut env)
       → ~150 builtins 登録 (= bi_car / bi_cons / bi_intern …)
    3. STDLIB_IMAGES 15 個を順に decode_image
       → 各 image bytes (= NELIMG v2) を Vec<Sexp> に逆構築
       → for form in &forms { eval(form, &mut env) }
    4. use_elisp_apply フラグ ON
#+end_example

(3) で Rust reader は呼ばれない (= 起動 path から完全切離) が *eval* は
依然として走る. 各 STDLIB image の中身は本質的に=defvar=, =defun=,
=defalias=, =defmacro=, =put= の連発であり、評価結果は env の
=globals= map を mutating 更新するだけ.

** §1.2 計測 (= 2026-05-08 baseline)

=Env::new_global= は CI box (Linux x86_64, release build) で:

- builtins install: ~3ms
- 15 STDLIB images の decode + eval: ~80-100ms
  - decode 自体は ~5ms 程度 (= bytes parse は速い)
  - 残り ~75ms+ は eval (= defvar/defun の loop で globals に挿入)

つまり起動オーバーヘッドの大半は *image 化されているにも関わらず
eval が走る* ことに起因. =defun=で =Sexp::Cons= chain を function cell
に詰めるだけの仕事だが、=eval= が macro 展開 / =sf_*= 特殊形式 /
=apply_combiner= を辿るため見かけ以上に重い.

** §1.3 Phase 9 frozen heap が解こうとする問題

仮に =Env::new_global= の最終状態 (= globals map + 内部の Sexp tree)
を build 時に snapshot し、起動時に *直接 deserialize* できれば:

- builtins install (= Rust 側の =install_builtins=) はそのまま実行
- STDLIB の 15 image decode + eval を *スキップ* 可能
- かわりに 1 個の =frozen-env.image= を deserialize して =globals= 直挿入

期待効果:

| 項目 | 現状 (Stage 7.7.b) | Stage 7.7.e 目標 |
|------+--------------------+------------------|
| =Env::new_global= 全体 | ~80-100ms | ~5-10ms |
| reader skip | 達成済 | 維持 |
| eval skip | 未達 | *新規* |
| binary size 増分 | image 計 ~190KB | snapshot ~250-400KB 想定 |

** §1.4 既存 STDLIB AOT bake (Stage 7.7.a/b) との差別化

| 観点 | Stage 7.7.a/b (= AOT bake) | Stage 7.7.e (= frozen heap) |
|------+----------------------------+-----------------------------|
| 対象 | 各 .el の *source forms* | =Env::new_global= の *最終 env state* |
| 単位 | file ごと (= 15 image) | env 全体 (= 1 image) |
| 起動時の処理 | decode → *eval* 各 form | decode → *globals 直挿入* |
| Sexp 単位 | source 由来 = closure 不在 | env 内 closure / Frame ref が混在 |
| cycle / shared 構造 | 起こりにくい (= source は tree) | 起きうる (= 同 cons が複数 globals に共有) |
| invalidation | =.el= 変更で baker 再走 | 上記 + Rust builtin 変更でも要再走 |
| 移植性 | image format v2 で確定 | format v3 拡張要、互換 break |

つまり frozen heap は AOT bake の "次の段" — source 形のままだった
形式を、評価済 env 状態へと押し上げる.

** §2.1 Stage 分解

Phase 9 は規模が大きいため、論理的に独立な 3 stage に分割:

** §2.2 Stage 7.7.e.1 — snapshot 形式設計

*** §2.2.1 表現する必要があるもの

=Env::new_global= 完了時の env 状態:

| 構成要素 | 既存 image v2 で対応? | frozen heap で必要 |
|----------+----------------------+--------------------|
| =globals: HashMap<String, SymbolEntry>= | × (= top-level 構造) | ○ |
| =SymbolEntry { value, function, plist, constant }= | × | ○ |
| =Sexp::Nil/T/Int/Float/Str/Symbol= | ○ (= TAG_NIL..STRING) | ○ (= 流用) |
| =Sexp::MutStr= | △ (= Str に flatten) | identity 必要なら拡張 |
| =Sexp::Cons(Rc<RefCell>,Rc<RefCell>)= | ○ (= TAG_CONS、ただし *identity 喪失*) | identity 保持版が必要 |
| =Sexp::Vector= | ○ (= 同上、identity 喪失) | identity 保持版が必要 |
| =Sexp::CharTable= / =Sexp::BoolVector= | ○ (= TAG_CHAR_TABLE/BOOL_VECTOR) | 流用 |
| =Sexp::Cell(Rc<RefCell<Sexp>>)= | △ (= 内側を平坦化) | identity 保持 + cycle 対策必要 |
| =Sexp::Record { type_tag, slots }= | × (= 現状 NotImplemented) | 新規 |
| =Frame= ref を持つ closure | × | *対象外* (= 後述 §2.2.4) |
| =extern_builtins=, =use_elisp_apply= 等 | × | 起動時再構築 |

*** §2.2.2 NELIMG v3 schema 概要

Doc 72 で確立した NELIMG v2 を拡張:

#+begin_example
NELIMG\0\x03                                  (8 byte magic, version=3)
u32  IMAGE_ABI_VERSION = 3
u8   IMAGE_KIND        = 0x01 (= "frozen-heap"、0x00 は v2 互換 form-list)
─── KIND 0x01 only ───
u32  N_NODES                                  ; shared object pool size
[N_NODES] {
   u8   NODE_TAG                              ; TAG_NIL..TAG_RECORD
   ...payload...                              ; 子参照は u32 NODE_INDEX で記述
}
u32  N_GLOBALS
[N_GLOBALS] {
   u32  NAME_LEN ; bytes NAME
   u8   FLAGS    ; bit0=has_value bit1=has_function bit2=has_plist bit3=constant
   if has_value:    u32 NODE_INDEX
   if has_function: u32 NODE_INDEX
   if has_plist:    u32 NODE_INDEX
}
#+end_example

*key 性質*:

- Sexp tree を *DAG* として正規化 — 同 =Rc::ptr_eq= のセルは同 NODE_INDEX
- =Cons= の car/cdr は子 Sexp を *NODE_INDEX 参照* で表す (= v2 では再帰 inline)
- これにより shared 構造 (= 同じ symbol object が 100 globals から指される、
  cycle list、self-referential cons) が identity 保ったまま round-trip
- v2 既存 TAG (= TAG_NIL..TAG_BOOL_VECTOR) は値の表現を流用しつつ
  *子参照だけ NODE_INDEX に置き換え*

*** §2.2.3 既存 image v2 との互換

- v2 image (= Stage 7.7.b の STDLIB image) は KIND=0x00 (= form-list mode) と
  解釈、現行 =decode_image= が処理
- v3 frozen-heap image は KIND=0x01 で新 path =decode_frozen_heap=
- 両者は同じ magic prefix 末尾 byte (= ABI version) で区別
- v3 reader は v2 image を読めるよう *両 KIND を支持* するか、
  *別 entrypoint* (= =decode_image_v2= / =decode_frozen_heap_v3=) として
  分けるか — 推奨は後者 (= 用途が異なるため別 surface)

*** §2.2.4 closure / FrameCell の扱い (= core risk)

現状 =Env::new_global= 完了後の =globals= 内には、各 =defun= が installed
した =Sexp::Cons= で表現された closure が居る:

#+begin_src text
function cell of `my-fn':
   (closure CAPTURED-ENV (ARGS...) BODY...)
where CAPTURED-ENV is an alist ((name . FrameCell) ...)
#+end_src

問題:

1. CAPTURED-ENV の =FrameCell= (= =Rc<RefCell<Sexp>>=) は Cons 形式で
   =(name . Cell-wrapped-value)= に折られているが、Cell の identity は
   "同じ let-binding を共有する別 closure からの write-through" を支える
2. =Frame= 自体は Vec<HashMap> で =new_global= 完了時には空 (= 全 frame
   pop 済) だが、closure CAPTURED-ENV alist 内の FrameCell は heap で生存
3. これらを serialize して deserialize 後も identity を保つには、
   =FrameCell= も DAG node として encode 必要

*戦略 A: 完全 frozen* — closure / Cell も含めて全 env DAG を snapshot.
identity 保持で 100% 等価復元. *実装難度高、bug リスク大*.

*戦略 B: builtins + plain data 限定* — 起動時に:
- builtins は Rust から再 install (= 既存 =install_builtins=)
- =defvar= の値 (= 純粋 Sexp tree) は frozen heap から復元
- =defun= / =defmacro= / =defalias= は *起動時に再 eval* で closure 再生成

戦略 B では closure 部分だけ起動時 eval が必要 — つまり frozen heap は
"source eval を deferred eval へ" ではなく "値の serialize は frozen /
closure は再生成" の hybrid. 期待 speedup は若干薄まる (~80ms → ~30ms 想定).

*戦略 C: 戦略 B + closure body のみ pre-decode* — =defun= の引数 list と
body Sexp tree は frozen 化 (= source decode + macro 展開後の form を
保存)、CAPTURED-ENV は起動時に空 alist (= top-level defun は lexical env
を捕捉していない) として再構築. 多くの defun は CAPTURED-ENV が =nil= の
ため、これで大半のケースが通る.

*推奨*: *戦略 C*. STDLIB 内の =defun= はほぼ全て top-level (= CAPTURED-ENV
が空) であり、closure-with-captures が居るのは下位で =let= 内 lambda が
=defalias= された限定パターンのみ. それは fallback として "起動時再 eval
が必要な form" として別 list で持たせる.

*** §2.2.5 NODE_TAG 拡張

NELIMG v2 から追加する tag (= NODE_TAG として識別):

| TAG | 値 | payload |
|-----+----+--------|
| TAG_CELL | 0x0A | u32 NODE_INDEX (= 中身 Sexp) |
| TAG_RECORD | 0x0B | u32 type_tag NODE_INDEX, u32 slot 数, [slot] u32 NODE_INDEX |
| TAG_MUT_STR | 0x0C | u32 LEN, bytes UTF-8 |

子参照を *インライン* せず *NODE_INDEX 参照化* するため、TAG_CONS /
TAG_VECTOR の payload も v2 から変更:

| TAG | v2 (form-list mode) | v3 (frozen heap mode) |
|-----+---------------------+-----------------------|
| TAG_CONS | inline car / inline cdr (= 再帰) | u32 car_idx, u32 cdr_idx |
| TAG_VECTOR | u32 len, inline items × n | u32 len, [u32 idx] × n |
| TAG_CHAR_TABLE | inline subtype/default/entries | NODE_INDEX 参照版 |

つまり KIND=0x01 では tag 値は同じだが payload 形式が DAG 化する.
*混同を避けるため別 decoder 関数を用意* — 既存 =decode_value= は v2 専用、
新規 =decode_node= は v3 frozen heap 専用.

** §2.3 Stage 7.7.e.2 — cycle-safe encoder 実装

*** §2.3.1 algorithm (= DAG 正規化 + post-order 出力)

#+begin_src text
fn encode_env(env: &Env) -> Vec<u8>:
  let mut interner = NodeInterner::new()
  let mut globals_table = Vec::new()
  for (name, entry) in &env.globals (sorted for determinism):
    let value_idx = entry.value.as_ref().map(|v| intern(&mut interner, v))
    let function_idx = entry.function.as_ref().map(|v| intern(&mut interner, v))
    let plist_idx = entry.plist.as_ref().map(|v| intern(&mut interner, v))
    globals_table.push((name, value_idx, function_idx, plist_idx, entry.constant))
  serialize: header + interner.to_bytes() + globals_table.to_bytes()

fn intern(interner: &mut NodeInterner, sexp: &Sexp) -> u32:
  match sexp {
    Sexp::Nil | Sexp::T => atomic_intern(...)
    Sexp::Int(_) | Sexp::Float(_) | Sexp::Symbol(_) | Sexp::Str(_) =>
       value-keyed dedupe (= same Int n → same idx)
    Sexp::Cons(car, cdr):
       let key = ConsKey { car_ptr: Rc::as_ptr(car), cdr_ptr: Rc::as_ptr(cdr) }
       if let Some(idx) = interner.cons_seen.get(&key): return idx
       let placeholder_idx = interner.alloc_placeholder()
       interner.cons_seen.insert(key, placeholder_idx)   // mark before recursion
       let car_idx = intern(interner, &car.borrow())
       let cdr_idx = intern(interner, &cdr.borrow())
       interner.fill(placeholder_idx, NodeBody::Cons { car_idx, cdr_idx })
       return placeholder_idx
    // Sexp::Vector / Sexp::Cell / Sexp::Record も同パターン
    // (= Rc::as_ptr で identity key、placeholder で cycle 対応)
  }
#+end_src

*key 設計*:

- =Rc::as_ptr(rc)= で Rc の generation を捕捉、cycle 中の self-ref で
  二重訪問しない (= Doc 50 §3.5 の cycle-safe equal pattern を踏襲)
- placeholder 戦略 = 訪問開始時に NODE_INDEX を予約、子訪問が完了して
  から body を書き込む (= cycle 中に "親が見えていない" 状況回避)
- Sexp::Symbol / Str など atomic は value-keyed dedupe (= "true" symbol
  が 100 箇所から指されても node 1 つ)

*** §2.3.2 Closure 戦略 C の落とし穴

戦略 C では =defun= installed Sexp は基本 =(closure nil ARGS BODY...)=
の形 (= CAPTURED-ENV = =Sexp::Nil=). しかし以下が混在しうる:

1. =(closure ((cell . FrameCell-Sexp)) ARGS BODY...)= = let 内 lambda が
   defalias で global に出たケース
2. =(macro . (lambda ARGS BODY))= = =defmacro= 形式
3. =(autoload ...)= = =autoload= 設定 (= STDLIB ではほぼ未使用、念のため)

戦略:
- (1) は frozen heap の対象外として =Env::new_global= 起動時 fallback
  list (= "再 eval forms") に格納、起動時に source 再 eval
- (2) は内部の lambda body だけが Sexp tree なので普通に encode 可
- (3) は =Symbol= + =List= の組み合わせ、普通に encode 可

判定: encoder は =Sexp::Cons(head, body)= の =head= を見て、
=head == Symbol("closure")= かつ =body.car != Nil= なら fallback 行き、
それ以外は通常 encode.

** §2.4 Stage 7.7.e.3 — deserializer + 起動時組み込み

*** §2.4.1 deserializer

#+begin_src text
fn decode_frozen_heap(bytes: &[u8]) -> Result<DecodedEnv, ImageError>:
  parse header (= magic + version=3 + kind=0x01)
  let n_nodes = read_u32
  let mut nodes: Vec<Option<Sexp>> = vec![None; n_nodes]
  // 1st pass: パース node body (子参照は idx で持つ、まだ Sexp 化しない)
  let mut bodies: Vec<NodeBody> = parse n_nodes 個
  // 2nd pass: cycle-safe build — 各 idx に Rc allocation しつつ填める
  for idx in 0..n_nodes:
    nodes[idx] = Some(allocate_shell(&bodies[idx]))   // RefCell-empty placeholder
  for idx in 0..n_nodes:
    fill_in(&mut nodes[idx], &bodies[idx], &nodes)    // inner Sexp 設定
  // 3rd: globals_table parse
  let n_globals = read_u32
  let mut globals: HashMap<String, SymbolEntry> = HashMap::with_capacity(n_globals)
  for _ in 0..n_globals:
    parse name + flags + idxs
    populate SymbolEntry
  Ok(DecodedEnv { globals, fallback_forms: ... })
#+end_src

*key*:
- 2-pass で =Rc<RefCell>= の placeholder allocation → 中身設定. これで
  cycle (= self-referential cons) も復元可
- =Cons(Rc::clone(&nodes[car_idx]).downcast_to_cell(), ...)= の代わりに
  各 node ごとに =Rc<RefCell<Sexp>>= を保持し、parent-child は Rc::clone で
  share

*** §2.4.2 =Env::new_global= の bootstrap loop 切替

before (Stage 7.7.b):

#+begin_src rust
super::builtins::install_builtins(&mut env);
for (name, image_bytes) in STDLIB_IMAGES {
    let forms = image::decode_image(image_bytes)?;
    for form in &forms {
        super::eval(form, &mut env)?;
    }
}
#+end_src

after (Stage 7.7.e.3):

#+begin_src rust
super::builtins::install_builtins(&mut env);
const FROZEN_HEAP: &[u8] = include_bytes!("../../../lisp/frozen-env.image");
let decoded = image::decode_frozen_heap(FROZEN_HEAP)?;
env.globals = decoded.globals;
for src in decoded.fallback_forms {
    let forms = read_all_via_elisp(&src, &mut env)?;
    for form in &forms {
        super::eval(form, &mut env)?;
    }
}
#+end_src

戦略 C の fallback_forms は通常空 list — STDLIB が closure-with-captures
を持たないなら fallback path は走らない.

*** §2.4.3 invalidation policy

frozen-env.image は依存の transitive closure が広い:
1. =lisp/*.el= の内容
2. =install_builtins= が登録する関数 list
3. NELIMG v3 schema 自体
4. =Sexp= 型の derive 順 / Variant 並び (= JIT inline-emit と同じ理由)

build.rs で再 bake trigger:
- =cargo:rerun-if-changed=lisp/= = (1)
- =cargo:rerun-if-changed=build-tool/src/eval/builtins.rs= = (2)
- =cargo:rerun-if-changed=build-tool/src/image.rs= = (3)
- =cargo:rerun-if-changed=build-tool/src/eval/sexp.rs= = (4)

frozen-env.image 自体は Stage 7.7.b の =lisp/*.image= と並んで
git track するか、build artifact として OUT_DIR に置くかは設計判断:

- *git track 案*: 新人 / CI box でも cargo build 1 step で走る (= 既存
  Stage 7.7.b 流儀踏襲)
- *artifact 案*: image size ~300KB でリポジトリ膨張 (= Stage 7.7.b
  でも debate あり、結局 git track 採用)

→ *git track 推奨*. Stage 7.7.b 流儀継続.

* §3 Risks / Mitigation

** §3.1 format breakage の影響範囲

NELIMG v3 schema が一度 ship されると、=Sexp= variant の追加 / 並び替え /
struct field 追加で format が壊れる. =Sexp= 型は既に
=#[repr(C, u8)]= 制約 (= Doc 62 Phase 5 inline-emit unlock) で並びが固定
されているが、frozen heap はさらに *値の binary representation* も pin する.

- *Mitigation*: =IMAGE_ABI_VERSION= を v3 → v4 に bump する pre-flight
  check を =image-baker= 側に持たせる. ABI bump は Stage 7.7.b の
  =UnsupportedVersion(1)= reject と同じ pattern (= image.rs:549 参照)
- *Mitigation*: =SEXP_TAG_*= constants と NODE_TAG が同期しているか
  check する unit test (= =frozen_heap_tags_match_sexp_variants= 相当)

** §3.2 初期化順序保証 (= globals の install 順)

=defvar= / =defun= には load 順依存があるケースがある (= ある defvar
の値が別 defun の closure body に直接埋まっている). frozen heap は
"全 globals を一括復元" するため、現状の load 順仮定が破れていないか
検証必要.

- *Mitigation*: =Env::new_global= 通常 path と frozen heap path で
  =globals= 内容を =equal= で比較する cross-equiv test (= 全 entry の
  =SymbolEntry= が一致することを確認). この test を CI gate に置く

** §3.3 closure fallback (= 戦略 C ↔ 戦略 B 切替)

戦略 C (= top-level defun は frozen、closure-with-captures だけ fallback)
が *実は STDLIB に closure-with-captures が多い* と判明したケース、
fallback ratio が高くなり期待 speedup が消える.

- *Mitigation*: Stage 7.7.e.1 で実測 — =Env::new_global= 完了後に
  globals を walk し、=Sexp::Cons(Symbol("closure"), Cons(env, ...))=
  の env-cell が =Nil= 以外の数を count. 多ければ戦略 B (= defun も
  起動時再 eval) に retreat
- *Mitigation*: 戦略 retreat path を Stage 7.7.e.2 着手前に決定、本 Doc
  §9 改判断 record に明記

** §3.4 cycle / shared identity の正当性

=Sexp::Cons= に cycle が STDLIB 内で発生するかは現状未調査. 仮に発生
していると =equal= の cycle-safe 実装 (= Doc 50 §3.5) が前提だが、
serialize でも同じ visited-set pattern を使えば対応可能 (= §2.3.1
algorithm).

- *Mitigation*: encoder の cycle handling に specific test (= self-ref
  cons / 相互参照 list) を ERT で追加. Stage 7.7.e.2 検証必須項目
- *Mitigation*: 万一 STDLIB に予期せぬ cycle が居て encoder が無限再帰
  したら =std::collections::HashSet<*const RefCell<Sexp>>= で訪問記録、
  既訪問は idx 返却で抜ける (= placeholder pattern と同じ思想)

** §3.5 binary size 増分

STDLIB image (= Stage 7.7.b 完了形) は計 ~190KB. frozen heap は env 全体
を持つため 1.5x〜2x 想定 = ~250-400KB. =nelisp= binary に
=include_bytes!= で埋め込むと binary size が膨張.

- *Mitigation*: 既存 STDLIB image (= 15 個) を *frozen heap が成立した
  時点で deprecate*. =Env::new_global= は frozen heap 1 個だけ参照すれば
  良いので、STDLIB images は =cfg(feature = "image-baker")= 下に下げる
  (= bake-time の中間生成物に格下げ).  これで net 増は ~250KB - 190KB
  = +60KB 程度
- *Mitigation*: frozen heap を gzip 圧縮した形で埋め込み、起動時 inflate
  も検討 (= gzip on ~300KB → ~80KB 程度の見込み). ただし inflate コスト
  が ~5ms 加わるので net trade-off

** §3.6 development iteration cost

elisp 開発者が =lisp/*.el= を編集するたびに frozen heap 再 bake が走る:

- 現状 (Stage 7.7.b): 1 ファイル変更 → 該当 image だけ再 bake (= 1秒)
- 将来 (Stage 7.7.e): 1 ファイル変更 → frozen heap 全体再 bake (= 5-10秒
  想定、=Env::new_global= の現状 80ms + I/O)

- *Mitigation*: dev mode で =NELISP_NO_FROZEN_HEAP=1= 環境変数を見る、
  この場合は Stage 7.7.b path を走らせる (= per-file bake で iteration
  速度維持). production / CI のみ frozen heap path

** §3.7 既存 escape hatch との両立

=NELISP_USE_RUST_APPLY= (= Doc 70 Stage 7.4.e.2) は frozen heap 内に
保存される =use_elisp_apply= フラグ初期値とは独立. 起動時に frozen heap
deserialize → =env.use_elisp_apply= の re-set (= 環境変数読み直し) を
順序で行う.

- *Mitigation*: =Env::new_global= の末尾で現状通り環境変数 check し、
  flag 上書き. frozen heap 自体は flag 値を含めない (= runtime 設定なので)
