* §1 背景 + frozen heap が指す具体物

** §1.1 Phase 7 Stage 7.7.a/b の到達点 (= 起動時 Rust reader skip)

Doc 72 で SHIPPED した AOT bake によって、起動時の bootstrap path は以下:

#+begin_example
[Production binary 起動時]
  Env::new_global
    1. globals + frames を空で初期化
    2. install_builtins(&mut env)
       → 93 builtins 登録 (= 2026-05-09 grep 実数;
          install_builtins fn 内 names list の "..." count)
    3. STDLIB_IMAGES 16 個を順に decode_image
       → 各 image bytes (= NELIMG v2) を Vec<Sexp> に逆構築
       → for form in &forms { eval(form, &mut env) }
    4. use_elisp_apply フラグ ON
#+end_example

(3) で Rust reader は呼ばれない (= 起動 path から完全切離) が *eval* は
依然として走る. 各 STDLIB image の中身は本質的に =defvar=, =defun=,
=defalias=, =defmacro=, =put= の連発であり、評価結果は env の
=globals= map を mutating 更新するだけ.

** §1.2 計測 (= 2026-05-09 baseline)

=Env::new_global= は CI box (Linux x86_64, release build) で:

- builtins install: ~3ms
- 16 STDLIB images の decode + eval: ~80-100ms
  - decode 自体は ~5ms 程度 (= bytes parse は速い)
  - 残り ~75ms+ は eval (= defvar/defun の loop で globals に挿入)

つまり起動オーバーヘッドの大半は *image 化されているにも関わらず
eval が走る* ことに起因. =defun= で =Sexp::Cons= chain を function cell
に詰めるだけの仕事だが、=eval= が macro 展開 / =sf_*= 特殊形式 /
=apply_combiner= を辿るため見かけ以上に重い.

** §1.3 frozen heap が指す具体物 — 何を凍結するか

=Env::new_global= の最終状態を「凍結する」とは:

| 構成要素 | 構造 | 凍結対象? | 備考 |
|----------+------+----------+------|
| =globals: HashMap<String, SymbolEntry>= | top-level | ○ | 主対象 |
| =SymbolEntry { value, function, plist, constant }= | 4 field struct | ○ | 全 field 含む |
| =Sexp::Nil/T= | atomic singleton | ○ (= TAG 自明) | 既存 v2 流用 |
| =Sexp::Int(i64) / Float(f64)= | atomic primitive | ○ | 既存 v2 流用 |
| =Sexp::Symbol(String) / Str(String)= | atomic owned | ○ (= value-keyed dedupe) | 既存 v2 流用 |
| =Sexp::MutStr(NlStrRef)= | NlRc shared | ○ (= identity 保持必要) | format v3 拡張 |
| =Sexp::Cons(NlConsBoxRef)= | NlRc shared | ○ (= DAG 化必須) | format v3 拡張 |
| =Sexp::Vector(NlVectorRef)= | NlRc shared | ○ (= 同上) | format v3 拡張 |
| =Sexp::CharTable(NlCharTableRef)= | NlRc shared | ○ | format v3 拡張 |
| =Sexp::BoolVector(NlBoolVectorRef)= | NlRc shared | ○ | format v3 拡張 |
| =Sexp::Cell(NlCellRef)= | NlRc + RefCell | ○ (= identity 必要) | format v3 拡張 |
| =Sexp::Record(NlRecordRef)= | NlRc tagged | ○ (= 現状 NotImplemented) | 新規 |
| =Frame= ref を持つ closure | 動的 | × (= 戦略 C で fallback) | 後述 §1.5 |
| =extern_builtins: HashMap= | runtime config | × | 起動時再構築 |
| =use_elisp_apply: bool= | runtime config | × (= env var で覆る) | 起動時再構築 |
| =max_recursion: usize= / =frames= 等 | runtime state | × | 起動時 default |

凍結対象 = =globals= map + その値が指す Sexp DAG + (plist 内の Cell /
shared cons identity).

** §1.4 NlRc heap layout pinning との互換性 (= Doc 77c Phase A.5.1 SHIPPED)

Doc 77c Phase A.5.1 (2026-05-09 SHIPPED) で 7 個の =NlXxxRef= 型が
=#[repr(transparent)]= に固定済 (= NlConsBoxRef / NlVectorRef /
NlBoolVectorRef / NlCharTableRef / NlRecordRef / NlStrRef / NlCellRef).
これは frozen heap encoder/decoder にとって *重要な前提条件*:

- 各 NlXxxRef は内部 =NlRc<NlXxxBox>= の transparent newtype
- =NlRc<T>= は =Rc<UnsafeCell<NlRcInner<T>>>= 形式 (= eval/nlrc.rs)
- =NlRc::as_ptr(&rc)= で identity key 取得可 (= encoder の DAG dedup key
  に使える、§3 Stage 9.2 で詳述)
- runtime 中の Sexp tree が共有する Rc cell は *安定 layout* を持つ →
  serialize 後の byte-level relocation は不要 (= 起動時に *新規 NlRc 割当*
  で Rc cell を再構築すれば同 logical identity が保たれる、§3 Stage 9.3 で詳述)

つまり Phase 9 は heap layout pin (= 物理 byte address の永続化) ではなく
*logical DAG identity の serialize 後再生* で十分. これは Phase 9 の
複雑度を大きく下げる (= mmap 化 / pointer relocation table 等は不要).

** §1.5 closure / FrameCell の取扱い (= 戦略 C 採用)

=Env::new_global= 完了後の =globals= 内には、各 =defun= が installed
した =Sexp::Cons= で表現された closure が居る:

#+begin_src text
function cell of `my-fn':
   (closure CAPTURED-ENV (ARGS...) BODY...)
where CAPTURED-ENV is an alist ((name . FrameCell) ...) or `nil'
#+end_src

戦略候補:

- *戦略 A: 完全 frozen* — closure / Cell も含めて全 env DAG を snapshot.
  identity 保持で 100% 等価復元. 実装難度高、bug リスク大
- *戦略 B: builtins + plain data 限定* — defvar の値 (= 純粋 Sexp tree) は
  frozen heap から復元、defun / defmacro / defalias は起動時に再 eval で
  closure 再生成. speedup 薄まる (~80ms → ~30ms 想定)
- *戦略 C: 戦略 B + closure body のみ pre-decode* — defun の引数 list +
  body Sexp tree は frozen 化、CAPTURED-ENV は起動時に空 alist として
  再構築. 多くの defun は CAPTURED-ENV が =nil= のため大半のケースが通る

*推奨 = 戦略 C*. STDLIB 内の defun はほぼ全て top-level (= CAPTURED-ENV
が空) であり、closure-with-captures が居るのは下位で =let= 内 lambda が
=defalias= された限定パターンのみ. それは fallback として「起動時再 eval
が必要な form」として別 list で持たせる.

* §4 Bootstrap circularity risk

** §4.1 image loader と heap の chicken-and-egg

frozen heap image を *loader* が読む段階で:
- =Sexp= 型は Rust の =enum= として既に定義されている (= eval/sexp.rs)
- =NlRc<T>= は eval/nlrc.rs で実装済 (= Doc 77c Phase A.1 SHIPPED 2026-05-09)
- =NlConsBoxRef= 等の reference 型は Doc 77c Phase A.5.1 で
  =#[repr(transparent)]= 確定済

つまり loader は *self-host する前の Rust core* で動き、Sexp tree を
=NlConsBoxRef::new(NlConsBox { car, cdr })= 等で *新規 allocation* で
復元する. 既存 STDLIB image v2 decode と同じ pattern.

** §4.2 Phase 7 AOT bake (Stage 7.7.b) との比較

Stage 7.7.b SHIPPED 後の bootstrap path:
- =Env::new_global= → =install_builtins= → =STDLIB_IMAGES decode + eval=

ここで =eval= は Rust 実装. eval が走るためには:
- =sf_*= special forms (= sf_quote / sf_if / sf_let / sf_lambda 等、
  special_forms.rs:572 LOC) が installed
- =bi_*= builtins (= 93 個、builtins.rs) が installed
- =Sexp= 型 + =NlXxxRef= 型 (= Doc 77c Phase A の成果) が安定

これらは全て *Phase 7 段階で SHIPPED*. Phase 9 frozen heap は
"eval 後の env state を pre-compute して serialize" するので、
eval を走らせる Rust core は *引き続き必要* (= dev path / re-bake で使う).

*chicken-and-egg はない*: Phase 9 は eval 結果を *起動時 short-circuit*
するだけで、eval 本体の Rust 実装を消すわけではない. eval 本体は
*post-bootstrap user code* (= eval-string / load 等) のために残る.

** §4.3 frozen heap 自体の生成 path

=nelisp-baker --frozen-heap= 経由:
1. baker bin が =Env::new_global()= を呼ぶ (= 通常 bootstrap、~80ms 走る)
2. =image::frozen_heap::encode_env(&env)= で =Env= → bytes 化
3. =lisp/frozen-env.image= に書き出し

baker 自体は cfg(feature = "image-baker") 下で動く dev tool. production
binary には含まれない. これも Stage 7.7.a で確立した pattern (= Doc 72) と同.

** §4.4 frozen heap が「導入する」circularity リスクは小さい

逆に Phase 9 は existing circularity を *軽減* する側 — 起動時に Rust eval
を呼ばないことで:
- macro 展開 race condition (= Stage 7.5.c で観測した nelisp--expand-macro
  vs install 順序問題) が起きえない (= 既に展開済の Sexp が直ロード)
- =use_elisp_apply= flag の bootstrap 中 OFF → post-bootstrap ON という
  延期切替も不要に近い (= flag 状態は frozen heap に内包しない設計、§5.4 参照)

* §5 Image format 設計 (= NELIMG v3, frozen heap mode)

** §5.1 Stage 7.7.a/b (= NELIMG v2) との互換性

| 観点 | NELIMG v2 (= Stage 7.7.b SHIPPED) | NELIMG v3 (= Phase 9 frozen heap) |
|------+-----------------------------------+-----------------------------------|
| magic | =NELIMG\\0\\x02= (= 8 bytes) | =NELIMG\\0\\x03= |
| version | u32 = 2 | u32 = 3 |
| KIND byte | 暗黙 0 (= form list) | 0x01 = frozen-heap |
| 子参照 | inline 再帰 | u32 NODE_INDEX 参照化 |
| 同 Rc 共有 | identity 喪失 (= deep copy) | identity 保持 |
| cycle 対応 | 不可 (= 再帰で stack overflow) | placeholder 戦略で対応 |
| Sexp::Record | NotImplemented (= 拒否) | 新規 TAG_RECORD |
| Sexp::MutStr | TAG_STRING に flatten | TAG_MUT_STR で identity 保持 |
| 用途 | 単一 .el ファイル → form list | env 全体 → globals + DAG |

v2 image (= Stage 7.7.b の STDLIB image) は引き続き baker dev path で生成、
=cfg(feature = "image-baker")= 配下で decode 可. *両 KIND を別 entrypoint*
(= =decode_image= for v2 / =decode_frozen_heap= for v3) として分離する.

** §5.2 v3 schema 詳細

#+begin_example
NELIMG\\0\\x03                                 (8 byte magic, version=3)
u32  IMAGE_ABI_VERSION = 3
u8   IMAGE_KIND        = 0x01 (= frozen-heap、0x00 は v2 互換 form-list)
─── KIND 0x01 only ───
u32  N_NODES                                   ; shared object pool size
[N_NODES] {
   u8   NODE_TAG                               ; TAG_NIL..TAG_RECORD
   ...payload...                               ; 子参照は u32 NODE_INDEX で記述
}
u32  N_GLOBALS
[N_GLOBALS] {
   u32  NAME_LEN ; bytes NAME (UTF-8)
   u8   FLAGS    ; bit0=has_value bit1=has_function bit2=has_plist bit3=constant
   if has_value:    u32 NODE_INDEX
   if has_function: u32 NODE_INDEX
   if has_plist:    u32 NODE_INDEX
}
u32  N_FALLBACK_FORMS                          ; closure-with-captures 等
[N_FALLBACK_FORMS] {
   u32  SOURCE_LEN ; bytes UTF-8 source
}
#+end_example

key 性質:

- Sexp tree を *DAG* として正規化 — 同 =NlRc::as_ptr= のセルは同 NODE_INDEX
- =Cons= の car/cdr は子 Sexp を *NODE_INDEX 参照* で表す
- shared 構造 (= 同 symbol が 100 globals から指される、cycle list、self-ref
  cons) が identity 保持で round-trip
- 末尾 =N_FALLBACK_FORMS= は戦略 C 救済 list (= top-level でない closure
  installed の defun source)

** §5.3 NODE_TAG 一覧

| TAG | hex | payload (v3 frozen heap mode) | 既存 (= v2) との対応 |
|-----+-----+-------------------------------+----------------------|
| TAG_NIL | 0x00 | none | 同 |
| TAG_T | 0x01 | none | 同 |
| TAG_INT | 0x02 | i64 | 同 |
| TAG_FLOAT | 0x03 | u64 (= bits) | 同 |
| TAG_SYMBOL | 0x04 | u32 LEN + UTF-8 bytes | 同 |
| TAG_STRING | 0x05 | u32 LEN + UTF-8 bytes | 同 |
| TAG_CONS | 0x06 | u32 car_idx, u32 cdr_idx | 違 (v2 = inline) |
| TAG_VECTOR | 0x07 | u32 len, [u32 idx] × n | 違 (v2 = inline) |
| TAG_CHAR_TABLE | 0x08 | NODE_INDEX 参照版 (= subtype/default/entries が NODE_INDEX) | 違 (v2 = inline) |
| TAG_BOOL_VECTOR | 0x09 | u32 len, [u8 bit] × n | 同 |
| TAG_CELL | 0x0A | u32 NODE_INDEX (= 内側 Sexp) | 新規 |
| TAG_RECORD | 0x0B | u32 type_tag NODE_INDEX, u32 slot 数, [u32 idx] × n | 新規 |
| TAG_MUT_STR | 0x0C | u32 LEN + UTF-8 bytes | 新規 (= 区別保持) |

子参照を *インライン* せず *NODE_INDEX 参照化* するため、TAG_CONS /
TAG_VECTOR / TAG_CHAR_TABLE の payload も v2 から変更. 同じ tag 値だが
payload semantics が KIND で switch.

** §5.4 NlRc refcount initial state

deserialize 後の =NlRc<T>= の strong_count 初期値は *自然な refcount*
で良い (= encoder が DAG 化した時の参照数を post-load 環境で再現する
必要はない、Rc は新規 allocation で再構築されるため).

具体的には:
- decoder が NODE_INDEX 0..N_NODES の各 idx に対して =NlRc::new= で
  shell allocation → fill phase で car/cdr を Rc::clone で参照
- globals が同じ NODE_INDEX を参照する場合、refcount は自動的に N+1 に
  なる (= 新規 NlRc::new で 1 + globals 参照 N + 内部 cons 参照)
- Phase 9 起動後の env 状態は Stage 7.7.b 起動後と *Sexp 値レベルで等価*
  (= globals 内容 / function-cell の closure tree が =equal=). Rc identity
  も保たれる (= 同 NODE_INDEX を共有する 2 globals は同 =NlRc::ptr_eq=).

** §5.5 cycle 復元 (= placeholder pattern)

self-referential cons (= =(let ((x (cons 1 nil))) (setcdr x x))=) を
serialize → deserialize する時:

1. encoder: visit 開始時に NODE_INDEX を予約 (= placeholder), 子訪問が
   cycle で同 cell に戻ってきたら predict NODE_INDEX を返す, 訪問後に
   body を fill
2. decoder: 1st pass で全 NODE_INDEX の =NlConsBoxRef::new(empty placeholder)=
   を allocate. 2nd pass で各 cons の car/cdr に NODE_INDEX 参照を解決

これは Doc 50 §3.5 cycle-safe equal pattern を踏襲 (= visited-set
strategy). 既に nelisp 内で確立した pattern なので新規発明なし.

** §5.6 IMAGE_ABI_VERSION bump policy

- v2 → v3 は schema 互換 break (= KIND byte 追加 + payload semantics 変更)
- 起動時 v2 image を mistake で渡された場合: =UnsupportedVersion(2)= reject
  (= image.rs:108 既存 check と同 pattern)
- 将来 v3 → v4 が必要になる契機:
  - =Sexp= variant 追加 / 並び替え (= =#[repr(C, u8)]= 制約あり)
  - =NlRc<T>= layout 変更
  - 新規 NODE_TAG 追加
- ABI version は *Rust core* の image.rs に const として持ち、baker と
  loader が同値を使う

** §7.1 戦略 C の closure-with-captures 数 (= Stage 9.1.b で実測待ち)

** §7.2 frozen heap image を git track するか build artifact にするか

  → image format 変更時に repo size churn なし、ただし baker 経路必須

→ 推奨 = git track. Stage 7.7.b 流儀継続. binary review 不能の問題は
=git diff --stat lisp/frozen-env.image= で size delta 観察可能、

** §7.3 user 2026-05-09 monotonic shrink rule の "Rust LOC" 解釈

Phase 9 単独では net +75 LOC (§2.5). Phase 8 Stage 8.4 prereq と bundle

- *厳格解釈* (= 物理 LOC 全 .rs ファイル合計): Phase 9 単独で違反、
  bundle で OK
- *緩和解釈* (= production binary build graph 内 LOC): Phase 9 単独でも
  reader/* 1679 LOC が cfg gate で消えるので OK

→ user judgment 待ち. 本 Doc は緩和解釈を *推奨* としつつ厳格解釈でも

** §7.5 frozen heap 圧縮 (= gzip / zstd) で binary size 軽減するか

- 起動時間 short circuit が主目的なら inflate cost は許容外
- binary distribution size を気にするなら採用候補

→ Stage 9.6 の bench gate 結果次第. Phase 9 出口時点では非圧縮.
