* §1 Scope

** 1.1 motivation (= Doc 28 §11.5 errata の future Phase 担当)

Doc 28 v2 §11.5 errata で *fib 30x bar の post-v1.0 defer* を確定、
"Future Phase (post-v1.0): Phase 7.7 (or rename) = recursive inliner +
escape analysis、期待効果 fib 30x+ gate 達成可能性、v1.x feature
target、v1.0 ship gate からは除外" と明記された。本 doc がその Phase
7.7 担当 design doc。

T96 finding =Specific findings — why fib's 30x gate is unreachable= (= bench
result md §) は 3 因子で fib 30x unreachability を analyze:

1. *Emacs native-comp 自身が fib(30) で bytecode の 5% 以内* — libgccjit
   でも escape analysis + scalar replacement of closure environment 無し
   では recursive int arithmetic は specialise できない
2. *NeLisp native at 12x bytecode = Emacs native-comp の 12.5x* — 同
   compiler quality envelope 内では既に best-in-class
3. *Per-call overhead = recursion structure dominated* — 50ms / 10 iter
   / 2.7M calls = 1.85 ns / call = 5-6 cycle / call、prologue 単独 ~6
   cycle で frame elimination 無しでは shrink 不可

= 3 因子全てが *recursion を unrolling / inline で排除する* 方向の
optimization 必要を指している。本 Phase 7.7 = まさにその optimization
class を Phase として組成する。

** 1.2 4 sub-phase の concrete scope

| sub-phase | scope                                           | 期待効果 (fib 30x への寄与)             |
|-----------+-------------------------------------------------+----------------------------------------|
| 7.7.1     | escape analysis (closure env が caller frame に閉じるか SSA で証明) | inliner safety 保証、SROA gate 開放 (indirect) |

| 7.7.4     | lambda lift (free var → param 化 + flat dispatch) | nested lambda の constant overhead 排除 (= closure-heavy axis 専用、*fib 30x 直接寄与 0*) |

* §2 Architecture

** 2.1 SSA pass extension としての位置

既存 Phase 7.1 SSA IR (=src/nelisp-cc.el= =src/nelisp-cc-callees.el=)
を拡張、新 pass を追加する形で実装。SSA 自体の構造変更は最小限:

- 既存 =nelisp-cc--ssa-instr= の =:meta= alist に inliner 用 hint
  (=:inline-cost= / =:escape-class= / =:recursive-depth-budget=) を追加
- 新 pass = =nelisp-cc--escape-analyse= / =nelisp-cc--inline-simple= /
  =nelisp-cc--inline-recursive= / =nelisp-cc--lambda-lift= を順序付き
  pipeline に挿入
- backend (=src/nelisp-cc-x86_64.el= / =src/nelisp-cc-arm64.el=) は
  未変更が原則 (= inliner 後の SSA を従来通り emit)

= 既存 SSA invariant checker (=nelisp-cc--ssa-verify-function=) で
inline 後 SSA の well-formedness を validate、pass 内バグの早期検出。

** 2.2 cost model

inliner の暴走を防ぐ central piece。3 軸 budget:

| 軸                | budget                | 理由                                |
|-------------------+-----------------------+-------------------------------------|
| .text size        | 関数あたり ~4 KB hard cap | I-cache 64 KB / hot loop 16 関数想定 |
| compile time      | 関数あたり ~500 ms hard cap | edit-compile cycle 体感維持        |
| recursion depth   | 8 levels hard cap     | exponential explosion guard         |

各 budget 越えで inliner は *partial inline* (=途中まで unroll、残りは
recursive call 維持) に degrade、=:meta :inline-aborted-at= で抜けた
位置を記録。

** 2.3 heuristics

cost model を満たす範囲で inline するか否かを決める predicate:

- *leaf bias*: callee が leaf (= 自身が再帰呼び出ししない) なら +50 score
- *small body bias*: callee SSA instr count ≤ 20 なら +30 score
- *hot callsite bias*: profile feedback (Phase 7.7 第 2 iteration 以降
  で profile-guided 化、初版は static heuristics only)
- *closure env tiny bias*: closure free var ≤ 2 なら escape analysis
  経由で SROA、+40 score
- *recursive depth ≥ 4 expand bias*: fib 等 deep recursion で 4 階 unroll
  すると base case に近づく、+25 score

= 累積 score ≥ 80 で inline GO、それ以下は abort。score formula は
=defcustom nelisp-cc-inline-score-threshold= で tunable。

** 2.4 escape analysis の 3 class

- *class A — closed*: closure env が caller frame に閉じる (= 上方
  reference 無し)、SROA 適用可能、inliner GO
- *class B — upward*: closure env が caller frame の外に escape
  (return value / heap store)、SROA 不可、inliner は env 保存付きで
  GO
- *class C — unknown*: SSA で証明不能 (= dynamic dispatch 経由など)、
  conservative に escape 扱い、inliner abort

= class A の比率が fib 30x 達成可能性の主指標。fib body は class A
100% 想定、本 Phase で実証する。

** 3.3 Phase 7.7.3 — recursive inline (~3-5 週間、本 phase 主軸)

*scope*:
- recursive function (= self-call を含む callee) を depth budget 内
  unroll、cost model gate で abort 制御
- depth 4-8 unroll で base case に近づき、残り再帰部分は recursive call
  維持 (partial inline)
- inline 後 SSA に対し既存 backend の TCO + self-direct + load-var
  elision がそのまま効くこと検証
- =defcustom nelisp-cc-inline-recursive-depth= (default =4=、tunable)

*deliverables*:
- =src/nelisp-cc-inline.el= 拡張 (~+400 LOC)
- ERT delta +18 (=test/nelisp-cc-inline-recursive-test.el=)
- bench: =fib(30) 12.0x → 30x+= attempt、達成失敗時は §6 risk
  mitigation で iterate

*50-100% 上乗せ根拠 (T116 review D-7 + T125 review B-1 reflected、再導出)*:

T125 finding で v1 の calculation 「8x-15x 上乗せ × T96 baseline 12x =
18-30x」が単位混乱 (上乗せの定義不明 + 算術不整合) を起こしていたため、
本 v2 で *call-count saving 起点の per-iteration cycle ベース* で再導出:

1. *call-count saving*: T96 で fib(30) = 2.7M call。depth 4 unroll で
   fib leaf call 数を 1/16 (= 2^4) に削減 = 1.6M call 削減、残 1.1M call は
   recursive depth 4 超で partial inline abort
2. *cycle saving*: 削減 call 1.6M × per-call overhead 6 cycle = 9.6M cycle
   = ~3 ms (3 GHz host 想定)
3. *baseline iter cost*: T96 fib(30) bench は 50ms / 10 iter = 5 ms / iter
4. *net speedup*: 3 ms saving / 5 ms baseline = *60% speedup* で baseline
   12x → ~19x (= **+58% 上乗せ達成**、§1.2 表「50-100% 上乗せ」の下端と整合)
5. *上限要因*: 残 30-40% の theoretical gain は以下で吸収され、実測 50-100%
   範囲 = *18x-24x* に着地が現実予測:
   - I-cache pressure (= 4x copy で hot path 0.5-1 KB → 8-16 KB 程度に膨張)
   - branch predictor warm-up (= 新 inline body に対する BTB miss)
   - register pressure 増 (= depth 4 unroll で live SSA value 4x、spill 増)
6. *30x 到達条件*: depth 6+ unroll が必要 (= 2^6 = 64x call reduction)、
   ただし §2.2 cost model の =.text size hard cap 4 KB / 関数= で抑制される
   ため structural には reachable 限定的。GO gate (≥30x) は *stretch goal*、
   DEGRADE gate (20x ≤ fib < 30x) が *default outcome* と位置付け

*予測確率付き risk acknowledge (T125 finding B-1 mitigation)*:
- *GO gate (≥30x) ≈ 20%*: depth 6+ unroll + cost model 例外運用 + heuristics
  positive スコープが揃ったときのみ
- *DEGRADE gate (20x ≤ fib < 30x) ≈ 50%*: depth 4 default + 18x-24x の
  実測予測中央値、最有力 outcome
- *ABORT gate (<20x) ≈ 30%*: cost model abort 多発 + I-cache pressure 想定
  超 + branch predictor 悪化の重複時
- = stakeholder の expectation calibration 用、Phase 7.7.3 着手時 §7.1
  re-prioritise gate でも参照

*完遂 gate (本 doc 主目的、3 段階 binary 判定 — T116 review C-6 reflected)*:
- *"GO" gate*: =fib(30) ≥ 30x= で正規 ship + bench-results md で 30x lock
  + Doc 28 §11.5 errata の defer 先 mission complete 宣言
- *"DEGRADE" gate*: =20x ≤ fib(30) < 30x= で fallback ship +

  "30x 再 attempt path" 追加 (= Phase 7.7 第 2 iteration への引き継ぎ)

  本 sub-phase は revert
- =fact-iter / alloc-heavy= 既存 PASS 維持 (regression-free 必須、3 tier
  gate 全てで条件)
- code size budget 順守 (.text 肥大 < 4 KB / 関数)
- compile time budget 順守 (< 500 ms / 関数)
- 既存 ERT regression-free

* §6 Risks (= code size explosion / compile time / cache pressure)

** 6.1 code size explosion (.text 肥大、I-cache pressure)

*risk*: recursive inline depth 4 で fib body が ~16x copy、.text 肥大
で I-cache miss 増、speedup を相殺するか負方向

*mitigation*:
- §2.2 cost model の .text size hard cap (4 KB / 関数)
- =defcustom nelisp-cc-inline-text-size-cap= で実機 tuning
- bench で .text 計測 + I-cache miss rate 計測 (=perf stat -e
  L1-icache-load-misses=) を 7.7.3 ship 前に実行
- 越えた場合 partial inline (= 途中で unroll 停止 + recursive call 復活)
  に degrade

** 6.2 compile time spike

*risk*: inline + escape pass で 1 関数あたり ~500 ms 越え、edit-compile
cycle 体感劣化

*mitigation*:
- §2.2 cost model の compile time hard cap (500 ms / 関数)
- 1-pass cost model (linear walk、=O(n)= where n = SSA instr count)
- profile feedback iteration は Phase 7.7 第 2 iteration 以降に defer
- =nelisp-cc-inline-compile-time-cap= で実機 tuning

** 6.3 cache pressure (D-cache + I-cache 同時)

*risk*: inline 後 hot path 関数群が I-cache 64 KB 超 + closure SROA で
D-cache 経路変化、cache replacement で speedup 食う

*mitigation*:
- hot loop 16 関数想定 (16 × 4 KB = 64 KB) を I-cache budget 上限
- =nelisp-cc-inline-stats= の =:text-size-delta= 監視
- =perf stat -e LLC-load-misses= で D-cache miss 計測、bench harness
  に組み込み

** 6.4 fib 30x 達成不能 (本 phase 主目的の miss)

*risk*: recursive inline + escape analysis で fib(30) が 20x 止まり、
30x gate 未達 (= Doc 28 §11.5 errata で defer された旧 gate を再 attempt
するも未達)

*mitigation*:
- §5.3 fallback target =20x= で degrade 受容、bench-results md で
  achievable 値 lock
- *本 doc は v1.0 ship gate 外 = release blocker ではない*、未達でも
  v1.x release は可能
- Phase 7.7 第 2 iteration (profile-guided + より aggressive heuristics)
  で再 attempt の余地

** 6.5 SSA invariant 破壊 (inliner pass バグ)

*risk*: substitute / return rewrite で dangling reference / phi node
矛盾、backend で silent wrong-result

*mitigation*:
- §5.2 SSA invariant checker を全 pass 直後 強制実行 (default =t=)
- ERT +30〜+50 で各 substitute / rewrite path covered
- =nelisp-cc-inline-verify-after-pass= = =t= を ship 時も維持 (cost
  許容範囲、後続 ship で =nil= 化検討)

** 6.6 escape analysis false negative (class A → 実は B)

*risk*: closure env が実は upward escape しているのに class A 誤判定、
SROA で wrong-result

*mitigation*:
- §2.4 class C デフォルト conservative (=defcustom
  nelisp-cc-escape-conservative= = =t=)
- =nelisp-cc--escape-walk-uses= で全 use chain 走査 (= reachable from
  any return / store)
- ERT +12 (=test/nelisp-cc-escape-test.el=) で known escape patterns
  全 cover
- ambiguous case 専用 stress test (1000 random closure expression
  generated + verify)

** 6.7 lambda lift で free var capture semantics 破壊

*risk*: lambda lift で free var を param 化する際、原 closure semantics
(= captured value が caller scope で mutate されたとき visible) を破壊

*mitigation*:
- escape class A 限定で lift (= caller scope mutation 無し前提)
- mutation detect は escape walk と同 pass で実施
- ERT +10 で mutate-after-capture pattern 全 cover

** 6.8 heuristics false positive (= 不要な inline で速度低下)

*risk* (T116 review D-8 reflected): §2.3 heuristics の累積 score ≥ 80
で inline GO 判定でも、実機 bench で「inline すると遅くなる」 corner
case があり得る:
- caller の register pressure 増、L1d-cache spill 増
- branch predictor の cold path 増加
- I-cache footprint 拡大による LRU eviction 増

*mitigation*:
- per-function bench harness で inline 前後比較、negative speedup 検出時
  は cost model score 閾値を per-function override
- =defcustom nelisp-cc-inline-score-threshold-override= alist
  (= ((SYMBOL . SCORE) ...) で関数単位 override)
- ~5% 以上の regression は inline 自動 abort + log 記録
  (= =:meta :inline-aborted-by-regression= で記録)
- 7.7.3 ship 後 1 週間の bench soak で regression-prone function を
  detect、override alist の seed list を bench-results md に固定
