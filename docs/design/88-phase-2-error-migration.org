** §2.3 condition hierarchy

GNU Emacs の =(put 'TAG 'error-conditions LIST)= property により condition
hierarchy が表現される. =condition-case= は handler clause の tag を
=error-conditions= list と照合する.

現状 (= Phase 1 末端) の =error.rs::is_error_subtype= は **bootstrap
minimal hierarchy** のみ実装:
- exact match
- =t= = catch all
- =error= = catch all non-=quit=

GNU Emacs full hierarchy (= 例: =arith-error= → =(arith-error error)=,
=void-variable= → =(void-variable error)=) は §86.2 で elisp 側
=(put 'TAG 'error-conditions ...)= を設定して実現する.

提案 hierarchy (= =nelisp-stdlib-error.el= 冒頭で install):

#+begin_src emacs-lisp
(put 'error 'error-conditions '(error))
(put 'quit 'error-conditions '(quit))
(put 'wrong-type-argument 'error-conditions '(wrong-type-argument error))
(put 'arith-error 'error-conditions '(arith-error error))
(put 'void-variable 'error-conditions '(void-variable error))
(put 'void-function 'error-conditions '(void-function error))
(put 'wrong-number-of-arguments
     'error-conditions '(wrong-number-of-arguments error))
(put 'no-catch 'error-conditions '(no-catch error))
(put 'setting-constant 'error-conditions '(setting-constant error))
(put 'invalid-read-syntax 'error-conditions '(invalid-read-syntax error))
(put 'end-of-file 'error-conditions '(end-of-file error))
#+end_src

これが install されたら Rust 側 =is_error_subtype= は **データ駆動** に
変えるか、もしくは elisp に subtype check 自体を移管する.  推奨:
=condition-case= の matching logic を elisp =error-conditions= 経由に
切り替える (= Doc 89 で =condition-case= special form の env aware 部を
elisp 化する流れと統合可能).

(defun nelisp--read-error->signal (kind msg line col)

    ((eq kind 'lex)   (nelisp--signal-read-lex   msg line col))

    ((eq kind 'eof)   (nelisp--signal-read-eof   msg line col))

         (data (cdr err-obj))
         (msg (get tag 'error-message)))
    (cond
      ((null msg) (format "%S" err-obj))
      ((null data) msg)
      (t (concat msg ": " (format "%S" data))))))

         (conds (cons name parent-conds)))
    (list 'progn
          (list 'put (list 'quote name) ''error-conditions
                (list 'quote conds))
          (list 'put (list 'quote name) ''error-message message))))

* §4 Rust residual (= ~30-50 LOC kept)

** §4.1 残置 variant

唯一 Rust に残す variant は **=EvalError::Internal(String)=**.

理由:
- panic 等価 = elisp で recover 不能な invariant breakage を表現
  (= Sexp variant の予期しない shape、UTF-8 invalid bytes、I/O syscall
  failure 等)
- Rust callsite が ~53 ヶ所と最頻出 = elisp 移管しても "should-never-
  happen" branch を elisp 化する benefit が薄い
- =condition-case= で =(error ...)= clause で catch 可能 (= 完全に
  unrecoverable ではない、bug surface としての value あり)

** §4.2 Rust file 縮減後の skeleton

post-Doc-88 の =eval/error.rs= 推定構造:

#+begin_src rust
//! Evaluator panic-equivalent error type — Phase 8.0.2 + Doc 88 (§86.2).
//!
//! After Doc 88 ship, every user-visible error variant moved to
//! elisp `signal' callsites (see `lisp/nelisp-stdlib-error.el').
//! The remaining `EvalError::Internal' represents bug-surface only:
//! invariant breakage that elisp cannot meaningfully recover from
//! (= e.g. malformed `Sexp' variant payload, UTF-8 corruption in
//! pre-validated bytes, I/O syscall returning EBADF on a known-good
//! fd).  These still surface as elisp `(error "internal: ...")' so
//! `condition-case' with an `error' clause catches them, but no
//! finer-grained variant remains.

use std::fmt;

#[derive(Debug, Clone, PartialEq)]
pub struct EvalError {
    pub message: String,
}

impl EvalError {
    pub fn internal(msg: impl Into<String>) -> Self {
        Self { message: msg.into() }
    }
    /// Map to the elisp `(TAG . DATA)' cons that `condition-case' binds
    /// to.  All Rust-side `EvalError' instances are `error' tag now.
    pub fn signal_data(&self) -> super::sexp::Sexp {
        // (error "internal: MSG")
        super::sexp::Sexp::cons(
            super::sexp::Sexp::Symbol("error".into()),
            super::sexp::Sexp::list_from(&[
                super::sexp::Sexp::Str(format!("internal: {}", self.message)),
            ]),
        )
    }
    pub fn error_tag(&self) -> &str { "error" }
}

impl fmt::Display for EvalError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "internal: {}", self.message)
    }
}

impl std::error::Error for EvalError {}

// ReadError stays — see §5.  `From<ReadError> for EvalError' goes
// away because ReadError now signals elisp directly via
// `nelisp--read-error->signal' at the eval/read boundary.
pub use super::reader_pos::{ReadError, SourcePos};
#+end_src

LOC 試算: ~40 LOC (= 旧 308 LOC との差分 -268).

* §5 SourcePos preservation strategy

** §5.1 Rust 側 representation

=SourcePos { line: u32, col: u32 }= は **reader 内部の Rust struct
として残す**:

- =reader/lexer.rs= / =reader/parser.rs= の internal state は
  byte-level offset を tracking する (= UTF-8 multibyte char で col が
  byte width 単位) = Rust の方が効率的
- elisp に運ぶのは error path のみ = hot path に impact なし

** §5.2 elisp 側 transit shape

elisp 側 signal data に =:line= / =:col= keyword pair で attach する.
これは Emacs-compatible な慣行 (= GNU Emacs の =scan-error= 等が
=(scan-error MSG START END)= で position を運ぶのと類似).

#+begin_src emacs-lisp
(signal 'invalid-read-syntax
        (list "unmatched paren" :line 42 :col 7 :kind 'parse))
#+end_src

handler 側 access:
#+begin_src emacs-lisp
(condition-case e
    (read-from-string "(foo (bar)")
  (invalid-read-syntax
    (let* ((data (cdr e))
           (msg (car data))
           (line (plist-get (cdr data) :line))
           (col  (plist-get (cdr data) :col)))
      (format "%s at %d:%d" msg line col))))
#+end_src

** §5.3 reader → eval boundary での conversion

Rust =reader/= が =Result<Sexp, ReadError>= を返す path は、Doc 88 で
は elisp =signal= に **直接** 変換する (= =From<ReadError> for
EvalError= は廃止):

旧 path:
#+begin_src
reader → ReadError → From → EvalError::Read(rerr) → condition-case Rust
                                                  → e.signal_data()
                                                  → Sexp cons
#+end_src

新 path (= Doc 88 後):
#+begin_src
reader → ReadError → reader_to_signal_helper(rerr) → eval =(funcall
                                                           'nelisp--read-error->signal
                                                           kind msg line col)=
                                                  → elisp signal
                                                  → unwind via condition-case
#+end_src

実装上 =reader_to_signal_helper(env, rerr)= は Rust fn として書き、
内部で =env.eval(funcall-form)= を呼ぶ (= 4-arg helper invocation).
これにより Rust=>elisp boundary で SourcePos が plist として運ばれる.

** §5.4 backward compat

既存 ERT (= =reader/= test 群) が =EvalError::Read(ReadError::Lex {
... pos })= を pattern match していた箇所は、=condition-case= で
=invalid-read-syntax= を catch + =plist-get :line / :col= で抽出する
形式に書き換える.  test side change: ~30 ERT site (推定).

* §8 Risk / unknowns

** §8.1 Backtrace info loss / gain trade-off

*** §8.1.1 Rust 側 backtrace の loss

post-Doc-88 では Rust =EvalError= が =Internal(msg)= 1 variant のみ =
=cargo test= 時の Rust backtrace は粒度を落とす (= "wrong-type-argument
in =bi_car=" のような Rust file:line 情報が直接 enum variant に embedded
されない).

*** §8.1.2 elisp 側 backtrace の gain

逆に elisp =signal= path に乗ると:
- =backtrace= primitive (= =eval/builtins.rs::bi_backtrace= 既存) が
  elisp call stack を walk 可能
- =condition-case= handler が =error= を catch した時点の elisp dynamic
  stack を access できる
- ユーザは =(message "%s" (error-message-string e))= で human-readable
  diagnostic を得る

*** §8.1.3 trade-off 結論

end-user-facing は **elisp backtrace の方が valuable** (= "Rust
internal" な情報は user has no way to fix).  developer debug は
=EvalError::Internal= の =msg= field に Rust file:line を embed する
慣行 (= =format!("error.rs:{}: invariant", line!())=) で部分的に救済.

ただし net info loss は **存在する** = §88.5 ship 時に design memo
として「Rust enum 由来の static variant info を失った」を記録.

** §8.2 condition hierarchy migration race

Phase 1 末端で =error.rs::is_error_subtype= が hardcode した hierarchy
(= =error= catches non-=quit=) と、Doc 88 で elisp =error-conditions=
property に移すタイミングの不一致.

具体的: §88.1 で put-property を install しても、Rust 側
=is_error_subtype= は依然 hardcode のままなら、新 condition tag (=
=void-variable= 等) を catch する rule が **2 source-of-truth** になる.

mitigation:
- §88.1 で put-property install + Rust 側 =is_error_subtype= が
  =get 'TAG 'error-conditions= を look up するように rewrite (= ~10 LOC
  Rust 増)
- これにより hierarchy の SoT が elisp に統一される
- 既存 ERT (= =error= catches =arith-error=) の挙動は不変

** §8.3 reader boundary の Rust=>elisp call cost

§5.3 で articulate した =reader_to_signal_helper(env, rerr)= は Rust
fn 内から elisp =funcall= を発火する = boundary cross が発生.
rough cost: 1 elisp funcall ≈ 100-500 ns (= JIT 経由).  reader error
は 1 read あたり最悪 1 回発火 = total cost ~1 µs/error = 許容.

ただし =read-from-string= を loop で呼ぶ pathological case で error
を毎回踏むと cost が積もる.  unlikely in practice = §88.5 で perf
ERT 1 case 追加で gate (= 1000 error/s 最低 throughput).

** §8.4 SourcePos struct の Rust 側依存

reader 内の Rust struct =SourcePos= は eval 側からも参照される (=
=eval/error.rs:: pub struct SourcePos=) = post-Doc-88 では reader
crate の internal struct に retire.  callsite migration で
=use= path の書換が ~5 site 必要.

** §8.5 =UserError= 経由の任意 tag

=EvalError::UserError { tag, data }= は user code (= =(signal 'my-tag
DATA)=) の sink.  post-Doc-88 では =bi_signal= 内で **elisp の signal
callable に直 dispatch** = Rust enum 介在無し.

ただし =bi_signal= 自身は Rust builtin として keep (= elisp は =signal=
の primitive をどこかで踏まないと立ち上がらない、self-reference).
=bi_signal= の internal は Doc 80 で既に articulated (= =arith-error=
/ =wrong-type-argument= は dedicated variant に canonical 化、それ以外
は =UserError=).  Doc 88 で =UserError= variant 自体は消える = =bi_signal=
は **直接** elisp condition unwind を起動する形に refactor 必要.

具体的 design:
- =bi_signal(args)= が =Result<Sexp, EvalError>= を返すのではなく、
  =Result<Sexp, ElispSignal>= 等の dedicated control-flow type に変える
- =ElispSignal { tag: Sexp, data: Sexp }= = =EvalError= とは独立、
  =condition-case= matching が直接食う

これは Doc 88 内では **scope に入れない** (= =eval/= 内 control flow
の architectural rewrite に近い) = §88.5 で残課題として note + 別 Doc
候補化 (= "Doc 88-A: signal control-flow refactor").

仮実装としては: =UserError= の Rust enum は keep (= ~5 LOC)、ただし
"migrate target" 表記から外す = §1.1 row 6 を **partial migrate** に
修正する選択肢.

代替: §88.x 内で =EvalError= を完全に消すために =Result=Sexp,
ElispSignal>= type alias を導入する (= ~30 LOC Rust 増、純粋移管 net
は維持).  この alternative を採用するかは §88.5 で実装側判断.

** §8.6 unknowns 一覧

- =define-error= macro が =defmacro= 経由で正しく boot するか (= Doc
  80 substrate の =defmacro= self-boot pattern + put-property install
  順序の interplay).  smoke test 必須.
- =error-conditions= 経由の subtype check が =bi_signal= 内 hardcode
  と整合するか (= §8.2 と関連).
- reader test の plist-get migration (= ~30 site) の労力試算が適切か.
- =bi_signal= の =UserError= dependency を Doc 88 内で解決するか別 Doc
  に escalate するか (= §8.5).
