** 目的

release bar 後半 ("Emacs core 置換") を達成するため、Emacs C で
実装されている 3 核心 (buffer / redisplay / event-loop) の NeLisp
実装を新構築する。*既存 C モジュールを置き換え*、NeLisp 上で
最小限の "editor" が動作することで、NeLisp が Emacs の母体として
働ける substrate であることを実証する。

release bar 前半 (Phase 5-A Doc 12 SHIPPED-2026-04-24) で
"NeLisp 単独でパッケージ作成自立" は達成済。Phase 5-B 完了で
"NeLisp 単独で Emacs 代替 substrate 動作" を目指す。

** 背景

Phase 0 で lock された v1.0 完遂条件 4 項:
1. self-host 到達 — Phase 2 達成 (2026-04-23)
2. *3 大 core module (buffer / redisplay / event-loop) Elisp 化*
3. AI-critical path 5 領域が NeLisp 上で動作
4. 残存 C は SBCL-style 4 核 (syscall / allocator / GC inner / coding)

本 Phase 5-B は項目 2 に対応。anvil.el 開発中に感じた host C
由来の限界 (file-notify freeze / mutex cross-session 不能 /
subprocess 迂回 / Emacs server filter starvation 等) を、buffer
/ redisplay / event-loop の 3 核を Lisp で新設することで
structural に解消する下地を作る。

Rhodes 2008 "SBCL: A Sanely-Bootstrappable Common Lisp" 方針
(Phase 0 decision Bootstrap A) と並走: host C を SBCL 4 核
(syscall / allocator / GC inner / coding) のみに絞り、それ以外
は NeLisp 上に置く。

** Non-goals

- GUI redisplay (X / Windows / Cocoa) は本 Phase 範囲外。
  terminal-only から着手、GUI は Phase 6+ で検討。
- Major / minor mode 実装互換は本 Phase 範囲外。substrate が
  揃えば mode は Elisp で書ける (Phase 5-A の loader で load 可)。
- Unicode normalization の完全対応は範囲外 (encode/decode は
  host に委譲、bytes→chars 変換は Phase 7+)。
- Network I/O substrate (process / network connection) は
  Phase 5-C 以降。本 Phase は端末入出力と buffer I/O のみ。
- =redisplay= の bidi / ligature / font shaping 等は範囲外。
  ASCII + CJK full-width 2 分類の monospace grid のみ扱う。
- =package.el= 互換 installer は Phase 5-A と同じく別課題。

** §2.1 Buffer data structure

*Q*: buffer のテキスト表現をどう持つか。Emacs C は gap buffer
(2-part contiguous array + gap region) を使う。

選択肢:
A. Gap buffer (Emacs 同型) — 単純、挿入/削除 O(N/W)、Emacs
   semantics 完全再現しやすい
B. Rope (平衡二分木) — O(log N) 全操作、実装複雑、marker
   整合も再設計要
C. Piece table (VS Code 他) — undo-friendly、実装は比較的
   簡潔、marker は位置リスト保持

*推奨: A (Gap buffer)*。Emacs semantics 再現優先、perf は後続
Phase で移行可。rope/piece-table 移行は data-abstraction を
interface で切る前提で将来 swap 可能に書く。

** §2.2 Text representation

*Q*: バッファ内の文字単位。

選択肢:
A. Host Elisp =string= (multibyte 含む) — Emacs 同等、encode
   / decode は host 委譲
B. Byte vector + 明示 codec layer — 純粋だが encode/decode を
   NeLisp 内実装必要
C. Code-point vector (fix32 配列) — unicode 直球、memory 重い

*推奨: A (string)*。Phase 5-B は correctness が先、codec は
SBCL 4 核に残して host 委譲のままで良い。

** §2.3 Marker / overlay 実装

*Q*: =marker= / =overlay= をどう保持するか。

選択肢:
A. buffer-local doubly-linked list (weak-ref ではなく GC root) —
   Phase 3c mark-sweep が reach 可能、insert/delete で毎 marker
   offset 更新
B. Interval tree — 多 overlay 時に O(log N) query、実装複雑
C. B-tree (Emacs 流、itree) — Emacs 2024+ 方針に準拠

*推奨: A (linked list)*。MVP は overlay 数 < 1000 想定、
linked list で十分。interval tree は perf profile 後移行可。

** §2.4 Redisplay 戦略

*Q*: screen 描画をどう組むか。

選択肢:
A. Terminal-only + buffer 変更毎 full-repaint — 最単純、
   flicker あり
B. Terminal-only + dirty-region incremental (row 単位) —
   flicker 無し、実装中規模
C. Terminal + GUI 両対応 — scope 過大

*推奨: B (terminal dirty-region)*。A は UX 悪すぎ、C は本
Phase 範囲外。window を row x col grid に抽象、modified row
だけ terminal escape で再描画。

** §2.5 Event loop モデル

*Q*: input event ループの基盤。

選択肢:
A. Standalone event loop (new recv loop) — 独立実装、Phase 4
   actor は使わない
B. Phase 4 actor を再利用 — input source を 1 actor として
   spawn、buffer update / redisplay も actor message で trigger
C. Hybrid — keyboard は actor、timer は host

*推奨: B (Phase 4 actor)*。anvil.el pain の多くが event loop
blocking に起因、actor の yield point を活用。Phase 4 の
=nelisp-actor-tick-interval= をそのまま event drain 周期に使う。

** §2.6 Host Emacs との共存戦略

*Q*: NeLisp buffer が host Emacs buffer と同居するとき、どう
名前空間分離するか。

選択肢:
A. Parallel — NeLisp buffer は =nelisp-buffer-*= API、host
   buffer は触らない (Phase 5-B 内は完全隔離)
B. Shim — NeLisp buffer ops は host buffer を wrap
   (bootstrap、perf 同等)
C. Replacement — host buffer 概念を NeLisp-buffer で完全置換
   (Phase 5-B 終点の理想形)

*推奨: A (Parallel)*。NeLisp-buffer の correctness を独立に
検証可、host Emacs の安定性を損なわない。C は Phase 5-C 以降、
Emacs substitute として起動する境地まで行ってから。

** §2.7 Concurrency モデル

*Q*: buffer に対する並行アクセスをどう制御するか。

選択肢:
A. Single-threaded (Emacs 流) — 1 event loop、buffer 操作
   atomic
B. Per-buffer actor (Phase 4) — buffer 毎 1 actor、message
   駆動
C. STM (Phase 4b optional) — log-replay 方式、複雑

*推奨: A (single-threaded)*。Phase 5-B の目的は core semantics
再現、concurrency の上積みは Phase 6+。ただし event loop は
actor で組むので「buffer 操作 = actor message」の自然な延長で
B にいつでも migrate 可能なように interface を切る。

** §2.8 Test strategy

*Q*: NeLisp-buffer の正しさをどう担保するか。

選択肢:
A. ERT unit (insert/delete/point/mark 単体) のみ
B. A + host buffer parity test (同操作で NeLisp-buffer と
   host-buffer の text/point/mark が一致)
C. A + B + round-trip (serialize → restore 一致)

*推奨: B (A + parity)*。Phase 5-A と同じく ~50 ERT 目標、
parity test で semantics 漏れを早期検出。serialize は Phase
5-C で。

** §2.9 Merge gate 成功条件 (§3.6 equivalent)

選択肢:
A. 全 §3 sub-phase SHIPPED + make test green + NeLisp-buffer
   上で =insert= / =delete-region= / =goto-char= 動作
B. A + event loop から key input → buffer update が動く
C. B + redisplay を terminal で visible 描画

ではない。

** §3.4 Event loop (actor integration)

目的: input event → buffer update → redisplay の flow。

作業:
- =src/nelisp-eventloop.el= 新設
- keyboard-input actor: =tty-raw-mode= stdin fd から byte
  読み取り、key event として dispatch message を送信
- main event-loop actor: message 受信 → dispatch table (key
  → command) → command 実行 → =nelisp-redisplay=

  change + redisplay output 検証
