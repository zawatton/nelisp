* Goal

Run anvil module tools (=anvil-memory= / =anvil-worklog= / future =anvil-shell-filter=)
under NeLisp standalone — the =anvil-runtime mcp serve= binary — without depending on a
host Emacs runtime.  Final shape: =tools/call memory-add= / =memory-list= / =memory-search= /
=worklog-add= / =worklog-list= / =worklog-search= dispatched end-to-end through the chain
=anvil-runtime → build-tool eval → anvil-memory.el (pure elisp) → emacs-sqlite-ffi.el (pure
elisp) → in-process libffi → libnelisp_runtime.so:nl_sqlite_* → real SQLite DB=.

* Design directives (user, 2026-05-03)

1. *Rust 核は最小化* — no per-feature Rust glue; only generic primitives + bridges.
2. *elispで実装可能なものは elispで* — every higher-level layer (sqlite-* / cl-defun
   handling / anvil-memory logic) stays in pure elisp, even when slower / longer.
3. *SQLite は必須* — no in-memory fallback; the read AND write paths must persist
   to the canonical =Notes/.anvil-{memory,worklog}/*.db= stores.
4. *PR は不要* (zawatton/* repos = personal) — commits + push only, no PR creation.
5. =nelisp-ffi= リポジトリ使用 (= =zawatton/nelisp-ffi= elisp-ffi fork).

* Architecture (Pattern C)

#+begin_example
MCP request {tools/call memory-list ...}
  ↓
anvil-runtime (Rust binary)
  ↓ AnvilModuleRegistry::call
  ↓   - introspect handler formals
  ↓   - map JSON args by formal name (-/_ aliasing)
  ↓   - emit (handler POS... :K1 V1 ...)
build-tool eval (`<self-host-fallback>` path)
  ↓   - sf_cl_defun expanded &key into &rest --cl-keys + let* wrapper
  ↓   - the let* binds K via (or (car (cdr (memq :K --cl-keys))) DEFAULT)
anvil-memory--tool-list  (anvil-memory.el, pure elisp)
  ↓
anvil-memory-list  (cl-defun, pure elisp)
  ↓
sqlite-select  (emacs-sqlite-ffi.el polyfill, pure elisp)
  ↓
nl-ffi-call PATH FUNC SIG ARGS...  (build-tool/eval/ffi.rs, libffi-rs)
  ↓
nl_sqlite_query (libnelisp_runtime.so, rusqlite)
  ↓
real DB at /home/.../anvil-memory-index.db
#+end_example

** Why this shape

- =build-tool= keeps its Sweep-6 *zero-internal-crate-deps* invariant.  Only
  *external* crates (libffi, libloading, sha1, sha2, md5, chrono) get pulled in.
- =sqlite-* / nelisp-syscall-* / cl-* / pcase= live in elisp where each can
  evolve without touching Rust.
- =libnelisp_runtime.so= already contains rusqlite — no Rust *functionality*
  added, only a generic FFI bridge to call into it.

* Known limitations / Phase 8+ candidates

- =cl-defun &key= now expanded by Rust =sf_cl_defun=, but cl-defmethod (= &dispatch on
  type) still routes through elisp =cl-defmethod= stub — works for simple specializers
  via intern hack but not full multi-method dispatch.
- =rx= macro (= regex DSL) currently stubbed as no-op.  All anvil callsites use raw
  regex strings so this is harmless today, but blocks any module that builds regexes
  via =(rx ...)= constructor.
- =nelisp-eval-form= takeover path is the *minimal* fallback (= =(defun
  nelisp-eval-form (form env) (eval form))=).  The FULL =nelisp-eval.el= /
  =nelisp-macro.el= load fails during =bootstrap_self_host=; we get away with the
  minimal because build-tool's eval is sufficient for the anvil module load
  surface.  Future Phase 8: fix the FULL bootstrap so macros use =nelisp--macros=
  hashtable consistently.
- =anvil-memory-list= / =-search= row-shape is currently *list of plists* (= Emacs 30
  default) but anvil-host's MCP serializer renders the plist KEYS as JSON object
  property names paired oddly with the LIST values.  The data is correct; the
  presentation needs an =anvil-server-encode-result= polyfill that emits proper
  JSON arrays of objects.
- =anvil-shell-filter= tools (= =shell-filter= / =shell-run= / =shell-tee-grep=) work
  via the *Rust-side* compat registry already; not yet routed via Pattern C
  (= they don't need to be — the Rust path is faster).
