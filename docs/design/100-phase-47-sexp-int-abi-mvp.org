* §0 TL;DR

** §0.1 What §99.C left unfinished

§99.A-C (commits ca217eb8 / 9775a3a4 / d687f743, 2026-05-11) shipped
the *infrastructure* for elisp source → ET_REL `.o' → static archive
→ link → C-callable symbol.  Verified end-to-end via
=bi_nl_fact_i64= (= cargo test `nelisp_fact_i64_matches_rust_reference'
+ `nelisp_fact_i64_known_values' 2/2 green, range 0..=20! all
match).

But the §99.C swap pattern is *i64-only*: every elisp-compiled
function exports `extern "C" fn(i64, i64, ...) -> i64'.  The Rust
`bi_*' shim does Sexp-unwrap → i64 → call → i64 → Sexp-wrap.

This means §99.C's machinery is *useful only for purely-numeric
unary/binary builtins on i64 fixnums*.  None of Doc 90's D1/D2/D3
targets fit that shape:

| Doc 90 deferral | Target                           | Inputs              | Sexp variants needed |
|-----------------+----------------------------------+---------------------+----------------------|
| D1              | =env.rs= 19 methods              | symbol + value pair | Symbol / Cons / Vector |
| D2              | =parse_let_binding=              | cons-pair           | Cons                 |
| D3              | =bind_formals=                   | cons-chain + values | Cons / Vector        |

Doc 100 closes that gap by introducing the *first* Sexp-typed ABI
boundary: =Sexp::Int= (= fixnum-only).  Cons / Vector / Symbol /
Str are deferred to Doc 101+ (= one variant per follow-up doc).

** §0.2 Why start with Sexp::Int when D1/D2/D3 all need Cons

Three reasons:

1. *ABI fragility is the load-bearing risk* (Doc 99 §5.1).  Picking
   the simplest payload (= i64 fixnum) lets us prove the ABI shape
   in isolation, *before* layering on the heap-pointer +
   refcount story that Cons / Str carry.  If the ABI is wrong, we
   want to discover that with a 3-line builtin, not a 100-line one.

2. *=bi_truncate= Int branch is a degenerate-but-real swap target*.
   It's a unary function on `Sexp::Int → Sexp::Int' with identity
   semantics (`Sexp::Int(n) => Ok(Sexp::Int(*n))').  Trivial enough
   that any test failure points at the ABI, not at the algorithm.
   Real enough that it lives in production `eval/builtins.rs:1236'
   and is exercised by `(truncate N)' from elisp `format'.

3. *Phase 47 grammar today is i64-only* (Doc 97 §97.b/c).  Adding
   Cons / Vector / Str support requires Phase 47 to gain heap
   allocation + tagged-pointer arithmetic + GC root registration
   semantics.  None of those are in scope for an ABI MVP.  Sexp::Int
   needs only one new grammar form (= `extern-call') + one new
   relocation type emission (= R_X86_64_PLT32 against SHN_UNDEF).

* §1 Background

** §1.1 Where §99.C's i64 ABI hits its ceiling

The §99.C spike (= `bi_nl_fact_i64' / `nelisp_fact_i64') works
because factorial's full state fits in one i64 register: arg in
RDI, return in RAX, no Sexp values cross the FFI boundary.  The
Rust shim handles all Sexp-aware logic *before* calling the elisp
`.o':

#+begin_src rust
  // build-tool/src/eval/builtins.rs:1262 — §99.C shim shape.
  fn bi_nl_fact_i64(args: &[Sexp]) -> Result<Sexp, EvalError> {
      require_arity("nl-fact-i64", args, 1, Some(1))?;
      let n = as_int("nl-fact-i64", &args[0])?;          // Sexp → i64
      if !(0..=20).contains(&n) { ... }                  // range check
      let result = crate::elisp_cc_spike::fact_i64(n);   // i64 → i64 across FFI
      Ok(Sexp::Int(result))                              // i64 → Sexp
  }
#+end_src

This pattern handles *every*  builtin whose only meaningful state
is a fixnum.  But anything that needs to inspect cons-pair
structure, dispatch on symbol identity, or build a result list
inside the elisp body cannot use it.  =bind_formals= would have to
unwrap the entire arglist + cons-chain in Rust before calling the
elisp `.o', which leaves almost nothing for the elisp side to
implement (= defeats the swap).

** §1.2 What the Phase 47 chain already supports

Doc 97 §97.b/c shipped these forms:

- `(defun NAME (PARAMS...) BODY)' — SysV AMD64 callee with up to 6
  args in RDI/RSI/RDX/RCX/R8/R9
- `(NAME ARG ...)' — *self-recursive* call resolved by the
  defun-alist; emits direct `call' with rel32 fixup resolved by
  `nelisp-asm-x86_64-resolve-fixups' against in-buffer label
- `(if T E1 E2)' / `(while T BODY)' / `(cond ...)' / `(and/or)'
- arithmetic `+ - *' and comparisons `< > <= >= ='

Crucially absent: *cross-translation-unit calls*.  Every `call'
instruction emitted today resolves *within* a single Phase 47 input
buffer.  When the parser sees `(foo-name 42)' and `foo-name' is
*not* in `defuns', it signals `:not-value-expr' (see
=nelisp-phase47-compiler.el:331=).

The ELF writer (Doc 91) already defines the constants for external
references — `nelisp-elf--shn-undef = 0' /
`nelisp-elf--r-x86-64-plt32 = 4' — but the compiler has no code
path that registers a symbol as undefined.  Today's symtab
contains only `STT_FUNC' + `SHN_LOCAL_TEXT' entries.

** §1.3 The Sexp enum at runtime

=build-tool/src/eval/sexp.rs:58= declares:

#+begin_src rust
  #[derive(Debug, Clone, PartialEq)]
  #[repr(C, u8)]
  pub enum Sexp {
      Nil,                       // 0
      T,                         // 1
      Int(i64),                  // 2 + i64 payload
      Float(f64),                // 3 + f64 payload
      Symbol(String),            // 4 + heap String
      Str(String),               // 5 + heap String
      MutStr(NlStrRef),          // 6 + refcounted handle
      Cons(NlConsBoxRef),        // 7 + refcounted handle
      Vector(NlVectorRef),       // 8 + refcounted handle
      // ... (record / hash-table / etc. retired in Doc 50)
  }
#+end_src

Variants are heterogeneous in size: `Int' = 8-byte payload,
`Symbol' = 24-byte `String', `Cons' = 8-byte ref-handle.  Rust's
`#[repr(C, u8)]' pads every variant to the largest payload (=
the `String' variants today), so a `Sexp' value is currently
~32 bytes per slot.

Doc 99 §5.1 flags this layout as *load-bearing fragility*: any
field reorder or variant add would silently invalidate every
compiled `.o'.  An ABI that pins the byte layout is exposed to
that risk; an ABI that goes through *helpers* is not.

* §2 ABI design (= Sexp::Int MVP)

** §2.1 The three candidate schemes

We considered three shapes for the Sexp-typed FFI boundary:

*Option A — repr(C) Sexp pass-through (direct)*

#+begin_src c
  // elisp .o sees `Sexp' as a 32-byte struct; reads tag byte + payload directly.
  Sexp nelisp_truncate_int(Sexp arg);
#+end_src

- Pros: 0 helper-call overhead; closest to "real" elisp semantics
- Cons:
  - Doc 99 §5.1 layout fragility — Phase 47 must emit exact byte
    offsets matching Rust's `#[repr(C, u8)]'
  - Sexp variants are heterogeneous; the elisp .o would need to
    know all their sizes, not just Int's
  - Phase 47 grammar today has no struct-passing convention; adding
    one is a substantial extension

*Option B — i64 + tag byte separately (= unwrap at ABI boundary)*

#+begin_src c
  // Pass each Sexp as 2 args: tag byte + i64 payload.  Return likewise.
  void nelisp_truncate_int(uint8_t tag_in, int64_t payload_in,
                            uint8_t *tag_out, int64_t *payload_out);
#+end_src

- Pros: no struct layout dependency at all; Phase 47 stays in
  scalar/i64 land (= no grammar extension needed beyond `extern-call')
- Cons:
  - Doesn't extend to non-Int variants (Cons handle / Str heap
    pointer don't fit in one i64 cleanly)
  - When Doc 101 adds Cons we'd rewrite the ABI; defeats the
    "MVP that scales" property
  - 2 args per logical Sexp = burns half the SysV arg-reg budget

*Option C — opaque Sexp handle + FFI helpers (= this doc's choice)*

#+begin_src c
  // elisp .o sees `Sexp' as `void *' (= opaque handle).
  // Helpers (= Rust extern "C") create / inspect / unwrap handles.
  extern void *nl_sexp_int_new(int64_t n);
  extern int64_t nl_sexp_int_unwrap(const void *h);
  extern int32_t nl_sexp_is_int(const void *h);
  // .... extends to nl_sexp_cons_new / nl_sexp_cons_car etc. in Doc 101.

  void *nelisp_truncate_int(const void *h_in);
#+end_src

- Pros:
  - *Opaque*: Sexp layout can change in Rust without breaking any
    `.o'; only the helper signatures are load-bearing
  - *Extensible*: Doc 101 adds `nl_sexp_cons_new' / `_car' / `_cdr'
    helpers; the elisp .o gains new ops without rewriting existing ones
  - *Naturally compatible with future GC*: helpers can register
    roots without changing the calling convention
  - Phase 47 needs only ONE new grammar form (= `extern-call') and
    ONE new relocation type emission (= R_X86_64_PLT32 against
    SHN_UNDEF) — both already partially present in the ELF writer
- Cons:
  - Per-Sexp ops cost a function call (= ~2-5 ns on modern x86)
  - The helper API surface grows linearly with covered Sexp variants
  - Allocation per result handle (= until §2.4 ownership rules
    introduce a free-list / arena)

*Decision*: Option C.  Reasons:
1. Layout fragility (Doc 99 §5.1) is the single biggest design
   risk; Option C eliminates it.
2. The ABI must extend to Cons/Str/Symbol within the next 2-3 docs;
   Option C extends linearly, Option B doesn't extend, Option A
   compounds the fragility.
3. For Sexp::Int specifically, the per-op overhead is negligible
   compared to actual builtin work — `bi_truncate' does ~5 ns of
   useful work either way; an extra 5 ns on FFI boundary is in the
   noise.

** §2.2 The Sexp::Int MVP API surface (Rust side)

New crate module: =build-tool/src/elisp_abi.rs=.  All functions
=#[no_mangle] pub extern "C"= so the linker exposes them to the
elisp `.o' via PLT32 references:

#+begin_src rust
  /// Allocate a fresh Sexp::Int handle.
  /// Returns Box::into_raw — caller takes ownership and MUST eventually
  /// pass the handle to `nl_sexp_drop' or transfer it to another
  /// consumer.  For v0 the only caller paths are elisp .o (= consumed
  /// via the Rust shim's `Box::from_raw') and the shim itself.
  #[no_mangle]
  pub extern "C" fn nl_sexp_int_new(n: i64) -> *mut Sexp {
      Box::into_raw(Box::new(Sexp::Int(n)))
  }

  /// Read the i64 payload of an Sexp::Int handle.
  /// Pre-condition (= UB if violated): h must point to a live
  /// Sexp::Int variant.  The elisp `.o' MUST call `nl_sexp_is_int(h)'
  /// first when the type is not known at compile-time.
  /// MVP semantics: returns 0 for any non-Int variant (= defensive),
  /// but callers should treat that as a bug.
  #[no_mangle]
  pub extern "C" fn nl_sexp_int_unwrap(h: *const Sexp) -> i64 {
      let s = unsafe { &*h };
      match s {
          Sexp::Int(n) => *n,
          _ => 0,
      }
  }

  /// 1 if h points to an Sexp::Int variant, 0 otherwise.
  /// Returns int32_t to match SysV AMD64 ABI for `bool'-like values.
  #[no_mangle]
  pub extern "C" fn nl_sexp_is_int(h: *const Sexp) -> i32 {
      let s = unsafe { &*h };
      matches!(s, Sexp::Int(_)) as i32
  }

  /// Drop a handle.  Called by the elisp `.o' when it abandons an
  /// argument handle without returning it, OR by the Rust shim when
  /// taking ownership and re-wrapping into the public `Sexp' API.
  ///
  /// MVP: simple `Box::from_raw'.  Doc 101 will extend this to handle
  /// refcounted variants (Cons / MutStr / Vector).
  #[no_mangle]
  pub extern "C" fn nl_sexp_drop(h: *mut Sexp) {
      if !h.is_null() {
          unsafe { drop(Box::from_raw(h)) };
      }
  }
#+end_src

*Total surface*: 4 functions, ~60 LOC.

** §2.3 The Phase 47 grammar extension

One new value-producing form: =(extern-call SYM ARG ...)=.

- SYM is a symbol whose name becomes an SHN_UNDEF entry in the
  output `.o''s symtab.
- ARG... are existing value forms (= i64-producing).
- Up to 6 args (= SysV register budget).
- Result is in RAX (= caller treats it as `*mut Sexp', which on
  x86_64 is also pointer-sized = i64-compatible).
- No type checking at parse time — the elisp author is responsible
  for matching the C signature.  An ABI mismatch becomes a runtime
  SEGV, not a compile-time error, in v0.

IR shape added to =parse-value=:

#+begin_src elisp
  (:kind extern-call :name SYM :args (IR ...))
#+end_src

Code-gen (=--emit-extern-call=):

1. Emit code for each ARG in RDI/RSI/.../R9 order (= identical to
   intra-unit `call' IR).
2. Emit `call rel32' with placeholder zero in the rel32 field.
3. Record a fixup with `:kind 'extern-plt32', `:symbol SYM',
   `:offset <buf-offset-of-rel32-field>'.
4. Result is in RAX (= caller-saved per ABI; cooperate with
   existing register-spilling code as if it were a regular call).

** §2.4 Ownership / lifetime rules (v0)

MVP ownership is *transfer-only* (= no refcount, no GC root
registration):

- *Input arguments* — the Rust shim allocates each arg handle via
  `Box::into_raw' before calling the elisp `.o'.  Caller transfers
  ownership to callee; the elisp `.o' is responsible for dropping
  each arg (via `nl_sexp_drop') if it doesn't consume the handle.
- *Return value* — the elisp `.o' allocates a fresh handle via
  `nl_sexp_int_new' (or whichever variant constructor) and returns
  it.  The Rust shim takes ownership via `Box::from_raw' and
  re-wraps as a public `Sexp'.
- *No retention* — the elisp `.o' MUST NOT keep handles past the
  call return.  No global state, no static refs.  This is enforced
  by the language (= no `static' / `global' forms in Phase 47 v1).

This is wasteful for the identity-case (`bi_truncate(Int(n)) → Int(n)'
allocates twice when it could pass through).  Doc 101 will introduce
*handle move-in/move-out* semantics that let the elisp side return
its input unchanged.  v0 keeps it simple.

** §2.5 ELF / linker mechanics

Phase 47 chain → ET_REL `.o' with extern symbol references:

1. The compiler collects every `extern-call' form's SYM name during
   parse, accumulates a set =extern-syms=.
2. After defun emit, before `resolve-fixups', each extern fixup
   recorded in step (3) of §2.3 stays *unresolved* in the buffer
   (= rel32 field remains 0).
3. ELF writer:
   - Adds one symtab entry per name in =extern-syms= with
     `st_shndx = SHN_UNDEF', `st_info = STB_GLOBAL | STT_NOTYPE',
     `st_value = 0', `st_size = 0'.
   - Adds one `.rela.text' relocation per extern fixup with
     `r_info = (sym_idx << 32) | R_X86_64_PLT32', `r_offset' =
     fixup buffer offset, `r_addend = -4' (= standard for PLT32
     `call' instructions).
4. `ld' at static-link time matches each SHN_UNDEF symbol against
   the Rust `.rlib`'s exported `nl_sexp_int_new' / `_unwrap' /
   `_is_int' / `_drop' globals.  Standard linker-resolution flow.

Rust side keep-alive: each helper carries
`#[used] #[no_mangle] pub extern "C"' so cargo doesn't strip it
during dead-code elimination.  The elisp `.o''s PLT32 reference
*should* be enough to retain the symbol at link time, but `#[used]'
guards against the build edge-cases where the `.rlib' is processed
before the elisp `.o' is added to the link line.

* §5 Risks / unknowns

** §5.1 Memory leak on partial-eval / unwind

The v0 ownership rule (= callee always drops or returns) is
fragile in the face of mid-call unwinds (= `signal' / `panic').
If the elisp `.o' calls a helper that signals via env-side
mechanism, partially-built handles leak.

*Mitigation*: v0 elisp .o source has no error-signaling paths (=
the only ops are `extern-call' to helpers that return success
always).  Doc 101's Cons ABI will introduce `nl_sexp_drop_all'
helpers + a per-call arena that auto-drains on return regardless
of unwind state.

** §5.2 Phase 47 grammar collision with future macro forms

`extern-call' name collision risk: if a future elisp source
contains a function literally named `extern-call', the parser
ambiguates.  Probability low (= names with hyphens + verb-ish are
not typical in current `lisp/*.el'), but worth guarding.

*Mitigation*: §100.A reserves `extern-call' as a Phase 47 special
form name; the validator signals
`:reserved-name-as-function-name' if a `defun' tries to define it.

** §5.3 Linker order: `.rlib' vs static archive

`cargo' typically lists `.rlib' files after `cargo:rustc-link-lib=static=...'
archives, which means PLT32 references in elisp `.o' to Rust-exported
helpers *should* resolve.  But linker behavior depends on
`-Wl,--as-needed' / `--start-group' flags Cargo may emit by
default.

*Mitigation*: §100.B verifies resolution with `nm' + an integration
test before §100.C lands.  If link ordering is wrong, the fix is
likely a `println!("cargo:rustc-link-arg=-Wl,--start-group");' /
`--end-group' pair around the static archive in `build.rs'.

** §5.4 Sexp variant layout drift over time

This ABI is *opaque* in v0 (= elisp `.o' only touches handles via
helpers).  But Doc 101 will face the same Sexp-layout question for
Cons (= 1 helper call per car/cdr access is acceptable for D2/D3
but a Cons-heavy elisp body would regress vs Rust).  Future docs
may want to opt into Option A (= direct layout) for hot Cons code.

*Mitigation*: Doc 101 will benchmark per-Cons-access overhead on
D2/D3-shaped workloads.  If unacceptable, escalate to a hybrid (=
hot-path Cons via direct layout + Doc 99 §5.1 frozen-layout
CI gate, cold-path via helpers).
