** 1.2 4 核の定義 (=docs/03-architecture.org §0= LOCKED)

SBCL が 25 年間維持してきた 4 核境界を NeLisp に transfer:

| #   | 核                              | 役割                                                                  |
|-----+---------------------------------+-----------------------------------------------------------------------|
| 1   | *syscall*                       | OS との接点 — file I/O / process / signal / time / network primitive  |
| 2   | *allocator*                     | heap memory primitive — malloc / mmap 抽象化、page 管理                 |
| 3   | *GC inner*                      | mark-sweep の hot loop (Phase 3c で algorithm 確立済、それを native 化) |
| 4   | *coding (encoding)*             | UTF-8 / Latin-1 / Shift-JIS / EUC-JP 等 char ↔ byte 変換                |

これ以外 (eval / reader / VM / buffer / redisplay / event-loop / actor /
state / I/O orchestration) は全て NeLisp 上で動作 (= Phase 2-5 SHIPPED)。

*** 1.4.1 Rhodes 5-stage methodology

Rhodes "SBCL: A Sanely-Bootstrappable Common Lisp" (LNCS 5146, 2008)
の 5-stage genesis を NeLisp に transfer (=docs/03-architecture.org §1=):

| Rhodes stage | NeLisp 関与                                                                |
|--------------+----------------------------------------------------------------------------|
| host-1       | Host Emacs が =nelisp-cc.el= を byte-compile (= A1 produces A2 input)      |
| target-1     | *Phase 7.0 syscall stub + Phase 7.1 seed compiler bootstrap substrate*     |
| host-2       | A2 (seed native compiler) が NeLisp source を再 compile → A3 candidate     |
| genesis-2    | A3 candidate と A2 の semantic diff (golden hash 比較、Doc 28 §3.5 stage2) |
| target-2     | A3 を authoritative に昇格、cold-init で warm phase 走行 (Doc 28 §3.5 stage3) |

*** 1.4.2 NeLisp artifacts (A1/A2/A3) 遷移 — Doc 28 v2 §1.4 LOCKED

Phase 7+ は *parent abstraction で 3 artifact* を扱う:

| artifact | 役割                                       | 実装 phase                |
|----------+--------------------------------------------+---------------------------|
| A1       | host Emacs bytecode compiler               | Phase 5-X SHIPPED         |
| A2       | seed native compiler (=nelisp-cc.el= byte-compile 後 result) | Phase 7.1.1〜7.1.4 で構築 |
| A3       | self-compiled native compiler (A2 が =nelisp-cc.el= を native compile した result) | Phase 7.1.5 で確立 (4-stage bootstrap protocol) |

*artifact 遷移*:
1. A1 が =nelisp-cc.el= を byte-compile (host Emacs 経由)
2. A2 = A1 の output、bytecode VM 上で動く seed compiler
3. A2 が =nelisp-cc.el= source を native compile → A3 candidate
4. A3 candidate と A2 の semantic diff PASS で A3 = authoritative 昇格
5. A3 が =nelisp-cc.el= を再 native compile (self-recompile)、結果 A3 と
   semantic equivalent なら self-host bootstrap 完成

詳細 4-stage bootstrap protocol は *Doc 28 v2 §3.5 sub-phase 7.1.5* 参照。
本 parent doc では artifact level の説明のみ、stage 詳細は子 doc に分離。

*** 1.4.3 Phase 7+ の Rhodes stage における位置づけ

Phase 7+ は *Rhodes target-1 / host-2 / genesis-2 / target-2 にまたがる*:
- target-1 = Phase 7.0 syscall stub (Rust) + Phase 7.1.1〜7.1.4 native
  compiler skeleton
- host-2 = Phase 7.1.5 stage1 (A2 → A3 candidate)
- genesis-2 = Phase 7.1.5 stage2 (semantic diff)
- target-2 = Phase 7.1.5 stage3 (A3 authoritative + cold-init)

target-2 cold-init の困難は *Phase 5-B/5-C で既に確立済の elisp 実装*
を target binary 上で再現可能なため、SBCL が 25 年支払ったコストの
大部分を NeLisp は支払わない。Phase 7.5 integration で final standalone
binary 化。

** 2.2 4 核 implementation 言語 [LOCKED 2026-04-25 = E "syscall 最小 Rust + 他 3 核 NeLisp"]

最重要決定。user 2026-04-25 判断で *NeLisp purity max path* に LOCK:

| 候補 | 言語             | 利点                                                          | 欠点                                                  |
|------+------------------+---------------------------------------------------------------+-------------------------------------------------------|
| A    | C                | Emacs C / SBCL precedent と直接 align、移植性最大、依存 0     | safety / build complexity 高                          |
| B    | Rust 4 核全て    | safety, Cargo 単一 build、modern toolchain                    | LLVM 依存、binary 大、bootstrap 複雑                  |
| C    | Zig 4 核全て     | C ABI compat + safety、cross-comp first-class                 | community 小、stability 中、知名度低                  |
| D    | mixed (Rust+C)   | 核ごとに最適                                                  | 複雑度爆発、build 系 multi、debug 困難                |
| E    | *syscall stub のみ Rust + 他 3 核 NeLisp* (*LOCKED 2026-04-25*) | NeLisp purity max、SBCL precedent を超える研究価値、Rust 量極小 (~300-500 LOC)  | 実装 difficulty 飛躍 (cold-init 地獄の深刻化、native compiler 必須、要 §2.12) |

*LOCKED E 理由* (user 2026-04-25):
- "100% Lisp" project mission に最 align、SBCL が 25 年やらなかった
  precedent を超える NeLisp 独自の研究価値
- syscall は CPU と最初に話す瞬間 (Linux なら =syscall= 命令) なので
  Lisp で書きようがない部分 = ここだけ Rust で必要最小限
- allocator / GC / coding は pure logic (memory bookkeeping / object
  graph traversal / bit twisting) = NeLisp + native compiler で完結可能
- "user 体験 0 影響 = 永遠にやらない deferral pattern。project 開始
  期に lock-in する" (user logic)

*Rust 範囲限定* (Phase 7.0 で固定):
- Linux/macOS/Windows の syscall ABI thin wrapper のみ
- read/write/open/close/mmap/munmap/exit/clock_gettime/getpid/
  fork/exec/wait/signal/socket/bind/connect/accept/recv/send 等
  の OS 直叩き
- それ以外 (file path 解析 / encoding / process tree management /
  signal handler logic 等) は全て NeLisp 側

*native compiler 必須前提* (§2.12 参照): allocator / GC は hot path
なので bytecode VM では遅すぎる、NeLisp source → machine code の
native compiler が前提条件 = Phase 7.1 で新設。

** 2.4 着手順序 [LOCKED 2026-04-25 = A' "syscall stub → native compiler → allocator → GC → coding → integration"]

§2.2 LOCKED E (NeLisp purity max path) に基づき着手順序を再設計:

| 候補 | 順序                                                                                    | 理由                                          |
|------+-----------------------------------------------------------------------------------------+-----------------------------------------------|
| A    | (旧) syscall → allocator → GC → coding (4 核 Rust 想定)                                  | §2.2 LOCKED E で adopted されない             |
| A'   | *syscall stub (Rust) → native compiler (NeLisp) → allocator → GC → coding → integration* (*LOCKED*) | 依存順 + native compiler を 2 番目に新設       |
| B    | allocator から先 (memory 系優先)                                                         | bootstrap 不可 (syscall 無しで何も始まらない) |
| C    | coding から先 (I/O demo 優先)                                                            | bootstrap 不可                                |

*LOCKED A' 理由*:
1. =syscall stub Rust= = OS との 0 番目接点、これが無いと何も始まらない
2. =native compiler NeLisp= = allocator / GC / coding の hot path を
   実用速度で動かす前提、Phase 7.1 で新設 = Phase 7+ 最大の山場
3. =allocator NeLisp= = mmap (syscall) を借りて Lisp で memory 管理
4. =GC inner NeLisp= = allocator が動いてから object graph 走査を構築
5. =coding NeLisp= = bit twisting、最終、Phase 7.4
6. =integration= = 全部繋いで standalone E2E、Phase 7.5

各 sub-phase の MVP scope は §3 で具体化。Phase 7.1 (native compiler)
の追加で 6 sub-phase 構成、当初 DRAFT の 6 sub-phase と同数だが
内訳が異なる (Phase 7.0 syscall は Rust 縮小版に、Phase 7.1 native
compiler 新設、Phase 7.2-7.4 は NeLisp 化で重 work)。

の native compiler を前提とする。理由: allocator / GC / coding は hot
path で、現 NeLisp の bytecode VM 経由では実用速度が出ない (~10-100x
slower than C/Rust)。

| 候補 | 戦略                                                                | 利点                                              |
|------+---------------------------------------------------------------------+---------------------------------------------------|
| A    | *Phase 7.1 で NeLisp native compiler を新設* (*LOCKED*)             | NeLisp purity max path の必須前提、SBCL 同 pattern |
| B    | bytecode で耐える                                                   | 不可 (allocator hot path で 10-100x slow)         |
| C    | C 製 LLVM backend を call out                                       | NeLisp purity 主張破綻、=2.2 E= の趣旨に反する     |

*LOCKED A 理由*:
- SBCL も Common Lisp source → machine code の native compiler を持つ
  (= sb-c module、~30000 LOC CL)
- NeLisp 版は規模縮小で OK (Elisp subset、x86_64 + ARM64 のみ MVP、
  optimization 簡易) で 5000-10000 LOC NeLisp 想定
- Phase 7.1 = Phase 7+ 最大の山場、ただし完成すれば残 3 核 (allocator
  / GC / coding) は self-implement の道筋が立つ

*Native compiler のレイヤ案* (Phase 7.1 で詳細 design):
1. =nelisp-cc= = NeLisp source → SSA IR (NeLisp 内 data structure)
2. =nelisp-cc-x86_64= = SSA → x86_64 machine code (Linux/macOS/Windows)
3. =nelisp-cc-arm64= = SSA → ARM64 machine code (macOS Apple Silicon
   + Linux ARM)
4. =mmap= 経由 (syscall stub) で executable page に書込 + jump

*リスク (§6.7 で詳述)*: Phase 7.1 単体で 1-2 ヶ月 (anvil leverage 前提)。
未完成だと Phase 7.2 以降が完全に止まる = critical path。

* §6 想定リスク

** 6.1 cold-init の困難 (Rhodes §3.2 "most fragile")

SBCL の 25 年で最も繰り返し fix されてきた領域。NeLisp の優位:
*host Emacs debugger が cold-init 中に使える* (Elisp 実装ゆえ)。
ただし target-2 stage を Rust binary で実行する時点では Emacs が
無いため、cold-init 中の crash は Rust gdb / printf debug に頼る。
Phase 7.0 で *cold-init 用 trace 機構* を最初に整備するべし。

** 6.2 言語 lock-in (Rust 採用後の戦略変更困難)

§2.2 推奨 B (Rust) を LOCK すると Phase 7+ 着手後は変更困難。
mitigation: Phase 7.0 scaffold で C ABI =#[repr(C)]= 厳守、Rust 部分
は *別 crate* 化、最悪 C / Zig 移植が可能な layer 分離。

** 6.3 工数 6-18 ヶ月の overrun (典型 software engineering risk)

mitigation:
- sub-phase 単位 (7.0/7.1/.../7.5) で SHIPPED gate (=Doc 13 と同)
- 4 ヶ月 milestone で再 estimate (= Phase 7.0 syscall SHIPPED 時点 (本日 2026-04-25 達成済)
  での残工数見直し)
- 6 ヶ月以内 overrun 許容 (= 12 → 18 ヶ月)、それ超で scope 削減判断

** 6.4 user 単独 dev capacity

Phase 7+ は集中度高い work。memory =user_collaboration_philosophy.md=
通り user は AI の自己拡張プロセス自体を価値とする = Claude による
sub-phase 並行 dispatch が work model。session 並行性 + worktree 隔離
+ design doc LOCK の規律 (本 doc + Doc 13/14/18 pattern) で進行管理。

** 6.5 既存 Stage D Phase 6.1 user の混乱

- Phase 7.1 (native compiler) 単体で 1-2 ヶ月の sub-deadline、超過
  時は scope 削減判断

** 6.7 Phase 7.1 (native compiler) 単独失敗 = critical path 全停止

** 6.9 finalizer queue separation 失敗 (v2 新設、Doc 30 §6.9 parent reflection)

Doc 30 v2 §6.9 BLOCKER 2 mitigation で sweep 中 finalizer 直接実行を回避
し、queue 分離 + post-GC mutator context 実行に LOCK。本 parent
risk として独立記載:

*影響*: finalizer 内 allocation / mutation / resurrection が collector
不変条件を破壊、silent corruption / undefined behavior / post-mortem
debug 不可能。

*mitigation (Doc 30 v2 §6.9 LOCK 反映)*:
- finalizer queue (bounded buffer 1024 default) + post-GC mutator
  context 実行 (=nelisp-gc-drain-finalizer-queue=)
- resurrection policy LOCK = =:once= (default、1 回限り復活可)、
  =:nopolicy= (in-loop resurrection、SBCL/Java 風) は v2.0 検討
- Phase 7.3.6 sub-phase で finalizer queue executor 完成
- ERT で finalizer queue overflow / resurrection cycle / error in
  finalizer thunk を必ず cover

** 6.10 coding table provenance / regeneration drift (v2 新設、Doc 31 §6 parent reflection)

provenance / licensing / regeneration risk を明示化。本 parent risk
として独立記載:

*影響*: ~14000 entry の Shift-JIS (CP932 拡張部含む) / EUC-JP table が
source-of-truth (Microsoft 公式 CP932 mapping / Unicode UCD) と divergence
すると silent encoding bug、Windows 日本語 user の Excel / file name
