** 1.2 4 核の定義 (=docs/03-architecture.org §0= LOCKED)

SBCL が 25 年間維持してきた 4 核境界を NeLisp に transfer:

| #   | 核                              | 役割                                                                  |
|-----+---------------------------------+-----------------------------------------------------------------------|
| 1   | *syscall*                       | OS との接点 — file I/O / process / signal / time / network primitive  |
| 2   | *allocator*                     | heap memory primitive — malloc / mmap 抽象化、page 管理                 |
| 3   | *GC inner*                      | mark-sweep の hot loop (Phase 3c で algorithm 確立済、それを native 化) |
| 4   | *coding (encoding)*             | UTF-8 / Latin-1 / Shift-JIS / EUC-JP 等 char ↔ byte 変換                |

これ以外 (eval / reader / VM / buffer / redisplay / event-loop / actor /
state / I/O orchestration) は全て NeLisp 上で動作 (= Phase 2-5 SHIPPED)。

** 1.4 SBCL Rhodes 5-stage methodology (NeLisp 適応版)

Rhodes "SBCL: A Sanely-Bootstrappable Common Lisp" (LNCS 5146, 2008)
の 5-stage genesis を NeLisp に transfer (=docs/03-architecture.org §1=):

| Rhodes stage | NeLisp 類比                                                       |
|--------------+-------------------------------------------------------------------|
| host-1       | Host Emacs が NeLisp cross-compiler (=nelisp-xc=) を byte-compile   |
| target-1     | *残余 C (syscall/allocator/GC inner/coding) を build* ← 本 Phase 7+ |
| host-2       | NeLisp xc で NeLisp source 再コンパイル、独自 bytecode 生成          |
| genesis-2    | target image を Emacs buffer/vector で構築、不可 action を defer    |
| target-2     | cold-init (最難)、warm phase で system load                         |

本 Phase 7+ は *target-1 stage に相当* = "残余 C 4 核 build" の実装。
target-2 cold-init の困難は *Phase 5-B/5-C で既に確立済の elisp 実装*
を target binary 上で再現可能なため、SBCL が 25 年支払ったコストの
大部分を NeLisp は支払わない。

** 2.2 4 核 implementation 言語 [LOCKED 2026-04-25 = E "syscall 最小 Rust + 他 3 核 NeLisp"]

最重要決定。user 2026-04-25 判断で *NeLisp purity max path* に LOCK:

| 候補 | 言語             | 利点                                                          | 欠点                                                  |
|------+------------------+---------------------------------------------------------------+-------------------------------------------------------|
| A    | C                | Emacs C / SBCL precedent と直接 align、移植性最大、依存 0     | safety / build complexity 高                          |
| B    | Rust 4 核全て    | safety, Cargo 単一 build、modern toolchain                    | LLVM 依存、binary 大、bootstrap 複雑                  |
| C    | Zig 4 核全て     | C ABI compat + safety、cross-comp first-class                 | community 小、stability 中、知名度低                  |
| D    | mixed (Rust+C)   | 核ごとに最適                                                  | 複雑度爆発、build 系 multi、debug 困難                |
| E    | *syscall stub のみ Rust + 他 3 核 NeLisp* (*LOCKED 2026-04-25*) | NeLisp purity max、SBCL precedent を超える研究価値、Rust 量極小 (~300-500 LOC)  | 実装 difficulty 飛躍 (cold-init 地獄の深刻化、native compiler 必須、要 §2.12) |

*LOCKED E 理由* (user 2026-04-25):
- "100% Lisp" project mission に最 align、SBCL が 25 年やらなかった
  precedent を超える NeLisp 独自の研究価値
- syscall は CPU と最初に話す瞬間 (Linux なら =syscall= 命令) なので
  Lisp で書きようがない部分 = ここだけ Rust で必要最小限
- allocator / GC / coding は pure logic (memory bookkeeping / object
  graph traversal / bit twisting) = NeLisp + native compiler で完結可能
- "user 体験 0 影響 = 永遠にやらない deferral pattern。project 開始
  期に lock-in する" (user logic)

*Rust 範囲限定* (Phase 7.0 で固定):
- Linux/macOS/Windows の syscall ABI thin wrapper のみ
- read/write/open/close/mmap/munmap/exit/clock_gettime/getpid/
  fork/exec/wait/signal/socket/bind/connect/accept/recv/send 等
  の OS 直叩き
- それ以外 (file path 解析 / encoding / process tree management /
  signal handler logic 等) は全て NeLisp 側

*native compiler 必須前提* (§2.12 参照): allocator / GC は hot path
なので bytecode VM では遅すぎる、NeLisp source → machine code の
native compiler が前提条件 = Phase 7.1 で新設。

** 2.4 着手順序 [LOCKED 2026-04-25 = A' "syscall stub → native compiler → allocator → GC → coding → integration"]

§2.2 LOCKED E (NeLisp purity max path) に基づき着手順序を再設計:

| 候補 | 順序                                                                                    | 理由                                          |
|------+-----------------------------------------------------------------------------------------+-----------------------------------------------|
| A    | (旧) syscall → allocator → GC → coding (4 核 Rust 想定)                                  | §2.2 LOCKED E で adopted されない             |
| A'   | *syscall stub (Rust) → native compiler (NeLisp) → allocator → GC → coding → integration* (*LOCKED*) | 依存順 + native compiler を 2 番目に新設       |
| B    | allocator から先 (memory 系優先)                                                         | bootstrap 不可 (syscall 無しで何も始まらない) |
| C    | coding から先 (I/O demo 優先)                                                            | bootstrap 不可                                |

*LOCKED A' 理由*:
1. =syscall stub Rust= = OS との 0 番目接点、これが無いと何も始まらない
2. =native compiler NeLisp= = allocator / GC / coding の hot path を
   実用速度で動かす前提、Phase 7.1 で新設 = Phase 7+ 最大の山場
3. =allocator NeLisp= = mmap (syscall) を借りて Lisp で memory 管理
4. =GC inner NeLisp= = allocator が動いてから object graph 走査を構築
5. =coding NeLisp= = bit twisting、最終、Phase 7.4
6. =integration= = 全部繋いで standalone E2E、Phase 7.5

各 sub-phase の MVP scope は §3 で具体化。Phase 7.1 (native compiler)
の追加で 6 sub-phase 構成、当初 DRAFT の 6 sub-phase と同数だが
内訳が異なる (Phase 7.0 syscall は Rust 縮小版に、Phase 7.1 native
compiler 新設、Phase 7.2-7.4 は NeLisp 化で重 work)。

** 2.12 NeLisp native compiler 必須前提 [LOCKED 2026-04-25 = A "Phase 7.1 で新設"]

§2.2 LOCKED E (NeLisp purity max path) は *NeLisp source → machine code*
の native compiler を前提とする。理由: allocator / GC / coding は hot
path で、現 NeLisp の bytecode VM 経由では実用速度が出ない (~10-100x
slower than C/Rust)。

| 候補 | 戦略                                                                | 利点                                              |
|------+---------------------------------------------------------------------+---------------------------------------------------|
| A    | *Phase 7.1 で NeLisp native compiler を新設* (*LOCKED*)             | NeLisp purity max path の必須前提、SBCL 同 pattern |
| B    | bytecode で耐える                                                   | 不可 (allocator hot path で 10-100x slow)         |
| C    | C 製 LLVM backend を call out                                       | NeLisp purity 主張破綻、=2.2 E= の趣旨に反する     |

*LOCKED A 理由*:
- SBCL も Common Lisp source → machine code の native compiler を持つ
  (= sb-c module、~30000 LOC CL)
- NeLisp 版は規模縮小で OK (Elisp subset、x86_64 + ARM64 のみ MVP、
  optimization 簡易) で 5000-10000 LOC NeLisp 想定
- Phase 7.1 = Phase 7+ 最大の山場、ただし完成すれば残 3 核 (allocator
  / GC / coding) は self-implement の道筋が立つ

*Native compiler のレイヤ案* (Phase 7.1 で詳細 design):
1. =nelisp-cc= = NeLisp source → SSA IR (NeLisp 内 data structure)
2. =nelisp-cc-x86_64= = SSA → x86_64 machine code (Linux/macOS/Windows)
3. =nelisp-cc-arm64= = SSA → ARM64 machine code (macOS Apple Silicon
   + Linux ARM)
4. =mmap= 経由 (syscall stub) で executable page に書込 + jump

*リスク (§6.7 で詳述)*: Phase 7.1 単体で 1-2 ヶ月 (anvil leverage 前提)。
未完成だと Phase 7.2 以降が完全に止まる = critical path。

* §3 Sub-phase breakdown (LOCKED §2 全 12 項目 = NeLisp purity max path)

** Phase 7.0 — Rust syscall stub + scaffold (~2-3 週間 / anvil leverage 前提)

- =cargo new --lib nelisp-runtime=
- =mod syscall= = ~300-500 LOC Rust (OS direct primitives only)
  - Linux/macOS/Windows の =read / write / open / close / stat /
    mmap / munmap / mprotect / exit / clock_gettime / getpid /
    fork / exec / wait / signal handler attach / socket / bind /
    connect / accept / recv / send / poll / epoll/kqueue=
  - 上記の C ABI =extern "C"= wrapper のみ (logic 0)、NeLisp 側から
    =call-syscall= で叩く form を提供
- =build.rs= で NeLisp .el を pre-compile (host-1 stage)
- C ABI =#[repr(C)]= struct で Lisp_Object 互換
- =bin/anvil-runtime= 仮 main で initialize reply のみ
- ERT: =test/nelisp-runtime-test.el= +5 (smoke 起動 / syscall round-trip)
- 完遂 gate: =./anvil-runtime --syscall-smoke= が green、=read/write=
  経由で stdin/stdout 双方向通信できる

** Phase 7.1 — *NeLisp native compiler 新設* (~1-2 ヶ月、Phase 7+ 最大の山場)

§2.12 LOCKED A、Phase 7+ 中最も困難。NeLisp source → x86_64 / ARM64
machine code の compiler を新設、後続 Phase 7.2-7.4 の hot path 性能
を担保。

- =src/nelisp-cc.el= = NeLisp → SSA IR (~1500-3000 LOC NeLisp)
  - 既存 =nelisp-compile.el= bytecode compiler の SSA 拡張
  - constant folding / dead code elim / register allocation 簡易版
- =src/nelisp-cc-x86_64.el= = SSA → x86_64 machine code (~1500-2500 LOC)
  - System V AMD64 calling convention (Linux/macOS) + Win64 (Windows)
- =src/nelisp-cc-arm64.el= = SSA → ARM64 machine code (~1500-2500 LOC)
  - macOS Apple Silicon + Linux ARM
- =mmap (syscall)= 経由 executable page に書込 + jump
- ERT: =test/nelisp-cc-test.el= +30 (basic compile / arithmetic /
  call / loop / GC interop)
- 完遂 gate: =fib(30)= が bytecode 比 ~50x speedup (cargo bench で
  実測)、cross-arch (x86_64 + ARM64) 両方緑

** Phase 7.2 — allocator NeLisp 化 (~3-4 週間)

§2.2 LOCKED E に基づき *NeLisp + native compiler* で実装:

- =src/nelisp-allocator.el= = page-grained allocator (~800-1500 LOC NeLisp)
  - =mmap= (syscall) でページ確保、bookkeeping は Lisp で
  - free list / size class / per-thread arena (Phase 4 actor 対応)
- 既存 =alloc-cons / alloc-vector= 等 primitive を NeLisp 実装に切替
- Phase 7.1 native compiler で hot path を JIT、bytecode より ~30-50x
- regression: NeLisp ERT 1327/1333 緑 (allocator 切替後も不変)
- 完遂 gate: =allocator-stress= bench で C-equivalent throughput
  (~10M alloc/sec) 達成

** Phase 7.3 — GC inner NeLisp 化 (~1-2 ヶ月、最難)

mark-sweep hot loop (Phase 3c で algorithm 確立済) を NeLisp に port:

- =src/nelisp-gc-inner.el= = mark-sweep + write barrier (~1200-2000 LOC NeLisp)
- =mark phase= = object graph DFS、Phase 7.1 native compile で C 並み速度
- =sweep phase= = free list 統合、Phase 7.2 allocator と協調
- =root scanning= = stack / global / actor mailbox 全 walk
- 並行性: Phase 4 actor との yield 整合 (=yield-during-mark=)
- ERT: =test/nelisp-gc-stress-test.el= +20 (heavy allocation /
  cycle / weak ref / actor concurrent)
- 完遂 gate: =gc-stress 1M cycle= で C 比 ~80% throughput (許容)、
  pause time ~5ms 以下

** Phase 7.4 — coding NeLisp 化 (~2-3 週間)

UTF-8 / Latin-1 / Shift-JIS / EUC-JP encode/decode を NeLisp で実装:

- =src/nelisp-coding.el= = encode/decode loop (~600-1000 LOC NeLisp)
- bit twisting なので Phase 7.1 native compile で十分速い
- char ↔ byte 変換 = NeLisp string primitive 互換
- regression: 日本語含む ERT (=anvil-org-index= 等) 緑
- 完遂 gate: =coding-roundtrip 100MB= で C 比 ~70% throughput (許容)

** Phase 7.5 — integration + standalone E2E (~2 週間)

- =./anvil-runtime mcp serve= で Emacs 不在で initialize / tools/list
- =./anvil-runtime --self-test= で NeLisp ERT 1327/1333 緑
- Stage D tarball = =stage-d-v2.0= layout で binary 同梱
  (Phase 6.1 anvil-lib bundle と co-exist、§2.8 LOCKED A)
- README で OS install instructions
- GitHub Release v1.0.0 + 公開告知 (Reddit / HN / blog)

* §6 想定リスク

** 6.1 cold-init の困難 (Rhodes §3.2 "most fragile")

SBCL の 25 年で最も繰り返し fix されてきた領域。NeLisp の優位:
*host Emacs debugger が cold-init 中に使える* (Elisp 実装ゆえ)。
ただし target-2 stage を Rust binary で実行する時点では Emacs が
無いため、cold-init 中の crash は Rust gdb / printf debug に頼る。
Phase 7.0 で *cold-init 用 trace 機構* を最初に整備するべし。

** 6.2 言語 lock-in (Rust 採用後の戦略変更困難)

§2.2 推奨 B (Rust) を LOCK すると Phase 7+ 着手後は変更困難。
mitigation: Phase 7.0 scaffold で C ABI =#[repr(C)]= 厳守、Rust 部分
は *別 crate* 化、最悪 C / Zig 移植が可能な layer 分離。

** 6.3 工数 6-18 ヶ月の overrun (典型 software engineering risk)

mitigation:
- sub-phase 単位 (7.0/7.1/.../7.5) で SHIPPED gate (=Doc 13 と同)
- 4 ヶ月 milestone で再 estimate (= Phase 7.1 syscall SHIPPED 時点
  での残工数見直し)
- 6 ヶ月以内 overrun 許容 (= 12 → 18 ヶ月)、それ超で scope 削減判断

** 6.4 user 単独 dev capacity

Phase 7+ は集中度高い work。memory =user_collaboration_philosophy.md=
通り user は AI の自己拡張プロセス自体を価値とする = Claude による
sub-phase 並行 dispatch が work model。session 並行性 + worktree 隔離
+ design doc LOCK の規律 (本 doc + Doc 13/14/18 pattern) で進行管理。

** 6.5 既存 Stage D Phase 6.1 user の混乱

stage-d-v0.x ↔ stage-d-v2.0 シリーズ並行 (§2.8 推奨 A) で:
- v0.x = Emacs 必要 / smaller tarball (~624K) / dev / power user 向け
- v2.0 = Emacs 不要 / larger binary (~5-15MB) / non-Emacs user 向け

README で位置づけ明確化必須。

** 6.6 SBCL precedent 不適用 (= NeLisp 独自 path)

§2.2 LOCKED E (NeLisp purity max path) は SBCL が 25 年維持してきた
"4 核 C 固定" precedent を超える choice。SBCL community が "やらなかった"
理由 (= user 体験影響なし vs 工数膨大の trade-off で defer) を NeLisp
は逆判断する。

mitigation:
- Phase 7.0/.1 で fail-fast: 4 週間以内に native compiler MVP が
  syscall 経由 hello-world machine code execution できる確認
- できなければ Rust 4 核 (Doc 27 当初 DRAFT) に rollback option を
  残す (§2.2 候補 B = LOCKED E の安全網)
- Phase 7.1 (native compiler) 単体で 1-2 ヶ月の sub-deadline、超過
  時は scope 削減判断

** 6.7 Phase 7.1 (native compiler) 単独失敗 = critical path 全停止

§2.12 LOCKED A により Phase 7.1 が "最も困難 + 後続 Phase 全依存" の
critical path。Phase 7.1 が完成しないと Phase 7.2-7.4 が完全停止。

mitigation:
- Phase 7.1 内部を更に sub-sub-phase 分割 (7.1.1 SSA IR / 7.1.2
  x86_64 backend / 7.1.3 ARM64 backend / 7.1.4 mmap exec page +
  GC interop) で SHIPPED 刻み
- 各 sub-sub-phase で 1-2 週間 milestone、超過時 scope 削減
  (例: ARM64 Phase 7.5 へ後送り、x86_64 のみで MVP)
- Phase 7.1.1 (SSA IR) 完成時点で Phase 7.2 (allocator) スケッチ
  着手、parallel に walk

** 6.8 anvil leverage 想定の overrun

期間 4-8 ヶ月見積は anvil.el + 並列 session full leverage 前提。実際に
1 dev (anvil なし) で進めると 14-24 ヶ月になる (Doc 27 当初 DRAFT 値)。

mitigation:
- Phase 7.0 完遂時点で leverage 効率測定 (commit/週、並列 session 数)
- 想定下回ったら scope 削減 / Phase 7.5 deadline 後送り判断
- =project_nelisp.md= description に実工数を log し続ける
