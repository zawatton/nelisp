** 1.2 4 核の定義 (=docs/03-architecture.org §0= LOCKED)

SBCL が 25 年間維持してきた 4 核境界を NeLisp に transfer:

| #   | 核                              | 役割                                                                  |
|-----+---------------------------------+-----------------------------------------------------------------------|
| 1   | *syscall*                       | OS との接点 — file I/O / process / signal / time / network primitive  |
| 2   | *allocator*                     | heap memory primitive — malloc / mmap 抽象化、page 管理                 |
| 3   | *GC inner*                      | mark-sweep の hot loop (Phase 3c で algorithm 確立済、それを native 化) |
| 4   | *coding (encoding)*             | UTF-8 / Latin-1 / Shift-JIS / EUC-JP 等 char ↔ byte 変換                |

これ以外 (eval / reader / VM / buffer / redisplay / event-loop / actor /
state / I/O orchestration) は全て NeLisp 上で動作 (= Phase 2-5 SHIPPED)。

** 1.3 Stage D 関係 (Doc 18) と本 Phase の差

| Stage    | 内容                                            | 状態                  |
|----------+-------------------------------------------------+-----------------------|
| Stage A  | NeLisp substrate Elisp 化                        | Phase 5-F SHIPPED     |
| Stage D  | Emacs install 前提だが distributable             | Phase 6.1 SHIPPED     |
| Stage E  | (本 Doc 27 = Stage D v2.0 / Option B 相当) 真 standalone | Phase 7+ で対応 *本 doc* |

Stage D は "Emacs ありき" の配布形態 (Phase 6.1 で =bin/anvil mcp serve=
が headless profile で動作するが内部は =emacs --batch= subprocess)。
本 Phase 7+ で =emacs --batch= ですら不要な真 standalone を実現する。

** 1.4 SBCL Rhodes 5-stage methodology (NeLisp 適応版)

Rhodes "SBCL: A Sanely-Bootstrappable Common Lisp" (LNCS 5146, 2008)
の 5-stage genesis を NeLisp に transfer (=docs/03-architecture.org §1=):

| Rhodes stage | NeLisp 類比                                                       |
|--------------+-------------------------------------------------------------------|
| host-1       | Host Emacs が NeLisp cross-compiler (=nelisp-xc=) を byte-compile   |
| target-1     | *残余 C (syscall/allocator/GC inner/coding) を build* ← 本 Phase 7+ |
| host-2       | NeLisp xc で NeLisp source 再コンパイル、独自 bytecode 生成          |
| genesis-2    | target image を Emacs buffer/vector で構築、不可 action を defer    |
| target-2     | cold-init (最難)、warm phase で system load                         |

本 Phase 7+ は *target-1 stage に相当* = "残余 C 4 核 build" の実装。
target-2 cold-init の困難は *Phase 5-B/5-C で既に確立済の elisp 実装*
を target binary 上で再現可能なため、SBCL が 25 年支払ったコストの
大部分を NeLisp は支払わない。

** 1.5 NeLisp 独自 adaptation gap (Phase 7+ で扱うべき 3 点)

=docs/03-architecture.org §1 wave 2 audit= で判明:

1. =eval-when= 非存在 → phase flag (=nl-phase=) / file 分離で代替
2. =make-load-form= 非存在 → =print/read= 組合せ or 独自 encoder 設計
3. Buffer-centric primitive の扱い → cross-compiler image での

これら 3 点は本 Phase 7+ の §3 sub-phase で具体化する。

** 2.1 Phase 番号付け [推奨 = B]

NeLisp の現 Phase は 5-F (Stage A) + 6.x (Stage D) で進行中。本 phase
を何と呼ぶか:

| 候補 | 番号付け                       | 利点                                       |
|------+--------------------------------+--------------------------------------------|
| A    | Phase 7 単独                   | シンプル                                   |
| B    | Phase 7.0/7.1/.../7.4 (sub分割) | 4 核個別管理、SHIPPED 刻みやすい *推奨*    |
| C    | Phase 5-G + 5-H + ... (5 系列拡張) | 連続性                                     |

*推奨 B 理由*: 4 核は logical に独立 (syscall / allocator / GC / coding)。
sub-phase 分割で各核の SHIPPED を独立に gate 可能、Phase 4.1〜4.7 と
同 pattern。

** 2.2 4 核 implementation 言語 [推奨 = B Rust]

最重要決定。残余 C 4 核を何で書くか:

| 候補 | 言語     | 利点                                                          | 欠点                                                  |
|------+----------+---------------------------------------------------------------+-------------------------------------------------------|
| A    | C        | Emacs C / SBCL precedent と直接 align、移植性最大、依存 0     | safety / build complexity 高                          |
| B    | *Rust*   | safety, Cargo 単一 build、modern toolchain *推奨*             | LLVM 依存、binary 大、bootstrap 複雑                  |
| C    | Zig      | C ABI compat + safety、cross-comp first-class                 | community 小、stability 中、知名度低                  |
| D    | mixed    | 核ごとに最適 (e.g. allocator = C、GC = Rust)                  | 複雑度爆発、build 系 multi、debug 困難                |

*推奨 B (Rust) 理由*:
- safety = NeLisp が *研究 / dev tool* なので memory bug 即 trace 可能性
  が valuable (gdb で C の core dump 解析より rust panic 解析が短時間)
- Cargo single command = build / cross-comp / test / fmt が統一
- C ABI 互換 (=#[repr(C)]=) で Emacs ABI 継承 LOCK 維持
- bootstrap 複雑度: Rust 自身も Rust-on-OCaml/C bootstrap 経由なので
  「依存少なく」とは言えないが、Rhodes の "host CL は何でも" 思想を
  そのまま transfer すれば OK (NeLisp host = Emacs であり Rust 自体の
  bootstrap 問題は NeLisp の問題ではない)

代替: Zig (候補 C) は Rust より新しく軽量、stage-d-v2.0 リリース時に
"Zig + NeLisp" の対が話題性高いが、project complexity を考えると Rust
の community 厚さが優位。

** 2.3 Bootstrap 戦略 [推奨 = A Rhodes 5-stage 完全踏襲]

| 候補 | 戦略                                  | 利点                                  |
|------+---------------------------------------+---------------------------------------|
| A    | Rhodes 5-stage 完全踏襲 *推奨*         | 学術的に確立、SBCL 25 年実績          |
| B    | Direct compile (cross-comp なし)      | シンプル、しかし self-host fixpoint 不可 |
| C    | Hybrid (host-1/target-1 のみ Rhodes、host-2 以降簡略) | 折衷案                                |

*推奨 A 理由*:
- Phase 2 self-host 達成済 = Rhodes stage host-1/target-1 の概念は
  既に NeLisp 内で機能している
- target-1 stage = 本 Phase 7+ で書く 4 核 = 後続 stage が自動的に
  動作する (Phase 5-B/5-C の Elisp 実装が target-2 上で動く)
- SBCL の 25 年間の bootstrap pitfall (cold-init 順序 / =eval-when=
  phase 制御 etc) を全て継承可能 (=docs/papers/rhodes-2008-sbcl.org=
  詳細 reading 済)

** 2.4 4 核 module 着手順序 [推奨 = A syscall → allocator → GC → coding]

| 候補 | 順序                                              | 理由                                                |
|------+---------------------------------------------------+-----------------------------------------------------|
| A    | *syscall → allocator → GC inner → coding* *推奨*  | 依存順 (syscall は最基礎、coding は最応用)          |
| B    | allocator → GC → syscall → coding                 | memory 系から先                                     |
| C    | coding → syscall → allocator → GC                 | I/O 系から先 (実用 demo 早く出る)                   |

*推奨 A 理由*:
- syscall = OS との接点で他 3 核がほぼ全て依存 (mmap で allocator、
  brk で GC、read/write で coding)
- coding は応用層 = syscall/allocator が動かないと test できない

各 sub-phase の MVP scope は §3 で具体化。

** 2.5 build 系 [推奨 = A Cargo 単一]

| 候補 | 系統                            | 利点                                                |
|------+---------------------------------+-----------------------------------------------------|
| A    | *Cargo 単一* *推奨*              | Rust crate 1 つ、=cargo build= で完結               |
| B    | Cargo + Make 混在                | NeLisp 既存 Makefile と統合                         |
| C    | Bazel / Buck                    | scaling、ただし dev 1 名では oversize               |

*推奨 A 理由*: Rust 採用 (§2.2) 前提で Cargo 一択が rational。NeLisp
既存 Makefile (=make test= / =make stage-d-tarball=) は別系として残し、
本 phase 7+ は =cargo= で完結、tarball に binary をコピーする形で
Stage D と統合。

** 2.6 distribution surface [推奨 = B per-OS multi binary]

| 候補 | 形態                                  | 利点                                       |
|------+---------------------------------------+--------------------------------------------|
| A    | Linux 単独 binary                     | scope 最小、release 早い                   |
| B    | *Linux + macOS + Windows*             | non-Emacs user 全 OS 対応 *推奨*           |
| C    | Linux/macOS は Phase 7、Windows は Phase 8 | 段階的、Windows complexity 後送り          |

** 2.7 test 戦略 [推奨 = A multi-layer]

* §3 Sub-phase breakdown (推奨 §2 全 default 採用前提)

** Phase 7.0 — scaffold + cross-compiler skeleton (1-2 ヶ月)

- =Cargo.toml= 新規 / =cargo new --lib nelisp-runtime=
- 4 核 mod: =mod syscall; mod allocator; mod gc; mod coding;=

- ERT: =test/nelisp-runtime-test.el= +5 (smoke 起動 / abi compat)

** Phase 7.1 — syscall core (2-3 ヶ月)

- file I/O primitives: =open / read / write / close / stat / mkdir /
  rename / unlink / readlink=
- process primitives: =fork / exec / wait / signal / kill=
- time: =gettimeofday / clock_gettime=
- network: =socket / bind / connect / accept / recv / send=
- =mod syscall= = ~1000-2000 LOC Rust + 数十 unit test
- NeLisp 側 nelisp-process / nelisp-network が syscall 経由で動く

** Phase 7.2 — allocator core (1-2 ヶ月)

- =mmap= ベース page-grained allocator (4KB / 8KB page)
- 既存 Phase 3c GC algorithm の memory layout 維持
- per-thread allocator (Phase 4 actor / future thread 対応)
- =mod allocator= = ~500-1000 LOC + bench
- regression: NeLisp ERT 1327/1333 緑 (allocator 切替後)

** Phase 7.3 — GC inner core (2-3 ヶ月)

- mark-sweep hot loop (Phase 3c で algorithm 確立済) の Rust port
- write barrier (generational GC への足場)
- root scanning (stack / global / actor mailbox)
- =mod gc= = ~800-1500 LOC + GC stress test
- 並行性: actor (Phase 4) との yield 整合 (=yield-during-mark=)

** Phase 7.4 — coding core (1-2 ヶ月)

- UTF-8 / Latin-1 / Shift-JIS / EUC-JP encode/decode
- char ↔ byte 変換 = NeLisp string primitive 互換
- multibyte buffer ↔ external bytes
- =mod coding= = ~500-800 LOC + unit test
- regression: 日本語含む ERT (=anvil-org-index= related etc) 緑

** Phase 7.5 — integration + standalone E2E (1 ヶ月)

- =./anvil-runtime mcp serve= で Emacs 不在で initialize / tools/list
- =./anvil-runtime --self-test= で NeLisp ERT 1327/1333 緑
- Stage D tarball = =stage-d-v2.0= layout で binary 同梱
- README で OS install instructions
- GitHub Release v1.0.0 + 公開告知 (Reddit / HN / blog)

** 4.3 anvil.el との関係

architecture α delegate chain は Phase 7+ でも保持:
- anvil-XXX (Emacs Lisp side、anvil.el repo)
- nelisp-XXX (NeLisp Elisp side、nelisp main repo)

anvil-XXX → nelisp-XXX → (Phase 7+ binary 上で動作) の順序。anvil-* tools
は変わらず、container だけ変わる (Emacs → standalone binary)。

** 4.4 並行 dev session

- session-A: Phase 7.0 + 7.5 (orchestrator)

- session-C (if available): Phase 7.2 (allocator) + 7.3 (GC)

branch。Cargo workspace で crate 分割すれば conflict 0。

* §6 想定リスク

** 6.1 cold-init の困難 (Rhodes §3.2 "most fragile")

SBCL の 25 年で最も繰り返し fix されてきた領域。NeLisp の優位:
*host Emacs debugger が cold-init 中に使える* (Elisp 実装ゆえ)。
ただし target-2 stage を Rust binary で実行する時点では Emacs が
