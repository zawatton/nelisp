* §0 TL;DR

This doc extends the Sexp ABI to cover the *read-only* surface of
=Sexp::Cons= so Phase 47-compiled elisp =.o= objects can walk lists in
place.  Two new value-producing grammar forms:

- =(sexp-cons-car PTR)= → returns the =*const Sexp= of the cons cell's
  =car= field, borrowed for the lifetime of the input Cons.
- =(sexp-cons-cdr PTR)= → returns the =*const Sexp= of the cons cell's
  =cdr= field, borrowed similarly.

Plus a frozen-doc extension to =docs/arch/sexp-abi.md= covering the
=NlConsBox= layout (= car @ 0, cdr @ 32, refcount @ 64, total 72 bytes
under =#[repr(C)]=) and matching Rust =const_assert!= entries in
=build-tool/src/eval/sexp_abi_assert.rs=.

A =sexp-is-cons= form is *not* added because =(= (sexp-tag PTR) 7)= is
already expressible in §100.B grammar — adding a dedicated form would
duplicate dispatch behavior without saving any Rust LOC.

Fresh =Sexp::Cons= construction is *out of scope* for this doc.  The
elisp side has no allocator and the Rust =NlConsBox::new= path is
heap-allocating + refcount-initializing.  §5 enumerates the three
candidate strategies for a future fresh-Cons form.

* §1 Background

** §1.1 NlConsBox byte layout (= frozen contract extension)

=build-tool/src/eval/nlconsbox.rs:57=:

#+begin_src rust
  #[repr(C)]
  pub struct NlConsBox {
      pub car: Sexp,           // offset 0,  32 bytes
      pub cdr: Sexp,           // offset 32, 32 bytes
      pub refcount: AtomicUsize, // offset 64, 8 bytes
  }                              // total = 72 bytes
#+end_src

=NlConsBoxRef= is =#[repr(transparent)]= around =NonNull<NlConsBox>=
(= 8 bytes), so reading =[Sexp + 8]= for a =Sexp::Cons= value yields
the box pointer directly.  See =nlconsbox.rs:117=.

The =car= / =cdr= byte offsets are *new frozen contract values* —
Doc 100 v2 §100.B already covers the outer =Sexp= layout but did not
need =NlConsBox= internals.  Doc 101 adds 5 new entries to the ABI
export table:

| Name             | Value | Source                                            |
|------------------|------:|---------------------------------------------------|
| =cons-box-size=  | 72    | =size_of::<NlConsBox>()=                          |
| =cons-box-car=   | 0     | =offset_of!(NlConsBox, car)= (=hardcoded)         |
| =cons-box-cdr=   | 32    | =offset_of!(NlConsBox, cdr)=                      |
| =cons-box-refct= | 64    | =offset_of!(NlConsBox, refcount)=                 |
| =cons-box-align= | 8     | =align_of::<NlConsBox>()=                         |

These ride the same =make sexp-abi-check= diff that §100.B established.

** §1.2 Why borrow-returning is the right MVP

Returning a borrowed =*const Sexp= (= pointer into the live Cons box)
keeps elisp inside the §100.B caller-owned-slot ABI: no allocator is
needed on the elisp side, no refcount has to be touched, and the
existing =sexp-int-unwrap= / =sexp-tag= / =sexp-int-make= forms can
operate against the borrowed pointer transparently.

The borrow lifetime is "until the input Cons is dropped" which in
practice means "until the Rust shim's =Ok(...)= returns" — every Doc
101+ swap goes through a Rust shim that holds the input =&[Sexp]= for
the duration of the extern call.  No analogue to GC-style relocation
exists in the current evaluator, so the box pointer is stable.

* §2 ABI design (= Sexp::Cons read-only MVP)

** §2.1 The chosen scheme: borrowed pointer return

For =(sexp-cons-car PTR)=:

#+begin_src asm
  ;; PTR is in rax (computed by sub-expression)
  mov rdi, rax                   ; rdi = Sexp* (the Cons)
  mov rax, [rdi + 8]             ; rax = NlConsBox* (= car address since car @ 0)
                                 ; total: 7 bytes
#+end_src

For =(sexp-cons-cdr PTR)=:

#+begin_src asm
  mov rdi, rax                   ; rdi = Sexp* (the Cons)
  mov rax, [rdi + 8]             ; rax = NlConsBox*
  add rax, 32                    ; rax = NlConsBox + 32 = cdr address
                                 ; total: 11 bytes (= mov+mov+add)
#+end_src

The =add rax, 32= can be expressed as a 4-byte =48 83 C0 20= (= REX.W
+ ADD r/m64, imm8) — Phase 47 already emits =add-imm32= via
=--emit-imm32-binop= but that is 7 bytes; a future grammar refinement
might switch to imm8 for byte savings.  The MVP just uses
=add-imm32=.

** §2.3 The bi_nl_cons_car swap (= §101.C, see §3)

Current Rust =cons_primitives.rs:63=:

#+begin_src rust
  pub fn bi_nl_cons_car(args: &[Sexp]) -> Result<Sexp, EvalError> {
      require_arity("nl-cons-car", args, 1, Some(1))?;
      let r = require_cons("nl-cons-car", &args[0])?;
      Ok(r.car.clone())
  }
#+end_src

Post-§101.C:

#+begin_src rust
  pub fn bi_nl_cons_car(args: &[Sexp]) -> Result<Sexp, EvalError> {
      require_arity("nl-cons-car", args, 1, Some(1))?;
      require_cons("nl-cons-car", &args[0])?; // type check stays in Rust
      #[cfg(all(target_os = "linux", target_arch = "x86_64"))]
      {
          let car_ptr = unsafe {
              crate::elisp_cc_spike::cons_car_ptr(&args[0] as *const Sexp)
          };
          Ok(unsafe { (*car_ptr).clone() })
      }
      #[cfg(not(all(target_os = "linux", target_arch = "x86_64")))]
      {
          let r = require_cons("nl-cons-car", &args[0])?;
          Ok(r.car.clone())
      }
  }
#+end_src

The elisp =.o= computes the =car= address (= a 2-instruction
extraction).  Rust holds onto the clone + refcount-bump
responsibility because:

1. =Sexp::Clone= for boxed variants (Cons / MutStr / Vector / Cell /
   etc.) requires an atomic refcount increment, which is unsafe to
   express in elisp without crossing back to Rust.
2. Cloning into a 32-byte =Sexp= return value via memcpy from elisp
   would skip the refcount bump and break the ownership graph.

Net Rust algorithm LOC delta: −1 (= the =r.car.clone()= becomes
=(*car_ptr).clone()=, but the =r= variable goes away).  The Rust
shim is now structurally identical to the §100.C =bi_truncate=
pattern.  This is a *small* swap; its main value is *proving that
borrowed-pointer return works end-to-end* and laying the cleanup
shape for the bigger Cons-walking swaps in Doc 102+.

** §2.4 Allocation question (= deferred)

Fresh =Sexp::Cons= construction needs:

- A 72-byte heap allocation aligned to 8.
- Refcount init to 1.
- Two =Sexp= field initializations (= 32 bytes each, with their own
  refcount bumps if the field values are boxed variants).

None of these can be expressed in the current Phase 47 grammar
because:

- elisp has no heap allocator primitive.
- Refcount bumps are atomic ops + must be paired with the Rust
  ownership tree.

Three candidate strategies for the future fresh-Cons form (= §101.E
or a separate Doc 102 sub-stage):

*A. Rust allocator helper* (= 5..10 LOC =extern "C"= shim that just
calls =NlConsBoxRef::new=).  Smallest grammar surface but adds a Rust
codepath the elisp can't audit at compile time.

*B. Pre-allocated cons pool* (= Rust pre-fills a freelist of empty
boxes, elisp pops one and writes car/cdr into it).  No new Rust
algorithm for the allocator (= reuse =Vec<NonNull<NlConsBox>>=) but
requires a refcount initialization primitive elisp can call.

*C. Grammar form =(sexp-cons-new CAR CDR)=* (= compiles to a
=call __nl_cons_alloc= extern + writes via the standard layout
template).  Reuses the §100.A =extern-call= grammar; requires one
12-LOC Rust ABI helper.

The right choice depends on how often new-Cons fires in the swap
candidate set.  =bi_nl_cons_alloc= itself is one obvious target
(= maybe overhead-equivalent to staying in Rust); the bigger payoff
is list-construction helpers like =bi_reverse= where the elisp body
allocates many cells.

Per =feedback_nelisp_pure_elisp_means_rust_loc_decreases=, scheme *C*
is preferred (= reuses extern-call grammar from §100.A, adds only
the minimum Rust ABI helper).  Scheme *A* is the "Option C" anti-
pattern in disguise.  Final selection lives in Doc 102.

* §5 Risks / unknowns

** §5.1 Borrowed-pointer lifetime

The =car_ptr= returned by elisp is valid only while the input Cons
is alive — i.e. while the Rust shim's =args[0]= is in scope.  This
holds for the §101.C swap because =bi_nl_cons_car= holds =&[Sexp]=
for the entire body.  Future swaps that try to *cache* a borrowed
pointer across primitive calls would break (= no Rust lifetime
checker on the elisp side).  Doc 101 explicitly prohibits this; any
caller that needs to retain a car/cdr value must clone immediately.

** §5.2 Refcount drift via raw pointer read

Reading =[Sexp + 8]= as a =*const NlConsBox= and dereferencing its
fields does *not* touch the refcount — that's exactly why elisp can
do it safely.  But it means the elisp body has no protection against
the input Cons being concurrently dropped.  Single-threaded NeLisp
makes this currently sound; multi-threaded future will require either
a borrow scope marker or a refcount bump-then-deref.  This is the
same race as the existing =Sexp::cons_box_ptr= unsafe API.

** §5.3 Cdr offset = 32 depends on size_of::<Sexp>()

The =cdr= offset of 32 is computed as =offset_of!(NlConsBox, car) +
size_of::<Sexp>()= = 0 + 32.  If =size_of::<Sexp>()= ever changes
(= new variant with a payload exceeding 24 bytes), the cdr offset
shifts.  The §101.A =const_assert!= catches this at compile time;
the =make sexp-abi-check= catches it at CI time.  Both gates trip
*before* any =.o= would be silently miscompiled.

** §5.4 Allocator deferral creates dependency on Doc 102

Doc 101 by itself does not unlock the largest Cons swap targets
(= =bi_reverse=, =bi_assq=, =bi_member=, etc.) — those all construct
fresh cells.  Doc 102 (or a §101.E follow-up) must land a
=(sexp-cons-new CAR CDR)= form (per §2.4 scheme C) before the
heavy-list-construction =bi_*= can swap.  Doc 101 alone delivers
~2 LOC reduction, which is small but proves the read-only ABI.
