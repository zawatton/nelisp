* §0 TL;DR

Doc 77c §2.3 (= "Phase C: GC を elisp に migrate, 1-2 ヶ月") の詳細設計.

- *現状*: Rust 内に refcount-based GC が ~1,936 LOC (= 8 ファイル合計 non-test).  Drop 本体だけで 174 LOC, =NlRc<T>= core 240 LOC, 各 =NlXxxRef= の =Clone= / =Drop= / refcount API が残り.
- *target*: GC の *policy* (= cycle 検出、bulk free、finalizer queue) を elisp に持ち上げ、Rust 側は *mechanism* (= layout-pinned heap + atomic refcount + alloc/dealloc primitive) のみ ~300 LOC に縮める.
- *推奨 approach*: =Option C (hybrid)= = refcount mechanism は Rust 残置 (= layout pin 維持 / hot path 維持)、cycle collector + bulk finalizer scheduler を elisp に新設.
- *sub-stage*: C.0 audit → C.1 elisp boundary primitive 拡張 (+5 prim) → C.2 cycle collector elisp 実装 (+~400 LOC elisp) → C.3 finalizer queue elisp 化 (+~150 LOC elisp) → C.4 Drop 本体の thin 化 (-~100 LOC Rust) → C.5 self-host gate.
- *Rust 削減 target*: 8 ファイルの =Drop= body から refcount→0 の payload drop / dealloc を残し、cycle 関連の余地 (= 現状 Drop 本体は cycle 無視) は変えない.  =cons_primitives.rs= の =rc_inc_raw= / =rc_dec_raw= API surface 削減なし (= 既に Layer 2 から駆動可能).  純削減は ~-100〜-300 LOC Rust、純追加 ~+550-700 LOC elisp.
- *bootstrap circularity*: GC が eval を必要としない設計 (= alloc / dealloc / inc / dec はすべて pure C-like loop) で回避.  cycle collector は eval-time にのみ走るので boot 中 (= image decode 期) は呼ばない.

* §1 動機

** §1.1 100% pure elisp self-host commitment

memory =project_nelisp_100_percent_elisp_commitment= (user 2026-05-09 directive) と Doc 77c §2.3 の framing に従い、GC を含む runtime 全層を elisp で記述可能にする.  Phase C の完了で「Rust 側は heap allocator + atomic 操作 + layout pin の primitive のみ」という最終姿勢に近づく.

** §1.2 elisp 化のメリット

- *Cycle collection が hackable*: Bacon-Rajan / Recycler / Sapphire 等のアルゴリズム選択が elisp 修正のみで可能.  Rust recompile + ABI 同期不要.
- *Finalizer hook の自由度*: weak ref / phantom ref / 大規模 buffer の段階的解放など elisp 側で defcustom できる.
- *self-host depth の articulation*: =project_nelisp_100_percent_elisp_commitment= で謳う「runtime ABI も elisp」の証跡として GC は最後の砦.
- *future 移植 cost*: Rust→Zig / Rust→C 等の seed core 言語変更時、GC policy の elisp 部分は無関係に流用可能 (= Doc 47 phase 6 の C-runtime 自前化に効く).

** §1.3 elisp 化しない選択肢の reason

実は GC を elisp 化しない選択肢もあり得る (= Lua / Squirrel / Tcl 流: GC は VM の責務、language は触らない).  しかし NeLisp は

- 「Elisp が runtime 全責務を持つ」という Doc 73 / 77c の設計の柱
- =defadvice= 文化に基づく、ユーザーが GC policy を override できる Emacs Lisp 的な hackability
- Doc 47 phase 6 の C-runtime self-impl roadmap

から、elisp 化を採る.

** §2.2 Refcount プロトコル (NlRc<T> 由来、全 Ref で再現)

各 =NlXxxRef= は:

1. =new(payload)= → =std::alloc::alloc(Layout::new::<NlXxx>())= で heap 確保 → refcount=1 で初期化、=NonNull<NlXxx>= 保持.
2. =Clone= → =refcount.fetch_add(1, Relaxed)= bump、新 handle 返却.
3. =Drop= → =refcount.fetch_sub(1, Release)=、prev==1 なら =fence(Acquire)= → =drop_in_place(payload)= → =alloc::dealloc=.
4. =Deref= → =&NlXxx= 借用、固定 offset access.

=AtomicUsize= refcount は offset 0 (= NlRc) または末尾 (= NlConsBox) に locked、Phase A.5 JIT trampoline + Phase A.3 elisp =nl-ffi-write-i64= が直接読書きする contract.

** §2.4 Cycle handling 現状

- =NlConsBox= / =NlVector= / =NlCell= 等は acyclic invariant (= Phase A の cons_primitives.rs §記述通り).  ユーザーが =setcdr= で cycle を作れるが、Rust =Drop= は cycle 検出なし → =refcount > 0= のまま leak.
- =NlCharTable= の =parent= field は self-reference 不可 (= API gating、コメント nlchartable.rs §1.5).
- 結果: *cycle leak は現状 Rust に修復ロジックがない*.  既存 elisp test (= =equal_two_isomorphic_cycles_compare_equal= 等) は cycle を作って exit するが GC せず終了に頼っている.

** §2.5 Allocator boundary

全 Box struct は =std::alloc::alloc= = libc =malloc= 直 (= Rust default GlobalAlloc).  arena / pool / bump allocator なし.  third-party GC crate (= bdwgc, rsgc, gc-arena, shifgrethor, broom) は Cargo.toml 検索結果 *zero hit*.

ffi.rs の =Box::leak(Box::new(lib))= は plugin 動的 lib の lifetime 管理で GC 対象外.

** §2.6 既存 elisp boundary primitive

=cons_primitives.rs= (305 LOC, ~140 non-test) で Phase A.3 出荷済:

- =nl-cons-alloc=, =nl-cons-car=, =nl-cons-cdr=, =nl-cons-set-car=, =nl-cons-set-cdr=
- =nl-rc-inc=, =nl-rc-dec=, =nl-rc-count=

Cons 専用.  Phase C で =nl-vector-*= / =nl-str-*= / =nl-cell-*= / =nl-record-*= / =nl-bool-vector-*= / =nl-chartable-*= の similar surface が必要 (= elisp 側が allocate/free/inc/dec を全 boxed kind で driving).

* §3 Migration target — 3 option 比較

** §3.1 Option A: keep refcount in Rust + expose alloc/free as primitives

- *content*: 現状の =Drop= 本体を残し、=alloc/dealloc/inc/dec/count= を elisp primitive 化、cycle collector のみ elisp 新設.
- *Rust 削減*: ほぼ 0 (= cycle 関連 helper だけ).  cons_primitives.rs 系の追加で actually +200 LOC.
- *elisp 追加*: cycle collector ~300 LOC + finalizer scheduler ~150 LOC = ~450 LOC.
- *risk*: 低.  cycle leak 対応のみが elisp 仕事.
- *self-host depth*: 低 (= Rust の Drop 本体を温存).

** §3.2 Option B: replace refcount entirely with mark-sweep tracing GC

- *content*: =NlRc= / =NlXxxRef= refcount logic 全削除、Rust は heap region + roots set を expose、elisp 側 mark/sweep walker が evict.
- *Rust 削減*: ~-1,500 LOC (= refcount Clone/Drop 全廃 + atomic 操作削除).
- *elisp 追加*: ~+1,200-1,500 LOC (= mark/sweep + tri-color + write barrier shim).
- *risk*: 高.  Sexp ABI 大改修 (= refcount field 削除 = layout 再 pin = JIT trampoline rewrite = Phase A.5 の成果と衝突).  bootstrap circular (= mark/sweep が eval を必要 → eval は alloc を必要 → alloc は GC を必要 という circle).
- *self-host depth*: 最高.  ただし Phase A 投資の大部分が無駄になる.

** §3.3 Option C: hybrid — refcount for hot path + elisp cycle collector on demand

- *content*: =NlRc= / =NlXxxRef= の refcount logic を保持 (= JIT 互換性、layout pin 維持).  Drop 本体は payload drop + dealloc に縮め、elisp 側 cycle collector は =rc-inc/dec= の上で Bacon-Rajan を実装.  finalizer scheduler も elisp.
- *Rust 削減*: ~-100〜-300 LOC (= Drop 本体の重複を =nl_rc_dealloc_generic= 一本化、各 =Drop= は薄 wrapper).
- *elisp 追加*: ~+550-700 LOC (= cycle collector ~400 + scheduler ~150 + per-kind primitive wrapper ~100).
- *risk*: 中.  Phase A の成果を温存しつつ elisp 化が前進.
- *self-host depth*: 中.  Rust に refcount は残るが policy は elisp.

** §3.4 推奨: Option C

reasoning:
1. Phase A の Sexp ABI pin (= 8 boxed kind の layout-pin) は JIT direct emit (= Doc 78 Phase D-A) の前提条件であり、refcount field を消す Option B は Phase A 投資全廃.
2. Doc 77c §2.3 の elisp pseudo code (= =(defun cons (a b) (let ((box (nl-cons-alloc a b))) (nl-rc-inc a) (nl-rc-inc b) box))=) は明確に hybrid 路線.
3. Bacon-Rajan は Option C で機能する (= 必須前提が「mutator が refcount を bump/dec する」).
4. bootstrap circularity 回避が容易 (= cycle collector 不在でも alloc/free は動く、boot phase は cycle 無視で OK).

* §6 Risk + mitigation

** §6.1 Performance regression

- *risk*: cycle collector が elisp 駆動 → trace phase が Rust の Drop より 10-30x 遅い可能性.
- *mitigation*:
  - =gc-collect-cycles= は eval-time の idle hook でのみ走る (= mutator hot path 0 cost).
  - Phase A.5 の JIT trampoline で =nl-rc-inc= / =nl-rc-dec= は native code 化、refcount op は inline level.
  - Stage C.5 で 10% threshold gate.  超過したら C.2 algorithm を sync→async に変更 / write-barrier 追加.

** §6.2 Cycle leak (correctness)

- *risk*: cycle collector 未呼出時 / collector bug 時に rc 残留 → leak.
- *mitigation*:
  - finalizer queue (C.3) で =(garbage-collect)= API を提供、user / image-baker から explicit 呼出可能.
  - boot 期 (= image decode 中) は cycle 形成不可 contract (= Doc 75 PAUSED の image decode は cycle-free).
  - Rust =Drop= 本体は残置 (= Option C 採用).  cycle 以外の non-shared box は Rust =Drop= で即 free.

** §6.3 Bootstrap circularity

- *risk*: cycle collector が elisp で書かれる → elisp eval が必要 → eval は alloc を必要 → alloc は GC を必要 → 無限 loop.
- *mitigation*:
  - alloc / dealloc / inc / dec は pure C-level op、GC を呼ばない (= =std::alloc::alloc= 直 + atomic op のみ).
  - cycle collector は =run-with-idle-timer= か explicit =(garbage-collect)= でのみ起動、boot phase は呼ばない.
  - boot 期 leak は process exit で OS が回収 (= 実害なし; build 1 回 < 10ms cycle 形成余地小).
  - finalizer queue も idle-timer hook で driving、boot 期 disabled.

** §6.4 Atomic ordering subtlety

- *risk*: elisp が =rc-inc= / =rc-dec= を call する際、Rust 内部の =Release= / =Acquire= ordering が elisp 側 race と整合しない.
- *mitigation*:
  - elisp は single-threaded (= Doc 76 の thread 化議論は別 phase).  cross-thread race そのものが現状発生せず.
  - Phase A.1 の =NlRc= test (= =refcount_atomic_round_trip=) で raw ptr access の round-trip 検証済.
  - Phase D の native compiler が thread emit を始めたら、本 Doc を update (= memory barrier elisp primitive 追加).

** §6.5 Phase A 投資の保護

- *risk*: 本 Doc が Option B 路線に shift すると Phase A の Sexp ABI pin / JIT trampoline / cons_primitives.rs (~700 LOC) が無駄.
- *mitigation*: Option C 採用で Phase A 投資 100% 温存.  本 Doc §3.4 の reasoning 1 が gate.

* §7 Test plan

** §7.1 Correctness ERT

- 既存 =equal_two_isomorphic_cycles_compare_equal= / =equal_cycles_with_different_payloads_are_unequal= 等が Stage C.2 完了後 *cycle 解除を観測できる* (= 走った後に =gc-collect-cycles= 呼んで全 box が free) ように ERT 強化.
- per-kind primitive (§4.2) に arity / wrong-type / refcount-balance ERT 各 3 case = ~66 ERT 追加.
- bootstrap test: =cargo run= 経由 + bin =nelisp= 単体経由の両方で同 image が動く.

** §7.2 Bench

- =bench/gc-cycle-pressure.el= 新設、5 シナリオ:
  1. 10M cons alloc-recycle (cycle なし)
  2. 1M vector alloc-free (large payload)
  3. 100k cycle root (5-cycle each) → collect.
