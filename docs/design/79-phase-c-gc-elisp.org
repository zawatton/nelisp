* §0 TL;DR

Doc 77c §2.3 (= "Phase C: GC を elisp に migrate, 1-2 ヶ月") の詳細設計.

*v2 refactor (2026-05-09)*: v1 の per-kind 22 prim 展開 (= +423 Rust) は user 2026-05-09 monotonic shrink rule (= 全 commit が net Rust LOC を減らす) に違反.  Phase B "−408 統合" alibi も独立 audit で *invalidated* (= GATE FAILED、=strategy.rs= に削除すべき cons-arm 不在).  本 v2 では Stage C.1 を *generic NlRc primitive surface* (= opaque ptr + kind tag、5-8 fns) に再設計し、Phase C 単独で net Rust LOC が減るようにする.

- *現状*: Rust 内に refcount-based GC が ~1,936 LOC (= 8 ファイル合計 non-test).  Drop 本体だけで 208 LOC, =NlRc<T>= core 240 LOC, 各 =NlXxxRef= の =Clone= / =Drop= / refcount API が残り.
- *target*: GC の *policy* (= cycle 検出、bulk free、finalizer queue) を elisp に持ち上げ、Rust 側は *mechanism* (= layout-pinned heap + atomic refcount + 単一 generic alloc/dealloc primitive set) のみに縮める.
- *推奨 approach*: =Option C (hybrid) + generic boundary= = refcount mechanism は Rust 残置 (= layout pin 維持 / hot path 維持)、boundary は kind-tag dispatch の *generic NlRc primitive 5-8 件* で全 boxed type を覆い、cycle collector + bulk finalizer scheduler を elisp に新設.
- *sub-stage*: C.0 audit → *C.1+4 generic boundary primitive + Drop 本体 thin 化 (1 atomic stage)* → C.2 cycle collector elisp (+~400 LOC) → C.3 finalizer queue elisp (+~150 LOC) → C.5 self-host gate.
- *Rust 純 delta (Phase C 単独)*: C.1+4 -56〜-101 / 残 stage ±0 = *-56〜-101 LOC Rust* ✅.  monotonic shrink rule *per-commit* で充足 (= C.1+4 を atomic 1 commit ship、Phase B alibi 不要).
- *elisp 純 delta*: ~+660 LOC (= cycle collector +400 + finalizer +150 + wrapper +60 + bench +50).
- *bootstrap circularity*: GC が eval を必要としない設計 (= alloc / dealloc / inc / dec はすべて pure C-like loop) で回避.  cycle collector は eval-time にのみ走るので boot 中 (= image decode 期) は呼ばない.

* §1 動機

** §1.1 100% pure elisp self-host commitment

memory =project_nelisp_100_percent_elisp_commitment= (user 2026-05-09 directive) と Doc 77c §2.3 の framing に従い、GC を含む runtime 全層を elisp で記述可能にする.  Phase C の完了で「Rust 側は heap allocator + atomic 操作 + layout pin の primitive のみ」という最終姿勢に近づく.

** §1.2 elisp 化のメリット

- *Cycle collection が hackable*: Bacon-Rajan / Recycler / Sapphire 等のアルゴリズム選択が elisp 修正のみで可能.  Rust recompile + ABI 同期不要.
- *Finalizer hook の自由度*: weak ref / phantom ref / 大規模 buffer の段階的解放など elisp 側で defcustom できる.
- *self-host depth の articulation*: =project_nelisp_100_percent_elisp_commitment= で謳う「runtime ABI も elisp」の証跡として GC は最後の砦.
- *future 移植 cost*: Rust→Zig / Rust→C 等の seed core 言語変更時、GC policy の elisp 部分は無関係に流用可能 (= Doc 47 phase 6 の C-runtime 自前化に効く).

** §1.3 elisp 化しない選択肢の reason

実は GC を elisp 化しない選択肢もあり得る (= Lua / Squirrel / Tcl 流: GC は VM の責務、language は触らない).  しかし NeLisp は

- 「Elisp が runtime 全責務を持つ」という Doc 73 / 77c の設計の柱
- =defadvice= 文化に基づく、ユーザーが GC policy を override できる Emacs Lisp 的な hackability
- Doc 47 phase 6 の C-runtime self-impl roadmap

から、elisp 化を採る.

** §2.2 Refcount プロトコル (NlRc<T> 由来、全 Ref で再現)

各 =NlXxxRef= は:

1. =new(payload)= → =std::alloc::alloc(Layout::new::<NlXxx>())= で heap 確保 → refcount=1 で初期化、=NonNull<NlXxx>= 保持.
2. =Clone= → =refcount.fetch_add(1, Relaxed)= bump、新 handle 返却.
3. =Drop= → =refcount.fetch_sub(1, Release)=、prev==1 なら =fence(Acquire)= → =drop_in_place(payload)= → =alloc::dealloc=.
4. =Deref= → =&NlXxx= 借用、固定 offset access.

=AtomicUsize= refcount は offset 0 (= NlRc) または末尾 (= NlConsBox) に locked、Phase A.5 JIT trampoline + Phase A.3 elisp =nl-ffi-write-i64= が直接読書きする contract.

** §2.4 Cycle handling 現状

- =NlConsBox= / =NlVector= / =NlCell= 等は acyclic invariant (= Phase A の cons_primitives.rs §記述通り).  ユーザーが =setcdr= で cycle を作れるが、Rust =Drop= は cycle 検出なし → =refcount > 0= のまま leak.
- =NlCharTable= の =parent= field は self-reference 不可 (= API gating、コメント nlchartable.rs §1.5).
- 結果: *cycle leak は現状 Rust に修復ロジックがない*.  既存 elisp test (= =equal_two_isomorphic_cycles_compare_equal= 等) は cycle を作って exit するが GC せず終了に頼っている.

** §2.5 Allocator boundary

全 Box struct は =std::alloc::alloc= = libc =malloc= 直 (= Rust default GlobalAlloc).  arena / pool / bump allocator なし.  third-party GC crate (= bdwgc, rsgc, gc-arena, shifgrethor, broom) は Cargo.toml 検索結果 *zero hit*.

ffi.rs の =Box::leak(Box::new(lib))= は plugin 動的 lib の lifetime 管理で GC 対象外.

** §2.6 既存 elisp boundary primitive

=cons_primitives.rs= (305 LOC, ~140 non-test) で Phase A.3 出荷済:

- =nl-cons-alloc=, =nl-cons-car=, =nl-cons-cdr=, =nl-cons-set-car=, =nl-cons-set-cdr=
- =nl-rc-inc=, =nl-rc-dec=, =nl-rc-count=

Cons 専用.  Phase C で =nl-vector-*= / =nl-str-*= / =nl-cell-*= / =nl-record-*= / =nl-bool-vector-*= / =nl-chartable-*= の similar surface が必要 (= elisp 側が allocate/free/inc/dec を全 boxed kind で driving).

* §3 Migration target — 3 option 比較

** §3.1 Option A: keep refcount in Rust + expose alloc/free as primitives

- *content*: 現状の =Drop= 本体を残し、=alloc/dealloc/inc/dec/count= を elisp primitive 化、cycle collector のみ elisp 新設.
- *Rust 削減*: ほぼ 0 (= cycle 関連 helper だけ).  cons_primitives.rs 系の追加で actually +200 LOC.
- *elisp 追加*: cycle collector ~300 LOC + finalizer scheduler ~150 LOC = ~450 LOC.
- *risk*: 低.  cycle leak 対応のみが elisp 仕事.
- *self-host depth*: 低 (= Rust の Drop 本体を温存).

** §3.2 Option B: replace refcount entirely with mark-sweep tracing GC

- *content*: =NlRc= / =NlXxxRef= refcount logic 全削除、Rust は heap region + roots set を expose、elisp 側 mark/sweep walker が evict.
- *Rust 削減*: ~-1,500 LOC (= refcount Clone/Drop 全廃 + atomic 操作削除).
- *elisp 追加*: ~+1,200-1,500 LOC (= mark/sweep + tri-color + write barrier shim).
- *risk*: 高.  Sexp ABI 大改修 (= refcount field 削除 = layout 再 pin = JIT trampoline rewrite = Phase A.5 の成果と衝突).  bootstrap circular (= mark/sweep が eval を必要 → eval は alloc を必要 → alloc は GC を必要 という circle).
- *self-host depth*: 最高.  ただし Phase A 投資の大部分が無駄になる.

** §3.3 Option C: hybrid — refcount for hot path + elisp cycle collector on demand

- *content*: =NlRc= / =NlXxxRef= の refcount logic を保持 (= JIT 互換性、layout pin 維持).  Drop 本体は payload drop + dealloc に縮め、elisp 側 cycle collector は =rc-inc/dec= の上で Bacon-Rajan を実装.  finalizer scheduler も elisp.
- *Rust 削減*: -56〜-101 LOC (= 後述 §4 generic boundary refactor で C.1 +115〜160 を C.4 -216 が上回る).
- *elisp 追加*: ~+660 LOC (= cycle collector ~400 + scheduler ~150 + generic wrapper ~60 + bench ~50).
- *risk*: 中.  Phase A の成果を温存しつつ elisp 化が前進.  generic API は kind-tag corruption / type-erased pointer debugging という新 risk を導入 (§6.6 参照).
- *self-host depth*: 中.  Rust に refcount は残るが policy は elisp.

** §3.4 推奨: Option C + generic boundary

reasoning:
1. Phase A の Sexp ABI pin (= 8 boxed kind の layout-pin) は JIT direct emit (= Doc 78 Phase D-A) の前提条件であり、refcount field を消す Option B は Phase A 投資全廃.
2. Doc 77c §2.3 の elisp pseudo code (= =(defun cons (a b) (let ((box (nl-cons-alloc a b))) (nl-rc-inc a) (nl-rc-inc b) box))=) は明確に hybrid 路線.
3. Bacon-Rajan は Option C で機能する (= 必須前提が「mutator が refcount を bump/dec する」).
4. bootstrap circularity 回避が容易 (= cycle collector 不在でも alloc/free は動く、boot phase は cycle 無視で OK).
5. *v2 追加 reasoning*: per-kind 22 prim (v1) ではなく =nl-rc-*= 5-8 fns + kind tag dispatch にすれば、Stage C.1 Rust delta が +423 → +115〜160 に圧縮、C.4 の Drop body 統一 (-216) と合わせて Phase C 単独で net 負 delta.  Phase B 統合に依存せずに monotonic shrink rule を満たせる.

* §4 Boundary primitives — Rust → elisp 露出する API

*v2 refactor*: v1 の per-kind 22 prim (= +423 Rust) を *generic NlRc primitive 5-8 fns + kind-tag dispatch* に再設計.  payload access (= car/cdr/aref 等) は本 Doc 79 範囲外として Phase B (Sexp 操作 elisp 化) に投げる (= Phase B は元々 =strategy.rs= 大改修を含むので per-kind access prim はそちらの自然責務).  本 Doc 79 は *GC mechanism boundary* に絞る.

** §4.1 既存 (Phase A 出荷済、Phase C で再利用)

- =nl-cons-alloc=, =nl-cons-car=, =nl-cons-cdr=, =nl-cons-set-car=, =nl-cons-set-cdr=
- =nl-rc-inc=, =nl-rc-dec=, =nl-rc-count=

cons 専用に出ているこれらは Phase B で =nl-rc-*= generic 経由に rewire 予定 (= Phase B 範疇).  本 Doc では既存 surface を破壊しない.

** §4.2 Phase C で追加 (generic NlRc primitive set)

設計の柱:

1. *kind tag*: 単一 byte enum =NlRcKind= (= Phase A.5.1 の =Sexp::tag()= 由来 byte をそのまま再利用).  値 = =0=Cons / =1=Cell / =2=MutStr / =3=Vector / =4=BoolVector / =5=Record / =6=CharTable / =7=Generic.  kind tag は alloc 時に refcount trailer の隣 (= 1 byte 追加 field、各 NlXxxBox layout に既存の padding を使えば *layout pin 不変*) に書かれる.
2. *DROP_FN const*: 各 =impl NlXxxBox= に =const DROP_FN: unsafe fn(*mut c_void)= を追加 (= ~5 LOC × 8 kind = +40 LOC).  drop dispatch は =NLRC_DROP_TABLE: [unsafe fn(*mut c_void); 8]= から kind tag で index lookup.
3. *generic primitive 5-7 fns* (= 各 ~10-20 Rust LOC、bodies + extern 宣言 + arity check 込みで合計 ~75-120 LOC).

| 新規 prim              | sig (Rust extern)                                            | 用途                                                    | LOC 見積 |
|------------------------|--------------------------------------------------------------|---------------------------------------------------------|----------|
| =nl-rc-alloc=          | =unsafe extern "C" fn(kind: u8, payload_size: usize) -> *mut c_void= | kind dispatch で対応 =NlXxxBox::new()= 内部呼出、opaque ptr 返却 | 20       |
| =nl-rc-dealloc=        | =unsafe extern "C" fn(ptr: *mut c_void) -> ()=                | trailer の kind 読取 → =NLRC_DROP_TABLE[kind]= dispatch    | 15       |
| =nl-rc-inc-strong=     | =unsafe extern "C" fn(ptr: *mut c_void) -> ()=                | trailer の AtomicUsize fetch_add(1, Relaxed)             | 12       |
| =nl-rc-dec-strong=     | =unsafe extern "C" fn(ptr: *mut c_void) -> usize=             | fetch_sub(1, Release) → 新 count 返却 (auto-drop *なし*; elisp が explicit dealloc) | 15       |
| =nl-rc-strong-count=   | =unsafe extern "C" fn(ptr: *mut c_void) -> usize=             | trailer load(Acquire)                                    | 10       |
| =nl-rc-kind=           | =unsafe extern "C" fn(ptr: *mut c_void) -> u8=                | trailer の kind byte 読取 (cycle collector の dispatch 用)  | 8        |
| =nl-rc-payload-ptr=    | =unsafe extern "C" fn(ptr: *mut c_void) -> *mut c_void=       | interior pointer (= NlXxx struct の先頭 byte) を返却、=nl-ffi-read-bytes-at= で elisp が struct walk | 10       |
| *合計 generic prim*    | *7 fns*                                                      |                                                         | *~90*    |

加えて、各 =impl NlXxxBox= に =DROP_FN= constant を追加:

#+begin_src rust
impl NlConsBox {
    pub const DROP_FN: unsafe fn(*mut c_void) = nlconsbox_drop_thunk;
    // ...
}
unsafe fn nlconsbox_drop_thunk(ptr: *mut c_void) {
    let p = ptr as *mut NlConsBox;
    std::ptr::drop_in_place(p);
    std::alloc::dealloc(p as *mut u8, std::alloc::Layout::new::<NlConsBox>());
}
#+end_src

8 kind × ~5 LOC = +40 LOC.  drop table は単一 const array:

#+begin_src rust
const NLRC_DROP_TABLE: [unsafe fn(*mut c_void); 8] = [
    NlConsBox::DROP_FN,
    NlCell::DROP_FN,
    NlStr::DROP_FN,
    NlVector::DROP_FN,
    NlBoolVector::DROP_FN,
    NlRecord::DROP_FN,
    NlCharTable::DROP_FN,
    nl_rc_generic_drop_thunk, // for NlRc<T> pure refcount payloads
];
#+end_src

加えて GC 用 helper 3 fns (= cycle collector が必要):

| 新規 prim                 | sig                                  | 用途                                                   | LOC 見積 |
|---------------------------|--------------------------------------|--------------------------------------------------------|----------|
| =nl-gc-walk-children=     | =unsafe extern "C" fn(*mut c_void) -> *mut c_void= (returns Sexp pointer of List) | 1 box の outgoing edge 列挙、kind tag dispatch で内部 layout walk | 30       |
| =nl-gc-buffered-decs=     | =unsafe extern "C" fn() -> *mut c_void= (returns Sexp List)                       | recent dec の buffer (Bacon-Rajan roots candidate)       | 25       |
| =nl-gc-finalize=          | =unsafe extern "C" fn(*mut c_void)=  | refcount=0 強制 finalize (cycle 解除後 dealloc shortcut) | 15       |
| *合計 GC helper*          | *3 fns*                              |                                                        | *~70*    |

*Stage C.1 Rust 合計*: 90 (generic prim) + 40 (DROP_FN) + 70 (GC helper) = *+200 LOC* — 上限見積.  arity check / wrong-type test を tight に書けば +115〜160 LOC range に収まる.  本 Doc では保守側 *+115〜160* を target、+200 を upper bound として記載.

(Note: =nl-gc-trace-roots= / =nl-gc-rc-snapshot= は v1 にあったが、=nl-rc-strong-count= で代用可能 / mutator の root set は elisp 側で stack-walk すべきと判断し v2 で削除.  surface 縮小に貢献.)

** §4.3 elisp 側の generic API 駆動

elisp wrapper の典型形:

#+begin_src elisp
(defun nl-cons-alloc (a b)
  "Allocate a new cons cell."
  (let ((ptr (nl-rc-alloc 0 (nl-cons-payload-size))))  ; kind=0 = Cons
    (nl-rc-inc-strong (nl-sexp-as-ptr a))
    (nl-rc-inc-strong (nl-sexp-as-ptr b))
    ;; payload write は Phase B の per-kind access prim 経由
    (nl-cons--init-payload ptr a b)
    (nl-ptr-as-sexp 0 ptr)))  ; tag=0 = Cons sexp
#+end_src

cycle collector 中の kind dispatch:

#+begin_src elisp
(defun gc-walk-children (ptr)
  (let ((kind (nl-rc-kind ptr)))
    (cond
     ((= kind 0) (gc--cons-children ptr))      ; car + cdr
     ((= kind 3) (gc--vector-children ptr))    ; len 個 element
     ((= kind 5) (gc--record-children ptr))    ; type-tag + slots
     ;; ...
     (t nil))))
#+end_src

** §4.4 elisp wrapper 命名規約

Phase B (= Sexp 操作 elisp 化) との衝突回避:
- 公開 =cons= / =car= / =cdr= 等は Phase B で elisp 再実装、=nl-cons-*= / =nl-rc-*= の上に build.
- Phase C 追加分は =nl-rc-*= prefix 維持 (= 「raw, not-for-public」signal).
- elisp 公開 GC API は =gc-= prefix で別 layer (= =(gc-collect-cycles)=, =(gc-stats)= 等).

* §5 Stage breakdown

Doc 77c §2.3 の "1-2 ヶ月" estimate を 6 sub-stage に展開.  各 stage は monotonic shrink rule に従う (= test_count_unchanged_or_grow + Rust LOC monotonically non-increasing on net total).

** §5.1 Stage C.0 — audit + design freeze (1 週間)

- 本 Doc 79 の review + commit.
- generic NlRc primitive surface 確定 (= §4.2 の 7 generic + 3 GC helper = 10 fns 確定).
- Bacon-Rajan アルゴリズム選定確認 (= alternative: Lins, Recycler, Synchronous CC) → Bacon-Rajan が elisp 実装 cost / 動作 trace 容易性 / Doc 77c §2.3 言及済 で確定.
- *Rust delta*: ±0.  *elisp delta*: ±0.
- *commit-rule check*: net Rust delta = 0、monotonic shrink rule 充足 (= 増加なし).

** §5.2 Stage C.1+4 — generic boundary primitive set + Rust Drop 本体 thin 化 (4-5 週間)

- §4.2 の generic NlRc primitive 7 fns + GC helper 3 fns を =build-tool/src/eval/rc_primitives.rs= 単一 file に集約 (= 8 file 新設しない、ここが per-kind 22 prim 案との最大差).
- 各 =impl NlXxxBox= に =DROP_FN: unsafe fn(*mut c_void)= constant 追加 (8 kind × ~5 LOC).
- =NLRC_DROP_TABLE: [unsafe fn(*mut c_void); 8]= const array を =nlrc.rs= に追加 (~10 LOC).
- Rust ERT: =nl_rc_alloc= 各 kind / =nl_rc_dealloc= round-trip / =nl_rc_inc_strong= + =dec_strong= balance / =nl_rc_kind= byte read / =nl_rc_payload_ptr= interior valid / =nl_gc_walk_children= per-kind 列挙.
- elisp wrapper は薄 (= 単に generic prim を kind 引数で wrap、policy は C.2 以降で書く).
- *Rust delta (C.1 portion)*: +115〜160 LOC (target) / +200 LOC (upper bound).  内訳: generic prim ~90 + DROP_FN const ~40 + GC helper ~70 (upper) → arity/error check tight 化で ~115〜160 に収まる見込み.
- *elisp delta*: +~60 LOC (= 8 kind 用 thin wrapper、cons の既存 surface に合わせる adapter).

*v3 merge (= 旧 §5.5 Stage C.4 を本 stage に統合)*:

generic =NLRC_DROP_TABLE= + =DROP_FN= constants (上記 C.1 portion で導入) を活用し、各 =impl Drop= 本体を 1 行 dispatch に統一.

- =nlconsbox.rs= (35 LOC) / =nlcell.rs= (28) / =nlstr.rs= (24) / =nlvector.rs= (21) / =nlboolvector.rs= (21) / =nlrecord.rs= (22) / =nlchartable.rs= (23) + =nlrc.rs= (34) 各 =Drop= の本体を:

#+begin_src rust
impl Drop for NlConsBox {
    fn drop(&mut self) {
        unsafe { (Self::DROP_FN)(self as *mut _ as *mut c_void) }
    }
}
#+end_src

の ~3 LOC pattern に統一.

- 8 kind × ~30 LOC (現行平均 26、generic は 34) → ~3 LOC each = 8 × ~3 = 24 LOC.  削減 = 208 (=§2.3 audit 合計) - ~24 - 8 (各 file の use 1 行) = *~-176 LOC*.  保守側で *-216 LOC* を target、-176 を lower bound として記載.  realistic mid-point は *~-200 LOC*.
- =impl NlXxxRef= の =Clone= は touch しない (= refcount inc は generic API 経由で elisp が呼ぶが、Rust 内部利用も残るので =Clone= は維持).
- *Rust delta (C.4 portion)*: -216 LOC (target) / -176 LOC (lower).

*** §5.3.1 Bacon-Rajan algorithm overview

Bacon-Rajan = "Concurrent Cycle Collection in Reference Counted Systems" (ECOOP 2001).  本 Doc は *synchronous* 版 (= concurrent mutator なし、elisp single-thread 前提) を採用.  4 phase で構成:

- *Phase 1: Mark roots (= "suspect" buffering)*
  - 各 boxed が =nl-rc-dec-strong= で refcount dec されたとき、新 count が 0 にならず *かつ* 自身が cycle 候補となり得る kind (= Cons / Vector / Cell / Record / CharTable) なら =gc-cycle-roots-buffer= に push.
  - 不要な root (= MutStr / BoolVector 等 child を持たない kind) は最初から除外する.
  - 既に buffer 済の box は重複 push 抑止 (= "purple" color 検査).

- *Phase 2: Trial deletion*
  - root buffer の各 box から =nl-gc-walk-children= で reachable graph を DFS walk.
  - 訪問するたびに *internal* refcount を =gc-internal-rc-table= で 1 つ仮想 dec.
  - 訪問済 box は "gray" color で mark、cycle (= back edge) は walk 終了.
  - 結果として、*真の external root* から到達できる box は internal_rc < strong_count、cycle のみで参照される box は internal_rc == strong_count.

- *Phase 3: Scan*
  - root buffer を再 walk.  internal_rc < strong_count (= external 参照あり) の box は再 inc して "black" mark (= live).
  - internal_rc == strong_count の box は "white" mark (= cycle、garbage).
  - ※ scan は再帰的: white を mark する際、その children も white に下り mark (= 後段の collect-white で free を漏らさないため).

- *Phase 4: Collect white*
  - white-marked box の =nl-gc-finalize= を強制呼出 (= refcount 無視で payload drop + dealloc).
  - large payload (= NlVector size > threshold) は Stage C.3 finalizer queue に enqueue (= synchronous free を 避ける).

reference: Bacon-Rajan paper §4 ("Algorithm 4: Synchronous Cycle Collection") に対応.  本 Doc では *Algorithm 4* を simplified 形で採用 (= weak ref / phantom ref 抜き、Stage C.3 で finalizer queue 拡張).

*** §5.3.2 elisp data structures

| 名前                       | 型                        | 用途                                                |
|----------------------------+---------------------------+-----------------------------------------------------|
| =gc-cycle-roots-buffer=    | list of opaque ptr        | dec で 0 にならず purple mark された suspect box list |
| =gc-color-table=           | hash (ptr → symbol)       | =black= (live) / =gray= (visiting) / =white= (cycle、garbage) / =purple= (root candidate buffered) |
| =gc-internal-rc-table=     | hash (ptr → int)          | trial deletion 中の virtual refcount.  初期値 = =nl-rc-strong-count= ptr |
| =gc-finalize-queue=        | list of opaque ptr        | Stage C.3 idle-timer finalizer に手渡す large box    |
| =gc-cycle-roots-limit=     | int (defcustom = 32768)   | buffer overflow protect、超過時は immediate fire     |
| =gc-stats-counters=        | plist                     | observability: cycles-collected / boxes-freed / passes 統計 |

すべて elisp object、Stage C.1.b の =nl-rc-*= primitive を介して Rust heap と接続.  =gc-color-table= / =gc-internal-rc-table= は =gc-collect-cycles= の各 invocation で =clrhash= して reuse (= GC pressure 自体を増やさない).

color table の "kind tag" は §4.2 の =NlRcKind= = 0..=12 の sexp tag 派生.  Cons / Vector / Cell / Record / CharTable のみ root-candidate kind とする (= MutStr / BoolVector / Generic は children 0 で cycle 不可、early-skip).

*** §5.3.3 Per-phase pseudo-elisp

以下は articulation のための pseudo (= 実装時に微調整).  =cond= / =when= / =unless= / =null= / =not= / =nelisp--signal-*= は Doc 80 substrate (= STDLIB_FILES[0]) で利用可能.  =gethash= / =puthash= / =clrhash= / =maphash= は =nelisp-stdlib-hash.el= で提供済.

#+begin_src elisp
;; --- Phase entry point ---
(defun gc-collect-cycles ()
  "Bacon-Rajan synchronous cycle collection (Doc 79 §5.3)."
  (let ((roots (gc--snapshot-roots)))
    (when roots
      (clrhash gc-color-table)
      (clrhash gc-internal-rc-table)
      (gc--mark-roots roots)            ; Phase 1 (snapshot already; this seeds purple)
      (gc--trial-delete-pass roots)     ; Phase 2
      (gc--scan-pass roots)             ; Phase 3
      (gc--collect-white-pass roots)    ; Phase 4
      (setq gc-cycle-roots-buffer nil))
    (gc--bump-stat 'passes 1)))

;; --- Phase 1 helper: snapshot + mark purple ---
(defun gc--snapshot-roots ()
  "Drain `gc-cycle-roots-buffer' into a list and mark each purple."
  (let ((acc nil)
        (drained gc-cycle-roots-buffer))
    (setq gc-cycle-roots-buffer nil)
    (dolist (ptr drained acc)
      (unless (eq (gethash ptr gc-color-table) 'purple)
        (puthash ptr 'purple gc-color-table)
        (setq acc (cons ptr acc))))))

(defun gc--mark-roots (roots)
  "No-op in Algorithm 4 (purple already set in snapshot).  Hook for future async."
  (ignore roots))

;; --- Phase 2: trial deletion ---
(defun gc--trial-delete-pass (roots)
  (dolist (ptr roots)
    (gc--trial-delete-walk ptr)))

(defun gc--trial-delete-walk (ptr)
  (let ((color (gethash ptr gc-color-table 'black)))
    (unless (eq color 'gray)
      (puthash ptr 'gray gc-color-table)
      ;; Initialize internal rc snapshot lazily.
      (unless (gethash ptr gc-internal-rc-table)
        (puthash ptr (nl-rc-strong-count ptr) gc-internal-rc-table))
      (let ((children (gc--children ptr)))
        (dolist (child children)
          (when (gc--root-candidate-kind-p child)
            (puthash child (- (or (gethash child gc-internal-rc-table)
                                  (nl-rc-strong-count child))
                              1)
                     gc-internal-rc-table)
            (gc--trial-delete-walk child)))))))

;; --- Phase 3: scan ---
(defun gc--scan-pass (roots)
  (dolist (ptr roots)
    (gc--scan-walk ptr)))

(defun gc--scan-walk (ptr)
  (let ((color (gethash ptr gc-color-table)))
    (when (eq color 'gray)
      (let ((internal (gethash ptr gc-internal-rc-table 0))
            (strong   (nl-rc-strong-count ptr)))
        (cond
         ((> strong internal)
          (gc--scan-black ptr))            ; live, has external ref
         (t
          (puthash ptr 'white gc-color-table)
          (dolist (child (gc--children ptr))
            (when (gc--root-candidate-kind-p child)
              (gc--scan-walk child)))))))))

(defun gc--scan-black (ptr)
  "Re-mark `ptr' and its children black (= live), restoring internal rc."
  (puthash ptr 'black gc-color-table)
  (dolist (child (gc--children ptr))
    (when (gc--root-candidate-kind-p child)
      (puthash child (+ (or (gethash child gc-internal-rc-table)
                            (nl-rc-strong-count child))
                        1)
               gc-internal-rc-table)
      (unless (eq (gethash child gc-color-table) 'black)
        (gc--scan-black child)))))

;; --- Phase 4: collect white ---
(defun gc--collect-white-pass (roots)
  (dolist (ptr roots)
    (when (eq (gethash ptr gc-color-table) 'white)
      (gc--collect-white-walk ptr))))

(defun gc--collect-white-walk (ptr)
  (when (eq (gethash ptr gc-color-table) 'white)
    (puthash ptr 'black gc-color-table)
    (dolist (child (gc--children ptr))
      (when (gc--root-candidate-kind-p child)
        (gc--collect-white-walk child)))
    ;; Hand to Stage C.3 finalizer queue if large; else inline.
    (if (gc--large-payload-p ptr)
        (push ptr gc-finalize-queue)
      (nl-gc-finalize ptr))
    (gc--bump-stat 'boxes-freed 1)))

;; --- kind dispatch helpers ---
(defun gc--children (ptr)
  "List of child opaque ptrs reachable from PTR (= Cons car/cdr, Vector elts, ...)"
  (let ((kind (nl-rc-kind ptr)))
    (cond
     ((= kind 7)  (gc--cons-children ptr))      ; CONS = car + cdr
     ((= kind 8)  (gc--vector-children ptr))    ; VECTOR = N elts
     ((= kind 11) (gc--cell-children ptr))      ; CELL = single payload
     ((= kind 12) (gc--record-children ptr))    ; RECORD = type + slots
     ((= kind 9)  (gc--chartable-children ptr)) ; CHAR_TABLE = parent + slot vec
     (t nil))))

(defun gc--root-candidate-kind-p (ptr)
  "Return non-nil iff PTR's kind has child edges (= cycle-possible)."
  (let ((kind (nl-rc-kind ptr)))
    (or (= kind 7) (= kind 8) (= kind 11) (= kind 12) (= kind 9))))
#+end_src

(各 helper 5-15 LOC、合計 §5.3.3 で ~150 elisp LOC.  per-kind children extraction = §5.3.5.b で別途 ~80 LOC.)

*** §5.3.6 Risk + mitigation

**** §5.3.6.1 Performance regression (elisp GC slower than Rust refcount)

- *risk*: =gc-collect-cycles= 内の hash table op + per-kind cond dispatch + recursive walk が Rust =Drop= に比べて 10-30x 遅い可能性.
- *mitigation*:
  - cycle collector は *eval-time idle hook* でのみ走る (= mutator hot path 0 cost).  =run-with-idle-timer= で 1s idle で fire.
  - =nl-rc-inc-strong= / =-dec-strong= は Phase A.5 JIT trampoline で native code 化 (= primitive 自体の op cost は inline level).
  - hash table は =clrhash= で reuse、GC pressure を増やさない.
  - *threshold gate*: Stage C.5 で baseline (= Phase A) 比 -10% 以下 (= 90% 速度維持) に収束させる.  超過したら C.2 algorithm を Algorithm 5 (= concurrent CC、async background) に shift / write-barrier 追加.

**** §5.3.6.2 Concurrent mutator

- *risk*: cycle collector が trial deletion 中に mutator が refcount を変更すると inconsistent state (= Bacon-Rajan の有名 race).
- *mitigation*:
  - elisp は *single-threaded* (= §6.4 既存 articulation).  cross-thread race そのものが現状発生せず.
  - eval-time に走るが、=gc-collect-cycles= 自体が elisp なので invocation 中 mutator は走らない (= sequential consistency).
  - Phase D が thread emit を始めたら (= multi-thread eval)、本 §を re-articulate (= memory barrier primitive 追加 / Algorithm 5 への shift / weak-ref ロック).

**** §5.3.6.3 Bootstrap circularity (verified absent)

- *risk*: GC が elisp で書かれる → eval が必要 → eval は alloc を必要 → alloc が GC を必要 → 無限 loop.
- *mitigation* (= verified absent):
  - =nl-rc-alloc= / =-dealloc= / =-inc-strong= / =-dec-strong= は pure C-like primitive (= Stage C.1.b articulation 済).  GC を呼ばない.
  - cycle collector は =run-with-idle-timer= か explicit =(garbage-collect)= でのみ起動.  boot phase (= image decode 中) は disabled、=gc-cycle-roots-buffer= も image decode 中は push されない (= initialization order で =run-with-idle-timer= は最終).
  - boot 期 cycle leak は process exit で OS が回収 (= 実害なし; build 1 回 < 10ms cycle 形成余地小).
  - finalizer queue (Stage C.3) も idle-timer hook で driving、boot 期 disabled.
  - *確認*: alloc path に elisp 呼出無し / cycle-collector が require する関数群 (=cond= / =when= / =null= / =gethash= / =puthash= / =clrhash= / =dolist=) は Doc 80 substrate + =nelisp-stdlib-hash.el= で boot 完了済.

**** §5.3.6.4 Walk-children for unboxed Sexp

- *risk*: =nl-gc-walk-children= が unboxed Sexp (= Nil / T / Int / Float / Symbol / Str inline tag) を渡されたら crash / SEGV.
- *mitigation*:
  - =nl-rc-kind= が tag 0..=5 (= unboxed range) を返したら =gc--root-candidate-kind-p= が nil を返す = early-skip.
  - =gc--children= も =cond= の default 節で nil 返却.
  - debug build で =nl-rc-kind= が 13..=255 を返したら =error= raise (= type-confusion catch、§6.6 articulation 援用).
  - ERT: 各 unboxed kind を root buffer に強制 push して collector が clean に skip すること.

**** §5.3.6.5 Cycle root buffer overflow

- *risk*: dec rate が cycle collection rate を上回り、=gc-cycle-roots-buffer= が肥大化、heap pressure を悪化させる.
- *mitigation*:
  - =gc-cycle-roots-limit= (defcustom = 32768) で hard cap.  超過時は =gc-collect-cycles= immediate fire.
  - Stage C.5 bench で adaptive threshold (= alloc rate × ratio) を articulate.

**** §5.3.7.3 Stress test

- 100k allocation, 1% cycle rate (= 1k cycle、各 cycle 5 box 平均) = 5k cycle box + 95k acyclic box.
- 全 dec → cycle box 5k は =gc-cycle-roots-buffer= に集積、acyclic 95k は Rust =Drop= で即 free.
- =gc-collect-cycles= N 回 (= ceil(5k / 32k limit) = 1 回で足りる) で 5k 全 collect.
- Stage C.5 bench gate: 100k workload total time が Phase A baseline 比 +10% 以内.

**** §5.3.7.4 既存 cycle test の collect 観測 mode

- =equal_two_isomorphic_cycles_compare_equal= 等の既存 cycle-using ERT を *collect 観測 wrapper* で再走 (= test 終了後 =gc-collect-cycles=、strong_count 0 確認).
- 既存 test 自体は変更せず、=ert-deftest gc-existing-cycle-collect-observe= で別 test として追加.

*** §5.3.8 Integration with Stage C.3 finalizer queue

Stage C.3 (= Doc 79 §5.4 articulation) は large payload の incremental free を articulate.  本 Stage C.2 は collect-white phase の最終で finalize 経路を 2 分岐:

#+begin_src elisp
(if (gc--large-payload-p ptr)
    (push ptr gc-finalize-queue)
  (nl-gc-finalize ptr))
#+end_src

handoff API:

- *=gc-finalize-queue=* (= elisp dynamic var、list of opaque ptr): Stage C.2 が enqueue、Stage C.3 が =run-with-idle-timer= で =gc-finalize-step= 経由 dequeue.
- *=gc--large-payload-p ptr=* (= predicate): Stage C.2 では threshold = nil (= 常に false、全 inline finalize) で start.  Stage C.3 着手時に =nl-rc-payload-size= primitive (= §5.3.4 articulate 候補) を追加して threshold check 化.
- *=gc-stats-counters '(deferred-bytes)=* (= Stage C.3 で追加): finalizer queue 経由で deferred な bytes 累計 (observability).

ordering invariant: =gc-finalize-queue= に入った box は次 idle cycle で free.  その間に =gc-collect-cycles= が再 fire しても、queue 上の ptr は color = black (= 既 collect 済) で gc--children walk からも保護される (= queue に乗せた時点で children link を null化、tomb 化).

*Stage C.2 単独 ship 時の振る舞い*: =gc-finalize-queue= は常に空、全 box が inline =nl-gc-finalize=.  Stage C.3 着手で threshold > 0 化、queue が活性化.  C.2 ship 時点で *handoff API は wired*、threshold だけ flip すれば C.3 が機能する設計.

* §6 Risk + mitigation

** §6.1 Performance regression

- *risk*: cycle collector が elisp 駆動 → trace phase が Rust の Drop より 10-30x 遅い可能性.
- *mitigation*:
  - =gc-collect-cycles= は eval-time の idle hook でのみ走る (= mutator hot path 0 cost).
  - Phase A.5 の JIT trampoline で =nl-rc-inc= / =nl-rc-dec= は native code 化、refcount op は inline level.
  - Stage C.5 で 10% threshold gate.  超過したら C.2 algorithm を sync→async に変更 / write-barrier 追加.

** §6.2 Cycle leak (correctness)

- *risk*: cycle collector 未呼出時 / collector bug 時に rc 残留 → leak.
- *mitigation*:
  - finalizer queue (C.3) で =(garbage-collect)= API を提供、user / image-baker から explicit 呼出可能.
  - boot 期 (= image decode 中) は cycle 形成不可 contract (= Doc 75 PAUSED の image decode は cycle-free).
  - Rust =Drop= 本体は残置 (= Option C 採用).  cycle 以外の non-shared box は Rust =Drop= で即 free.

** §6.3 Bootstrap circularity

- *risk*: cycle collector が elisp で書かれる → elisp eval が必要 → eval は alloc を必要 → alloc は GC を必要 → 無限 loop.
- *mitigation*:
  - alloc / dealloc / inc / dec は pure C-level op、GC を呼ばない (= =std::alloc::alloc= 直 + atomic op のみ).
  - cycle collector は =run-with-idle-timer= か explicit =(garbage-collect)= でのみ起動、boot phase は呼ばない.
  - boot 期 leak は process exit で OS が回収 (= 実害なし; build 1 回 < 10ms cycle 形成余地小).
  - finalizer queue も idle-timer hook で driving、boot 期 disabled.

** §6.4 Atomic ordering subtlety

- *risk*: elisp が =rc-inc= / =rc-dec= を call する際、Rust 内部の =Release= / =Acquire= ordering が elisp 側 race と整合しない.
- *mitigation*:
  - elisp は single-threaded (= Doc 76 の thread 化議論は別 phase).  cross-thread race そのものが現状発生せず.
  - Phase A.1 の =NlRc= test (= =refcount_atomic_round_trip=) で raw ptr access の round-trip 検証済.
  - Phase D の native compiler が thread emit を始めたら、本 Doc を update (= memory barrier elisp primitive 追加).

** §6.5 Phase A 投資の保護

- *risk*: 本 Doc が Option B 路線に shift すると Phase A の Sexp ABI pin / JIT trampoline / cons_primitives.rs (~700 LOC) が無駄.
- *mitigation*: Option C 採用で Phase A 投資 100% 温存.  本 Doc §3.4 の reasoning 1 が gate.

** §6.6 Generic API type erasure (v2 新規 risk)

- *risk*: =nl-rc-alloc= が =*mut c_void= を返す = Rust 側 type checker が NlConsBox / NlVector の取り違えを検出できない.  kind tag corruption (= 1 byte 破壊) で全 Drop dispatch が wrong DROP_FN を引き、SEGV / heap corruption / silent UAF.
- *mitigation*:
  - kind tag は refcount trailer の 1 byte (= AtomicUsize の隣).  layout pin を破壊しないこと、固定 offset であることを Phase A.5.1 の Sexp ABI helper に追記.
  - =nl-rc-alloc= が返す opaque ptr を elisp 側で =Sexp= に packing するときに、tag-from-Sexp と =nl-rc-kind= の値が一致することを assert (= cheap byte compare、ERT で coverage).
  - debug build で =nl_rc_dealloc= が =NLRC_DROP_TABLE[kind]= を呼ぶ前に「magic word」(= alloc 時に書いた 4 byte sentinel) を check、type-confusion を catch.  release build では 0 cost.
  - cycle collector の =gc-walk-children= も kind dispatch を持つので、kind 不正値 = 8〜255 は =error= raise (= silent leak より visible 失敗).
  - ERT 追加: =nl_rc_alloc(0, …) → nl_rc_kind == 0=、各 kind round-trip、unknown kind の panic / error 動作.

** §6.7 Generic API debugging difficulty (v2 新規 risk)

- *risk*: =*mut c_void= が gdb / lldb で type 表示されない、bug 解析時に kind を毎回手動 lookup.
- *mitigation*:
  - elisp 側 =gc-describe-ptr= helper で kind tag + payload preview を pretty-print.
  - rust-gdb pretty printer 拡張 (= =impl fmt::Debug for NlRcOpaque= を debug-only feature で provide).
  - cycle collector の trace log に必ず kind name を記録 (= =[gc] walk Cons 0x7f.. children 2 internal-rc 1=).
