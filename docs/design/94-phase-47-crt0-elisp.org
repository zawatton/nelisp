* §0 TL;DR

** §0.1 crt0 とは

crt0 (= "C runtime 0") = OS kernel が =execve(2)= で binary を load し
た直後、user-space で **main() の前に走る最小の起動 stub**。

通常の libc (glibc / musl) では =crt1.o= / =Scrt1.o= として ~20-50 行
の assembly で実装されており、role は:

1. =_start= label = ELF entry point (= Ehdr.e_entry が指す address) =
   kernel が直接 jump する場所
2. Stack pointer (=rsp= / =sp=) 上に kernel が積んだ argv / envp /
   auxiliary vector を unpack
3. =argc= と =argv= を System V ABI に従って register に load
4. =main(argc, argv, envp)= を call
5. main の戻り値 (= =rax= / =x0=) を引数に =exit(2)= syscall

crt0 の特徴は **libc 非依存**: kernel は ELF を mmap してから =_start=
に jump するだけで、=write= / =printf= / =malloc= 等は user 側で raw
=syscall= instruction を emit するしかない (= dynamic loader を使わない
static binary 前提)。

** §0.2 elisp で書ける根拠

crt0 は **pure-functional bytewise machine code emit** に分解できる:

- input = main 関数の symbol name (= elisp string)
- output = byte sequence (= unibyte string、x86_64 機械語 ~30-50 bytes)
- 副作用 = なし (= caller が ELF =.text= section に concat するだけ)
- syscall は =write(2)= / =exit(2)= の 2 個のみ、両方とも x86_64 SysV
  ABI が規定する register 配置で =syscall= instruction を 1 個 emit
  すれば済む

elisp で =unibyte-string= / =vconcat= / 16進 literal が動けば実装可能。
Doc 92 macro assembler の output (= byte vector) を Doc 91 ELF writer
の =.text= section に流し込むだけで、binary が executable 化する。

** §0.3 なぜ Phase 47 が crt0 を必要とするか

Phase 47 = "0 Rust" 目標。現状の =./nelisp= binary は:

- =bin/anvil-runtime= (= shell wrapper、62 LOC) → =NELISP_BIN= に
  exec
- =nelisp-runtime/src/main.rs= (= Rust binary entry、130 LOC) → Rust
  std =fn main()= → libc =__libc_start_main= → =_start=
- =build-tool/src/bin/nelisp.rs= (= production binary entry、243 LOC) →
  同上

これら 3 entry はすべて **Rust std + libc startup** を経由する。
=fn main()= が呼ばれる時点で、libc は既に argv 整形 / TLS setup /
exception handling table 登録 / atexit 連鎖を完了している。Phase 47 は
この依存連鎖そのものを切り、=_start= から **直接 elisp evaluator entry
を call する** ことを目標とする。

crt0 elisp が SHIPPED すると:
1. =_start= は 5-10 命令の elisp-emitted stub
2. その stub が elisp-emitted argv unpacker を call
3. unpacker が elisp-emitted main (= NeLisp eval entry) を call
4. main が return すると elisp-emitted =exit(2)= で終了

**この時点で binary 内に Rust も libc も存在しない**。

** §2.2 =nelisp-crt0-emit-entry FUNC-NAME=

#+begin_src elisp
(defun nelisp-crt0-emit-entry (func-name &optional opts)
  "Emit a crt0 entry stub that ends in calling FUNC-NAME.

Returns a `nelisp-asm-block' (= label + bytes + relocations) suitable
for splicing into a `.text` section.

OPTS plist:
  :arch        x86_64-linux (default) | arm64-linux
  :exit-policy 'main-return-as-exit-code (default) | 'always-zero

The emitted block has label `_start' and is the ELF e_entry target."
  ...)
#+end_src

emit する instruction sequence (x86_64 SysV、~30 LOC):

#+begin_src text
_start:
    ;; Kernel hands us:
    ;;   rsp        → argc
    ;;   rsp+8      → argv[0..argc-1]
    ;;   rsp+8*(argc+1) → envp[]
    ;;   後ろに auxv (= ELF auxiliary vector)

    xor   rbp, rbp                ; ABI: clear frame pointer at entry
    mov   rdi, [rsp]              ; argc
    lea   rsi, [rsp + 8]          ; argv
    ;; envp is argv[argc+1] = [rsp + 8*(argc+1)]
    lea   rdx, [rsi + rdi*8 + 8]  ; envp

    and   rsp, -16                ; SysV: 16-byte align before call
    sub   rsp, 8                  ; (call pushes 8B, lands aligned)

    call  FUNC_NAME               ; relocation R_X86_64_PLT32

    ;; main returned in rax — pass to _exit
    mov   rdi, rax
    mov   rax, 60                 ; SYS_exit
    syscall
    ;; unreachable
    ud2
#+end_src

byte 数: ~30-40 bytes (= 命令長は ~3-7B 多くが平均 4B)。

** §2.3 =nelisp-crt0--unpack-stack=

System V AMD64 ABI が規定する process entry stack layout
(= /usr/include/sys/auxv.h + execve(2) man):

#+begin_src text
high address
+------------------------+
| environment strings    |  ← 実体 (envp[i] が指す)
+------------------------+
| argument strings       |  ← 実体 (argv[i] が指す)
+------------------------+
| AT_NULL                |
| auxv[]                 |  ← Elf64_auxv_t の配列
+------------------------+
| NULL                   |
| envp[]                 |
+------------------------+
| NULL                   |
| argv[argc-1..0]        |
+------------------------+
| argc                   |  ← rsp はここを指す
+------------------------+
low address
#+end_src

§2.2 の entry 自体は argc/argv/envp を register に load するだけで auxv
は触らない (= 第 4 引数 =rcx= に envp 末尾以降の pointer を渡せる
=AT_*= consumer は §2.6 main caller の責任)。

elisp 実装の outline (= ~40 LOC):

#+begin_src elisp
(defun nelisp-crt0--unpack-stack (arch)
  "Return assembler instructions that unpack argc/argv/envp/auxv.

Output is a list of `nelisp-asm-insn' for ARCH (x86_64-linux | arm64-linux).
Caller splices these directly after `_start:' label."
  (pcase arch
    ('x86_64-linux
     (list
       (nelisp-asm-insn 'xor 'rbp 'rbp)
       (nelisp-asm-insn 'mov 'rdi (nelisp-asm-mem 'rsp 0))     ; argc
       (nelisp-asm-insn 'lea 'rsi (nelisp-asm-mem 'rsp 8))     ; argv
       (nelisp-asm-insn 'lea 'rdx
         (nelisp-asm-mem-scaled 'rsi 'rdi 8 8))                ; envp = rsi + rdi*8 + 8
       (nelisp-asm-insn 'and 'rsp -16)                          ; align
       (nelisp-asm-insn 'sub 'rsp 8)))
    ('arm64-linux
     (list
       (nelisp-asm-insn 'mov 'fp 'xzr)
       (nelisp-asm-insn 'ldr 'x0 (nelisp-asm-mem 'sp 0))       ; argc
       (nelisp-asm-insn 'add 'x1 'sp 8)                        ; argv
       (nelisp-asm-insn 'add 'x2 'x1 (nelisp-asm-mem-scaled 'x0 8 8)) ; envp
       ;; arm64 SP must already be 16-aligned at entry per AAPCS — no
       ;; explicit align needed
       ))))
#+end_src

** §2.4 =nelisp-crt0--syscall-write FD BUF LEN=

syscall calling convention (x86_64 Linux):
- syscall number → =rax=
- args 1..6 → =rdi rsi rdx r10 r8 r9= (= 第 4 が =rcx= ではなく =r10=
  になる点が function call と違う)
- =syscall= instruction
- 戻り値 → =rax= (負の場合 =-errno=)

write(2) = syscall number 1。

#+begin_src elisp
(defun nelisp-crt0--syscall-write (fd buf-label len)
  "Emit `write(FD, BUF, LEN)' syscall.  BUF-LABEL is a relocation symbol
into `.rodata'."
  (list
    (nelisp-asm-insn 'mov 'rdi fd)
    (nelisp-asm-insn 'lea 'rsi (nelisp-asm-rip-rel buf-label))   ; PC-rel load
    (nelisp-asm-insn 'mov 'rdx len)
    (nelisp-asm-insn 'mov 'rax 1)                                ; SYS_write
    (nelisp-asm-insn 'syscall)))
#+end_src

5 命令、~15 LOC。

** §2.5 =nelisp-crt0--syscall-exit STATUS=

exit(2) = syscall number 60 (x86_64 Linux)、=_exit= は 231 (=
=exit_group=、Phase 47 ではどちらも acceptable、まずは 60)。

#+begin_src elisp
(defun nelisp-crt0--syscall-exit (status)
  "Emit `_exit(STATUS)' syscall.  STATUS is integer or `:rax' (= pass
through current rax)."
  (let ((insns (list)))
    (unless (eq status :rax)
      (push (nelisp-asm-insn 'mov 'rdi status) insns))
    ;; if :rax then caller already has the return value in rax → rdi
    (when (eq status :rax)
      (push (nelisp-asm-insn 'mov 'rdi 'rax) insns))
    (push (nelisp-asm-insn 'mov 'rax 60) insns)                  ; SYS_exit
    (push (nelisp-asm-insn 'syscall) insns)
    (push (nelisp-asm-insn 'ud2) insns)                          ; unreachable
    (nreverse insns)))
#+end_src

3-4 命令、~15 LOC。

** §2.6 =nelisp-crt0--call-main FUNC=

PLT-style relocation を使った関数 call。Doc 91 ELF writer が emit する
=R_X86_64_PLT32= relocation entry を Doc 93 linker が解決する。

(defun nelisp-crt0--call-main (func-name)

* §3 platform-specific invariants

** §3.1 x86_64 SysV ABI (Linux)

Process entry state (= execve 直後): =rsp= → argc on stack、他汎用 reg
は undefined、=rbp= は ABI 上 0 にすべき、DF clear。

Syscall ABI (= function call ABI と異なる、特に第 4 引数が =r10=):

| =rax= | syscall number |
| =rdi=, =rsi=, =rdx=, =r10=, =r8=, =r9= | arg 1-6 |
| =rcx=, =r11= | clobbered |

主要 syscall number: =read=0, =write=1, =open=2, =close=3, =mmap=9,
=exit=60, =exit_group=231.

Call alignment: =call= 直前で =rsp= が 16-byte aligned (= =AND -16= →
=SUB 8=)、関数 entry では =rsp+8= が 16-aligned (= return address 分)。

** §3.2 arm64 (AArch64) ABI (Linux)

Entry state: =sp= → argc、=x29= (fp) は ABI 上 0、=x0-x30= undefined。

Syscall ABI (AAPCS64): =w8= = syscall number (32-bit)、=x0-x5= = arg
1-6、=svc #0= で kernel 入。=x0= に戻り値。

主要 syscall number: =read=63, =write=64, =openat=56 (=open= は arm64
非存在), =close=57, =mmap=222, =exit=93, =exit_group=94.

Call alignment: =sp= は常に 16-byte aligned (= 関数 entry / call とも
に)、x86_64 のような pre-call adjust 不要。

** §3.3 macOS / Windows is OOS

macOS = Mach-O format + BSD/Mach syscall mixed numbering + =LC_MAIN=
entry convention、PE = =mainCRTStartup= → =__scrt_common_main_seh= で
MSVCRT 全依存 + Wow64 syscall 歴代非互換、共に Linux invariants と
一切共有しない。本 doc §2 は Linux 専用、macOS は "Doc 96 — Phase 48
candidate"、Windows は Phase 49+ candidate として defer。

* §4 testing strategy

** §4.1 minimum viable test

      (rod  (nelisp-asm-section :name '.rodata)))
  (nelisp-asm-emit-data rod "hi_msg" "hi")
  (nelisp-crt0-emit-entry "minimal_main"
    :arch 'x86_64-linux)
  (nelisp-asm-emit-fn asm "minimal_main"

    (nelisp-asm-insn 'mov 'rax 0)

  (nelisp-elf-write-binary "/tmp/hi"
    :sections (list asm rod)

#+end_src

* §6 risk / unknowns

** §6.1 libc 不在の影響

- =printf= 使えない → =write(2)= raw + 整数 → ASCII 変換を elisp で
  emit (= Doc 91 / 92 / 93 が独立に解決するが、crt0 が最初に必要に
  なる)
- =malloc= / =free= 使えない → Phase 47 の static-PIE binary は
  =mmap(2)= で heap 確保、free は省略 (= NeLisp は GC、free は GC が
  =munmap= で行う)
- =strerror= / =perror= 使えない → errno → string mapping を elisp
  emit (= optional、stderr に raw errno 数字でも debug 可能)

これらは crt0 の責務ではないが、§4.1 の minimum viable test を組む
時点で write の代替を持っていないと debug 不可能。crt0 SHIPPED 前に
=nelisp-crt0--syscall-write= だけは必須。

** §6.2 signal / setjmp / longjmp

- signal handler: NeLisp model に exception (= unwind) は無い、
  =condition-case= は eval evaluator 内で完結する。signal handler は
  Phase 47 では emit しない (= kernel default = SIGSEGV → coredump)
- setjmp / longjmp: GCC throw で使われるが NeLisp は使わない。Phase
  47 OOS

** §6.3 dynamic loader (=ld-linux-x86-64.so.2=)

static-PIE only (= Ehdr.e_type = ET_DYN, PT_INTERP なし)。Doc 91 ELF
writer が PT_INTERP を emit しない default なので crt0 側で考慮不要。

将来 dynamic linking が必要になった場合 (= =libc.so= / =libdl.so=
linked):
- =PT_INTERP= = =/lib64/ld-linux-x86-64.so.2= emit
- =DT_NEEDED= entries
- =.plt= / =.got.plt= section emit
- crt0 entry が =__libc_start_main= に jump する form に書き換え

= 別 Phase / 別 doc。

** §6.4 kernel ABI version drift

Linux kernel の execve stack layout は ABI として stable だが、稀に:
- =AT_RANDOM= (= ASLR 用 16-byte) の auxv entry 挿入位置
- =AT_PHDR= / =AT_PHENT= / =AT_PHNUM= 等の追加 (= LD_PRELOAD 系)
- =VDSO= = =linux-vdso.so.1= の auxv entry =AT_SYSINFO_EHDR=

がある。crt0 は **auxv を一切 read しない** ことで影響回避 (= Phase
47 の射程では =__vdso_clock_gettime= 高速化等は不要)。

CI matrix での test:
- Linux 5.x, 6.x (= mainline)
- musl-based distro (= alpine) で execve hand-off 確認
- rootless container (= unprivileged user namespace) での挙動

** §6.5 stack alignment + PIE invariants

x86_64 SysV は "function entry で =rsp+8= が 16-aligned" を要求。kernel
から =_start= 時点の =rsp= は 16-aligned とは限らないので §2.2 で
=AND rsp, -16= → =SUB rsp, 8= で確実化。これを忘れると main 内 SSE/AVX
命令 (=movaps= 等) が SIGSEGV する典型 trap。

PIE 側: Phase 47 は static-PIE only (= ET_DYN + PT_INTERP なし)、命令内
絶対 address 参照禁止、すべて RIP-relative。=call FUNC_NAME= は
=R_X86_64_PLT32= relocation で Doc 93 linker が解決。

** §6.6 top risk for future implementer

最大 risk = **stack alignment + auxv layout の kernel-version-specific
debug**。=AND -16= を忘れると SIGBUS / SIGSEGV が main 関数内で出て
crt0 ではなく main 側のバグに見える (= debug 1 日浪費の典型)。strace
+ gdb で =_start= 直後の =rsp= が 16-byte boundary かを最初に確認。

** §7.2 §94.b — arm64 Linux entry (SHIPPED)

*SHIPPED 2026-05-11* on branch =agent-doc-94b-impl=.

scope:
- =nelisp-crt0-arm64-entry-bytes= (= 20 byte =_start= stub, 5 inst)
- =nelisp-crt0-arm64-write-stdout-bytes= (= 20 byte inline =write(2)=)
- =nelisp-crt0-arm64-exit-bytes &optional STATUS-IMM= (= 8 or 12 byte
  inline =exit(2)=)
- =nelisp-crt0-arm64-emit-hello-world= (= ELF emit + =e_machine=
  in-place patch to EM_AARCH64 = 183)
- arm64 syscall number table inlined as named byte-string constants
  (=mov x8,#64= = SYS_write, =mov x8,#93= = SYS_exit)
- AAPCS64 alignment 検証: sp は kernel exec で 16-byte aligned に
  入る、entry stub は sp を一切いじらず x0 / x1 だけ書く

実装上の注意:
- =sf=1, opc=10= MOVZ 64-bit encoding を defun =nelisp-crt0-arm64--movz=
  に集約 (= cf. Arm ARM §C6.2)
- =bl= rel26 は 4-byte aligned 必須 (=:= encoder が error 出す)、未指定
  時は 0 placeholder (R_AARCH64_CALL26 で linker patch 可能)
- =adr Xd, +rel21= の immlo/immhi split (= bits 30..29 + bits 23..5) を
  defun =nelisp-crt0-arm64--adr= で正しく分配
- Doc 94 §3.2 が "MOVZ X8 #93 = 0xD2800B28" と書いていたが、これは実は
  =mov x8, #89= (= 0x59 << 5) であり SYS_exit ではない。本実装は正しく
  0xD2800BA8 を emit する。byte-pattern ert に同事実をコメント
- Doc 91 ELF writer の rich path は =:machine= が hard-coded
  =EM_X86_64= のため、emit 直後に file offset 0x12 の 2 byte を
  EM_AARCH64 (= 183 = 0xB7 0x00) に in-place patch (= 将来 ELF writer
  に =:machine= plumbing が入ったら redundant)

LOC: =lisp/nelisp-crt0-arm64.el= 245 (code) + 135 (docstring + cmt) =
380 行。Sub-goal +30 (= §7.4) には超過したが、§94.a と code 共有なし
+ ADR/MOVZ encoder を NeLisp 内製したため (= Doc 92 macro assembler
SHIPPED 後は collapse 可能)

工数: 1 day (= 2026-05-11)
gate: byte-pattern ert 36/37 pass + =readelf -h= が =Machine: AArch64=
を報告 + =file(1)= が =ARM aarch64= を tag。Live exec (= /tmp/hi が
"hello\n" 出力 + exit 0) は aarch64 host = qemu-aarch64 で確認待ち
(= skip-unless で gate)
