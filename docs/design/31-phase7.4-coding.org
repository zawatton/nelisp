** 2.1 supported encodings (MVP) [推奨 = A 4 encoding + CP932 拡張、user signoff 2026-04-25 で v2 改訂]

v2 改訂 (user signoff 2026-04-25 option B): Shift-JIS scope を v1 の
"strict JIS X 0208 only" から *CP932 (Windows-31J) 拡張部含む* に拡張。
日本語 Windows user の Shift-JIS は事実上 CP932 = Excel / file name /
Windows tool 連携に CP932 必須のため。

| 候補 | 戦略                                                   | 利点                                       |
|------+--------------------------------------------------------+--------------------------------------------|
| A    | UTF-8 / Latin-1 / Shift-JIS (CP932) / EUC-JP (default、4 種) | NeLisp 主用途 (英 / 西欧 / 日本 Windows-31J 含む) cover |
| B    | UTF-8 のみ                                             | scope 1 週間、日本語 file 既存資産 取り込め無い |
| C    | A + UTF-16 / GBK / Big5 / Korean                       | scope 1.5-2x 拡大、Phase 7.5 着手遅延 risk |

*推奨 A 理由 (v2 改訂)*:
- UTF-8 = JSON-RPC / 現代 web mandatory、Phase 7.5 MCP 経路必須
- Latin-1 = 単 byte trivial、ASCII fast path
- *Shift-JIS (CP932 拡張部含む)* = NeLisp 日本 Windows user の事実上の
  Shift-JIS、CP932 (Windows-31J) は JIS X 0208 strict + Microsoft 拡張
  (NEC 特殊文字 / IBM 拡張) を含み、Windows 環境の Excel / file name /
  legacy 日本語 Windows tool 連携に必須
- EUC-JP = JIS X 0208 + JIS X 0212 + 単 byte ASCII、Unix legacy 日本語 file
- UTF-16 / GBK / Big5 / Korean = v2.0 で =defcustom= 経由 opt-in 化

*Shift-JIS scope detail (v2 LOCKED)*:
- *MVP 含む*: ASCII + JIS X 0201 kana + JIS X 0208 + CP932 拡張
  (NEC 特殊文字 ~83 字 + IBM 拡張 ~388 字 + NEC 選定 IBM 拡張 ~374 字 =
  Microsoft 公式 CP932 mapping)
- *MVP out*: JIS X 0212 (= Shift-JIS では encode 不可、EUC-JP のみ対応)
- *table size*: ~7000 + 845 拡張 = ~7845 entry (CP932 包含で ~12% 増)
- *generator script*: =tools/coding-table-gen.el= = Unicode UCD の
  Microsoft CP932 mapping (=public domain `WindowsBestFit932.txt`= or
  =Unicode CP932.TXT=) を入力、source-of-truth は Microsoft 公式 mapping

** 2.4 error handling [推奨 = A 3 strategy contract LOCK (v2 改訂)]

v2 改訂 (codex 観点 3.7 反映): 当初は 3 strategy 名のみで実質的 contract
が不明だったため、各 strategy の return value / signal type / byte
offset / streaming atomicity を本 v2 で *contract LOCK*。

| strategy   | return on invalid                | signal raised          | offset reporting | streaming atomicity |
|------------+----------------------------------+------------------------+------------------+---------------------|
| =replace=  | partial result string (replacement char inserted) | none (returns normally) | last replaced byte offset in =:invalid-positions= plist | invalid byte 検出時 chunk boundary 維持、内部 buffer flush |
| =error=    | signal、partial result discarded | =nelisp-coding-invalid-byte= condition (data = invalid byte offset + strategy) | signal data alist =(:offset N :byte B :strategy 'error)= | streaming 中 signal 時 internal state は consumer 復旧可能形 |
| =strict=   | signal、abort process            | =nelisp-coding-strict-violation= condition (uncatchable in streaming) | signal data + immediate process abort | streaming 即終了、buffer 状態 undefined |

*replace strategy detail (default)*:
- invalid byte → =U+FFFD REPLACEMENT CHARACTER= (UTF-8 spec)
- 連続 invalid byte は 1 個の =U+FFFD= に collapse (= per WHATWG Encoding Standard)
- =:invalid-positions= plist field で全 replacement 位置を返す

*error strategy detail*:
- API caller が =condition-case= で catch 可能、resume 不能
- streaming 中で signal 時、internal =coding-state= struct は pre-error
  位置で停止、=resume-coding= primitive (Phase 7.5 で実装) で続行可能

*strict strategy detail*:
- batch script 用、signal は uncatchable (= NeLisp =error= signal の
  =:noresume= flag)、process exit code 1
- streaming 中 invalid 検出で即 process abort、buffer state undefined

*default*: =:replace= (Emacs =coding-system-error-handler= 風 fail-safe)、
=defcustom nelisp-coding-error-strategy= で global override 可。

*v2 contract LOCK 根拠*: codex review 観点 3.7 で「=:error= と =:strict=
の差不明」「partial output/byte offset/atomicity 不明」指摘を上記 4 列
table で全 contract 化、silent corruption / API break risk を解消。

** 2.5 streaming API [推奨 = A chunk-based callback]

| 候補 | 戦略                                                                | 利点                                       |

| A    | chunk-based callback (=nelisp-coding-decode-stream=) (default)      | Emacs =accept-process-output= 互換、large file OK |

  (=nelisp-coding--state= struct = pending byte buffer + decoder state)

** 3.1 Phase 7.4.1 — UTF-8 encode/decode + BOM handling

新規 module: =src/nelisp-coding-utf8.el= (~400-700 LOC NeLisp)。

*** scope

- BOM (=EF BB BF=) strip on read / no emit on write
- error handling 3 strategy (=replace= / =error= / =strict=)、default =replace=
- =U+FFFD REPLACEMENT CHARACTER= emit on invalid byte (=replace= 時)
- emoji / BMP 外 (=U+10000-U+10FFFF=) supplementary plane support
- Phase 7.1 native JIT 前提の hot path (=declare (inline t)=)

*** ERT smoke (+10)
- ASCII (1-byte) encode/decode round-trip
- Latin extended (2-byte) encode/decode
- 日本語 BMP (3-byte) encode/decode (=あ= = =E3 81 82=)
- emoji (4-byte) encode/decode (=🎉= = =F0 9F 8E 89=)
- BOM strip on read (=EF BB BF= prefix → 除外)
- BOM no-emit on write (default)

** 3.2 Phase 7.4.2 — Latin-1 encode/decode (single-byte trivial)

新規 module: =src/nelisp-coding-latin1.el= (~200-400 LOC NeLisp)。

*** scope

- single-byte trivial (table 不要、direct cast)
- BOM 処理 = なし (Latin-1 spec)
- error handling = encode 時 =U+0100+= で =error= or =replace=、decode 時
  invalid byte なし (256 値全て valid)
- ASCII fast path (=U+0000-U+007F= は UTF-8 と互換)

*** ERT smoke (+6)
- ASCII round-trip (Latin-1 / UTF-8 互換 confirm)
- Latin-1 拡張 (=U+0080-U+00FF=) round-trip

- 全 256 byte decode invertibility (data integrity)
- round-trip 100MB Latin-1 file (Phase 7.1 native JIT で 300MB/sec target、

*** scope

*** ERT smoke (+12)

* §6 Risks (6 項目)

** 6.1 BOM emit policy (Windows tool 互換 risk)

§2.3 推奨 A で UTF-8 BOM = read で strip / write で no-emit を default
としたが、Windows-only tool (Notepad 旧版 / Excel CSV) は BOM 期待で
char 化けする known issue。

*影響*: Phase 7.5 user (Windows、Excel CSV) で =./nelisp-runtime= が
書出した CSV を Excel で開いて文字化け。

*mitigation*:
- =--coding-bom-emit= flag (CLI、startup option) で emit 強制可
- =nelisp-write-file :bom-emit t= argument で per-file customize
- Phase 7.5 release note に Windows compatibility section 明記

** 6.2 Latin-1 encode 時 =U+0100+= 範囲の placeholder

§2.4 + §3.2 で =replace= strategy = =U+0100+= → =3F= (=?=) emit を
default としたが、情報損失で user friendly でない可能性。

*影響*: Latin-1 encode 時 emoji や日本語入力で全部 =?= に化け、debug
困難。

*mitigation*:
- =:encoding-error-replacement= 引数で customize 可
  (例 =:encoding-error-replacement (lambda (codepoint) (format "&#%d;"
  codepoint))= で HTML numeric reference)
- default =replace= は =3F= (Emacs precedent) 維持、=error= strategy
  で fail-fast 推奨を doc 化

** 6.3 Shift-JIS / EUC-JP table data の source LOC 増

§3.3 で Shift-JIS / EUC-JP 各 ~7000 entry の bidirectional table を
source 内 alist / vector で持つことで NeLisp purity 重視。ただし source
LOC 増 (~50 LOC ではなく実際は ~14000 entry × 2 = 28000 行 risk)。

*影響*: =src/nelisp-coding-tables.el= が異常に長くなり review / load 時間悪化。

*mitigation*:
- table 形式 = vector (codepoint indexed、char → byte 単方向、O(1)
  lookup)
- inverse direction (byte → char) は hash-table backed (load 時 build、
  source LOC 不要)
- LOC 計上は table file ~50 LOC (vector literal は format 1 行で長く
  書ける) 維持、実 byte 数 ~100KB 程度

** 6.4 table lookup overhead (UTF-8 比 2x 遅い)

§3.3 で Shift-JIS / EUC-JP target 100MB/sec、UTF-8 200MB/sec の半分。
table lookup hash overhead が原因。

*影響*: Phase 7.5 で日本語 file 処理が UTF-8 比 2x 遅い、user
perception 悪化。

*mitigation*:
- hash-table backed lookup + Phase 7.1 inline expansion
  (=declare (inline t)=) で hash op を JIT
- vector backed (codepoint indexed direct lookup) で hash op 1 個削減
- ベンチ未達なら Phase 7.5 で =defcustom nelisp-coding-japanese-fast-mode=
  + binary table file (.dat) を opt-in 化

** 6.5 chunk 越境 state 持ち越し bug (silent data corruption)

§3.4 で multi-byte char chunk 越境 state 持ち越し (=pending byte buffer
+ decoder state=) を実装、bug = silent data corruption (検出困難)。

*影響*: 1GB file decode で 1 char 化け = file 全体 garbage、user 信頼失う。

*mitigation*:
- ERT で random chunk size (1〜64KB) × multi-byte 全境界位置 (UTF-8
  4 byte char の 1/2/3/4 byte 目で chunk break) を 100K iter
- =nelisp-coding-decode-stream= 内部 invariant assertion (state buffer
  size ≤ 4 byte for UTF-8、≤ 2 byte for Shift-JIS / EUC-JP)
- Phase 7.5 soak test で 1GB file × 4 encoding 全 round-trip

** 6.6 anvil leverage 過大評価 (Doc 28 §6.6 / Doc 29 §6.6 同 pattern)

Doc 27 で SBCL 25 年 → 2-3 週間の見積。本 doc 7.4 で 2-3 週間に維持
したが、なお下振れ risk あり。Doc 28 §6.6 / Doc 29 §6.6 の week 1/2
replan gate 同 pattern で *2 replan gate* を導入。

*影響*: Phase 7.5 (integration) の着手遅延 = Doc 27 全体 STATE 遅延。

*mitigation (Doc 28 §6.6 v2 / Doc 29 §6.6 と同 pattern)*:

確定。

pair precedent 踏襲、Phase 7.5 で正式 freeze。

** 9.4 streaming API の interface (callback vs generator)

§2.5 で =accept-process-output= 互換 callback を default としたが、
NeLisp に generator primitive (Python yield 風) が将来 phase で導入
される可能性。

*default*: 本 phase は callback 1 択、generator-based interface は v2.0
で =cl-loop= / Python yield 風 primitive 追加時に検討。

** 9.5 Shift-JIS / EUC-JP の方言 (CP932 / Windows-31J 等)

§2.1 で Shift-JIS = JIS X 0208 strict、Windows 拡張 (CP932 = JIS X
0208 + 機種依存文字) は本 phase scope 外。

*default*: Phase 7.4 は JIS X 0208 strict、CP932 は v2.0 で
=defcustom nelisp-coding-shift-jis-cp932-mode= 経由 opt-in 化。
