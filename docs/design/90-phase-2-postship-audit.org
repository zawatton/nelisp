* §3 §86.3.c specific deferral articulation

§86.3.c は Phase 2 最大 sub-stage で v2 reconciled -1,214 target に対し
ship 実績 -396 (= -818 short)。Doc 89 §10.1 が定めた hot-path policy
を ship agent が遵守した結果、4 個の deferral が発生した。本節は各

** §3.1 deferral 1: env.rs 19 method body un-reduced (-176 expected)

Doc 89 §4.3.1 Step 3 が「19 method を thin slim-primitive forwarder
化、env.rs 935 → ~50 LOC stub = -885」を spec していた。actual = env.rs
935 → 597 (= -338) で stop、19 method body は **そのまま残置**。

ship agent (= Doc 86 §12.9) の根拠:
- 各 =Env::*= method の body は既に 3-24 LOC のタイト実装
- slim primitive と同じ =Env::globals= HashMap を共有 (= storage は
  single source of truth)
- thin forwarder 化するには slim primitive の =(args: &[Sexp], env:
  &mut Env)= signature を refactor 必要 = 侵襲的、switch-over 目的
  と直交

Doc 89 §10.1 hot-path policy 適用判断: =env.lookup_value= は eval
ループ内で全 symbol resolve で hit する。Rust =Env::*= method を
slim primitive 経由 (= 同 HashMap への 2 段 indirection) にすると、
eval loop の hot path で indirect call cost を払う = ~10-50x slowdown
risk。ship agent は **Rust internal callsite を Rust =Env= method
直呼び出しに維持**することで policy 遵守。

代わりに shadow-verify retirement (= -338 of -514 target) で env.rs
内部を slim な形に整理 = §86.3.b で +210 cfg-gated として上乗せして
いた verify infrastructure (= 7 helper + frame_stack predicate +
shadow_sexp_eq) を §86.3.c で full retire = -338 inside file。

なぜ正しいか: Doc 89 §10.1 が明示するように、shim 経由は user-visible
API + advice hook 用、Rust internal callsite は Rust =Env= method
直呼び出し維持が canonical。19 method body 残置は policy compliant。

Phase 47 unlock 条件: Doc 85 spike が完了し native compiler が elisp
round-trip を free (= zero-overhead) にできた時点で、19 method を
elisp =nelisp--env-globals-*= 直呼びに switch 可能。Sexp DSL で global
hash-table 表現を elisp 化できれば storage 側も Rust 不要。expected
recover: ~-176 LOC (= 597 → 597 − 176 = ~421) または更に深く ~-547
(= ~50 LOC stub 着地、Doc 89 §4.4 完遂値)。

** §3.2 deferral 2: parse_let_binding hot-path preserved (-30 expected)

Doc 89 §5.2 が「=parse_let_binding= 30 LOC を elisp =nelisp--parse-
let-binding= に push、Rust 側薄い shim ~10 LOC」を spec。actual =
elisp shim defun は exported (= =lisp/nelisp-stdlib-env-shim.el=) が、
Rust =parse_let_binding= は **そのまま残置** (= callsite から funcall
されない)。

ship agent (= Doc 86 §12.9) の根拠:
- 全 =let= / =let*= 評価で binding ごとに elisp dispatch round-trip が
  発生 = ~50x slowdown estimate (Doc 89 §10.1)
- Rust =parse_let_binding= は 30 LOC including comments で yield 小
- Phase 3 D-B 時に elisp shim を canonical 化する用意は完了

Doc 89 §10.1 hot-path policy 適用判断: =let= / =let*= は eval ループ
内で頻繁に hit する hot path。binding parse は 1 binding あたり ~10
LOC の処理で、毎 evaluation ごとに elisp dispatch を介すと ~50x
overhead 発生 risk。

なぜ正しいか: =parse_let_binding= の semantic は inline 30 LOC で
完結し、複雑度低 = Rust 側 maintain cost も低い。一方 elisp 化の
overhead cost は eval ループ全体に prorated される = ROI 著しく低い。
ship agent は Rust 内 hot path を維持し、elisp shim を Phase 3 D-B
向け canonical 化 unlock として配置 = policy compliant。

Phase 47 unlock 条件: Phase 3 D-B (= Doc 77c master) が ship し native
compiler 経由で elisp =parse-let-binding= が Rust 同等 perf で動く時点。
expected recover: -30 LOC (= special_forms.rs から =parse_let_binding=
削除)。

- §86.1.f (= drift +297 / +Rust 増)

特徴: Tier 1/2 builtin を =bi_*= helper から JIT trampoline 経由 elisp
wrapper に migrate する場合、trampoline body は =bi_*= body と **構造的
isomorphic** になる (= 同 Sexp variant → tag-flip logic / 同 primitive
operation 呼び出し)。理由は Sexp 変換 / IEEE-754 / =Rc::ptr_eq= 等の
underlying primitive 自体が Rust-resident な fact = 言語側 (Rust) の
expressibility に locked。

migration の **conceptual win** は dispatch + arity + type-error shape を
elisp 化 = =advice-add= / =fset= override / user-visible signal contract
が elisp ergonomics に乗る。ただし Rust LOC delta は body isomorphism
で構造的に limited (= ±~50-200 LOC per arm)。

ROI 改善 condition: Phase 47 (= Doc 85 spike) で **Sexp DSL** が
trampoline body を elisp で表現可能にする = =Rc::ptr_eq= / IEEE-754 /
=DefaultHasher= state machine 等の primitive を Sexp DSL 経由で elisp
記述 = trampoline body 自体が elisp 化可能。

** §4.2 (b) Hot-path policy preservation (= Doc 89 §10.1)

該当 sub-stage:
- §86.2 残 -145 deferred (= eval ループ Result type architectural cost)
- §86.3.c 4 deferral (§3 articulated、合計 -436 ~ -807 expected recover)

特徴: eval ループ内 hot path (= =lookup_value= / =bind_formals= /
=parse_let_binding= / Tier 3 dispatch / Result type) を elisp 化すると
~10-50x slowdown が apply chain 全体に loaded する。Doc 89 §10.1 が
explicit policy として established、ship agent が遵守。

trade-off: Phase 2 単独削減量 vs Phase 3 D-B 完了後の native compiler
free-overhead unlock。Phase 2 で push しても perf regression revert を
要する risk = 遅らせる方が architecturally clean。

ROI 改善 condition: Phase 3 D-B (= Doc 77c master plan) 完了で native
compiler が elisp dispatch を free にする時点 = hot-path policy
restriction が外れる。
