** §2.1 port 範囲 (MVP scope)

- A. 全 6 public API + 6 deftool を一気に port
- *B. MVP = rebuild + search 2 本のみ、残は Phase 6.5.2-3 で個別判断 ← RECOMMENDED (handoff §3.3)*
- C. 全 API + tool だけ MVP (実装は spy stub)

推奨理由: handoff §1.1 必達条件 = 本日中 SHIPPED。MVP 縦串で動作確認、
残 4 API は Phase 6.5.2-3 stretch。MVP として最小有用 set (project ナビ
ゲートの本丸 = name → file:line) が rebuild + search の 2 本。

** §2.2 DB path

- A. =user-emacs-directory= 追従 (anvil-defs と同じ =~/.emacs.d/...=)
- *B. =~/.cache/nelisp/nelisp-defs.db= (XDG cache、Emacs 非依存) ← RECOMMENDED*
- C. env var 必須 (explicit)

推奨理由: nelisp-state と同方針 (Doc 17 §2.2 B)。Stage D 配布 binary
からも使うため =user-emacs-directory= 依存禁止。XDG cache は Linux/macOS
共通、=NELISP_CACHE_DIR= で override 可。

** §2.3 schema 互換性

- A. anvil-defs schema v1 と完全同一 (table 名 / column 名 / DDL)
- *B. 同一 + schema_meta version 1 で固定、migration は drop-and-rebuild ← RECOMMENDED*
- C. 独自 schema

推奨理由: anvil-defs MVP rebuild の cost ~数秒、migration 不要。
schema mismatch 時は drop-and-recreate が最小実装。

** §2.4 defcustom 扱い

- A. defcustom のまま port (NeLisp customize 未実装でも host で動く)
- *B. defvar 化 (anvil-defs-paths / anvil-defs-exclude-patterns / DB path) ← RECOMMENDED*
- C. defconst (運用変更不可)

推奨理由: nelisp-state / nelisp-http と同方針 (Phase 5-A.2 / Doc 24 §2.9)。
host 動作と self-host 動作の差を作らない。

** §2.5 with-transaction macro

- A. NeLisp defmacro で port (anvil-defs と同じ defmacro)
- *B. nelisp-state--with-transaction の private 形を inline 再定義 ← RECOMMENDED*
- C. 関数化 (callback)

推奨理由: nelisp-state 側の private macro を re-export する手間より、
inline 再定義 (~10 LOC) が独立性高い。anvil-defs と同形。

** §2.6 scanner 戦略

- A. anvil-sexp.el を含めて port (~600 LOC)
- *B. inline simple scanner: insert-file-contents + read を loop、top-level
  form の car が defun/defvar/defcustom/cl-defun/defmacro/defconst/
  define-* なら index に投入 ← RECOMMENDED*
- C. host elisp-mode でパース (font-lock 依存)

推奨理由: anvil-sexp は Doc 12 で 600 LOC、現時点 NeLisp 側 port 価値なし。
MVP は top-level の car symbol だけ見れば ~80% useful。引数アリティや
docstring 抽出は Phase 6.5.2 で精緻化。

** §2.7 ERT scope

- A. anvil-defs-test 全 port (~30 ERT、anvil-sexp 前提含む)
- *B. MVP scope の 8-10 本 (rebuild → row count、search known / missing /
  kind filter、schema PRAGMA table_info、enable/disable 冪等性、
  exclude pattern) ← RECOMMENDED*
- C. integration smoke のみ (ERT 0)

推奨理由: handoff §1.1 必達 = ERT 8-10 本。MVP 表面の往復 (insert →
search → row 取得 → 復元) が回ることを保証。Phase 6.5.2 で参照解析
ERT を追加。

** §2.8 MCP tool

- A. 全 6 tool 一気に expose (deftool 全部書いて未実装は stub)
- *B. MVP = defs-index-rebuild + defs-search の 2 本のみ、smoke +2 assert ← RECOMMENDED (handoff §3.3)*
- C. tool 0 本、Phase 6.5.2 で expose

推奨理由: handoff §1.1 必達条件 4 = 「最低 2 deftool」。MVP スコープと
align。実装無いなら expose しない方針 (anvil-defs と同じ "実装済の API
だけ tool 化")。
