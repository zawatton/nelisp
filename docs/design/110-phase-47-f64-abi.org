* §0 TL;DR

** §0.1 What §100.D left unfinished

§100.D Stage 2/3 (= commit ae4bb08a, 2026-05-12) shipped 12
=nl_jit_arith_*= trampolines as Phase 47-compiled =.o= files and
*deleted* =build-tool/src/jit/arith.rs= (= −181 LOC) on three target
triples (linux-x86_64, linux-aarch64, macos-aarch64).

But §100.D path is =extern "C" fn(i64, i64) -> i64= only — every
trampoline argument and return rides general-purpose registers (rdi,
rsi, rax on x86_64; x0, x1, x0 on aarch64).  The f64-shaped
trampolines in =jit/float.rs= (8 fns) + =jit/math.rs= (3 fns) cannot
be swapped under §100.D's ABI because:

1. Their arguments arrive in xmm0/xmm1 (x86_64) or d0/d1 (aarch64),
   not in rdi/rsi or x0/x1.  Phase 47's =arg-regs= constant
   (=lisp/nelisp-phase47-compiler.el= line 121) is GP-only.
2. f64 arithmetic uses SSE2 / NEON instructions that Phase 47's
   =asm-x86_64.el= + =asm-arm64.el= helpers do not encode.  See
   =lisp/nelisp-asm-x86_64.el:103= which explicitly states "Covers
   only integer GPRs — xmm/ymm/zmm extension is out of scope".
3. The mixed-return shapes (= =nl_jit_float_eq_eps= takes f64 args
   but returns i64 in rax) require coordinated register class
   handling within a single function body.

** §0.2 The f64 ABI gap

Three architectural directions exist:

*Option A (extern-call only)*: keep f64 trampolines in Rust, but
expose them via Phase 47's existing =extern-call= grammar.  Elisp
=.o= cannot *be* the trampoline; it can only call into the Rust
one.  Net Rust delta: 0 (= trampoline stays).  *Rejected* — defeats
the purpose.

*Option B (full f64 register class)*: teach Phase 47 about xmm /
NEON register class.  Add f64 arg / return ABI, f64 arithmetic emit,
f64 compare emit, f64 → i64 / i64 → f64 cast.  Elisp =.o= becomes
the trampoline directly.  Net Rust delta: −49 LOC after §110.E + F.
*Chosen*.

*Option C (mixed: GP-stash, SSE in epilogue)*: smuggle f64 values
through GP registers (= rax holds the i64 bit-pattern of the f64,
then movq rax, xmm0 just before return).  This avoids most asm
encoding work but produces obscure code paths Phase 47 readers will
hate, and breaks for any f64 arithmetic intermediate (= you can't
ADDSD across a GP value).  *Rejected* — would only work for
=nl_jit_float_float= (= identity) and nothing else.

Doc 110 follows Doc 100 §3 stage breakdown methodology:

|--------+---------------------------------------------+-----------------+---------------|

** §0.4 Scope of Doc 110

In scope:
- =extern "C" fn(f64, f64) -> f64=                  (= 4 ops: add/sub/mul/div in float.rs)
- =extern "C" fn(f64, f64) -> i64=                  (= 4 ops: eq-eps/lt/gt/le/ge in float.rs)
- =extern "C" fn(f64) -> f64=                       (= 3 ops in math.rs)

* §1 Background

** §1.1 Why elisp `.o' cannot use f64 today

Phase 47's emit pipeline picks ABI registers from a single GP-only
table at =lisp/nelisp-phase47-compiler.el= line 121:

#+begin_src elisp
  (defconst nelisp-phase47-compiler--arg-regs
    '((x86_64 . (rdi rsi rdx rcx r8 r9))
      (aarch64 . (x0 x1 x2 x3 x4 x5 x6 x7))))
#+end_src

When the compiler emits =(defun fn (a b) BODY)= it walks =arg-regs=
in order, binding parameter =a= to =rdi= and =b= to =rsi= on
x86_64.  There is no f64-aware variant — the parameter type is
inferred from grammar use (= =(+ a b)= treats =a=, =b= as i64), and
the register pool offers no xmm members.

Even if the compiler were taught to emit MOVSD instead of MOV, the
arg binding step would still hand back =rdi=, and =MOVSD xmm0,
xmm_rdi= is meaningless (= no such instruction exists; the encoding
demands xmm0..15 source).

** §1.2 The SysV AMD64 f64 ABI

For the trampoline signatures in =jit/float.rs= + =jit/math.rs= on
linux-x86_64, the relevant SysV AMD64 ABI rules are:

| Element                      | Register   | Notes                          |
|------------------------------+------------+--------------------------------|
| 1st-8th f64 argument         | xmm0..xmm7 | Caller-saved (= scratch)       |
| f64 return                   | xmm0       | Same register class as args    |
| i64 return (mixed shape)     | rax        | f64 args still in xmm0/xmm1    |
| Stack alignment at call      | 16-byte    | Per ABI                        |

For =extern "C" fn(f64, f64) -> f64= specifically:
- =a= arrives in xmm0
- =b= arrives in xmm1
- result returns in xmm0

For =extern "C" fn(f64, f64) -> i64= (= comparison shape):
- =a= in xmm0, =b= in xmm1
- result returns in rax (= GP register)
- No interaction between xmm and rax inputs — the function can do
  UCOMISD xmm0, xmm1 then SETcc al + MOVZX rax, al.

For =extern "C" fn(f64) -> f64= (= unary shape):
- =x= in xmm0, result in xmm0

** §1.3 The aarch64 AAPCS f64 ABI

aarch64 vector / FP register ABI rules:

| Element                      | Register | Notes                              |
|------------------------------+----------+------------------------------------|
| 1st-8th f64 argument         | v0..v7   | Lower 64 bits = "d0..d7"           |
| f64 return                   | v0 (d0)  |                                    |
| i64 return (mixed shape)     | x0       | f64 args still in d0/d1            |

The =vN= register is a 128-bit SIMD/FP register; for scalar f64 we
address its lower 64-bit slice as =dN=.  AArch64 encodes scalar f64
arithmetic via the SIMD&FP instruction subset (= FADD/FSUB/FMUL/FDIV
with the =.D= operand suffix in the canonical mnemonic, but the
machine encoding identifies double via the =ftype= field).

** §1.4 What 49 LOC are at stake

The swap targets in detail:

*=jit/float.rs= (= 20 LOC body, 11 LOC #[no_mangle] + comments)*

#+begin_src rust
  pub extern "C" fn nl_jit_float_add(a: f64, b: f64) -> f64 { a + b }
  pub extern "C" fn nl_jit_float_sub(a: f64, b: f64) -> f64 { a - b }
  pub extern "C" fn nl_jit_float_mul(a: f64, b: f64) -> f64 { a * b }
  pub extern "C" fn nl_jit_float_div(a: f64, b: f64) -> f64 { a / b }
  pub extern "C" fn nl_jit_float_eq_eps(a: f64, b: f64) -> i64 {
      if (a - b).abs() < 1e-15 { 1 } else { 0 }
  }
  pub extern "C" fn nl_jit_float_lt(a: f64, b: f64) -> i64 { (a < b) as i64 }
  pub extern "C" fn nl_jit_float_gt(a: f64, b: f64) -> i64 { (a > b) as i64 }
  pub extern "C" fn nl_jit_float_le(a: f64, b: f64) -> i64 { (a <= b) as i64 }
  pub extern "C" fn nl_jit_float_ge(a: f64, b: f64) -> i64 { (a >= b) as i64 }
#+end_src

Pure compute: no allocation, no refcount, no clone, no syscall.  All
8 fit Phase 47 swap rules once f64 ABI lands.

*=jit/math.rs= (= 29 LOC body)*

#+begin_src rust
  pub extern "C" fn nl_jit_float_float(x: f64) -> f64 { x }
  pub extern "C" fn nl_jit_float_exp(x: f64) -> f64 { x.exp() }
  pub extern "C" fn nl_jit_float_log(x: f64) -> f64 { x.ln() }
#+end_src

=nl_jit_float_float= is identity — trivially swappable.
=nl_jit_float_exp= / =nl_jit_float_log= delegate to Rust libm via
=f64::exp= / =f64::ln=.  Swap requires Phase 47 =extern-call=
grammar to support f64 args + f64 return (= §100.A extern-call is
i64-only; §110.F extends it).  The libm symbols (=exp=, =log=) are
already linked into the binary via std's libm dependency, so the
=.o= can name them directly.

* §2 ABI design

** §2.1 Register class model

Add to =lisp/nelisp-phase47-compiler.el=:

#+begin_src elisp
  (defconst nelisp-phase47-compiler--reg-class-alist
    '((x86_64
       (gp  . (rdi rsi rdx rcx r8 r9))
       (xmm . (xmm0 xmm1 xmm2 xmm3 xmm4 xmm5 xmm6 xmm7)))
      (aarch64
       (gp . (x0 x1 x2 x3 x4 x5 x6 x7))
       (fp . (d0 d1 d2 d3 d4 d5 d6 d7))))
    "Register class → ordered argument register list per arch.
  CLASS is `gp' for integer / pointer args, `xmm' (x86_64) or `fp'
  (aarch64) for f64 args.  Position N within a class names that
  class's Nth argument; classes are independent so a function with
  signature `(int, f64, int)' uses (rdi, xmm0, rsi) on x86_64
  (= per SysV AMD64 §3.2.3).")
#+end_src

The existing =arg-regs= constant is retained as a thin alias for
the =gp= class (= avoids touching i64-only call sites).  New f64
emit goes through the class-aware lookup.

** §2.2 Phase 47 grammar additions

Add 7 new value / IR forms to =nelisp-phase47-compiler--parse-value=
dispatch:

*1. =(f64 LITERAL-OR-VAR)= — value form*

Materialize a value as f64.  LITERAL-OR-VAR is either:
- An elisp =float= literal (= =1e-15= etc.) — embedded in rodata,
  loaded via =movsd xmm0, [rip + offset]=.
- A parameter symbol already bound to an xmm register — passthrough.

IR: =(:kind f64 :form 'literal :value FLOAT)= or
=(:kind f64 :form 'param :reg SYM)=.

*2-5. =(f64-add A B)=, =(f64-sub A B)=, =(f64-mul A B)=, =(f64-div A B)=*

Standard f64 arithmetic.  IR: =(:kind f64-binop :op ADD|SUB|MUL|DIV
:lhs IR :rhs IR)=.

Emit (x86_64):
#+begin_src
  ; lhs in xmm0, rhs in xmm1 (per usual 2-input convention)
  ADDSD xmm0, xmm1    ; (or SUBSD/MULSD/DIVSD)
  ; result in xmm0
#+end_src

Emit (aarch64):
#+begin_src
  ; lhs in d0, rhs in d1
  FADD d0, d0, d1     ; (or FSUB/FMUL/FDIV)
  ; result in d0
#+end_src

*6. =(f64-cmp A B :op LT|GT|LE|GE|EQ-EPS)= → i64 0/1*

Compare two f64 values, materialize result as i64 0/1 in rax (x86_64)
or x0 (aarch64).

IR: =(:kind f64-cmp :op OP :lhs IR :rhs IR)=.

Emit (x86_64):
#+begin_src
  ; lhs in xmm0, rhs in xmm1
  UCOMISD xmm0, xmm1   ; sets ZF/PF/CF per IEEE 754 unordered-aware compare
  SETcc al             ; cc per :op (= setb / seta / setbe / setae / sete)
  MOVZX rax, al        ; zero-extend to i64
#+end_src

For =:EQ-EPS= specifically: emit a small inline sequence
=SUBSD xmm0, xmm1; ANDPS xmm0, [abs-mask]; COMISD xmm0, [eps-1e-15];=
=SETB al; MOVZX rax, al= where =abs-mask= is the 64-bit mask
=0x7FFF_FFFF_FFFF_FFFF= and =eps-1e-15= is =1e-15= as f64, both in
rodata.

Emit (aarch64):
#+begin_src
  ; lhs in d0, rhs in d1
  FCMP d0, d1          ; sets NZCV flags
  CSET x0, cc          ; condition code per :op
#+end_src

*7. =(f64-call EXTERN-NAME ARG)= — unary libm shape*

Call an externally-linked f64 → f64 function (= libm =exp=, =log=).

IR: =(:kind f64-call :name STRING :arg IR)=.

Emit (x86_64): direct =CALL exp= via PLT32 relocation (= reuses
§100.A SHN_UNDEF / R_X86_64_PLT32 wiring).  =xmm0= holds the arg on
call; SysV ABI guarantees =xmm0= holds return on resume.  Local
xmm spill before the call only required if intermediate f64
results are live across the call (= rare for the 3-LOC libm cases).

Emit (aarch64): =BL exp= via R_AARCH64_CALL26 relocation (= same as
§100.D ARM64 path).  =d0= holds arg / return.

** §2.3 Mixed-return ABI (= float-in, int-out)

For =nl_jit_float_eq_eps= / =lt/gt/le/ge=: the elisp =(defun)= body
ends with a =(f64-cmp ...)= form whose IR materializes the result in
rax (x86_64) or x0 (aarch64).  The function epilogue then returns
from rax/x0 (= unchanged from §100.D arith.rs return convention).

No additional ABI work needed beyond §2.2.6 — the f64-cmp emit
already terminates in a GP register, which the existing epilogue
already returns.

** §2.4 Stack alignment + spill policy

§110.A-F do *not* introduce xmm spill (= the 8 float.rs + 3 math.rs
trampolines are all ≤ 2-arg, ≤ 1-temp, fitting entirely in
xmm0/xmm1 without spilling).  Spill is deferred to a future Doc 112
when complex f64 expressions arrive.

Stack alignment at call (= 16-byte boundary) is preserved by the
existing Phase 47 prologue (= already enforces 16-byte alignment for
GP-only code).  =CALL exp= within =(f64-call)= works without
additional pad insertion.

** §2.5 ELF / Mach-O / linker mechanics

Reuses §100.A + §100.D infrastructure:
- ELF: =SHN_UNDEF= entry + =R_X86_64_PLT32= relocation for libm
  externs.  Phase 47's existing relocation emit handles the
  encoding once symbol name + addend are filled in.
- Mach-O 64: =N_UNDF= entry + =X86_64_RELOC_BRANCH= for libm
  externs.  Phase 47's §100.D Mach-O writer covers this.
- AArch64: =R_AARCH64_CALL26= for BL targets.

The libm symbol names (=exp=, =log=) link against the C library
that =bin/nelisp= already loads — no new dependency.

** §2.6 Rodata for f64 literals

f64 literals (= =1e-15= and the abs-mask) live in the section that
§100.A established for elisp string rodata.  Add an 8-byte alignment
constraint when emitting f64 entries (= existing string entries are
1-byte aligned).  Emit:

#+begin_src
  .section .rodata
  .align 8
  nelisp_cc_f64_eps_1e_15:    .double 1e-15
  nelisp_cc_f64_abs_mask:     .quad 0x7FFFFFFFFFFFFFFF
#+end_src

* §5 Risks / unknowns

** §5.1 Register pressure with mixed-class function bodies

Functions like =nl_jit_float_eq_eps= use xmm0 + xmm1 + rax
simultaneously.  Phase 47's existing register allocator tracks GP
liveness only; teaching it about xmm liveness is a structural
change.

*Mitigation*: §110.A introduces explicit per-class allocators (=
GP allocator and xmm allocator are independent because the
classes' register sets don't overlap).  Liveness analysis runs per
class.  Verified bit-exact via §110.C ert tests before §110.E ships.

** §5.2 NaN handling in =:EQ-EPS=

=UCOMISD= sets PF on NaN inputs; the SETcc / setnp / setpe choice
affects whether NaN compares as not-equal-but-not-less-or-greater.
Rust's =(a - b).abs() < 1e-15= treats NaN inputs as never equal
(= NaN comparisons return false).  The elisp emit must match.

*Mitigation*: §110.C smoke tests include NaN cases.  Use
=COMISD= + check PF + ZF = bit-exact match of Rust semantics.

** §5.3 aarch64 cross-arch verification

§110.D ships f64 emit but linux-x86_64 hosts cannot directly execute
aarch64 binaries.  Same constraint as §100.D Stage 2/3 — use
=aarch64-linux-gnu-objdump= to verify instruction encoding + a
qemu-user smoke run (= optional, behind feature flag).

*Mitigation*: identical to §100.D Stage D verification.  CI matrix
covers actual aarch64 via release-qualification.yml.

** §5.4 IEEE 754 subnormal handling

Some i64↔f64 casts handle subnormals differently across CPUs.  =FTZ=
/ =DAZ= MXCSR flags affect SSE behavior.

*Mitigation*: §110.E+F do not introduce i64↔f64 casts (= float.rs
and math.rs don't use them).  Defer to Doc 112+ if needed.

** §5.5 LOC estimate accuracy

§100.D Stage 2/3 estimated ~280 LOC elisp scaffolding for ARM64 +
Mach-O; actuals were within ~15% (= confirms our methodology).  Doc
110 estimates ~300 LOC elisp scaffolding total across §110.A-F.

*Mitigation*: per-stage LOC estimate is a budget, not a promise.
Each stage commit records actuals; deviations >25% trigger a DRAFT
revision before the next stage.

** §5.6 What if Phase 47 reveals a fundamental design issue at §110.B?

E.g., the existing IR data structures (= =(:kind ...)= plists)
don't accommodate per-node register class.

*Mitigation*: §110.A is the design-validation stage — its single
literal-load smoke proves the register class model end-to-end before
§110.B-F commit to it.  If §110.A reveals a design issue, this DRAFT
gets a v2 rewrite before further code lands (= same protocol as
Doc 100 v1 → v2).

* §6 Out of scope

- *f32 / single-precision*: Elisp =float= is always f64 in NeLisp.
  No path covers SSE2 single-precision SS variants (= ADDSS etc.)
  or AArch64 single-precision FP encoding.

- *Vector / SIMD operations*: 128-bit xmm vector ops (= ADDPS etc.)
  are not introduced.  Future Doc 112+ if vector compute reaches
  Phase 47.

- *Long double / x87*: x87 80-bit FP stack is not introduced.  All
  Doc 110 paths use SSE2 scalar double, not x87.

- *f64 in Sexp pointer args*: =extern "C" fn(*const Sexp, ...)=
  signatures where one field is f64 stay out of scope (= the Sexp
  layout itself doesn't expose unboxed f64 fields; =Sexp::Float= is
  read via =sexp-float-unwrap= which is a future Doc 101 grammar
  form, not Doc 110).

- *Vector loads / SIMD GP register interop*: =MOVQ rax, xmm0= and
  similar bit-pattern transfer is not introduced (= rejected as
  Option C in §0.2).

- *Doc 102-109 env.rs deletion*: parallel track, unrelated capability
  expansion.  Doc 110 ships independently and does not depend on or
  block Doc 102+.
