* §0 TL;DR

** §0.1 What Stage 9.5 actually shipped (= the gap)

Doc 75 §3.5 marked Stage 9.5 as SHIPPED 2026-05-10.  The actual change
was a rewire of =compile_elisp_to_image=:

#+begin_src rust
pub fn compile_elisp_to_image(source: &str) -> Result<Vec<u8>, ImageError> {
    encode_v3_with_fallback(&Env::empty(), &[source.to_string()])
}
#+end_src

This wraps =source= as a single strategy-C fallback form inside an
NELIMG v3 envelope with an *empty* node pool (= =&Env::empty()=
contributes 0 globals).  The wire format is v3, but the semantics are
unchanged from the v2 form-list era: the boot path still has to parse
+ re-eval source strings.

** §0.2 What "real" Stage 9.5 means

A frozen-heap image with *non-empty node pool + empty fallback*.  At
boot, =decode_v3_into= populates =env.globals= directly from the node
pool, no re-parse, no re-eval.  This requires:

1. Baker must *evaluate* source into a populated =Env=, not stash raw
   strings.
2. Baker must iterate the STDLIB chain (= each file builds on prior
   files' env state) — the first file boots from builtins-only env,
   each subsequent file boots from the prior cumulative env.
3. =Env::new_global= must consume frozen-heap images via
   =decode_v3_into(&mut env, bytes)= directly, NOT via =decode_image=
   (which currently discards the decoded env via =Env::empty()=).

** §0.3 Why this is the *common* prerequisite

3 distinct migration tracks blocked on Stage 9.5-real in the 2026-05-11
session:

| Track                            | Surface block             | Common cause                |
|----------------------------------+---------------------------+-----------------------------|
| Doc 90 deferral 2 (parse_let)    | void-function during boot | env-shim loads at pos 4     |
|                                  |                           | (= jit-substrate position 1 |
|                                  |                           | uses =let= before shim live) |
| Stage B reader/ 全消滅          | prod binary panic         | non-empty fallback in .image |
| Stage C image.rs 全消滅          | hot-path on every boot    | decoder needed for fallback  |

All three resolve once images carry full frozen-heap node pool + empty
fallback section.  reader/ becomes dispensable, env-shim ordering
becomes irrelevant (= shim functions are already in globals at boot),
image.rs decoder becomes a pure deserializer of pre-evaluated state.

* §1 Background

** §1.1 NELIMG v3 wire format (= current state)

Per Doc 75 §3.3.1 (SHIPPED), the v3 wire format is:

#+begin_example
NELIMG\0\x03                  (8 byte magic)
u32  IMAGE_ABI_VERSION = 3
u8   KIND              = 0x01 (= frozen-heap)
u32  N_NODES
[N_NODES] { u8 NODE_TAG ; payload ... ; child refs = u32 NODE_INDEX }
u32  N_GLOBALS
[N_GLOBALS] { NAME ; FLAGS ; [value_idx, function_idx, plist_idx] }
u32  N_FALLBACK_FORMS
[N_FALLBACK_FORMS] { source_text }
#+end_example

The current baker writes:
- =N_NODES= = 0 (empty pool)
- =N_GLOBALS= = 0
- =N_FALLBACK_FORMS= = 1 (the entire source text)

Real Stage 9.5 baker will write:
- =N_NODES= = ~1000-5000 (= reachable Sexps from env globals)
- =N_GLOBALS= = ~200-500 (= defun / defvar / defmacro etc.)
- =N_FALLBACK_FORMS= = 0 (= ideal) or small residue (= side-effect
  forms that don't show up as env globals, e.g., =(provide 'X)=)

** §1.2 Existing encoder is ready

=encode_v3= / =encode_v3_with_fallback= (= image.rs lines 205, 214)
are already implemented.  They walk =env.globals=, intern reachable
Sexps via box-pointer dedupe + structural atom dedupe, emit NODE_INDEX-
referenced records.  The encoder is callable with any populated =Env=.

What's missing is a *driver* that populates Env iteratively + invokes
the encoder per-file with the *diff* of globals.

** §1.3 Strategy-C fallback semantics (= what to do with them)

Doc 75 §1.5 articulated strategy-C as "re-eval source at boot for
forms that capture lexical context which can't be serialised as a
closure".  Common examples:

1. =(provide 'feature)= — registers feature name in env.features
2. =(autoload 'sym "file")= — sets up lazy-load metadata
3. =(eval-when-compile ...)= — host-Emacs compile-time only
4. closures with non-empty captured env (= rare in stdlib)

Strategy: either capture these as additional env state (= extend env
schema), or keep them as small fallback forms.  Practical baker should
*minimize* fallback to forms that genuinely can't be serialised.
Initial pilot: accept any residue as fallback, count it, articulate in
§98.1 roundtrip test.  If residue is large (= > a few forms per file),
revisit schema.

"every commit must net-decrease Rust LOC, primitive 追加と Rust dispatch
削除を必ず同 commit で対にする".

Doc 98's Stage 98.1 *violates* this rule by design (= +300 LOC
infrastructure with no offsetting deletion).  Justification:

1. The +300 is *infrastructure investment* toward an atomic large-
   delta payoff (= §98.3 -1,679).
2. Cumulative across §98.1〜§98.5: *-2,200〜-2,700 net* (= 7-9x rule
   compliance over the bundle).
3. Sliced commit boundaries are unavoidable — atomic single-commit
   for "infrastructure + rebake + new_global cutover + reader delete +

   un-reviewable.
4. user pre-approval at Doc 98 acceptance gates the rule waiver.

Memory entry =feedback_rust_core_minimization_principle= recommends
recording such planned violations explicitly so future bisects find
the rationale.  Each commit message in §98.1〜§98.2 should reference

* §5 Risks / unknowns

** §5.1 Strategy-C residue (= fallback non-emptiness)

Some stdlib forms may not be capturable as env globals:
- =(provide 'X)= (= unless env.features schema is extended)
- closures with captured non-empty env at top level (= probably zero
  in stdlib but unverified)
- =(eval-when-compile ...)= (= host-Emacs-only; possibly nil at runtime)

§98.1 roundtrip test articulates residue count.  Mitigation options:
- Extend Env schema to capture =features=, =autoload=, etc.
- Keep small residue (= 1-5 forms per file) as fallback
- Synthesize equivalent forms at decode time

** §5.2 NlConsBox identity at decode-time

The encoder dedupes by box-pointer identity.  Decoder rebuilds via
=NlConsBox::new(empty)= placeholders + fill pass.  Any encoder/decoder
mismatch on identity semantics will cause "looks the same" globals to
silently diverge.

§98.1 roundtrip test must compare via =Sexp::equal_structural= not
=Sexp::eq= to verify identity is preserved.

** §5.3 Order-dependent iteration

The iterative baker relies on STDLIB_FILES order being identical at
bake and boot.  Doc 75 §3.5 already established this invariant
(=STDLIB_FILES in baker = STDLIB_IMAGES in env.rs=, kept in sync by
manual review).  §98.2 driver should assert order parity explicitly.

** §5.4 cargo test runs use STDLIB_IMAGES at test time

Cargo tests construct =Env::new_global()= which loads STDLIB_IMAGES.
After §98.2 rebake, these images are frozen-heap.  Tests should
continue to work because they're testing eval/special form semantics
against the post-load env — they don't care whether env was populated
via fallback re-eval or frozen-heap decode.

Risk: some tests may rely on side effects of fallback re-eval (= e.g.,
=(message ...)= side effects during boot).  §98.1 cargo test set
should be a canary.
