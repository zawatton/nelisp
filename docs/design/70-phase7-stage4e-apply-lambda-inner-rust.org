* §0 TL;DR

Stage 7.4.d で発見した *frame-capture leak* を、
=nelisp--apply-lambda-inner= の *elisp defun を Rust builtin に降ろす* こと
で解消する。helper の formal slot (= =--nl-ali-captured= /-formals/-body/
-args) が elisp lexical env に出ないため、=defun= が body 内で実行されても
slot を捕捉できない。

* §1 動機 (= Doc 68 §3.4 の再要約)

** §1.1 観測 bug (= Stage 7.4.d default-on spike で fail)

#+begin_example
$ NELISP_USE_ELISP_APPLY=1 cargo test load_evaluates_file_contents
...
panic: assertion failed: result == 99
note: signaled wrong-number-of-arguments: (lambda 0 1)
#+end_example

具体経路:

1. =(load "/tmp/zz.el")= — =load= は elisp 関数 (= Doc 67 で migrate 済).
2. =load= body 内で各 top-level form を =eval= する.
3. ある form が =(defun X () 99)= の場合、=defun= は special form 判定
   後に =sf_lambda= で *現在の lexical env を captured-env として snapshot*.
4. snapshot 時の lexical env スタック:
   - =[load の call-frame: file, ...]=
   - =[apply-lambda-inner の call-frame: --nl-ali-captured, --nl-ali-formals, --nl-ali-body, --nl-ali-args]=
   - =[captured frame: load の closure]=
   - =[inner formals frame: file=...]=
5. =capture_lexical= は innermost-first で walk するので、=--nl-ali-*=
   slot 4 つも snapshot に含まれる.
6. 後で =(X)= 呼出 → =push-captured= で =--nl-ali-formals= 等が new
   captured frame に install.
7. 次に =apply-lambda-inner= が呼ばれると、helper 自身の formal binding
   が leaked slot を *shadow* (= bind-local 時に topmost frame を target
   するが、leaked frame の slot を上書き) → state machine 破綻.

** §1.2 なぜ Rust 化で解決するか

apply-lambda-inner を Rust builtin にすると、helper の state
(= captured / formals / body / args / 中間 pairs / last result) は
*Rust の関数 stack frame* に置かれる. elisp 側の =Env::frames= には
出ない → =capture_lexical= の walk 対象外.  defun が body 内で snapshot
を取っても helper の slot は含まれない.

apply-fn / apply-closure / apply-lambda の formal slot (= =fn=, =closure=,
=args=) は *elisp lexical env に残る* が、それらの leak は本 Stage の
scope 外 (= 観測される failure を起こさない範囲).  必要なら Stage 7.4.f
以降で扱う.

** §1.3 Rust 側に既に基盤がある

=build-tool/src/eval/mod.rs= line 476-512 に既存の
=fn apply_lambda_inner(captured, formals, body, args, env)= があり、
Rust dispatch 経路 (= =Env::use_elisp_apply= 未設定) で常用.
本 Stage はそれを *builtin として exposed する* だけ → ロジック書き起し
不要、~25 LOC のラッパで済む.

=builtins.rs=:

#+begin_src rust
/// `(nelisp--apply-lambda-inner CAPTURED FORMALS BODY ARGS)` —
/// Rust builtin replacing the elisp defun of the same name (Stage
/// 7.4.b dormant install → Stage 7.4.e Rust takeover).  All four
/// state slots plus intermediate `pairs' / `last' live on the Rust
/// call stack — nothing enters the elisp lexical env, so a `defun'
/// executed within BODY cannot snapshot this builtin's formal slots.
/// Surgical fix for the Stage 7.4.d capture-leak bug.
fn bi_apply_lambda_inner(args: &[Sexp], env: &mut Env) -> Result<Sexp, EvalError> {
    require_arity("nelisp--apply-lambda-inner", args, 4, Some(4))?;
    let captured = &args[0];
    let formals = &args[1];
    let body_list = &args[2];
    let args_list = &args[3];
    let body_vec = super::list_elements(body_list)?;
    let args_vec = super::list_elements(args_list)?;
    super::apply_lambda_inner(captured, formals, &body_vec, &args_vec, env)
}
#+end_src

* §5 Risks / Open issues

** §5.1 apply-fn / apply-closure formal slot leak (= 残存)

apply-fn の =fn=, =args= 等の formal slot は依然として lexical env に
入る.  user defun が同名 formal を declared していなければ unobservable
だが、observable な場合は default-on で test fail の可能性.

mitigation: e.2 で全 test 走らせて問題があれば e.3 として apply-fn /
apply-closure も Rust builtin 化.

** §5.2 expand-macro / bf-compute slot leak

同じく defun のため formal slot が残る.  =macro-form= / =arg-forms=
/ =formals= / =args= が user code と衝突する場合に problematic.
特に =formals= / =args= は common name なので注意.

mitigation: e.2 で fail 観測があれば、bf-compute だけでも Rust 化
(= ~30 LOC) → bf-compute は dispatch logic ではなく state計算なので
Doc 68 §1 の "dispatch logic を elisp に" 趣旨を毀損しない.

** §5.3 bench regression > 2x

elisp dispatch が outermost-only (= delegation_depth=0 のみ) でも
function 1 回 call ごとに =delegate_to_elisp_apply= の dispatch_form
構築 + eval が走る.  100 万回呼ぶ bench で 2x 超える可能性あり.

mitigation: Phase 5 lower entry に =apply-fn= path を追加し、Rust
builtin 直接 lower で bypass.  Stage 7.4.f として切出.
