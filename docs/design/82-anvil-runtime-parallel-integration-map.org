* §5 Risk + mitigation

** §5.1 二重 commit 衝突 (= 同 file を両 session で編集)

Risk: 両 session が同じ anvil-runtime/ Rust file を異なる branch で
同時編集する.

Mitigation:
- §3.2 で示した通り *layer 直交* (= parallel session は NeLisp side、
  my session は anvil.el repo + anvil-runtime delete).  Phase B 進行
  中 (= parallel session 主体) は my session の Stage 74.1 (= Rust
  compile fix) のみ ship、Stage 74.2 以降は parallel session の Phase
  B5 SHIPPED 確認後に dispatch.
- Doc 82 に \"並行 session 状態\" を articulate し、各 session 開始時
  に本 Doc を確認.

** §5.2 framing 不整合 (= rule 解釈 / 工数 estimate / scope boundary)

Risk:
- monotonic shrink rule の解釈差 (= parallel session の \"投資 LOC は
  Final B で回収\" vs my session の \"per-stage で必ず Δ ≤ 0\")
- 工数 estimate (= parallel = 1-2 ヶ月、my = 7 day) の不一致

Mitigation:
- Doc 82 §3.2 で layer 直交を articulate、両 session が *同じ commit
  の中で Rust 増加を ship しない* 限り rule 違反は発生しない.  parallel
  session の elisp 投資は Rust LOC 0 increment で進められている (= 確認済み:
  Phase B2-B4 commit body の Rust delta = +84 のみ、それも Phase B6+
  で回収予定).
- 工数 estimate は両者が独立に立てた数値で *grand total では同じ
  destination*.  本 Doc §4.2 で reconciliation table を残す.

** §5.3 phase 順序衝突 (= my session の Stage 74.x が parallel の Phase B より先に出る場合)

Risk: my session が Stage 74.2 以降を ship → anvil.el repo に新 elisp
file が増える → parallel session の Final B が anvil.el repo を 1 行も
触らない方針と矛盾、結果 anvil.el repo に dead code が残る.

Mitigation:
- Doc 82 §4.4 Option α 採用 → my session の Stage 74.2-74.5 は parallel
  session Final B の進捗を見て決める
- Stage 74.1 (= Rust compile fix) のみ即時 ship 可、それ以降は dispatch
  gate を Doc 82 §6 で明記

** §5.4 LOC accounting double-count

Risk: =anvil-runtime crate 削除= を両 session が独立に \"自分の delta\"
として claim する.

Mitigation:
- Doc 82 §4.3 で canonical accounting を固定 (= anvil-runtime crate -5,667
  は 1 度のみ計上、reader cfg-gate -1,679 は Doc 73 Stage 8.4 accounting)
- 各 session の commit message に \"per Doc 82 §4.3 canonical accounting\"
  を入れる慣行を提案
