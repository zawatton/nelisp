** 2.1 process spawn primitive: =posix_spawn= 優先 + =fork+exec= fallback [推奨 = A]

A. *推奨*: =posix_spawn= を default、=fork+exec= は fallback (= macOS
   では posix_spawn が推奨経路、Linux でも glibc 内部で =vfork= 経由
   実装で =fork= より高速)
B. =fork+exec= 単独 (= POSIX 古典、portable 最大)
C. =vfork+exec= 単独 (= memory 効率 max、ただし非標準動作 risk)
D. =posix_spawn= 単独、=fork= 完全廃止

採用理由 (A、T85 MAJOR-2.1 反映):
- macOS = =posix_spawn= が *推奨経路*、=fork= は thread-safe ではない
  (Apple 公式 deprecation prospect)、=posix_spawn= で安全。macOS の
  =posix_spawn= は libdispatch 経由実装で thread-safe、ただし wait queue
  overhead あり (= Linux 系 vfork path とは性能特性が異なる)
- Linux glibc の =posix_spawn= は *条件付き* で vfork-clone fast path:
  - *fast path 条件*: =POSIX_SPAWN_USEVFORK= attr 含むか、または "no-op
    spawn attrs" 時 (= 単純 =file_actions= ≤ 5 + sigmask / sigdef
    複雑度なし)
  - *fallback 条件*: 複雑な =file_actions= / =POSIX_SPAWN_SETSIGMASK= /
    =POSIX_SPAWN_SETSIGDEF= / =POSIX_SPAWN_SETSID= 等の組合せで glibc
    が generic fork path に fallback (=glibc sysdeps/unix/sysv/linux/
    spawni.c= 参照、glibc 2.24+ で =clone(CLONE_VM | CLONE_VFORK)=
    base 新実装、ただし複雑 file_actions で fork fallback あり)
  - *anvil 想定 use case*: attrs 単純 + file_action ≤ 5 (= stdin /
    stdout / stderr / 1-2 inherit fd) で fast path 期待、ただし
    benchmark 必須
- =fork+exec= fallback で =posix_spawn= が utility 不足のケース
  (= file action / signal action の複雑な組合せ + Linux 旧 glibc) を救済
- 両 OS で benchmark 必須 = macOS = wait queue overhead あり、Linux =
  vfork path、性能特性が異なるため anvil 想定 workload で実 measure
- B は macOS で thread-safety 警告、C は portability risk、D は
  =posix_spawn= の attr API が複雑なケースで escape hatch なし

mitigation:
- =posix_spawn= 失敗時の fallback path は ERT で fault inject 確認
- macOS では =posix_spawn= が default、Linux では =glibc 2.24+= 確認
  (= fallback to =fork+exec= for older systems)
- Linux fast path 期待は §5 measurement protocol で macOS / Linux 別
  spawn latency p95 を計測、両 OS で gate 確認
- 平均的 anvil workload (= attrs 単純 + file_action ≤ 5) で fast path 期待、
  fallback 経路は glibc 旧 + 複雑 file_actions でのみ走行

** 2.2 eventloop primitive: =select(2)= MVP + =poll(2)= upgrade path [推奨 = A]

A. *推奨*: =select(2)= で MVP (= POSIX 古典、portable 最大)、=poll(2)=
   は v2.1+ で upgrade
B. =poll(2)= MVP (= fd 数 1024 制限なし、現代的)
C. =epoll(7)= Linux + =kqueue(2)= macOS (= async IO 性能 max)

採用理由 (A、T85 MAJOR-2.2 反映):
- =select(2)= = POSIX 共通、Linux + macOS で identical 動作、portability
  最大
- *fd budget 試算 (anvil 同時起動時)*:
  - child fd 32 (= anvil-orchestrator 5 LLM provider × ~6 fd/child) +
    state SQLite 1 + org-index SQLite 1 + http connection pool 16 +
    log fd 4 + stdin/stdout/stderr 3 ≒ *60 fd*
  - FD_SETSIZE 1024 内で十分余裕、ただし future memory-engine 経由
    SQLite WAL fd 急増 risk あり (= 1 SQLite WAL = 3 fd / table、tables
    増で 100+ fd 達)
  - 安全 margin = 1024 - 60 = ~960 fd 余裕、ただし stress 増で逼迫 risk
- anvil 既存 use case (= 並行 child process 数 ≤ 32、Doc 38 audit) は
  =select= の fd 1024 制限内で十分、ただし memory-engine SQLite WAL
  fd 数監視必要
- *Linux FD_SETSIZE 拡張* = recompile 必要 (= libc constant)、runtime
  tunable ではない、libc rebuild は出荷経路で非現実的、=poll= upgrade が
  正しい mitigation
- =poll= upgrade は v2.1+ で必要時 (= fd 1024 超 use case 出現時、また
  は memory-engine WAL fd 急増時)
- =epoll/kqueue= は Phase 10+ で Layer 1 separation 後、async IO 専用
  module として spec
- B は fd 数優位だが API 互換性 break (= struct pollfd vs fd_set)、本
  phase は MVP 優先
- C は OS 別実装 = portability cost 高 (= 同一 source で Linux / macOS
  両対応のための conditional compile 必要)

mitigation:
- fd 数制限 (1024) で運用上限を spec 内明記
- *fd budget audit ERT を blocker gate* (T85 MAJOR-2.2 反映): fd 数
  50 / 200 / 500 の stress test、= select(2) 動作確認、500 fd 時 tick
  overhead < 200us p95 確認
- =poll= upgrade path = 本 phase で API surface を fd-based abstract に
  抑制 (= =eventloop-fd-set-add= / =eventloop-fd-set-remove= の OS-
  independent abstract API、内部 implementation を select → poll に
  swap 可能、API break なし)
- §6.6 Architecture B reference は本 phase scope 外 (= memory feedback
  は Emacs 経路の話、本 phase NeLisp standalone は別 process model)、
  reference のみ留め、本 phase の risk としては *扱わない*

** 2.3 filter callback dispatch model: synchronous + GC barrier [推奨 = A]

A. *推奨*: filter callback を *eventloop main thread 内 synchronous*
   実行、GC barrier (= filter 中 GC 不可、return 後 GC trigger) で
   reentrancy 抑制
B. async dispatch (= filter callback を separate goroutine / task で
   実行、Lisp main thread と並行)
C. queue dispatch (= filter callback を queue に積み、main thread の
   =accept-process-output= で順次 dispatch)

採用理由 (A):
- Emacs 既存 semantic との parity = filter は main thread 内 synchronous
  (Emacs C runtime の =process_send_signal= と同等)
- B は multi-thread 前提、Phase 7.x single-thread MVP と矛盾
- C は queue lag が =process-send-string= response 遅延を導入、interactive
  use case で体験劣化
- GC barrier semantics = filter 中の =cons= / =allocate= が GC trigger
  不可 = filter 内で大量 allocation する code は事前 reserve 必要 (Emacs
  標準 pattern)

mitigation:
- filter 中 GC trigger は debug build で assertion (= =gc-inside-filter-p=
  flag)
- filter 内 allocation は =with-process-filter-arena= macro で local
  arena 提供
- *precedent reference* (T85 MAJOR-4.2 反映): Emacs C runtime の
  =process.c= 内 =gcpro= / =Vinhibit_quit= bind pattern と同等の barrier
  semantic を採用 (= filter return まで GC 抑止、return 後 next eventloop
  tick で deferred GC trigger)。Doc 30 v2 §6.9 finalizer queue separation
  pattern は GC sweep 中の write barrier 競合解消用で本 phase の filter
  reentrancy / GC barrier 議論とは目的が異なるため reference しない
- Emacs 互換 = filter callback signature = =(lambda (proc string) ...)=、
  string buffer 寿命 = filter return まで

** 2.4 sentinel ordering: =waitpid= → status report atomicity [推奨 = A]

A. *推奨*: sentinel callback は *=waitpid= 完了後、status report 確定
   後 single shot* で fire (= Emacs 既存 semantic と同等、ordering 強保証)
B. sentinel = =SIGCHLD= 受信時即 fire (= waitpid は sentinel 内で user
   実行)
C. sentinel = poll 経由 (= =process-status= が =run= → =exit= 遷移検出
   時 fire)

採用理由 (A):
- Emacs 既存 semantic との parity = sentinel が fire した時点で
  =process-exit-status= が確定値返す (= waitpid 完了)
- B は SIGCHLD handler 内で =waitpid= = signal-safe primitive 制限あり、
  user code に signal-safety を要求すると体験劣化
- C は polling lag (= eventloop tick 間隔分の遅延)、interactive use
  case で体験劣化
- *atomicity* = =waitpid= 結果 → =process-exit-status= 更新 → sentinel
  fire の順序を eventloop 内で保証、間に他 callback 介在不可

mitigation:
- sentinel ordering ERT (= =delete-process= → sentinel fire → 100ms
  後 =process-status= 確認、=exit= 確定確認) を blocker gate
- 並行 child の SIGCHLD storm 対応 = =waitpid(WNOHANG)= loop で全 zombie
  収穫、sentinel は zombie 順 fire

** 2.5 fd lifetime / close-on-exec hygiene [推奨 = A]

A. *推奨*: 全 fd を =O_CLOEXEC= で open、=pipe2(O_CLOEXEC)= 使用、parent
   process 経由 leak 0 確保
B. =fcntl(F_SETFD, FD_CLOEXEC)= で post-open に設定 (= race window あり)
C. fd 管理を user 責任化 (= =delete-process= で明示 close)

採用理由 (A):
- =O_CLOEXEC= で race window 0、= multi-thread parent でも safe
- B は =open= → =fcntl= 間で =fork= が他 thread から発火すると leak
- C は user error 多発 risk (= forget-to-close pattern が anvil bench
  で頻発、Doc 38 audit)
- POSIX 2008 で =O_CLOEXEC= 標準化、Linux 2.6.23+ / macOS 10.7+ 対応、
  portability 問題なし

mitigation:
- fd leak 検知 ERT = 100 child spawn cycle 後 =lsof -p $$= で fd 数
  baseline 一致確認、blocker gate
- =close-on-exec= 違反 (= 意図的に inherit したい fd) は
  =process-inherit-fds= keyword で明示 opt-in

** 2.6 zombie cleanup: =waitpid(WNOHANG)= eventloop tick [推奨 = A]

A. *推奨*: eventloop tick (~10ms 間隔) ごとに =waitpid(-1, WNOHANG)=
   loop で全 zombie 収穫、=process-status= 自動更新
B. =SIGCHLD= handler で =waitpid= 即実行 (= signal-safe primitive 制限)
C. user 明示 =waitpid= API (= =process-wait= primitive 提供、user 責任)

採用理由 (A):
- POSIX zombie cleanup standard pattern (= polling)
- B は signal-safety 制限、user code との reentrancy issue
- C は user error risk (= forget-to-wait pattern で zombie 蓄積)
- ~10ms tick = interactive 体験 OK (= sentinel fire 遅延上限 10ms)、
  background batch では問題なし

mitigation:
- zombie 検知 ERT = 100 child spawn 後 =ps aux | grep defunct= で 0
  確認、blocker gate
- tick 間隔 tunable (=process-eventloop-tick-ms= defcustom、default 10、
  range 1-100)

** 2.7 process-send-string buffering [推奨 = A]

A. *推奨*: =process-send-string= は *blocking* (= write completion

C. user 明示 flush (= =process-flush-output= API)

採用理由 (A):

mitigation:

* §4 Architecture (=nelisp-process= delegate chain + eventloop)

#+begin_src text
[ user code ]
   ↓ (make-process :name "x" :command '("/bin/echo" "hello") :filter ... :sentinel ...)
[ src/nelisp-process.el (Layer 2 = Emacs API 互換 wrapper) ]
   ↓ keyword parse + child config build
   ↓ pipe2(O_CLOEXEC) → stdin/stdout/stderr fd
   ↓ posix_spawn(file_actions, attrs) [primary path]
   ↓ fallback fork+exec [if posix_spawn fails]
[ nelisp-runtime/src/syscall/process.rs (Layer 1 = Rust syscall thin wrapper) ]
   ↓ POSIX syscall → child process spawned
[ child process (separate address space) ]
   ↓ exec /bin/echo hello → stdout writes "hello\n" → exit 0
[ parent eventloop tick (src/nelisp-eventloop.el) ]
   ↓ select(2) detects stdout fd readable
   ↓ filter callback fire (synchronous, GC barrier)
   ↓ "hello\n" → user-defined filter
[ child SIGCHLD ]
   ↓ eventloop tick: waitpid(WNOHANG) → child reaped
   ↓ process-status updated to 'exit
   ↓ sentinel callback fire (after waitpid completion, atomic ordering)
   ↓ "finished\n" → user-defined sentinel
[ user observes process complete ]
#+end_src

*失敗経路 (= posix_spawn fail → fork+exec fallback)*:

#+begin_src text
[ make-process call ]
   ↓ posix_spawn(file_actions, attrs) → -1 (e.g. EINVAL on macOS attr complexity)
[ Layer 2 fallback decision ]
   ↓ try fork+exec
   ↓ fork() → child / parent
[ child branch ]
   ↓ dup2(pipe_read_fd, 0) / dup2(pipe_write_fd, 1) / dup2(stderr_fd, 2)
   ↓ close non-essential fds
   ↓ execve(path, argv, envp) → no return on success / exit on fail
[ parent branch ]
   ↓ register child pid + fd in process table
   ↓ eventloop integration as primary path
#+end_src

*timer + filter ordering*:

#+begin_src text
[ eventloop tick start ]
   ↓ select(2) with timeout = next_timer_fire_time
   ↓ if fd ready: filter callback fire (sync)
   ↓ if timeout reached: timer callback fire (sync, after filters)
   ↓ if both: filter first, timer deferred to next tick
[ next tick ]
   ↓ ...
#+end_src

* §6 Risks (9 項目、T85 MAJOR-4.1 反映で §6.9 child resource exhaustion 新設、T93 MINOR-D4 反映で header 件数 update)

** 6.1 macOS posix_spawn / fork+exec の reliability divergence

macOS では =fork= が thread-safe ではない (Apple 公式 deprecation
prospect)、=posix_spawn= 推奨だが file action / signal action の attr
API が複雑、stable 動作要 stress test。

mitigation:
- §2.1 採用 A の =posix_spawn= 優先 + =fork+exec= fallback dual-path
- 8 child 並行 spawn ERT で macOS arm64 で stress test、failure rate
  5%+ なら fallback path 検証強化
- macOS 12+ (= =posix_spawn= modern API stable) を min support
  baseline、macOS 11 以前は best-effort

** 6.2 zombie process / PID race

POSIX zombie cleanup の standard pattern (= =waitpid(WNOHANG)= polling)
で cover、ただし child SIGCHLD を miss すると永続 zombie。

mitigation:
- §2.6 採用 A の eventloop tick (10ms 間隔) で =waitpid(-1, WNOHANG)=
  loop 実行
- ERT で 100 child spawn 後 =ps aux | grep defunct= 0 確認、blocker
  gate
- zombie 蓄積検知 (= =process-list-stats= で defunct count > 0 時 warn)

** 6.3 fd leak (close-on-exec hygiene)

=O_CLOEXEC= で全 fd 保護 (§2.5 採用 A)、ただし =:process-inherit-fds=
keyword で明示 opt-in した fd は parent 経由 leak risk。

mitigation:
- §2.5 採用 A の =O_CLOEXEC= default + opt-in inherit
- 100 cycle ERT で =lsof -p $$= baseline 一致確認、blocker gate
- inherit fd 利用 module は ERT で明示 cleanup verify

** 6.4 filter callback reentrancy / GC interaction

§2.3 採用 A の synchronous + GC barrier model で 5 段までは OK、超え
たら error。anvil-worker / anvil-orchestrator で =process-send-string=
を filter 内で呼ぶ pattern が深い再帰すると深刻。

precedent reference (T85 MAJOR-4.2 反映): Emacs C runtime の
=process.c= 内 =gcpro= / =Vinhibit_quit= bind pattern を本 phase で
採用、Doc 30 v2 §6.9 finalizer queue separation pattern (= GC sweep
中の write barrier 競合解消用) とは目的が異なるため reference しない。

mitigation:
- §2.3 採用 A の =eventloop-reentry-depth= guard (5 段超 error)
- ERT で reentrancy 1 / 3 / 5 段 fire 確認 + 6 段 error 確認
- filter 内 GC trigger は debug build assertion (=
  =gc-inside-filter-p= flag)

** 6.5 multi-thread expansion (Phase 7.x single-thread MVP との関係)

Phase 7.x は single-thread MVP 確定 (Doc 29 v2 §2.9)、本 phase も
single-thread eventloop。multi-thread 化は Phase 10+ 候補だが、anvil-
orchestrator dispatch で 8+ child 並行が常態化すると single-thread
eventloop tick が saturate する risk。

mitigation:
- single-thread eventloop で 32 child 並行 OK 確認 (= bench gate)
- 32 超は v2.1+ で multi-thread eventloop / async IO (=epoll / kqueue=)
  upgrade を spec
- anvil-orchestrator で並行 child を 32 内に capacity-throttle

** 6.6 emacs-server 並列 access / eventloop lock contention

Emacs server (= =emacsclient -e=) からの並列 connection と eventloop
の interaction = single-thread eventloop で eval handler 中に新 server
connection 来ると eval が yield するか?

(本 risk は memory =feedback_emacs_accept_process_output_no_server_yield.md=
で既知 = =accept-process-output= は新 server connection の eval 実行

mitigation:
- 本 phase scope では Architecture B (= 別 MCP server) を *前提化*、

** 6.7 cancellation propagation (delete-process → SIGTERM → waitpid)

** 6.8 timer + filter contention (run-at-time が filter 中 fire)

* §9 Open questions

** 9.1 fork or posix_spawn 優先の最終決定

§2.1 採用 A = =posix_spawn= 優先 + =fork+exec= fallback、ただし:
- Linux glibc =posix_spawn= は *条件付き* vfork-clone fast path (= no-op
  attrs / 単純 file_actions)、複雑な attrs / file_actions / sigmask /
  setsid 等で fork fallback (= glibc =sysdeps/unix/sysv/linux/spawni.c=
  参照、§2.1 採用理由 mitigation)
- macOS =posix_spawn= の =POSIX_SPAWN_SETSIGMASK= が Apple Silicon
  arm64 で stable か (= macOS 12 / 13 / 14 で test 必要)
- macOS posix_spawn は libdispatch 経由実装、wait queue overhead あり、
  Linux fast path とは性能特性が異なる (= 両 OS benchmark 必須)

確定 trigger (T85 MINOR-4.3 反映): *Phase 9d.J ERT smoke 緑時点で確定*、
8 child 並行 spawn ERT で 3 OS (Linux x86_64 + macOS arm64 + Linux arm64)
× spawn latency p95 を比較、fast path が anvil 想定 workload (= attrs
単純 + file_action ≤ 5) で stable に取れるか確認。

** 9.2 filter callback 内 allocation の GC trigger timing

§2.3 採用 A = synchronous + GC barrier (= Emacs precedent =process.c
gcpro / Vinhibit_quit pattern=)、ただし *filter callback 内 allocation
の GC trigger timing* で 3 候補:

- A: filter return 後 immediate GC trigger (= return 直後 sync)
- B: next eventloop tick で deferred GC (= filter return → tick 末尾 → GC)
- C: immediate GC during filter (= 現在の Emacs C runtime に近い、
  =gcpro= 内なら safe)

確定 trigger (T85 MINOR-4.3 反映): *Phase 9d.K eventloop tick ERT 緑
時点で確定*、3 候補から bench (= filter overhead vs latency tradeoff)
で最適を選定。

(Doc 30 v2 §6.9 finalizer queue separation pattern は GC sweep 中の
write barrier 競合解消用で本 question とは目的が異なる、T85 MAJOR-4.2
反映で reference 削除済。Doc 30 maintainer との adjustment は不要。)

** 9.3 sentinel ordering の atomicity 保証範囲

§2.4 採用 A = waitpid → status report → sentinel fire の atomic
ordering、ただし:
- 並行 child の SIGCHLD storm (= 100 child 同時 exit) で sentinel fire
  順序が deterministic か
- =delete-process= cascade (= SIGTERM → SIGKILL → waitpid) の sentinel
  が *1 回のみ fire* される保証

確定 trigger (T85 MINOR-4.3 反映): *Phase 9d.L SIGCHLD storm ERT 緑
時点で確定*、100 child 同時 exit ERT で sentinel fire 順序 deterministic
+ =delete-process= cascade で sentinel 1 回確認。

** 9.4 multi-thread expansion (decision、本 phase scope 外)
