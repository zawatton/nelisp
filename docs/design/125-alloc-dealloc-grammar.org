* §0 TL;DR

** §0.1 Why this doc exists

Doc 124 §5 names a single gap: the NlBox Drop elisp kernels cannot
express the *if-zero-refcount free* branch because AOT grammar
has *no `dealloc` op*.  The existing `nlrc_drop_box!` Rust macro
relies on `Layout::for_value()` / `Layout::new::<T>()` resolved at
Rust *compile time*, so a pure-elisp body that wants to free a heap
block needs *either* a runtime `(size, align)` pair *or* one extern
per type that pre-bakes the layout.

This doc adds *both* paths:

- §125.A — generic `(alloc-bytes SIZE ALIGN)` + `(dealloc-bytes PTR
  SIZE ALIGN)` ops wrapping `Layout::from_size_align`.  Foundation
  for the substrate (= the elisp caller can call the typed wrappers
  *or* drop down to byte-level for opaque arenas / GC pools later).
- §125.B-F — per-type extern wrappers `nl_dealloc_consbox(ptr)` /
  `nl_dealloc_vector(ptr)` / `nl_dealloc_cell(ptr)` / `nl_dealloc_
  record(ptr)` / `nl_dealloc_str(ptr)` that internally invoke
  `alloc::dealloc(ptr, Layout::new::<NlT>())` so the Doc 124.G-K
  Drop kernels can call a single-arg helper.

The AOT grammar surface (= §125.A) lands first; §125.B-F are
trivial 3-LOC Rust externs gated on each §124.G-K stage shipping
its own probe.

** §0.2 Why generic + per-type both

Two reasons to keep both:

1. *Type-specific layout pin*.  Doc 124 §3.3 already establishes the
   per-type `const_assert_eq!(offset_of!(NlT, refcount), N)` pattern.
   The §125.B-F wrappers extend this discipline to the layout —
   `nl_dealloc_consbox` uses `Layout::new::<NlConsBox>()` which fails
   the build if the type definition drifts.  Bypassing this and
   emitting `(dealloc-bytes ptr 72 8)` from the elisp Drop kernel
   loses the compile-time layout-pin and trades a tiny LOC saving
   for substantial maintenance risk.
2. *Future GC / arena freedom*.  The Doc 126-128 bridge GC plan
   needs to free opaque buffers (= concatenated leaf slabs, hash
   bucket arrays) whose `Layout` is computed at runtime.  Those
   callers go through §125.A directly with `(alloc-bytes SIZE
   ALIGN)`.  Without the generic op, every new arena size needs
   its own extern.

So: §125.A = grammar foundation + GC use case; §125.B-F = stable
typed helpers for the Drop kernels.

* §2 Stage breakdown

** §2.A — Generic byte-level `alloc-bytes` / `dealloc-bytes` grammar

Adds two AOT grammar ops:

- `(alloc-bytes SIZE ALIGN)` → i64 (= raw `*mut u8` re-cast to i64,
  0 on `from_size_align` failure or OOM).  Substrate gate for the
  Doc 126-128 GC arena allocator.
- `(dealloc-bytes PTR SIZE ALIGN)` → i64 = 1 sentinel for `and`-chain
  composition.  Substrate gate for §125.B-F + the Doc 124.G-K Drop
  kernels' generic-path fallback.

Both lower to a single Rust extern call (= the same emit pattern as
the §122.E `atomic-fetch-add` op):

#+begin_src rust
#[no_mangle]
pub unsafe extern "C" fn nl_alloc_bytes(size: i64, align: i64) -> *mut u8 {
    let Ok(layout) = Layout::from_size_align(size as usize, align as usize) else {
        return std::ptr::null_mut();
    };
    if layout.size() == 0 { return std::ptr::null_mut(); }
    unsafe { alloc::alloc(layout) }
}

#[no_mangle]
pub unsafe extern "C" fn nl_dealloc_bytes(ptr: *mut u8, size: i64, align: i64) {
    let Ok(layout) = Layout::from_size_align(size as usize, align as usize) else {
        return;
    };
    if ptr.is_null() || layout.size() == 0 { return; }
    unsafe { alloc::dealloc(ptr, layout) }
}
#+end_src

** §2.B-F — Per-type dealloc externs (= deferred to §124.G-K)

Each NlBox type gets a 3-LOC extern wrapper that pre-bakes its
`Layout` at Rust compile time:

#+begin_src rust
#[no_mangle]
pub unsafe extern "C" fn nl_dealloc_consbox(ptr: *mut u8) {
    unsafe { alloc::dealloc(ptr, Layout::new::<NlConsBox>()) }
}

#[no_mangle]
pub unsafe extern "C" fn nl_dealloc_vector(ptr: *mut u8) {
    unsafe { alloc::dealloc(ptr, Layout::new::<NlVector>()) }
}

#[no_mangle]
pub unsafe extern "C" fn nl_dealloc_cell(ptr: *mut u8) {
    unsafe { alloc::dealloc(ptr, Layout::new::<NlCell>()) }
}

#[no_mangle]
pub unsafe extern "C" fn nl_dealloc_record(ptr: *mut u8) {
    unsafe { alloc::dealloc(ptr, Layout::new::<NlRecord>()) }
}

#[no_mangle]
pub unsafe extern "C" fn nl_dealloc_str(ptr: *mut u8) {
    unsafe { alloc::dealloc(ptr, Layout::new::<NlStr>()) }

Each landing pairs with the matching Doc 124.G-K Drop stage; the
elisp Drop kernel calls `(extern-call nl_dealloc_consbox ptr)` after
the `(atomic-fetch-add ... -1)` returns 1 (= pre-sub count was 1,
post-sub is 0).

* §3 Strategy

** §3.1 §125.A grammar surface

Parse + emit follow the §122.E precedent.  The `(alloc-bytes SIZE
ALIGN)` parse-value clause is a 2-arg variant of the `(atomic-fetch-
add PTR DELTA)` pattern (= eval both args, push onto stack, pop to
rdi/rsi, call extern, pop scratch slot).  `(dealloc-bytes PTR SIZE
ALIGN)` adds a third arg for rdx and matches the §122.E `(ptr-write-
u64 PTR OFFSET VAL)` emit shape exactly (= 3 pushes, pops to rdi/rsi/
rdx, extern-call, restore stack, `mov rax, 1` sentinel).

Current AOT aarch64 emit wires both ops as normal GP helper-call
paths, so macOS arm64 builds compile them directly instead of routing
through `--emit-aarch64-unsupported`.

** §3.2 §125.B-F per-type externs

Each is a one-line `unsafe extern "C" fn nl_dealloc_<T>(ptr: *mut u8)
{ alloc::dealloc(ptr, Layout::new::<T>()) }` in the matching
`build-tool/src/eval/nl{consbox,vector,cell,record,str}.rs` file.
No new elisp grammar — the Doc 124 Drop kernel just calls them via
the existing `extern-call` op:

#+begin_src elisp
(defun nelisp_nlconsbox_drop (box-ptr)
  (let ((prev (atomic-fetch-add (+ box-ptr 64) -1)))
    (when (= prev 1)
      (extern-call nl_dealloc_consbox box-ptr))))
#+end_src

The §124.G probe verifies this end-to-end.

** §3.3 Probe coverage (= §125.A this commit)

`build-tool/tests/elisp_cc_alloc_dealloc_probe.rs` exercises:

1. *Round-trip*: `alloc-bytes 64 8` → write a byte pattern via
   `ptr-write-u64` (§122.E) → read it back via `ptr-read-u64` → free
   via `dealloc-bytes ptr 64 8`.  Verifies the elisp grammar
   primitives compose end-to-end and the returned pointer is valid
   for read+write.
2. *Different sizes*: 3 separate allocs at sizes (= 8, 64, 4096) all
   align=8.  Verifies the layout arithmetic handles small/medium/
   page-sized blocks.  Each is freed via the matching dealloc.
3. *16-byte aligned alloc*: `alloc-bytes 128 16` returns a pointer
   whose low 4 bits are zero.  Verifies the alignment arg is honored
   (= matches the `Layout::from_size_align` contract for SIMD-friendly
   alignments that NlBox types use).

All cases run through the public wrapper in
`nelisp_build_tool::elisp_cc_spike::alloc_bytes` /
`::dealloc_bytes` (= same pattern as `atomic_fetch_add` wrapping
`nelisp_atomic_fetch_add`).

* §5 Alloc counterpart — `(alloc-bytes SIZE ALIGN)` rationale

NlBox lifecycle does *not* need `alloc-bytes` strictly — each NlBox
type already has a dedicated `nl_alloc_<T>()` extern shipped earlier:

- `nl_alloc_consbox` (= Doc 101 §101.D, `eval/nlconsbox.rs:113`).
- `nl_alloc_vector` (= Doc 115 §115.1, `eval/nlvector.rs`).
- `nl_alloc_cell` (= Doc 111 §111.D, `eval/nlcell.rs`).
- `nl_alloc_record` (= Doc 111 §111.B, `eval/nlrecord.rs`).
- `nl_alloc_str` / `nl_alloc_symbol` (= Doc 122 §122.A,
  `eval/nlstr.rs`).
- `nl_alloc_mut_str` (= Doc 122 §122.B, `eval/nlstr.rs`).

Shipping `alloc-bytes` in §125.A is *foundational*:

- Symmetry with `dealloc-bytes` — AOT grammar should not have
  a free op without an alloc counterpart (= teaches confusion).
- Doc 126-128 bridge GC (= concatenated leaf slabs, hash table
  bucket arrays, weak-ref tombstones) needs arena-style fresh
  `*mut u8` from elisp without per-shape externs.
- Future arena pooling experiments — having the grammar op lets us
  prototype `(arena-make POOL-SIZE)` etc. without touching Rust.

The cost is one extra extern (= ~25 LOC) + one extra parse-value
clause + emit fn (= ~30 LOC elisp).  Within the §125.A LOC envelope.

* §6 Risks + open questions

** §6.1 `Layout::from_size_align` failure modes

`from_size_align(size, align)` rejects when:

- align is not a power of two.
- size, rounded up to nearest multiple of align, would overflow
  isize.

The §125.A op returns 0 (= null pointer re-cast to i64) on both
cases.  This is the *same sentinel* as OOM (= `alloc::alloc`
returning null).  The elisp caller must inspect for 0 and either
`signal` an error or retry with smaller params.  The wrapper
`alloc_bytes` doc-comments this contract.

Decision: ship §125.A with the null-on-failure sentinel.  If a
future caller needs distinguishing layout-error vs OOM, add a
follow-up `(alloc-bytes-checked)` op that returns a 2-element
slot (= ptr + error code).  Not needed by Doc 124.G-K.

** §6.2 Should §125.A track allocations for double-free detection?

Adding a host-side `HashMap<ptr, Layout>` would catch caller bugs
(= free with wrong size, double free, free of stack pointer).
*Decision*: do not add tracking in §125.A.

Rationale: (a) every existing alloc path in `nl{consbox,vector,
cell,record,str}.rs` already runs without tracking, (b) tracking
adds atomic lock contention on every alloc / dealloc (= defeats
the purpose of moving the body to elisp), (c) the layout-pin via
`const_assert_eq!` (= §125.B-F per-type externs) catches the
common drift bug at *compile* time.

A debug-only Doc 125.A.+ follow-up could add tracking under a cfg
flag, mirroring the §122.E.2 proposal pattern.  Not on the
critical path.

** §6.3 Should §125.A also handle realloc?

Realloc would be useful for the GC arena resize path (= bump
allocator running out of slab room).  *Decision*: defer to a future
Doc 125.G stage when a real caller materialises.  Avoid speculative
surface area — AOT grammar ops are easier to add than retract.

** §6.4 Alignment of `Layout::new::<NlT>()` vs runtime alignment

The per-type externs (§125.B-F) use `Layout::new::<T>()` which pulls
both size and align from `T`.  The generic `dealloc-bytes` op takes
size + align from the elisp caller.  These *must* match for a
sound free — `alloc::dealloc` is UB if the layout doesn't equal the
layout used at `alloc::alloc` time.

For Doc 124 Drop kernels: callers go through §125.B-F (= layout
baked at compile time) so this risk is non-issue.  For Doc 126-128
GC: the arena code must track its own (size, align) per allocation
and pass them back at free.  The grammar op's contract is the same
as Rust's `alloc::dealloc` — caller responsibility.
