** §1.2 Doc 80 §7 の再評価

Doc 80 §7 は "out of scope" と articulate しているが、その理由は:
- (a) Float arith / cmp は Doc 80 が pre-stdlib substrate (= =defmacro=/=cond=/=when= 等の language rule) に focus したため、Float primitive は別 Doc 担当という分業.
- (b) 内部 box accessor は =Sexp::MutStr= 等の variant box の private field を elisp 露出させない方針だが、これは Doc 80 の判断であり、本 Doc は逆判断 = **trampoline ABI 経由で露出** とする.
- (c) =bi_syscall_nr_resolve= は libc::SYS_* compile-time constant の elisp 経路が無いだけ.  build.rs codegen で解決.

つまり Doc 80 の "permanent OOS" は **Doc 80 の scope に対する permanent**、**system 全体への permanent ではない**.  Doc 84 が Doc 80 の scope の外側で 15 fn を elisp 化する.

elisp 化阻害要因:
- pre-stdlib substrate (Doc 80) に Float arithmetic primitive が無い.
- =Sexp::Float(f64)= variant は elisp に露出されているが、Float ↔ f64 marshal の primitive (= =nl-float-payload= / =nl-float-make=) が無い.
- IEEE 754 epsilon comparison (=1e-15=) も pre-stdlib では表現不可.

** §2.2 Group 2: Syscall table (1 fn × 2 cfg)

実装位置: =strategy.rs= line 142-155.

呼出経路: =eval::builtins::dispatch= の =nelisp--syscall-nr-resolve= arm が =strategy::bi_syscall_nr_resolve= を call.  elisp 側 =nelisp-jit-strategy.el= は wrapper を持たず、直接 builtin 経由で呼ぶ.  Linux variant は =crate::eval::builtins::syscall_nr= 経由で =libc::SYS_*= を返す.

elisp 化阻害要因:
- =libc::SYS_*= は compile-time constant (= Rust 側 platform-dependent symbol catalog).
- elisp は build-time に Rust crate に直接アクセスできない.
- runtime に primitive 経由で問合せるのは可能だが、syscall name → number lookup は build-time constants の static table 化が圧倒的に効率的.

#+begin_example
#[no_mangle] pub extern "C" fn nl_jit_mut_str_len(s: *const NlMutStrRef) -> i64 {
    unsafe { (*s).value.chars().count() as i64 }
}

#[no_mangle] pub extern "C" fn nl_jit_bool_vector_len(v: *const NlBoolVectorRef) -> i64 {
    unsafe { (*v).value.len() as i64 }
}

#[no_mangle] pub extern "C" fn nl_jit_str_codepoint_at(s: *const Sexp, idx: i64) -> i64 {
    // 0..u32::MAX → codepoint, -1 → out-of-range
    let s = unsafe { &*s };
    let s_str = match s {
        Sexp::Str(rc) => &rc[..],
        Sexp::MutStr(rc) => &unsafe { &*rc }.value[..],
        _ => return -1,
    };
    if idx < 0 { return -1; }
    s_str.chars().nth(idx as usize).map(|c| c as i64).unwrap_or(-1)
}

#[no_mangle] pub extern "C" fn nl_jit_mut_str_set_codepoint(
    s: *const NlMutStrRef, idx: i64, cp: i64) -> i64 {
    // returns 0 on success, -1 on out-of-range or invalid codepoint
    unsafe {
        let rc = &*s;
        let new_ch = match char::from_u32(cp as u32) {
            Some(c) => c, None => return -1,
        };
        let chars: Vec<char> = rc.value.chars().collect();
        if idx < 0 || (idx as usize) >= chars.len() { return -1; }
        let new_str: String = chars.iter().enumerate()
            .map(|(i, c)| if i == idx as usize { new_ch } else { *c }).collect();
        rc.set_value(new_str);
        0
    }
}

#[no_mangle] pub extern "C" fn nl_jit_char_table_aref(
    t: *const NlCharTableRef, idx: i64) -> Sexp {
    unsafe { char_table_get(&*t, idx) }
}

#[no_mangle] pub extern "C" fn nl_jit_char_table_aset(
    t: *const NlCharTableRef, idx: i64, val: Sexp) -> i64 {
    unsafe {
        (*t).with_inner_mut(|i| char_table_set_one(i, idx, val.clone()));
    }
    0
}
#+end_example

* §11 Open questions (= reviewer 検討事項)

** §11.1 Float marshalling の代替設計

選択肢 A (本 Doc 採用): =nl-float-payload= / =nl-float-make= primitive で f64 marshal、trampoline は f64 直接受取.

選択肢 B: =Sexp::Float= を直接 trampoline argument にする (= =:trampoline-binary= の float variant).  marshalling primitive 不要だが、ABI signature が =fn(*const Sexp, *const Sexp, *mut Sexp) -> i64= で xmm レジスタ未使用 (= 性能退行リスク).

選択肢 A を推奨 (= xmm0/xmm1 直接使用、Float arith hot path の高速性維持).

** §11.2 Box accessor の payload pointer 露出

選択肢 A (本 Doc 採用): =nl-mut-str-payload= 等が =NlMutStrRef= の生 pointer (= =Box<UnsafeCell<...>>= の中身) を返す.  trampoline は raw pointer を受取り unsafe 操作.

選択肢 B: 全 box 操作を =nl-rc-*= primitive 経由に統一し、box 種類別 trampoline を作らない.  ただし =nl-rc-payload-ptr= が type-erased pointer を返すため、box 種類ごとの type cast が trampoline 内に必要.

選択肢 A の方が Stage 81.2/81.3 の cons/vector pattern と integrity 高い.

** §11.3 Stage 84.2 の build.rs runtime cost

build.rs が libc crate を pull する増 build time vs. runtime に primitive 経由で問合せる approach の trade-off.

build.rs codegen は **build-time コスト一回 vs runtime コスト無し** で、production binary 影響なし → 推奨.

** §13.1 Stage 84.3 Box accessor — payload-ptr design からの逸脱
