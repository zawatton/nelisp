* §0 要旨

Doc 18 §2.10 で Stage D MVP scope = "Phase 5-E 7 tool + *anvil-http full
port*" と LOCK 済。Phase 5-F.1 で anvil-state、Phase 5-F.2 で file/git/data

audit 結果 (§1.1) で anvil-http.el = 2452 LOC、ピュア Elisp port は

本 doc では:

wrapper、=nelisp-network.el= 経由)。これは超基本 GET only、cache や
extract 等の anvil-http の機能群を持たない。

- 既存 =http-get= は keep (smoke test と Phase 5-E 互換性)

* §4 実装計画 (Phase 6.2.1 着手時の指針)

** 4.1 cache key 戦略

anvil-http と互換 schema にする (=nelisp-state-source-tag= = "nelisp"
で source 列で区別):
- =:ns= = =anvil-http-cache=
- =:key= = =sha1(URL)= (=secure-hash 'sha1=)
- =:value= = plist =(:url URL :status N :headers ALIST :body STR :fetched-at INT)=
- =:ttl= = =anvil-http-cache-ttl-sec= 既定 86400

これで anvil + nelisp が同 DB 同 ns で coexist 可能 (各 source タグで区別)。

** 4.2 url-retrieve のラッピング

=url-retrieve-synchronously= は副作用 buffer 生成 + parse 必要。
nelisp-http--parse-response が:
- buffer から status line + headers + body 切り出し
- buffer kill (memory leak 防止)
- plist で返却

非同期 retry / backoff は MVP 範囲外 (Phase 6.2.1 は同期 only)。

** 4.3 ERT 移植戦略

anvil-http-test.el (= anvil 側、~1500 LOC ERT) のうち MVP 該当部分のみ
sample 移植:
- cache hit / miss / TTL expiry の 3 本
- HEAD method の 2 本
- cache management 4 本

dom / robots / batch / post test は本 phase scope 外、anvil 側 ERT を
そのまま keep (anvil 側 develop で継続維持)。

** 4.4 nelisp-http と既存 nelisp-network の関係

既存 =nelisp-network.el= = Phase 5-C.0 で追加した低レベル
=nelisp-http-get URL= primitive (network process 経由)。これは
*削除しない* — Phase 5-E の =http-get= tool が依存。

新 =nelisp-http.el= は url package 経由の高レベル (cache + retry なし
だが MVP では十分)。既存 nelisp-network はそのまま、nelisp-http が
別 namespace で並走。
