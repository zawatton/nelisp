* §0 TL;DR

** §0.1 macro assembler とは何か (= 1-line)

elisp DSL = =(emit-mov rax 42)= → bytes =0x48 0xb8 0x2a 0x00 0x00 0x00=
を返す pure-functional encoder。input = mnemonic + operand 群 (= elisp
symbol / integer)、output = byte list / unibyte vector。副作用なし、
syscall 呼ばず、libc にも libffi にも依存しない。

** §0.2 既に elisp 側にあるもの (= Phase 7.x prior art)

=src/nelisp-cc-x86_64.el= (2,861 LOC) と =src/nelisp-cc-arm64.el= (1,513
LOC) は **Phase 7.1.2 / 7.1.3 で既に SHIP 済** = native code emit を行う
JIT compiler の backend。本 doc が articulate するのは「**新規**
assembler を起こす」のではなく、「**既存 encoder を full self-contained
にする** = Rust helper 依存 (= 検査時点では実は **無い**) を確認した上で
Phase 47 用に opcode coverage を広げ、Doc 93 linker と握る relocation
marker API を追加し、Doc 91 writer に流せる format を整備する」こと。

残る Rust 依存は:
1. =build-tool/src/eval/ffi.rs= (820 LOC) の =libffi= / =libloading= /
   =CString= を経由した **任意 C 関数 invocation** (= JIT compile 結果の
   byte vector を実機で実行するために =mmap= → =write bytes= →
   =mprotect(PROT_EXEC)= → 関数 pointer 化 → 呼び出しの最終 stage で
   libffi 経由で C ABI dispatch する箇所)
2. =build-tool/src/jit/bridge.rs= (1,084 LOC) の =bi_nl_jit_call_*=
   trampoline 群 = elisp が emit した byte vector を「Rust 側 fn ptr に
   cast → C ABI で呼ぶ」boundary (= encoder 自体ではないが、encoder の
   output を実機で動かすための最後の Rust 接点)

本 doc が articulate する対象は **(2) の trampoline 経路を完全 elisp
化するために必要な encoder 拡張** = 「現在 1.4 と 1.5 で出力された
byte sequence を Rust 経由で実行している」依存を「elisp 自身が syscall
mmap して実行する」path に切り替えるための encoder side 整備。

* §2 Macro assembler design

** §2.1 設計方針

§1 の inventory から **encoder 自体は概ね既存**。本 doc が追加する extension は:

1. *opcode coverage 拡充*: Phase 47 の JIT trampoline path で必要な
   ~30 mnemonic のうち、既存 38 helper (x86_64) / 21 helper (arm64)
   でカバー済の subset を確認し、欠けている分を埋める (§2.2)
2. *register encoding table の参照容易化*: 既に exhaustive (16 entry
   x86_64 / 32 entry arm64)、このまま据え置き (§2.3)
3. *REX prefix computer*: 既存 =--rex-byte= 据え置き、ただし elision
   policy (= REX.WRXB 全 0 のとき省略可) を Doc 91 ELF writer 連携用に
   helper 化する option を §2.4 で articulate
4. *ModR/M + SIB byte encoder*: 既存 =--modrm-byte= 据え置き、SIB byte
   encoder を新規 (§2.5)
5. *Relocation marker API*: 新規 =nelisp-cc-asm-reloc= (§2.6) — 本 doc
   の主成果物、Doc 93 linker との handshake protocol
6. *binary buffer 化*: 既存 buffer は reverse list of bytes、Phase 47
   で millions of byte 単位の build で performance critical になる可能
   性、§2.7 で =unibyte-string= / =insert-byte= 切替の design

** §2.4 REX prefix computer (既存 =--rex-byte= 強化)

現状 =nelisp-cc-x86_64--rex-byte (W R X B)= は line 220-234 で定義、
constant 0x40 + W<<3 | R<<2 | X<<1 | B を返す。常に valid byte を返し、
**elision (= 全 0 のとき省略) は呼び出し側の責務**。Phase 47 拡張案:
=--rex-byte-with-elision= helper を追加 = =(BYTE . NEEDED-P)= を返す
~10 LOC。NEEDED-P が nil なら呼び出し側が prefix を drop。
ROI: =.text= byte count 削減 (= REX 不要箇所の 1 byte/instr 節約)。

** §2.5 ModR/M + SIB byte encoder

ModR/M = 既存 =--modrm-byte (mod reg rm)= line 236-245 据え置き。
SIB byte encoder (= =--sib-byte (scale index base)= = =(scale<<6) |
(index<<3) | base= + invariant check 3 つ) を新設、~15 LOC。SIB は
indexed addressing (= =[base + index*scale + disp]=) で必要。Phase
47 minimal scope では (a) [rbp-disp] / (b) [rip+disp] / (c) [reg+
disp] のみ使用、SIB 基本不要だが将来 array indexing 用に preemptive
配備。

** §2.6 Relocation marker API (= 本 doc の主成果物)

新規 =nelisp-cc-asm-reloc= struct:
- =TYPE= — =:r-x86_64-pc32= (= 4-byte rel32) / =:r-x86_64-64= (= abs64)
  / =:r-x86_64-plt32= / =:r-aarch64-call26= (= 26-bit BL imm) /
  =:r-aarch64-jump26= / =:r-aarch64-condbr19= (= B.cond) /
  =:r-aarch64-adr-prel-pg-hi21= (= ADRP) / =:r-aarch64-add-abs-lo12-nc=
- =SYM-NAME= — symbol naming the target (= Doc 93 linker resolves via
  =.symtab=)
- =OFFSET= — byte offset within current section (= first byte of the
  imm field, NOT the instruction start)
- =ADDEND= (optional, default 0) — matches ELF =Elf64_Rela.r_addend=
- =SECTION= (optional, default =:text=) — Doc 91 ELF writer dispatches
  into =.rela.text= / =.rela.data= per section

LOC: ~30 (= struct 12 + constructor 6 + invariant check 12).

buffer 拡張: 既存 =nelisp-cc-x86_64--buffer= に =relocs= slot を追加
した v2 struct + =--buffer-emit-reloc (buf type sym-name &optional
addend section)= helper を新設。emit-reloc は (a) reloc struct を
=relocs= list に push、(b) placeholder zero bytes (= 4 or 8 byte) を
byte stream に書き込み offset を進める。LABELS / FIXUPS は intra-
section (= 同 =.text= 内 local fixup) を継続担当、RELOCS は inter-
section (= =.rodata= constant、=.data= global、external symbol) を
扱う 2 層分離が Doc 91 ELF writer の =.rela.*= 出力との clean
boundary になる。

LOC: ~50 (= buffer-v2 struct + emit-reloc + arm64 sibling)。
合計 §2.6 LOC: **~80 elisp LOC**.

** §2.7 binary buffer optimization (= performance path)

現状 =--buffer-bytes= は **reversed list of integers**。Phase 47 で
typical NeLisp build:
- core elisp ~17,000 LOC × ~50 instr/LOC × ~5 bytes/instr ≒ 4 MB byte
- list-of-integers 4M × cons cell 16 byte = ~64 MB Emacs heap
- =--buffer-finalize= = =reverse= O(N) + =concat= O(N) → ~8M cons walk

代替設計: =--buffer-v3= struct (= include v2) で =bytes= slot を **grow-
only unibyte-string + fill pointer + capacity** に切り替え、容量
exhaustion で =make-string (* cap 2) 0= → =aset= で複製 → 容量倍化
(= amortized O(1) append)。=nelisp-cc-x86_64-fast-buffer= defcustom
で v1 / v3 を切替、既存 ert golden が v1 比較で生きる backward-
compat 維持。=finalize= は active variant を detect し dispatch。

LOC: ~80 (= struct + emit-byte + emit-bytes + finalize 切替 + arm64 同等)。

ROI: 4 MB build で list 経路 ~10s vs unibyte-string 経路 ~0.5s 見込み
(= benchmark 未実施、Doc 91 §91.d performance discussion と同じ orders
of magnitude を仮定)。

* §3 byte-level invariants

** §3.1 Endianness

x86_64: little-endian。すべての imm32 / imm64 / disp32 は LSB 先頭。
=--imm32-bytes= line 247-266 で =(low (>> 8) (>> 16) (>> 24))= の順。

arm64: little-endian。32-bit instruction word を 4 byte LE で並べる。
=--buffer-emit-instruction= line 551-564 で =(byte 0..3 of u32, LSB
first)=。

big-endian arm64 (= ARM legacy) は **OOS**、Linux / macOS は LE。

** §3.2 Prefix order

x86_64: =[Legacy prefix(es)] [REX] [Opcode] [ModR/M] [SIB] [Disp] [Imm]=

- Legacy prefix (= 0x66 / 0x67 / lock 等): Phase 47 minimal で不要
- REX: 0x40-0x4F、必ず opcode の **直前** (= legacy prefix と opcode の間)
- Opcode: 1-3 byte (= 0x0F escape 含む)
- ModR/M: opcode が要求するときのみ
- SIB: ModR/M.rm = 100b (= rsp/r12 base or scaled-index) のときのみ
- Disp: ModR/M.mod = 01 (= disp8) or 10 (= disp32) のときのみ
- Imm: opcode が要求する size (= imm8 / imm16 / imm32 / imm64)

既存 emitter 群はこの順序で list を構築 (= =(rex 0x89 modrm)=、
=(rex 0xC7 modrm imm32-bytes)= 等)、§3.2 invariant **既に準拠**。

arm64: 4-byte fixed-width、prefix 概念なし。命令毎に opcode field と
operand field が 32-bit word 内で固定 bit position (= 既存 encode-*
helper が直接 logior で組み立て)。

** §3.3 Opcode disambiguation

同じ mnemonic が operand type で異なる opcode に lower:

x86_64 例:

| mnemonic     | operand type   | opcode               |
|--------------+----------------+----------------------|
| MOV r64, r64 | reg, reg       | 0x89 (MR) または 0x8B (RM) |
| MOV r64, imm32 | reg, imm32   | 0xC7 /0              |
| MOV r64, imm64 | reg, imm64   | 0xB8+rd (10 byte)    |
| MOV r64, mem | reg, mem       | 0x8B (RM)            |
| MOV mem, r64 | mem, reg       | 0x89 (MR)            |

elisp 側の現状は **mnemonic-shape pair で別関数名** =
=--emit-mov-reg-reg= / =--emit-mov-reg-imm32= / =--emit-mov-rax-rip-rel=
等。auto-dispatch は無い。Phase 47 で macro-style entry point
=(emit-mov dst src)= を追加する場合、=src= の type で dispatch する
elisp macro が必要 (= ~30 LOC、§7 で articulate)。

arm64: addressing mode が固定 (= 命令ごとに operand encoding が一意)、
disambiguation はほぼ不要。例外: =MOV (= ORR/ADD/MOVZ alias)= は alias
chain をどの命令に解決するか implicit、現状 =--encode-mov-reg-reg= は
=ORR Xd, XZR, Xm= に固定済 (line 203-214)。

** §3.4 Instruction length restrictions

x86_64: 最大 15 byte。Phase 47 で生成する命令は最長 10 byte (= MOV
imm64) 以下、制約に触れない。

arm64: 完全固定 4 byte、長さ制約なし。

** §3.5 alignment

x86_64: byte alignment、命令境界に制約なし (= self-modifying code 含めて
任意 byte boundary 開始可)。

arm64: 命令は 4 byte aligned **必須** (= ARM ARM B1.1.1)。
=--encode-bl/b/cbz= 等の offset arg は =(zerop (logand offset #x3))=
check で enforce 済 (line 301-307 等)。

** §3.6 RIP-relative vs absolute

x86_64: RIP-relative は ModR/M.rm = 101b (= mod=00) で disp32 が
**次命令 から** の displacement。既存 =--emit-mov-rax-rip-rel= line
505-515 が `48 8B 05 <rel32>` で実装、Doc 93 linker が rel32 を patch。

arm64: pc-relative は LDR (= literal pool) / ADR / ADRP で実現。Phase 47
minimal scope では LDR literal pool は使わず、MOVZ + MOVK chain で
absolute address を materialize する path が単純 (= §2.2 MOVK 追加が
ここで efficacy 発揮)。

* §4 testing strategy

** §4.1 round-trip oracle

各 instruction encoder helper の検証は **disassemble-and-compare**
方針: =(emit-X args)= で得た byte list を =unibyte-string= 化 → temp
=.bin= file に =write-region= → =objdump -D -b binary -m i386:x86-64
$tmp= の出力を regex で mnemonic / operand 一致確認。arm64 は
=-m aarch64=。

** §4.2 fixture matrix

x86_64: MOV (reg-reg / reg-imm32 / reg-imm64 / mem-reg / reg-mem) ×
(legacy 8 reg + REX 8 reg) × (rax/rsp/rbp/r-rN base) ≒ 100 fixture。
ADD/SUB/CMP/IMUL ~30 each。制御転送 (CALL/JMP/JCC) ~20。

arm64: MOV (reg-reg / MOVZ / MOVK / MOVN) × 32 reg ≒ 120 fixture。
ADD/SUB/MUL ~50。BL/BLR/B/B.cond/CBZ ~30。

合計 ~400 ert fixture、各 1 sec 以内、make test-all 影響 ~7 min。

(= 0 Rust + 0 GNU) では検証 path 喪失、§6.4 risk で articulate。

各 encoder の invariant check (= imm32 out-of-range、unknown register、
misaligned arm64 offset 等) で ~50 negative fixture (=
=should-error= type =nelisp-cc-x86_64-encoding-error=)。

** §4.5 既存 test との関係

* §6 Risk / unknowns

** §6.1 性能 (= byte build hot path)

§2.7 で articulate: x86_64 instruction emit は ~10-50 byte/instr ×
1000s instr per JIT compile = millions of byte ops。elisp の
=unibyte-string= grow vs =insert-byte= vs =push to list + reverse= の
performance order:

- =push to list + reverse + concat=: ~64 MB heap, O(N) finalize
- =unibyte-string + grow-doubling=: ~4 MB heap, O(1) amortized append
- =insert-byte to temp buffer= (= text buffer): O(1) append via
  =insert-char=、Emacs buffer の specialized path

Doc 91 §91.d performance discussion で "elisp byte emit は GiB/sec
achievable" と articulate (= host Emacs の =unibyte-string= が C
implemented)。本 doc も同 assumption を継承。

懸念: byte-compile されない interpreted elisp で ~5x 遅くなる可能性
あり。Phase 47 toolchain は byte-compile 必須 (= AOT pipeline で
=.elc= 経由) として risk 軽減。

** §6.2 opcode coverage

Phase 47 minimal で必要な mnemonic は §2.2 で ~30 個と articulate。
ただし NeLisp の elisp source が evolve すると追加要件発生:
- string mutation primitive の inline = REP MOVS / REP STOS が要
- bignum 演算の inline = ADC / SBB (= carry chain) が要
- atomic op の inline = LOCK prefix + CMPXCHG が要

これらは Phase 47.x post-ship で都度追加。本 doc minimal scope は
**core integer + control flow** のみ。

** §6.3 arm64 parity

x86_64 38 helper / arm64 21 helper = arm64 が現状やや coverage 薄。
Phase 47 で linux-x86_64 を first ship target にするなら arm64 拡充は
deferred、ただし macos-arm64 を fall-behind させない方針を前提に
**両 platform で同 coverage** が望ましい (= §92.a / §92.b を並列実装)。

arm64 fixed-width 4-byte encoding は x86_64 variable-width より
**simpler** = LOC は同程度に収束 (§5.2 で 150/150 評価)。

** §6.4 disassembler dependency for testing

§4.3 で articulate: =objdump= host 依存。Phase 47 完遂後 (= 0 GNU
toolchain world) では検証 path が消える。代替案: (a) 各 fixture で
objdump 出力を 1 度取得し以降は byte vector を constant として比較
(= host-tool 依存ゼロ)、(b) 約 500 LOC の elisp 逆 disassembler を
別 doc で起こす (= post-Phase 47 enhancement)、(c) zydis / iced-x86
を bin として cargo workspace 削除前に oracle にする。Phase 47
minimal で (a) を採用。

** §6.5 既存 helper との backward-compat

§2.7 binary buffer 切替が **既存 ert fixture を破壊しない** ことが
gate 条件: =--buffer-bytes= が list-of-int → unibyte-string に変わる
が、=finalize= 戻り値 type (= unibyte-string) は不変、API surface
据え置き。risk: 既存 caller が accessor 直読みしている場合は破綻、
ただし grep snapshot では accessor 直読みは無し (= =finalize= 経由
のみ)。Phase 47 着手時に grep 再実行で確認。

** §6.6 implicit register liveness

macro-style auto-dispatch entry =(emit-mov dst src)= は operand type
で opcode 選択 = elisp runtime function (= integer? symbol? cons-of-
disp?) で十分、macro 不要。implicit flag clobber (= ADD は OF/SF/ZF
触る、MOV は触らない) は user の責務、本 doc の encoder layer は
flag 可視化を持たない (= register allocator との handshake で対処)。
