* 1. Purpose

This document defines the efficient development order for the current NeLisp
performance work:

- runtime image/cache
- native compile hot path
- GC/allocator pressure reduction

The immediate goal is not to add more surface area.  The goal is to make the
fast path measurable, keep it stable, and then move the remaining fixed costs
out of per-command startup.

* 2. Current State

The source-command cache path is now default-on after =make standalone-reader=
and materially faster than the marker-disabled full source fallback for a tiny
adjacent =.neln= smoke:

#+begin_example
cache_default_load rc=0 ms=2251 out=bench-native-smoke
cache_default_eval rc=0 ms=2906 out=42
fallback_load rc=0 ms=13238 out=bench-native-smoke
fallback_eval rc=0 ms=14471 out=42
cache_source_fallback_load rc=0 ms=841 out=source-fallback
cache_source_fallback_eval rc=0 ms=921 out=42
#+end_example

The cache path improves startup by:

- bypassing the full =lisp/nelisp-artifact.el= command runtime;
- bypassing general artifact runtime cache replay for source commands;
- reading only =:format= and =:features= eagerly from adjacent artifacts;
- replaying =:module-init= item-by-item instead of materializing the full
  module list;
- skipping =:native= object/base64 payload parsing on the startup-oriented
  source-command path.

The repeatable local gate is now:

#+begin_src sh

tools/nelisp-nelix-command-gate.sh
tools/nelisp-native-artifact-gate.sh
tools/nelisp-nelix-native-hot-gate.sh

tools/nelisp-source-command-substrate-gate.sh

It emits =gate_result= wall-time lines, =gate_stats= arena-stat lines, verifies
cache fallback/restore behavior, and runs the focused artifact ERT set.  The
Nelix command gate uses DSL v1 =nelix-environment= manifests and a fake Nix
profile so command output and timings are repeatable without touching the
user's real profile.  It covers =list=, =audit=, =upgrade-plan=, =plan
--dry-run=, and =apply --dry-run= through both the AOT cache lane and the
direct standalone NeLisp lane.  The native artifact gate verifies the generic
=.el= -> adjacent =.neln= contract through the =target/nelisp= command
itself.  The runtime-image cache gate verifies the runtime-image
cache/freshness contract.  The source-command substrate gate keeps exposing the
exact evaluator/reader/bytecode source as an independent whole-substrate
research unit.

The Nelix command gate also keeps a 512-target manifest fixture on both the
production AOT cache lane and the direct =:nelisp-fast= lane.  For the AOT
cache route, =audit= and =upgrade-plan= must report the expected present,
missing, extra, upgrade, and pinned sets with =:nelisp-aot-cache=.  For the
direct route, the same fixture remains an allocator-pressure diagnostic with
explicit arena delta limits.

Current 512-target gate evidence:

#+end_example

Interpretation: the production AOT cache lane handles larger manifests
quickly, while the direct =:nelisp-fast= diagnostic still spends most of its
allocator budget in generic result formatting/printing.  The gate keeps that
pressure visible and bounded while later work moves more JSON/S-expression
output onto compact native/AOT paths.

The current native artifact contract is generic but not yet full-native:

- every readable =.el= should be compilable to a valid =.neln= cache;
- native-eligible top-level =defun= forms are recorded in the =:native=
  section and installed through the normal native-first loader;
- unsupported forms and unsupported =defun= bodies remain covered by bytecode
  / module-init fallback;
- =:native-report= records coverage gaps so callers can audit them;
- =--native-policy required= is the CI gate for hot modules that must not have
  native coverage gaps.

This distinction is intentional.  A universal =.neln= cache format prevents
per-file special cases, while required native policy is used only where the
performance contract demands it.

User-facing policy: arbitrary package code should enter the same generic
=.el= -> =.neln= pipeline, not a Nelix-only special case.  The default
artifact is allowed to be mixed native/fallback so package installation stays
robust across platforms and language coverage gaps.  Hot modules are promoted
only after =--native-policy required= and =audit-elisp-artifacts --required=
prove that the required defuns have native coverage and that normal source
loading will use the adjacent artifact.

Interpretation: the generic CLI can bulk-build adjacent =.neln= files, enforce

The largest remaining startup cost in source commands is still the
evaluator/bytecode substrate bootstrap, but the default source-command route no
longer replays it from raw source.  The generated artifact runtime cache
contains a filtered command runtime and installs source-command fast functions
from the cache:

- =target/nelisp-artifact-runtime.el.nelc=
- =target/nelisp-artifact-runtime.el.nelc.enable=

Deleting the enable marker forces the full source fallback.  Re-running
=make standalone-reader= rebuilds the cache and restores the enable marker
with a sha256 line, making the cache/fallback state explicit.

The runtime-image artifact cache is correct but not a performance win for tiny
images:

Interpretation: the cache creates a sibling artifact, records the runtime-image

stale artifact after the image changes.  It is slower for this tiny fixture

therefore a correctness/freshness gate, not the source-command startup fix.

A tiny eval-only artifact does compile and load correctly, so the policy and
CLI path are functional.  The remaining whole-substrate blocker is the cost of
reading and processing real substrate Elisp in =target/nelisp=, not only
bytecode lowering.  This is useful negative evidence: the substrate boundary is
explicit and reproducible, but the current artifact compiler/reader path is too
slow to make whole-substrate compile a default gate.  The production Stage B
path uses the filtered artifact runtime cache instead; whole-substrate compile
remains open until either:

- the substrate is split into smaller cacheable chunks;
- the artifact compiler/reader can compile the whole substrate within the gate
  budget;
- or the cache path can load a lower-level precompiled representation without
  replaying 370 KB of source forms.

An opt-in =--profile-stages= flag now emits stage timings from
=compile-elisp-artifact=.  A second opt-in =--profile-forms= flag emits
per-form reader timings as =artifact_profile_form= lines.  On a tiny two-form
eval-only artifact, both profile paths are functional.

The first reader bottleneck is now closed for non-per-form-profiled artifact
compiles: =nelisp--read-all-from-string-native= is callable from interpreted
Elisp in the standalone command context and is used by
=nelisp-artifact--read-top-level-forms= when =--profile-forms= is disabled.
On =src/nelisp-macro.el=, the former =read-forms= cost collapsed from tens of
seconds to single-digit milliseconds:

#+begin_example

artifact_profile stage=read-forms elapsed_ms=9.5 detail=(:forms 23 ...)
artifact_profile stage=module-build elapsed_ms=39.5 detail=(:forms 23 :module-policy eval-only)

Before the native read-all bridge, the same file spent about =55651.5ms= in
=read-forms=.  The new path makes reading practical for ordinary file-sized
inputs, but it does not make whole-substrate artifact compilation practical
yet.  The next optimization target is therefore artifact serialization and
whole-substrate materialization, especially =artifact-string= and any list
printing it forces, not the bytecode compiler.

Per-form profiling on =src/nelisp-macro.el= shows that the cost is not isolated
to one malformed form.  The later forms trend slower, which is consistent with
the current standalone =read-from-string= implementation reparsing or copying a

#+end_example

The portable per-form reader path remains available when =--profile-forms= is
enabled because it emits one timing row per top-level form.  For normal
artifact compilation, the default is now the native read-all bridge.

The serializer now exposes nested =artifact-module-string= and
=artifact-wrap-string= profile rows when =--profile-stages= is enabled.  This
confirmed that the remaining =artifact-string= cost is concentrated in
=:module-init= printing:

#+begin_example
artifact_profile stage=artifact-module-string elapsed_ms=5157.5 detail=(:source-slices nil :items 23 :bytes 6835)
artifact_profile stage=artifact-wrap-string elapsed_ms=202.5
artifact_profile stage=artifact-string elapsed_ms=5390.5 detail=(:bytes 7282)
#+end_example

Three compact-writer trials were checked and rejected as the default artifact
format:

- per-form source slicing avoids object printing, but the standalone Elisp
  scanner took about =19969.5ms= on =src/nelisp-macro.el=, worse than printing;
- one-item source replay =((:eval (progn ...)))= reduced
  =src/nelisp-macro.el= =artifact-string= to about =215.5ms= and let the
  368 KB substrate compile in about =5.4s=, but normal artifact load no longer
  preserved =defun= bindings for required-native =.neln= artifacts.  That
  violates the generic artifact contract, so it is not used by
  =nelisp-artifact-compile-file=;
- raw eval-source blocks avoid both source scanning and suffix scanning, and
  reduce =src/nelisp-macro.el= load/eval to about =793ms= when used in a
  standalone direct artifact.  2026-06-19 follow-up fixed one concrete raw
  replay defect: non-ASCII docstrings/comments now serialize into an
  ASCII-readable raw block, with string contents preserved through =\u= / =\U=
  escapes.  This moved the full-substrate raw failure past the early
  =dolist= docstring bootstrap.  The same raw block path still fails on the
  full source-command substrate artifact, however.  Prefix probes show that
  the full self-replay reaches the evaluator bootstrap and then fails at, or
  after, the top-level =(nelisp--install-primitives)= call; removing that call
  still produced a segfault.  This means the current blocker is no longer just
  raw reader syntax.  It is the unsafe act of re-evaluating the running
  evaluator/source-cache substrate into itself.  Therefore
  =nelisp-artifact-raw-eval-source-threshold= remains intentionally =nil= by
  default.  The parser/loader scaffold remains for the next isolated fix, but
  standard artifacts stay on the safe per-form replay representation.

The next successful Stage B fix likely needs a compact representation that
preserves the normal per-form replay semantics, or a lower-level artifact
loader that can install definitions without printing and rereading large Elisp
lists.  It must also avoid full evaluator self-replay; a correct split should
load only the missing source-command layer into the direct artifact command, or
move the minimal artifact loader closer to the standalone driver so the full
substrate artifact is not evaluated inside a copy of itself.

* 3. Development Principles

** 3.1 Separate speed work from safety work

Performance experiments must not be mixed with unrelated feature work.  Each
iteration should have:

- one speed hypothesis;
- one measurement;
- one implementation change;
- one verification set;
- one design/worklog update.

** 3.2 Fix gates before chasing more speed

Every change to this area should preserve a small, repeatable gate:

#+begin_example
normal_load
normal_eval
cache_load
cache_eval
cache_source_fallback_load
cache_source_fallback_eval
focused artifact ERT
marker_absent
#+end_example

The gate should report wall time and, once available, arena/GC stats.

** 3.3 Keep marker and normal loader roles distinct

The normal =.neln= loader is the correctness/native-dispatch path:

- validates manifests;
- installs native-first wrappers;
- preserves native coverage diagnostics;
- supports broader artifact operations.

The marker source-command path is the startup path:

- loads adjacent =.neln= / =.nelc= quickly;
- skips unused native object payload;
- prioritizes startup allocation reduction;
- has a compact direct source fallback for artifact misses;
- may use bytecode payload only unless a later fast native dispatch path is
  explicitly added.

Do not blur these roles without a dedicated design change.

* 4. Recommended Work Order

** 4.1 Stage A - Add performance and arena gates

Add an opt-in measurement path for:

- wall time;
- arena stats before source-command bootstrap;
- arena stats after substrate bootstrap;
- arena stats after artifact load;
- arena stats after requested form evaluation;
- optional =garbage-collect= result.

The output should be machine-readable and disabled by default.  Suggested
activation:

#+begin_example
touch target/nelisp-source-command-cache.stats
target/nelisp eval-elisp-source FILE.el FORM
rm -f target/nelisp-source-command-cache.stats
#+end_example

Stats are intentionally marker-based for now.  The standalone source-command
path does not yet have a safe OS-environment reader at the point where the
compact marker loader runs; direct raw =envp= scanning is unsafe in this
bootstrap position.  The artifact runtime cache enable marker is no longer a
stats opt-in: =make standalone-reader= writes it together with the cache
artifact, and deleting it is the supported fallback switch.

Acceptance:

- normal output remains unchanged when the stats marker is absent;
- stats appear on stderr when enabled;
- stats marker file is not left behind;
- focused artifact ERT stays green.

** 4.2 Stage B - Precompile the evaluator substrate

The original marker path source-evaluated the evaluator/bytecode substrate.
That was the main startup cost.

Implement one of these in order of lowest risk:

1. compile the substrate to a private =.nelc= artifact and replay it;
2. compile the substrate to a private =.neln= artifact where native-eligible
   substrate functions are wrapped;
3. bake the substrate into a runtime image format that avoids source read/eval;
4. move the adjacent-artifact loader closer to the standalone driver.

Acceptance:

- source-command cache path remains correct for =load-elisp-source= and
  =eval-elisp-source=;
- cache load/eval smoke improves over the source fallback baseline;
- substrate cache freshness is explicit and rejects stale input;
- fallback behavior is documented.

Current status:

- =make standalone-reader= now builds =target/nelisp-artifact-runtime.el.nelc=
  and writes =target/nelisp-artifact-runtime.el.nelc.enable= with the cache
  sha256.  With that marker present, source commands use the filtered
  artifact-runtime cache path by default;
- deleting the enable marker forces the full source fallback, and rebuilding
  restores the marker.  =tools/nelisp-performance-gate.sh= proves both paths;
- measured 2026-06-19 gate sample: default cache =eval-elisp-source= around
  =2906ms=, marker-disabled fallback around =14471ms=, source fallback with
  no adjacent artifact around =921ms=;
- runtime-image cache freshness is now covered by
  =tools/nelisp-runtime-image-cache-gate.sh=;
- source replay vs artifact replay timings are emitted by that gate;
- the real evaluator substrate source can now be emitted without command
  dispatch by =nelisp-standalone--artifact-source-command-substrate-src=;
- =--module-policy eval-only= exists for large replay-only bootstrap caches;
- whole-substrate eval-only =.nelc= compile through =target/nelisp= is still
  too slow for default gating and raw full-substrate self-replay remains
  unsafe.  That is now tracked as a lower-level representation research item,
  not the required Stage B runtime path.

** 4.2.1 Generic native artifact contract

Before adding more Nelix-specific optimization, preserve the generic artifact
contract:

#+begin_example
target/nelisp compile-elisp-artifact --kind neln --input FILE.el --output FILE.el.neln
target/nelisp compile-elisp-artifacts --kind neln DIR
target/nelisp audit-elisp-artifacts [--required] DIR
target/nelisp load-elisp-source FILE.el
target/nelisp eval-elisp-source FILE.el FORM
#+end_example

Expected behavior:

- the compile commands work for arbitrary readable =.el= files;
- adjacent =FILE.el.neln= is preferred over source replay;
- the normal loader uses native-first wrappers when =:native= exists;
- bytecode/module-init fallback preserves correctness when native coverage is
  incomplete;
- =audit-elisp-artifacts= reports missing artifacts, invalid artifacts, and
  native coverage gaps for directories and individual files;
- =audit-elisp-artifacts --required= exits non-zero unless every audited
  adjacent =.neln= has complete native coverage for the defuns recorded in its
  manifest;
- =--native-policy required= fails before writing partial artifacts when any
  top-level =defun= cannot enter native code.

This is the long-term maintainability rule: Nelix and user packages should not
grow one-off native handling.  They should emit ordinary =.el=, call the common
artifact builder, and let =audit-elisp-artifacts=, =:native-report=, and
required policy expose missing compiler coverage.

** 4.2.2 Native compile status and policy

The current answer to "can every =.el= be native compiled?" is deliberately
two-layered:

- every readable =.el= should be buildable into a reusable =.neln= artifact;
- a =.neln= artifact is not native-only: it carries portable replay/bytecode
  fallback plus any native functions the compiler can prove;
- normal loads prefer adjacent =.neln= and install native-first wrappers when
  =:native= metadata exists;
- =--native-policy required= is the enforcement mechanism for hot modules,
  not the default for arbitrary user files.

This keeps the system generic.  Nelix should not add a new special path every
time a module becomes performance-sensitive.  Instead, the module is compiled
with the common artifact builder, audited with =audit-elisp-artifacts
--required=, and only promoted into the hot path after the required-native gate
passes.

Native-only for all Elisp is not yet true.  Unsupported top-level forms,
unsupported defun bodies, dynamic constructs, and platform-specific lowering
gaps still use fallback.  Treating those files as "compile failed" by default
would make ordinary package loading brittle.  Treating them as valid =.neln=
with an explicit =:native-report= gives us a scalable migration path while
still allowing CI to fail for the modules that must be fast.

This is the answer to the maintainability concern: do not patch every future
module by hand.  Package/profile installation should bulk-build adjacent
=.neln= artifacts with the generic builder, load through the normal
native-first loader, and run a required-native audit only for the modules that
are declared performance-critical.  The remaining work is to make the generic
standalone builder fast enough that this is practical for daily use.

** 4.4 Stage D - Isolate Nelix hot modules

Move the command-specific hot operations into small modules:

- manifest target name extraction;
- package/profile name set construction;
- audit set difference;
- upgrade-plan set difference;
- compact line/JSON output writers.

Compile these with:

#+begin_example
target/nelisp compile-elisp-artifact --kind neln --native-policy required ...
#+end_example

Acceptance:

- hot modules pass =--native-policy required=;
- normal loader installs native-first wrappers;
- marker path remains startup-oriented unless a specific native fast dispatch
  is designed for these modules.

Current status:

- =scripts/nelix-aot-native-cli-proof.el= passes required native coverage
  through the common =.neln= artifact builder.
- =scripts/nelix-aot-native-subset.el= passes required native coverage through
  the same common artifact builder for the current bitset/ID hot kernel set.
- =target/nelisp audit-elisp-artifacts --required= can verify the small
  =cli-proof= artifact generated by that builder.
- =target/nelisp native-exec-elisp-artifact= can execute the public
  =cli-proof= string lane from the generated =.neln= artifact.

working fix is lower in the reader driver dispatch layer, before the generated
artifact command source is evaluated at all.

** 4.5 Stage E - Reduce allocator pressure from data shape

Avoid plist/list/alist-heavy processing in Nelix hot paths.  Prefer:

- name-only lists where possible;
- hash/name-set indexes;
- compact records;
- minimal JSON/line writers;
- single-pass comparison routines.

Acceptance:

- arena stats show lower allocation after each command phase;
- command output stays stable;
- data-shape changes are covered by focused tests.

Current status:

- =nelix-fast.el= uses name-only profile paths, hash/name-set indexes, compact
  manifest targets, and direct JSON writers for the standalone NeLisp
  =list=, =audit=, and =upgrade-plan= paths;
- the AOT cache line protocol keeps package names, target IDs, installed IDs,
  pin flags, and output rows compact instead of expanding the full Emacs Lisp
  manifest shape;
- =tools/nelisp-nelix-command-gate.sh= now verifies output stability and
  machine-checks allocator pressure in the post-load command phases.

This closes Stage E for the fixture command path.  The remaining long-term
allocator work is to lower the fixed standalone preload itself and repeat the
same style of delta gate on larger real-world manifests.

** 5.8 Artifact load and direct CLI fast path

Host Emacs can compile the full source-command substrate into a private
artifact quickly when the substrate is treated as ordinary Elisp input:

#+begin_example
compile_substrate_bytecode_artifact total=42.4ms bytes=261604 forms=682
compile_substrate_eval_only_artifact total=23.8ms bytes=245192 forms=682
#+end_example

Before the load fast path, =target/nelisp eval-elisp-artifact= timed out at
=120s= on both eval-only and bytecode substrate artifacts.  The failure was not
native compilation itself; it was runtime materialization:

- full payload parsing read the whole =:module-init= plist before replay;
- command invocation could also choose the large artifact command source path
  before it ever reached the user artifact.

The loader now has a generated-private streaming path:

- scan generated keyword positions for load-critical keys;
- replay =:module-init= item by item instead of materializing the whole list;
- use the same replay helper for full-parse fallback and streaming load;
- expose opt-in =nelisp-artifact-profile-load= stage logging for future
  loader profiling.

The standalone driver also has direct =eval-elisp-artifact= /
=exec-elisp-artifact= source generation.  These commands now use the compact
source-command loader substrate, load the requested artifact, and evaluate
FORM strings from =nelisp-standalone-argv=, avoiding the heavyweight artifact
command source when compile/audit functionality is not needed.

It does not yet mean every Elisp form is natively lowered.  Unsupported
top-level effects and unsupported defuns still fall back to bytecode/eval
according to the artifact manifest and native audit report.

* 6. Near-Term Roadmap

The most efficient next sequence is now:

1. keep the default source-command artifact runtime cache enabled by
   =make standalone-reader= and prevent regressions with
   =tools/nelisp-performance-gate.sh=;
2. keep =eval-elisp-artifact= / =exec-elisp-artifact= on the direct compact
   loader path and continue reducing its substrate startup cost;
3. profile and accelerate the =target/nelisp= reader/artifact compiler path for
   real substrate Elisp, especially =read-forms= and =artifact-string=, or
   introduce a lower-level precompiled representation;
4. compare the default filtered runtime cache against any future
   whole-substrate artifact/image representation before replacing it;
5. use the new =NELIX_NELISP_STATS=1= direct-runner phase stats on larger real
   user manifests and compare against the fixture gate;
6. promote the larger native-subset public standalone execution path after it
   fits the same cached general driver budget as the =cli-proof= lane;
7. expand native compiler coverage until more hot modules pass
   =--native-policy required= without local workarounds;
8. reduce the fixed standalone preload allocation before =after-cli-load=.

* 7. Completion Criteria

This work is complete for the default source-command path and fixture-backed
Nelix command path when all three goals are proven:

- runtime image/cache: source-command and real Nelix commands avoid per-command
  full source replay and have repeatable freshness/fallback behavior;
- native compile hot path: hot Nelix modules pass native-required gates and are
  used by the normal native-first loader;
- GC/allocator pressure: arena stats show reduced allocation for startup,
  artifact load, and command execution phases.
