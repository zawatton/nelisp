* §0 TL;DR

Doc 77c Phase A 完遂 (= 全 boxed Sexp variant の =NlXxxRef= 化 + =#[repr(transparent)]= + =Sexp::*_box_ptr()= 直接 accessor SHIPPED) により、JIT trampoline 層を elisp wrapper で書換える FFI 基盤が揃った.  Doc 77c §2.4 articulation を refine し、**Phase D-A = Cranelift Rust crate を約 38 個の C-style FFI shim で expose し、IR builder を elisp 駆動に置換する** path を本 Doc で詳細設計する.

D-A 着地時の core 構成:
- Rust 側: cranelift crate + 約 38 個の =cl_*= 薄い shim function (~400 LOC、unsafe 集中)
- elisp 側: =nl-cl-*= primitive 経由で IR を組立、=cl_jit_finalize= で実行ポインタ取得
- 削除対象: =build-tool/src/jit/{cons,access,arith,predicate,syscall,dsl}.rs= の builder 関数群 (~3000 LOC) + =bridge.rs= の half (~400 LOC)

合計 net delta: **-~2400 LOC Rust** (= 既存 4733 LOC jit/ → 残 ~400 LOC shim + ~600 LOC bridge bookkeeping = ~1000 LOC).  対応する elisp 側 LOC = +~1500 LOC (= IR builder DSL + 既存 strategy.el の再配線).

工数: 1-2 週間 (= Doc 77c §2.4.1 推定維持). Phase B / C 完了後に着手、ただし shim 設計は前倒し可能.

* §1 Background

** §1.1 Phase A で揃った FFI 基盤 (= 2026-05-09 着地)

Phase A.1〜A.5 で以下が確定:
- 全 7 boxed Sexp variant が layout-pinned struct (=NlConsBox / NlCell / NlStr / NlVector / NlBoolVector / NlRecord / NlCharTable=) + =#[repr(C)]= + =value(s) @ offset 0, refcount @ trailer= shape
- 7 件の =NlXxxRef= が =#[repr(transparent)]= で =NonNull<NlXxx>= に固定
- =Sexp::tag()= で discriminant byte read、=Sexp::*_box_ptr()= で box pointer 取得
- =SEXP_PAYLOAD_OFFSET = 8= 定数 + compile-time layout assertion
- =nl-cons-*= / =nl-rc-*= 8 件 elisp primitive (Phase A.3 SHIPPED)

これにより elisp 側から:
1. raw =Sexp= 値の variant tag が読める (= =nl-sexp-tag= primitive 1 行 wrap)
2. raw box pointer が elisp の int 値として手に入る (= =nl-cons-box-ptr= 等の wrapper、現状 nl-cons-car 経由で間接)
3. refcount を elisp 駆動で増減できる (= =nl-rc-inc/-dec/-count= SHIPPED)

これは Phase B (Sexp 操作 elisp 化) には十分.  Phase D-A はさらに **cranelift IR generator を elisp 駆動にする** ので、別軸の FFI surface 拡張が必要 = 本 Doc の主題.

** §1.2 Phase D-A vs D-B vs Doc 77c §2.4 articulation

Doc 77c §2.4 では:
- D-A = Cranelift wrapper (= "~50 fn 薄い shim")、1-2 週間
- D-B = Cranelift 完全置換 (= elisp が x86_64/ARM64 byte sequence 直 emit)、3-6 ヶ月

本 Doc は D-A に焦点.  D-B は将来別 Doc (= Doc 79 候補) で articulate.

D-A を選ぶ理由:
- Cranelift の register allocator / instruction selection / ELF linkage は production quality (= Bytecodealliance + wasmtime と共有)、これを elisp で書換えるのは ROI 悪
- 完全 self-host が gate でない (= "Cranelift binding crate に依存" は許容、Cranelift が Rust crate であることは elisp self-host の障害ではない、cranelift の internals に手を入れていないので "external library として使う" 立ち位置)
- 1-2 週間で着地する見込み = Phase B/C 完了後の自然な流れ

* §3 State management 設計

cranelift API は state-ful (= =FunctionBuilder= は =&mut Context= 経由、=Module= は =&mut JITModule=).  elisp 側でこれを保持する方法:

** §3.1 案 A: opaque pointer + nl-rc 経由 (= 推奨)

各 cranelift state を =NlRc<Box<T>>= で wrap:

#+begin_src rust
#[no_mangle]
pub unsafe extern "C" fn cl_make_module() -> *mut std::ffi::c_void {
    let module = JITModule::new(JITBuilder::new(...));
    let boxed = Box::into_raw(Box::new(module));
    boxed as *mut _
}

#[no_mangle]
pub unsafe extern "C" fn cl_drop_module(handle: *mut std::ffi::c_void) {
    drop(Box::from_raw(handle as *mut JITModule));
}
#+end_src

elisp 側は =Sexp::Int(handle as i64)= で保持、=cl-drop-module= 呼出責任を持つ.  または =NlRc<JITModule>= で wrap して既存 =nl-rc-*= primitive 経由で refcount management.

** §3.2 案 B: elisp-side state machine

Cranelift state を Rust に持たず、IR を elisp side で alist として組立、=cl_compile_module= 1 fn で一括 lower.  問題: Cranelift API は callback-style で =FunctionBuilder= の各 ins() が直 IR emit、これを deferred batch にすると elisp 側で IR DSL を full reimplement する必要 = D-B に近い分量になり D-A の趣旨と外れる.

→ **採用 = 案 A**.

** §3.3 handle ライフサイクル

| handle 種別        | alloc                | drop                 | refcount   |
|--------------------+----------------------+----------------------+------------|
| =JITModule=        | =cl_make_module=     | =cl_drop_module=     | nl-rc 任意 |
| =Context=          | =cl_make_context=    | =cl_clear_context=   | nl-rc 任意 |
| =FunctionBuilder=  | =cl_make_builder=    | =cl_finalize_builder= | nl-rc 任意 |
| =Signature=        | =cl_make_signature=  | (= module 所有)       | -          |
| =FuncId=           | =cl_declare_function= | (= module 所有)       | -          |
| =Block= / =Value= / =Inst= | (= ID は単なる u32) | -                    | -          |

elisp 側は =Sexp::Int= で raw handle (= 数値) を保持.  =Block= / =Value= / =Inst= は cranelift 内部 ID なので drop 不要.

* §4 elisp DSL shape

** §4.1 直書き shim 呼出 (= primitive level)

#+begin_src elisp
(let* ((mod (cl-make-module))
       (sig (cl-make-signature mod)))
  (cl-sig-add-param sig cl-type-i64)
  (cl-sig-add-param sig cl-type-i64)
  (cl-sig-add-return sig cl-type-i64)
  (let* ((fn-id (cl-declare-function mod "add2" cl-linkage-local sig))
         (ctx (cl-make-context mod))
         (fb (cl-make-builder ctx))
         (entry (cl-create-block fb)))
    (cl-append-block-params fb entry)
    (cl-switch-to-block fb entry)
    (let* ((a (cl-block-params fb entry 0))
           (b (cl-block-params fb entry 1))
           (sum (cl-emit-iadd fb a b)))
      (cl-emit-return fb (vector sum)))
    (cl-seal-block fb entry)
    (cl-finalize-builder fb)
    (cl-define-function mod fn-id ctx)
    (cl-clear-context mod ctx)
    (cl-finalize-definitions mod)
    (cl-get-finalized-function mod fn-id)))  ; → *u8 ptr
#+end_src

(= ~20 行で 2-引数 add 関数を JIT compile)

* §7 Risks

** §7.1 cranelift API drift

cranelift crate は upstream で API 変更頻度高 (= 2024-2026 で =FunctionBuilder= の seal API 等が複数回変更).  shim 経由 = elisp side が isolation されるが、shim Rust 側で API 追従が必要.

Mitigation: shim 数を最小限に抑える (= ~38)、cranelift version pin.

** §7.2 Performance regression

shim 経由 = 各 IR instruction 1 個に対し =nl-ffi-call= 1 回 = function call overhead 加算.  典型 trampoline は ~20 IR ins = 20 ffi-call.  Phase A 時点の bench で許容範囲内か検証必要.

Mitigation:
- =nl-ffi-call= overhead の bench (= empty fn 1 call の cost を測定)
- 必要なら hot path だけ Rust に残す (= "Rust core" 約 500 LOC まで膨らむ妥協)
- IR emit を batch 化 (= elisp 側で IR を一旦 alist に組立、=cl_emit_batch= 1 call で渡す = 案 B 折衷)

** §7.3 Debugging difficulty

cranelift IR の検査は Rust 側で =cranelift-codegen= の DSL を使うのが標準.  elisp 側 IR は alist で持つしかなく、debug print が劣る.

Mitigation: =cl_dump_module= / =cl_dump_ir= shim を追加、elisp から =(cl-dump-module mod)= で stdout に Cranelift native fmt を出力可能に.

** §7.4 unsafe surface 増加

shim 38 fn は全 =#[no_mangle] pub unsafe extern "C"= = audit 表面が広がる.  elisp 側のバグ (= 不正 handle 渡し) が UB に直結.

Mitigation:
- 各 shim 入口で handle null-check + magic-number check (= each 1 LOC)
- miri + cargo-fuzz で shim の random input fuzzing
- 案 A の =NlRc<Box<T>>= 化で handle lifetime を refcount 管理

* §9 Decision points

Phase D-A 着手前に以下を resolve:

1. **Phase B 完了が必須 gate か?** (= §3.1 dependency graph 上は yes、ただし B が =strategy.rs= 削除で D-A の dsl.rs 削除に近い shape なので並行可能性あり)
2. **shim 数の妥協範囲** (= 案 A の opaque ptr + 38 fn vs. batch IR の ~10 fn): Performance bench 結果次第
3. **elisp 側 IR debug 機能の優先度** (= =cl_dump_module= の必須度): 小規模なら yes、大規模なら =disasm= shim も追加
4. **cranelift version pin 戦略**: =Cargo.toml= で =cranelift = "=0.x.y"= として shim audit を担保
