* §0 TL;DR

** §0.1 ELF writer とは

ELF writer = source program の compile 結果 (= 機械語 + symbol +
relocation) を **OS が exec() できる ELF64 binary file** として disk に
書き出す component。input は assembler の output (= section 群 + symbol
table + relocation entry の memory representation)、output は ELF64
形式の単一 file (= =./nelisp= のような実行可能 binary)。

責務:
1. ELF64 header (Ehdr) emit = magic / arch / entry point address
2. Program header (Phdr) emit = OS loader が =mmap= する segment 定義
3. Section header (Shdr) emit = =readelf= / =objdump= が見る section
   metadata
4. Section bytes 書き込み = =.text= (機械語) / =.rodata= (定数) /
   =.data= (初期化済 global) / =.bss= (zero-fill) / =.symtab= /
   =.strtab=
5. Relocation entry emit = =.rela.text= / =.rela.data=

** §0.2 elisp で書ける根拠

ELF writer は **pure-functional bytewise serialization**:
- input = section 群 (= elisp data structure)
- output = byte sequence (= unibyte string)
- 副作用 = =write-region= で file へ flush 1 回のみ
- syscall は =write(2)= と =chmod(2)= (= +x bit) のみ、他の OS surface
  に依存しない

elisp で =insert-byte= / =unibyte-string= / =write-region= が安定的に
動けば実装可能。性能は §7.1 risk として articulate するが、
correctness 上の障害なし。

** §0.3 なぜ必要か

現状: =cargo build --release= が =target/release/nelisp= ELF binary を
生成している。これは Rust toolchain (= =rustc= + =ld= + =cargo=) を
**毎回の build で必要** とする。

Phase 47 = 0 Rust runtime 完遂時:
- =./nelisp= は **elisp toolchain によって生成** される必要
- 既存 =./nelisp= が source elisp を read → assembler が x86_64 機械語
  emit → linker が symbol resolve → **ELF writer が disk 上の ELF
  binary に書き出す** という self-host loop
- このループの最終 stage が本 doc が articulate する component

代替なし: ELF binary を出力する手段は (a) Rust ld、(b) GNU binutils ld、
(c) 自前 emit の 3 択。(a)(b) を保留すると Phase 47 の "0 Rust" /
"toolchain 不要" goal が崩壊する。よって (c) elisp 自前 emit が唯一の
path。

* §2 ELF format primer

** §2.1 ELF64 全体構成

=./nelisp= (= 静的 link された executable) の典型的 layout:

#+begin_src text
+--------------------------------------+ offset 0x0000
|  Ehdr         (64 bytes)             |
+--------------------------------------+ offset 0x0040
|  Phdr[0..n]   (56 bytes each)        | <-- OS loader が読む
+--------------------------------------+
|  .text        (x86_64 machine code)  | <-- assembler emit
|  .rodata      (constants, strings)   |
|  .data        (initialized globals)  |
|  .bss         (zero-init globals,    | <-- file 上は size 0、mem で zero
|               file image 不在)        |
|  .symtab      (symbol table)         |
|  .strtab      (symbol name strings)  |
|  .shstrtab    (section name strings) |
+--------------------------------------+
|  Shdr[0..m]   (64 bytes each)        | <-- readelf / objdump が読む
+--------------------------------------+ EOF
#+end_src

** §2.2 ELF64 Ehdr (Elf64_Ehdr) byte-layout

| offset | size | name        | value (= x86_64 ELF executable)        |
|--------+------+-------------+----------------------------------------|
|      0 |    4 | e_ident[0..4] | =0x7F 'E' 'L' 'F'= (= magic)         |
|      4 |    1 | e_ident[4]    | =2= (= ELFCLASS64)                   |
|      5 |    1 | e_ident[5]    | =1= (= ELFDATA2LSB = little-endian)  |
|      6 |    1 | e_ident[6]    | =1= (= EV_CURRENT)                   |
|      7 |    1 | e_ident[7]    | =0= (= ELFOSABI_NONE / SYSV)         |
|      8 |    8 | e_ident[8..15] padding | zero                        |
|     16 |    2 | e_type      | =2= (= ET_EXEC) または =3= (= ET_DYN) |
|     18 |    2 | e_machine   | =0x3E= (= EM_X86_64)                   |
|     20 |    4 | e_version   | =1=                                    |
|     24 |    8 | e_entry     | entry point virtual address            |
|     32 |    8 | e_phoff     | Phdr table offset (= typically 0x40)   |
|     40 |    8 | e_shoff     | Shdr table offset                      |
|     48 |    4 | e_flags     | =0=                                    |
|     52 |    2 | e_ehsize    | =64= (= sizeof Ehdr)                   |
|     54 |    2 | e_phentsize | =56= (= sizeof Phdr)                   |
|     56 |    2 | e_phnum     | Phdr count                             |
|     58 |    2 | e_shentsize | =64= (= sizeof Shdr)                   |
|     60 |    2 | e_shnum     | Shdr count                             |
|     62 |    2 | e_shstrndx  | =.shstrtab= section index              |

** §2.3 Elf64_Phdr byte-layout

| offset | size | name      | meaning                                  |
|--------+------+-----------+------------------------------------------|
|      0 |    4 | p_type    | =1= PT_LOAD / =6= PT_PHDR / =3= PT_INTERP |
|      4 |    4 | p_flags   | =4= R / =2= W / =1= X (bit OR)          |
|      8 |    8 | p_offset  | file offset (= section 開始位置)         |
|     16 |    8 | p_vaddr   | mapped virtual address                   |
|     24 |    8 | p_paddr   | physical address (= 0 でも OK)            |
|     32 |    8 | p_filesz  | file image 上の size                      |
|     40 |    8 | p_memsz   | memory image 上の size (= bss は filesz < memsz) |
|     48 |    8 | p_align   | alignment (= 0x1000 typically)           |

** §2.4 Elf64_Shdr byte-layout

| offset | size | name         | meaning                              |
|--------+------+--------------+--------------------------------------|
|      0 |    4 | sh_name      | =.shstrtab= 内の name string offset  |
|      4 |    4 | sh_type      | =1= PROGBITS / =8= NOBITS / =2= SYMTAB / =3= STRTAB / =4= RELA |
|      8 |    8 | sh_flags     | =1= WRITE / =2= ALLOC / =4= EXECINSTR (bit OR) |
|     16 |    8 | sh_addr      | mapped virtual address (= ALLOC のみ) |
|     24 |    8 | sh_offset    | file offset                          |
|     32 |    8 | sh_size      | section size in bytes                |
|     40 |    4 | sh_link      | RELA → SYMTAB index, SYMTAB → STRTAB index |
|     44 |    4 | sh_info      | RELA → target section index          |
|     48 |    8 | sh_addralign | alignment                            |
|     56 |    8 | sh_entsize   | entry size (= SYMTAB 24, RELA 24)    |

** §2.5 Elf64_Sym byte-layout (= .symtab entry)

| offset | size | name      | meaning                              |
|--------+------+-----------+--------------------------------------|
|      0 |    4 | st_name   | =.strtab= 内 name offset             |
|      4 |    1 | st_info   | bind << 4 \vert type (= 1<<4\vert 2 = global func) |
|      5 |    1 | st_other  | visibility (= 0 default)             |
|      6 |    2 | st_shndx  | section index                        |
|      8 |    8 | st_value  | symbol address (= section + offset)  |
|     16 |    8 | st_size   | symbol size                          |

** §2.6 Elf64_Rela byte-layout (= .rela.text entry)

| offset | size | name        | meaning                            |
|--------+------+-------------+------------------------------------|
|      0 |    8 | r_offset    | relocation target offset           |
|      8 |    8 | r_info      | (sym << 32) \vert type             |
|     16 |    8 | r_addend    | signed addend                      |

x86_64 で頻出する r_info type:
- =1= R_X86_64_64    = 64-bit absolute
- =2= R_X86_64_PC32  = 32-bit PC-relative
- =4= R_X86_64_PLT32 = 32-bit PC-rel via PLT (= static link で R_X86_64_PC32 同等)

** §2.7 magic / endian / alignment 不変条件

- Magic = =0x7F 0x45 0x4C 0x46= (= =\x7FELF=) at offset 0
- endian = ELFDATA2LSB = **all multi-byte fields little-endian**
- alignment = 8-byte (= 64-bit struct 内 field は naturally aligned)
- segment alignment = 0x1000 (= 4 KiB page) typically
- bss = =sh_type = NOBITS=、=p_filesz < p_memsz= で zero-fill 表現

** §3.2 byte-int conversion helpers (= ~50 LOC)

#+begin_src elisp
;; 8-bit, 16-bit, 32-bit, 64-bit little-endian writers.
;;   buf = unibyte string accumulator (= elisp string + insert)
(defun nelisp--write-u8 (buf v)
  (insert (unibyte-string (logand v #xff))))

(defun nelisp--write-le16 (buf v)
  (insert (unibyte-string (logand v #xff)
                          (logand (ash v -8) #xff))))

(defun nelisp--write-le32 (buf v)
  (let ((b0 (logand v #xff))
        (b1 (logand (ash v -8) #xff))
        (b2 (logand (ash v -16) #xff))
        (b3 (logand (ash v -24) #xff)))
    (insert (unibyte-string b0 b1 b2 b3))))

(defun nelisp--write-le64 (buf v)
  ;; elisp 64-bit fixnum on 64-bit Emacs; bignum fallback on 32-bit.
  ;; See §7.2 risk.
  (dotimes (i 8)
    (insert (unibyte-string (logand (ash v (- (* i 8))) #xff)))))

(defun nelisp--write-bytes (buf bytes)
  (insert bytes))

(defun nelisp--write-zero-pad (buf n)
  (insert (make-string n 0)))

(defun nelisp--write-strz (buf s)
  ;; Null-terminated string for .strtab.
  (insert s)
  (insert (unibyte-string 0)))
#+end_src

工数 = 50 elisp LOC。

(defun nelisp-elf-write-ehdr (buf hdr)
  "Write Ehdr (64 bytes). HDR is a plist with :entry :phoff :shoff
   :phnum :shnum :shstrndx ..."

** §3.4 top-level orchestration =nelisp-elf-write-binary= (~80 LOC)

#+begin_src elisp
(defun nelisp-elf-write-binary (file-path sections)
  "Emit static-linked ELF64 executable to FILE-PATH.
SECTIONS = plist:
  :text       unibyte-string of x86_64 machine code
  :rodata     unibyte-string
  :data       unibyte-string
  :bss-size   integer (= zero-init region size)
  :symbols    list of (name section offset size binding type)
  :relocs     list of (section target-offset sym-index type addend)
  :entry-sym  symbol name (= e_entry resolution target)"
  (let ((buf (generate-new-buffer "*nelisp-elf*")))
    (with-current-buffer buf
      (set-buffer-multibyte nil)
      ;; Phase 1: layout calculation = decide Phdr / section offsets
      (let* ((layout (nelisp-elf--compute-layout sections))
             (ehdr   (nelisp-elf--ehdr-from-layout layout))
             (phdrs  (nelisp-elf--phdrs-from-layout layout))
             (shdrs  (nelisp-elf--shdrs-from-layout layout)))
        ;; Phase 2: emit in order
        (nelisp-elf-write-ehdr buf ehdr)
        (dolist (p phdrs) (nelisp-elf-write-phdr buf p))
        (nelisp-elf--emit-sections buf layout sections)
        (dolist (s shdrs) (nelisp-elf-write-shdr buf s))
        ;; Phase 3: flush
        (write-region (point-min) (point-max) file-path nil 'silent)
        (set-file-modes file-path #o755)))   ; +x bit
    (kill-buffer buf)))
#+end_src

helper:
- =nelisp-elf--compute-layout= = ~30 LOC = section offset / vaddr 計算
- =nelisp-elf--emit-sections= = ~30 LOC = section bytes + symtab +
  strtab + rela serialize 順序

合計 ~80 LOC。

* §4 byte-level invariants

** §4.1 endianness

ELF64 x86_64 = **little-endian** (= ELFDATA2LSB / =e_ident[5] = 1=)。
elisp 側で multi-byte int を書くとき必ず =le16/le32/le64= helper 経由。
Big-endian は出ない (= aarch64 でも依然 LE が main configuration、
本 doc は LE 前提)。

** §4.2 alignment

- struct field = 8-byte (= 64-bit struct は naturally aligned)
- segment / section = 16-byte または 8-byte (= =sh_addralign= で個別
  指定)
- PT_LOAD segment = **0x1000 page-aligned** (= file offset と vaddr を
  両方 page boundary に)
- =.bss= = =sh_addralign 8= (= or 16)、=sh_size= は実 size、=p_memsz=
  は roundup

elisp 実装方針: =compute-layout= helper 内で =mod= / =roundup= を一手
集約、各 writer は zero padding を =nelisp--write-zero-pad= 経由で挿入。

** §4.3 padding (= zero-fill)

- e_ident[8..15] = 9-byte zero pad (= ABIVER 1 byte + reserved 7)
- segment 間 = file 上は連続でも OK (= =p_offset= で個別位置指定)
- =.bss= = file image **不在** = =sh_type = NOBITS= かつ =sh_offset= は
  次 section の開始 offset を指す convention
- struct 末尾 padding なし (= ELF64 struct は exactly 64/56/24/24 byte)

** §4.4 magic numbers

| 用途                | byte sequence                          |
|---------------------+----------------------------------------|
| ELF magic           | =0x7F 0x45 0x4C 0x46= (= =\x7FELF=)    |
| EM_X86_64           | =0x3E 0x00= (= LE 16-bit)              |
| EI_CLASS = 64       | =0x02=                                 |
| EI_DATA = LSB       | =0x01=                                 |
| ET_EXEC             | =0x02 0x00=                            |
| EV_CURRENT          | =0x01 0x00 0x00 0x00=                  |
| PT_LOAD             | =0x01 0x00 0x00 0x00=                  |

これらは `defconst` で集約 (= =nelisp-elf--magic-elf= /
=nelisp-elf--em-x86-64= 等)、source 中に hard-coded literal を散らさ
ない。

** §4.5 elisp 上の保存方針

elisp string は default で multibyte。ELF binary は **必ず unibyte**。
=set-buffer-multibyte nil= で buffer を unibyte 化、=unibyte-string= で
byte literal 構築。=write-region= は unibyte buffer なら raw bytes を
そのまま file に書き出す。

* §7 Risk / unknowns

** §7.1 elisp 性能 (= 1-2 MB ELF を bytewise で書く速度)

1 MB ELF を 1 byte ずつ =insert-byte= で書くと:
- =insert= は buffer position update を毎回 = O(n) 単位
- ~1M calls × O(1) = 数秒〜十秒 オーダー
- =unibyte-string= で N byte chunk を一度に作って =insert= = ~ms 単位

*対策*:
1. =unibyte-string= で 4-byte / 8-byte chunk を作って =insert= する形に
   helper を統一 (= 既に §3.2 で採用方針)
2. 全 ELF を 1 大 unibyte string に build → =write-region= 1 回 (=
   buffer 経由しない方が速い場合あり、要 benchmark)
3. =benchmark-run= で計測、5 sec 以上なら C primitive (= =write-bytes=
   builtin 追加) で escape hatch

Doc 85 §7.2 が articulate した "production debug tools 喪失" と並ぶ
重要 risk。

** §7.2 integer overflow on 32-bit Emacs

elisp =most-positive-fixnum= = (32-bit Emacs では) 2^29 - 1 = 536M。
ELF64 は 64-bit address / offset を要求 → fixnum 越えする entry が
ある (= e_entry, p_offset 等)。

*対策*:
1. Emacs 27+ は automatic bignum、64-bit Emacs では fixnum 余裕あり
2. elisp 側で =le64= helper を bignum 安全に書く (= bit-shift で 1 byte
   ずつ取り出す pattern、§3.2 実装通り)
3. 32-bit Emacs support は OOS = 64-bit Emacs を前提とする

** §7.3 cross-platform (= aarch64 / Mach-O / PE COFF)

本 doc は **x86_64-linux ELF only**。

| platform        | ELF64 / format       | 工数 prediction     |
|-----------------+----------------------+---------------------|
| x86_64-linux    | ELF64 + EM_X86_64    | 本 doc 対象 (~500 LOC) |
| aarch64-linux   | ELF64 + EM_AARCH64   | +50 LOC (= magic 違いのみ) |
| x86_64-macos    | Mach-O               | 別 module ~600 LOC  |
| x86_64-windows  | PE COFF              | 別 module ~700 LOC  |

aarch64-linux は **本 doc の小幅拡張で対応可** (= EM 値 + reloc type
table の差し替え)。Mach-O / PE は完全別実装、Doc 91-extension として
Phase 47 完遂 *後* に着手。

** §7.4 dynamic linking 不要前提の制約

本 doc は **static linked binary のみ** emit:
- =.dynamic= / =.dynsym= / =.dynstr= section emit せず
- =PT_INTERP= (= =/lib64/ld-linux-x86-64.so.2=) emit せず
- libc 動的 link は非対応

対応方針: musl static or pure syscall binary (= libc 内蔵) のみ
generate。Phase 47 全体の方向性 (= Doc 93 syscall inline) と整合。

** §7.5 DWARF debug info

Doc 85 §7.2 risk と同じ。本 doc では DWARF emit 非対応 = =gdb= 上で
Rust source level debug 不可。production 運用前に follow-up doc (= Doc
91-debug 候補) で DWARF section emit を articulate する必要。
