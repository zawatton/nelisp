** 2.1 binary integration: nelisp-runtime cdylib + bin embed [推奨 = A]

A. *推奨*: =nelisp-runtime= cdylib (Phase 7.0 SHIPPED) を =bin/anvil=
   が静的 link、host Emacs 不要モード起動 (=--no-emacs= flag)
B. dynamic link (=.so= / =.dylib=) で起動時 dlopen
C. embed しない、別 binary (=nelisp-runtime= 単独 exe) で =bin/anvil=
   から spawn

採用理由 (A):
- *single-file binary* = =stage-d-v2.0= の核要件、tarball 不要
- LTO + strip で binary size 最適化可能 (target ~10MB)
- runtime path resolution の OS 差異吸収不要 (Linux / macOS / Windows)
- B は Linux glibc / macOS dyld / Windows DLL hell の互換性 layer 増、
  C は IPC overhead が cold-init 経路で許容できない (~50ms+)

** 2.2 cold-init protocol: Doc 28 §3.5 4-stage bootstrap embed [推奨 = A]

A. *推奨*: Doc 28 §3.5 4-stage bootstrap を =bin/anvil --self-compile=
   に embed、stage0 (host bytecode 不要、bytes 直接) → stage1 (seed) →
   stage2 (diff) → stage3 (self-recompile) を bin 内で完結
B. 4-stage を 2-stage に簡略化 (stage0 + stage3 のみ)
C. cold-init を別 sub-binary (=nelisp-bootstrap=) に分離

採用理由 (A):
- Doc 28 §3.5 で確立した failback chain (=A2 fallback=) を runtime で
  再現、authoritative compiler 昇格に対し safety net 維持
- B は stage2 semantic diff 検証を skip、silent miscompile risk
- C は binary 数を増やす、=stage-d-v2.0= single-file requirement 違反

** 2.3 stage-d-v2.0 release artifact format [推奨 = A]

A. *推奨*: tarball ではなく single-file binary (ELF / Mach-O)、cdylib
   link 静的 (full LTO + strip)、~10MB target size (anvil-lib bundle なし)
B. tarball (=stage-d-v2.0.tar.gz=) で binary + license + README 同梱
C. AppImage (Linux) / .dmg (macOS) で installer 化

採用理由 (A):
- Doc 18 Option B の "single-file" 思想直接実現
- =install.sh= は curl + chmod のみで済む (~5 line)、user 体験最簡
- B は v0.x シリーズ (Phase 6.1 anvil-lib) で既に存在、v2.0 は別形式
- C は OS 依存度高、Linux / macOS / Windows 全部別 packaging system 必要

** 2.4 cross-arch CI: Linux x86_64 + macOS arm64 + Linux arm64 [推奨 = A]

** 2.5 backward compat: Stage D dual-mode 維持 [推奨 = A]

A. *推奨*: =bin/anvil mcp serve= (Stage D Phase 6.1 既存) と
   =--no-emacs= mode の dual path 維持、user は =--no-emacs= explicit
   opt-in、default は host Emacs 経路 (Stage D)
B. =--no-emacs= mode を default、=--use-emacs= で旧経路に明示 fallback
C. v2.0 release で旧経路 deprecate、3 月以内に v0.x 廃止

採用理由 (A):

- B は cold-init unstable (= 24h soak gate 未達 risk) の場合に user 体験
  退化 risk
- C は *NeLisp 公開告知 trigger* (本 phase) と同時 deprecate は user 急変
  すぎる、6-12 ヶ月の grace period が安全

** 2.6 fallback: =--no-emacs= 起動 fail 時 host Emacs 経路 auto-fallback [推奨 = A]

A. *推奨*: =--no-emacs= mode で起動 fail 時 host Emacs path に
   auto-fallback (warning 出力、=--strict-no-emacs= で抑止可能)
B. fail で即 exit、retry は user 責任
C. fail 時 =--use-emacs= を user に suggest しつつ exit

採用理由 (A):
- cold-init 失敗は本 phase 24h soak gate 達成しても 0 にはならない
  (= memory pressure / disk full / mmap failure 等の OS 的失敗)
- auto-fallback で *user 体験は Stage D Phase 6.1 と同等以上* を保証、
  v2.0 を試す user の安心感確保
- B は v2.0 試用ハードル上昇 (= adoption rate 低下 risk)
- C は user の 1 step 増、auto-fallback の方が体験良い
- =--strict-no-emacs= flag で fallback 抑止 (CI / debugging 用途)

* §4 Architecture (=bin/anvil --no-emacs= 起動経路)

#+begin_src text
[ user ]
   ↓ ./bin/anvil mcp serve --no-emacs
[ bin/anvil launcher (Rust thin shell) ]
   ↓ flag dispatch
[ Phase 7.0 syscall stub init (Rust cdylib link 静的) ]
   ↓ mmap arena (~64MB nursery + ~256MB tenured slot)
[ Phase 7.2 allocator init (heap span = nursery 4MB + tenured 64MB) ]
   ↓ bump pointer + free-list pool ready
[ Phase 7.3 GC scheduler init (mark trigger + safe-point poll) ]
   ↓ collector idle、safe-point poll registered
[ Phase 7.4 coding init (UTF-8 default + IO codecs setup) ]
   ↓ stdin/stdout に UTF-8 codec attach
[ 4-stage cold-init (Doc 28 §3.5) ]
   ↓ stage0: bin 内 embed bytecode → A2 seed
   ↓ stage1: A2 が core lib を native compile → A3 candidate
   ↓ stage2: A3 ↔ A2 semantic diff (golden hash)
   ↓ stage3: A3 が core lib を self-recompile + diff → authoritative 昇格
[ Phase 7.1 native compiler authoritative (A3) ]
   ↓ JIT-ready
[ MCP server start (anvil-server-run-batch-stdio、headless profile、~28 tools) ]
   ↓ stdin / stdout で JSON-RPC ready
[ mcp serve listening、JSON-RPC over stdio ]
   ↓ Claude Code MCP client connects
   ↓ initialize → tools/list → tools/call → ... → shutdown
[ shutdown ]
   ↓ GC sweep、resource release、exit 0
#+end_src

*失敗経路 (=--no-emacs= → fallback)*:

#+begin_src text
[ Phase 7.X init failure (例: stage1 native compile fail) ]
   ↓ fallback decision
[ if --strict-no-emacs ]
   → exit 1 (即座中断)
[ else ]
   → warning 出力: "cold-init failed at stage1, falling back to host Emacs path"
   → bin/anvil mcp serve (= Stage D Phase 6.1 経路) を internal fork
   → Stage D 経路で MCP server 起動継続 (user 体験不変)
#+end_src

** 6.1 cold-init time spike (4-stage bootstrap で stage1/2 の compile 時間)
