* 0. One-line

Replace the opaque Rust ~*mut Env~ threaded through the elisp eval core
with an ~EvalCtx~ *Sexp Record we define* (slot layout under our control →
known offsets), and port the eval *driver* (~nelisp_eval_call~ /
~nl_eval_inner_cons~ / ~apply_special~ / ~eval~ / ~apply_function~ /
~eval_arg_list~) to elisp that threads it. This *dissolves* the 14
env-group undefined symbols (they become ~EvalCtx~-slot accessors), deletes
the Rust eval driver (net-negative Rust), and adds ZERO new Rust.

* 0.5 Implementation vehicle — nelisp-sys, NOT raw phase47 cc-files (B1, user decision 2026-05-30)

The driver port is written in the *typed nelisp-sys DSL*, not hand-written
~lisp/nelisp-cc-*.el~ phase47. Rationale + capability (grep-confirmed in
~packages/nelisp-sys/src/~):

- ~sys:extern~ (+ ~nelisp-sys-frontend--parse-extern~) declares/calls the EXISTING
  C-ABI accessors (~nelisp_env_lookup_value~, ~nelisp_frame_push/pop~, …) → the
  current Sexp/mirror/frames data model is REUSED, no model rewrite.
- ~sys:defstruct~ + ~sys:load-field~/~sys:store-field~ defines ~EvalCtx~ as a TYPED
  struct → slot offsets known by construction (solves the non-repr-C blocker),
  pinned to the C-ABI Record layout via ~sys:static-assert~.
- ~sys:alloc~/~sys:addr-of~/~sys:ptr-add~/~sys:peek-u64~/~ptr-read,write~ + ~sys:call-ptr~
  cover all raw-memory / indirect-call needs.
- The ~type checker~ (~E-SYS-*~) catches at COMPILE time the bug class that has been
  the campaign's main pain (offset/i32-i64/ABI mistakes are otherwise latent
  until the build is green — untestable, crash-prone). This is the decisive win.
- *CORRECTION (Stage 135.0)*: the earlier claim "the 6 non-env symbols are
  already nelisp-sys grammar ops" is MISLEADING. The ops ~sys:str-to-float~ /
  ~sys:char-table-get,set~ / ~sys:mut-str-push-byte,codepoint~ exist, but each
  *lowers to a `(extern-call nl_…)` / `(mut-str-push-byte …)` that CALLS the very
  deleted symbol* (verified: ~lisp/nelisp-phase47-compiler.el:10604~ emits
  ~call nl_mut_str_push_byte~; backend ~nelisp-sys-backend.el:159-204~ emits
  ~extern-call nl_char_table_*~ / ~nl_str_to_float~). So the ops are CALLERS,
  not definitions — using them to define the symbol is self-recursive. All 6
  non-env symbols need REAL low-level implementations (push needs Vec realloc-
  grow; set-codepoint needs char-count + UTF-8; str-to-float is strtod;
  char-table is a tree-walk). nelisp-sys still helps (type-checked low-level
  code) but they are NOT free.

* 1. Why Doc 134's per-symbol re-provision fails for the env group

Doc 134 re-provided deleted helpers as standalone zero-Rust ~.o~ (succeeded for
the 7 data-ops + ~nl_eval_is_truthy~ + ~nl_sexp_clone_into~). Stage 134.C
proved the remaining ~14 env-group symbols CANNOT be done that way:

- ~nelisp_eval_call(form, env, out)~ and ~nl_eval_inner_cons(head, tail, env, out)~
  both do ~let r = &mut *(env as *mut Env)~ (~env as *mut Env~ appears 9× in
  ~build-tool/src/eval/mod.rs~). So the ~env~ threaded through the WHOLE elisp
  eval core (~nl_eval_inner~ → ~nl_env_lookup_val~; ~nl_sf_setq~ → ~nl_env_set_value~;
  ~nl_apply_lambda_inner~ → ~nl_push_and_bind~ / ~nl_env_pop_frame~; …) is the
  *opaque Rust ~Env~ struct pointer*.
- ~Env~ is *NOT ~#[repr(C)]~*:
  ~{max_recursion:u32, current_recursion:u32, extern_builtins:HashMap, use_elisp_apply:bool, delegation_depth:u32, globals_record:Sexp (=mirror), unbound_marker:Sexp, frames_record:Sexp (=frames)}~.
  Its field offsets are rustc-chosen and not stable.
- The shipped C-ABI accessors (~nelisp_env_lookup_value(mirror_ptr, frames_ptr, name, out)~,
  ~nelisp_env_set_value~, ~nelisp_env_bind_local~, ~nelisp_frame_push~) take
  mirror+frames as *separate Sexp pointers*. To re-provide e.g.
  ~nl_env_lookup_val(name, env, out)~ as elisp we'd need ~&env.globals_record~ +
  ~&env.frames_record~ — field addresses inside a non-repr-C Rust struct.
- All three ways to get those are closed: (a) ~#[repr(C)]~ + offset arithmetic
  = a forbidden ~eval/mod.rs~ edit; (b) offset-read a repr(Rust) struct =
  build-dependent + unverifiable until the build is green (latent crash);
  (c) a Rust accessor ~nelisp_env_globals_ptr(env)~ = forbidden new Rust.

Conclusion: ~fa8932eb~ deleted helpers that are an *inseparable coupling to
the Rust ~Env~*. They are not "fillable" in isolation. The build reaches green
only by changing what ~env~ *is*.

* 2. The fix — env becomes an EvalCtx Sexp Record we control

Define an ~EvalCtx~ as a ~Sexp::Record~ whose slots WE lay out, threaded as the
single ~env~ Sexp pointer through the elisp eval core. Because it is a Record,
slot access is the established ~record-slot-ref~ / ~nlrecord-set-slot~ idiom at
*known* offsets — exactly the pattern the env group needs and the Rust struct
denied us.

** 2.1 EvalCtx slot layout (proposed)

| slot | name              | type   | replaces (Env field)              |
|------+-------------------+--------+-----------------------------------|
|    0 | type-tag          | Symbol | (record type marker, "eval-ctx")  |
|    1 | mirror            | Record | globals_record                    |
|    2 | frames            | Record | frames_record                     |
|    3 | unbound-marker    | Sexp   | unbound_marker                    |
|    4 | recursion-cur     | Int    | current_recursion                 |
|    5 | recursion-max     | Int    | max_recursion                     |
|    6 | flags             | Int    | use_elisp_apply | delegation_depth<<1 (bitpacked) |
|    7 | builtin-registry  | Sexp   | extern_builtins (see §3)          |

Mirror+frames already exist as Records (constructed by the shipped
~nelisp_env_install_empty_globals_frames(globals_out, frames_out, scratch, _pad)~).
EvalCtx just *holds* them in slots 1/2, so any elisp ~.o~ recovers them with
~record-slot-ref ctx 1~ / ~record-slot-ref ctx 2~ and feeds the existing
~nelisp_env_lookup_value~ / ~_set_value~ / ~_bind_local~ accessors unchanged.

** 2.2 How each env-group symbol dissolves (no more standalone re-provision)

After the port, the *callers* (eval-inner, sf-*, apply-lambda-inner) thread
~ctx~ and call the C-ABI accessors with ~(record-slot-ref ctx 1)~ / ~(… ctx 2)~
directly — so most deleted helpers are no longer referenced at all:

| deleted symbol            | post-port disposition                                              |
|---------------------------+--------------------------------------------------------------------|
| nl_env_lookup_val         | caller calls nelisp_env_lookup_value(ctx.mirror, ctx.frames, …)    |
| nl_env_set_value          | caller calls nelisp_env_set_value(ctx.mirror, ctx.frames, …)       |
| nl_bf_bind_sym            | caller calls nelisp_env_bind_local(ctx.mirror, ctx.frames, …)      |
| nl_env_pop_frame          | caller calls the shipped elisp nelisp_frame_pop(ctx.frames, …)     |
| nl_push_and_bind          | thin elisp: frame_push(ctx.frames) + bind-formals loop (ctx)       |
| nl_let_setup              | thin elisp: frame_push(ctx.frames) + bind-all loop (ctx)           |
| nl_cc_match_and_bind      | thin elisp: clause-match + frame_push + var-bind (ctx)             |
| nl_bf_bind_optional/rest  | thin elisp: list-walk + nelisp_env_bind_local (ctx)                |
| nl_bf_err_arity/type/dangling_rest | thin elisp: build signal + nelisp_env_set_value(ctx, "nelisp--last-signal-data", …) |
| nl_env_push_captured      | thin elisp: frame_push(ctx.frames) + bind captured alist          |
| nl_env_capture_lexical    | DEAD-REF (no live caller found in 134.C) → drops out               |

All become *direct ctx-slot operations over already-shipped C-ABI*, zero Rust.

** 2.3 LOCKED EvalCtx + nelisp-sys idioms (Stage 135.B, type-checked 2026-05-30)

The EvalCtx representation is type-checked (E-SYS-* clean) and LOCKED. Model A
(by-value Sexp fields — nelisp-sys DOES support by-value aggregate struct
fields). Supersedes the §2.1 proposal (no separate type-tag/builtin-registry
slots: extern_builtins has only 1 entry per §3.0, so no registry slot needed).

#+begin_src elisp
;; 32-byte runtime Sexp value (tag@0, payload@8, pad@16..31).
(sys:defstruct sexp (:repr c)
  (tag u8) (payload u64) (pad (array u8 16)))

;; EvalCtx — replaces the Rust Env. 120 bytes, align 8 (x86_64):
;;   mirror@0  frames@32  unbound@64  rec_cur@96  rec_max@104  flags@112
;; flags bit0 = use_elisp_apply, bits 1.. = delegation_depth.
(sys:defstruct eval_ctx (:repr c)
  (mirror sexp) (frames sexp) (unbound sexp)
  (rec_cur i64) (rec_max i64) (flags i64))

;; C-ABI accessor (shipped, defined): mirror+frames are separate Sexp ptrs.
(sys:extern nelisp_env_lookup_value
  (:symbol "nelisp_env_lookup_value" :abi c :unsafe t)
  ((mirror_ptr usize) (frames_ptr usize) (name_ptr usize) (out_ptr usize))
  i64 (:alloc none :ffi may :unsafe may))

;; Accessor proof — the ctx→C-ABI path (this is the shape every env-group
;; symbol takes post-cutover; adjust arg order to match each deleted sig).
(sys:defun nl_eval_ctx_lookup
    ((ctx (ptr eval_ctx)) (name usize) (out usize)) i64
  (:alloc none :ffi may :unsafe may)
  (let ((mirror_ptr usize (+ (sys:cast usize ctx) (sys:offsetof eval_ctx mirror)))
        (frames_ptr usize (+ (sys:cast usize ctx) (sys:offsetof eval_ctx frames))))
    (sys:unsafe (nelisp_env_lookup_value mirror_ptr frames_ptr name out))))
#+end_src

Lowers to (validated): ~(defun nl_eval_ctx_lookup (ctx name out) (let ((mirror_ptr
(+ ctx 0)) (frames_ptr (+ ctx 32))) (nelisp_env_lookup_value mirror_ptr frames_ptr
name out)))~ — sys:offsetof folds to 0/32, sys:cast/sys:unsafe erase, extern emits a
direct ~call~. nelisp_env_lookup_value is a defined shipped symbol → resolves.

*nelisp-sys idioms LOCKED for all driver code* (reuse these):
- By-value 32-byte Sexp field: define a ~sexp~ ~sys:defstruct (:repr c)~, embed by name.
- Field address (→ ~*const Sexp~ = usize): ~(+ (sys:cast usize PTR) (sys:offsetof STRUCT FIELD))~.
  (~sys:addr-of~ is for FUNCTION names only, not struct fields.)
- Call a shipped C-ABI symbol: ~sys:extern~ decl with ~:abi c :unsafe t~, args as ~usize~
  for every pointer; CALL inside ~(sys:unsafe …)~; the wrapper defun MUST carry
  ~:unsafe may :ffi may~ and match the extern's ~:alloc~ (none) or hit E-SYS-ALLOC-001.

** 3.0 RESOLVED (Stage 135.A, 2026-05-30) — VERDICT-B: mirror resolves, existing bridge dispatches

Read-only investigation answered the #1 risk. Findings (build-tool/src/eval/mod.rs):
- ~install_builtins(env)~ registers ALL 61 builtins into the mirror function-cells

- ~extern_builtins~ has exactly ONE real entry (~nelisp--env-globals-op~); the other
  60 builtins live in the Rust ~builtin_dispatch!~ macro (58 arms). The HashMap is a
  near-empty fallback, NOT the main registry. → it collapses into EvalCtx slot 7 /

- The DISPATCH BRIDGE ALREADY EXISTS: ~nelisp--apply-builtin-dispatch(name, args)~ is
  itself a registered builtin (a ~builtin_dispatch!~ arm) that fans out to
  ~builtins::dispatch(name, args, env)~. The elisp driver invokes any builtin by
  calling it — *no new Rust*; the 58-arm dispatcher is existing Rust that stays
  until Doc 136+ ports its arms (21 are already ~bi_*~ .o; ~37 are Rust-only).
- An ELISP APPLY STACK ALREADY EXISTS: ~lisp/nelisp-stdlib-eval-core.el~ has
  ~nelisp--apply-fn~ / ~nelisp--builtinp~ / ~nelisp--closurep~ / ~nelisp--apply-closure~
  / ~nelisp--apply-lambda~, dispatching ~(builtin NAME)~ → ~nelisp--apply-builtin-dispatch~
  and closures → ~nelisp--apply-lambda-inner~. The 135.B port can build on this.

⚠️ CAVEAT for 135.B/E (the agent's VERDICT-B glossed this): every ~builtin_dispatch!~
arm takes ~&mut Env~. The env-INDEPENDENT arms (make-vector, string-bytes, truncate,
syscalls, I/O, JIT…) ignore env and are safe to call across the cutover. But the
env-USING arms (~eval~, ~funcall~, ~apply~, ~nelisp--push/pop-frame~, ~nelisp--bind-local~,
~nelisp--apply-lambda-inner~, ~nelisp--env-globals-op~) deref ~env as *mut Env~. After
135.E (env = EvalCtx Sexp) those would receive an EvalCtx and mis-cast. RESOLUTION:
the env-USING ones are EITHER the driver itself (eval/funcall/apply = ported, not
dispatched) OR the frame/bind ops the env-group symbols already replace — so they do
NOT go through ~nelisp--apply-builtin-dispatch~ post-cutover; only env-independent
builtins are dispatched through the bridge. 135.B must split the dispatch accordingly
(env-independent → bridge; env-using → driver/ctx-ops). This is tractable but is the
real shape of the cutover, to be pinned in 135.B.

** 3.1 (original framing, superseded by 3.0)

~extern_builtins: HashMap<String, Rc<dyn Fn(&[Sexp], &mut Env)>>~ is Rust
closures = the dispatch table for Rust-implemented builtins. It cannot live in
a Sexp slot as-is. Two facts make this tractable:

1. Most builtins are ALREADY elisp ~.o~ (the ~bi_*~ family: ~bi_make_vector~,
   ~bi_string_bytes~, …) — invoked today through the Rust ~builtin_dispatch!~
   macro. The registry is largely a *name → which ~.o~* table.
2. Function resolution already has an elisp path: ~nelisp_env_lookup_function~ +
   the mirror function-cells (~nelisp_mirror_set_function~ etc.).

Plan: builtins resolve via the *mirror function-cells*, not a Rust HashMap.
Slot 7 (builtin-registry) holds the mirror-backed builtin table (or is simply
folded into the mirror). During transition, ~apply_function~'s builtin path may
call a *stable C-ABI builtin dispatcher* ~nelisp_builtin_dispatch(name_ptr,
args_ptr, ctx_ptr, out)~ that fans out to the existing ~bi_*~ ~.o~. This
dispatcher is itself an elisp ~.o~ (a name→.o switch), NOT new Rust. Porting
the long tail of ~bi_*~ that are still Rust is *out of scope for Doc 135* —
they remain callable via their existing C-ABI symbols (already defined), and
Doc 136+ ports them. Doc 135 only needs them *reachable* from the elisp driver.

NOTE: this is the riskiest area; Stage 135.A produces a focused spike proving
one builtin resolves through the ctx-threaded path before the full cutover.

* 4. Driver surface to port (Rust → elisp, then delete the Rust)

From ~build-tool/src/eval/mod.rs~ (all own ~&mut Env~):

- ~eval(form, env) -> Result<Sexp, EvalError>~ — main tag dispatch (atom →
  self; symbol → lookup; cons → ~nl_eval_inner_cons~). ~nl_eval_inner~ elisp ~.o~
  already does the atom/symbol/cons split; extend it to own the cons handling.
- ~eval_arg_list(args, env) -> Vec<Sexp>~ — map eval over an arg list (cons-walk).
- ~apply_function(func, args, env)~ — closure vs builtin application.
- ~apply_special(name, args, env) -> Option<Sexp>~ — 10-way string dispatch:
  quote/function/if/let/let*/lambda/setq/while/condition-case/unwind-protect/progn.
  *Each arm already calls a shipped elisp ~.o~* (~sf_quote_call~, ~sf_if_call~, …)
  passing ~env~ — so this is a name-dispatch + ctx-threading port, the bodies
  are done.
- ~eval_stash_err(env, result, out) -> i64~ — error→out marshalling.
- C-ABI entries: ~nelisp_eval_call(form, env, out)~, ~nl_eval_inner_cons(head,
  tail, env, out)~ — the boundary that casts ~env as *mut Env~ today.

The special-form ~.o~ (~sf_*~) and bind-formals ~.o~ currently take ~env~ as
~*mut c_void~ (= ~*mut Env~). After the port they take ~ctx~ (= ~*mut Sexp~ to the
EvalCtx Record). Their bodies change only where they touch env (lookup/bind):
those become ctx-slot + C-ABI accessor calls.

** 5.5.2 The combiner (the core intellectual work)

~nl_eval_inner_cons(head, tail, env, out)~ (Rust) is the apply-combiner: eval head →
~e.lookup_function(name)~ → if special-form → ~apply_special~ (calls the sf_* .o) ; if
