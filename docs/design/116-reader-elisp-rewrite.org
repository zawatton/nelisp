* §0 TL;DR

** §0.1 The Reader is the classical Lisp self-host milestone

A Lisp implementation that reads its own source in its own
language is — by tradition — the canonical self-host marker.
NeLisp already runs the *production* read path through elisp
(see §0.2 below), so the conceptual milestone is partially
achieved.  Doc 116 closes the remaining gap: the *baker* path
(= `nelisp-baker` dev tool that constructs frozen-heap images
from `.el` source) currently lexes and parses in Rust.  After
Doc 116 SHIPPED, every byte of Lisp source the project consumes
is read by Lisp.

| File             | LOC | Role                                  |
|------------------+-----+---------------------------------------|
| reader/mod.rs    | 260 | Public surface (`read_str' / `read_all') + ERTs |
| reader/lexer.rs  | 885 | Hand-rolled tokenizer state machine   |
| reader/parser.rs | 485 | Recursive-descent s-expression parser |
| *Total*          | *1,630* | (mod includes 200 LOC test fixtures) |

Excluding test fixtures, the live code is ~1,500 LOC of Rust to
remove.

** §0.2 cfg-gating context (= already dev-only)

Per Doc 98 §98.3 (2026-05-11), the *production* `nelisp` binary
never touches `reader::*`:

- Boot reads pre-baked NELIMG v3 frozen-heap globals via
  `image::decode_v3_into' (= dump-image fast path, zero parsing).
- User-visible `(read ...)` / `(read-from-string ...)` route to
  =lisp/nelisp-stdlib-reader.el= (= production elisp reader,
  already shipped).
- `eval_str' / `eval_str_all' route through
  `eval::read_all_with_line_via_elisp' (= elisp reader path with
  line-tracking, see =build-tool/src/eval/mod.rs:37,72=).

Surviving Rust callsites (= "dev tool only"):
- =build-tool/src/image.rs:169,1518= — `image::iterative_bake_one`
  parses each stdlib source on the way to its v3 image.
- =build-tool/src/bin/nelisp-baker.rs:221= — baker CLI entry.
- =build-tool/src/reader/*= ERT inner tests.

The `#[cfg(any(test, feature = "image-baker"))]` gate on `pub mod
reader;` (= =lib.rs:58=) means non-baker production builds already
exclude the module from the binary.

** §0.3 Why elisp化 anyway

Despite cfg-gating, ~1,500 LOC of Rust reader logic still ships in:
- `nelisp-baker` builds (= image-baker feature)
- `cargo test` (= `cfg(test)`)
- Image regeneration pipelines (CI + release)

Per repo charter (=CLAUDE.md=: 純 elisp 化 が repo の存在目的, Rust
削減 が唯一の評価軸) the reader being Rust is a contradiction —
NeLisp parses NeLisp source with NeLisp code.  Doc 116 removes the
contradiction.

** §1.2 =build-tool/src/reader/lexer.rs= (885 LOC, ~675 LOC live)

*Entry points (= public to mod.rs)*:

| Function                                                          | Returns                            |
|-------------------------------------------------------------------+------------------------------------|
| `pub fn tokenize(input: &str) -> Result<Vec<PositionedToken>, ReadError>` | All-tokens vector                  |
| `pub enum Token { LParen, RParen, ..., Symbol(String), ... }` (12 variants) | Token kinds                      |
| `pub struct PositionedToken { pub token: Token, pub pos: SourcePos }`    | Token + source location          |

*Algorithm* — single-pass hand-rolled tokenizer with byte-precise
column tracking:

1. *Outer loop*: `skip_whitespace_and_comments` → look at first byte → dispatch.
2. *Dispatch table* (line 159-209):
   - `(`, `)`, `[`, `]`, `'`, `` ` `` — single-byte tokens.
   - `,` — peek next, emit `Comma` or `CommaAt`.
   - `"` → `read_string` (escape resolution, `\n`/`\t`/`\xNN`/`\C-`/`\M-`/
     `\(` / `\<unknown>` fallthrough — Doc 51 Phase 3-A'' compat).
   - `#` → `read_sharpsign` (radix `#x` / `#o` / `#b`, function-quote
     `#'`, byte-code `#[` NYI, record `#s(`).
   - `?` → either bare symbol `?` (whitespace/EOF follower) or
     `read_char_literal` (chars accept any byte, escape codes,
     `\C-X`, `\M-X` nesting, `\xNN`).
   - else → `read_atom` (scan non-terminator bytes, try-parse as
     Int → Float → Symbol).
3. *State*: byte-offset position, line+column tracking with `\r\n`
   normalisation.

*Sub-functions*:
- `read_string` (~95 LOC, line 214-310) — escape resolution incl. line continuation, hex `\xNN`, named escapes (`\n`, `\t`, `\r`, `\e`, `\s`, `\b`, `\d`, `\a`, `\f`, `\v`, `\0`), unknown-`\X` fallthrough.
- `read_sharpsign` (~60 LOC, line 312-372) — `#x`/`#o`/`#b`/`#'`/`#s(`/`#[` NYI dispatch.
- `read_radix_int` (~40 LOC, line 374-414) — generic radix integer with optional sign + `_` digit separators.
- `read_atom` (~25 LOC, line 422-448) — accumulate non-terminator bytes, decide Int vs Float vs Symbol via `try_parse_int` / `try_parse_float`.
- `read_char_literal` (~145 LOC, line 450-594) — character escape DSL: `?a`, `?\n`, `?\xNN`, `?\C-a`, `?\M-a`, nested `?\M-\C-a`, `?\(` etc.
- Free helpers: `is_whitespace`, `is_atom_terminator`, `hex_digit`, `try_parse_int`, `try_parse_float` (~80 LOC, line 597-674).

*Sexp types produced*: none directly — emits `Token` enum values
that the parser converts.

*External dependencies*:
- `eval::error::{ReadError, SourcePos}` for error construction (= `ReadError::lex`, `::unexpected_eof`, `::not_yet_implemented`).
- Stdlib: `String::push`, `i64::from_str_radix`, `f64::from_str`,
  `str::parse`, `String::chars().filter()`.  Each of these is a
  Phase 47 grammar concern (see §3).

*Live code estimate*: ~675 LOC (= 885 - 210 `mod tests`).

** §1.3 =build-tool/src/reader/parser.rs= (485 LOC, ~290 LOC live)

*Entry points (= public to mod.rs)*:

| Function                                                       | Returns                       |
|----------------------------------------------------------------+-------------------------------|
| `pub fn parse_one(tokens: &[PositionedToken]) -> Result<(Sexp, usize), ReadError>` | First form + index next-unread |
| `pub fn parse_all(tokens: &[PositionedToken]) -> Result<Vec<Sexp>, ReadError>` | All forms                   |

*Algorithm* — single-class recursive-descent:

1. `Parser { tokens: &[PositionedToken], pos: usize }` carries the
   cursor.
2. `parse_form(depth, quasiquote_depth)` (line 74-139) — pops one
   token, dispatches:
   - `LParen` → `parse_list` (proper / dotted)
   - `LBracket` → `parse_vector`
   - `SharpsParen` → `parse_record` (= `#s(TYPE V0 ...)` per Doc 52)
   - `Quote` / `Backquote` / `FunctionQuote` → recursive parse +
     wrap in `(quote X)` / `(backquote X)` / `(function X)`
   - `Comma` / `CommaAt` → recursive parse + wrap (= Doc 51 Phase
     3-A''-3 — bare comma outside backquote allowed at parse time)
   - `Int` / `Float` / `Str` → direct Sexp leaf
   - `Symbol` → `symbol_to_sexp` (= recognise `nil` → `Sexp::Nil`,
     `t` → `Sexp::T`, else `Sexp::Symbol`)
   - `RParen` / `RBracket` / `Dot` → error (= unexpected close)
3. `parse_list` (line 151-200) — loop accumulating items; on `Dot`
   token, parse one tail form, require `RParen`, build improper
   list via `build_dotted`.
4. `parse_vector` (line 202-230) — same shape, rejects `Dot`.
5. `parse_record` (line 236-272) — first item = type tag, rest =
   slots; empty `#s()` rejected.

*Safety net*: `MAX_DEPTH = 256` recursion cap (line 26, ample
headroom for bootstrap source's ~10-deep nesting).

*Sexp types produced* (= the full reader output vocabulary):
- `Sexp::Nil`, `Sexp::T`
- `Sexp::Int(i64)`, `Sexp::Float(f64)`, `Sexp::Str(String)`,
  `Sexp::Symbol(String)`
- `Sexp::cons(car, cdr)` — proper list cons chains + dotted pairs
- `Sexp::list_from(&[...])` — convenience that folds via cons
- `Sexp::vector(Vec<Sexp>)` — `[...]` literals
- `Sexp::record(type_tag, Vec<Sexp>)` — `#s(...)` literals
- `Sexp::quote(x)` = `(quote x)`, `::backquote`, `::function`,
  `::comma`, `::comma_at` (= macro-shape helpers)

*External dependencies*:
- `super::lexer::{PositionedToken, Token}` — input
- `eval::error::{ReadError, SourcePos}` — error type
- `eval::sexp::Sexp` + its 10+ constructor helpers

*Live code estimate*: ~290 LOC (= 485 - 195 `mod tests`).

* §2 Phase 47 elisp化 plan — 4 stages

** §116.A — Lexer (= lisp/nelisp-cc-reader-lexer.el)

*Goal*: tokenizer state machine in pure elisp.  Single source
file holding `defconst …--source' that gets Phase 47 compiled into
`target/<...>/elisp-objects/nelisp_reader_lexer.o`.

*Output shape*: instead of `Vec<PositionedToken>` (= Rust-owned
heap), the elisp tokenizer returns a `Sexp::Vector` of token
records `#s(nelisp-token KIND POS-LINE POS-COL PAYLOAD)`.  Tokens
are pure-elisp `Sexp` values that the §116.B parser walks.

*Phase 47 ops needed*:
- =str-len=, =str-byte-at= — already shipped (Doc 101 §101.C).
- =int-eq=, =int-lt=, =int-le=, =int-ge=, =arith add/sub= — shipped (Doc 100).
- =cond=, =while= — Phase 47 core.
- =cons-make=, =cons-car=, =cons-cdr= — shipped (Doc 101).
- =vector-make=, =vector-slot-set=, =vector-len=, =vector-ref= — shipped (Doc 111).
- =record-make=, =record-slot-ref=, =record-slot-set= — shipped (Doc 111).
- *NEW* =mut-str-builder= (= push byte, finalize to `Sexp::Str`) — *Doc 122.B dependency*.
- *NEW* =mut-str-append-byte= — Doc 122.B.
- *NEW* `int-logior` / `int-logand` for hex escape composition —
  already shipped (Doc 115 §0.2 audit confirmed via Doc 100 §100.D arith op set).
- *Optional* char classification helpers: =is-ascii-digit=,
  =is-hex-digit=, =is-whitespace=.  Can be implemented in pure
  elisp via byte-range checks (= `(and (>= b 0x30) (<= b 0x39))`),
  or as Doc 122.C extern-call wrappers to libc `isdigit`/`isalpha`.
  *Decision*: pure-elisp byte-range checks — no extern needed,
  identical perf after Phase 47 lowering.

*Probe test plan*:
- New `build-tool/tests/elisp_cc_reader_lexer_probe.rs` (= Rust
  integration test that calls into the compiled elisp `.o`).
- Migrate the 16 existing `mod tests` cases in =lexer.rs= as
  probe cases: skips_whitespace_and_comments, parens_and_brackets,
  quote_prefix, quasiquote_prefixes, function_quote_prefix,
  dotted_pair_dot_is_token, integer_shapes, radix_integers,
  float_shapes, string_with_escapes, string_backslash_newline,
  unterminated_string_errors, string_unknown_escape_drops_backslash,
  string_paren_escapes, char_literal_shapes,
  char_literal_paren_escapes, char_literal_meta_modifier,
  char_literal_bad_hex_errors, symbol_with_punctuation.
- Add 5 cross-comparison probes: lex the same input with Rust + elisp
  side-by-side and assert token-stream equality (shake-out gate).

*Estimated effort*: 2-3 Agent dispatch days.  ~675 LOC Rust → ~700-800
LOC elisp (denser, slightly longer due to explicit `cond` arms vs
Rust pattern matching).

** §3.2 Gaps Doc 122 must close

Per the pure-elisp roadmap memory entry:
- Doc 122.A — *raw memory primitives*: `ptr-read-u8`, `ptr-read-u64`,
  `ptr-write-u8`, `ptr-write-u64`.  Needed by mut-str (= append
  bytes to a growable buffer).
- Doc 122.B — *mut-str builder*: `mut-str-new`, `mut-str-append-byte`,
  `mut-str-finalize` (= produce immutable `Sexp::Str`).  *This is the
  critical blocker for §116.A* — the lexer's `read_string`,
  `read_atom`, `read_radix_int` all build their output string char-
  by-char.  Without a mut-str primitive, the elisp port would have
  to construct strings via cons-list-of-bytes + final fold, which
  is correct but ~10× slower.
- Doc 122.C — *atomic ops* (atomic-load/store/cas).  *Not needed by
  the reader* (= reader is single-threaded, no shared state).  Skip.
- Doc 122.D — *libc binding for malloc / free*.  *Not needed by the
  reader* — mut-str builder uses Rust-allocated boxes per the existing
  NlBox pattern (= refcount-aware, allocator-stable).  Skip.

** §3.3 String slicing — design decision

Two approaches for the lexer's `read_atom` (= scan a run of non-
terminator bytes, then try-parse Int / Float / Symbol):

1. *Byte-range substring* — Phase 47 `(str-sub STR START END)` op.
   Requires a new grammar primitive.  Best perf.  *Cost*: +1 grammar
   op (= ~80 LOC asm helper + parser + emit).
2. *Byte-by-byte append into mut-str* — uses only Doc 122.B mut-str
   primitives.  No new grammar surface.  Slightly slower but the
   reader is already O(n) per source byte; the constant factor is
   negligible.

*Decision*: option 2 (= byte-by-byte mut-str append).  Adds zero
new grammar ops beyond Doc 122.B.  Lexer cold-init speed is dwarfed
by Phase 47 emit time; perf is not a deciding factor.

** §3.4 Char classification primitives

Lexer needs: is-digit, is-hex-digit, is-whitespace, is-atom-terminator,
is-radix-digit (for `#x` / `#o` / `#b`).

*Options*:
- Pure-elisp byte-range tests (= `(and (>= b 0x30) (<= b 0x39))`).
  ~5 LOC each, no new ops.  *Chosen*.
- libc `isdigit` / `isalpha` via extern-call.  Requires Doc 122.D
  libc binding for the char-class table.  Overkill.

*Decision*: pure-elisp byte-range checks.  Adds zero new grammar ops.

** §3.5 Error reporting — `(sexp . error-msg)` tuple

Rust's `ReadError` carries:
- enum kind: `Lex` / `Parse` / `UnexpectedEof` / `NotYetImplemented`
- `SourcePos { line: u32, col: u32 }`
- `String` message

Elisp port returns a `Sexp::cons(result-or-nil, error-record-or-nil)`
two-tuple from every entry point.  The Rust `nelisp_cc_read_str`
wrapper (§116.C) inspects the cdr; if non-nil, builds the matching
`ReadError` variant and returns `Err(...)`.  This pattern matches
how Doc 115 §115.7 FNV-1a returns its result (= bare Sexp::Int, no
error path needed) — and how Doc 117 Tier A handlers will surface
errors after the throw / signal primitive lands.

*New Phase 47 ops needed for error path*: zero.  All cons + record
+ Int + Str ops already shipped.

* §4 Risks + open questions

** §4.1 The reader is the bootstrap entry

Even though Doc 98 §98.3 removed reader from the production boot
path, the *image baker* still uses it to construct every NELIMG v3
image from `.el` source.  If the elisp reader has a bug, the
baker silently produces a broken image that boots into a corrupt
heap — symptom appears at first use of the affected stdlib
function, not at boot.

*Mitigation*:
- Mega-probe (§116.C) lex+parse-compares full =lisp/= corpus
  against the Rust legacy reader.  Must be 100% identical per-form

** §4.2 Test migration risk

The reader module is cfg-gated to `#[cfg(any(test, feature = "image-baker"))]`.

the gate (= elisp reader is cheap; no reason to gate it).

*Decision*: §116.D leaves the gate in place to minimise change
surface; Doc 121 (image baker elisp化) revisits gating.

** §4.5 Performance regression

The Rust reader is a tight byte-loop with `&[u8]` indexing — very
fast.  Phase 47-compiled elisp is also a tight loop after lowering,
but each `(str-byte-at STR I)` call has the Phase 47 extern-call
overhead (= ~3-5 cycles for cdecl prologue).

*Concrete estimate*: Rust reader on a 50KB elisp source = ~50µs.
Phase 47 elisp version = ~80-100µs (= 2× slower due to per-byte call
overhead).

*Verdict*: not a regression for the baker use-case (= images bake
once, then are reused).  Mega-probe (§116.C) includes 1 timing
assertion: full =lisp/= corpus reader wall-clock < 5× legacy Rust.

*If perf matters more later*: a future Doc 122.E could add inline-
mode hint to `(str-byte-at)` (= the compiler inlines the i64 load
instead of emitting an extern-call), recovering most of the gap.
