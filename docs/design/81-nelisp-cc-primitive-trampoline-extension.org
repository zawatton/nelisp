* §1 動機

** §1.1 Phase 7.1.6 prerequisite

Doc 28 §3.6 (= Phase 7.1.6 eval-site primitive lowering takeover) は
*Cranelift JIT が emit している =nl_jit_*= trampoline 群を nelisp-cc
backend で直 emit する経路*.  しかし 2026-05-09 時点の nelisp-cc は:

1. *SSA IR 上の primitive lowering opcode が無い* — =cons= / =car= /
   =cdr= / =setcar= / =setcdr= は =:call= opcode の callee meta で
   identify されるが、host machine code emit pattern は generic CALL
   rel32 (= System V AMD64 / AAPCS64 ABI 経由) しか持たない.
   =build-tool/src/jit/cons.rs= が emit する *inline NIL fast path* (=
   tag-byte load + immediate compare + conditional branch + helper
   tail-call) は表現不能.

2. *Sexp memory layout を読む opcode が無い* — Phase A.5.1 で =Sexp=
   は =#[repr(C, u8)]= に pin されており、tag-byte = offset 0 / payload
   ptr = offset 8 (=SEXP_PAYLOAD_OFFSET=) として inline で読める.  しかし
   nelisp-cc には =mov al, byte ptr [rdi]= / =mov rax, qword ptr
   [rdi+8]= 相当の SSA-level abstraction が無く、=:load-var= /
   =:store-var= は heap-cell / spill-slot 専用 (= elisp i64 wrapper).

3. *外部 C symbol への CALL emit path が無い* — 現 =--lower-call= は
   =nelisp-defs-index= 経由で elisp callee fixup を解決 (= 同一 mmap
   page 内 letrec / inner closure).  =nl_jit_cons_car= のような *外部
   linker symbol* を =Linkage::Import= 相当で resolve する path が無い.

4. *Trampoline ABI mode が無い* — 現
   =nelisp-cc-runtime-compile-and-allocate= の出力 page は host ABI
   (=System V AMD64= 6-arg int register / =AAPCS64= x0..x7) の
   =extern "C" fn(i64, i64, ..., i64) -> i64= 形状.  Cranelift trampoline
   は =extern "C" fn(*const Sexp, *mut Sexp) -> i64= (= unary) /
   =extern "C" fn(*const Sexp, *const Sexp, *mut Sexp) -> i64= (= binary
   constructor) で *out-pointer をもらって `i64` で OK/ERR を返す* 別 ABI.

* §2 Current nelisp-cc state

** §2.1 既存 SSA IR opcode set

=src/nelisp-cc.el= (= 1,985 LOC) + 13 module total *~12,263 LOC elisp*
(Doc 28 §0 LATEST に articulate 済).  SSA opcode set は backend pcase
dispatch (=src/nelisp-cc-x86_64.el::nelisp-cc-x86_64--lower-instr=
line 1984 / =src/nelisp-cc-arm64.el::nelisp-cc-arm64--lower-instr=
line 1111) で 10 種:

#+begin_example
  pcase opcode
    'const          → load literal i64 into def reg
    'load-var       → load named var (cell / spill / register)
    'store-var      → store def value to named var
    'call           → CALL rel32 + ABI marshalling (System V AMD64 / AAPCS64)
    'call-indirect  → CALL r/m64 (function-pointer call、closure 展開後)
    'closure        → LEA rax, [rip + cell:inner:N] (or alloc-closure rel32)
    'copy           → MOV reg, reg (phi-out 後の resolution)
    'branch         → TEST + JE/JNE (current = i64 reg vs 0 比較)
    'jump           → JMP rel32 (unconditional)
    'return         → MOV rax, value-reg + epilogue + RET
    'phi            → ERROR (= phi-out pass 必須)
#+end_example

これに =:tail-call=, =:tail-call-self=, =:self-direct-call=,
=:inner-function-index= 等の meta が plist で乗る (= Phase 7.7 inliner
の各 pass が annotate).

** §2.2 既存 backend emit dispatch (x86_64 / arm64)

各 backend は:
- *prologue / epilogue* = System V AMD64 (rbp save + rsp align) /
  AAPCS64 (fp/lr save + sp align)
- *register file* = 16 x86_64 GPR (rax..r15) / 32 arm64 GPR (x0..x30 + sp)
- *spill slot* = stack frame の RBP-relative / FP-relative slot
- *fixup table* = call-fixups (= elisp letrec 名 → 同 mmap page 内
  CALL rel32 disp32 patch) / cell-fixups (= =cell:NAME= label → RIP-rel
  load patch)

外部 C symbol fixup table は *存在しない*.

** §2.3 既存 entry-point ABI (= nelisp-cc-runtime-compile-and-allocate)

=src/nelisp-cc-runtime.el::nelisp-cc-runtime-compile-and-allocate=
(line 1336、1408 LOC total) の 8-step pipeline:

#+begin_example
  AST → SSA (T6 frontend)
      → Phase 7.7 passes (T158: escape / inline / rec-inline / lift)
      → safe-points (T11)
      → linear-scan (T4)
      → backend compile (T9 x86_64 / T10 arm64)
      → tail-call rewrite (`--lower-tail-call`)
      → safe-point stub injection
      → exec-page allocation (W^X mmap PROT_EXEC)
      → real exec via subprocess `nelisp-exec-bytes`
#+end_example

返り値 plist に =:exec-page=, =:final-bytes=, =:exec-result= 等.
ABI shape は固定 = =extern "C" fn(i64, i64, ..., i64) -> i64=.

** §2.4 既存 FFI call resolution (= --lower-call の制約)

=src/nelisp-cc-x86_64.el::nelisp-cc-x86_64--lower-call= (line 1415):
- =meta :fn CALLEE-SYM= = elisp symbol (= letrec 名 / inner closure 名)
- 出力 = =CALL rel32= with =disp32 = 0= placeholder + =call-fixups=
  entry =(fixup-offset . CALLEE-SYM)=
- Phase 7.5 link pass (= =nelisp-cc-x86_64-compile-with-link=) が
  =nelisp-defs-index= 経由で CALLEE-SYM の同 page 内 entry offset を
  resolve、disp32 に patch

*制約*: CALLEE-SYM は *同 mmap page 内の elisp letrec / inner* に限る.
外部 linker symbol (= =nl_jit_cons_car= 相当) は resolve 不能.

* §3 Extension scope

** §3.1 New entry-point ABI mode (= primitive trampoline shape)

=nelisp-cc-runtime-compile-and-allocate= に *第 4 引数 ENTRY-ABI* を
追加.  既定 =:host-int= (= 既存 =extern "C" fn(i64, ..., i64) -> i64=).
新値 =:trampoline-unary=, =:trampoline-binary-ctor=,
=:trampoline-binary-mut= で primitive trampoline shape を選択.

#+begin_src elisp
  ;; 既存 (= shape 1):
  (nelisp-cc-runtime-compile-and-allocate LAMBDA-FORM 'x86_64 EXEC-ARGS)
    ;; → page :: extern "C" fn(i64, i64, ..., i64) -> i64

  ;; 新 (= shape 2 = unary out-param trampoline):
  (nelisp-cc-runtime-compile-and-allocate LAMBDA-FORM 'x86_64 nil
                                          :entry-abi :trampoline-unary)
    ;; → page :: extern "C" fn(*const Sexp, *mut Sexp) -> i64
    ;; arg0 = arg_ptr (rdi / x0), arg1 = out_ptr (rsi / x1)
    ;; 返り値 = TRAMPOLINE_OK (0) / TRAMPOLINE_ERR (1)
#+end_src

実装 = backend prologue emit pass で =--frame-layout-mode=
(=:host-int= / =:trampoline-unary= / =:trampoline-binary-ctor= /
=:trampoline-binary-mut=) を選択.  =nelisp-cc-x86_64--emit-prologue= /
=nelisp-cc-arm64--emit-prologue= に mode 引数追加.

** §3.2 New SSA opcodes (= ssa-call-primitive, ssa-load-tag, ssa-load-payload-ptr, ssa-cmp-tag-imm)

| opcode               | operands                | meta            | semantics                                   |
|----------------------+-------------------------+-----------------+---------------------------------------------|
| =ssa-call-primitive= | (PRIM-SYM ARG-VALS...) | =:abi=          | 外部 C symbol への CALL.  =:abi= で          |
|                      |                          | (= shape の指定) | trampoline-unary / -binary-ctor / -binary-mut |
| =ssa-load-tag=       | (PTR-VAL)              | nil             | =mov al, byte ptr [PTR-VAL]= = tag byte     |
|                      |                          |                  | (8-bit zero-extend to i64)                   |
| =ssa-load-payload-ptr= | (PTR-VAL)            | nil             | =mov rax, qword ptr [PTR-VAL+8]= = payload   |
|                      |                          |                  | ptr (=NlConsBox*= / =NlVectorBox*= 等)       |
| =ssa-cmp-tag-imm=    | (TAG-VAL IMM-VAL)      | =:cond=         | i8 immediate compare + branch (= TAG_NIL    |
|                      |                          | (=:eq= / =:ne=) | / TAG_CONS / TAG_INT 等の constant 比較)     |

*注*: =ssa-call-primitive= は既存 =call= opcode と独立 (= meta merge で
 統合する option もあるが、call-fixups table の扱いを分離した方が
 link pass の責務が clean).  =ssa-load-tag= / =ssa-load-payload-ptr= は
 *unsafe memory read* (= GC safe-point の在 / 不在で扱いが変わる).

** §3.5 FFI call resolution path (= external C symbol via =declare-extern-c=)

新 fixup table: =extern-c-fixups= (= alist =((PATCH-OFFSET . SYMBOL-NAME) ...)=).
Phase 7.5 link pass (= =nelisp-cc-x86_64-compile-with-link= /
=nelisp-cc-arm64-compile-with-link=) で resolve.

resolve 戦略:
1. =mmap PROT_EXEC= page allocation 後、=dlsym= 相当で host runtime の
   =nl_jit_cons_*= symbol address を query (= Linux =dlsym(RTLD_DEFAULT,
   "nl_jit_cons_car")=、macOS =dlsym(RTLD_DEFAULT, ...)=)
2. *initial implementation* = =nelisp-cc-runtime= が elisp 側で symbol
   address を =anvil-runtime= module call で query (= 既存
   =nelisp-cc-runtime--exec-real= と同 subprocess bridge 経由)
3. *future optimization* = nelisp-cc-runtime に *known-extern-c
   table* (= alist =((SYM . ABS-ADDR) ...)= ) を keep して dlsym roundtrip
   を回避

*caveat*: System V AMD64 で CALL rel32 = ±2GB range、=dlsym= が返す
absolute address が mmap page から ±2GB を超える場合は *trampoline
indirection* (= =mov rax, abs64 / call rax=) に fallback.  arm64 BL
rel26 = ±128MB なので同様の indirection が頻出.

* §6 Risk + mitigation

** §6.1 性能回帰 (= elisp-side IR overhead vs Cranelift native)

risk: nelisp-cc が emit する trampoline shape の machine code 品質が
Cranelift baseline 未達 (= linear-scan vs Cranelift regalloc2 + LLVM
ISA encoder の差)

mitigation:
- Stage 81.4 の =cons-heavy bench= を *gate*.  baseline 未達なら 81.4
  blocked、emit pattern tuning を Stage 81.5 (= 仮設) で実施
- Phase 7.7 inliner との合算で Cranelift より速くなる可能性が高い (=
  Phase 7.1.5 fib 3964x の前例)
- worst-case = 5-10% perf gap → *user 2026-05-09 strategic directive
  「100% 純 elisp」を優先*、5-10% gap は acceptable

** §6.2 ABI 不一致 (= System V AMD64 vs AAPCS64)

risk: trampoline ABI shape (=*const Sexp, *mut Sexp -> i64=) は
=System V AMD64= = =rdi, rsi -> rax=、=AAPCS64= = =x0, x1 -> x0=、
両者で *out-pointer 経由の return value writeback* が必要.

mitigation:
- Stage 81.1 で x86_64 のみ実装、out-pointer writeback は明示的な
  =MOV qword ptr [rsi], src= emit
- Stage 81.2 で arm64 移植、=STR Xn, [X1]= で同等
- ERT に *cross-platform parity test* (= same elisp source、x86_64 /
  arm64 両方 compile + same Sexp output) を入れる

** §6.3 FFI call resolution の external C symbol 解決

risk: =dlsym= roundtrip が exec page 起動毎に走ると overhead 大、また
mmap page から symbol address ±2GB / ±128MB を超える場合の indirection
fallback.

mitigation:
- *known-extern-c table* (= compile-time hardcode、=anvil-runtime=
  module から symbol absolute address を取得して elisp side に baked
  in、=nelisp-cc-runtime-primitive-table= の field として保持)
- range overflow 時は *trampoline indirection* (= =mov rax, abs64 +
  call rax=) に自動 fallback、emit pattern を =--lower-call-primitive=
  内で range check で分岐
- ERT に *long-range fixup test* (= mmap page を high address に
  allocate して indirection path 強制) を入れる

** §6.4 Phase 7.1.5 LOCKED 維持 (= 既 close した perf gate を破らない)

risk: Doc 81 の SSA IR 拡張 (= 4 新 opcode + recognition pass) が
既存 =fib= / =fact-iter= / =alloc-heavy= 3-軸 bench に regression 入れる.

mitigation:
- Stage 81.1〜81.4 各 stage で =make bench-actual= を *gate*.
  regression 検知 (= baseline -5% 以下) なら stage 戻し
- primitive recognition pass は *opt-in* (= existing IR は =:call=
  opcode のまま、新 =:ssa-call-primitive= opcode は明示的に migrate
  された primitive のみ)
- existing 1086/1091 ERT 全 PASS を *hard gate* (= ERT 5 件
  failing は既存、Phase 7.7 fix で着手中、新規 fail は許容しない)
