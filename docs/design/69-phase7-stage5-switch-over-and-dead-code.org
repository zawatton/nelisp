** §1.3 "完全削除" が aspirational である理由

*** §1.3.1 bootstrap chicken-and-egg

=Env::new_global= bootstrap 中、=STDLIB_SOURCES= の elisp ファイルを順次
=eval= する.  =nelisp-stdlib-eval-core.el= は **最後** の entry なので、
それ以前の defun / defmacro / defvar 評価時点では elisp dispatcher
(=nelisp--apply-fn= 等) はまだ install されていない.  したがって
bootstrap 中の =apply_combiner= は **必ず** Rust =apply_function= 経路
を取る.

→ =apply_function= / =apply_closure= / =apply_lambda= / =apply_lambda_inner=
/ =bind_formals= は bootstrap path として削除不可.

*** §1.3.2 Rust API 多数 callsite

=apply_function= は build-tool 内の **多数の Rust 関数** から呼ばれる
(= 単に "elisp 関数を呼ぶ" Rust API として):

| caller | 用途 |
|--------+------|
| =eval/mod.rs= =apply_combiner= | dispatch 本体 (= Rust path 時) |
| =eval/mod.rs= =expand_macro= | macro lambda 適用 |
| =eval/builtins.rs= =bi_funcall= | =(funcall FN ARGS)= |
| =eval/builtins.rs= =bi_apply= | =(apply FN ARGS)= |
| =eval/builtins.rs= =bi_mapcar= | =(mapcar FN LIST)= |
| =eval/builtins.rs= =bi_load_file= | =load= で評価された関数の呼出 |
| =eval/builtins.rs= 他 5+ 箇所 | =run-hook-with-args= 等 |
| =bridge/takeover.rs= | Phase 8.0.3 self-host bridge (= 別 layer) |

→ =apply_function= 関数本体を削除すると上記すべての callsite が壊れる.
個別 callsite を =delegate_to_elisp_apply= 経由に書換える方法もあるが、
helper 検出 / depth 増減 / dispatch_form 構築 overhead が builtin 内
ループで嵩む (= mapcar 等で性能 regression リスク).

*** §1.3.3 helper exemption の必要性

=is_elisp_apply_helper= で除外している 11 helper (= apply-fn / apply-closure
/ apply-lambda / bind-formals--compute / -required-count / 4 predicate
/ expand-macro) は、user の defun と同じく =defun= で定義された
**普通の elisp 関数**.  ただし dispatcher 自身がこれらを呼ぶ際に再度
=nelisp--apply-fn= を経由すると infinite recursion (= consp が consp
を呼ぶ) するため、Rust =apply_function= で **直接** 適用する必要が
ある.

→ 仮に =apply_function= を delete しても、helper を呼ぶ専用の Rust
path は何らかの形で残さざるを得ない.

** §1.4 結論: Stage 7.5 の現実的 scope

Doc 63 §2.5 が予告した "~600 LOC 削減" は、現状の bootstrap + API 構造
では達成不能.  Stage 7.5 の現実的 deliverable は:

1. **検証**: make test 全 ERT + bench gate を両 mode で完走
2. **観察期間の体系化**: 何を watch するか、何が出たら escape hatch で
   切戻すか、を doc 化
3. **保守的削除**: Rust dispatch 経路で *呼ばれない* 関数 (= 現状の
   apply_combiner で見ると =is_macro= / =expand_macro= の Rust 版が
   候補) を elisp 経路に振替えて削除.  ~30-50 LOC 程度.
4. **将来計画**: bootstrap path を AOT bake (= Doc 63 §3.3) に移すなら
   更なる削除可能、その場合 Stage 7.7 として切出 (= Phase 7 範囲外).

* §3 削除候補の詳細

** §3.1 =is_macro= / =expand_macro= elisp 経由化

*** 現状

=apply_combiner= line ~196:

#+begin_src rust
if let Ok(func) = env.lookup_function(name) {
    if is_macro(&func) {
        let expansion = expand_macro(&func, tail, env)?;
        return eval(&expansion, env);
    }
    ...
}
#+end_src

bootstrap でも post-bootstrap でも同じ Rust path.  Stage 7.4.b で
elisp 側に =nelisp--macrop= / =nelisp--expand-macro= が install 済だが、
=apply_combiner= からは呼ばれない.

*** 案

elisp dispatcher の =nelisp--apply-fn= 内では既に =nelisp--macrop= が
使われている (= 内部 cond chain).  =apply_combiner= で macro を
=nelisp--apply-fn= に丸投げすると、elisp dispatcher が macro 展開も
担当 → Rust =is_macro= / =expand_macro= 不要.

ただし bootstrap 中は =nelisp--apply-fn= が無いので、bootstrap では
Rust 側を使う必要がある.  →

#+begin_src rust
// Stage 7.5.c (Doc 69)
if let Ok(func) = env.lookup_function(name) {
    if env.delegation_depth == 0
        && !is_elisp_apply_helper(name)
        && !is_builtin_value(&func)
        && env.has_function("nelisp--apply-fn")  // self-pacing
    {
        // 一括 elisp 委譲: macro / closure / lambda 全て delegate
        // 先で判定.  args は un-evaluated のまま渡す (= macro 用)
        return delegate_to_elisp_apply_with_unevaluated_args(&func, tail, env);
    }
    // Rust path (bootstrap / helper / builtin / escape hatch)
    if is_macro(&func) {
        let expansion = expand_macro(&func, tail, env)?;
        return eval(&expansion, env);
    }
    let args = eval_arg_list(tail, env)?;
    return apply_function(&func, &args, env);
}
#+end_src

ただし *args un-evaluated 経路* の追加は新たな contract で、=delegate_to_elisp_apply=
の仕様変更となる (= 現状は evaluated args を quote で渡す).  Risk
非小.

Stage 7.5.c で *調査+spike* し、リスクが小さければ採用.  困難なら
=is_macro= / =expand_macro= は据え置き、合計削減 ~27 LOC のみ

** §3.2 =apply_closure= / =apply_lambda= の inline

*** 現状

#+begin_src rust
pub fn apply_function(func: &Sexp, args: &[Sexp], env: &mut Env) -> Result<Sexp, EvalError> {
    match func {
        Sexp::Cons(head, _) => match &*head.borrow() {
            Sexp::Symbol(s) if s == "builtin" => apply_builtin(func, args, env),
            Sexp::Symbol(s) if s == "closure" => apply_closure(func, args, env),
            Sexp::Symbol(s) if s == "lambda" => {
                apply_lambda(func, &Sexp::Nil, args, env)
            }
            ...
        },
        ...
    }
}

fn apply_closure(closure: &Sexp, args: &[Sexp], env: &mut Env) -> Result<Sexp, EvalError> {
    let parts = list_elements(closure)?;
    if parts.len() < 3 { return Err(...) }
    apply_lambda_inner(&parts[1], &parts[2], &parts[3..], args, env)
}

fn apply_lambda(lambda: &Sexp, captured: &Sexp, args: &[Sexp], env: &mut Env) -> Result<Sexp, EvalError> {
    let parts = list_elements(lambda)?;
    if parts.len() < 2 { return Err(...) }
    apply_lambda_inner(captured, &parts[1], &parts[2..], args, env)
}
#+end_src

=apply_closure= / =apply_lambda= は単に list_elements + apply_lambda_inner
の薄いラッパ.  callsite が =apply_function= 1 箇所のみなので inline 化
で削除可.

*** 案

#+begin_src rust
// Stage 7.5.d
pub fn apply_function(func: &Sexp, args: &[Sexp], env: &mut Env) -> Result<Sexp, EvalError> {
    match func {
        Sexp::Cons(head, _) => match &*head.borrow() {
            Sexp::Symbol(s) if s == "builtin" => apply_builtin(func, args, env),
            Sexp::Symbol(s) if s == "closure" => {
                let parts = list_elements(func)?;
                if parts.len() < 3 {
                    return Err(EvalError::Internal("closure missing env / args / body".into()));
                }
                apply_lambda_inner(&parts[1], &parts[2], &parts[3..], args, env)
            }
            Sexp::Symbol(s) if s == "lambda" => {
                let parts = list_elements(func)?;
                if parts.len() < 2 {
                    return Err(EvalError::Internal("lambda missing args / body".into()));
                }
                apply_lambda_inner(&Sexp::Nil, &parts[1], &parts[2..], args, env)
            }
            Sexp::Symbol(s) if s == "macro" => Err(EvalError::WrongType { ... }),
            _ => Err(EvalError::WrongType { ... }),
        },
        _ => Err(EvalError::WrongType { ... }),
    }
}
#+end_src

LOC 削減: ~27 (= 関数 2 つ削除 - inline コードの増加).

*** Risk

低い.  callsite 1 箇所のみ、ロジック完全等価、ERT regression 検出が
容易.

=(macro lambda ARGS BODY)= の =sf_lambda= 評価時 captured-env に
=apply_combiner= の状態が混入する subtle case があるかも.  Stage 7.4.d
で観測した capture leak の類似 surface.  Stage 7.5.c spike で慎重に
検証する.

mitigation: spike が FAIL なら §4.3 を skip、§4.4 のみで確定.

** §7.2 bench wallclock < 1.8x 未達

elisp dispatch overhead が予想以上に大きく、bench gate FAIL する case
が出るかも.  特に =bench-funcall-builtin-1m= (= builtin を 100 万回
funcall) は dispatch overhead がモロに見える.

mitigation: Phase 5 native cc lowering (= Doc 62) で =funcall= /
=apply= / hot-path 関数を JIT lower 化.  Stage 7.5.b の bench 結果次第
で Stage 7.5.e として計画.

** §7.3 Phase 8.0.3 bridge との二重管理

=bridge/takeover.rs= は別の self-host 機構.  Phase 7 と Phase 8 が
共存するなら維持コストが二重.  Stage 7.5 の scope 外だが、Stage 7.6
以降で統合検討.
