*** §2.4.1 Reader feature gate concept overview (= dev-only gating)

Stage 8.4 の本質は *reader files を物理削除しない* ことにある.
=build-tool/src/reader/{lexer,parser,mod}.rs= を:

1. =cfg(any(test, feature = "image-baker"))= attribute で *production
   build から conditional に exclude*
2. *default features 設定 = =[]== (= image-baker は opt-in、現状の
   =Cargo.toml= 19-30 行で確認済)
3. *production runtime* (= image-baked path、Stage 7.7.b SHIPPED) は
   reader module を build graph から外す → binary size 削減
4. *elisp side reader* (= =lisp/nelisp-stdlib-reader.el= Phase 7.2 SHIPPED)
   が post-bootstrap 唯一の reader、bridge / production code の read 操作は
   全て elisp 経由

「物理削除しない」理由 = =image-baker= feature build (= dev tooling)、
=cargo test --features image-baker= (= cargo unit fixture) で reader が
必要なため.  source は git track 維持、production graph 上の effective LOC
だけが削減される.

注: user's task spec で言及された =reader= feature 追加 alternative も
検討したが、既存 =image-baker= feature が reader callsite 唯一の survivor

*** §2.4.5 Risk + mitigation

**** Risk 1 — Bootstrap circularity (= image bake 自体に reader 必要)

- 現象: Phase 9 frozen heap baker (= Doc 75 Stage 9.3+9.4) は内部で
  =image::compile_elisp_to_image= → =reader::read_all= を呼ぶ.  reader が
  cfg gate 下に降りた状態で baker が build できないと bootstrap circularity
  発症
- mitigation: Phase 9 image baker は =cargo build --release --bin nelisp-baker --features image-baker=
  で build される (= =Cargo.toml:44 required-features = ["image-baker"]=
  で既に enforced).  baker 専用 binary が image-baker feature 下にある間、
  reader が gate されても *baker build path では reader 入る* ので circularity
  なし
- verify step: Stage 8.4.d で =cargo build --release --bin nelisp-baker
  --features image-baker= が PASS することを確認

**** Risk 2 — elisp reader perf parity (= post-Stage 8.4 唯一の post-bootstrap reader)

- 現象: Stage 8.4 後 production runtime は =lisp/nelisp-stdlib-reader.el=
  (= elisp reader、Phase 7.2 SHIPPED) のみ使用.  仮に elisp reader が Rust
  reader 比 ~10x 遅い場合、production load 時間 regression
- mitigation: Phase 7.2 SHIPPED 時に perf 確認済 (= Doc 66 §X perf gate、
  =bridge_bootstrap= bench で elisp reader 経由 + image cache が Rust reader
  経由比 ≦ 1.05x).  Phase 7.7.b の AOT image bake (= STDLIB 15 ファイル
  pre-baked) で起動 reader 呼び出し回数自体が ~0 に近い
- verify step: Stage 8.4.d で =bench/bridge_bootstrap= に perf gate 追加
  (= Phase 7 baseline 維持、≦ +5% latency)

**** Risk 3 — Test coverage (= reader が cfg-gated されると Rust-side reader test は dev only)

- 現象: =build-tool/src/reader/*.rs= 内 =#[cfg(test)] mod tests= は cfg gate
  下に入ると =cargo test --release= (= default features) では実行されない.
  reader regression が dev mode test でしか検出されない silent gap
- mitigation: cfg gate を =cfg(any(test, feature = "image-baker"))= に

** §2.5 Stage 8.5 — bootstrap eval frozen heap (= Stage 7.7.e、Phase 9 候補)

STDLIB 15 files を起動毎に eval するのではなく、build 時に env を
構築済 image として serialize、起動時に直接 deserialize.

- =Env::new_global= の所要時間 ~80ms+ → ~5ms (deserialize のみ)

  serialize 形式で扱う必要

  は不可 — 現実的には builtins / defvar / defun の result snapshot
  のみ frozen 化、closures は起動時再構築

- Phase 8 (= reader 削除) と独立に扱える

  という user feedback が出る可能性あり

* §4 Risks / Mitigation

** §4.1 anvil-runtime の error type 互換性
