** §1.1 Phase 7 完了形 (= 2026-05-08 時点)

** §1.2 Phase 8 の本質的目的

- *Production binary から Rust reader/eval を完全切離*: 産品 =nelisp= +
  =anvil-runtime= が =reader/= module を一切 link しない状態にする
- *Rust LOC 大幅削減*: =reader/{lexer,parser,error,mod}.rs= = 1,779 LOC
  を =cfg(image-baker)= gate に降ろし、production build から落とす
- *Sexp 型の論理 home を eval に明確化*: 現状 =reader/sexp.rs= 671 LOC が
  Sexp 値型を所有しているが、Stage 7.7.c.1 以降 reader は parse-only
  なので Sexp 型は =eval= が所有すべき

  type を返すので、=eval/error.rs= に rehome すれば feature gate
  問題が解消.

** §2.3 Stage 8.3 — bridge/loader.rs migrate (= line-tracking variant 必要)

bridge bootstrap が =reader::read_all= + =reader::lexer::tokenize= +
=reader::parser::parse_one= を呼ぶのは、各 form に行番号を付けて
error 時に =BridgeError::EvalError("line N in ...")= を出すため.

elisp reader をそのまま使うと line 情報が失われるので、まず elisp
reader 側に line-tracking variant を追加.

*** §2.3.a — =nelisp--read-all-with-line= 追加

=lisp/nelisp-stdlib-reader.el= に新 entrypoint:

#+begin_src emacs-lisp
(defun nelisp--read-all-with-line-from-string-impl (input)
  "Like `nelisp--read-all-from-string-impl' but each result is
\(LINE . FORM\) where LINE is the 1-origin source line at which
FORM started."
  ...)
#+end_src

実装は既存 reader internal state (= =:position= cursor) を流用、
parse 開始時の position を record して 1-origin line に変換.

*** §2.3.b — bridge =parse_tracked_forms= migrate

=bridge/loader.rs::parse_tracked_forms= を migrate:

- 既存 =reader::lexer::tokenize= + =reader::parser::parse_one= の二段
  parse を、=eval::read_all_with_line_via_elisp(source, env)= 1 callに
- =TrackedForm { form, line }= の取得元が変わるが struct 自体は維持
- =reader::fmt_sexp= 呼出 (= 行 306, 308) は Stage 8.1 で
  =eval::sexp::fmt_sexp= に migrate 済

*** Stage 8.3 が *達成しなかった* こと

- bridge bootstrap が STDLIB_IMAGES の elisp reader に依存 — chicken-
  and-egg は無し (= STDLIB は image deserialize で先に load 済、bridge
  はその後)
- 5 canonical files を image bake する選択肢は別途 Stage 8.5 (= frozen
  heap) で扱う

*** 検証

- =cargo test --release= 全 PASS
- ERT =bridge::tests::*= 全 PASS (4 case)
- 既存 anvil-runtime smoke test (= bootstrap_self_host 完走) PASS

*** 推定コスト

- 2 commit (8.3.a + 8.3.b)、~100 LOC + reader.el ~30 LOC
- elisp reader extension は moderate 難度 (line tracking は既存 cursor
  state から導出可)

** §2.5 Stage 8.5 — bootstrap eval frozen heap (= Stage 7.7.e、Phase 9 候補)

STDLIB 15 files を起動毎に eval するのではなく、build 時に env を
構築済 image として serialize、起動時に直接 deserialize.

*** 期待効果

- =Env::new_global= の所要時間 ~80ms+ → ~5ms (deserialize のみ)
- bridge canonical files も同様に frozen heap 化すれば bridge bootstrap
  も skip 可

*** Risks / 設計課題

- =Sexp= cycle (= cycle-safe equal pattern を Doc 50 で構築済) を
  serialize 形式で扱う必要
- =Rc<RefCell<...>>= の identity を deserialize 後も保つには、内部
  pointer を index に置換する custom serializer 必要
- closures (= =Sexp::Function= variant、=Frame= ref を keep) の serialize
  は不可 — 現実的には builtins / defvar / defun の result snapshot
  のみ frozen 化、closures は起動時再構築

*** Phase 9 に分離する判断

- 規模が大きい (~500 LOC + image format v3 拡張)
- Phase 8 (= reader 削除) と独立に扱える
- frozen heap は startup speed 改善であり、現状の AOT bake で十分実用
  という user feedback が出る可能性あり

→ *Phase 8 から外す*. 別 Doc (= Doc 74 候補) で扱う.

* §4 Risks / Mitigation

** §4.1 anvil-runtime の error type 互換性

- Stage 8.2 で =ReadError= → =EvalError= に migrate するが、
  =anvil-runtime= の =AnvilModuleRegistryError::ModuleParse= /
  =AnvilHostRegistryError::AnvilHostParse= が =message: String= で
  保持しているため、外側 (= MCP error) には影響しない
- *Mitigation*: error message の prefix が変わる (=nelisp-read-error: ...= →
  =elisp signal "error" with: ...=) ので test fixture の文字列 match
  を確認.  影響あれば test だけ書き換え

** §4.2 elisp reader の line tracking 精度

- Stage 8.3.a で =position= cursor → 1-origin line 計算する際、
  multi-byte char や =\r\n= 改行で off-by-one リスクあり
- *Mitigation*: =\n= count + Rust reader が出していた行番号と
  cross-equiv ERT を 5 case 用意 (= 既存 bridge 行追跡で動いていた
  fixture を流用)

** §4.3 Sexp rehome の git diff churn

- Stage 8.1 で ~10 ファイルの import 一括書き換え + =reader/sexp.rs= →
  =eval/sexp.rs= の物理 move (= git mv で履歴保持)
- *Mitigation*: 1 commit に集約、commit message で =git log --follow=
  での参照方法を articulate.  =git mv= で blame 履歴は維持される

** §4.4 reader module の cfg gate が IDE / rust-analyzer に与える影響

- =cfg(feature = "image-baker")= 下に降ろした reader が rust-analyzer
  で grey-out される (= LSP completion から消える)
- *Mitigation*: dev で reader を触る際は =cargo +nightly check --features image-baker=
  を README に記述、IDE 側設定で =rust-analyzer.cargo.features = ["image-baker"]=
  を推奨

** §4.5 frozen heap の serialize 複雑度 (Stage 8.5 = Phase 9 候補)

- 既述の通り cycle / closure 周辺で実装コスト大
- *Mitigation*: Phase 8 では Stage 8.5 を out-of-scope とし、Phase 9
  Doc 74 で別途扱う.  本 Doc は §2.5 で articulate 済

* §9 改判断 record

(2026-05-08, Claude session continuation; Phase 7 main scope SHIPPED 後)

設計時点での判断:

1. *Stage 8.5 (= Stage 7.7.e frozen heap) を Phase 9 に分離*: 当初
   Phase 8 は "Rust reader/eval 完全削除 + frozen heap" を含む scope
   だったが、frozen heap は serialize 形式設計 + cycle / closure 対応で
   ~500 LOC 規模の独立タスクと判明.  Phase 8 は "reader 物理削除 +
   eval 完全切離" に focus し、frozen heap は別 Doc 74 で扱う.
2. *Stage 8.1 (Sexp rehome) を最初に置く*: type-only refactor で
   behavior change なし、後続 sub-stage の前提.  逆順 (= 先に bridge
   migrate) も検討したが、Sexp が =reader/sexp.rs= に居る間は
   "reader が Sexp の home" という言外の依存が残り、削除コストが
   上がる.
3. *=ReadError= を =eval/error.rs= に移す*: =reader/error.rs= に残す
   と =cfg(image-baker)= gate 下になり、production code の
   =EvalError::Read= variant がbroken.  =eval/error.rs= rehome で
   feature gate 整合性を確保.
4. *=read_one_via_elisp= の cfg(test) gate を解除する判断は保留*:
   現状 cargo test 専用、Stage 8.2 で anvil-runtime からも使いたく
   なるが、anvil-runtime は =read_all_via_elisp= で十分 (= 1 form
   ケースは =into_iter().next()= で取得可).  =read_one_via_elisp=
   は test 専用のままにする.
