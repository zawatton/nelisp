* 1. 目標と非目標

** 1.1 目標 (Phase 3b 完了条件)

1. =src/nelisp-bytecode.el= 新規: compiler + VM + dispatch table
2. 既存 =nelisp-eval= の evaluator path と *共存* (coexist) — 単独の
   form は interpreter 経由、closure 呼び出しは VM fast-path
3. Phase 1/2 の 50+ ERT + Phase 3a 20 ERT が *全て* VM 経由でも pass
4. cycle-1 = cycle-2 fixpoint が VM ON で引き続き green
5. =nelisp-bench-fib-30= の wall-time が host interpreter (non-byte-compiled
   同等 NeLisp form を recursive eval した場合) の 1/10 以下 (性能 10x 改善
   を最低、roadmap §4 の 5x 目標は stretch)
6. Lambda の引数が =defvar= 済 symbol の場合、VM bind op が dynamic 側に
   push する。Phase 2 test =nelisp-dynamic-test.el= の lambda param
   ケースが pass

** 1.2 非目標 (Phase 3b では入れない)

- *Rhodes 5-stage genesis* (shared-tables 置換え) — Phase 3b 後に独立
  phase (v0.4.x / v0.5.0 候補)。bytecode ABI が stable になってから着手
- *=.elc= 互換 reader/writer* — Phase 5+ で community contrib として検討
- *Native comp* — Phase 3 architecture doc §1 Candidate C が PARKED。
  NeLisp 独自 bytecode が先
- *Generational GC* — =05-roadmap.org= §4 "generational は Phase 3 後半で
  検討"。Phase 3b の範囲は mark-sweep 見込み、実装自体は Phase 3c に回す
- *Fiber yield point の実装* — =04-concurrency.org= §5 Phase 3 対応は
  dispatch loop に yield 埋め込む「設計余地を残す」レベルまで。
  実動 fiber primitive は Phase 3c 後半 / Phase 4
- *Tail call optimization (TCO)* — Phase 1 interpreter も TCO 未実装。
  Phase 3b は explicit =apply= 経由の tail call のみ非再帰化。Proper TCO は
  Phase 3b.7 で余裕あれば

* 2. Key decisions (user signoff 待ち)

以下 5 点は Phase 3b 着手前に合意が必要。各項目に推奨と代替を併記。

** 2.1 Bytecode format

| 選択肢 | 内容 | 評価 |
|---|---|---|
| **推奨 B** | *独自 bytecode* (Emacs inspired だが非互換) | 設計自由度、fiber yield 埋込容易 |
| A | Emacs bytecode 完全互換 | =.elc= loader future、validation via Emacs、op 呼び出しの quirks 継承 |
| C | Hybrid (op 名称 Emacs 準拠、encoding 独自) | 移植性 vs 自由度の中庸、doc 量最多 |

推奨 *B*。NeLisp は既に Phase 2.5 で host recursive から離脱した。
Emacs =.elc= 互換は将来 loader 層で翻訳できる余地がある。

** 2.2 Stack machine vs register machine

- *推奨: Stack machine*
  - Pro: Emacs 準拠、Phase 1-2 evaluator の mental model 継承
  - Pro: Compiler が単純 (各式 → push 命令列)
  - Con: Register 比 ~20% inst 増 (我々の 5-10x 目標には影響軽微)
- 代替: Register machine (Lua / Dalvik 流)
  - Pro: dispatch 回数少、inst あたり仕事大
  - Con: Register allocation 複雑、Phase 3b の実装工数が 1.5-2 倍

Phase 3b は *stack* 決定。Register は Phase 5+ で re-eval 余地。

** 2.3 Interpreter との共存 (coexist) or 置き換え

- *推奨: Coexist* (Phase 3b 完了時点)
  - =nelisp-eval-form= + =nelisp-tramp-eval= は残置
  - Closure に *compiled code* 属性が付いていれば VM dispatch、
    無ければ interpreter
  - Top-level form (defun / defvar / defmacro) は interpreter 処理、
    closure 内の body は compile 済
- 代替: 完全置き換え (全 form を compile、interpreter 除去)
  - Con: Phase 3b scope が膨大、cycle-2 fixpoint の risk 高
  - Pro: 将来 performant。Phase 4 で再評価

** 2.4 Lambda dynamic binding 統合

- *推奨: Phase 3b 内で対応* (§1.1 goal 6)
  - VM compile 時、各 lambda param を =nelisp--specials= で check
  - Special な param → `BIND-DYNAMIC` op、通常 → stack slot (`BIND-LEX`)
  - 理由: VM の binding ops 設計と同時に決める方が frame layout が安定
- 代替: Phase 3c / 独立 phase
  - Pro: Phase 3b 完了を早められる
  - Con: 後から VM binding semantics を変えるのは risk

** 2.5 性能目標

- *推奨: 最低 10x (host interp 比)、stretch 5x (Emacs bytecode 比)*
  - Roadmap §4 の 5x は Emacs bytecode 比の stretch
  - Phase 3b の merge gate は「Phase 3a 以前の test が全 pass」と
    「benchmark が host interp の *少なくとも* 10x faster」
- 代替: Roadmap 原文通り 5x (Emacs bytecode 比) を hard gate
  - Con: Phase 3b.7 で gate 不達なら phase 長期化
  - Pro: 告知時に defensible claim

** 3b.1 Skeleton (設計固め)

** 3b.2 Minimal VM (const / return / call0)

- =(nelisp-bc-run BCL)= で定数返しだけ動く
- ERT: const literal return / stack-ref 等、~10 test

** 3b.3 Control flow + arithmetic

  =PLUS=, =MINUS=, =LESS=, =GREATER=, =EQ=, =NOT=

  に落とす

** 3b.4 Special form coverage

実装の粒度が大きいので、3b.4 は a/b/c の三段構成で ship する。
各段階で make test green を必須にし、commit を分ける。

*** 3b.4a Variable access + binding / setq / while (SHIPPED)

- Op 追加: =VARREF= (16) / =VARSET= (17) / =VARBIND= (18) / =UNBIND= (19)
  / =STACK-SET= (20) / =DISCARDN= (21)
- Compiler が lex-env (symbol → abs stack-slot) を track
- =setq= / =let= / =let*= / =while= / symbol reference をサポート
- =let= は parallel (init を outer env で全評価 → slot 割当 / VARBIND)、
  =let*= は sequential (binding ごとに lex-env 拡張)
- defvar 済 special は VARBIND/UNBIND で specpdl に save/restore、
  VM が非 local exit しても globals を復元 (=unwind-protect= 経由)
- ERT: 25 本追加 (3b.4a-*)、357/357 green

*** 3b.4b List primitives + generic CALL (SHIPPED)

- Op 追加: =CAR= (22) / =CDR= (23) / =CONS= (24) / =LIST1..LIST4=
  (25-28) / =LISTN= (29) / =CALL= (30)
- =cons= / =car= / =cdr= / =list= を primitive として compile
- generic =(f arg...)= は =CALL= に落とし、VM 側で =nelisp--apply=
  経由で primitive / interpreter closure を dispatch
- 未対応の special form (=lambda= / =function= / =defun= / =and= /
  =or= / =when= / =unless= / =prog1= / =prog2= / =catch= / =throw= /
  =unwind-protect= / =condition-case= / =defvar= / =defconst= /
  =defmacro=) は明示的に =nelisp-bc-unimplemented= を signal
- ERT: 14 本追加 (3b.4b-*)、371/371 green

*** 3b.4c Error-handling forms (SHIPPED)

- Op 追加: =PUSH-CATCH= (31) / =POP-HANDLER= (32) / =THROW= (33) /
  =PUSH-UNWIND= (34) / =PUSH-CC= (35)
- dispatcher を =cl-labels= で re-entrant 化、host =catch= /
  =unwind-protect= / =condition-case= を per-op で wrap
- =catch= / =throw= / =unwind-protect= / =condition-case= をサポート
  - =catch= は fresh cons sentinel で「body 正常終了」と「throw 値
    がたまたま = sentinel」を区別
  - =unwind-protect= は cleanup を nested dispatch で再入実行、
    非 local exit 時は sp / specpdl を pre-body snapshot にリセット
  - =condition-case= は handler spec を inline にディスパッチ、match
    は host helper =nelisp-bc--cc-match-p= に CALL で委譲、VAR は
    lexical/dyn/nil 3 パターン対応、no-match 時は =(signal (car err)
    (cdr err))= で re-raise
- ERT: 19 本追加 (3b.4c-*)、390/390 green

Phase 3b.5 (closure + lambda dynamic binding) と束ねて取り扱う。
現状 compile-and-run で動くのは body 内に =lambda= / =defun= を

** 3b.5 Closure + lambda dynamic binding

3b.4 と同様に a/b/c の三段構成で進める。

*** 3b.5a Lambda + arg binding (no capture) (SHIPPED)

- 新規 op 不要 (既存 VARBIND/UNBIND/DISCARDN/STACK-REF を再利用)
- =(lambda PARAMS BODY)= → sub-bcl を template として CONSTS に
  詰め、parent code は =CONST= で push
- sub-bcl の preamble: special param 毎に =STACK-REF + VARBIND=
  (param が dyn なら specpdl に save-and-install)
- sub-bcl の postamble: =UNBIND num-dyn=, =DISCARDN N-positionals=,
  =RETURN=
- =nelisp-bc-run bcl args= が PARAMS spec をもとに &optional /
  &rest を materialise、stack[0..N-1] に prefill して PC=0 開始
- =nelisp--apply= に bcl dispatch を追加 (host eval / VM 共存)
- 自由 lex 参照は parent-lex-shadow で検出して
  =nelisp-bc-unimplemented= を signal (3b.5b 待ち)
- ERT: 14 本追加 (3b.5a-*)、404/404 green

*** 3b.5b Free-lex capture (SHIPPED)

- 新規 op: =MAKE-CLOSURE N= (36, 1 byte) / =CAPTURED-REF idx= (37, 1 byte)
- ctx に =captures= alist (sym → capture-idx) を追加。compile-var-ref が
  自由 lex を見つけたら自動 allocate
- compile-lambda の sub-ctx は親の lex-env / captures / parent-lex-shadow
  を全て継承して shadow-source とする → multi-level closure を自然対応
- compile-lambda 完了後、sub.captures があれば parent code に
  「captures push (各 sym を parent の compile-var-ref で resolve) →
  CONST template → MAKE-CLOSURE N」を emit
- runtime: =MAKE-CLOSURE n= は stack 上の N captures + 1 template を
  pop、新規 bcl を env=(cap0..capN-1) で構築
- runtime: =CAPTURED-REF idx= は現在 bcl の env から N 番目を読む
- ERT: 12 本追加 (3b.5b-*)、416/416 green。multi-level / closure-from-
  closure / 3 段 chain / capture snapshot 不変性 / let* / capture +
  special 共存もカバー

*** 3b.5c defun auto-compile hook (SHIPPED, opt-in)

- =nelisp-bc-try-compile-lambda= (env params body) — env が nil の
  時のみ =(lambda PARAMS BODY)= を compile して bcl 返す。失敗
  (nelisp-bc-error 系) は nil を返して interpreter fallback
- =nelisp--make-closure= は =nelisp-bc-auto-compile= が non-nil で
  かつ try-compile が bcl を返したら、それを installed function に
  使う
- =nelisp-bc-auto-compile= 既定は **nil**。理由 2 つ:
  1. =setq= on captured lex を bcl で表現する CAPTURED-SET op が
     未実装 — そういう closure が含まれると compile が bail し、
     ネスト全体が fallback する (3b.5d 候補)
  2. tramp / self-host テストが NeLisp-on-NeLisp で深く再帰し、
     auto-compile の数 frame でも =max-lisp-eval-depth= を超える
- tramp dispatch も bcl-aware に (=nelisp--apply= 経由で走らせる)
- 既定 nil でも、benchmark / 個別テスト時に
  =(let ((nelisp-bc-auto-compile t)) ...)= で opt-in 可
- ERT: 9 本 + flag default の 1 本 = 10 本追加。426/426 green

*** 3b.5d (将来) — CAPTURED-SET / 部分 fallback

- =setq= on captured lex を bcl 化する CAPTURED-SET op を追加すれば
  3b.5c の bail 範囲がさらに減る
- 内側 lambda が unsupported でも外側を bcl 化する partial-fallback も
  検討余地あり (現状は 1 個でも bail で全体 fallback)

* 4. Bytecode 構造 (草案)

** 4.1 Bytecode closure object

=(nelisp-bcl ENV PARAMS CONSTS CODE STACK-DEPTH SPECIAL-MASK)=

- =ENV=: captured lexical env (alist、interpreter closure と同じ表現)
- =PARAMS=: 引数 spec list (Phase 1 の lambda-list、&optional / &rest 含む)
- =CONSTS=: constant vector (symbols / literals / nested closure)
- =CODE=: 命令列 (Phase 3b.1 は Elisp vector of ints、後で unibyte
  string に最適化余地)
- =STACK-DEPTH=: 最大 stack 深さ (VM が alloc サイズ決定)
- =SPECIAL-MASK=: PARAMS の i 番目が special なら bit_i = 1 の整数。
  Phase 3b.5 で使用

Phase 3b.1 では =SPECIAL-MASK= は常に 0、=CODE= は空 vector。

** 4.2 Opcode encoding (草案)

- 1-byte opcode (= 最大 256 op。Phase 3b では 50 以下で十分)
- 引数付き op は後続 1-2 byte で index / offset を持つ
- Phase 3b.1 は int vector で prototype、3b.7 で unibyte string 化検討

** 4.3 Stack frame

Phase 3b の stack は単一の大きい vector。frame 内訳:

#+begin_src
[ ... caller frame ...
  ret-pc
  saved-sp
  local 0  (= param 0 if lexical)
  local 1
  ...
  temp 0
  temp 1
  ... <- sp ]
#+end_src

=BIND-DYNAMIC= は locals に入れずに =nelisp--globals= を直接 push/pop。

* 5. Interpreter ↔ VM interop

- VM 内 =CALL= op:
  - target が =nelisp-bcl= → VM 直接 dispatch (no host recursion)
  - target が =nelisp-closure= → work-stack に marker push し
    trampoline に handoff → 結果戻る
  - target が primitive (elisp lambda / subr) → host funcall
- Interpreter 内 =nelisp--apply-closure=:
  - closure 形が =nelisp-bcl= → =nelisp-bc-run= に委譲
  - それ以外 Phase 2.5 通り

これにより、*どちらからでも* 他方を呼べる。cold init の top-level は
interpreter、hot loop は VM。
