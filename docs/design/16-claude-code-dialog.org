* 観察ポイント

- *id:1 initialize* — =serverInfo=, =capabilities.tools.listChanged
  = false=, =protocolVersion = 2024-11-05= が MCP 2024-11-05 spec
  準拠の shape
- *notifications/initialized* に対して reply 無し (JSON-RPC 2.0
  notification は明示的に reply を返さない規約)
- *id:2 tools/list* — §3.3 で port した 7 tool 全てが =name= /
  =description= / =inputSchema= 揃って返る。=inputSchema= の
  =required= は JSON array として =["path"]= 形式で serialize
- *id:3 tools/call git-log* — subprocess spawn 経由で実 git 出力、
  =:exit 0= と =:log= string 両方が MCP =content.text= envelope
  に prin1 form で wrap
- *id:4 / id:5 data-set-path → data-get-path* — 同一 session 内
  の state round trip。=42= (integer) が JSON 経由で element 往復
  した後も型保持

* 既知の限界 (Phase 5-E 範囲)

- state は single process = 単一 emacs --batch 起動中のみ保持。
  Claude Code が MCP server を再起動 (session restart / config
  reload) すると data-store は消える
- =http-get= は同期 =url-retrieve-synchronously= wrap、MCP call
  中は全 request が直列化する (Phase 5-D worker 層を通す
  async integration は Phase 5-F+)
- destructive tool (=file-replace-string= / =file-batch= 等) は
  §2.4 non-goal で port 済でない、Phase 5-F+ で risk 管理しながら
