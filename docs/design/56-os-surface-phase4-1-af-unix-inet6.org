* §0 TL;DR

Doc 55 Phase 4 で AF_INET の socket family 専用 primitive (=bi_syscall_bind_inet=
/ =-connect-inet= / =-accept-inet=) を ship した。本 Phase 4.1 はその直接拡張で
AF_UNIX + AF_INET6 を同形に追加する。socket / listen / setsockopt-int / poll /
read / write / close は family 非依存なので既存のものをそのまま再利用、family
個別なのは sockaddr 構築 + accept の peer 情報 unpack のみ。

scope = 6 専用 primitive + 6 elisp wrapper + 7 定数 + 2 ERT。

| 区分    | 名前                                  | 提供層        |
|---------+---------------------------------------+---------------|
| AF_UNIX | bind / connect / accept (filesystem path) | 専用 primitive |
| AF_INET6 | bind / connect / accept (sockaddr_in6)    | 専用 primitive |

設計指針 (= Doc 53/54/55 を継承): *Rust 追加を最小化する*。

- =socket= / =listen= / =setsockopt-int= / =poll= / =read= / =write= / =close=
  は family 非依存なので Doc 55 / 53 の wrapper をそのまま使う。本 Phase は
  bind/connect/accept の sockaddr 構築/解体だけ family 別の専用 primitive で
  受ける (= 計 6 本)。
- AF_UNIX path は filesystem path 限定 (Linux abstract socket = sun_path[0]=NUL
  prefix は scope 外、Phase 4.1.1 で別途検討)。
- AF_INET6 アドレスは elisp 側で *length-8 list of 16-bit integers* (= group
  単位、host byte order) として扱う。e.g. =::1= は ='(0 0 0 0 0 0 0 1)=、
  Rust 側で各 group を big-endian にパックして in6_addr に積む。

Path A (Linux) のみ。Path B (Darwin/Windows) fallback は他 Phase と同じく
deferred。

* §1 動機

Doc 55 で TCP loopback が elisp 完結したが、IPC ばかりは AF_UNIX / 現代的な
modern stack には AF_INET6 が必須。両 family の socket 構築は AF_INET と
*formula 同一* (sockaddr → libc::bind/connect/accept) なので、追加コストは
sockaddr_un / sockaddr_in6 をそれぞれ 1 ペアの primitive で受けるだけで済む。

実用 use case:

- *AF_UNIX*: Cowork 環境内の elisp ↔ elisp プロセス間 RPC、editor ↔ daemon の
  IPC、=anvil= 系工具との bridge を nl-ffi-call なしで elisp で書ける line。
- *AF_INET6*: dual-stack TCP server / client。最近の loopback でも ::1 経由が
  multipath として要求されるケースがある (= IPv6-only host)。

将来 (Phase 4.1.1 候補): AF_UNIX abstract namespace、SOCK_SEQPACKET、
SCM_RIGHTS で fd-passing。
