* §1 Env / SymbolEntry internals (= field-by-field analysis)

** §1.1 SymbolEntry struct (= 4 fields)

#+begin_src rust
pub struct SymbolEntry {
    pub value: Option<Sexp>,
    pub function: Option<Sexp>,
    pub plist: Option<Sexp>,
    pub constant: bool,
}
#+end_src

| field | 型 | 意味 | 現 Rust 用途 | elisp 表現 |
|-------+-----+------+---------------+-------------|
| =value= | =Option<Sexp>= | symbol の value cell (= =defvar= / =setq= の格納先) | =lookup_value= / =set_value= で読み書き | =(plist :value V)= の =V= or =:unbound= sentinel |
| =function= | =Option<Sexp>= | symbol の function cell (= =defun= / =defalias= / =fset= の格納先) | =lookup_function= / =set_function= | =(plist :function F)= |
| =plist= | =Option<Sexp>= | property list (= =put= / =get= 用) | 未使用 (= 初期 nil)、TODO comment | =(plist :plist P)= or skip (=t= = 全 symbol がまず empty plist 持ち) |
| =constant= | =bool= | =setq= 不可 flag (= =nil= / =t= / keyword) | =set_value= で SettingConstant signal | =(plist :constant t)= or 別 hash-set =nelisp--constant-symbols= |

elisp 側の natural 表現は **plist** (= elisp 自身の =get= / =put= で
直接操作可能) または小さな **vector record** (= =(record 'nelisp-symbol-
entry VALUE FUNCTION PLIST CONSTANT)= による cl-defstruct 風).

判断: =plist= が low-overhead (= record は =NlRecord= 経由で boot
order の depth を増やす).

** §1.2 Env struct (= 7 fields)

#+begin_src rust
pub struct Env {
    pub globals: HashMap<String, SymbolEntry>,
    pub frames: Vec<Frame>,                          // Frame = HashMap<String, FrameCell>
    pub max_recursion: u32,
    pub current_recursion: u32,
    pub extern_builtins: HashMap<String, ExternBuiltin>,
    pub use_elisp_apply: bool,
    pub delegation_depth: u32,
}
pub type FrameCell = crate::eval::nlcell::NlCellRef;
pub type Frame = HashMap<String, FrameCell>;
#+end_src

*** §1.2.1 globals (= 主犯、~80% の env op が触る)

- 型: =HashMap<String, SymbolEntry>=
- 用途: =defvar= / =defun= / =fset= / =defconst= / Rust builtin の install /
  primitive constant (= =nil= / =t=) の保存先
- 操作: =get= / =insert= / =or_insert_with= / =get_mut= が基本
- elisp 表現: =nelisp--global-env= = =make-hash-table :test 'equal= 1
  個、各 entry は =(NAME . SYMBOL-ENTRY)= or 直 plist
- migration 工法: option A (alist + hash) を base、Tier 1/2 は Tier 0
  shim 経由

*** §1.2.2 frames (= lexical scope stack)

- 型: =Vec<Frame>= where =Frame = HashMap<String, FrameCell>=
- 各 frame = =let= / =let*= / lambda body 1 個に対応 (= push_frame / pop_frame
  でペア管理)
- =FrameCell= = layout-pinned =NlCellRef= (= Phase A.4 で
  =Rc<RefCell<Sexp>>= から migrate 済) = closure write-through 用
- elisp 表現:
  - **option A1**: elisp =let= form の native frame stack を直接利用
    (= =sf_let= 自身が elisp 化されれば自動的に動く)
  - **option A2**: =nelisp--frame-stack= = nested alist =(((a . cell-a)
    (b . cell-b)) ((c . cell-c)) ...)=、明示的に push / pop する Rust
    builtin =bi_push_frame= / =bi_pop_frame= は既存 (= Doc 68 Stage 7.4.a
    で SHIPPED)
- 判断: option A2 が現実的 (= option A1 は Rust =sf_let= を完全 elisp
  化する必要があり、§86.3 を超えた scope)
- 既存 helper: =bi_push_frame= / =bi_pop_frame= / =bi_push_captured= /
  =bi_bind_local= が builtins.rs にすでに存在 (= line 2389-2428)、scope
  chain elisp 経路の primitive はほぼ揃っている

*** §1.2.3 max_recursion / current_recursion (= u32 counters)

- recursion guard、=eval= 入口で increment / 出口で decrement
- elisp 表現: =defvar nelisp--current-recursion 0= + =defvar nelisp--
  max-recursion 1024=、Rust 側 =eval= は elisp の =nelisp--enter-recursion=
  / =nelisp--exit-recursion= を呼ぶ shim に変える ... ただしこれは hot
  path (= 全 eval call で touch)、Rust 側 u32 incr/decr (= ~2 instruction)
  → elisp helper 経由 (= ~1000+ instruction) は perf-prohibitive
- 判断: counter 自体は **Rust 側に残置** (= elisp 化の意味 0、§86.3 OOS)、
  elisp が読み取りたい場合のみ =bi_get-current-recursion= primitive を
  追加

*** §1.2.4 extern_builtins (= host crate registry)

- 型: =HashMap<String, ExternBuiltin>= where =ExternBuiltin = Rc<dyn Fn(&[Sexp], &mut Env) -> Result<Sexp, EvalError>>=
- 用途: =nelisp-emacs-gtk= 等の host crate が GTK / SDL primitive を
  register
- elisp 化不可: Rust closure を elisp が hold する手段なし (= Sexp::Cell
  に dyn Fn は wrap できない)
- 判断: **Rust 側に永久残置** (= §86.3 OOS、Tier 0 minimal env に含める)

*** §1.2.5 use_elisp_apply / delegation_depth (= apply chain delegate flags)

- =use_elisp_apply= = bool、bootstrap 完了後 ON、ERT で flip 可能
- =delegation_depth= = u32、=delegate_to_elisp_apply= 内で incr/decr で
  recursion gate
- 既に Stage 7.4.c〜7.4.e で elisp 経路と互換、=bi_set_use_elisp_apply= /
  =bi_get_use_elisp_apply= で elisp visibility あり
- 判断: counter 系は Rust 残置 (=current_recursion= と同理由)

** §1.3 fields の elisp 化対象 / Rust 残置 分類

| field             | elisp 化対象? | 理由                                             |
|-------------------+---------------+--------------------------------------------------|
| =globals=         | YES           | §86.3 中核、~80% の op が touch              |
| =frames=          | YES (= partial) | 既 primitive 経由で elisp visible、明示 push/pop は elisp 駆動 |
| =max_recursion=   | NO            | hot-path counter、elisp 化 = perf prohibitive |
| =current_recursion= | NO          | 同上                                             |
| =extern_builtins= | NO            | Rust closure を elisp で hold できない         |
| =use_elisp_apply= | NO (= leave as-is) | 既に elisp visible (= bi_set/get)、bool は移管メリット 0 |
| =delegation_depth=| NO            | hot-path counter                                 |

つまり **真に elisp 化対象は =globals= と =frames= の 2 field のみ**.
他は Rust 側 Tier 0 stub に残置して shim 経由で elisp に visibility を
提供する.

** §2.3 image.rs serialisation (= migration の隠れ依存)

=build-tool/src/image.rs= は globals を AOT bake 形式で serialise する:

- =image.rs:229= = =env.globals.keys()= の iterate
- =image.rs:530= = =SymbolEntry::default()= で entry 復元
- =image.rs:544= = =env.globals.insert(name, entry)= で復元

elisp 化すると **bake / unbake の両 sides で wire format が変わる**.
shadow path (= §86.3.b) では:

1. Rust =Env::globals= が canonical、elisp =nelisp--global-env= は
   mirror
2. image.rs は Rust 側を read/write して既存 wire format 維持
3. switch-over (= §86.3.c) で wire format を v4 に bump (= elisp
   hash-table の serialisation)、または elisp 側を Rust =globals= の
   thin view に維持して image.rs 不変

判断: 後者が安全 (= image.rs の wire format change は orthogonal scope、
切り離す).  =Env::globals= は Rust 側に永久残置の Tier 0 minimal env
として、elisp =nelisp--global-env= は **その上のキャッシュ view** =
read-through、write-through の dual-update.

これで image.rs は zero change、Phase 3 (= 究極 D-B) で wire format
revisit のみ.

* §3 Implementation strategy (= option A/B/hybrid trade-off)

** §3.1 option A: alist + hash (= 純 elisp)

elisp =make-hash-table :test 'equal= 1 個 = =nelisp--global-env=、各
entry は plist =(:value V :function F :plist P :constant nil)=.

#+begin_src emacs-lisp
(defvar nelisp--global-env (make-hash-table :test 'equal))

(defun nelisp--symbol-entry (name)
  "Return the SymbolEntry plist for NAME, creating empty if absent."
  (let ((e (gethash name nelisp--global-env)))
    (or e
        (let ((fresh (list :value nil :function nil :plist nil :constant nil)))
          (puthash name fresh nelisp--global-env)
          fresh))))

(defun nelisp--lookup-value (name)
  "Look up NAME's value cell."
  (let ((e (gethash name nelisp--global-env)))
    (cond
     ((null e) (signal 'void-variable (cons name nil)))
     (t (let ((v (plist-get e :value)))
          (if (eq v :unbound)
              (signal 'void-variable (cons name nil))
            v))))))
#+end_src

*** trade-off

- *優*: 純 elisp、Rust 側 layout pin 不要、Phase 3 (= 究極 D-B) との
  整合性 100%
- *劣*: hash lookup overhead が elisp interpreter loop 経由 = ~10-50x の
  hot path slowdown (= Doc 84 §84.3 Box accessor elisp 化と同 pattern)
- *劣*: =frames= の =FrameCell= = =NlCellRef= は Rust struct なので、
  elisp =hash-table= の value 側に `Sexp::Cell` 経由で wrap する layer
  が必要

** §3.2 option B: payload pointer 経由 (= Doc 81 ABI mode 拡張)

Rust =Env::globals= の memory layout を pin、elisp が =nl-jit-call-out-*=
trampoline で payload pointer 直接アクセス.

#+begin_src emacs-lisp
(defun nelisp--lookup-value (name)
  (nl-jit-call-globals-lookup-value name))  ; → Rust ptr → SymbolEntry → value cell
#+end_src

Rust 側に slim primitive を 8-12 件追加:

|----------------------------------------+---------+-------------------|
| =bi_globals_get_value=                 | 8       | =lookup_value=    |
| =bi_globals_set_value=                 | 10      | =set_value=       |
| =bi_globals_get_function=              | 8       | =lookup_function= |
| =bi_globals_set_function=              | 6       | =set_function=    |
| =bi_globals_clear_value=               | 5       | =makunbound=      |
| =bi_globals_clear_function=            | 5       | =fmakunbound=     |
| =bi_globals_is_bound=                  | 6       | =boundp=          |
| =bi_globals_is_fbound=                 | 6       | =fboundp=         |
| =bi_globals_is_constant=               | 5       | =constant= flag check |
| =bi_globals_set_constant=              | 5       | =defconst=        |
| =bi_globals_capture_lexical=           | 17      | (= =capture_lexical=) |
| 既存 =bi_push_frame= / =bi_pop_frame= 等は流用 | — | (= 既 SHIPPED) |

*** trade-off

- *優*: hot path ほぼ無 overhead (= Rust 直接 lookup と同等)、image.rs
  wire format 不変
- *優*: 既 SHIPPED の Stage 7.4.a primitive (= =bi_push_frame= 等) と
  同一 pattern = leverage 可能
- *劣*: elisp 化「度」が低い = Phase 3 究極 D-B で再 audit が必要 (=
  Phase 3 で Rust =Env= 自体を消す path が必要)
- *劣*: env.rs LOC 削減効果は薄い (= primitive が +80 LOC で打ち消す)

** §3.3 option C: hybrid (= recommended)

- *Tier 0 stub* (= Rust 残置、~50 LOC):
  - =Env::globals= = =HashMap<String, SymbolEntry>= 維持 (= image.rs
    wire format / extern_builtins / recursion counter のため)
  - =Env::frames= の =push_frame= / =pop_frame= / =push_captured= /
    =bind_local= は既存 primitive (= Stage 7.4.a) のまま、Rust 側で
    frame stack 管理
  - =SymbolEntry= struct も維持 (= image.rs 互換)
- *option B 経路* (= 8 件 slim primitive 追加、~80 LOC):
  - =bi_globals_get_value= / =-set_value= / =-get_function= / =-set_function=
    / =-clear_value= / =-clear_function= / =-is_bound= / =-is_fbound=
  - 既存の Tier 3 dispatch arm (= =symbol-value= / =fset= / etc.) は
    これらの slim primitive に reduce (= 各 arm 平均 -5 LOC body、12
    arm × -5 = -60 LOC builtins.rs)
- *option A 経路* (= elisp 側 ~700 LOC):
  - =nelisp--global-env-shim.el= = elisp 側 primitive helper (=
    =nelisp--lookup-value= / =-set-value= / etc.)、内部で slim primitive
    を呼ぶ
  - =nelisp--bind-local= 等の既存 primitive と統合
  - 連動先 (= =mod.rs= の =bind_formals= / =special_forms.rs= の
    =sf_let= / etc.) を elisp 経由に書き換え

*** §3.3.1 option C choice 理由

1. *Phase 2 単独 着地点* = ~17,000-19,000 LOC を満たす最短経路
2. *Phase 3 (D-B) との互換*: option B の slim primitive は Phase 3 で
   trampoline 経由のままにできる、option A の hash-table 経路は Phase 3
   の Rust =Env= 消滅と整合

* §7 Lexical vs dynamic binding handling

** §7.1 NeLisp 現状 = "全部 lexical" (= bootstrap 制約)

env.rs comment line 11-14:

#+begin_quote
Even though Elisp default is dynamic binding, the bootstrap interpreter
implements *lexical* binding for `let`, `lambda` argument lists, and
`let*` because (a) the prompt §6.5 names this scope explicitly and (b)
the Phase 7.1 native compiler takeover only sees lexical-binding code
anyway.
#+end_quote

つまり NeLisp 全体が =-*- lexical-binding: t -*-= 前提.  Emacs Lisp 的
な dynamic =defvar= "special variable" semantics (= 同名の =let= が
dynamic shadowing する) は **NeLisp では実装されていない**.

** §7.2 elisp 化後の binding 実現

| binding 種別        | 現 Rust 経路                | elisp 経由             |
|---------------------+-----------------------------+------------------------|
| lexical =let=       | =push_frame= + =bind_local= + body + =pop_frame= | 既 primitive 経由 (= 不変) |
| lexical =let*=      | 同上 (= 順次 bind)         | 同上                   |
| lexical lambda 引数 | =bind_formals= + =push_frame= | =nelisp--bind-formals= elisp 化 |
| dynamic =defvar=    | =env.defvar= = global value cell 上書きのみ | shim 経由             |
| dynamic =setq-default= | (= 未実装、defvar 同等) | 同上                  |

NeLisp の dynamic binding は **global value cell 上書き** (= Emacs と
比べて単純化されている) なので、shim 経由でそのまま実現可能.

** §7.3 elisp native =let= form との interaction

elisp =let= form は NeLisp =sf_let= (= Tier 1 special form、Rust 残置)
が実装.  =sf_let= が =bind_local= 経由で elisp shim を呼ぶ場合:

#+begin_src text
sf_let body 評価
   ↓
push_frame (= Rust direct, Tier 0 op)
   ↓
各 binding に対して:
  bind_local (= Rust direct, Tier 0 op)
   ↓
body 評価 (= recursive eval)
   ↓
pop_frame (= Rust direct, Tier 0 op)
#+end_src

frame stack 操作は **Tier 0 direct**、value lookup は **shim 経由** =
hot path も safe.  =setq= だけが shim を呼ぶ (= =set_value= 経由).

** §7.4 special variable semantics

Emacs Lisp の "special variable" = =defvar= した symbol が =let= で
dynamic binding される.  NeLisp では:

- =defvar= した symbol も全 lexical = =let= は frame に bind、global
  value cell には触れない
- =setq= が **frame で先 hit**、frame 不在で global cell に fall-through
  する `set_value` semantics (= env.rs:351-374)

elisp 化後も同 semantics 維持.  shim の =nelisp--set-value= は frame
walk を Rust 側で行い (= primitive 経由)、無ければ global hash-table
に書く.

* §10 Risk / unknowns

** §10.1 hot path perf regression (HIGH)

=lookup_value= は eval ループ内で **全 symbol resolve** で hit する.
shim 経由 (= Rust → slim primitive → elisp shim defun → elisp lookup)
は ~10-50x slowdown 可能性.  Doc 84 §84.3 の Box accessor elisp 化と
同 pattern.

mitigation:
- slim primitive を **直接 dispatch** で hit させる (= elisp shim defun
  経由しない)、elisp shim はあくまで user-visible API + advice hook 用
- Rust 内部 callsite (= =mod.rs= eval ループ) は引き続き Rust =Env=
  method 経由、shim は user code のみ
- bench gate (= Doc 62 Phase 5 / Doc 76 bench) を §86.3.c 完了直後に
  re-run、threshold 5% regression なら revert

** §10.2 image.rs wire format (MEDIUM)

§2.3 で articulate 通り、image.rs は Rust =Env::globals= を canonical
扱い、shim は read-through cache のため wire format 不変.  ただし future
Phase 3 で wire format v4 への bump が必要 = 別 doc で扱う.

** §10.3 chicken-and-egg (MEDIUM, mitigated)

§6 で Tier 0 permanent stub 戦略で解決.  shim load 中の self-call 回避
は実 ship で検証必要.

** §10.4 extern_builtins (LOW)

Rust closure を elisp で持てない問題は §1.2.4 で articulate.  Tier 0
永久残置で OK、host crate (= =nelisp-emacs-gtk=) も既存 API 維持.

** §10.5 Doc 86 §4.3 試算誤り (LOW)

Doc 86 §4.3 の "ffi.rs -100 LOC" は grep 0 件 = 誤り.  Doc 86 § net
delta -1,500 → -1,214 へ訂正.  本 Doc 89 SHIPPED 後に Doc 86 に correction
note を追加.

** §10.6 Tier 3 arm 数の不確実性

Doc 86 §2.2.3 で "Tier 3 ~12 arm" としたが、本 Doc §2.2.3 grep で 12
user-visible + 4 helper = 16 件確認.  Doc 86 試算は user-visible のみで
妥当、helper は §86.3.a 既 SHIPPED の primitive (= 不変).

** §10.7 §86.1 Tier 3 vs §86.3 の dispatch arm 削除順序

§86.1 Tier 3 (= 12 arm 削減 -200 LOC) と §86.3.c (= 同 12 arm の env
shim 経由化) の重複に注意.  本 Doc では **§86.3.c が canonical** = arm
削除 + slim primitive 経由化を §86.3 で実施、§86.1 Tier 3 は post-§86.3
の clean-up sub-stage に縮退する案.

→ Doc 87 (= §86.1 detailed design) で再 articulate 必要.

** §10.8 ERT covered surface (LOW)

env shim の defun が 11 件、各 5-10 ERT で計 ~70 ERT 必要.  既存 ERT
(= 1300+) は env API 経路 100% カバーしているので shadow phase で
divergence 検知は確実.
