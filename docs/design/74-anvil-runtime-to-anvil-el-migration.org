* §1 背景

** §1.1 現状の Rust / elisp 分離

architecture_correction_anvil_uses_nelisp_2026-04-25.org §1.2 LOCKED:

| layer | 担当 | LOC |
|-------+------+-----|

| =anvil.el= リポジトリ | Emacs-attached helpers (= anvil-host.el 等) | ~多数 |

=anvil-runtime= は "emacs-less standalone" の MCP server. 起動時に
=anvil-host.el= を =read_all= でパース → =eval_via_self_host= で
評価 → tool registry 構築. その後 MCP request を受けて tool dispatch
を Rust 側で組み立て、=eval_via_self_host= で elisp 関数を呼出.

** §1.2 なぜ anvil.el 側に移すのか

User durable feedback =feedback_nelisp_minimize_rust= 通り:

#+begin_quote
Rust 最小化、elisp 化可能なら elisp。JIT + native comp で速度問題なし。
Rust 実装も migrate していく
#+end_quote

=anvil-runtime= の orchestration (= file load / tool registry / dispatch
form 構築 / formal introspection / JSON ↔ Sexp 変換) は本質的に
elisp で記述できる内容で、Rust に書く必然性が薄い. 残すべき Rust
は:
- MCP transport (= stdio JSON-RPC 受信 / 送信ループ)
- =nelisp= standalone binary (= eval entrypoint)

それ以外の orchestration は =anvil.el= に移す方が:
- *単一 source of truth*: tool dispatch logic が Emacs-attached / -less
  両モードで共通化される (= 現状は Rust dispatch + Emacs server dispatch
  の二重実装)
- *iteration 速度*: elisp で書けば再 build 不要、=eval-buffer= で即反映
- *Rust LOC 削減*: 5,667 LOC → ~500-800 LOC (= MCP transport 残り) に
  減る見込み

** §2.4 移行困難な部分

- *MCP stdio transport*: JSON-RPC over stdio loop は elisp で書ける
  が、毎メッセージで Lisp object → JSON string 変換 + flush + lock
  管理が必要. Rust で書く方が remote 実装と互換性高い.
  → *残す* (= ~500 LOC、=mcp/= 配下 + =bin/anvil-runtime.rs= の minimal shim)
- *NeLisp interpreter binary*: =nelisp= command 自体は Rust + elisp
  混合実装. これは Phase 7 で既に最小化済 (= eval は Rust、reader は
  elisp).  *残す* (= 既存)

** §3.2 Rust ↔ elisp boundary

- *Rust 側*:
  - stdio JSON-RPC 1 byte 単位の receive loop
  - JSON parse / serialize
  - request を 1 form (= =(anvil-mcp-handle-request JSON)=) に詰めて
    elisp に投げる
  - elisp の return を JSON で stdout に送出
- *elisp 側*:
  - =anvil-mcp-handle-request= = MCP request ディスパッチ
  - tool registry (= 起動時に各 anvil-*-tools.el を load)
  - tool call → JSON args 変換 → elisp 関数呼出 → return JSON 変換
  - error 時 JSON-RPC error 形式に整形

要するに Rust = transport、elisp = orchestration + business logic.

- Rust 側 =anvil_host_registry.rs= は =--elisp-dispatch= mode で skip、

*検証*: =anvil-mcp-host-info= 等が Rust dispatch / elisp dispatch 両

* §5 Risks / Mitigation

** §5.1 性能

- elisp dispatch は Rust dispatch より overhead 大 (= function lookup,
  apply, JSON encode/decode を全部 elisp で行う)
- *Mitigation*: bench 測定、典型 tool で latency p99 が 5ms 以下なら
  許容. JIT (Cranelift, Phase 5) が有効化されれば diff 大半解消

** §5.2 既存 anvil-runtime tests の port

- Rust の =anvil_host_registry.rs= 等には ~1,000 LOC の test fixture
  がある (= anvil-host parse / formal introspection / dispatch shape)
- *Mitigation*: 各 elisp port stage で対応する ERT を追加、Rust test
  は段階的に削除 (= elisp port 完了で Rust test ignore + 後で削除)

** §5.3 anvil.el repo 側の build / load order

- elisp ファイル間 dep (= require) graph が複雑化する可能性
- *Mitigation*: =provide= / =require= で明示的に整理、各新規 file の
  header に dep 列挙

** §5.4 user 既存 workflow への影響

- =anvil mcp serve --no-emacs= で起動する user は Rust binary 経由で
  動く前提
- *Mitigation*: Stage A で =--elisp-dispatch= mode 追加、default は
  既存 Rust dispatch のまま. 全 stage 完了で switch.  Rust binary 自体
  の CLI surface は不変

** §5.5 cross-PC 互換 (Linux / Windows)

- elisp port は OS-agnostic に書く (= =system-type= 分岐は anvil-host.el
  既存 pattern を踏襲)
- Windows でも shell command 動作確認
