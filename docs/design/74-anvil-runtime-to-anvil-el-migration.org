* §4 Migration option 比較

** §4.1 Option A — anvil-runtime を current Sexp ABI に追従させる (= rejected)

Phase A.2.1 / A.4.x ABI 変更を anvil-runtime 内に追従:
- =Sexp::Cons(car, cdr)= → =Sexp::Cons(rc)= (= 10 callsite を rewrite)
- =NlCellRef.borrow()= → =NlCellRef.deref()= (= N callsite)
- ERT も同期させる

| 観点 | 評価 |
|------+------|
| Rust LOC delta | *+0 〜 +50* (= 修正コードの方が冗長になる場合あり) |
| user 2026-05-09 commitment | *違反* (= net-decrease 不可) |
| 工数 | ~1 day |
| 中長期 ABI 変更追従コスト | 継続発生 (= Phase A.5+ で再 sync 必須) |
| Rust 最小化方針 | 反する (= elisp 化可能 orchestration を Rust に温存) |

→ *rejected*.

** §4.2 Option B — 一気に elisp port + crate 削除 (= big-bang)

anvil-runtime 全 5,667 LOC の elisp port を 1 stage で作成、Rust 側を
1 commit で全削除:

| 観点 | 評価 |
|------+------|
| Rust LOC delta | *-5,667* (= 1 commit) |
| user commitment | 1 commit で達成、しかし途中 commit 不在で daily granularity に合わない |
| 工数 | ~7 day 連続 |
| risk | 巨大 commit、cross-mode equivalence 検証困難、回帰時に bisect 困難 |
| reader feature gate (Doc 73 Stage 8.4) unblock | crate 削除 commit 直後 |

→ *rejected*.  monotonic shrink rule を 1 commit で満たすが、daily
operating discipline と整合しない (= 各日に Rust LOC 純減 commit を
1 個以上出す前提の方が user の operating model と合致).

** §4.3 Option C — fix-then-incremental-delete (= *recommended*)

Stage 74.1 で *minimum-effort fix* (= compile error 14 個解消、Rust LOC
delta = ±0 ぴったり、または -10 程度) で anvil-runtime を再びビルド可能
にする.  これが Stage 8.4 unblock の "前段の前段". その後 Stage 74.2-74.k
で各 registry を 1 stage = 1 elisp port + 対応 Rust source 削除で
incremental migrate. Stage 74.final で残った crate を削除.

| 観点 | 評価 |
|------+------|
| 各 stage Rust LOC delta | =≤ 0= (= 制約付きで articulate、§5) |
| user commitment | 各 commit で satisfy |
| 工数 | 各 stage ~1 day、計 ~7 day だが pipeline 化可 |
| risk | per-stage equivalence 検証で回帰局所化 |
| reader feature gate unblock | Stage 74.4 完了 (= =read_all= callsite 全消滅) で達成 |

注: Stage 74.1 (= compile fix) は厳密に言うと "user 2026-05-09 ルールの
gray zone" (= ±0 commit).  しかし無修正のまま放置すると以降の Phase A
sync 毎に technical debt が膨らむため、**例外的に net-zero commit を
許容** する判断.  代替として Stage 74.1 を skip して Stage 74.2 から
始めることも可能だが、その場合 Stage 74.2 の elisp port 作業中に
Rust 側 anvil-runtime が依然として broken state なので integration test
が走らない (= cross-mode equivalence 不可能).  Stage 74.1 で一時的に
build を通す方が 検証 baseline 確立の点で優位.

→ *Option C を推奨*.

* §6 Risk

** §6.1 Cross-crate ABI risk

§3 で実証した通り anvil-runtime は leaf crate. 他 Rust crate からの
import 0 件、external surface = =bin/anvil-runtime= binary のみ.  crate
削除時の Rust side breakage は workspace =Cargo.toml= の =members= 1
行と build script の binary path 1 行のみ.

逆に anvil-runtime → nelisp-build-tool 方向の依存は強い (= Sexp ABI /
eval / reader を経由). この dep 鎖が Phase A.2.1 compile error の
根本原因. elisp port 完了後はこの dep 鎖が断たれる (= elisp 側は Sexp
を opaque に扱う).

** §6.3 性能 risk (= perf-critical 関数の identification)

elisp dispatch overhead が懸念される代表的 hot path:

| 関数 | 役割 | 推定 latency 影響 |
|------+------+-------------------|
| =anvil_host_registry::call_via_eval= | MCP tool call の毎回 dispatch | 各 call ~1ms (= JSON encode + form 構築) |
| =anvil_host_registry::compat_info= | =anvil-host-info= の per-call response 構築 | 各 call ~5-20ms (= disk_info + shell_capture) |
| =anvil_host_registry::sexp_to_json= | tool 結果 → JSON 変換 | 各 call ~0.5ms |
| =mcp/dispatch::serve_loop= | per-request transport (= 残す前提) | n/a |

elisp port では:
- =call_via_eval= 相当: 元々 elisp eval にディスパッチしていたので、
  Rust side で form 構築する時間が消えるのみ (= 改善側になる可能性)
- =compat_info= 相当: 100% 重複なので Rust 削除で純減
- =sexp_to_json= 相当: =json-encode= の elisp impl は C実装ベースなので
  C-bound、性能差小

**bench gate**: Stage 74.3 完了で =anvil mcp bench --tool anvil-host-info=
を実施、p99 latency が baseline + 5ms 以下に収まることを確認.  超過時は
JIT (Phase 5) 待ち or 該当関数のみ Rust に戻す pivot 判断.

/ anvil-server.el にのみ依存させ、cross-anvil-mcp dep は最小化.
