* §0 TL;DR

** §0.1 The gap, in one sentence

The Phase 47 compiler (=lisp/nelisp-phase47-compiler.el=, ~5,900 LOC)
is a frontend for a *restricted Sexp DSL* — values are raw =i64= in
registers + rbp spill slots, with no heap, no GC integration, no
tagged-value calculus, no closures, no dynamic dispatch.  It was built
for *Phase 47 self-application* (= rewriting nelisp's own Rust
primitive bodies as elisp that emits identical asm), not for compiling
arbitrary elisp.  Compiling =anvil-server.el=-class code requires
turning that "i64 calculator" into a full "Lisp value machine".

** §0.2 What the current grammar covers (= SHIPPED, Doc 97.a/b/c)

Statement forms: =write=, =exit=, =seq=, =let= (single binding,
constant-or-foldable value), =defun=, intra-program =call=,
=static-imm32-table-define=.

Value forms: integer literal, param/let symbol ref, =+= / =-= / =*=
(compile-time fold + runtime emit), =/= shifts + bitwise, comparisons
(=<= =>= =<== =>== ===), =and= / =or= / =if= / =while= / =cond=,
=extern-call= (6 i64 args + i64 ret), direct =syscall=, f64 arithmetic
(Doc 118), Sexp-ABI direct field access (Doc 100), hash-table desugar
(Doc 49 Wave 11).

Architecture: parse (Sexp → IR) → collect strings/tables/defuns →
two-pass emit (fixed-width instructions → address resolution →
relocations) → =nelisp-elf-write-binary=.  SysV AMD64 (+ arm64).

* §1 Mechanism groups (= the actual work, grouped by what they share)

Each group is an independent implementation effort; they are ordered
so earlier groups unblock later ones.

** §1.A Tagged-value calculus (= the foundational shift)

Today an IR value is a raw =i64= in a register.  Real elisp values are
tagged =Sexp= (Int / Float / Cons / Str / Symbol / Vector / closure).
Every value-producing op must box/unbox and type-dispatch.

- New IR value representation: a slot holds a =Sexp= (tagged pointer),
  not a bare integer.  Arithmetic unwraps Int/Float, re-boxes result.
- Reuses Doc 100 Sexp-ABI access ops (=sexp-tag=, =sexp-int-unwrap=,
  =sexp-int-make=, =sexp-float-unwrap=) already in the grammar.
- This is the *largest* single change: it rewrites =--emit-value= /
  =--emit-arith= to thread tagged values, and adds a runtime
  type-error path (signal wrong-type-argument).

*Why first*: every other group manipulates tagged values.

*** §1.A.1 Shipped MVP: boxed-boundary Int functions

The first 129.2 implementation deliberately avoids rewriting the
entire Phase 47 raw-i64 IR.  Instead, =defun-sexp-int= defines a
boxed-boundary function:

#+begin_src elisp
(defun-sexp-int add (out a b) (+ a b))
#+end_src

This lowers to:

#+begin_src elisp
(defun add (out a b)
  (sexp-int-make out
    (+ (sexp-int-unwrap a)
       (sexp-int-unwrap b))))
#+end_src

The result is ABI-useful for runtime/object integration: callers pass
a mutable =Sexp= output slot plus =Sexp::Int= input pointers, and the
compiled code writes the boxed result.  It does not yet provide
general tagged locals, GC roots, type errors, boxed truth values, or
arbitrary user-elisp value semantics.

** §1.B Heap + GC root integration

=cons=, =make-hash-table=, strings, vectors, and closures all
allocate.  Phase 47 has no heap today.

- Allocation: call the existing allocator via =extern-call= (Doc 122.A
  =nl_alloc_str= / =nl_alloc_symbol=, Doc 125 alloc/dealloc grammar).
  No new allocator — delegate.
- *GC roots*: the hard part.  The bytecode VM (=nelisp-bc-run=)
  already registers in-flight stack via =nelisp-gc--active-vms= (Doc
  3c).  AOT-compiled frames must do the equivalent: spill-slot values
  that are live across an allocation must be reachable by the
  collector.  Options: (a) a shadow stack map emitted per call site,
  (b) conservative scan of the spill region, (c) pin live values into
  a frame-local GC-rooted vector.  Recommend (c) for correctness-first
  (matches the VM's active-vm vector), optimise later.

*Why second*: closures / hash / strings in later groups all allocate.

*** §1.B.1 Shipped MVP: static root map for annotated Sexp params

Stage 129.5A adds the compiler-side metadata needed before runtime
root registration can be correct:

#+begin_src elisp
(defun make-str ((slot :type sexp) bytes len)
  (sexp-write-str slot bytes len))
#+end_src

The parameter still uses the GP ABI path, but the =ref= node carries
=:root-p t=.  If the defun body contains a known allocator
(=sexp-write-str=, =sexp-write-symbol=, =cons-make=,
=cons-make-with-clone=, =vector-make=, =record-make=, =cell-make=,
=mut-str-make-empty=, or =alloc-bytes=), the defun IR receives
=:gc-root-slots= with the referenced annotated Sexp slots.

This deliberately stops short of claiming full GC integration: the
compiled prologue does not yet register a root vector, no allocation
site consults a stack map, and runtime =let= locals cannot yet be
typed as Sexp roots.  It is the static map that later 129.5 work will
consume for frame-local GC-rooted vectors.

*** §1.B.2 Shipped MVP: AOT frame roots in the runtime root set

Stage 129.5B adds the runtime-side root category:

#+begin_src elisp
(nelisp-gc--with-active-aot-frame root-vector
  ...)
#+end_src

While the dynamic extent is active, =nelisp-gc-root-set= includes a
=(:kind aot-frame :value ROOT-VECTOR)= entry.  The existing
=nelisp-gc-reachable-set= walker already traverses vectors, so each
slot in =ROOT-VECTOR= becomes reachable through the same mark path as
bytecode VM stack roots.

This is still not full native-frame GC integration.  The missing
bridge is to have Phase 47 / Doc 99 call-boundary code materialize a
root vector from =:gc-root-slots= and push/pop it around compiled
function execution, or emit equivalent extern-call hooks from the
compiled prologue and epilogue.

*** §1.B.3 Shipped MVP: call-boundary root-vector bridge

Stage 129.5C connects the 129.5A static map to the 129.5B runtime
substrate without yet pretending the backend emits native push/pop
code.

Compiler side:

#+begin_src elisp
(nelisp-phase47-compiler--gc-root-descriptor-for-defun defun-ir)
;; => (:name make-str :slots (0) :param-count 3 :rt-slot-count 0)
#+end_src

Runtime side:

#+begin_src elisp
(let ((roots (nelisp-cc-runtime-aot-root-vector-from-slots frame '(0))))
  (nelisp-cc-runtime-call-with-aot-roots roots
    (lambda () ...)))
#+end_src

Inside the dynamic extent, =nelisp-gc-root-set= exposes =roots= as an
=aot-frame= root and the normal reachable-set walker traverses it.
This is the Doc 99 handoff contract for the interpreter/compiled-code

equivalent extern-call hooks in compiled prologue/epilogue, or have
the final Doc 99 dispatcher build/register the frame before jumping

*** §1.B.4 Shipped MVP: runtime let locals as Sexp roots

Stage 129.5D extends the static root map beyond params:

#+begin_src elisp
(defun make-str (raw)
  (let (((slot :type sexp) (id raw)))
    (sexp-write-str slot raw raw)))
#+end_src

The annotated runtime =let= local still lowers through the existing
=let-rt= / =let-rt-n= spill-slot path, but its =FENV= entry carries
=:root-p t=.  When the body contains an allocation and references that
local, the defun's =:gc-root-slots= includes the runtime slot (for the
example above, slot 1: param slot 0 plus one runtime local).

This keeps the 129.5C descriptor format unchanged.  It makes the
descriptor more useful by allowing frame-local Sexp temporaries, not
just annotated incoming params, to be copied into the AOT root vector.

** §1.C General =let= / =let*= + lexical environment

Replace the single-constant-binding =let= with an N-binding lexical
scope: a chain of frame slots (or a heap lexframe — Doc 115
=env_lexframe= shims already exist for the substrate).  Distinguish
lexical from dynamic (=defvar='d special) variables.

*** §1.C.1 Shipped MVP: multi-binding runtime let

Stage 129.4 adds a narrow but useful =let-rt-n= IR for multi-binding
=let= in =defun= bodies.  Runtime initializers are parsed against the
original =ENV/FENV=, so later bindings do not see earlier bindings
(parallel =let=, not =let*=).  Each non-foldable initializer gets a
frame slot and the body parses with all slots visible through the
existing =ref= load path.

Compile-time-foldable bindings still fold into =ENV=.  Mixed
compile-time/runtime multi-binding forms are allowed, but this is
still an MVP: duplicate variables are rejected, special-variable
binding is not modeled, and no heap lexframe / GC-root map exists yet.

** §1.D =setq= / =defvar= / =defconst= / =defcustom= — symbol cells

Writing a global needs symbol value-cell access.  =nl_env_set_value=
extern already exists (used by =nelisp-cc-sf-setq.el= for the
substrate setq swap) — reuse it.  =defvar= / =defcustom= become
=defvar=-then-conditional-set sequences at top level.

*** §1.D.1 Shipped MVP: boxed Int setq delegation

Stage 129.3A adds an explicit ABI surface rather than full user
=setq= syntax:

#+begin_src elisp
(defun-sexp-int-setq set-x (out mirror frames scratch name a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun set-x (out mirror frames scratch name a)
  (seq
   (sexp-int-make out (+ (sexp-int-unwrap a) 16))
   (extern-call nelisp_env_set_value
                mirror frames name out scratch 0)
   out))
#+end_src

The caller still supplies =name= as a =Sexp::Symbol= pointer and a
prebuilt env scratch vector.  That keeps the commit inside existing
grammar/extern coverage and avoids pretending full user =setq= is
available before symbol literal materialisation, =defvar= boundp
checks, and general tagged values land.

*** §1.D.2 Shipped MVP: symbol literal materialisation for boxed setq

Stage 129.3B adds a narrow variant for compile-time-known variable
names:

#+begin_src elisp
(defun-sexp-int-setq-symbol set-x x
    (out mirror frames scratch name-slot a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun set-x (out mirror frames scratch name-slot a)
  (seq
   (sexp-write-symbol-lit name-slot "x")
   (sexp-int-make out (+ (sexp-int-unwrap a) 16))
   (extern-call nelisp_env_set_value
                mirror frames name-slot out scratch 0)
   out))
#+end_src

=sexp-write-symbol-lit= calls the existing =nl_alloc_symbol= helper,
but its UTF-8 bytes are materialized on the stack for the duration of
the call.  That avoids adding =.rodata= to object-mode output, which
is still forbidden by the Doc 99 ET_REL spike path.

This removes the need for the caller to prebuild the symbol object,
but it remains a boxed-boundary helper: the caller still provides
=OUT=, environment pointers, scratch storage, and a writable
=NAME-SLOT=.

*** §1.D.3 Shipped MVP: symbol literal defvar conditional set

Stage 129.3C adds a sibling helper for =defvar=-style conditional
initialization:

#+begin_src elisp
(defun-sexp-int-defvar-symbol ensure-x x
    (out mirror frames scratch name-slot a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun ensure-x (out mirror frames scratch name-slot a)
  (seq
   (sexp-write-symbol-lit name-slot "x")
   (if (= (extern-call nelisp_mirror_is_bound
                       mirror name-slot out)
          1)
       name-slot
     (seq
      (sexp-int-make out (+ (sexp-int-unwrap a) 16))
      (extern-call nelisp_env_set_value
                   mirror frames name-slot out scratch 0)
      name-slot))))
#+end_src

The return value is the materialized symbol slot in both branches,
matching the useful =defvar= result shape for this boxed-boundary
helper.  The =OUT= slot is dual-use: before the bound check, the
caller must initialize it with the runtime unbound-marker Sexp; on the
unbound branch, compiled code overwrites it with the boxed initial
value and delegates the actual value-cell write to the existing env
setter.

This proves the conditional-write substrate without changing the
top-level program ABI.  General user-facing =defvar= forms still need
the Doc 99/runtime entry bridge that supplies env pointers and the
unbound marker automatically.

*** §1.D.4 Shipped MVP: symbol literal defconst value + constant flag

Stage 129.3D adds the =defconst= sibling:

#+begin_src elisp
(defun-sexp-int-defconst-symbol const-x x
    (out mirror frames scratch name-slot a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun const-x (out mirror frames scratch name-slot a)
  (seq
   (sexp-write-symbol-lit name-slot "x")
   (sexp-write-nil out)
   (extern-call nelisp_mirror_set_constant
                mirror name-slot out)
   (sexp-int-make out (+ (sexp-int-unwrap a) 16))
   (extern-call nelisp_env_set_value
                mirror frames name-slot out scratch 0)
   (sexp-write-t out)
   (extern-call nelisp_mirror_set_constant
                mirror name-slot out)
   name-slot))
#+end_src

The first =nelisp_mirror_set_constant= clears an existing constant flag
so the existing =nelisp_env_set_value= constant guard does not reject a
reinitializing =defconst=.  After the value write, the final call marks
the symbol constant again.  =OUT= is reused as the Nil/T flag slot and
as the boxed BODY value slot; =NAME-SLOT= remains the function result.

As with 129.3C, this is still an explicit boxed-boundary helper.
Direct top-level execution needs the Doc 99/runtime entry ABI to
supply env pointers, scratch storage, and result/name slots.

*** §1.D.5 Shipped MVP: top-level defvar/defconst/defcustom AOT init helpers

Stage 129.3E connects ordinary =defvar= / =defconst= top-level syntax
to the helper substrate for object output, Stage 129.3F extends the
same path to =defcustom= initial values, Stage 129.3G exposes the
matching =defcustom= metadata as compiler-side descriptors, and Stage
129.3H adds the runtime scheduling plan shape:

#+begin_src elisp
(seq
  (defvar x 42 "doc")
  (defconst y 7 "doc")
  (defcustom z 9 "doc" :type 'integer))
#+end_src

The frontend lowers that to generated helper defuns:

#+begin_src elisp
(seq
  (defun-sexp-int-defvar-symbol nelisp_aot_var_0_x x
      (out mirror frames scratch name-slot)
    42)
  (defun-sexp-int-defconst-symbol nelisp_aot_const_1_y y
      (out mirror frames scratch name-slot)
    7)
  (defun-sexp-int-defvar-symbol nelisp_aot_custom_2_z z
      (out mirror frames scratch name-slot)
    9))
#+end_src

Those helpers then lower through 129.3C / 129.3D.  The generated
symbols are deterministic and linker-friendly, so =compile-to-object=
emits normal GLOBAL FUNC symbols that the Doc 99 runtime layer can
call during module initialization.  Declaration-only =(defvar x)= is
stripped when other top-level forms remain; no special-variable
metadata is emitted yet.

For =defcustom=, the compiler also exposes a side descriptor:

#+begin_src elisp
(:name z
 :helper nelisp_aot_custom_2_z
 :standard 9
 :docstring "doc"
 :options (:type (quote integer)))
#+end_src

This validates the docstring and requires customization options to be
keyword/value pairs.  The descriptor is the handoff shape for Doc 99
module scheduling; runtime customization-table registration is still
pending.

For all top-level variable initializers, the compiler exposes ordered
init-helper descriptors:

#+begin_src elisp
((:kind defvar
  :name x
  :helper nelisp_aot_var_0_x
  :index 0)
 (:kind defconst
  :name y
  :helper nelisp_aot_const_1_y
  :index 1)
 (:kind defcustom
  :name z
  :helper nelisp_aot_custom_2_z
  :index 2))
#+end_src

=nelisp-cc-runtime-aot-module-init-plan= then normalizes the ordered
init descriptors, =defcustom= metadata descriptors, and 129.5C root
descriptors into one pure Doc 99 module plan:

#+begin_src elisp
(:helper-order (nelisp_aot_var_0_x
                nelisp_aot_const_1_y
                nelisp_aot_custom_2_z)
 :custom-by-helper ((nelisp_aot_custom_2_z . ...))
 :root-descriptors (...))
#+end_src

=nelisp-cc-runtime-run-aot-module-init-plan= executes that plan in
order once Doc 99 supplies the native helper caller and boundary
context:

#+begin_src elisp
(nelisp-cc-runtime-run-aot-module-init-plan
 plan
 '(:out OUT :mirror MIRROR :frames FRAMES
   :scratch SCRATCH :name-slot NAME-SLOT)
 call-helper-fn
 register-custom-fn)
#+end_src

The runtime side validates the context keys and fires
=register-custom-fn= only after the matching =defcustom= helper has
run.  =call-helper-fn= is still the Doc 99 loader boundary: it resolves
the symbol, performs the native call, and owns any platform-specific
FFI details.

When =register-custom-fn= is nil, runtime registration defaults to
=nelisp-cc-runtime-register-aot-custom-metadata=.  Metadata is stored
by variable name in =nelisp-cc-runtime--aot-custom-table= and can be
queried through =nelisp-cc-runtime-aot-custom-metadata= or snapshotted
with =nelisp-cc-runtime-aot-custom-table-snapshot=.  The table stores
only descriptor data plus the init helper kind/index, not the
transient =OUT/MIRROR/FRAMES/SCRATCH/NAME-SLOT= pointers.

=nelisp-cc-runtime-aot-init-helper-caller= is the standard
=call-helper-fn= implementation for Doc 99.  It resolves helper
symbols through =nelisp-cc-runtime-resolve-aot-init-helper=, which
wraps the existing Doc 81 =nelisp-cc-runtime-resolve-symbol= contract:

#+begin_src elisp
(:helper nelisp_aot_var_0_x
 :status :resolved        ;; or :host-stub / :not-found
 :addr #x...
 :descriptor ...)
#+end_src

If Doc 99 supplies a native-call callback, runtime invokes it as
=(NATIVE-CALL RESOLUTION CONTEXT INIT-DESCRIPTOR)=.  On host Emacs,
the default resolver still returns =:host-stub= so ERT can inspect the
contract without dereferencing a native address.  =:not-found= raises
before any native-call callback is invoked.

This still is not full top-level execution inside Phase 47 itself.
The compiled object exposes the initializer entry points and the
runtime now has a validated scheduling/execution hook, metadata store,
and helper resolution/call contract; actual object loading, real
native ABI invocation, scratch/name/out allocation, and real
environment passing remain Doc 99 call-boundary responsibilities.
Initializer values are currently limited to the same boxed Int
expression subset accepted by =defun-sexp-int-*= helpers.

** §1.E Closures + =funcall= / =apply= dispatch

=(lambda (x) (+ x cap))= = heap env record (the captured bindings) +
a code pointer.  =funcall= / =apply= dispatch through the runtime's
existing callable-resolution (=apply_function= / =nelisp--apply=) via
=extern-call= — *delegate to the runtime dispatcher rather than
re-implementing arity/&optional/&rest handling*.  Mirrors the
bytecode VM's MAKE-CLOSURE / CALL opcodes.

*** §1.E.1 Shipped MVP: one-argument =funcall= dispatcher bridge

Stage 129.7A adds the first higher-order call surface without yet
allocating closures.  A boxed-boundary defun may write:

#+begin_src elisp
(defun call-fn
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)

#+end_src

The parser lowers the body to:

#+begin_src elisp

#+end_src

The runtime-side bridge has the same ABI order:

#+begin_src elisp

#+end_src

It dispatches through =nelisp--apply= when available, falls back to
host =funcall= for host symbols/functions, writes the result to =OUT=,
and returns =OUT=.  This fixes the call-boundary contract for an
already-materialized function designator/value.  It does not yet build
heap closures from =lambda= forms, capture lexical environments, support
multi-argument =funcall=, or implement =apply=.

** §1.F Exception machinery

=condition-case= / =catch= / =throw= / =unwind-protect= / =signal=
need non-local exit.  Two designs:
- *Handler stack* (recommended): a thread-local stack of (tag,
  landing-pad, saved-sp) records; =throw= / =signal= walks it, runs
  =unwind-protect= cleanups, restores sp, jumps to the pad.  Mirrors
  the bytecode VM's PUSH-CATCH / PUSH-CC / PUSH-UNWIND / POP-HANDLER
  opcodes (= the VM already implements this shape in elisp).
- setjmp/longjmp via extern — simpler but fights the spill-slot model.

*** §1.F.1 Shipped MVP: runtime handler-stack substrate

Stage 129.8A fixes the runtime-side contract before the compiler emits
handler records:

#+begin_src elisp
(nelisp-cc-runtime-aot-push-catch TAG LANDING-PAD SAVED-SP)
(nelisp-cc-runtime-aot-push-condition CONDITIONS LANDING-PAD SAVED-SP)
(nelisp-cc-runtime-aot-push-unwind CLEANUP LANDING-PAD SAVED-SP)
(nelisp-cc-runtime-aot-throw TAG VALUE)
(nelisp-cc-runtime-aot-signal ERROR-TAG DATA)
#+end_src

The elisp bridge is intentionally a simulator: =throw= / =signal= return
a landing descriptor instead of jumping:

#+begin_src elisp
(:kind catch
 :landing-pad PAD
 :saved-sp SAVED-SP
 :reason throw
 :tag TAG
 :value VALUE
 :cleanups (...))
#+end_src

For =condition= handlers the descriptor carries =:error (TAG . DATA)=.
Crossed =unwind= handlers run from inner to outer and their return values
are recorded in =:cleanups=.  Unmatched =throw= / =signal= still run
crossed cleanups, unwind the stack, and signal
=nelisp-cc-runtime-error=.

Compiler lowering for =(catch ...)= / =(condition-case ...)= /
=(unwind-protect ...)=, native stack-pointer restoration, and actual
landing-pad jumps remain later 129.8 work.

*** §1.F.5 Shipped MVP: source =catch= normal-exit lowering

Stage 129.8E starts synthesizing source-level handler control flow.
For a boxed-boundary function:

#+begin_src elisp
(defun catch-value
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (catch 'done value))
#+end_src

the compiler lowers the normal path as if it had written:

#+begin_src elisp
(seq
 (aot-push-catch 'done 0 0)
 (let (((aot-catch-value-N :type sexp) value))
   (seq
    (aot-pop-handler 'catch)
    aot-catch-value-N)))
#+end_src

The temporary is a runtime Sexp local so the BODY result survives the
pop bridge call.  The placeholder =0= landing pad / saved stack pointer
keeps this stage honest: it proves balanced source-form lowering and
object relocations, but it is not the final non-local exit path.

Consequently the compiler rejects =catch= bodies that explicitly
contain =throw=, =signal=, =condition-case=, or =unwind-protect=.  Those
forms still require native landing-pad labels, stack-pointer restore,
and branch-to-pad emission.  =condition-case= and =unwind-protect=
source lowering remain later 129.8 work.

*** §1.F.6 Shipped MVP: source =condition-case= normal-exit lowering

#+begin_src elisp
(defun cc-value

     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (condition-case err
      value
    (error out)))
#+end_src

lowers as if it had written:

#+begin_src elisp
(seq

native jump-to-handler emission remain later 129.8 work.

*** §1.F.7 Shipped MVP: source =unwind-protect= normal-exit lowering

Stage 129.8G covers the normal path of =unwind-protect=:

#+begin_src elisp
(defun unwind-value
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (unwind-protect
      value
    (identity value)))
#+end_src

lowers as if it had written:

#+begin_src elisp
(let (((aot-unwind-value-N :type sexp) value))
  (seq
   (identity value)
   aot-unwind-value-N))
#+end_src

This is normal-exit semantics only: cleanup forms run and the protected
BODY value remains the result.  Non-local exits crossing the protected
body still require real cleanup landing pads, native stack-pointer
restore, and branch-to-cleanup / branch-back emission.

*** §1.F.8 Shipped MVP: source =error= bridge lowering

Stage 129.8H adds the common single-argument =error= surface:

#+begin_src elisp
(defun error-value
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (error value))
#+end_src

This lowers through the existing signal bridge:

#+begin_src elisp
(seq
 (sexp-write-symbol-lit name_slot "error")
 (extern-call nelisp_aot_signal
              mirror frames name_slot value out scratch)
 out)
#+end_src

The argument is treated as already-boxed signal data.  The usual
formatted =error= shape, e.g. =(error "%s" value)=, still needs calln
and rest/list construction before it can be represented faithfully.

** §1.G macroexpand frontend (= cheapest group)

=defmacro= / =when= / =unless= / =dolist= / =cl-*= / =pcase= reduce to
core forms via compile-time macro expansion.  The Wave A21 bytecode
path already does broad =macroexpand-all= before =nelisp-bc-compile=.
For Phase 47, the shipped MVP is deliberately narrower: selectively
macroexpand value/call positions while preserving DSL structural forms
such as =defun=, because host =macroexpand-all= rewrites them to
=defalias= forms outside the Phase 47 grammar.  Multi-form macro
bodies lower through =value-seq= so the final child value is preserved
in value context.  Near-zero new asm.

*** §1.G.1 Shipped MVP: top-level module forms

Stage 129.6A extends the frontend with top-level =require= /
=provide= handling:

#+begin_src elisp
(seq
  (require 'cl-lib)
  (defmacro ...)
  (provide 'my-feature)
  ...)
#+end_src

=require= is evaluated during compilation, before top-level
=defmacro= bodies are installed, and then stripped from the Phase 47
IR.  Only literal arguments are accepted: =FEATURE= must resolve to a
symbol, =FILENAME= to nil or a string, and =NOERROR= follows host
=require= semantics.  =provide= is also stripped after validating its
feature symbol, but it does not yet emit runtime module registration.

This is deliberately a frontend/module-load MVP, not general builtin
delegation.  It lets application files with ordinary top-level module
boilerplate pass through the AOT parser.  Stage 129.6B below adds the
first explicit dispatcher helper, but ordinary runtime call lowering
for =format=, =mapcar=, =funcall=, and other real calls remains later
129.6 work.

** §1.H Builtin library delegation

=format= / =concat= / =substring= / =string-match= / =mapcar= /
=nth= / =assq= / ... = hundreds of builtins.  Pragmatic route: emit a
=call= into the runtime's existing implementation via =extern-call=
(the dispatcher already resolves builtins).  Consequence: compiled
output links against the nelisp runtime (= "compiled code + runtime"),
*not* a tiny standalone ELF.  This is acceptable for the
interpreter-embedded use case (Doc 99 Arch B/C) but is in tension with
Doc 97's standalone-binary lineage — call out explicitly per output
target.

*** §1.H.1 Shipped MVP: one-argument Sexp builtin delegation helper

Stage 129.6B adds an explicit boxed-boundary helper form:

#+begin_src elisp
(defun-sexp-builtin1-symbol call-symbol-name symbol-name
    (out mirror frames scratch name-slot arg))
#+end_src

It lowers to ordinary Phase 47 DSL:

#+begin_src elisp
(defun call-symbol-name
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name-slot :type sexp) (arg :type sexp))
  (seq
   (sexp-write-symbol-lit name-slot "symbol-name")
   (extern-call nelisp_aot_builtin_call1
                mirror frames name-slot arg out scratch)
   out))
#+end_src

The caller supplies =OUT/MIRROR/FRAMES/SCRATCH/NAME-SLOT= plus the
boxed Sexp =ARG=.  The dispatcher ABI writes the builtin result into
=OUT= and returns =OUT=, matching the boxed-boundary convention used by
the 129.2/129.3 helpers.  Object output exposes the compiled helper
symbol and unresolved relocations for =nelisp_aot_builtin_call1= and
=nl_alloc_symbol=.

This is intentionally not ordinary user syntax yet: =(symbol-name x)=
does not lower automatically, no varargs ABI exists, and heavier calls
such as =format=, =mapcar=, =apply=, and =funcall= are still pending.
The runtime C/Rust dispatcher body behind =nelisp_aot_builtin_call1=
also remains Doc 99 / later-129.6 integration work.

*** §1.H.2 Shipped MVP: runtime builtin-call1 bridge contract

Stage 129.6C adds the runtime-side analogue of the native dispatcher
ABI:

#+begin_src elisp
(nelisp-cc-runtime-aot-builtin-call1
 mirror frames 'symbol-name arg out scratch)
#+end_src

The argument order mirrors the native call emitted by 129.6B:

#+begin_src c
nelisp_aot_builtin_call1(mirror, frames, name, arg, out, scratch)
#+end_src

In host-side tests =OUT= is a caller-owned vector with at least one
slot.  The bridge resolves and dispatches the one-argument builtin
(or uses an injected dispatcher callback), writes the result to
=(aref OUT 0)=, and returns =OUT=.  This pins the Doc 99 calling
contract without pretending the final exported C ABI is already wired:
native code still needs a real =nelisp_aot_builtin_call1= symbol that
can recover/use the runtime =Env= behind =MIRROR/FRAMES=.

*** §1.H.3 Shipped MVP: constrained direct =symbol-name= call lowering

Stage 129.6D removes the custom surface form for the first user-call
case.  A normal defun with explicit boxed-boundary parameters may now
write:

#+begin_src elisp
(defun call-symbol-name
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name-slot :type sexp) (arg :type sexp))
  (symbol-name arg))
#+end_src

The parser lowers =(symbol-name arg)= to the same sequence used by
129.6B:

#+begin_src elisp
(seq
 (sexp-write-symbol-lit name-slot "symbol-name")
 (extern-call nelisp_aot_builtin_call1
              mirror frames name-slot arg out scratch)
 out)
#+end_src

This is intentionally constrained.  The surrounding defun must expose
the boundary symbols by name, the only shipped direct builtin is
=symbol-name=, and the call remains one-argument only.  It proves the
shape needed for ordinary user-call lowering without yet claiming
general =format= / =mapcar= / =funcall= coverage.

*** §1.H.4 Shipped MVP: unary builtin delegation table

Stage 129.6E widens the 129.6D direct-call path from a single builtin
to an explicit one-argument table:

#+begin_src elisp
identity ignore not null atom consp listp symbolp keywordp
numberp integerp float stringp vectorp
car cdr car-safe cdr-safe
length reverse nreverse copy-sequence
1+ 1- abs
symbol-name prin1-to-string number-to-string string-to-number
upcase downcase
#+end_src

Each table entry lowers to the same =nelisp_aot_builtin_call1=
sequence as =symbol-name=.  The parser checks the current defun table
first: if the compile unit defines a same-named Phase 47 function,
ordinary direct-call lowering to that function wins and builtin
delegation is not used.

This still stops at unary calls.  Multi-argument and rest-argument
builtins need a different ABI shape, likely an argument-vector or
list-backed =nelisp_aot_builtin_calln= helper rather than more fixed
GP arguments, because the current explicit boundary already consumes
the six-register call budget.

* §3 Honest scope assessment

§1.A + §1.B + §1.E + §1.F together constitute a *complete native-code
language runtime*.  "Compile =anvil-server.el= to native" is, in
honest terms, "build a native-comp backend for elisp" — the same
scope as GNU Emacs's libgccjit native-comp, except hand-emitted via
the Phase 47 asm chain instead of delegating to GCC.  Realistic
horizon: multi-month to multi-year, paced one group per engagement.

This is *not* a near-term cold-load fix.  It is the structural endgame
that aligns with Doc 99 Arch C ("every byte of code derives from elisp
the Phase 47 chain compiled").

* §4 Relationship to the other axes

- *Doc 99 (call boundary)*: gives the interpreter an ABI to call
  compiled functions.  Doc 129 gives those functions a language.  A
  single compiled =anvil-server= function needs *both* (Doc 99 ET_REL
  + ABI, Doc 129 grammar coverage).
- *Doc 122 (substrate primitives)*: supplies the alloc / mut-str /
  extended-extern-call primitives that Doc 129 groups call.  Doc 129
  is "Doc 122 consumed at application level".
- *Doc 126 (eval-boot)*: precedent — replacing "interpret source" with
  a faster path.  Doc 129 is the user-code analogue of that move.

* §5 The pragmatic alternative (= why this is endgame, not now)

The Wave A21 bytecode VM (=byte-compile-file= → =.elc= → =nelisp-bc-run=)
*already* handles closures, exceptions, hash, and higher-order calls —
because it runs on the interpreter's runtime, not bare registers.  Its
only blocker for standalone use was a heap-corruption bug in the
reader (§6), now fixed.  Bytecode gives a 5–10x evaluator speedup with
*zero* new grammar.  Phase 47 AOT (this doc) gives native speed but
needs the full §1 stack.

Recommended sequencing: harvest the bytecode-VM win first (cheap,
grammar-free), pursue Doc 129 as the long-horizon native-comp endgame.

* §6 Prerequisite SHIPPED — reader heap-corruption fix (2026-05-25)

Loading a Wave A21 =.elc= with a large pre-compiled =defun= (e.g.
=anvil-server--handle-tools-call=, a 1,099-element bytecode CODE
vector) aborted standalone NeLisp with glibc =corrupted size vs.
prev_size= / =double free=.

Root cause: =build-tool/src/lib.rs= reader =PARSER_POOL_SIZE = 3 + 4 *
1024= (depth ≈1023).  The Phase 47 elisp parser
(=nelisp-cc-reader-parser.el=) recurses depth+1 per list/vector
*element*, indexing the slot-pool via *unchecked* =vector-ref-ptr=, so
any list/vector literal with ≳1024 elements overran the pool → heap
corruption (threshold confirmed: 1000-element OK, 1024 crash).

Fix (1 line, net 0 Rust LOC): size the pool to the input —
=vec![Sexp::Nil; (3 + 4 * input.len()).max(PARSER_POOL_SIZE)]=.  An
element count can never exceed the input byte count, so pool overflow
is structurally impossible.  Verified: 1,099 + 5,000-element vector
literals and the full =anvil-server.elc= (68 bcl defuns) load cleanly;
=mini.elc= regression + cargo =--lib= 10/10 green.

Proper follow-up (not blocking): the parser's depth-per-element
recursion is itself the latent flaw; an *iterative* list/vector parse
(reusing a fixed slot set, accumulating via a running tail) would
remove the input-sized-pool memory cost entirely.  Tracked as a Phase
47 reader-parser cleanup, independent of Doc 129.
