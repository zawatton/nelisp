- *129.2 MVP*: =defun-sexp-int= lowers a monomorphic
  =(defun-sexp-int NAME (OUT ARG...) BODY)= surface into ordinary
  Phase 47 DSL: =ARG= values are assumed =Sexp::Int= pointers and
  are read via =sexp-int-unwrap=; the raw integer result is written to
  caller-owned =OUT= via =sexp-int-make=.  This is intentionally a
  boxed-boundary step, not the full compiler-wide "every IR value is
  Sexp" rewrite.
- *129.3A MVP*: =defun-sexp-int-setq= extends the boxed Int boundary
  with explicit environment pointers:
  =(defun-sexp-int-setq NAME (OUT MIRROR FRAMES SCRATCH SYM ARG...) BODY)=
  boxes BODY into =OUT=, calls =nelisp_env_set_value= with
  =MIRROR/FRAMES/SYM/OUT/SCRATCH=, then returns =OUT=.  This proves
  the AOT path can delegate a symbol-cell write without adding Rust
  LOC.  It is not yet general =setq= syntax, =defvar= conditional set
  semantics, or tagged-value locals; 129.3B adds the literal-symbol
  variant below.
- *129.3B MVP*: =defun-sexp-int-setq-symbol= covers the next
  symbol-cell step: the caller supplies a writable =SYM-SLOT=, while
  the compiled function materializes a compile-time symbol literal via
  =sexp-write-symbol-lit= before calling =nelisp_env_set_value=.  The
  literal bytes are pushed as a temporary stack buffer rather than
  stored in =.rodata=, so the helper remains valid in ET_REL object
  mode.  This is still not full user =setq= syntax or top-level
  =defvar= / =defconst= semantics.
- *129.3C MVP*: =defun-sexp-int-defvar-symbol= adds a boxed-boundary
  conditional symbol-cell write for a compile-time variable name.  It
  materializes the symbol, asks =nelisp_mirror_is_bound= whether the
  value cell is already bound, and only on the unbound branch boxes
  BODY into =OUT= and delegates to =nelisp_env_set_value=.  To stay
  inside the current six-GP-argument Phase 47 ABI, =OUT= must initially
  contain the runtime unbound marker and is reused as the
  =nelisp_mirror_is_bound= marker argument before the unbound branch
  overwrites it with the boxed initial value.  This proves
  =defvar=-style conditional set, but it is still an explicit helper,
  not direct top-level execution.
- *129.3D MVP*: =defun-sexp-int-defconst-symbol= adds the matching
  boxed-boundary constant initializer.  It materializes the symbol,
  clears any existing constant flag via =nelisp_mirror_set_constant=
  with =Nil= in =OUT=, boxes BODY and delegates the value write through
  =nelisp_env_set_value=, then writes =T= into =OUT= and marks the
  symbol constant.  This proves the constant-flag substrate for
  compile-time-known names, but it is still an explicit helper rather
  than direct top-level execution.

* §0 TL;DR

** §0.1 The gap, in one sentence

The Phase 47 compiler (=lisp/nelisp-phase47-compiler.el=, ~5,900 LOC)
is a frontend for a *restricted Sexp DSL* — values are raw =i64= in
registers + rbp spill slots, with no heap, no GC integration, no
tagged-value calculus, no closures, no dynamic dispatch.  It was built
for *Phase 47 self-application* (= rewriting nelisp's own Rust
primitive bodies as elisp that emits identical asm), not for compiling
arbitrary elisp.  Compiling =anvil-server.el=-class code requires
turning that "i64 calculator" into a full "Lisp value machine".

** §0.2 What the current grammar covers (= SHIPPED, Doc 97.a/b/c)

Statement forms: =write=, =exit=, =seq=, =let= (single binding,
constant-or-foldable value), =defun=, intra-program =call=,
=static-imm32-table-define=.

Value forms: integer literal, param/let symbol ref, =+= / =-= / =*=
(compile-time fold + runtime emit), =/= shifts + bitwise, comparisons
(=<= =>= =<== =>== ===), =and= / =or= / =if= / =while= / =cond=,
=extern-call= (6 register i64 args plus SysV stack-GP call-spill MVP + i64 ret), direct =syscall=, f64 arithmetic
(Doc 118), Sexp-ABI direct field access (Doc 100), hash-table desugar
(Doc 49 Wave 11).

Architecture: parse (Sexp → IR) → collect strings/tables/defuns →
two-pass emit (fixed-width instructions → address resolution →
relocations) → =nelisp-elf-write-binary=.  SysV AMD64 (+ arm64).

* §1 Mechanism groups (= the actual work, grouped by what they share)

Each group is an independent implementation effort; they are ordered
so earlier groups unblock later ones.

** §1.A Tagged-value calculus (= the foundational shift)

Today an IR value is a raw =i64= in a register.  Real elisp values are
tagged =Sexp= (Int / Float / Cons / Str / Symbol / Vector / closure).
Every value-producing op must box/unbox and type-dispatch.

- New IR value representation: a slot holds a =Sexp= (tagged pointer),
  not a bare integer.  Arithmetic unwraps Int/Float, re-boxes result.
- Reuses Doc 100 Sexp-ABI access ops (=sexp-tag=, =sexp-int-unwrap=,
  =sexp-int-make=, =sexp-float-unwrap=) already in the grammar.
- This is the *largest* single change: it rewrites =--emit-value= /
  =--emit-arith= to thread tagged values, and adds a runtime
  type-error path (signal wrong-type-argument).

*Why first*: every other group manipulates tagged values.

*** §1.A.1 Shipped MVP: boxed-boundary Int functions

The first 129.2 implementation deliberately avoids rewriting the
entire Phase 47 raw-i64 IR.  Instead, =defun-sexp-int= defines a
boxed-boundary function:

#+begin_src elisp
(defun-sexp-int add (out a b) (+ a b))
#+end_src

This lowers to:

#+begin_src elisp
(defun add (out a b)
  (sexp-int-make out
    (+ (sexp-int-unwrap a)
       (sexp-int-unwrap b))))
#+end_src

The result is ABI-useful for runtime/object integration: callers pass
a mutable =Sexp= output slot plus =Sexp::Int= input pointers, and the
compiled code writes the boxed result.  It does not yet provide
general tagged locals, GC roots, type errors, boxed truth values, or
arbitrary user-elisp value semantics.

** §1.B Heap + GC root integration

=cons=, =make-hash-table=, strings, vectors, and closures all
allocate.  Phase 47 has no heap today.

- Allocation: call the existing allocator via =extern-call= (Doc 122.A
  =nl_alloc_str= / =nl_alloc_symbol=, Doc 125 alloc/dealloc grammar).
  No new allocator — delegate.
- *GC roots*: the hard part.  The bytecode VM (=nelisp-bc-run=)
  already registers in-flight stack via =nelisp-gc--active-vms= (Doc
  3c).  AOT-compiled frames must do the equivalent: spill-slot values
  that are live across an allocation must be reachable by the
  collector.  Options: (a) a shadow stack map emitted per call site,
  (b) conservative scan of the spill region, (c) pin live values into
  a frame-local GC-rooted vector.  Recommend (c) for correctness-first
  (matches the VM's active-vm vector), optimise later.

*Why second*: closures / hash / strings in later groups all allocate.

*** §1.B.1 Shipped MVP: static root map for annotated Sexp params

Stage 129.5A adds the compiler-side metadata needed before runtime
root registration can be correct:

#+begin_src elisp
(defun make-str ((slot :type sexp) bytes len)
  (sexp-write-str slot bytes len))
#+end_src

The parameter still uses the GP ABI path, but the =ref= node carries
=:root-p t=.  If the defun body contains a known allocator
(=sexp-write-str=, =sexp-write-symbol=, =cons-make=,
=cons-make-with-clone=, =vector-make=, =record-make=, =cell-make=,
=mut-str-make-empty=, or =alloc-bytes=), the defun IR receives
=:gc-root-slots= with the referenced annotated Sexp slots.

This deliberately stops short of claiming full GC integration: the
compiled prologue does not yet register a root vector, no allocation
site consults a stack map, and runtime =let= locals cannot yet be
typed as Sexp roots.  It is the static map that later 129.5 work will
consume for frame-local GC-rooted vectors.

*** §1.B.2 Shipped MVP: AOT frame roots in the runtime root set

Stage 129.5B adds the runtime-side root category:

#+begin_src elisp
(nelisp-gc--with-active-aot-frame root-vector
  ...)
#+end_src

While the dynamic extent is active, =nelisp-gc-root-set= includes a
=(:kind aot-frame :value ROOT-VECTOR)= entry.  The existing
=nelisp-gc-reachable-set= walker already traverses vectors, so each
slot in =ROOT-VECTOR= becomes reachable through the same mark path as
bytecode VM stack roots.

This is still not full native-frame GC integration.  Stage 129.5E
adds explicit native push/pop hooks; the remaining bridge is to have
Phase 47 / Doc 99 call-boundary code materialize a root vector from
=:gc-root-slots= and insert those hooks around compiled function
execution automatically.

*** §1.B.3 Shipped MVP: call-boundary root-vector bridge

Stage 129.5C connects the 129.5A static map to the 129.5B runtime
substrate without yet pretending the backend emits native push/pop
code.

Compiler side:

#+begin_src elisp
(nelisp-phase47-compiler--gc-root-descriptor-for-defun defun-ir)
;; => (:name make-str :slots (0) :param-count 3 :rt-slot-count 0)
#+end_src

Runtime side:

#+begin_src elisp
(let ((roots (nelisp-cc-runtime-aot-root-vector-from-slots frame '(0))))
  (nelisp-cc-runtime-call-with-aot-roots roots
    (lambda () ...)))
#+end_src

Inside the dynamic extent, =nelisp-gc-root-set= exposes =roots= as an
=aot-frame= root and the normal reachable-set walker traverses it.
This is the Doc 99 handoff contract for the interpreter/compiled-code
boundary.  Stage 129.5E gives the backend explicit extern-call hooks
to use for the prologue/epilogue side.

*** §1.B.4 Shipped MVP: runtime let locals as Sexp roots

Stage 129.5D extends the static root map beyond params:

#+begin_src elisp
(defun make-str (raw)
  (let (((slot :type sexp) (id raw)))
    (sexp-write-str slot raw raw)))
#+end_src

The annotated runtime =let= local still lowers through the existing
=let-rt= / =let-rt-n= spill-slot path, but its =FENV= entry carries
=:root-p t=.  When the body contains an allocation and references that
local, the defun's =:gc-root-slots= includes the runtime slot (for the
example above, slot 1: param slot 0 plus one runtime local).

This keeps the 129.5C descriptor format unchanged.  It makes the
descriptor more useful by allowing frame-local Sexp temporaries, not
just annotated incoming params, to be copied into the AOT root vector.

*** §1.B.5 Shipped MVP: explicit native root push/pop bridges

Stage 129.5E exposes the root-frame operation as ordinary native
relocs:

#+begin_src elisp
(aot-push-roots roots)
...
(aot-pop-roots roots)
#+end_src

The compiler lowers these internal forms to:

#+begin_src elisp
(extern-call nelisp_aot_push_roots mirror frames roots out scratch)
(extern-call nelisp_aot_pop_roots mirror frames roots out scratch)
#+end_src

Runtime side, =nelisp-cc-runtime-aot-push-roots-boundary= validates
that =ROOTS= is a vector, pushes it onto
=nelisp-gc--active-aot-frames=, writes the pushed vector to =OUT[0]=,
and returns =OUT=.  =nelisp-cc-runtime-aot-pop-roots-boundary= pops
the innermost AOT root vector, verifies the expected vector when
supplied, writes the popped vector to =OUT[0]=, and returns =OUT=.

This is a native ABI surface, not yet automatic root management for
every allocating defun.  Stages 129.5F/G connect it to boundary defun
emission and materialize root vectors from =:gc-root-slots= for that
surface.

*** §1.B.6 Shipped MVP: automatic root scope for boundary defuns

Stage 129.5F connects the explicit bridge to the ordinary defun emit
path when the function already has the Doc 99 boundary locals:

#+begin_src elisp
(defun make-str
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (roots :type sexp) bytes len)
  (sexp-write-str out bytes len))
#+end_src

The parser keeps the static descriptor =(:gc-root-slots (0))=, then
wraps the body in an internal =aot-root-scope=.  The x86_64 SysV
emitter calls =nelisp_aot_push_roots= before the allocating body and
=nelisp_aot_pop_roots= afterwards; it saves the body's GP result
across the pop call so the compiled defun's return value is unchanged.

This is the first automatic prologue/epilogue insertion step.  Stage
129.5G below removes the need for the caller to prebuild the =roots=
vector.

*** §1.B.7 Shipped MVP: root-vector materialisation bridge

Stage 129.5G adds the missing vector construction step:

#+begin_src elisp
(extern-call nelisp_aot_materialize_roots
             mirror frames 1 out scratch out)
#+end_src

For an =aot-root-scope= whose =:gc-root-slots= are =(0)=, the compiler
maps slot 0 back to its frame symbol, passes that value to the bridge,
and stores the returned root vector in the =roots= frame slot before
calling =nelisp_aot_push_roots=.  Runtime-side,
=nelisp-cc-runtime-aot-materialize-roots-boundary= validates the count,
builds a fresh vector from the root arguments, writes it to =OUT[0]=,
and returns it.

This is still a boundary-defun MVP: the defun must expose the Doc 99
locals =out=, =mirror=, =frames=, =scratch=, and =roots=.  Within that
surface, root vectors now come from =:gc-root-slots= instead of a
caller-provided prebuilt vector.

*** §1.B.8 Shipped MVP: hash-FRAMES root-vector materialisation bridge

Stage 129.5H adds the non-boundary-friendly sibling:

#+begin_src elisp
(extern-call nelisp_aot_materialize_frame_roots
             mirror frames 2 out scratch 0 'cap)
#+end_src

Runtime-side, =nelisp-cc-runtime-aot-materialize-frame-roots-boundary=
validates that =FRAMES= is the shared hash-table frame handle, looks up
each integer or symbol root slot, builds a fresh root vector, writes it
to =OUT[0]=, and returns it.  Missing slots signal instead of silently
rooting =nil= so standalone loader integration can catch stale root
maps early.

This does not yet make every non-boundary defun self-rooting.  It
removes the earlier requirement that every live root value be passed as
a GP argument to =nelisp_aot_materialize_roots=, giving the backend and
Doc 99 loader a concrete root materialisation ABI for ordinary native
frame handles.

*** §1.B.9 Shipped MVP: stack-held frame-root auto scope

Stage 129.5I lets Phase 47 emit an automatic root scope without a
caller-owned =roots= local.  A defun that exposes the usual AOT handles
and opts into frame-root materialisation with a =frame_roots= marker:

#+begin_src elisp
(defun make-str
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) frame_roots bytes len)
  (sexp-write-str out bytes len))
#+end_src

now wraps the allocating body in =aot-root-scope= with
=:root-storage stack= and =:materialize-kind frames=.  The x86_64 SysV
emitter calls =nelisp_aot_materialize_frame_roots=, keeps the returned
root vector in an aligned stack temporary across the body, and passes
that temporary to the existing =nelisp_aot_push_roots= /
=nelisp_aot_pop_roots= prologue/epilogue calls.  This removes the
previous requirement that the source signature carry a mutable =roots=
slot while preserving the older boundary-defun path for callers that do
provide one.

This remains a source-visible opt-in marker.  Stage 129.5J adds a
separate marker for loader-selected non-boundary rooting so generated
native objects can request the same ABI without reusing the
hand-written =frame_roots= spelling.

*** §1.B.10 Shipped MVP: loader-selected hash-FRAMES root-scope marker

Stage 129.5J gives the loader/native-object handoff its own selection
marker for the stack-held hash-FRAMES root scope:

#+begin_src elisp
(defun make-str
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) auto_frame_roots bytes len)
  (sexp-write-str out bytes len))
#+end_src

If the body allocates and the static root map is non-empty, the parser
now wraps the body in =aot-root-scope= with =:root-storage stack= and
=:materialize-kind frames= when =auto_frame_roots= is present.  Object
output therefore emits
=nelisp_aot_materialize_frame_roots= and balanced root push/pop relocs
for non-boundary defuns that already carry =out=, =mirror=, =frames=,
and =scratch= handles.  Defuns without =frame_roots= or
=auto_frame_roots= are still left unwrapped so ordinary AOT boundary
helpers do not accidentally grow a root-stack runtime dependency.

*** §1.B.11 Shipped MVP: object loader root metadata selection

Stage 129.5K connects the marker to object emission.  When the loader
or compiler driver calls:

#+begin_src elisp
(nelisp-phase47-compile-to-object source path :auto-frame-roots t)
#+end_src

Phase 47 first parses =source= to collect the Doc 129.5 root
descriptors, then rewrites only eligible defuns by inserting
=auto_frame_roots= before the final parse/emission pass.  Eligibility
requires:

- a non-empty root descriptor for the defun,
- the standard AOT handle params =out=, =mirror=, =frames=, and
  =scratch=,
- no caller-owned =roots= slot, and
- no existing =frame_roots= or =auto_frame_roots= marker.

This gives the loader/module metadata path a concrete switch for
non-boundary root prologue/epilogue selection while leaving ordinary
object output byte-compatible unless the new option is supplied.

** §1.C General =let= / =let*= + lexical environment

Replace the single-constant-binding =let= with an N-binding lexical
scope: a chain of frame slots (or a heap lexframe — Doc 115
=env_lexframe= shims already exist for the substrate).  Distinguish
lexical from dynamic (=defvar='d special) variables.

*** §1.C.1 Shipped MVP: multi-binding runtime let

Stage 129.4 adds a narrow but useful =let-rt-n= IR for multi-binding
=let= in =defun= bodies.  Runtime initializers are parsed against the
original =ENV/FENV=, so later bindings do not see earlier bindings
(parallel =let=, not =let*=).  Each non-foldable initializer gets a
frame slot and the body parses with all slots visible through the
existing =ref= load path.

Compile-time-foldable bindings still fold into =ENV=.  Mixed
compile-time/runtime multi-binding forms are allowed, but this is
still an MVP: duplicate variables are rejected, special-variable
dynamic binding is not modeled, and no heap lexframe / GC-root map
exists yet.

*** §1.C.2 Shipped MVP: special declaration guard

Stage 129.4B records top-level =defvar= / =defconst= / =defcustom=
names while extracting compile-time-only declarations.  During the
same compile unit, binding one of those names with =let= now signals
=:let-special-binding-pending= instead of silently lowering it to a
lexical compile-time binding or runtime frame slot.

This is intentionally a guard, not full dynamic binding.  Real special
=let= still needs value-cell save/restore, unwind cleanup, and
integration with the 129.8 non-local-exit path.

*** §1.C.3 Shipped MVP: special binding bridge substrate

Stage 129.4C adds the runtime-side bridge shape for dynamic special
binding:

#+begin_src elisp
(nelisp_aot_push_special mirror frames name value out scratch)
(nelisp_aot_pop_special  mirror frames handle out scratch)
#+end_src

The Emacs-side runtime simulator saves the previous value-cell state,
binds the symbol to the new value, returns a restore handle in =OUT=,
and restores the prior bound/unbound state on pop.  Phase 47 also
accepts explicit internal forms:

#+begin_src elisp
(aot-push-special 'dyn value)
(aot-pop-special handle)
#+end_src

Those forms lower to relocations against =nelisp_aot_push_special= and
=nelisp_aot_pop_special=, materializing quoted symbol names through
=name-slot=.  This stage only fixes the ABI and restore substrate;
source-level normal-exit lowering starts in 129.4D below.

*** §1.C.4 Shipped MVP: source special =let= normal exit

Stage 129.4D connects single-binding source =let= forms whose variable
is a tracked top-level special declaration:

#+begin_src elisp
(seq
 (defvar dyn)
 (defun f (out mirror frames scratch name-slot value)
   (let ((dyn value))
     value)))
#+end_src

The frontend lowers the body to the existing normal-exit bridge shape:

#+begin_src elisp
(seq
 (aot-push-special 'dyn value)
 (let (((aot-special-value-N :type sexp) BODY))
   (seq
    (aot-pop-special 0)
    aot-special-value-N)))
#+end_src

The =0= pop handle means "pop the innermost special binding" and is
used because the source-lowered normal path only needs stack discipline,
not an externally held handle.  The runtime still supports explicit
handle validation for lower-level =aot-pop-special= forms.

This normal-exit shape is extended by 129.4G for direct quoted-tag
=throw= bodies.  Other nested non-local bodies containing =signal=,
=error=, =condition-case=, or =unwind-protect= are rejected until the
129.8 landing-pad path can guarantee cleanup.  Multi-binding special
=let= starts in 129.4E below.

*** §1.C.5 Shipped MVP: all-special multi-binding =let= normal exit

Stage 129.4E extends the source special =let= lowering to forms whose
bindings are all tracked special declarations:

#+begin_src elisp
(seq
 (defvar dyn-a)
 (defvar dyn-b)
 (defun f (out mirror frames scratch name-slot value-a value-b)
   (let ((dyn-a value-a)
         (dyn-b value-b))
     value-b)))
#+end_src

To preserve ordinary parallel =let= initializer semantics, the
frontend first evaluates every initializer against the original
=ENV/FENV= into fresh Sexp temp slots, then pushes the special
bindings in source order, saves the BODY result, and pops the
innermost bindings:

#+begin_src elisp
(let (((aot-special-init-0 :type sexp) value-a)
      ((aot-special-init-1 :type sexp) value-b))
  (seq
   (aot-push-special 'dyn-a aot-special-init-0)
   (aot-push-special 'dyn-b aot-special-init-1)
   (let (((aot-special-value-N :type sexp) BODY))
     (seq
      (aot-pop-special 0)
      (aot-pop-special 0)
      aot-special-value-N))))
#+end_src

*** §1.C.6 Shipped MVP: mixed lexical/special multi-binding =let= normal exit

Stage 129.4F extends the same temp-first strategy to mixed forms:

#+begin_src elisp
(seq
 (defvar dyn)
 (defun f (out mirror frames scratch name-slot value-a value-b)
   (let (((a :type sexp) value-a)
         (dyn value-b))
     a)))
#+end_src

The lowering evaluates every initializer into a fresh temp slot before
any source binding becomes visible.  It then pushes the special
bindings, creates lexical aliases from their temp values, evaluates
BODY into a saved Sexp slot, pops the innermost special bindings, and
returns the saved BODY value:

#+begin_src elisp
(let (((aot-let-init-0 :type sexp) value-a)
      ((aot-let-init-1 :type sexp) value-b))
  (seq
   (aot-push-special 'dyn aot-let-init-1)
   (let (((a :type sexp) aot-let-init-0))
     (let (((aot-special-value-N :type sexp) BODY))
       (seq
        (aot-pop-special 0)
        aot-special-value-N)))))
#+end_src

This keeps ordinary parallel =let= initializer semantics: lexical
initializers and special initializers both see the original ENV/FENV,
not each other's new bindings.

*** §1.C.7 Shipped MVP: source special =let= direct =throw= cleanup

Stage 129.4G covers the first non-local cleanup path for source special
=let=:

#+begin_src elisp
(let ((dyn value))
  (throw 'tag result))
#+end_src

The compiler evaluates =result= into a runtime Sexp slot while the
special binding is active, emits one =aot-pop-special 0= for each
source special binding, then emits the existing =throw= bridge with the
saved value.  The same shape applies to all-special multi-binding and
mixed lexical/special =let= forms; mixed forms first establish lexical
aliases so the saved throw value can refer to lexical temporaries.

Stage 129.4H extends this direct cleanup discipline to one-level
conditional bodies:

#+begin_src elisp
(let ((dyn value))
  (if value
      (throw 'tag value)
    value))
#+end_src

The generated shape evaluates the =if= test while the special binding is
active, then emits one cleanup sequence per branch.  Throwing branches
save the payload, pop all active source special bindings, and dispatch to
=nelisp_aot_throw=; normal branches save the result, pop the same
bindings, and return that saved result.  Mixed lexical/special =let=
forms wrap the whole conditional in lexical aliases so the test and both
branches observe the same source bindings.

Stage 129.4I extends this from one-level =if= bodies to nested
conditional trees:

#+begin_src elisp
(let ((dyn value))
  (if value
      (if value
          (throw 'tag value)
        value)
    value))
#+end_src

Every leaf emits the same cleanup sequence: evaluate and save the leaf
value, pop all active source special bindings, then either return the
saved value or dispatch the saved payload through =nelisp_aot_throw=.
Arbitrary non-conditional non-local bodies still require the 129.8
landing-pad path to guarantee cleanup.

** §1.D =setq= / =defvar= / =defconst= / =defcustom= — symbol cells

Writing a global needs symbol value-cell access.  =nl_env_set_value=
extern already exists (used by =nelisp-cc-sf-setq.el= for the
substrate setq swap) — reuse it.  =defvar= / =defcustom= become
=defvar=-then-conditional-set sequences at top level.

*** §1.D.1 Shipped MVP: boxed Int setq delegation

Stage 129.3A adds an explicit ABI surface rather than full user
=setq= syntax:

#+begin_src elisp
(defun-sexp-int-setq set-x (out mirror frames scratch name a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun set-x (out mirror frames scratch name a)
  (seq
   (sexp-int-make out (+ (sexp-int-unwrap a) 16))
   (extern-call nelisp_env_set_value
                mirror frames name out scratch 0)
   out))
#+end_src

The caller still supplies =name= as a =Sexp::Symbol= pointer and a
prebuilt env scratch vector.  That keeps the commit inside existing
grammar/extern coverage and avoids pretending full user =setq= is
available before symbol literal materialisation, =defvar= boundp
checks, and general tagged values land.

*** §1.D.2 Shipped MVP: symbol literal materialisation for boxed setq

Stage 129.3B adds a narrow variant for compile-time-known variable
names:

#+begin_src elisp
(defun-sexp-int-setq-symbol set-x x
    (out mirror frames scratch name-slot a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun set-x (out mirror frames scratch name-slot a)
  (seq
   (sexp-write-symbol-lit name-slot "x")
   (sexp-int-make out (+ (sexp-int-unwrap a) 16))
   (extern-call nelisp_env_set_value
                mirror frames name-slot out scratch 0)
   out))
#+end_src

=sexp-write-symbol-lit= calls the existing =nl_alloc_symbol= helper,
but its UTF-8 bytes are materialized on the stack for the duration of
the call.  That avoids adding =.rodata= to object-mode output, which
is still forbidden by the Doc 99 ET_REL spike path.

This removes the need for the caller to prebuild the symbol object,
but it remains a boxed-boundary helper: the caller still provides
=OUT=, environment pointers, scratch storage, and a writable
=NAME-SLOT=.

*** §1.D.3 Shipped MVP: symbol literal defvar conditional set

Stage 129.3C adds a sibling helper for =defvar=-style conditional
initialization:

#+begin_src elisp
(defun-sexp-int-defvar-symbol ensure-x x
    (out mirror frames scratch name-slot a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun ensure-x (out mirror frames scratch name-slot a)
  (seq
   (sexp-write-symbol-lit name-slot "x")
   (if (= (extern-call nelisp_mirror_is_bound
                       mirror name-slot out)
          1)
       name-slot
     (seq
      (sexp-int-make out (+ (sexp-int-unwrap a) 16))
      (extern-call nelisp_env_set_value
                   mirror frames name-slot out scratch 0)
      name-slot))))
#+end_src

The return value is the materialized symbol slot in both branches,
matching the useful =defvar= result shape for this boxed-boundary
helper.  The =OUT= slot is dual-use: before the bound check, the
caller must initialize it with the runtime unbound-marker Sexp; on the
unbound branch, compiled code overwrites it with the boxed initial
value and delegates the actual value-cell write to the existing env
setter.

This proves the conditional-write substrate without changing the
top-level program ABI.  General user-facing =defvar= forms still need
the Doc 99/runtime entry bridge that supplies env pointers and the
unbound marker automatically.

*** §1.D.4 Shipped MVP: symbol literal defconst value + constant flag

Stage 129.3D adds the =defconst= sibling:

#+begin_src elisp
(defun-sexp-int-defconst-symbol const-x x
    (out mirror frames scratch name-slot a)
  (+ a 16))
#+end_src

This lowers to:

#+begin_src elisp
(defun const-x (out mirror frames scratch name-slot a)
  (seq
   (sexp-write-symbol-lit name-slot "x")
   (sexp-write-nil out)
   (extern-call nelisp_mirror_set_constant
                mirror name-slot out)
   (sexp-int-make out (+ (sexp-int-unwrap a) 16))
   (extern-call nelisp_env_set_value
                mirror frames name-slot out scratch 0)
   (sexp-write-t out)
   (extern-call nelisp_mirror_set_constant
                mirror name-slot out)
   name-slot))
#+end_src

The first =nelisp_mirror_set_constant= clears an existing constant flag
so the existing =nelisp_env_set_value= constant guard does not reject a
reinitializing =defconst=.  After the value write, the final call marks
the symbol constant again.  =OUT= is reused as the Nil/T flag slot and
as the boxed BODY value slot; =NAME-SLOT= remains the function result.

As with 129.3C, this is still an explicit boxed-boundary helper.
Direct top-level execution needs the Doc 99/runtime entry ABI to
supply env pointers, scratch storage, and result/name slots.

*** §1.D.5 Shipped MVP: top-level defvar/defconst/defcustom AOT init helpers

Stage 129.3E connects ordinary =defvar= / =defconst= top-level syntax
to the helper substrate for object output, Stage 129.3F extends the
same path to =defcustom= initial values, Stage 129.3G exposes the
matching =defcustom= metadata as compiler-side descriptors, and Stage
129.3H adds the runtime scheduling plan shape:

#+begin_src elisp
(seq
  (defvar x 42 "doc")
  (defconst y 7 "doc")
  (defcustom z 9 "doc" :type 'integer))
#+end_src

The frontend lowers that to generated helper defuns:

#+begin_src elisp
(seq
  (defun-sexp-int-defvar-symbol nelisp_aot_var_0_x x
      (out mirror frames scratch name-slot)
    42)
  (defun-sexp-int-defconst-symbol nelisp_aot_const_1_y y
      (out mirror frames scratch name-slot)
    7)
  (defun-sexp-int-defvar-symbol nelisp_aot_custom_2_z z
      (out mirror frames scratch name-slot)
    9))
#+end_src

Those helpers then lower through 129.3C / 129.3D.  The generated
symbols are deterministic and linker-friendly, so =compile-to-object=
emits normal GLOBAL FUNC symbols that the Doc 99 runtime layer can
call during module initialization.  Declaration-only =(defvar x)= is
stripped when other top-level forms remain; no special-variable
metadata is emitted yet.

For =defcustom=, the compiler also exposes a side descriptor:

#+begin_src elisp
(:name z
 :helper nelisp_aot_custom_2_z
 :standard 9
 :docstring "doc"
 :options (:type (quote integer)))
#+end_src

This validates the docstring and requires customization options to be
keyword/value pairs.  The descriptor is the handoff shape for Doc 99
module scheduling; runtime customization-table registration is still
pending.

For all top-level variable initializers, the compiler exposes ordered
init-helper descriptors:

#+begin_src elisp
((:kind defvar
  :name x
  :helper nelisp_aot_var_0_x
  :index 0)
 (:kind defconst
  :name y
  :helper nelisp_aot_const_1_y
  :index 1)
 (:kind defcustom
  :name z
  :helper nelisp_aot_custom_2_z
  :index 2))
#+end_src

=nelisp-phase47-compiler--module-init-plan= is the compiler-side
single-call handoff; it gathers top-level init descriptors,
=defcustom= metadata descriptors, 129.5C root descriptors, and 129.7
closure descriptors, then normalizes them through
=nelisp-cc-runtime-aot-module-init-plan= into one pure Doc 99 module
plan:

#+begin_src elisp
(:helper-order (nelisp_aot_var_0_x
                nelisp_aot_const_1_y
                nelisp_aot_custom_2_z)
 :custom-by-helper ((nelisp_aot_custom_2_z . ...))
 :root-descriptors (...)
 :closure-descriptors (...))
#+end_src

=nelisp-cc-runtime-run-aot-module-init-plan= executes that plan in
order once Doc 99 supplies the native helper caller and boundary
context:

#+begin_src elisp
(nelisp-cc-runtime-run-aot-module-init-plan
 plan
 '(:out OUT :mirror MIRROR :frames FRAMES
   :scratch SCRATCH :name-slot NAME-SLOT)
 call-helper-fn
 register-custom-fn)
#+end_src

The runtime side validates the context keys and fires
=register-custom-fn= only after the matching =defcustom= helper has
run.  =call-helper-fn= is still the Doc 99 loader boundary: it resolves
the symbol, performs the native call, and owns any platform-specific
FFI details.

When =register-custom-fn= is nil, runtime registration defaults to
=nelisp-cc-runtime-register-aot-custom-metadata=.  Metadata is stored
by variable name in =nelisp-cc-runtime--aot-custom-table= and can be
queried through =nelisp-cc-runtime-aot-custom-metadata= or snapshotted
with =nelisp-cc-runtime-aot-custom-table-snapshot=.  The table stores
only descriptor data plus the init helper kind/index, not the
transient =OUT/MIRROR/FRAMES/SCRATCH/NAME-SLOT= pointers.

=nelisp-cc-runtime-aot-init-helper-caller= is the standard
=call-helper-fn= implementation for Doc 99.  It resolves helper
symbols through =nelisp-cc-runtime-resolve-aot-init-helper=, which
wraps the existing Doc 81 =nelisp-cc-runtime-resolve-symbol= contract:

#+begin_src elisp
(:helper nelisp_aot_var_0_x
 :status :resolved        ;; or :host-stub / :not-found
 :addr #x...
 :abi-argv (OUT MIRROR FRAMES SCRATCH NAME-SLOT)
 :descriptor ...)
#+end_src

If Doc 99 supplies a native-call callback, runtime invokes it as
=(NATIVE-CALL RESOLUTION CONTEXT INIT-DESCRIPTOR)=.  On host Emacs,
the default resolver still returns =:host-stub= so ERT can inspect the
contract without dereferencing a native address.  =:not-found= raises
before any native-call callback is invoked.

=nelisp-cc-runtime-aot-init-helper-argv= is the single runtime-side
source of truth for the generated helper ABI.  Its order matches the
Stage 129.3E/F helper parameters:

#+begin_src elisp
(OUT MIRROR FRAMES SCRATCH NAME-SLOT)
#+end_src

=nelisp-cc-runtime-aot-native-call-entry= is the standard adapter for
Doc 99's real native-call thunk.  It receives the resolution already
decorated by =nelisp-cc-runtime-call-aot-init-helper=, requires
=:status :resolved=, and forwards only the native address and fixed ABI
argv as the call-critical pair:

#+begin_src elisp
(funcall (nelisp-cc-runtime-aot-native-call-entry native-thunk)
         '(:helper nelisp_aot_var_0_x
           :status :resolved
           :addr #x401000
           :abi-argv (OUT MIRROR FRAMES SCRATCH NAME-SLOT)
           :descriptor ...)
         context
         descriptor)
;; native-thunk sees:
;;   (ADDR ARGV CONTEXT DESCRIPTOR RESOLUTION)
#+end_src

Host-stub and =:not-found= resolutions are rejected before the thunk is
called, so standalone/native paths can share one checked entry point
instead of revalidating the resolver result at every call site.

=nelisp-cc-runtime-make-aot-init-context= fills the standard
helper-call boundary context:

#+begin_src elisp
(:out OUT :mirror MIRROR :frames FRAMES
 :scratch SCRATCH :name-slot NAME-SLOT
 :env ENV) ;; optional native loader/runtime handle
#+end_src

When the caller does not supply =MIRROR= or =FRAMES=, the runtime now
allocates fresh hash-table environment handles for those values.  When
the caller does not supply =OUT=, =SCRATCH=, or =NAME-SLOT=, the runtime
allocates fresh one-slot vectors for those values.  The convenience wrapper
=nelisp-cc-runtime-run-aot-module-init-plan-with-default-context=
uses the same allocation helper before delegating to the ordinary
module-init plan runner.  This gives Doc 99 a reusable boundary
context constructor while preserving the explicit callback point for
native helper invocation.  The optional =:env= entry is deliberately
not part of =nelisp-cc-runtime-aot-init-helper-argv=; it is a
loader/native-call side channel for carrying the runtime =Env= pointer
that backs or owns the =MIRROR= / =FRAMES= handles.

This still is not full top-level execution inside Phase 47 itself.
The compiled object exposes the initializer entry points and the
runtime now has a validated scheduling/execution hook, metadata store,
helper resolution/call contract, default allocation for the standard
mirror/frames/out/scratch/name slots, a canonical native helper argv,
default AOT closure descriptor registration, and a context field for
standalone Env pointer handoff.  It also has the checked native-call
entry adapter that hands a resolved address and ABI argv to the
platform thunk.  Stage 129.3Q adds the first concrete process-backed
thunk:

#+begin_src elisp
(nelisp-cc-runtime-aot-native-call-entry
 (nelisp-cc-runtime-aot-executable-native-thunk
  "/path/to/native-helper-shim"))
#+end_src

The default executable argv is:

#+begin_src text
ADDR HELPER KIND ARGC ARG0 ARG1 ...
#+end_src

where =ARG0...= are printable representations of
=(OUT MIRROR FRAMES SCRATCH NAME-SLOT)=.  The thunk captures stdout
and exit status, and treats non-zero exits as runtime errors unless the
caller opts into =:allow-nonzero=.  Native embedding of the plan remains
Doc 99 call-boundary responsibility.
Initializer values are currently limited to the same boxed Int
expression subset accepted by =defun-sexp-int-*= helpers.

** §1.E Closures + =funcall= / =apply= dispatch

=(lambda (x) (+ x cap))= = heap env record (the captured bindings) +
a code pointer.  =funcall= / =apply= dispatch through the runtime's
existing callable-resolution (=apply_function= / =nelisp--apply=) via
=extern-call= — *delegate to the runtime dispatcher rather than
re-implementing arity/&optional/&rest handling*.  Mirrors the
bytecode VM's MAKE-CLOSURE / CALL opcodes.

*** §1.E.1 Shipped MVP: one-argument =funcall= dispatcher bridge

Stage 129.7A adds the first higher-order call surface without yet
allocating closures.  A boxed-boundary defun may write:

#+begin_src elisp

    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (fn :type sexp) (arg :type sexp))
  (funcall fn arg))
#+end_src

The parser lowers the body to:

#+begin_src elisp

              mirror frames fn arg out scratch)
 out)
#+end_src

The runtime-side bridge has the same ABI order:

#+begin_src elisp

#+end_src

It dispatches through =nelisp--apply= when available, falls back to
host =funcall= for host symbols/functions, writes the result to =OUT=,
and returns =OUT=.  This fixes the call-boundary contract for an
already-materialized function designator/value.  It does not yet build
heap closures from =lambda= forms, capture lexical environments, support
multi-argument =funcall=, or implement =apply=.

Stage 129.7R starts the capture side for direct calls, without
claiming heap closure support.  When a literal lambda is applied
directly, the frontend computes free source variables and threads them
as leading parameters on the synthetic defun:

#+begin_src elisp
(defun caller (cap x)

#+end_src

becomes:

#+begin_src elisp

#+end_src

Runtime lexical slots follow the same rule:

#+begin_src elisp
(defun caller (x)
  (let ((cap x))
    (funcall (lambda (y) (+ y cap)) x)))
#+end_src

lowers to a =let-rt= slot for =cap= and then calls the synthetic defun
with =cap= as the first argument.

calls.  Higher-order callback positions such as =mapcar=,
=seq-filter=, and =cl-find-if= now use the 129.7U heap-closure path

arguments.  Captured =setq= uses the heap-closure path, with 129.7AA
providing runtime write-through cells for later native-frame
materialization.

Stage 129.7T connects that remaining heap-closure substrate to the AOT
runtime dispatcher without yet claiming compiler-side allocation.  When
compiled code passes an existing =nelisp-closure= record as the function
designator to =nelisp_aot_funcall1= / =funcall2= / =funcall3= /
=funcalln= / =apply= / =applyn=, =nelisp-cc-runtime= dispatches
directly through =nelisp-closure-apply=.  That preserves the canonical
closure module's captured environment and shared-cell mutation
semantics, including closures whose bodies mutate captured variables via
=setq=.

Stage 129.7U adds the first compiler-side captured-callback closure
materialization path.  When a higher-order builtin callback position
contains a literal lambda with captured lexical values, preprocessing
keeps a synthetic descriptor and rewrites the callback argument to an
internal =aot-closure-lambda= form.  Direct calln lowering then emits:

#+begin_src elisp
(seq
 (sexp-write-symbol-lit name_slot "mapcar")
 (sexp-write-symbol-lit scratch "nelisp_aot_closure_0")
 (extern-call nelisp_aot_make_closure
              mirror frames scratch 1 out scratch cap)
 (extern-call nelisp_aot_builtin_calln
              mirror frames name_slot 2 out scratch out xs)
 out)
#+end_src

The runtime bridge pairs the descriptor's capture symbols with the
captured values and constructs a canonical =nelisp-closure=, so the
129.7T dispatcher can call it through =nelisp-closure-apply=.  This
path is now backed by 129.7V module-init registration: closure
descriptors flow through =nelisp-cc-runtime-aot-module-init-plan= and
are registered by =nelisp-cc-runtime-run-aot-module-init-plan= before
callback materialization.

Stage 129.7X extends that materialization path from higher-order
callback positions to escaping value positions.  A literal lambda value
such as:

#+begin_src elisp
(defun make-closure
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (cap :type sexp))

now records the same closure descriptor shape and lowers to:

#+begin_src elisp
(seq

              mirror frames scratch 1 out scratch cap)
 out)
#+end_src

The resulting value is the canonical runtime closure object in =out=,
so callers can store or pass it through the existing 129.7T dispatcher
bridges.  Remaining closure work is broader automatic native object
selection of the standalone frame-slot ABI.

Stage 129.7Y extends direct literal-lambda =funcall= preprocessing for
captured mutation.  A form such as:

#+begin_src elisp
(defun caller
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (cap :type sexp) (x :type sexp))
  (funcall (lambda (y) (setq cap y)) x))
#+end_src

no longer attempts to lambda-lift =cap= as a by-value synthetic
argument.  Instead it records a closure descriptor whose body is
=((setq cap y))=, materializes that descriptor with
=nelisp_aot_make_closure=, and passes the resulting closure object as
the function value to =nelisp_aot_funcall1=.  This lets the canonical
runtime closure machinery own the captured mutation cell.  The remaining
native work is propagating that closure cell back into surrounding
compiled frame slots when later source code observes the mutated
captured variable.

Stage 129.7AA adds the runtime cell shape needed by that propagation:

#+begin_src elisp
(nelisp-cc-runtime-aot-capture-cell
 'cap initial-value
 (lambda (name new-value)
   ...write NAME's native frame slot...))
#+end_src

When such a cell appears as the value in a =nelisp-closure= environment,
=nelisp-special-forms-eval= reads the cell's current value for ordinary
symbol references and writes through the cell for lexical =setq=.  The
129.7AB then makes the cell available to compiled code:

#+begin_src elisp
(aot-capture-cell 'cap cap)
#+end_src

lowers to =nelisp_aot_capture_cell(mirror, frames, name, value, out,
scratch)= and records the bridge in the exported AOT C ABI descriptor
table.  Stage 129.7AC wires this into direct captured-mutation lambda
closures automatically, so:

#+begin_src elisp
(funcall (lambda (y) (setq cap y)) x)
#+end_src

now materializes the =cap= capture as =(aot-capture-cell 'cap cap)=
before the =nelisp_aot_make_closure= call.  The runtime bridge unwraps
capture-cell OUT vectors when building the closure environment.
129.7AF fixes the concrete writer target as =FRAMES[slot]= through
=nelisp_aot_frame_slot_set=.  129.7AG exposes explicit compiler forms
for that ABI; native object emission still needs to choose those forms
automatically from compiled slot metadata.

Stage 129.7AH starts that automatic selection for the first observable
captured-mutation shape:

#+begin_src elisp
(seq
 (funcall (lambda (y) (setq cap y)) x)

#+end_src

The frontend tracks that the literal-lambda call mutates =cap= through
the capture-cell writer, and rewrites the later =cap= read to:

#+begin_src elisp

#+end_src

Object output now emits the =nelisp_aot_frame_slot_ref= relocation for
that read path.  This is deliberately a sequence-local selection pass:
general dataflow across branches, loops, and nested lexical scopes
remains later 129.7/native-object work.

Stage 129.7AK extends that selection to the first branch-safe native
object shape:

#+begin_src elisp
(seq
 (if flag
     (funcall (lambda (v) (setq cap v)) x)
   (funcall (lambda (v) (setq cap v)) y))
 cap)
#+end_src

Because both branches definitely initialize the capture-cell's
write-through =FRAMES[cap]= slot, the later =cap= read is rewritten to
=aot-frame-slot-ref=.  The analysis deliberately uses branch
intersection: if only one branch mutates =cap=, the ordinary by-value
read is preserved to avoid reading a missing frame slot.

Stage 129.7AL extends the same intersection rule to exhaustive =cond=
forms:

#+begin_src elisp
(seq
 (cond
  (flag
   (funcall (lambda (v) (setq cap v)) x))
  (t
   (funcall (lambda (v) (setq cap v)) y)))
 cap)
#+end_src

The =t= clause proves that the =cond= cannot fall through without a
write.  The compiler intersects the definitely-updated captured
variables across the reachable clauses; a =cond= without a final =t=
clause remains conservative and preserves the ordinary by-value read.

Stage 129.7AM also accounts for the first operand of short-circuit
=and= / =or= forms:

#+begin_src elisp
(seq
 (and
  (funcall (lambda (v) (setq cap v)) x)
  flag)
 cap)
#+end_src

Because the leading operand is always evaluated, its captured mutation
is guaranteed after the short-circuit form regardless of the truth value
that controls later operands.  The analysis does not claim mutations in
second or later operands, since those operands may be skipped.  Stage
129.7AP extends the native object proof for this same rule to leading
=or= forms, so both short-circuit operators now have parse and object
coverage for the conservative first-operand case.

Stage 129.7AQ adds the matching intra-form read selection for
short-circuit operands:

#+begin_src elisp
(and
 (funcall (lambda (v) (setq cap v)) x)
 cap)
#+end_src

The preprocessor now visits =and= / =or= operands left-to-right and
passes the captured-mutation facts guaranteed by earlier operands into
later operands.  In the example, the second operand reads =cap= through
=aot-frame-slot-ref= because reaching that operand proves the first
operand has already run the capture-cell writer.  The same rule is valid
for =or=: if the leading operand is truthy the later read is skipped,
and if the later read runs the leading operand has already performed its
guaranteed mutation.

Stage 129.7AR applies the same left-to-right frame-slot fact propagation
to =if= condition expressions:

#+begin_src elisp
(if (funcall (lambda (v) (setq cap v)) x)
    cap
  cap)
#+end_src

The condition expression always runs before either branch.  If it
definitely performs a captured mutation, both branch bodies read that
captured variable through =aot-frame-slot-ref= after their ordinary source
preprocessing.  This is deliberately narrower than claiming facts from a
branch after the =if= exits; branch-exit guarantees still use the
intersection rule from 129.7AK.

Stage 129.7AO extends the frame-slot selection pass through lexical
=let= / =let*= wrappers:

#+begin_src elisp
(seq
 (let ((tmp x))
   (funcall (lambda (v) (setq cap v)) tmp))
 cap)
#+end_src

The initializer expressions and body are both evaluated on the normal
path, so a captured mutation that is definitely performed inside the
lexical wrapper now makes the later =cap= read use
=aot-frame-slot-ref=.  The rewrite is scope-aware: if a later =let=
binds =cap=, references in that body's lexical scope remain ordinary
locals instead of being rewritten to the outer capture-cell frame slot.
Stage 129.7AP also adds native object coverage for the =let*= variant,
closing the proof gap between the parser-level selection and emitted
relocations.

Stage 129.7AI adds the first native object metadata embedding step.
When an ELF ET_REL object has a non-empty module-init plan, the compiler
serializes that plan as a UTF-8/NUL-terminated printable payload in
=.rodata= and publishes a local object symbol:

#+begin_src text
nelisp_aot_module_init_plan
#+end_src

The embedded payload is intentionally loader-friendly rather than a
final binary protocol: it is the same normalized plan exposed by
=nelisp-phase47-compiler--module-init-plan=, carrying top-level
init-helper ordering, =defcustom= metadata, root descriptors, and
closure descriptors.  129.7AJ wires the first loader consumption path;
whole-object ABI selection remains later 129.7 work.

Stage 129.7AJ consumes that embedded metadata from the runtime side.
=nelisp-elf-read-symbol-bytes= can now recover a symbol payload from an
ELF64 =.symtab= entry, and
=nelisp-cc-runtime-read-aot-object-module-init-plan= decodes the
NUL-terminated payload back into a validated Doc 129 module-init plan.
The standalone loader grows the direct object entry:

#+begin_src elisp
(nelisp-cc-runtime-run-aot-standalone-loader-from-object
 path :resolver resolver :native-call native-call)
#+end_src

This keeps the native-object path honest: object output, symbol-table
readback, plan validation, custom metadata registration, and closure
descriptor registration are all tested through the same =.o= artifact.

*** §1.E.4 Shipped MVP: quoted function designators

Stage 129.7D lets the compiler handle symbol function designators
without requiring the caller to pass a pre-materialized =FN= slot:

#+begin_src elisp
(defun call-quoted
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (arg :type sexp))
  (funcall 'identity arg))
#+end_src

The parser lowers this to the same dispatcher path with one extra

#+begin_src elisp
(seq

 (extern-call nelisp_aot_funcall1
              mirror frames name_slot arg out scratch)

#+end_src

The same =name-slot= / =name_slot= convention is used for
=(funcall #'foo arg0 arg1)= and =(apply 'foo args)=.  This stage does
not allocate closure objects.  Non-capturing literal lambdas used by
=funcall= are covered by 129.7K; general heap closures and captured
environments remain part of the closure allocation/capture work.

** §1.F Exception machinery

=condition-case= / =catch= / =throw= / =unwind-protect= / =signal=
need non-local exit.  Two designs:
- *Handler stack* (recommended): a thread-local stack of (tag,
  landing-pad, saved-sp) records; =throw= / =signal= walks it, runs
  =unwind-protect= cleanups, restores sp, jumps to the pad.  Mirrors
  the bytecode VM's PUSH-CATCH / PUSH-CC / PUSH-UNWIND / POP-HANDLER
  opcodes (= the VM already implements this shape in elisp).
- setjmp/longjmp via extern — simpler but fights the spill-slot model.

*** §1.F.1 Shipped MVP: runtime handler-stack substrate

Stage 129.8A fixes the runtime-side contract before the compiler emits
handler records:

#+begin_src elisp
(nelisp-cc-runtime-aot-push-catch TAG LANDING-PAD SAVED-SP)
(nelisp-cc-runtime-aot-push-condition CONDITIONS LANDING-PAD SAVED-SP)
(nelisp-cc-runtime-aot-push-unwind CLEANUP LANDING-PAD SAVED-SP)
(nelisp-cc-runtime-aot-throw TAG VALUE)
(nelisp-cc-runtime-aot-signal ERROR-TAG DATA)
#+end_src

The elisp bridge is intentionally a simulator: =throw= / =signal= return
a landing descriptor instead of jumping:

#+begin_src elisp
(:kind catch
 :landing-pad PAD
 :saved-sp SAVED-SP
 :reason throw
 :tag TAG
 :value VALUE
 :cleanups (...))
#+end_src

For =condition= handlers the descriptor carries =:error (TAG . DATA)=.
Crossed =unwind= handlers run from inner to outer and their return values
are recorded in =:cleanups=.  Unmatched =throw= / =signal= still run
crossed cleanups, unwind the stack, and signal
=nelisp-cc-runtime-error=.

Compiler lowering for =(catch ...)= / =(condition-case ...)= /
=(unwind-protect ...)=, native stack-pointer restoration, and actual
landing-pad jumps remain later 129.8 work.

*** §1.F.2 Shipped MVP: =throw= / =signal= bridge lowering

Stage 129.8B connects the first user-visible exception forms to the
129.8A runtime stack:

#+begin_src elisp
(defun throw-tag
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (throw 'done value))
#+end_src

The parser lowers this to:

#+begin_src elisp
(seq
 (sexp-write-symbol-lit name_slot "done")
 (extern-call nelisp_aot_throw
              mirror frames name_slot value out scratch)
 out)
#+end_src

Dynamic tags avoid the materialization step:

#+begin_src elisp
(defun signal-tag
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (tag :type sexp) (data :type sexp))
  (signal tag data))
#+end_src

lowers to:

#+begin_src elisp
(seq
 (extern-call nelisp_aot_signal
              mirror frames tag data out scratch)
 out)
#+end_src

Runtime bridge shape:

#+begin_src elisp
(nelisp-cc-runtime-aot-throw-boundary
 mirror frames tag value out scratch)
(nelisp-cc-runtime-aot-signal-boundary
 mirror frames tag data out scratch)
#+end_src

On the elisp side both bridge functions write the simulator landing
descriptor into =OUT= and return =OUT=.  The native ABI names are
=nelisp_aot_throw= / =nelisp_aot_signal=; final codegen must still use
the descriptor to restore the saved stack pointer and jump to the
landing pad rather than merely returning it.

*** §1.F.3 Shipped MVP: handler-push bridge lowering

Stage 129.8C adds the registration side of the handler-stack ABI.  The
compiler accepts internal AOT forms:

#+begin_src elisp
(aot-push-catch TAG LANDING-PAD SAVED-SP)
(aot-push-condition CONDITIONS LANDING-PAD SAVED-SP)
(aot-push-unwind CLEANUP LANDING-PAD SAVED-SP)
#+end_src

For a quoted catch tag:

#+begin_src elisp
(aot-push-catch 'done 4096 64)
#+end_src

the lowering is:

#+begin_src elisp
(seq
 (sexp-write-symbol-lit name_slot "done")
 (extern-call nelisp_aot_push_catch
              mirror frames name_slot 4096 64 scratch))
#+end_src

Dynamic =CONDITIONS= / =CLEANUP= values are forwarded directly.  The
runtime boundary bridges are:

#+begin_src elisp
(nelisp-cc-runtime-aot-push-catch-boundary
 mirror frames tag landing-pad saved-sp scratch)
(nelisp-cc-runtime-aot-push-condition-boundary
 mirror frames conditions landing-pad saved-sp scratch)
(nelisp-cc-runtime-aot-push-unwind-boundary
 mirror frames cleanup landing-pad saved-sp scratch)
#+end_src

These functions install 129.8A handler records and return the handler
descriptor.  Source-level =catch= / =condition-case= / =unwind-protect=
lowering must still synthesize real landing pads, saved stack-pointer
values, cleanup entry points, and balanced pop paths.

*** §1.F.4 Shipped MVP: handler-pop bridge lowering

Stage 129.8D adds the normal-exit counterpart:

#+begin_src elisp
(aot-pop-handler 'catch)
#+end_src

lowers to:

#+begin_src elisp
(seq
 (sexp-write-symbol-lit name_slot "catch")
 (extern-call nelisp_aot_pop_handler
              mirror frames name_slot out scratch)
 out)
#+end_src

Runtime bridge shape:

#+begin_src elisp
(nelisp-cc-runtime-aot-pop-handler-boundary
 mirror frames expected-kind out scratch)
#+end_src

The bridge pops the innermost handler, optionally validates its kind,
writes the descriptor into =OUT=, and returns =OUT=.  This makes
balanced push/pop expressible in object output; 129.8E below uses it
for the first source-level =catch= normal-exit path.

*** §1.F.5 Shipped MVP: source =catch= normal-exit lowering

Stage 129.8E starts synthesizing source-level handler control flow.
For a boxed-boundary function:

#+begin_src elisp
(defun catch-value
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (catch 'done value))
#+end_src

the compiler lowers the normal path as if it had written:

#+begin_src elisp
(seq
 (aot-push-catch 'done 0 0)
 (let (((aot-catch-value-N :type sexp) value))
   (seq
    (aot-pop-handler 'catch)
    aot-catch-value-N)))
#+end_src

The temporary is a runtime Sexp local so the BODY result survives the
pop bridge call.  The placeholder =0= landing pad / saved stack pointer
keeps this stage honest: it proves balanced source-form lowering and
object relocations, but it is not the final non-local exit path.

Stage 129.8L covers the direct throw shape:

#+begin_src elisp
(catch 'done (throw 'done value))
#+end_src

The compiler pushes the catch handler, emits =nelisp_aot_throw=, and
then calls =nelisp_aot_landing_value= to move the landing descriptor's
=:value= payload back into =OUT=.  The throw bridge itself pops through
the matching catch handler, so this path does not emit the normal-exit
=aot-pop-handler=.

Stage 129.8N covers the first conditional throw shape:

#+begin_src elisp
(catch 'done
  (if pred
      (throw 'done value)
    fallback))
#+end_src

The compiler pushes one catch handler before the =if=.  The throwing
branch emits =nelisp_aot_throw= and =nelisp_aot_landing_value=; the
non-throwing branch saves its value, emits =aot-pop-handler 'catch=,
and returns the saved value.  Stage 129.8R generalizes the same lowering
to nested =if= trees whose throwing leaves target the same catch tag.
Mismatched-tag non-local forms in =catch= bodies still require native
landing-pad labels, stack-pointer restore, and branch-to-pad emission.
=condition-case= direct signal dispatch starts in 129.8M below; nested
condition paths and =unwind-protect= source lowering remain later
129.8 work.

*** §1.F.6 Shipped MVP: source =condition-case= normal-exit lowering

Stage 129.8F applies the same balanced source-form lowering to the
normal path of =condition-case=:

#+begin_src elisp
(defun cc-value
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (condition-case err
      value
    (error out)))
#+end_src

lowers as if it had written:

#+begin_src elisp
(seq
 (aot-push-condition 'error 0 0)
 (let (((aot-condition-value-N :type sexp) value))
   (seq
    (aot-pop-handler 'condition)
    aot-condition-value-N)))
#+end_src

The handler body is intentionally not compiled as a landing pad yet.
Stage 129.8J extends the accepted selector shape to condition lists:

#+begin_src elisp
(condition-case err
    value
  ((error quit) out))
#+end_src

This normal path lowers as two condition pushes with the same
placeholder landing metadata followed by two condition pops after BODY
returns:

#+begin_src elisp
(seq
 (aot-push-condition 'error 0 0)
 (aot-push-condition 'quit 0 0)
 (let (((aot-condition-value-N :type sexp) value))
   (seq
    (aot-pop-handler 'condition)
    (aot-pop-handler 'condition)
    aot-condition-value-N)))
#+end_src

Signalled handler dispatch, binding =VAR= to the error data, and native
jump-to-handler emission remain later 129.8 work.

Stage 129.8K also walks every handler clause on this normal path:

#+begin_src elisp
(condition-case err
    value
  (error out)
  (quit out))
#+end_src

lowers to two condition pushes before BODY and two matching pops after
BODY.  The handler bodies remain metadata for the future landing-pad
path; they are not executed unless a non-local signal path is compiled.

Stage 129.8M covers the first direct signalled path:

#+begin_src elisp
(condition-case err
    (signal 'error data)
  (error err))
#+end_src

The compiler statically selects the first matching handler, pushes a
condition handler for the direct tag, emits =nelisp_aot_signal= or the
specialised =error= bridge, then calls =nelisp_aot_landing_error= to
bind =err= to the descriptor's =:error= payload before compiling the
handler body.  The signal bridge itself pops through the matched
condition handler, so this path does not emit the normal-exit
=aot-pop-handler=.

Stage 129.8Q covers the first conditional signalled path:

#+begin_src elisp
(condition-case err
    (if value
        (signal 'error value)
      value)
  (error err))
#+end_src

The compiler pushes the statically selected condition handler before the
=if=.  A signalled branch emits the direct signal bridge, extracts the
landing descriptor's =:error= payload, and runs the handler body; a
normal branch saves its branch value, pops the condition handler, and
returns that saved value.  Stage 129.8R extends the same lowering to
nested =if= trees whose signalled leaves resolve to the same condition
handler.  Dynamic or mismatched condition paths still require native
landing-pad labels, stack-pointer restore, and branch-to-pad emission.

*** §1.F.7 Shipped MVP: source =unwind-protect= normal-exit lowering

Stage 129.8G covers the normal path of =unwind-protect=:

#+begin_src elisp
(defun unwind-value
    ((out :type sexp) (mirror :type sexp) (frames :type sexp)
     (scratch :type sexp) (name_slot :type sexp) (value :type sexp))
  (unwind-protect
      value
    (identity value)))
#+end_src

lowers as if it had written:

#+begin_src elisp
(let (((aot-unwind-value-N :type sexp) value))
  (seq
   (identity value)
   aot-unwind-value-N))
#+end_src

Stage 129.8O covers the first direct non-local protected body:

#+begin_src elisp
(unwind-protect
    (throw 'done value)
  (identity value))
#+end_src

The compiler saves =value= into a runtime Sexp slot, runs cleanup forms
in order, then emits =nelisp_aot_throw= with the saved value.  This
preserves cleanup ordering for the direct quoted-tag throw shape without
claiming general landing-pad support.

Stage 129.8P extends this to one-level conditional protected bodies:

#+begin_src elisp
(unwind-protect
    (if value
        (throw 'done value)
      value)
  (identity value))
#+end_src

Each branch saves the protected body's value, runs cleanup forms in
order, and then either returns the saved value or emits
=nelisp_aot_throw=.  Stage 129.8R extends this leaf cleanup to nested
=if= trees with direct quoted-tag throw leaves.  Non-conditional
protected non-local bodies still require real cleanup landing pads,
native stack-pointer restore, and branch-to-cleanup / branch-back
emission.

Stage 129.8S introduces the explicit landing-jump ABI used by later
native branch emission:

#+begin_src elisp
(aot-landing-jump out)
#+end_src

lowers to:

#+begin_src elisp
(extern-call nelisp_aot_landing_jump mirror frames out out scratch)
#+end_src

The runtime bridge unwraps the landing descriptor, validates that it
carries both =:landing-pad= and =:saved-sp=, and writes a simulator
jump plan:

#+begin_src elisp
(:kind landing-jump
 :landing-pad PAD
 :saved-sp SAVED-SP
 :reason REASON
 :landing DESCRIPTOR)
#+end_src

This makes the stack-restore/branch target an auditable ABI entry and
object relocation.

Stage 129.8T adds the first backend-native branch primitive:

#+begin_src elisp
(seq
 (aot-machine-landing-jump saved-sp catch-pad)
 (aot-landing-label catch-pad value))
#+end_src

The jump form evaluates =saved-sp= through the ordinary value emitter,
moves it through =r10= into =rsp=, then emits a direct =jmp rel32= to
the landing label.  This gives later =catch= / =condition-case= /
=unwind-protect= lowering a real machine target while keeping the
descriptor-to-label selection work explicit and auditable.

Stage 129.8U adds the matching saved-stack-pointer producer:

#+begin_src elisp
(aot-current-sp)
#+end_src

On x86_64 this emits:

#+begin_src text
mov rax, rsp
#+end_src

It can feed the 129.8T jump primitive directly:

#+begin_src elisp
(aot-machine-landing-jump (aot-current-sp) catch-pad)
#+end_src

Stage 129.8V threads those concrete stack snapshots into the first
source-level descriptors and maps direct catch/condition descriptors
back to compiler landing labels:

#+begin_src elisp
(seq
 (aot-push-catch 'done 'catch-pad (aot-current-sp))
 (throw 'done value)
 (aot-landing-label catch-pad
   (aot-landing-value out)))
#+end_src

The same shape now exists for direct =condition-case= signal/error
handlers via =aot-push-condition= and =aot-landing-error=.  Stage 129.8W
extends the mapping to conditional catch/condition trees that have
exactly one throwing or signalling leaf:

#+begin_src elisp
(seq
 (aot-push-condition 'error 'condition-pad (aot-current-sp))
 (if test
     (seq
      (signal 'error value)
      (aot-landing-label condition-pad
        HANDLER))
   NORMAL))
#+end_src

Stage 129.8X generalizes this to multi-leaf trees by pushing a
leaf-local handler before each non-local leaf:

#+begin_src elisp
(if test
    (seq
     (aot-push-catch 'done 'catch-pad-1 (aot-current-sp))
     (throw 'done value)
     (aot-landing-label catch-pad-1
       (aot-landing-value out)))
  (seq
   (aot-push-catch 'done 'catch-pad-2 (aot-current-sp))
   (throw 'done value)
   (aot-landing-label catch-pad-2
     (aot-landing-value out))))
#+end_src

Stage 129.8Y then gives =unwind-protect= cleanup code concrete entry
labels without changing the existing inline cleanup ordering:

#+begin_src elisp
(let (((aot-unwind-value-N :type sexp) VALUE))
  (aot-landing-label unwind-cleanup-pad
    (seq
     CLEANUP...
     (throw 'done aot-unwind-value-N))))
#+end_src

Stage 129.8Z wires those cleanup pads into the pushed unwind descriptor
and hands control back to the original non-local landing descriptor after
cleanup:

#+begin_src elisp
(seq
 (aot-push-unwind 0 'unwind-cleanup-pad (aot-current-sp))
 (let (((aot-unwind-value-N :type sexp) VALUE))
   (seq
    (throw 'done aot-unwind-value-N)
    (aot-landing-label unwind-cleanup-pad
      (seq
       CLEANUP...
       (aot-landing-jump out))))))
#+end_src

This remains the descriptor-level bridge: =aot-landing-jump= preserves
the original catch/condition target carried by =out=, while later native
selection can replace the simulator bridge with a concrete
=aot-machine-landing-jump= when the target label is statically known.

Stage 129.8AA performs that static selection for source =catch= and
=condition-case= throw/signal leaves whose handler landing label is
known at compile time:

#+begin_src elisp
(seq
 (aot-push-catch 'done 'catch-pad (aot-current-sp))
 (seq
  (throw 'done value)
  (aot-machine-landing-jump (aot-current-sp) catch-pad)
  (aot-landing-label catch-pad
    (aot-landing-value out))))
#+end_src

The same insertion is used for direct, single-leaf, nested, and
multi-leaf condition trees.  Dynamic descriptor-only cleanup handoff
continues to use =aot-landing-jump= until the original target label can
be proven statically.

Stage 129.8AB performs that selection for the first cleanup handoff
shape where the original target is statically visible: a =catch= whose
body is a direct =unwind-protect= throw to the same tag.

#+begin_src elisp
(seq
 (aot-push-catch 'done 'catch-pad (aot-current-sp))
 (seq
  (aot-push-unwind 0 'cleanup-pad (aot-current-sp))
  (let (((aot-unwind-value-N :type sexp) value))
    (seq
     (throw 'done aot-unwind-value-N)
     (aot-landing-label cleanup-pad
       (seq
        CLEANUP...
        (aot-machine-landing-jump (aot-current-sp) catch-pad)))
     (aot-landing-label catch-pad
       (aot-landing-value out))))))
#+end_src

This keeps the descriptor-level path for dynamic or not-yet-proven
cleanup targets, while avoiding the simulator bridge when the cleanup
pad can branch to a concrete same-function label.

Stage 129.8AC extends the same static selection to the first conditional
cleanup handoff where the target is still fully known:

#+begin_src elisp
(catch 'done
  (unwind-protect
      (if flag
          (throw 'done left)
        (throw 'done right))
    CLEANUP...))
#+end_src

The compiler only accepts all-throw trees for this static route.  Normal
leaves remain on the broader dynamic cleanup path because they must
preserve the outer handler pop and ordinary result flow.  For the
accepted all-throw case, each throw leaf pushes its own cleanup handler,
runs cleanup once, and then emits =aot-machine-landing-jump= to the
single catch landing label.

Stage 129.8AD keeps that catch-targeted static route when the protected
conditional body mixes matching throw leaves with ordinary leaves:

#+begin_src elisp
(catch 'done
  (unwind-protect
      (if flag
          (throw 'done value)
        value)
    CLEANUP...))
#+end_src

Throwing leaves still run the cleanup forms and branch with
=aot-machine-landing-jump= to the shared catch landing.  Ordinary leaves
save their branch result, run the same cleanup forms, pop the catch
handler with =aot-pop-handler=, and return the saved result.  This keeps
the normal path balanced while avoiding =aot-landing-jump= for the
statically-known cleanup-to-catch transfer.

Stage 129.8AE applies the same static cleanup-target selection to the
first =condition-case= cleanup handoff:

#+begin_src elisp
(condition-case err
    (unwind-protect
        (signal 'error value)
      CLEANUP...)
  (error err))
#+end_src

The compiler pushes the condition handler with a concrete landing label,
pushes the unwind descriptor for the cleanup pad, and lets the direct
=signal= path fall into that cleanup pad.  The pad runs cleanup and then
branches with =aot-machine-landing-jump= to the shared condition
landing, where =aot-landing-error= exposes the saved condition data to
the handler body.  Cleanup forms that contain their own non-local exits
remain rejected by the existing unwind cleanup guard.

Stage 129.8AF keeps that condition-targeted static route for the first
mixed conditional protected body:

#+begin_src elisp
(condition-case err
    (unwind-protect
        (if flag
            (signal 'error value)
          value)
      CLEANUP...)
  (error err))
#+end_src

The signalled leaf follows the cleanup pad to the shared condition
landing with =aot-machine-landing-jump=.  The ordinary leaf saves its
branch result, runs the same cleanup forms inline, pops the condition
handler with =aot-pop-handler=, and returns the saved result.  This is
the condition-case analogue of the 129.8AD catch-targeted mixed cleanup
selection.

** §1.G macroexpand frontend (= cheapest group)

=defmacro= / =when= / =unless= / =dolist= / =cl-*= / =pcase= reduce to
core forms via compile-time macro expansion.  The Wave A21 bytecode
path already does broad =macroexpand-all= before =nelisp-bc-compile=.
For Phase 47, the shipped MVP is deliberately narrower: selectively
macroexpand value/call positions while preserving DSL structural forms
such as =defun=, because host =macroexpand-all= rewrites them to
=defalias= forms outside the Phase 47 grammar.  Multi-form macro
bodies lower through =value-seq= so the final child value is preserved
in value context.  Near-zero new asm.

*** §1.G.1 Shipped MVP: top-level module forms

Stage 129.6A extends the frontend with top-level =require= /
=provide= handling:

#+begin_src elisp

  (defmacro ...)
  (provide 'my-feature)
  ...)
#+end_src

=require= is evaluated during compilation, before top-level

** §1.H Builtin library delegation

=format= / =concat= / =substring= / =string-match= / =mapcar= /
=nth= / =assq= / ... = hundreds of builtins.  Pragmatic route: emit a
=call= into the runtime's existing implementation via =extern-call=
(the dispatcher already resolves builtins).  Consequence: compiled
output links against the nelisp runtime (= "compiled code + runtime"),
*not* a tiny standalone ELF.  This is acceptable for the
interpreter-embedded use case (Doc 99 Arch B/C) but is in tension with
Doc 97's standalone-binary lineage — call out explicitly per output
target.

*** §1.H.1 Shipped MVP: one-argument Sexp builtin delegation helper

#+begin_src elisp

* §3 Honest scope assessment

§1.A + §1.B + §1.E + §1.F together constitute a *complete native-code
language runtime*.  "Compile =anvil-server.el= to native" is, in
honest terms, "build a native-comp backend for elisp" — the same
scope as GNU Emacs's libgccjit native-comp, except hand-emitted via
the Phase 47 asm chain instead of delegating to GCC.  Realistic
horizon: multi-month to multi-year, paced one group per engagement.

This is *not* a near-term cold-load fix.  It is the structural endgame
that aligns with Doc 99 Arch C ("every byte of code derives from elisp
the Phase 47 chain compiled").

* §4 Relationship to the other axes

- *Doc 99 (call boundary)*: gives the interpreter an ABI to call
  compiled functions.  Doc 129 gives those functions a language.  A
  single compiled =anvil-server= function needs *both* (Doc 99 ET_REL
  + ABI, Doc 129 grammar coverage).
- *Doc 122 (substrate primitives)*: supplies the alloc / mut-str /
  extended-extern-call primitives that Doc 129 groups call.  Doc 129
  is "Doc 122 consumed at application level".
- *Doc 126 (eval-boot)*: precedent — replacing "interpret source" with
  a faster path.  Doc 129 is the user-code analogue of that move.

* §5 The pragmatic alternative (= why this is endgame, not now)

The Wave A21 bytecode VM (=byte-compile-file= → =.elc= → =nelisp-bc-run=)
*already* handles closures, exceptions, hash, and higher-order calls —
because it runs on the interpreter's runtime, not bare registers.  Its
only blocker for standalone use was a heap-corruption bug in the
reader (§6), now fixed.  Bytecode gives a 5–10x evaluator speedup with
*zero* new grammar.  Phase 47 AOT (this doc) gives native speed but
needs the full §1 stack.

Recommended sequencing: harvest the bytecode-VM win first (cheap,
grammar-free), pursue Doc 129 as the long-horizon native-comp endgame.

* §6 Prerequisite SHIPPED — reader heap-corruption fix (2026-05-25)

Loading a Wave A21 =.elc= with a large pre-compiled =defun= (e.g.
=anvil-server--handle-tools-call=, a 1,099-element bytecode CODE
vector) aborted standalone NeLisp with glibc =corrupted size vs.
prev_size= / =double free=.

Root cause: =build-tool/src/lib.rs= reader =PARSER_POOL_SIZE = 3 + 4 *
1024= (depth ≈1023).  The Phase 47 elisp parser
(=nelisp-cc-reader-parser.el=) recurses depth+1 per list/vector
*element*, indexing the slot-pool via *unchecked* =vector-ref-ptr=, so
any list/vector literal with ≳1024 elements overran the pool → heap
corruption (threshold confirmed: 1000-element OK, 1024 crash).

Fix (1 line, net 0 Rust LOC): size the pool to the input —
=vec![Sexp::Nil; (3 + 4 * input.len()).max(PARSER_POOL_SIZE)]=.  An
element count can never exceed the input byte count, so pool overflow
is structurally impossible.  Verified: 1,099 + 5,000-element vector
literals and the full =anvil-server.elc= (68 bcl defuns) load cleanly;
=mini.elc= regression + cargo =--lib= 10/10 green.

Proper follow-up (not blocking): the parser's depth-per-element
recursion is itself the latent flaw; an *iterative* list/vector parse
(reusing a fixed slot set, accumulating via a running tail) would
remove the input-sized-pool memory cost entirely.  Tracked as a Phase
47 reader-parser cleanup, independent of Doc 129.
