** 1.2 既存 NeLisp memory layout (踏襲できる前提)

NeLisp は host Emacs の =Lisp_Object= layout を踏襲し、3-bit low-tag
pointer + 単一 heap (host Emacs allocator 経由) で動作してきた。Phase
7.2 では host Emacs allocator を捨て、NeLisp 自前 allocator が
=mmap= (Phase 7.0 syscall) でページ確保 → bookkeeping を Lisp で実装する。

| 既存資産                            | LOC | 7.2 での役割                                       |
|-------------------------------------+-----+----------------------------------------------------|
| =src/nelisp-types.el= (3-bit tag)   | -   | tag 定義踏襲、本 phase で変更しない                |
| =src/nelisp-runtime= (Phase 7.0)    | 689 | =mmap= / =munmap= / =mprotect= 経由でページ確保    |
| =src/nelisp-cc= (Phase 7.1)         | -   | hot-path (alloc / promotion) を JIT、bytecode 比 ~30-50x |
| =src/nelisp-defs-index= (Phase 6.5) | 793 | per-type pool の type 解決 1 query                 |

= Phase 7.2 allocator は *Phase 7.0 mmap 経由 / Phase 7.1 native JIT /
=nelisp-defs-index= type 解決* の interface が既に固定された位置で
動く。自由度が制約され設計が安定する (Doc 28 §1.2 と同 pattern)。

** 1.3 SBCL alloc.lisp + Emacs alloc.c dual precedent reference

NeLisp の allocator は SBCL =src/runtime/alloc.lisp= (CL ~3000 LOC、
25 年維持) と Emacs =src/alloc.c= (C ~6000 LOC、Emacs 1985-) の dual
precedent を踏まえ、*runtime purity 軸で両者を超え、allocator
capability 軸では両者の MVP subset* を狙う (Doc 28 §1.3 と同 pattern)。

| 軸           | SBCL alloc.lisp          | Emacs alloc.c            | NeLisp Phase 7.2          |
|--------------+--------------------------+--------------------------+---------------------------|
| 実装 lang    | CL                       | C                        | NeLisp                    |
| GC type      | generational mark-region | mark-sweep (主)          | Phase 7.3 で決定           |
| allocator    | bump + free-list         | block + free-list        | bump + free-list (2 gen)  |
| heap layout  | semi-space (Cheney)      | per-type block           | 2-gen (nursery / tenured) |
| LOC          | ~3000 CL                 | ~6000 C                  | 1500-3000 NeLisp          |
| thread model | multi (mutator + GC)     | single (主) / pthread mutex | single MVP (Phase 7.5+ multi) |

*SBCL precedent との差別化軸*:
1. *Purity*: NeLisp は host Emacs allocator を Phase 7.2 で永久放棄、
   pure NeLisp + mmap syscall pipeline を確立する
2. *Bootstrap simplicity*: SBCL は genesis-2 で半空間 image を構築する
   が、NeLisp は Phase 7.0 syscall stub の =mmap= 直叩きで簡略化
3. *Cheney semispace は Phase 7.3 で導入*: Phase 7.2 の本 phase は
   bump + free-list の moveable arena まで、コピー型は GC 着手と co-design

*Emacs alloc.c precedent との差別化軸*:
1. per-type block (=cons_block= / =string_block= 等) の発想は踏襲、
   ただし 2-generation 化 (nursery / tenured) で promotion を入れる
2. host C dependency を完全除去、=malloc= / =free= は Phase 7.0 syscall
   stub 経由 =mmap= に置換

** 1.4 Doc 28 §2.9 GC metadata contract と Phase 7.2 heap-region registry contract の分離 (v2 改訂)

v2 改訂 (codex 観点 1.2 反映): 当初は allocator を「GC metadata
contract provider」と書いていたが、これは Doc 28 §2.9 の compiler-side
contract (safe-point ID / live root bitmap / frame layout descriptor)
と allocator-side の heap structure metadata を混線させていた。本 v2
では *2 つの独立 contract* として明示分離:

| contract                              | 仕様 LOCK 場所       | producer       | consumer    |
|---------------------------------------+----------------------+----------------+-------------|
| *GC metadata contract v1* (Doc 28 §2.9) | Doc 28 v2 LOCKED    | Phase 7.1 native compiler | Phase 7.3 GC |
| *heap-region registry v1* (本 doc §2.6 新設) | Doc 29 v2 LOCKED  | Phase 7.2 allocator | Phase 7.3 GC |

両 contract を Phase 7.3 (Doc 30) が *別々に consume* する設計。
Phase 7.2 allocator が compiler ABI に touch しない構造で、Phase 7.1
backend と独立に進化可能。

heap-region registry v1 schema (本 doc §2.6 で詳細 LOCK):
- region-id (sequential uint32)
- start / end address (size_t pair)
- generation tag (=:nursery= / =:tenured=)
- object family (=:cons-pool= / =:string-span= / =:vector-span= /
  =:closure-pool= / =:large-object=)
- region table format version = =NELISP_HEAP_REGION_VERSION= constant
  (本 doc が独立 versioning、Doc 28 GC_METADATA_VERSION と無関係)

= Phase 7.2 allocator は *heap-region registry v1 の provider*、
*GC metadata contract の touch 禁止* (consumer も provider も
not、Phase 7.1 と Phase 7.3 のみ)。Phase 7.3 着手時に両 contract を
別々に review する。

** 2.1 allocator algorithm [推奨 = A bump + free-list 2-generation]

| 候補 | 戦略                                                 | 利点                                       |
|------+------------------------------------------------------+--------------------------------------------|
| A    | bump (nursery) + free-list (tenured) (default)       | SBCL/Emacs dual precedent middle path、moveable arena まで |
| B    | Cheney semispace 直導入                              | Phase 7.3 GC と同時導入、本 phase scope 過大 |
| C    | mark-region (= G1GC 風)                              | scope 過大、Phase 8+ で検討                 |

*推奨 A 理由*:
- Cheney semispace は Phase 7.3 (GC inner) で collector 本体と co-design
  するのが rational、本 phase は moveable arena まで
- bump allocator は hot-path で 1 命令 (pointer increment) = Phase 7.1
  native JIT で C 同等速度
- free-list (tenured) は size-class 化で fragmentation 抑制

** 2.2 heap layout [推奨 = A 2-generation (nursery / tenured)]

| 候補 | 戦略                                                | 利点                                     |
|------+-----------------------------------------------------+------------------------------------------|
| A    | 2-generation (nursery 4MB default / tenured 64MB) (default) | generational hypothesis 活用、Phase 7.3 で minor GC 容易 |
| B    | 単一 heap (generation なし)                         | 実装最簡だが GC pause time 不利           |
| C    | 3+ generation                                       | scope 過大、Phase 8+ で検討               |

*推奨 A 理由*:
- short-lived object (=let= local / closure / arithmetic intermediate)
  が 95%+ という generational hypothesis は Lisp で実証済 (SBCL 等)
- size 配分 (nursery 4MB / tenured 64MB) は user tunable
  (=defcustom nelisp-allocator-nursery-size=) とし MVP default は
  典型 Lisp workload に合わせる

** 2.3 page granularity [推奨 = A 4KB page + 16-page chunk]

| 候補 | 戦略                              | 利点                                     |
|------+-----------------------------------+------------------------------------------|
| A    | 4KB page + 16-page chunk (= 64KB) (default) | OS native page size align、TLB friendly  |
| B    | 2MB huge page                     | TLB miss 削減、allocator overhead 大     |
| C    | 1KB sub-page                      | mmap syscall overhead 過大               |

*推奨 A 理由*:
- 4KB は Linux/macOS/Windows で OS native page size、=mmap= overhead
  最小
- 16-page chunk (64KB) で =mmap= syscall 頻度を 1/16 に圧縮、
  fragmentation も許容範囲

** 2.4 type tag [推奨 = A 既存 NeLisp 3-bit low tag 踏襲]

| 候補 | 戦略                                              | 利点                                       |
|------+---------------------------------------------------+--------------------------------------------|
| A    | 既存 NeLisp 3-bit low tag 踏襲 (default)          | 互換性 100%、=defs-index= で確認済          |
| B    | 4-bit tag に拡張                                  | type 種別増、ABI 破綻                       |
| C    | NaN-boxing                                        | scope 過大、Phase 8+ で検討                 |

*推奨 A 理由*:
- 既存 NeLisp =Lisp_Object= layout (Phase 5-B SHIPPED) を破綻させない
- =defs-index= で symbol → type 1 query で解決可能 (Phase 6.5 SHIPPED)
- Phase 7.2 内で tag layout を変更しない (= ABI freeze)

** 2.5 pool 管理 + object family table [推奨 = A hybrid per-family]

v2 改訂 (codex 観点 2.5 反映): variable-size object (string / vector)
を per-type pool だけで扱えない問題を *object family 分割* で resolve。

| object family   | container        | layout            | free-list 単位     | use case             |
|-----------------+------------------+-------------------+--------------------+----------------------|
| cons-pool       | fixed-size pool  | 16 byte (cdr+car) | per-block bitmap   | =(cons a b)= 等       |
| closure-pool    | fixed-size pool  | 32 byte (header + 3 slot) | per-block bitmap | small closure |
| string-span     | size-classed span | header + payload  | size-class free-list (8/16/32/64/128/256 byte 等) | 大半の string |
| vector-span     | size-classed span | header + payload  | size-class free-list | 大半の vector |
| large-object    | dedicated mmap region | header + payload | LRU free-list | >4KB string/vector  |

選択規則:
- size <= 32 byte かつ fixed-size = cons-pool / closure-pool
- 32 byte < size <= 4KB = string-span / vector-span (size-class round up)
- 4KB < size = large-object path (= mmap 単独 region)

| 候補 | 戦略                                                | 利点                                       |
|------+-----------------------------------------------------+--------------------------------------------|
| A    | hybrid per-family (default、上記 5 family)          | variable-size 対応、Emacs alloc.c + jemalloc dual influence |
| B    | per-type pool 単一 (cons/string/vector/closure)     | variable-size で破綻 (Doc 29 v1 案、reject) |
| C    | 単一 free-list                                      | fragmentation 不利                          |

*推奨 A 理由 (v2)*:
- variable-size object family を fixed-size と size-class に明示分割
  することで、free-list ownership と alignment を contract 化
- Emacs =cons_block= / =string_block= 風の per-family pool が generational
  GC と協調しやすい (Phase 7.3 で minor GC 着手時)
- =defs-index= で type → family 1 query で解決
- large-object path は mmap 直叩きで fragmentation を回避

** 3.1 Phase 7.2.1 — bump allocator nursery (single generation)

*** scope
- =mmap= 経由 nursery 領域確保 (default 4MB、=defcustom=)

  Emacs allocator から離脱)

  本 phase は no-op + abort)
- region table (=nelisp-allocator--region-table=) 初版実装、
  GC metadata contract provider 側

*** ERT smoke (+8)

- 1M cons alloc で nursery 4MB 充足 → overflow 検出 test
- region table が allocator confirm 後 1 entry 追加されている test
- bump pointer 単調増加 invariant test

*** risks

*** scope

- size-class free-list (8 / 16 / 32 / 64 / 128 / 256 / 512 byte)
- promotion path (nursery → tenured、age-aware、Phase 7.3 minor GC で

* §4 Architecture (page allocation 経路)

#+begin_src text
[NeLisp source]
    │ nelisp-cc.el (Phase 7.1 SHIPPED)
    ▼
[native code: nelisp-alloc-cons / nelisp-alloc-string / etc]
    │
    │ Phase 7.2.3 inline JIT (per-type pool fast path)
    ▼
[per-type pool free-list (tenured) or bump pointer (nursery)]
    │
    │ overflow → nursery: trigger-minor-gc (Phase 7.3 stub)
    │ overflow → tenured: =mmap= via Phase 7.0 syscall stub
    ▼
[region table (= GC metadata contract provider)]
    │
    │ Phase 7.3 mark phase が consume (root scanning)
    ▼
[Phase 7.3 GC inner (NeLisp、本 phase scope 外)]
#+end_src

** 4.1 入力境界 (NeLisp source via Phase 7.1 native compile)

Phase 7.1 (Doc 28) SHIPPED 後、=nelisp-alloc-cons= primitive は native
JIT で inline expanded、bytecode VM 比 30-50x speedup。Phase 7.2 hot
path は per-type pool の bump pointer increment 1 命令で C 同等速度。

** 4.2 出力境界 (region table = GC metadata contract provider)

allocator が確保した nursery / tenured ページを =region-table= に
登録、Phase 7.3 mark phase が =GC_METADATA_VERSION= 経由で consume。
本 phase 7.2 は provider 側のみ実装、consumer (mark phase) は Phase
7.3 scope。

** 4.3 syscall 経路

Phase 7.0 syscall stub の =mmap= / =mprotect= に依存:
- nursery: =mmap MAP_ANONYMOUS | MAP_PRIVATE PROT_READ|PROT_WRITE= 4MB
- tenured: =mmap MAP_ANONYMOUS | MAP_PRIVATE PROT_READ|PROT_WRITE= 64MB

各 chunk (64KB = 16-page) 単位で =mmap= 拡張、tenured 領域 overflow
時は =munmap= の代わりに free-list 統合 (fragmentation 抑制)。

* §6 Risks (6 項目)

** 6.1 nursery overflow handling (Phase 7.3 GC co-design)

Phase 7.2.1 で nursery overflow 時 =trigger-minor-gc= stub のみ実装、
Phase 7.3 で actual collector 実装 = co-design 必須。

*影響*: Phase 7.3 で minor GC algorithm (Cheney semispace? mark-region?)
が確定するまで nursery → tenured promotion path の interface が確定
しない。

*mitigation*:
- Phase 7.2.1 で stub 化、Phase 7.3 着手時 orchestrator consensus review
  1 回 (Doc 28 §6.2 と同 pattern)
- region table format = =GC_METADATA_VERSION= で versioned (§2.6)、
  Phase 7.3 で incompatible 改訂が必要なら version bump

** 6.2 size-class 選定 (Emacs alloc.c precedent との divergence)

§2.5 推奨 A で per-type pool + size-class free-list を採用、size-class
(8 / 16 / 32 / ...) は Emacs alloc.c precedent 直接 transfer したが、
NeLisp workload (Phase 5-B selfhost) と差がある可能性。

*影響*: tenured fragmentation 増、Phase 7.5 で実測 tuning 必須。

*mitigation*:
- Phase 7.2.2 で precedent 直接 transfer、Phase 7.5 integration で
  =alloc-profile= 経由 size-class histogram 採取
- Phase 8+ で size-class tuning task として後続化

** 6.3 type tag conflict (既存 NeLisp 3-bit tag 制約)

§2.4 推奨 A で既存 NeLisp 3-bit low tag 踏襲、tag 種別 (cons / string
/ vector / closure / float / int / symbol / misc) は 8 種で固定。新
type 追加時に tag 不足 risk。

*影響*: Phase 7.2.3 per-type pool で type 種別追加 (例: bignum /
hash-table) 時に tag layout 改訂必須、ABI 破綻。

*mitigation*:
- Phase 7.2 内で tag layout 変更しない (= ABI freeze、§2.4)
- 新 type 追加は Phase 8+ で =defs-index= 経由 secondary tag 経路
  (=misc-tag= subtype) で対応、3-bit primary tag は不変

** 6.4 nursery / tenured promotion threshold tuning

§2.2 推奨 A で 2-generation default (nursery 4MB / tenured 64MB)、
promotion threshold (= age N で nursery → tenured 移送) は Phase
7.3 minor GC で確定。

*影響*: 現実的 Lisp workload (Phase 5-B selfhost) で promotion 過多
or 過少だと bench gate (§5.1) C 比 70-90% 未達 risk。

*mitigation*:
- Phase 7.2.2 で manual promotion API (=nelisp-alloc-promote=) のみ
  実装、auto promotion は Phase 7.3 で
- Phase 7.5 integration で =alloc-profile= 経由 promotion rate 採取、
  threshold tuning task

** 6.5 fragmentation accumulation

長時間動作で tenured 領域 fragmentation 累積、Phase 7.2.2 free-list
coalescing で 80%+ utilization 維持を target。defragmentation (=
mark-compact / Cheney semispace) は Phase 7.3 で導入 = Phase 7.2 単体
では fragmentation 累積止め切れない。

*影響*: long-running NeLisp process (例: anvil daemon) で OOM。

*mitigation*:
- Phase 7.2.2 free-list coalescing で 80%+ utilization 維持を ERT で
  固定 (=fragmentation-test=、§5.2)
- Phase 7.3 mark-compact 実装で defragmentation を補助、Phase 7.5
  integration で long-running soak test

** 6.7 generational correctness interim invariant (v2 新設)

codex 観点 3.7 反映: nursery / tenured と promotion を導入しつつ write
barrier を Phase 7.3 へ defer しているため、Phase 7.2 中の tenured →
nursery 参照 (= cross-generation pointer) handling が未定義のままだと、
Phase 7.2.2 promotion + Phase 7.3 minor GC の境界で silent corruption
発生 risk。

*影響*: Phase 7.2 単体動作中、Phase 7.3 着手前の "interim period" で
heap 不変条件が崩れる。Phase 7.3 minor GC が設計通り動作するための
前提が建たない。

*mitigation*:
- *Phase 7.2 中は minor GC を実行しない*: nursery が full に到達したら
  major GC (= mark-sweep + tenured 含む) のみ trigger、minor GC 経路は
  Phase 7.3.3 で初期化 (Doc 30 §3.3 と co-design)
- *tenured → nursery 参照は slow path で remembered-set 記録*: 通常の
  =setq= / =setcar= 等の write は Phase 7.2 段階では cross-generation を
  detect せず full-heap walk 前提、Phase 7.3 で write barrier 導入時に
  remembered-set を有効化
- 上記 2 つを §2.X で固定 (本 v2 で §2.5 末尾に "Phase 7.2 interim
  invariant" として記述)、Phase 7.3 着手時に Doc 30 §2.6 write barrier
  実装で解除
