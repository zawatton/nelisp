** 目的

anvil.el (56 モジュール / 約 42,000 LOC) の *spine 部分を NeLisp 上に
port* し、Claude Code から実際に MCP 接続して tool call できる
"anvil-on-nelisp" の最小構成を立ち上げる。これは Phase 0 v1.0 完遂
条件 3 "AI-critical path が NeLisp 上で動作" の *実運用検証路線*
(memory user 方針「最低限動く = anvil.el で実用運用」) の到達点
であり、v0.4.0 tag 発行 + GitHub push + 告知の前提条件 (memory
=project_nelisp.md= 参照)。

anvil 全 port は本 Phase scope 外 (Phase 5-F+ で逐次継続)。本 Phase
の完走条件は *spine 3-5 module が src/ に定着し、read-only 系 MCP
tool 5-10 本が Claude Code 実接続で動く* こと。

- *write-operation tool (file-replace / file-batch / Edit 相当)* は
  scope 外。read-only 系で MCP protocol 動作を実証、destructive
  operation の port は Phase 5-F 以降で risk 管理しつつ

  (terminal CLI) 1 経路で動作実証、他 client は後続
- *anvil-orchestrator / -memory / -worker-ui 等 heavy module の
  port* は scope 外
- *Windows 環境での動作検証* は scope 外。Linux (Debian) 1 環境で
  完走、Windows は Phase 5-F+ で anvil-pty-broker / conpty 罠を
  踏みながら対応

  bootstrap 方式を維持

** §2.1 Port scope

*Q*: 本 Phase で NeLisp 上に port する範囲をどこまで広げるか。

選択肢:
A. anvil 全 56 module / ~42 kLOC を port (multi-phase 化前提)
B. *spine 3-5 module + 初期 tool 5-10 本* (本 Phase 完走可能な
   最小 "anvil-on-nelisp" 動作セット)
C. spine のみ (tool port は 0、merge gate は server 起動のみ)

*推奨: B (spine + tool 5-10 本)*。A は scope 爆発で Phase 境界
無意味、C は "実運用検証の起点" として実感薄く user 方針
「最低限動く = 実用運用」を満たせない。B で Claude Code 実接続
demo が成立し、残モジュールは Phase 5-F+ で順次積む。

** §2.2 MCP framing

*Q*: MCP protocol の wire 層をどう実装するか。

選択肢:
A. anvil-server.el (1894 LOC) を NeLisp 互換で port、既存 register-
   tools / JSON 層を全て引き込む
B. *NeLisp 側で JSON-RPC 2.0 最小実装 re-impl* (Phase 5-C.4
   =examples/nelisp-daemon/main.el= の実装を母体に拡張)
C. stdio bridge — host anvil が MCP 受信 → tool exec のみ NeLisp
   に転送、JSON framing は host 担当

*推奨: B (NeLisp 側で最小 re-impl、Phase 5-C.4 demo 拡張)*。A は
anvil-server の Elisp イディオム (json.el / process filter の細かい
書き方) を NeLisp に引き込むと scope 爆発 + NeLisp primitive 穴
発見の rabbit hole。B なら Phase 5-C.4 で既に JSON-RPC 2.0 +
=tools/list= / =tools/call= が動作しており、framing (line-delimited)
+ error code 拡充 + notification 無視の 3 点で実用線。C は "NeLisp
上で動く" という Phase 目的を満たさない (host anvil 依存)。

** §2.3 Host / NeLisp 分担

*Q*: host Emacs と NeLisp の責務境界。

選択肢:
A. 全て NeLisp (cold-boot 前提、Rhodes genesis 完了後)
B. *起動・daemon は host、tool exec は NeLisp* (host Emacs 上で
   =nelisp-eval= で走らせる bootstrap 方式を維持)
C. 2 process (host-bridge daemon + nelisp-server 子 process、stdio
   forwarding)

*推奨: B (host 起動、NeLisp exec)*。NeLisp は Rhodes 5-stage genesis
が Phase 6+ のため真 cold-boot 未到達、A は原理的に不可。C は
process 分離の複雑度 + serialization cost が Phase 5-E 見た目と
釣り合わず、host 単 process 内で =nelisp-eval= で充分。B で "NeLisp
の tool dispatcher / handler / registry" が実体としてのメイン、
host は起動 shim に留まる。

** §3.2 =nelisp-deftool= macro + dispatcher (3-4 日)

- =src/nelisp-server.el= に =nelisp-deftool= macro を配置:
  #+begin_src elisp
  (nelisp-deftool file-read
    :description "Read a file; return its content as a string."
    :input-schema ((path :type "string" :description "Absolute path"))
    :handler (lambda (params)
               (let ((path (alist-get 'path params)))
                 ;; ... 実装 ...
                 result)))
  #+end_src
- macro expansion: =nelisp-server-tool-registry= への =push= + 既存
  name の duplicate-register 検出 + schema normalize

  inputSchema)= を列挙して alist → JSON
- =tools/call= dispatcher: name → registry lookup → handler 呼び出し、
  =params= は nested plist → alist converter 経由、error は =:error=
  envelope で wrap
- =test/nelisp-deftool-test.el= +8 ERT:
  - macro expansion 2 (正常 / duplicate name error)
  - registry round trip 2 (deftool → list → lookup)
  - schema normalize 2 (plist → JSON-safe alist)

- 完了条件: 空 handler の dummy tool 2 本が macro 経由で
