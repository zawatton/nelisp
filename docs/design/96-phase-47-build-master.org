* §0 TL;DR

** §0.1 What Doc 96 ships

Doc 96 = =lisp/nelisp-phase47-build.el= — *thin chain integrator* that
binds the five Phase 47 components into a single, callable orchestrator
plus three pre-built demo functions that produce runnable Linux
binaries.

The orchestrator is intentionally /minimal/: ~180 elisp LOC, no new
machinery, no new file format, no new ABI.  It just wires existing
Doc 91-94 entry points into the canonical sequence and adds three
demos that exercise representative shapes (hello-world fixed-message,
exit-with-status configurable code, argc-printer kernel-stack
unpack).

** §0.2 The Phase 47 thesis Doc 96 demonstrates

#+begin_quote
  Pure elisp can emit a static-linked Linux ELF that the kernel exec's.
#+end_quote

Before Doc 96 the chain was provable component-by-component (Doc 91
ELF round-trip, Doc 92 instruction encoding, Doc 93 reloc patching,
Doc 94 crt0 byte tables, Doc 95 Sexp byte layout), but no single
function demonstrated that the components compose into a working
binary.  Doc 96 closes that gap with three demos, each runnable in
under one second from a fresh elisp invocation.

** §0.3 What Doc 96 explicitly does NOT do

- *Does not* compile from Sexp source.  Source = pre-built compile
  units (= functions returning =:text= / =:rodata= / =:symbols=
  plists).  A Sexp → asm compiler is the next layer up; deferring it
  keeps Doc 96 ≤200 LOC and lets the chain demo land before the
  Sexp-IR work.
- *Does not* register in nelisp-baker STDLIB_FILES.  Like Doc 91-95,
  Doc 96 is freestanding spike-chain code; baker integration is
  deferred to whoever ships Phase 47 production.
- *Does not* introduce a new file format or wire protocol.  Every
  byte that lands on disk is produced by =nelisp-elf-write-binary=,
  which means =readelf= / =objdump= / =strace= all work unchanged.

* §1 Chain architecture

#+begin_example
   user                  Doc 96                 Doc 91-95
   ─────                ────────                ────────────────────────
   build-       ───►    build-program  ───►    (a) asm-x86_64 emit
   hello-world          ASM-FN              ───►(b) crt0 byte tables
   build-                                   ───►(c) static-linker patch
   exit-with-                               ───►(d) elf-write-binary
   status STATUS
   build-argc-          (no compiler here:
   printer              ASM-FN is a thunk)
#+end_example

The orchestrator =nelisp-phase47-build-program= takes /one/ caller-
supplied function (=ASM-FN=) plus a file path + (optional) arch +
entry symbol.  ASM-FN returns a plist describing the compile units to
link; the orchestrator threads them through Doc 93 's
=nelisp-link-units-2pass= (when more than one unit is supplied), then
hands the resulting section bytes + symtab to Doc 91 's
=nelisp-elf-write-binary= for on-disk emission.

For the three demos, ASM-FN is internal — each demo is a single
pre-built function that already encapsulates the assembly + crt0 byte
emission + final layout.  This keeps the demos /maximally short/ (1-3
LOC per demo) while still exercising the full chain at run time.

| Doc 94 crt0 argc  | =nelisp-crt0-emit-hello-with-argc=| =build-argc-printer=               |
| Doc 94 crt0 bytes | =nelisp-crt0--encode-rel32=  | =build-exit-with-status= text emit |
| Doc 93 linker     | =nelisp-link-units-2pass=    | (= orchestrator multi-unit path)   |
| Doc 92 asm        | =nelisp-asm-x86_64-*=        | (= reserved for future ASM-FN)     |
| Doc 95 Sexp DSL   | none (= reserved for Doc 97) | future                             |

** §1.2 Why a single orchestrator + three demos (and not three
orchestrators)

Each demo could have been an independent =nelisp-crt0-emit-*= helper
(= Doc 94 already ships two of them).  Doc 96 's contribution is the
/uniform top-level API/: every demo is reachable through
=nelisp-phase47-build-*= so a future tutorial / smoke harness / CI
gate can iterate the chain without needing to know which Doc 94
helper covers which demo.  The orchestrator surface is the public
landing zone; the helpers stay private to their owning module.

* §2 Source → binary pipeline

Pipeline step-by-step for =nelisp-phase47-build-hello-world=:

1. /Caller/ invokes
   =(nelisp-phase47-build-hello-world "/tmp/hello")=.
2. /Doc 96 orchestrator/ dispatches on ARCH (default =x86_64=).
3. /Doc 96/ calls
   =nelisp-phase47--build-x86_64-hello-units=, which is currently a
   thin wrapper that delegates to =nelisp-crt0-emit-hello-world=.
4. /Doc 94 crt0/ computes the .text layout (= 37-byte =_start= stub
   that writes "hello\n" + =SYS_exit(0)=) and the .rodata payload
   (= six-byte =h e l l o \n=).
5. /Doc 94 crt0/ hands the assembled section plist to
   /Doc 91 ELF writer/ via =nelisp-elf-write-binary=.
6. /Doc 91/ emits the Ehdr (= 64 bytes, =EM_X86_64=, ET_EXEC,
   =e_entry= = =_start= VA) + a single PT_LOAD Phdr (= 56 bytes) +
   the .text bytes + .rodata bytes + section header table.
7. /Doc 91/ writes the bytes to the file with mode #o755 (= +x).
8. /Linux kernel/ exec's the binary; =_start= prints "hello\n" via
   =write(2)= and exits via =exit(2)= = process exits with status 0.

For =build-exit-with-status STATUS=, step 4 is replaced by a tiny
inline emitter that produces a 7-byte =_start= stub (= =mov edi,
STATUS; mov eax, 60; syscall=) — no .rodata, no message.

For =build-argc-printer=, step 4 delegates to
=nelisp-crt0-emit-hello-with-argc= which exercises the SysV AMD64
argv-stack unpack (= 108-byte =_start= routine that formats argc as
decimal ASCII).

* §3 API

** §3.1 =nelisp-phase47-build-program= (= main orchestrator)

#+begin_src elisp
(nelisp-phase47-build-program ASM-FN FILE-PATH
                              &key (arch 'x86_64) (entry-sym "_start"))
#+end_src

Arguments:
- =ASM-FN= : 0-arg function returning a plist with keys =:units=,
  =:rodata=, =:data=, =:bss-size=.  =:units= is either a list of
  Doc 93 compile-unit plists (= will be passed to
  =nelisp-link-units-2pass=) /or/ the symbol =:prebuilt=, in which
  case the function must additionally return =:text= bytes +
  =:symbols= list ready for =nelisp-elf-write-binary= directly (=
  short-circuit for the three demos).
- =FILE-PATH= : string, target output path.  Created with mode #o755.
- =:arch= : =x86_64= (default) or =aarch64=.  Threaded into Doc 91 's
  =:machine= key — the writer picks =EM_X86_64= or =EM_AARCH64=
  accordingly.
- =:entry-sym= : string, name of the entry symbol.  Default =_start=
  matches Doc 94 crt0 convention.

Returns: =FILE-PATH= on success; signals =nelisp-phase47-build-error=
on missing arch / unresolved entry symbol / writer failure.

** §3.2 =nelisp-phase47-build-hello-world FILE-PATH &key ARCH=

Builds a static-linked ELF that prints =hello\n= to stdout and exits
with status 0.  Equivalent to =nelisp-crt0-emit-hello-world= on
x86_64; signals on aarch64 (= aarch64 hello-world is owned by Doc 94
§94.b, separate helper).

** §3.3 =nelisp-phase47-build-exit-with-status FILE-PATH STATUS &key ARCH=

Builds a static-linked ELF that immediately exits with the supplied
=STATUS= byte (0..255).  Inline emits a 7-byte =_start= stub —
showcase of the minimum possible binary in the Phase 47 chain.

Used for smoke-style CI gates: when you want to verify the kernel
loader picks up a binary at all, this is the cheapest call.

** §3.4 =nelisp-phase47-build-argc-printer FILE-PATH &key ARCH=

Builds a static-linked ELF that prints =argc=<N>\n= where <N> is the
argv count.  Demonstrates the Doc 94 §94.c SysV AMD64 argv-stack
unpack via =nelisp-crt0-emit-hello-with-argc=.

* §4 Demo cases

** §4.1 hello-world

#+begin_src bash
emacs --batch -L lisp -l nelisp-phase47-build \
  --eval '(nelisp-phase47-build-hello-world "/tmp/h")'
/tmp/h    # → stdout: hello   ; EXIT=0
#+end_src

The smallest demo that exercises *all four* of the Phase 47 byte-
emitting components (= ELF header, single PT_LOAD, .text with raw
syscall, .rodata payload).  108-byte file on disk for a 37-byte
=_start= routine.

** §4.2 exit-with-status

#+begin_src bash
emacs --batch -L lisp -l nelisp-phase47-build \
  --eval '(nelisp-phase47-build-exit-with-status "/tmp/e" 42)'
/tmp/e; echo $?   # → 42
#+end_src

The smallest /possible/ user-visible demo — no stdout activity, just
a kernel exit syscall.  Useful for cross-architecture validation
because the entry stub is trivial enough that the byte-pattern can be
asserted at the ELF-writer level.

** §4.3 argc-printer

#+begin_src bash
emacs --batch -L lisp -l nelisp-phase47-build \
  --eval '(nelisp-phase47-build-argc-printer "/tmp/a")'
/tmp/a              # → argc=1
/tmp/a x y z        # → argc=4
#+end_src

Exercises the SysV AMD64 process-entry handoff (= kernel pushes
=argc=, =argv[]=, =NULL=, =envp[]=, =NULL=, auxv onto the stack
before transferring control to =_start=).  This is the most
mechanically complex demo and is the canonical regression test for
Doc 94 §94.c.

* §5 Testing

** §5.1 Chain-level integration tests
   (=test/nelisp-phase47-build-test.el=)

At least 8 ert-deftest cases:

1. =nelisp-phase47-build-hello-world-exec= — emit + exec, assert
   stdout = "hello\n", exit code = 0.
2. =nelisp-phase47-build-hello-world-readelf= — emit + invoke
   =readelf -h=, assert =ELF64=, =EXEC=, =X86-64= strings appear.
3. =nelisp-phase47-build-exit-with-status-zero= — STATUS=0 → exit 0.
4. =nelisp-phase47-build-exit-with-status-42= — STATUS=42 → exit 42.
5. =nelisp-phase47-build-exit-with-status-255= — STATUS=255 → exit
   255 (= max single-byte status).
6. =nelisp-phase47-build-argc-printer-no-args= — exec with 0 args
   beyond argv[0] → stdout = "argc=1\n".
7. =nelisp-phase47-build-argc-printer-three-args= — exec with three
   trailing args → stdout = "argc=4\n".
8. =nelisp-phase47-build-aarch64-magic= — emit hello-world with
   ARCH=='aarch64, parse the e_machine field in the resulting ELF
   bytes, assert =EM_AARCH64= = 183.

Each exec test gates on =nelisp-phase47-build-test--x86_64-host-p=,
which checks =system-type= ∈ {=gnu/linux=, =gnu=} and
=system-configuration= matches "x86_64\\|amd64".  On non-x86_64 hosts
the exec tests skip while the emission + byte tests still run.

** §5.2 What the tests do NOT cover

- Performance: emit time is < 50 ms per binary on a 2026-era laptop,
  not bounded by ert.  Doc 91 §91.d already gates 1 MB emit perf.
- Stripping / =strip= round-trip: covered by Doc 91 tests.
- arm64 exec: not feasible on x86_64 CI host without qemu-aarch64;
  tag-byte check is sufficient evidence for Doc 96 scope.

* §6 Known limitations

** §6.1 No Sexp source → asm compiler

The orchestrator accepts an ASM-FN — a thunk supplied by the caller.
For the three demos, the thunk is hard-coded (= delegates to Doc 94
crt0 helpers).  /No part of Doc 96 reads Sexp source and emits
assembly./  A Sexp source compiler would be a separate ~500-1000 LOC
component; deferring it keeps Doc 96 minimal and the chain demo
landable.

** §6.2 Hand-crafted assembly for the spike chain

The three demos use byte sequences /pre-encoded by hand/ from Intel
SDM Vol 2 + SysV AMD64 psABI (= the Doc 94 byte tables).  The Doc 92
macro assembler is /not yet wired/ into the orchestrator; switching
would require an ASM-FN that uses =nelisp-asm-x86_64-mov-imm32= +
=nelisp-asm-x86_64-syscall= + ... to emit the same bytes.  This is a
mechanical refactor and orthogonal to demonstrating the chain.

** §6.3 x86_64 Linux only for exec

Doc 94 §94.b ships aarch64 crt0 byte tables, but Doc 96 does not yet
provide a =build-hello-world= path through =nelisp-crt0-arm64=
because the arm64 module is owned by a sibling agent (= §92.b / §94.b
branches).  The orchestrator accepts =:arch 'aarch64=, validates the
machine code (= EM_AARCH64 tag in the output ELF) but signals on
exec-time codegen for aarch64 in the hello-world demo.  Adding the
aarch64 demo is a follow-up = one more demo function + matching
ert test.

** §6.4 Deferral to "Doc 97 Sexp source compiler"

The natural next step is a Doc 97 that:
1. Reads Sexp source via Doc 95 's =nelisp-sexp-dsl-read=.
2. Lowers each Sexp form to Doc 92 macro-assembler calls.
3. Returns the resulting compile-unit list to Doc 96's
   =nelisp-phase47-build-program=.

This would close the loop = pure elisp source on disk → pure elisp
emits a Linux ELF that runs that source.  Doc 96 ships the bottom
half of the loop; Doc 97 would ship the top half.
