** 目的

純 NeLisp で書かれた package を NeLisp 自身の loader で =require= →
NeLisp eval で動作させる。"NeLisp 単独で package 作成が可能" の
release bar の前半を達成する。

** 背景

Audit Doc 11 §3-5 + Check-1 §8 より:
- NeLisp は T1 基礎データ op を VM 再実装せず、host C に pragmatic
  delegation する設計が確立済 (SBCL-style)。T1 純 Lisp 化は release
  bar 外 (Phase 6+)
- T3 (buffer/window/frame/redisplay) 依存ゼロ = core 置換は "新設"、
  既存コード置き換えではない
- 現状 "NeLisp で package loop を閉じる" ための loader / load-path /
  require / macroexpand host 非依存化 が未完

Phase 5-A はこれらを埋める。新規 C 依存は追加しない。

** Non-goals

- Emacs core (buffer/redisplay/event-loop) substrate — *Phase 5-B*
- T1 の純 Lisp 再実装 — *Phase 6+ (optional)*
- package.el 互換 installer — release bar 外
- 複数 NeLisp instance の並列 package 読み込み — actor で実現可能だが
  Phase 5-A scope 外

*** §2.1 Loader API の surface

選択肢:
A. =nelisp-load-file PATH= 単一関数 + =nelisp-require FEATURE=
B. A + =nelisp-load STR= (in-memory source eval、test に便利)
C. B + =nelisp-load-from-buffer BUFFER= (現状 host =load= との parity)

*推奨: B*。reader + eval の glue を test しやすく、=nelisp-load-file=
はその薄 wrapper。=load-from-buffer= は現状不要 (buffer 抽象は T3、
Phase 5-A 範囲外)。

*** §2.2 load-path の設計

選択肢:
A. host の =load-path= を借用 (shortcut、NeLisp 化しない)
B. =nelisp-load-path= defcustom を新設、host =load-path= と完全に
   独立
C. =nelisp-load-path= 新設 + =nelisp-load-path-include-host= boolean
   defcustom で host を optional に含める

*推奨: C*。toy package は NeLisp 専用 dir で管理したいし、実用時に
host library (cl-lib 等) を見つけたい cross-over 需要もある。
defcustom 2 本で両立。

*** §2.3 Require の circular detection

選択肢:
A. set-based (loading 中の feature set、circular → error)
B. depth 制限 (max-depth defcustom、超過 → error)
C. A + B (両方)

*推奨: A*。host =require= と挙動 parity、set 管理は nelisp--
functions hash と同じ pattern で実装が清潔。depth limit は将来
Phase 5-B の event-loop と合わせて再検討。

*** §2.4 Macro expansion の host 非依存化戦略

選択肢:
A. =nelisp-macroexpand-all-1= を新規実装 (NeLisp 内で完結)
B. host =macroexpand= を nelisp--primitive-symbols に登録、
   =nelisp--functions= 経由で常に NeLisp-aware wrapper (user macro
   を先に見る) を dispatch させる
C. A + B hybrid (既知 macro は A、未知は B fallback)

*推奨: A*。bootstrap 後は NeLisp 単独 macro 展開、host =macroexpand=
への依存は eliminate したい。=nelisp--install-core-macros= で入れた
macro が NeLisp 表で visible なので、実装は macro lookup + substitute
の short path で済む。Phase 5-A.4 で検証と実装を合わせる。

*** §2.5 Extension の選択

選択肢:
A. =.el= 既定 (host Emacs と同じ、tooling の default を活用)
B. =.nel.el= 既定 (NeLisp 専用 file と明示、host =load= と衝突しない)
C. =.el= 既定、=.nel.el= 優先読み順位

*推奨: A* (2026-04-24 更新、旧推奨 B から変更)。§2.2 C で
=nelisp-load-path= を host =load-path= から独立化済のため、host =load=
が NeLisp 専用 dir を誤走査する事故は path 分離で構造的に防げる。
extension 差別化の二重防御は不要。=.el= 統一によるメリット:
- flycheck / lsp-mode / tree-sitter-elisp / xref / imenu が素で動く
- find-function / ert-run-test-interactively の elisp-mode hook 資産活用
- NeLisp 実装完了後に host parity 化する際の rename 作業ゼロ

将来 host parity で衝突が顕在化した場合は defcustom
=nelisp-file-name-suffix= で =.nel.el= fallback を後付け可能。

*** §2.6 Toy package 成功条件 (acceptance test)

選択肢:
A. =hello-world= 1 関数 + unit test (minimum)
B. A + =nelisp-utils= 的な多 function package (2 level require 検証)
C. B + 自己参照 (NeLisp-built helper が NeLisp-built helper を
   require する pattern 確認)

*推奨: C*。1 段 require は A/B で OK だが、実用 package は依存
ツリーを持つので、Phase 5-A.5 で 2 階建て require 動作を担保した方が
Phase 5-B 以降の dog-food 前提が崩れない。
