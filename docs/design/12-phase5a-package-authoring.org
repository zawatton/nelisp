* STATE: DRAFT 2026-04-24 (PDCA Plan, awaiting §2 decisions signoff)

** 目的

純 NeLisp で書かれた package を NeLisp 自身の loader で =require= →
NeLisp eval で動作させる。"NeLisp 単独で package 作成が可能" の
release bar の前半を達成する。

** 背景

Audit Doc 11 §3-5 + Check-1 §8 より:
- NeLisp は T1 基礎データ op を VM 再実装せず、host C に pragmatic
  delegation する設計が確立済 (SBCL-style)。T1 純 Lisp 化は release
  bar 外 (Phase 6+)
- T3 (buffer/window/frame/redisplay) 依存ゼロ = core 置換は "新設"、
  既存コード置き換えではない
- 現状 "NeLisp で package loop を閉じる" ための loader / load-path /
  require / macroexpand host 非依存化 が未完

Phase 5-A はこれらを埋める。新規 C 依存は追加しない。

** Non-goals

- Emacs core (buffer/redisplay/event-loop) substrate — *Phase 5-B*
- T1 の純 Lisp 再実装 — *Phase 6+ (optional)*
- package.el 互換 installer — release bar 外
- 複数 NeLisp instance の並列 package 読み込み — actor で実現可能だが
  Phase 5-A scope 外

* §1 Phase 4 pattern 踏襲

Phase 4 で確立した workflow をそのまま適用:

1. §2 の decisions を列挙し user signoff を得る (本 doc の現状)
2. signoff 後に §3 の decomposition を phase branch で順次 ship
3. 各 sub-phase で design doc の STATE を DRAFT → SHIPPED-YYYY-MM-DD 更新
4. 全 sub-phase 完了で §8 merge gate に到達、master merge

* §2 Locked decisions (signoff 必須)

以下 6 項目を user に承認してもらう。承認後に LOCKED stamp を追加。

*** §2.1 Loader API の surface

選択肢:
A. =nelisp-load-file PATH= 単一関数 + =nelisp-require FEATURE=
B. A + =nelisp-load STR= (in-memory source eval、test に便利)
C. B + =nelisp-load-from-buffer BUFFER= (現状 host =load= との parity)

*推奨: B*。reader + eval の glue を test しやすく、=nelisp-load-file=
はその薄 wrapper。=load-from-buffer= は現状不要 (buffer 抽象は T3、
Phase 5-A 範囲外)。

*** §2.2 load-path の設計

選択肢:
A. host の =load-path= を借用 (shortcut、NeLisp 化しない)
B. =nelisp-load-path= defcustom を新設、host =load-path= と完全に
   独立
C. =nelisp-load-path= 新設 + =nelisp-load-path-include-host= boolean
   defcustom で host を optional に含める

*推奨: C*。toy package は NeLisp 専用 dir で管理したいし、実用時に
host library (cl-lib 等) を見つけたい cross-over 需要もある。
defcustom 2 本で両立。

*** §2.3 Require の circular detection

選択肢:
A. set-based (loading 中の feature set、circular → error)
B. depth 制限 (max-depth defcustom、超過 → error)
C. A + B (両方)

*推奨: A*。host =require= と挙動 parity、set 管理は nelisp--
functions hash と同じ pattern で実装が清潔。depth limit は将来
Phase 5-B の event-loop と合わせて再検討。

*** §2.4 Macro expansion の host 非依存化戦略

選択肢:
A. =nelisp-macroexpand-all-1= を新規実装 (NeLisp 内で完結)
B. host =macroexpand= を nelisp--primitive-symbols に登録、
   =nelisp--functions= 経由で常に NeLisp-aware wrapper (user macro
   を先に見る) を dispatch させる
C. A + B hybrid (既知 macro は A、未知は B fallback)

*推奨: A*。bootstrap 後は NeLisp 単独 macro 展開、host =macroexpand=
への依存は eliminate したい。=nelisp--install-core-macros= で入れた
macro が NeLisp 表で visible なので、実装は macro lookup + substitute
の short path で済む。Phase 5-A.4 で検証と実装を合わせる。

*** §2.5 =.nel.el= extension vs =.el= 採用

選択肢:
A. =.el= 既定 (host Emacs と同じ、差別化しない)
B. =.nel.el= 既定 (NeLisp 専用 file と明示、host =load= と衝突しない)
C. =.el= 既定、=.nel.el= 優先読み順位

*推奨: B*。"NeLisp 単独 package" の identity を extension で表現、
host =load= が誤って読み込む事故を構造的に防ぐ。将来 host parity
が必要になれば defcustom で .el fallback 可。

*** §2.6 Toy package 成功条件 (acceptance test)

選択肢:
A. =hello-world= 1 関数 + unit test (minimum)
B. A + =nelisp-utils= 的な多 function package (2 level require 検証)
C. B + 自己参照 (NeLisp-built helper が NeLisp-built helper を
   require する pattern 確認)

*推奨: C*。1 段 require は A/B で OK だが、実用 package は依存
ツリーを持つので、Phase 5-A.5 で 2 階建て require 動作を担保した方が
Phase 5-B 以降の dog-food 前提が崩れない。

* §3 Decomposition

§2 signoff 後に着手。各 sub-phase は =feat/phase5a.N-<name>= branch で
ship、Phase 4 と同じく =--no-ff= merge で phase branch に集約、全完了

** §3.0 Pre-req cleanup (audit Check-1 §8 fix)

- =maphash= wrapper を =nelisp--install-primitives= に追加
- primitive 表未登録の 12 件を整備 (=plist-get/put/vector/make-vector/
  caddr/cdddr/ash/logand/logior/car-safe/float/message=)
- 既存 ERT を全 pass 維持
- 期待スコープ: 4-6 時間

** §3.1 =nelisp-load-file= (reader + eval glue)

- =(nelisp-load-file PATH)= → file を =nelisp-read-all= → 各 form を
  =nelisp-eval=
- =nelisp-load= in-memory variant (§2.1 B 採用時)
- エラー伝播: reader エラー / eval エラーを位置付きで signal
- 期待スコープ: 1 週
- ERT: read+eval loop / 部分失敗 / 再入の 3 本以上

** §3.2 =nelisp-load-path= defcustom + search

- =nelisp-load-path= / =nelisp-load-path-include-host= (§2.2 C)
- =nelisp-locate-file FEATURE= で file 解決 (=.nel.el= 優先)
- 期待スコープ: 3 日
- ERT: 多パス / not-found / host fallback

** §3.3 =nelisp-require= + circular detection

- =nelisp-require FEATURE &optional FILENAME NOERROR=
- loading set で循環検出 (§2.3 A)
- =nelisp-provide= は既存 =--builtin-provide= を再利用
- 期待スコープ: 3 日
- ERT: happy / circular error / already-provided / noerror

** §3.4 Macroexpand host 非依存化 (§2.4 A)

- =nelisp-macroexpand-1 FORM= を新規実装 (symbol → user-macro か
  core-macro を nelisp--functions 経由で lookup、展開結果返却)
- =nelisp-macroexpand-all= recursive 版
- =nelisp--eval-form= / =nelisp-bc--compile-form= の macro 展開
  step を host =macroexpand-1= から差し替え
- 動的 trace で host =macroexpand= が call されないこと確認
- 期待スコープ: 1 週
- ERT: backquote / nested macro / user-defined macro の 3 本以上

** §3.5 Toy package E2E (§2.6 C)

- =examples/hello/hello.nel.el= — 1 defun + provide
- =examples/hello/hello-utils.nel.el= — helper defun + provide、
  hello が require する
- =examples/hello-test.el= — 両 file を =nelisp-require= → 動作確認
- docs/design/12 の §7 に E2E 成功 log を転記
- 期待スコープ: 1 週 (docs 含む)

** §3.6 Merge gate (§2.8 equivalent)

全 sub-phase SHIPPED + make test-all green + toy package E2E 成功で
master merge 可。merge 前に以下確認:
- [ ] host =macroexpand= への runtime 依存が trace で確認されないこと
- [ ] =nelisp-require= が host =load-path= 非依存で動作すること
- [ ] ERT 追加 ~20 本、既存 520 / 520 + 新規全 pass

* §4 期待成果と next

*** Phase 5-A 完了後の状態
- 純 NeLisp で書いた package が NeLisp 単独 loader + evaluator で動作
- host Emacs C への runtime 依存は T1/T2 委譲のみ (T0 macro / T4 eval は
  NeLisp 内完結)
- release bar 前半達成: "NeLisp 単独でパッケージ作成が可能"

*** Phase 5-B への橋渡し
- Phase 5-A で整った loader + eval を dog-food target として、
  buffer/redisplay/event-loop substrate を Lisp で新設していく
- Phase 5-A の ERT パターンが Phase 5-B テストの雛形

*** Out of scope (本 doc の範囲外)
- Emacs core 置換 (buffer/redisplay/event-loop) → Doc 13
- package.el 互換 installer → 別課題
- binary package format / byte-compile-on-NeLisp → Phase 6+ 検討
