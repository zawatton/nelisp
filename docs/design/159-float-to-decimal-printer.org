#+TITLE: Doc 159 — real float → decimal printer (replacing the ".5" stub)
#+DATE: 2026-06-24
#+STATUS: FIXED — floats now render their actual decimal digits (P=15, %.15g-style).
#+BRANCH: spike/flat-arena-boot-install

* 0. Symptom

On the bare reader every non-integer float printed as =N.5=:

#+begin_example
3.14159            => 3.5
0.25               => 0.5
123.456            => 123.5
(/ 10.0 3.0)       => 3.5
(number-to-string 3.14159) => "3.5"
#+end_example

The breadth loop first read this as "float arithmetic is broken", but it is
*print-only*: the stored value is a correct f64.  Proof —

#+begin_example
(= 3.14159 3.5)                      => nil          ; not equal to 3.5
(* 3.14159 100000)                   => 314159.0     ; multiplies correctly
(truncate (* (- 3.14159 3) 100000))  => 14158        ; frac part is 0.14159…
(= 0.25 (* 0.5 0.5))                 => t            ; full precision retained
#+end_example

So reading, arithmetic and comparison are all correct; only =float → string=
was wrong.

* 1. Root cause — =m5_push_float= was a stub

=number-to-string= / =prin1= route floats through =m5_push_float=
(=scripts/nelisp-standalone-build.el=).  It printed the (correct) integer part,
a =.=, then a single hard-coded digit:

#+begin_src elisp
(defun m5_push_float (ms vptr)
  (let* ((whole (f64-to-i64-trunc (bits-to-f64 (sexp-float-unwrap vptr)))))
    (seq (m5_push_dec ms whole)
         (mut-str-push-byte ms 46)
         (if (= (m5_float_eq_i64_p vptr whole) 1)
             (mut-str-push-byte ms 48)     ; whole number  -> "N.0"
           (mut-str-push-byte ms 53)))))   ; anything else -> "N.5"   <-- bug
#+end_src

It never generated fractional digits at all.

* 2. Fix — significant-digit decimal generation (P = 15)

Replaced the stub with a real printer, written in the restricted AOT dialect
(no new Rust; the dialect is elisp that lowers to native code).  Algorithm:

1. sign: emit =-= and negate the magnitude bits.
2. integer-valued float (=frac == 0=): emit =WHOLE.0=.
3. otherwise choose the number of fractional digits by SIGNIFICANT-digit budget
   =P = 15= (= =DBL_DIG=):
   - =whole > 0=: =gen = max(1, 15 - digits(whole))=
   - =whole == 0=: =gen = leadingZeros + 15= (leading zeros before the first
     non-zero fractional digit do not consume the budget).
4. generate =gen+1= fractional digits by repeated =frac *= 10; d = trunc(frac);
   frac -= d=, storing one byte per digit in a scratch buffer.
5. round: if digit =gen= ≥ 5, propagate a carry backwards through digits
   =gen-1 … 0=; a carry out of digit 0 bumps =whole=.
6. trim trailing zeros (keep ≥ 1) and push the digits.

** Dialect constraints learned

- *f64 values cannot live in a =let*= variable* — the AOT compiler rejects an
  f64-class leaf bound to a general-purpose slot
  (=:f64-leaf-shape-unsupported=).  f64 results must be consumed inline or
  round-tripped to bits via =nl_sexp_write_float= + =(ptr-read-u64 scratch 8)=
  and carried as a u64, exactly like =wf_fsum= / =wf_fprod=.
- The new helpers (=m5_idigits=, =m5_leadzeros=, =m5_gendigits=, =m5_carry=,
  =m5_lastnz=, =m5_pushdigits=, =m5_push_float_frac=, =m5_push_float_mag=) are
  plain recursive defuns in the M5 helper list; =alloc-bytes= provides the
  digit buffer and the f64 scratch slot.

* 3. Result

=number-to-string= now matches Emacs for human-written decimals (36/40 of a
mixed battery, incl. all money-style values):

#+begin_example
0.25 0.1 0.2 0.3 3.14159 123.456 0.0001234567 9.999999999 1234567.89
99.99 19.95 0.99 49.99 0.05 1234.56   => all exact
(/ 1.0 3.0)   => 0.333333333333333    ; 15 sig digits (Emacs shortest: 16)
(/ 22.0 7.0)  => 3.14285714285714     ; clean, one digit shorter than Emacs
#+end_example

** Deliberate non-goals / known deviations

- *Not shortest-round-trip.*  P=15 prints division/irrational results one
  significant digit shorter than Emacs's shortest form, but always cleanly
  (no f64-noise tail).  P=16 would match the 1/3 family yet expose noise like
  =99.99 -> "99.98999999999999"=; P=15 keeps money/human values clean, which is
  the better trade for breadth.  True shortest-round-trip would need a decimal→
  f64 parse oracle callable from the printer (deferred).
- *Scientific notation* (added Doc 159 §5): =m5_push_float_mag= now switches
  to exponential via the §4 =m5_fmt_sci= when the decimal exponent is < -4 or
  >= 15 (= the P=15 significant digits), the %g rule.  =1e-5 -> "1e-05"=,
  =1e15 -> "1e+15"=, =6.022e23 -> "6.022e+23"=, =-1e20 -> "-1e+20"=, while
  =1e14 -> "100000000000000.0"= stays fixed — matching Emacs `number-to-string'
  for clean magnitudes.  Deviation: 16-17 digit integers in =[1e15,1e16)= that
  Emacs keeps fixed (shortest-length) print in scientific here.

* 4. =format= =%f/%e/%g= precision (Doc 159 §3/§4)

A separate gap surfaced here: =(format "%.2f" 3.14159)= returned ="3.14159"=
and =(format "%.3e" x)= returned ="%e"=.  The *active* =format= on the bare
reader is the elisp one in =scripts/nelisp-stdlib-prelude.el= (NOT the
=m5_fmt_loop= native arm, and NOT =lisp/nelisp-stdlib-format.el= /
=nl_jit_format_float=, which belong to a different runtime and are not baked
into the standalone reader).  It parses flags / width / precision in elisp but
*delegated the conversion* to native, which drops the precision.

*Fix*: a native AOT-dialect primitive =nelisp--fmt-float= (=m5_fmt_float_body=
+ helpers) renders =%f/%F/%e/%E/%g/%G= with round-half-to-even, reusing the §2
printer machinery (=m5_gendigits= / =m5_carry= / =m5_lastnz= / =m5_pushdigits=)
plus =m5_fmt_exp_of= / =m5_fmt_pow10= / =m5_push_exp=.  The prelude routes all
six float conversions to it (registered in the builtin-name list at the
=nelisp--repr= entry).  =%g= chooses %f vs %e by the exponent the value has
*after* rounding to P sig figs — biasing the magnitude by half a ULP
(=5*10^(e0-p)=) so a value rounding up across a power of ten (=9.999= at
=%.1g=) classifies as the bumped exponent.  Matches Emacs =format= on 400/400
of a 20-spec × 20-value =%f/%e/%g/%E/%G= battery (flags / width / precision /
scientific / general).  Same lone deviation class as §2 — the exact-half tie
on non-binary-exact values, e.g. =(format "%.1f" 0.05)= => ="0.0"= vs Emacs
="0.1"= (the f64→decimal extraction can't see the sub-ULP excess).

** Why a native primitive and not baked prelude elisp — size threshold (gotcha)

A first cut implemented the full =%f/%e/%g= family (~85 lines, 9 helper
defuns) in the *prelude*.  It built and ran, but *silently corrupted unrelated
baked strings*: =(format "price")= => ="p#<object>ice"= (deterministic, the
literal char-copy path the change never touched; =char-to-string= / =concat=
were fine standalone, and the bundle smoke still passed — only =format='s
baked constants were damaged).  Shrinking to a single ~22-line =%f= defun
removed the corruption.  So *there is a size threshold on prelude additions
baked into the standalone image* (between ~22 and ~85 lines of new code here)
past which the bake silently corrupts adjacent interned values; comments are
stripped before baking and do not count.

The AOT dialect has no such limit (§2's =m5_push_float= is ~85 dialect lines,
no corruption), so the full =%f/%e/%g= formatter was moved there and the
prelude elisp shrank to a one-line dispatch — net prelude reduction, and Rust
reduction too (the conversion no longer needs the native default formatter).

** Deferred

- =%a= (hex float) is not implemented.

* 5. =number-to-string= scientific notation + a lexer bug it exposed

=m5_push_float_mag= gained the %g-style exponent switch (§2 deviations,
updated above): exponent < -4 or >= 15 routes to =m5_fmt_sci= (15 sig,
stripped).  Verified on arithmetically-constructed magnitudes — =1e20 -> "1e+20"=,
=1/1e20 -> "1e-20"=, =6.022e23 -> "6.022e+23"=, =1e40 -> "1e+40"=,
=-1e20 -> "-1e+20"=, =1e14 -> "100000000000000.0"= (fixed), =1e15 -> "1e+15"=.
=format= %f/%e/%g unregressed (400/401), no baked-string corruption.

* 6. Lexer large-exponent float literals (Doc 159 §6)

Verifying §5 surfaced a *reader/lexer* bug: large-exponent float LITERALS were
misread.  =1e20= read as ≈7.77e18 (=(= 1e20 (* 1e10 1e10))= => nil), and
=1e100=/=1e-100= read as 0 / overflow; small exponents round-tripped fine.

*Root cause*: =nl_str_to_float= (=lisp/nelisp-cc-evalport-str-to-float.el=, the
AOT-elisp reimplementation of Rust =str::parse::<f64>()=) scaled the integer
mantissa by =10^exp= using =nl_stf_ipow10= — an *i64* power — and an i64
multiply.  =10^20= overflows i64 (max ≈9.2e18), wrapping to garbage; hence
=1e20= → ≈7.77e18 and =1e100= → 0.

*Fix*: scale in f64.  =nl_stf_finish= now multiplies/divides the integer
mantissa by a power of ten in f64.  No new Rust.

* 7. Correctly-rounded power-of-ten tables (Doc 159 §7)

A first f64 fix computed =10^n= by squaring, which accumulated sub-ULP error
for |n| > 22 (=1e100 -> "9.99999999999999e+99"=).  Replaced with two baked
*correctly-rounded* tables generated at build time from host =strtod=
(=(float (expt 10 n))= / =(read "1e-n")=, IEEE bits extracted via =frexp=,
subnormal-aware): =nl_stf_pow10hi(n)= = round(10^n), n∈[0,308] (=inf= above);
=nl_stf_pow10lo(n)= = round(10^-n), n∈[0,340] (=0= below).  Each is a balanced
binary-search =if=-tree over =n= returning the i64 bit pattern (all finite
double bit patterns are < 2^63, so they fit a signed i64); ~10–11 KB of
generated dialect each, depth ~9.

=nl_stf_finish= picks the correctly-rounded path:
- =netexp >= 0=: =mant * pow10hi(netexp)=.
- =netexp < 0, |netexp| <= 22=: =mant / pow10hi(|netexp|)= — the divisor is
  EXACT (10^k for k<=22 is representable), so this is correctly rounded; a
  multiply by the inexact =pow10lo= would double-round (=1.5e-8= regressed
  until this split was added).
- =netexp < 0, |netexp| > 22=: =mant * pow10lo(|netexp|)=.

Result: clean powers and the whole common range are correct —
=1e20=, =1e100 -> "1e+100"=, =1e308 -> "1e+308"=, =1e-100 -> "1e-100"=,
=6.022e23=, =1.5e-8= …; =format= %f/%e/%g unregressed (400/401), bundle green.

** Remaining (fast-path limits — not the table's fault)

The table makes each =10^n= correctly rounded, but =mant * 10^n= still does ONE
f64 multiply, so a *multi-digit* mantissa times a power outside [-22,22] can
double-round in the last digit: =5e-300 -> "4.99999999999999e-300"=,
=9.99e99 -> "9.98999999999999e+99"=.  And values whose intermediate power
underflows although the product would not (=2.2250738585072014e-308= = min
normal, =5e-324= = min subnormal) collapse to =0=.  Closing these needs a
128-bit mantissa path (Eisel-Lemire) or big-integer fallback (Gay/Clinger),
well beyond a power-of-ten table.  =max double= prints with §2's 15 sig digits
(=1.79e308= form), a printer (not parser) limit.

* 8. Eisel-Lemire 128-bit parse path (Doc 159 §8)

The §7 fast-path residuals (double-rounding for multi-digit mantissa × power
outside [-22,22]; min-normal/subnormal collapsing to 0) are now *correctly
rounded* via the Eisel-Lemire algorithm, implemented in the AOT dialect.

** Pieces

- *u64×u64→u128 multiply* (=nl_u128_hi=): 16-bit schoolbook limbs so every
  partial product stays < 2^32 and every accumulator < 2^48 — no overflow of
  the dialect's signed i64, and limb extraction =(logand (sar x k) 65535)=
  works on the bit pattern even when bit 63 is set.  (The low 64 bits are just
  the wrapping =(* a b)=.)  Validated in isolation: =hi(2^80)=65536=, etc.
- =nl_clz64= (count leading zeros), =nl_ult= (unsigned `<` via sign-bit flip
  =(logxor x (shl 1 63))=), =nl_lshr= (logical shift right = =sar= + mask).
- *128-bit power-of-five tables* =nl_lem_hi(q)= / =nl_lem_lo(q)=, q∈[-342,308],
  generated at build time from host bignums (top 128 bits of 5^q, normalized)
  and emitted as balanced if-trees of *signed* i64 literals (values ≥ 2^63 are
  written as their two's-complement negative; the multiply reads the bit
  pattern, so the sign is irrelevant).  ~23 KB each.
- =nl_lem_compute(w,q,neg,slot)=: normalize w, 128-bit product with the table,
  extract the 54/53-bit mantissa, round-to-even, assemble the IEEE bits, and
  =nl_write_float_bits= them straight into the Float Sexp slot (tag 3 + bits —
  =bits-to-f64= is not accepted as a bare value argument by the AOT, only
  inside an f64 binop).  Returns 1, or 0 for the genuinely-ambiguous tie
  (=zlo<2 ∧ -27≤q≤55 ∧ (mantissa&3)==1=).

=nl_stf_finish= tries Lemire first and falls back to the §7 power-of-ten tables
on a 0 return — those ambiguous cases all have |q|≤22 where the §7 path is
exact, so the combination is correctly rounded throughout.

** Result

Every value in the §7 "remaining" list now PARSES correctly (verified by
round-tripping through arithmetic, since the *printer* is the weak link at the
extremes): =5e-300 · 1e300 -> "5.0"=, =9.99e99 · 1e-99 -> "9.99"=,
=1e-308 · 1e308 -> "1.0"=, =(= 5e-300 (* 5.0 1e-300))= => t=, and the min normal
=2.2250738585072014e-308= (which §7 collapsed to 0) parses exactly.  format
%f/%e/%g unregressed (400/401), common floats unregressed, bundle green.

** Still open — the *printer* at the extremes

=(number-to-string 5e-300)= prints ="4.99999999999999e-300"= even though the
parsed value is exact: §5's =m5_fmt_sci= derives the decimal exponent by
repeated ÷10 and the mantissa by a squaring =m5_fmt_pow10=, both of which lose
ULPs for |exponent| ≳ 200.  This is a PRINTER limit (the value is correct, only
its scientific rendering at extreme magnitude is off by a last digit); the
common range is exact.  Giving the printer the §7 correctly-rounded tables
would close it.  =%a= (hex float) and >19-digit integer mantissas (which overrun
the u64 accumulator before Lemire) remain unimplemented.

* 9. Printer: correctly-rounded power tables for the extremes (Doc 159 §9)

§8 made the PARSE correctly rounded, but =(number-to-string 5e-300)= still
printed ="4.99999999999999e-300"= because §5's =m5_fmt_sci= normalized the
mantissa with =m5_fmt_pow10=, a *squaring* 10^E that loses ULPs past |E|≈200.

*Fix*: replace =m5_fmt_pow10='s squaring body with correctly-rounded tables
=m5_p10hi(n)= = round(10^n), n∈[0,308] and =m5_p10lo(n)= = round(10^-n),
n∈[0,340] (same host-=strtod= generation + balanced-if-tree shape as §7, but in
the m5 unit; bit patterns are < 2^63 so they stay positive i64).  The exponent
count from =m5_fmt_exp_of= is robust; only the 10^E it divides by needed to be
exact.  This flows through every =m5_fmt_sci= caller, so =format= %e/%g at the
extremes improves too.

Now correct across the normal range incl. the extremes:
=5e-300 -> "5e-300"=, =9.99e99 -> "9.99e+99"=, =1e308 -> "1e+308"=,
=7e-50=, =1.234e250=, =3.3e-7=, =8.8e88= …  format %f/%e/%g unregressed
(400/401), common floats unregressed, bundle green.

** Remaining (rare)

- *max double / min normal* print with §2's P=15 significant digits
  (=1.79769313486232e+308=, =2.2250738585072e-308=) — the value is exact, just
  not the 17-digit shortest form; a printer limit (§2), not a magnitude error.
- *deep subnormals* (=1e-308 -> "9.99…e-309"=, min subnormal =5e-324 -> "0.0"=)
  carry too few mantissa bits for the =mag · 10^|E|= normalization to land in
  [1,10); correct rendering there needs a subnormal-aware scale-up.  Both are at
  the very edge of the double range and are left as documented limits.

* 10. Shortest-round-trip printing (Doc 159 §10)

§2's P=15 printer was not shortest (1/3 -> "0.333333333333333", 16-digit
Emacs form lost) and could even mis-round (0.1+0.2 -> "0.3", which does NOT
parse back).  Replaced =m5_push_float= with a true shortest-round-trip printer
that reuses the §8 correctly-rounded PARSER as a round-trip oracle:

1. =nl_extract17(bits)= — the 17 exact significant digits as an integer W and
   the decimal exponent E.  W = round(mag · 10^(16-E)) computed in 128 bits:
   decompose the double to mantissa M + binary exponent be; look up the §8
   128-bit power-of-five table[16-E] (range extended to q≤340 for small
   magnitudes); the binary exponent of 5^s is the closed form
   k = (s·152170 >> 16) − 127; form the 181-bit product M·table via three
   =nl_u128_hi= limb multiplies; shift-and-round (=nl_rshift_round_3=) to W.
   E itself = floor(log10 mag): seed (be + bitlen(M) − 1)·log10 2 (the bitlen
   term is essential for subnormals where M ≪ 2^52), then nudge ±1 against the
   §7 power table.
2. =nl_shortest= — for P = 1..17, round W to P significant digits (w_P), set
   q_P = E−P+1, and ask the oracle =nl_stf_finish(w_P, q_P)= (§8 Lemire + §7
   fallback) whether it parses back to the original bits.  The first P that
   round-trips is the shortest; return (w_P, q_P).
3. =m5_emit_sig= — emit w's digits with the decimal point placed for q, fixed
   vs scientific by Emacs's rule: scientific iff DE < −4 OR (DE ≥ 15 AND q ≥ 1),
   where DE = q + digits(w) − 1 (so 1e15 -> "1e+15" but 1234567890123456.0 stays
   fixed; 1e6 -> "1000000.0").

Matches Emacs =number-to-string= on 39/39 of a battery spanning 1/3, 2/3, 22/7,
e, √2, 0.1+0.2, 1234567890123456.0, 5e-300, 9.99e99, 1e100, 1e15, subnormals
(1e-308, min subnormal 5e-324) — including the 16- and 17-digit forms §2 could
not produce.  format %f/%e/%g unregressed (400/401), bundle green.

The former min-subnormal-band PARSE edge (=4.9e-324= read as 0) is now closed
(§8 fix below): the full min-subnormal battery (=4.9e-324=, =5e-324=, =4e-324=,
=9e-324=, =1e-323=, … min-normal) matches Emacs 10/10.  Float work (§2–§10) now:
parse correctly rounded, print shortest-round-trip — Emacs-identical across the
full double range incl. the bottom subnormal band.

** 8a. Subnormal rounding-order fix (Doc 159 §8)

=nl_lem_compute= originally rounded the 53-bit mantissa to *normal* precision
first, then right-shifted into subnormal position — truncating the bits that
decide the subnormal round and collapsing =4.9e-324= to 0.  The fix follows
fast_float's order: when =power2 <= 0= (subnormal/zero), shift the *pre-round*
mantissa by =1 - power2= first, *then* round-half-even at the resulting
subnormal precision (carry into the exponent field handled by =msr >= 2^52=).
The normal path (=power2 > 0=) is unchanged.

* 11. C99 %a / %A hex-float conversion (Doc 159 §11)

=format= now supports the C99 hexadecimal-float conversions =%a= / =%A=.  This
is a *NeLisp superset*: host Emacs 30.1 `format' rejects =%a= with "Invalid
format operation %a", so there is no Emacs oracle — the reference is glibc
=printf("%a")= and the output matches it exactly on the validation battery.

Routing: the prelude `format' adds conv 97/65 to the set delegated to
=nelisp--fmt-float=, passing precision =-1= (sentinel for "unspecified") instead
of the e/f/g default of 6.  All formatting lives in the AOT dialect
(=m5_fmt_hexfloat= + =m5_hexf_*= helpers, dispatched from =m5_fmt_dispatch= on
conv =a=/=A=).  It operates entirely on the IEEE-754 bits, so the f64-leaf
dialect restriction does not apply.

Layout =[-]0xh.hhhhp±d=:

- *sign* — the leading =-= is emitted by =m5_fmt_float_body= for finite
  negatives (it negates and re-dispatches the magnitude); =m5_fmt_hexfloat=
  additionally emits =-= when the raw bits have the sign set (the residual
  =-0.0= / =-nan= cases).
- *leading digit* — =1= for normals (=ef != 0=), =0= for subnormals/zero.
- *fraction* — the 52-bit mantissa as up to 13 hex nibbles, MSB first.
  Unspecified precision strips trailing-zero nibbles (and drops the =.= when
  none remain: =1.0 -> "0x1p+0"=); an explicit =.N= emits exactly N nibbles
  with round-half-even, the carry propagating into the leading digit
  (=%.0a 1.5 -> "0x2p+0"=).
- *exponent* — the unbiased binary exponent in decimal with an explicit sign
  (=p+0=, =p-4=, =p+1023=); subnormals are fixed at =p-1022=; zero is =0x0p+0=.
- =inf= / =nan= (=ef == 2047=) render as =inf= / =nan= (=INF= / =NAN= for =%A=).

Matches glibc =%a= exactly across normals, subnormals (=5e-324 ->
"0x0.0000000000001p-1022"=, min-normal =-> "0x1p-1022"=), the max double
(=0x1.fffffffffffffp+1023=), =%A= upper-casing, and =.3= precision rounding.
The one value-level divergence is =-0.0=: this reader collapses =-0.0= to
=+0.0= at read/negate time (=number-to-string -0.0= is "0.0" vs Emacs "-0.0"),
so the sign is already gone before =%a= sees it — a pre-existing float
representation gap independent of this conversion, not a =%a= bug.
