* §0 要旨

Stage A Phase 5-F.3 で =org-read-outline-tree= 単発 tool を NeLisp 側
=nelisp-tools.el= に port 済 (file 1 個ずつの構造抽出)。本 doc では
*複数 org file を SQLite で index 化* し、cross-file 検索 / headline
high-speed lookup を可能にする =nelisp-org-index= module を新設する。

port 元: =anvil-org-index.el= (Doc 02 SHIPPED、1419 LOC、user 実機で
1500 files / 71k headlines を 2.7s rebuild + 177x faster than
file-scan の実績)。session-B が完了させた =nelisp-defs-index= (Phase
6.5.1 9746ed6) と双子の構造 = SQLite-backed index + rebuild + search、
schema が 4 table で walker pattern も類似。

** なぜ port するか
- Stage D ユーザ (非 Emacs user 含む) で *業務 / 個人記録の org 利用
  者* に対する価値を提供。Claude Code から「この headline を探せ」
  「この tag が付いた entry list 化」が SQLite-backed で即時返却
- =nelisp-state= (Phase 5-F.1.1) + Phase 5-F.3 の org parser を再利用、
  実装は schema + scanner + search query 層のみ
- session-B の Phase 6.5 と並走: 両者とも「rebuild + search」MVP →
  「watcher」拡張 → 「deep walker」進化 の 3 phase 分割で同形

** 非目標
- org ファイル *編集* (read-only 運用、edit は anvil-org-index にも
  ない)
- =org-read-by-id= / =-read-headline= 等の単発 reader は Phase 5-F.3
  の =org-read-outline-tree= で既に対応済、本 doc では DEFER

* §6 想定リスク

** 6.1 org parser の Phase 5-F.3 surface
=nelisp-tools--org-flat-headlines= は private function (=--= prefix)。
本 phase で公開化する必要あり。対策: 名前を =nelisp-org-flat-headlines=
に rename し、session-B の Phase 6.5.2/3 が触らないファイル
(=nelisp-tools.el= 内 helper section) を session-B merge 後に touch。

** 6.2 SQLite WAL モード + 複数 process
Phase 6.5.1 と同じ DB ファイルではなく別 file (=nelisp-org-index.db=
vs =nelisp-state.db=) なので process 並行アクセスの問題は別個。
nelisp-state の transaction macro を再利用すれば OK。

** 6.3 file scanner の throughput
1500 files / 71k headlines を 2.7s で anvil-org-index は処理する。
NeLisp 側は host Elisp なので速度は同等以上のはず。MVP は
=insert-file-contents= + regex scanner で十分、=org-element= 不要。

** 6.4 org-id (=:ID:= property) lookup
anvil-org-index は =org-id= による lookup を提供するが、本 MVP では
DEFER。Phase 6.6.3 で再評価。
