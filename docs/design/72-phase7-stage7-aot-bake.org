** §1.1 Stage 7.6 が残した形

Stage 7.6.a/b SHIPPED (commit f15b16b) で post-bootstrap reader callsite
は **すべて** elisp reader (=nelisp--read-from-string-impl= /
=nelisp--read-all-from-string-impl=) 経由になった.  しかし *bootstrap
path* は依然 Rust reader が使われる:

#+begin_src rust
// build-tool/src/eval/env.rs:256-266
for (name, src) in STDLIB_SOURCES {
    let forms = match reader::read_all(src) {       // ← Rust reader
        Ok(forms) => forms,
        Err(e) => panic!("{} bootstrap failed: {}", name, e),
    };
    for form in &forms {
        if let Err(e) = super::eval(form, &mut env) { // ← Rust eval
            panic!("{} bootstrap failed: {}", name, e);
        }
    }
}
#+end_src

これが解消できないと:

- =reader/mod.rs= (292 LOC) を削除できない (bootstrap が依存)
- =reader/lexer.rs= (885 LOC) / =reader/parser.rs= (485 LOC) も同様
- =reader/sexp.rs= (671 LOC) は runtime data structure なので別問題

つまり Stage 7.6 §8 で articulated されていた "Rust reader 完全削除"
は bootstrap reader 依存の解消が前提.

** §1.2 既存の image 形式

=image.rs= (545 LOC) が既に Sexp tree の binary serialization 形式を
持っている:

| component                | LOC             | 役割                                       |
|--------------------------+-----------------+--------------------------------------------|
| =IMAGE_MAGIC=            | 1               | =NELIMG\0\x02= (8 bytes)                   |
| =IMAGE_ABI_VERSION=      | 1               | =u32= = 2                                  |
| TAG_NIL/T/INT/FLOAT/.../BOOL_VECTOR | 10  | Sexp variant 1 byte tag                    |
| =compile_elisp_to_image= | 4               | source → reader::read_all → encode_image   |
| =encode_image=           | ~10             | =[Sexp]= → bytes                           |
| =decode_image=           | ~20             | bytes → =Vec<Sexp>= (reader 不要)           |
| =eval_image=             | 3               | bytes → decode → =eval_forms=              |

*key 性質*:
- =decode_image= は **Rust reader を呼ばない** (=encode 時の tag 列を
  直接 Sexp::Cons / Sexp::Symbol 等に逆構築)
- 既に Phase 8.x 「smoke path」 として =bootstrap.el -> image -> eval=
  が動作確認済 (= image.rs の冒頭 docstring)

→ *bootstrap STDLIB_SOURCES を image にすれば Rust reader を起動時に
呼ばずに済む*.

* §2 Stage 7.7 sub-stages

** §2.1 Stage 7.7.a — image baker tooling

build 時に bundled elisp を image bytes に変換する pipeline を追加.

実装案 (= 実装難度の低い順):

*** Option A: separate baker binary + build.rs

#+begin_example
[Crate structure]
build-tool/
  src/
    bin/
      nelisp.rs               (既存: REPL + driver)
      nelisp-baker.rs         (新規: STDLIB → image bytes)
    eval/...
    reader/...
    image.rs
  build.rs                    (新規: cargo build 前に baker 実行)
#+end_example

build.rs:
1. cargo の =OUT_DIR= に =baked-stdlib/{name}.image= を書き出し対象として
2. =cargo:rerun-if-changed= で =lisp/*.el= 変更を hook
3. =cargo run --bin nelisp-baker -- --out OUT_DIR/baked-stdlib= で baker を呼ぶ
4. baker は STDLIB_SOURCES と同じ list を内部に持ち、各 file を
   =compile_elisp_to_image= で bake、=OUT_DIR/baked-stdlib/{name}.image= に
   書き出す

困難: build.rs が *自分の crate* (build-tool) を =cargo run= で呼ぶのは
循環依存になる.  解消法:
- (a) baker を独立 workspace member にする (= =image-baker/=)
  → reader+image+sexp の dependency 整理が必要
- (b) cargo の =[build-dependencies]= に =build-tool= を入れる
  → workspace の build graph で cycle を作るので不可
- (c) build.rs に reader+image のロジックを duplicate
  → メンテ二重化、却下
- (d) image bake を offline で = =cargo xtask bake-images= の
  別 task として手動実行、=lisp/*.image= を git track
  → 開発者が elisp 編集後に毎回手動で再 bake、UX 悪い

→ *Option A は (a) workspace 分離が現実的*.

*** Option B: pre-built image を git track

baker 実行責任を CI / 開発者の hook で外出しし、=lisp/*.el= とペアで
=lisp/*.image= を git track する:

| pro                                         | con                                      |
|---------------------------------------------+------------------------------------------|
| build.rs 不要、cargo build cycle 問題ゼロ      | =.image= は binary、git review 不能         |
| =include_bytes!= で zero-cost embed         | elisp 編集 → image 再生成漏れの risk       |
| 複雑な workspace 操作不要                     | image format 変更時に大量 churn          |

→ *Option B は image format が安定している前提なら現実的*. しかし
ABI_VERSION bump (= 例: tag 追加) が起きるたび全 .image 再生成.

*** Option C: 起動時 read を keep + Stage 7.7 を Phase 8 に格上げ

Stage 7.7 を skip、Stage 7.6 までで自然な区切りとする.

| pro                              | con                              |
|----------------------------------+----------------------------------|
| 実装コストゼロ                    | Rust reader/eval 完全削除の道筋が消える |
| 現状で動いているのを触らない      | Doc 63 §2.6 / §3.3 の articulated 計画が宙に浮く |

→ Option C は本 Doc の「設計しない」結論. user judgment 待ち.

*** 推奨

*Option A (workspace 分離)* が architectural には最も clean.  しかし
実装コスト (= reader+image+sexp の sub-crate 化) が大きいため、
*Option B (git-tracked .image)* を本 Doc では default 推奨.

理由:
- image ABI_VERSION は v2 で stabilize (=v2 拡張は Stage 7.7 中に必要に
  なれば bump するが、頻度低)
- =.image= 再生成は =make bake-images= target で一発、開発者の hook で
  自動化可能
- workspace 分離は将来 image format 拡張時に再検討

*** §2.3.1 Stage 7.7.c.1 — eval_str 族を elisp reader 経由に migrate

産品 =nelisp= binary の CLI entry (=eval EXPR= / =-l FILE= / =exec
FILE= / =-= stdin / =eval_str_all_at_path=) は内部で =reader::read_str=
/ =reader::read_all= を呼んでいた.  これを Stage 7.6.a の bi_read
pattern と同形で =nelisp--read-all-from-string-impl= 委譲に置換:

#+begin_src rust
fn read_all_via_elisp(input: &str, env: &mut Env) -> Result<Vec<Sexp>, EvalError> {
    let impl_fn = env.lookup_function("nelisp--read-all-from-string-impl")?;
    let arg = Sexp::Str(input.to_string());
    let mut list = apply_function(&impl_fn, &[arg], env)?;
    let mut out = Vec::new();
    loop {
        match list {
            Sexp::Nil => break,
            Sexp::Cons(car, cdr) => {
                out.push(car.borrow().clone());
                list = cdr.borrow().clone();
            }
            other => return Err(EvalError::Internal(...)),
        }
    }
    Ok(out)
}
#+end_src

これで産品 =nelisp eval EXPR= は **Rust reader を一切使わない** 経路に.

*** §2.3.2 Stage 7.7.c.2 — image::compile_elisp_to_image を feature gate

=image::compile_elisp_to_image= は baker tool が =reader::read_all= を
通る唯一の non-test, non-bridge callsite.  これを =cfg(feature =
"image-baker")= で gate:

#+begin_src rust
#[cfg(any(test, feature = "image-baker"))]
pub fn compile_elisp_to_image(source: &str) -> Result<Vec<u8>, ImageError> {
    let forms = reader::read_all(source)?;
    encode_image(&forms)
}
#+end_src

廃止 (= =nelisp-baker= bin が dev tooling として既存、機能重複).

["image-baker"]= を追加、=make bake-images= も =--features
image-baker= に更新.

*** §2.3.3 副産物 — elisp reader bug fix (=nelisp--read-tok-try-float=)

* §4 Risks / Mitigation

** §4.1 .image format breakage

=Sexp= variant 追加 (= 新 tag) が起きるたび ABI_VERSION bump、stale
=.image= は =UnsupportedVersion= で fail.

mitigation:
- =make bake-images= target を整備、=make compile= に dep として追加
- CI で =.el → .image= の rebuild を gate

** §4.2 baker 自体が壊れる

baker は Rust reader を使うので、reader bug が image にも propagate.
runtime と baker で同じ reader を使う限り対称性は保たれるが、image
の中で reader bug が *cached* される形になる.

mitigation:
- baker は cargo serial の前に常に再実行 (= =make bake-images= が
  cargo build より先)
- =eval_image= の round-trip ERT を追加 (= source → image → decode →
  eval した結果が source → reader → eval と同じ)

** §4.3 git binary churn

=lisp/*.image= を git track する場合、=.el= 編集の度に binary diff が
発生.  PR review で =.image= を見比べる意味は無い (= source-of-truth
は =.el=) ので、=.gitattributes= で =merge=binary= 扱いにする.

mitigation:
- =.gitattributes= に =*.image binary= を記述
- PR review では =.el= の diff を見て、=.image= は再生成済か checkbox
  で確認

** §4.4 build.rs cycle (Option A 採用時)

build.rs から =cargo run --bin nelisp-baker= を呼ぶと cargo の build
graph が cycle.  Option B (= git track) を default にすれば回避.

** §4.5 cargo unit test fixture の互換性

cargo unit test は Rust reader を直接呼ぶ fixture を多数持つ.  Stage
7.7.c で =reader= module を =cfg(test)= 又は =feature= gate にする際、
test build では variant が active なので互換性は保たれる.
