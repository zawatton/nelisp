* §0 要旨

standalone reader (=target/nelisp=、nelisp-cc 製 static ELF、Rust core 無し) は **現状 GC を一切走らせていない**
（実測 §1）。reuse / form-boundary collect が実際には発火せず、純 bump + chunk-growth で **never-free**。
結果:
1. *メモリ無限増*（実測: 2.4GB 確保で =COLL=0=、drop 参照も未回収、=used=2.44GB=）。
2. *anvil-pkg ERT suite (~13 ファイル + 182 テスト) を nemacs に load すると非決定的に abort*（Doc 142 gate5 系。
   heavy structured load 下で reader が落ちる。GC が無く arena が無秩序成長することが背景）。

本書は **sound な form-boundary tracing mark-sweep を「実際に稼働」させる**設計。既存資産の上に建てる:
- Doc 146: =rc は vestigial（=nl_dealloc_bytes= は no-op、実 freer は GC sweep のみ）→ *tracing が唯一の権威*=。
  pure-tracing は実質既得。残る健全性課題は **② root-coverage gap**（mark が live を取りこぼすと sweep が
  use-after-free）。
- =nelisp-gc-inner.el=: tri-color mark phase (=run-mark-phase=) + sweep phase (§7 per-family free-list rebuild) が
  *実装済*。だが reader の form-boundary には *配線/有効化されていない*（実測 COLL=0）。
- 旧 *epoch boundary-reclaim* は unsound として OFF（§2）。これは「sound tracing reclaim ができたら置換」と
  コメントに明記されている（=scripts/nelisp-standalone-build.el:929-936=）。

*評価 gate*: (a) =COLL>0= かつ heavy-alloc 後も =used= が bounded、(b) 既存 4438 PASS 維持、
(c) **anvil-pkg full suite が nemacs で完走（非決定 abort 消失）**。

* §1 現状（2026-06-14 実測）

** 1.1 GC は走っていない
=(nelisp--arena-stats)= tuple = =(base size bump used live next free COLL REUSE CHUNK-COUNT reserved used)=。
- boot 直後: =chunk=33 COLL=0 REUSE=0 used≈95MB=。
- =(make-string 100)= を ~18万 cons（途中 drop あり）後: =chunk=68 COLL=0 REUSE=0 used=2.44GB reserved=2.75GB=。
- = *COLL=0 のまま*。drop した参照も回収されず =used= が 2.44GB へ。**form-boundary collect / reuse は事実上 OFF**。

** 1.2 allocator 構成（=nelisp-standalone--arena-source=、nelisp-cc が native emit）
- chunked bump: =nl_alloc_bytes= → exact-fit free-list reuse（=nl_freelist_take=）→ 満杯なら =nl_chunk_alloc_new=。
- block header 8B: =BLOCK_TOTAL=（8-align）high bits + mark(0 live/1 marked/2 free) low 3 bits。object = hdr+8。
- =nl_dealloc_bytes= = *no-op*（rc-drop は何も解放しない）。実解放は GC sweep のみ。
- reuse-on は zero-fill 済（=nl_alloc_zero_fill=、Doc 146 §2.1 の reuse-correctness root-cause-fix）= dirty-memory
  問題は解決済。

** 1.3 既存 GC 機構（未配線）
- =nelisp-gc.el=: root-set 列挙のみ（globals 4 HT / vm-stack / aot-frame / aot-pinned-root）。Phase 3c.1。
- =nelisp-gc-inner.el=: tri-color mark（grey queue、=children-of-addr=、color）+ sweep（§7 free-list rebuild）+
  finalizer queue。Doc 30 (Phase 7.3)。
- =nelisp-gc-verify.el= / =-verify-hooks.el=: heap / mark-state-clean / free-list / card-table / poison-from-space
  検証。=NELISP_GC_VERIFY_DEBUG= で有効化。**だが現 standalone build には未収録**（nemacs で env を立てても無反応）。

* §2 なぜ旧 reclaim は unsound だったか（置換対象）

=scripts/nelisp-standalone-build.el:929-936=:
#+begin_quote
Doc 140 boundary reclaim DISABLED (was 1): the escape-epoch reset is unsound … it recycles per-form regions
holding values that escaped to globals via function-call returns / refcount-aliasing clones, a use-after-free.
Re-enable only with a sound tracing-GC reclaim.
#+end_quote

- 旧方式 = *epoch / region reset*: 「この form の epoch で確保した領域は form 終了で garbage」。
- *不健全な理由*: 値は form を *escape* する — 関数戻り値が global へ格納される、clone が aliasing する、等。
  epoch reset はこれらを reachability で追わず領域単位で捨てるため use-after-free。
- = *健全な基盤は唯一 reachability tracing*: 「root から到達可能なものは生存」。allocation epoch は無関係。

* §4 アルゴリズム — form-boundary stop-the-world tri-color mark-sweep

reader driver が *top-level form を 1 つ eval し終えた境界*で、watermark 閾値（前回 GC 以降の bump 量 > T）を
超えていれば collection を走らせる。

** 4.1 Root set（精密・完全）— §5 が要
1. *globals*: NeLisp の 4 global hash table（=nelisp-gc--globals-root=）。
2. *current VM stack(s)*: 実行中 VM state vector（=nelisp-gc--vm-stacks-root=）。
3. *live AOT frame roots*: 実行中 native frame の frame-local root vector スタック（=nelisp-gc--aot-frames-root=）。
4. *AOT pinned roots*: module-lifetime pinned root vectors（=nelisp-gc--aot-pinned-roots-root=）。
5. ***just-evaluated form の結果値（escape root）***: driver が *境界 GC の間、form の戻り値を登録 root スロットに
   保持*する。旧 epoch reclaim はこれを root 化せず捨てて use-after-free を起こした（§2）。**本設計の核心**:
   form result は global 格納前でも *driver が一時 root として pin* → reachable → 生存。
6. *imm-aware*（Doc 146 ④ が入った場合）: root slot が tagged-immediate word を持つなら =nl_val_is_imm= で skip。

** 4.2 Mark（=nelisp-gc-inner-run-mark-phase= を使用）
- root 値を grey queue へ。grey を pop → black、子を grey へ（tri-color）。
- *子 walk を型ごとに完全列挙*（§5）: Cons(car/cdr Sexp slot)、Vector(elems)、Record(fields)、HashTable(k/v)、
  CharTable / BoolVector（Doc 146 ② で明示された取りこぼし候補）、Closure(captured env)、Marker(buffer)、
  Str/MutStr(子無し)。各 box の storage form = 32B Sexp slot を辿る。

** 4.3 Sweep（=nelisp-gc-inner run-sweep-phase= 系 + emitted =nl_gc_sweep_one=）
- 全 chunk を走査（chunk-aware: head→next の chunk descriptor 連鎖、各 chunk の [hdr][obj]… 列）。
- mark1（live）→ mark0 に戻す。mark0（unmarked）→ free-list へ（family-keyed bucket、=nl_gc_free_block=）。
  mark2（既 free）→ skip。
- reuse は zero-fill 済（§1.2）= dirty-memory 健全。

** 4.4 Trigger
- 配線点: reader driver の top-level form 境界（現 epoch reclaim 呼出し位置を置換）。
- 閾値 T（bump-since-last-GC）で amortize。初期 T=64MB（Doc 146 §3.0.5 で 64MB が non-moving floor と実測、
  trigger 頻度の妥当値）。

* §5 Root-coverage 完全性（Doc 146 ②）— 最重要・S1/S3 の核心

mark の root/子 walk に *1 つでも取りこぼし*があれば、live box が unmarked → sweep が free → 次の alloc が
reuse → use-after-free = §1.2 の非決定 abort そのもの。健全化はここが全て。

** 5.1 監査項目
1. *全 root 種別の列挙完全性*: §4.1 の 1-5。特に **(5) escape root**（form 戻り値）と、
   **per-binding env scaffolding vector / scratch slot**（Doc 146 が「scratch-slot 監査」と名指し）。
2. *全 box 型の子 walk*: §4.2。Doc 146 ② が名指しする **CharTable 子 / BoolVector 子**の walk 欠落を最優先で塞ぐ。
3. *AOT frame の safe-point / frame-layout 整合*: =nelisp-gc-inner-collect-roots= が safe-point から root を正しく
   復元するか（vid→addr、=roots-from-safe-point=）。

** 5.2 差分検証（diagnostic-first）
- *GC verify を standalone build に収録*し（§6）、=poison-from-space= を ON にして collection を走らせる:
  free した領域を poison し、以後の touch を検出。anvil-pkg suite load を走らせ、**最初に poison-touch する
  アクセス = 取りこぼした live への参照**を捕捉 → その親型の子 walk / root 種別を §5.1 に追加 → 反復。
- これにより「非決定 abort」を「決定的な verify violation（どの型のどのスロットが未 mark か）」へ変換する。

* §6 検証計画

1. *GC verify を standalone build に入れる*（現状未収録、§1.3）: =nelisp-gc-verify= / =-verify-hooks= を

2. *単体*: =make standalone-chunk-growth-test=（既 PASS）+ 新 =standalone-gc-soundness-test=
   （alloc→drop→GC→arena-stats で =COLL>0= かつ =used= 減 を assert、escape 値が GC を跨いで正しい を assert）。
3. *回帰*: 既存 4438 ERT。
4. *Gate (G)*: nemacs で anvil-pkg full suite を load+run（=loadrun.el= 系）。
   - GC OFF（現状）: 非決定 abort / =used= 無限増。

5. *差分*: 同一 suite load で OFF/ON の arena-stats（COLL / used / reserved）を比較。

* §7 段階（各段で 4438 緑 + boot 健全を維持）

- *P1 配線（flag-off 既定）*: gc-inner mark+sweep を form 境界に配線、=collect-enabled= slot で gate。GC verify を
  standalone build に収録。default OFF（現状維持）。検証: build 緑 + verify モジュール load 確認。
- *P2 root-coverage 監査（S1/S3 の核心、§5）*: escape root の driver 配線 + CharTable/BoolVector 子 walk +

- *P3 有効化*: watermark T で collection ON。検証: gc-soundness-test + 4438 + **anvil-pkg suite 完走（gate G）** +
  =used= bounded。
- *P4 後続*: 増分/世代別 GC（perf）、Doc 146 ③ moving GC（peak/munmap）、④ tagged-immediate（imm-aware mark は
  §4.1.6 で既に考慮）。

* §11 P1 実測知見（2026-06-14）と設計訂正

P1 着手で source を精査・実測した結果、§0–§10 のうち *GC 機構の所在*に重要な訂正がある。

** 11.1 GC は native（elisp gc-inner/gc-verify は reader 非使用）
- reader の稼働 GC は =nelisp-standalone--arena-source=（elisp defconst → nelisp-cc が native emit）内の
  =nl_gc_collect= / =nl_gc_mark_*= / =nl_gc_sweep= / =nl_gc_free_block=。driver が top-level form 境界で
  =nl_gc_collect= を呼ぶ（=scripts/...:3228/3252/5634=）。
- =nelisp-gc-inner.el= / =nelisp-gc-verify.el=（§1.3/§6 で「build に収録」と書いた elisp 群）は *reader が使わない別実装*。
  → **§6.1「elisp gc-verify を build に収録」は誤り**。verify は *native 側*（swept block の poison: reuse OFF 時に
  全 payload poison、=scripts/...:732=）で行う。=NELISP_GC_VERIFY_DEBUG= は無関係（実測で nemacs 上無反応）。

** 11.2 collect が走らない真因 = gate でなく trigger/配線（実測）
- =nl_gc_collect= の RECLAIMER GATE = 固定アドレス =268435616=（= 静的 base 0x10000000 + 160）を読み、=1= で no-op。
- arena init は2系統: 静的 =--arena-init-metadata-forms=（base+160 = *1 disabled*）と
  動的 =--arena-init-metadata-forms-dynamic=（Doc 140 Stage 8、runtime mmap base、base+160 = *0 enabled*）。
- *実測*: build-time env で静的 init の base+160 を 0 に flip し rebuild → 3.4GB 確保しても =COLL=0=（collect 不発）。
  = **静的 init は active でない**。active は動的 init（base+160 既に 0=enabled）。*なのに collect が発火しない*。
- = 真因は documented gate ではなく、(a) =nl_gc_collect= の gate 読みが *固定アドレス 268435616* なのに動的 base は
  *runtime mmap アドレス*で不一致の疑い、(b) form 境界の collect 呼出しが trigger 閾値（=268435560= = 16MiB）で
  gate される配線、のいずれか/両方。**enable レバーは documented gate でなく trigger/アドレス整合側**。
- *実用上の reclaim*: 稼働しているのは =nl_gc_collect= でなく **per-eval LIFO scratch reset**
  （nested eval 戻り時、結果が inline immediate (tag<=3) かつ mutation-epoch 不変 = no-escape なら bump を entry
  mark へ reset。=scripts/...:1990+=）。純数値再帰は回収するが、*heap 結果を産む form（cons/string/defun = suite の
  大半）は escape gate で reset されず never-free* → §1 の無限増。

** 11.3 訂正後の P1（正しいレバー）
- *誤*: elisp gc-verify を build に収録。
- *正*: 動的 init 経路で =nl_gc_collect= が form 境界で実発火する配線を有効化する（trigger 閾値 + gate アドレス整合）。
  その上で reuse OFF（native poison）で suite を走らせ、root/mark gap が live を free → poison fault で §5 を決定化。
- ただし *この trigger/アドレス配線は native GC 著者の mental model が要*。本調査で固定アドレス vs runtime base、
  2系統 init、trigger gate の関係を静的読解で繰返し誤読した。確実な enable は rebuild 反復を伴う *専用の native-GC
  集中作業*で、root/mark gap 修正（§5、本丸）と一体。

** 11.4 結論
§5（root-coverage 健全性）が本丸である点は不変で、source コメント
（=「Keep collection disabled until the standalone root/mark gap is fixed」=）が直接裏付ける。だが「まず collect を
診断有効化する」P1 は、*documented gate を 1→0 するだけでは不足*（active な動的 init では既に 0、なお不発）で、
*trigger/アドレス配線*の native 修正が要ると実測で判明。P1 は native-GC 内部に踏み込む集中作業として再スコープし、
§5 の root/mark gap 監査と同一セッションで進めるのが妥当。

** 11.5 *決定的発見（2026-06-14 続行）= native GC は static 固定-base 模型で、現 dynamic-base build に未 wire*
§11.2 の「trigger/アドレス整合」を実機で詰めた結果、根因が確定:
- 稼働 reader で固定アドレスの gate slot を user elisp から読むと **SIGSEGV**:
  =./target/nelisp --load '(ptr-read-u64 268435616 0)'= → EXIT=139。一方 =(nelisp--arena-stats)=（runtime-base
  相対の読み）は正常に tuple を返す。
- = **現 Doc 140 Stage-8 dynamic-base build では固定アドレス領域 0x10000000+offset が unmapped**。arena は
  runtime mmap base に在り、=nl_gc_collect= が読む固定 gate（=268435616= = 静的 base+160）/ trigger（=268435560=）/
  reuse（=268435624=）/ mutation-epoch（=268435544=）は *この build に存在しない*。
- = =nl_gc_collect= と固定アドレス gate 群は *static 固定-base arena 向けに書かれており、Stage-8 dynamic-base へ
  port されていない*。よって collect は dead/未呼出（COLL=0・純 bump）。**§11.2-11.3 で「gate を flip すれば」と
  考えたのは誤りで、そもそも GC 機構全体が現 arena 模型に wire されていない**（私が静的読解で固定アドレスと
  runtime base を混同し続けた根因もこれ）。
- *per-eval LIFO scratch reset* が pure 数値再帰を回収できるのは、それが *runtime-base 相対のカーソル/mark* で
  動くため（固定アドレス mutation-epoch に依存する gate 部分は同様に未 wire の可能性、要確認）。

** 11.6 訂正後のスコープ（sound-GC 実装の本体）
sound-GC を現 reader で稼働させるには、以下を *native-GC 著者の mental model で一体実装*する必要がある（gate flip
ではない）:
1. **GC のアドレス模型を Stage-8 dynamic-base へ port**: gate / trigger / mutation-epoch / collect / mark / sweep の
   全固定アドレス参照を *runtime arena base 相対*へ。chunk descriptor 連鎖（head/next）を sweep が辿る。
2. **driver の form 境界で =nl_gc_collect= を実呼出し**（trigger 閾値を runtime base 相対 cursor で判定）。
3. **§5 root-coverage 完全性**: globals / vm-stack / aot-frame / pinned + escape root + CharTable/BoolVector 子walk +
   scratch-slot。
4. **reuse OFF poison で差分検証**（§5.2）→ root/mark gap を決定化 → 反復 → clean で本有効化（gate G）。
- これは「P1 で verify を入れる」より大きい *GC の dynamic-base 移植 + 健全化* で、Doc 140 Stage-8 の GC 未対応分の
  完了に相当。

** 11.7 *次セッションの一意 resume entry（2026-06-14 確定）*
§5 には 2 面ある: *(静的)* mark 関数の子walk 完全性のコード精査（collect 実発火 不要・独立に可能）と、
*(動的)* poison 差分検証（§5.2、collect 実発火 = §11.6-1 が前提）。よって作業順は:

1. **§5（静的）子walk 監査 ← 次セッションの開始点**。=nelisp-standalone--arena-source= の native mark
   （=nl_gc_mark_roots= / =-mark_slot= / =-mark_block= / =-mark_cons= / =-mark_vec_slots= / =-mark_pool=）の
   *type-tag dispatch を全列挙*し、=nelisp-sexp-layout.el= / =nelisp-heap-types.el= の全 box 型 × 子スロットと
   突き合わせ、*子を walk していない型を gap 表に列挙*（Doc 146 ② = CharTable / BoolVector が筆頭候補）。

* §11.8 §5.1 静的子walk 監査結果（2026-06-14 実施）

=nl_gc_mark_slot=（=scripts/nelisp-standalone-build.el:1379=）の type-tag dispatch を全列挙し、各 Sexp tag
（=lisp/nelisp-sexp-layout.el= の 0–12）の子walk 完全性を精査した。

| tag | 型 | 子/buffer | mark_slot の処理 | 判定 |
|-----+----+-----------+------------------+------|
| 0–3 | Nil/T/Int/Float | なし(immediate/inline payload) | else→0 | OK |
| 4 | Symbol(String) | name buf のみ | =nl_gc_mark_buf (ptr-read-u64 sp 16)= | OK（value/function/plist は global HT=root 側。box は name のみ） |
| 5 | Str | buf | =nl_gc_mark_buf (sp+16)= | OK |
| 6 | MutStr | buf | box + =nl_gc_mark_buf (box+8)= | OK |
| 7 | Cons | car/cdr | =nl_gc_mark_cons=（box+2子再帰） | OK |
| 8 | Vector | data buf + elem slots | box + buf + =mark_vec_slots(data,0,len)= | OK |
| *9* | *CharTable* | default/parent/entries値 + entries buf | *box block のみ mark* | **GAP-1** |
| *10* | *BoolVector* | bits buffer(別 heap block) | *box block のみ mark* | **GAP-2** |
| 11 | Cell | value word | box + value word(imm-aware 再帰) | OK |
| 12 | Record | type_tag + data buf + field slots | box + (box+0)slot + buf + slots再帰 | OK |

** GAP-1（確定・高確度）: tag 9 CharTable の子未 walk
=nl_gc_mark_slot= の tag-9 arm は =(nl_gc_mark_block (ptr-read-u64 sp 8))= で *box block を mark するだけ*
（コメント「tag 9/10 boxed-no-walk (do not occur)」= char-table 不出現の前提）。char-table get/set source
（=lisp/nelisp-cc-evalport-nonenv-char-table.el=）より box（=sp+8=）の子は:
- =box+32=: *default 値*（32B Sexp slot）
- =box+88=: *parent* char-table box ptr
- =box+64=: entries 配列 ptr、=box+80=: entries_len、stride 40 = =[key:u64@+0][value:32B Sexp slot@+8]=
  → entries buffer block + 各 entry の value slot
*fix*: tag-9 arm を Vector/Record 同様に拡張 — box mark + =nl_gc_mark_slot (box+32)=（default）+ parent box を
mark/再帰 + entries buffer を =nl_gc_mark_buf= + 各 i で =nl_gc_mark_slot (eptr + i*40 + 8)=。
*顕在化条件*: Emacs-compat（syntax/case/category table 等）が char-table を生成すると、default/parent/entry値が
char-table 経由でのみ到達可能な場合に sweep される use-after-free。前提「do not occur」が vendor/Emacs-compat load で破れる。

** GAP-2（確定）: tag 10 BoolVector の bits buffer 未 mark
NlBoolVector box layout（=lisp/nelisp-cc-jit-bool-vector-len.el=）: =box+0= = *value.ptr（Vec<bool> bits buffer、
別 heap block）*、=box+8= cap、=box+16= length、=box+24= rc。tag-10 arm は box block のみ mark し、=box+0= の
*bits buffer を mark しない* → bits buffer が sweep/reuse される。Sexp 子は無い（bits は raw）ので fix は
=nl_gc_mark_buf (ptr-read-u64 box 0)= の 1 edge 追加のみ。

** 結論 + 次手
- 子walk 漏れは *tag 9 CharTable と tag 10 BoolVector の 2 件*（Doc 146 ② の名指しと一致、offset 付きで確定）。
  他 tag（0–8, 11, 12）は子/buffer を walk 済。
- これらは「mark の取りこぼし」= sweep が free する候補。**§11.6-1 で collect 実発火後、reuse OFF poison で
  「char-table/bool-vector を多用する load が poison-touch するか」を動的確認** → 確認できたら上記 fix を適用。
- ただし *root 側*（§4.1 の globals/vm-stack/aot-frame/pinned + escape root）の完全性は本静的監査の範囲外
  （mark は root から辿るので、root 取りこぼしは別途 §11.6-2 の root 列挙監査で。escape root が筆頭）。
- = 静的監査の deliverable（GAP-1/2 の fix-spec）は確定。残りは collect 実発火（§11.6-1）+ poison 動的検証 +
  root 列挙完全性。

* §11.9 root 側 静的監査（§4.1）+ §11.5 訂正（2026-06-14）

** root 列挙は概ね完全（escape root は既に handled）
=nl_gc_mark_roots=（=scripts/...:1518=、引数 ctx/result/out/pool/src/cursor/bsym）が mark する root:
- =ctx+0=（mirror / globals）/ =ctx+32=（frame stack）/ =ctx+64=（unbound marker）
- =result=（current parsed form）/ ***out（in-flight / last result）***/ =pool=（parse-pool RAW buffer, cap slots）
- =src=（source string）/ =cursor=（reader cursor）/ =bsym= + shared symentry（268436328、block + Symbol slot で
  name buf も mark。過去 name-buf が growth chunk で未mark → munmap → SIGSEGV を起こした gap の修正がコメントに
  記録 = root-gap を crash 駆動で塞いだ前例）。
- = **§4.1.5 で「escape root が旧 epoch reclaim の罠」と強調したのは tracing GC には当たらない**: tracing GC は
  =out=（form 結果）を *既に root として mark* している。旧 epoch reclaim（region-reset、reachability を辿らない別
  機構）が escape を取りこぼしたのであって、tracing の root 側は escape を root 化済。**root 列挙に明白な欠落は
  見えない**（frame-stack/pinned の transitive 到達の完全性は動的確認が要るが、root カテゴリは網羅）。

** §11.5「固定アドレス unmapped」の訂正
collect call site（=scripts/...:3224=）は =(< bump@268436184 trigger@268435560)= を *native で読んで* collect を
gate しており、reader は crash せず動く = **native は 0x10000000 領域の固定アドレスを問題なく read している**。
§11.5 の SIGSEGV は =./target/nelisp --load '(ptr-read-u64 268435616 0)'= = *user-elisp の ptr-read-u64* 由来で、
これは arena オブジェクト境界の bounds-check 等で固定 gate アドレスを弾いた可能性が高い（native のアクセスとは別経路）。
= **§11.5/§11.6 の「GC は dynamic-base に未 wire / アドレス模型 port が要る」は早計**。native は固定 gate/trigger を
読めており、COLL=0 の原因は *gate 値（268435616 が 1 か）/ trigger 条件（bump<trigger）/ COLL カウンタの意味* の
いずれかに絞られる。

** COLL=0 の正確な原因は runtime instrumentation で（静的読解は誤診を反復）
静的 base+160 を 0 に build しても COLL=0 だった（§11.2）理由は、(a) 動的 init が別アドレスに gate を書く /
(b) trigger 条件 / (c) COLL カウンタ位置、のどれかだが *静的読解では確定できず繰返し誤診*。
= **§11.6-1 の正しい最初の一手は「アドレス模型 port」ではなく、=nl_gc_collect= と call site に
=nl_write_stdout_bytes= 等の debug print を 1 行差し込んで rebuild し、「collect が呼ばれるか / gate 読みは何か /
trigger 比較の両辺は何か」を実機で観測する instrumentation**。それで COLL=0 の真因（gate 値か trigger か）を確定
してから enable する。GAP-1/2（§11.8、child 側、offset 確定）は enable と独立に fix を用意できる。
