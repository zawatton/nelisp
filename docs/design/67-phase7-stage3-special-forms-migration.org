* §0 TL;DR

Phase 7 self-host (Doc 63) の 3 番目の足場。=build-tool/src/eval/special_forms.rs=
の 33 special forms を「**Tier 1 = irreducible primitive**」と「**Tier 2 =
elisp macro derivable**」に分類、Tier 2 (= 22 件) を elisp =defmacro=
として =lisp/nelisp-stdlib-eval-special.el= に移し、Rust 側 =sf_*=
ハンドラを撤去する。Stage 7.2 (= reader) で確立した *parallel path
+ takeover* pattern の 2 度目の application。

| layer | before | after |
|-+-+-|
| Rust =eval/special_forms.rs= (1,146 LOC) | 33 special forms | ~600 LOC (= Tier 1 13 forms 残) |
| Rust =eval/mod.rs= dispatcher | =apply_special= が 33 arm | 13 arm に縮退 |
| elisp =lisp/nelisp-stdlib-eval-special.el= | (新規) | ~400 LOC (= Tier 2 22 macros + tests) |

LOC 削減推定: Rust ~500 (= Tier 2 ハンドラ + helpers)、elisp +400。残り
~600 Rust LOC は Stage 7.4 (= apply / env) 完了後に連動して撤去 (=
Doc 63 §2.3 〜 §2.4 が言う合計 ~900 LOC 削減のうち、本 Doc 67 が ~500、
Doc 68 Stage 7.4 が ~400 を担当)。

Tier 1 = control flow / scoping / closure / non-local exit / signal の
プリミティブで *elisp 自身では書けない*。Tier 2 = Tier 1 を組み合わせ
れば書ける糖衣構文 — 33 件中 22 件 (= 67%) がこれに該当。Phase 5 JIT
lower 後、Tier 1 primitive 自身が host 命令直 emit され、Tier 2 macro
expansion 後の =if= / =while= / =setq= chain も同経路を通るので
wallclock degradation < 3x で gate (= Doc 63 §5.3)。

** §1.2 Stage 7.2 chicken-and-egg からの教訓

Stage 7.2 で発見した bootstrap dependency cycle:
- =reader.el= の body が =not= / =null= / =nreverse= / =floatp= 等
  「elisp で書かれた helper」を使う
- それら helper は =nelisp-stdlib*.el= で defun されている
- =reader.el= 自身を *load する* ためには Rust reader が必要 (=
  elisp reader が自分自身を読めない)

Stage 7.2 はこれを「Rust reader を Tier 1 残し、elisp reader は
=read-from-string= entry の takeover として install」で解決 (= Doc 66 §3)。

Stage 7.3 でも同形の罠がある:
- =nelisp-stdlib-eval-special.el= の =defmacro when= の body が
  =if= / =progn= 等の primitive を使う
- =if= / =progn= は Tier 1 (= Rust 残) なので動作するが、もし
  =when= macro 自身が他 stdlib helper (= =cl-list= 系) を body で
  使うと再帰の罠が発生
- 解決: *Tier 2 macro body は Tier 1 primitive のみで書く*。これに
  より load order に関わらず install 直後から動作する

この **Tier 1-only macro body 制約** が本 Doc の中核設計判断。

** §1.3 Phase 5 native cc lowering との相補関係

Tier 2 macro が =if= / =progn= / =setq= / =while= に展開される。Phase 5
SHIPPED 後 (= Stage 5.1〜5.6 commit b37cd72/9330f85)、これら Tier 1
primitive のうち arithmetic / cons / predicate / access 系は lower 経由
で host 命令直 emit。よって macro 展開後の hot path は Rust special
form 直叩きと比べて wallclock 増を最小限に抑制可能 (= Doc 63 §5.3
gate < 3x degradation)。

bench gate 例:
- =(let ((x 0)) (dotimes (i 10000) (setq x (+ x i))) x)= = =dotimes=
  macro 展開 → =while= + =setq= + =1+= = Phase 5 ArithIR JIT lower
  → Rust =sf_dotimes= 直叩きとの差は ~1.5x 以内見込み

** §1.4 既存の elisp macro infra

NeLisp は既に macro expander を持つ:
- =defmacro= 自身は Tier 1 special form (= Rust =sf_defmacro= が
  =macro_form= cons cell を =fcell= に install)
- macro expansion は =eval/mod.rs= の =apply_combiner= が cons head
  fcell lookup → 結果が =(macro . LAMBDA)= なら body call → 結果を
  改めて eval ループに戻す
- Doc 50 で =cl-defmacro= / =pcase= が elisp 化済 (= 既に同 path で
  動作実績あり)

本 Stage はこの infra を 22 件の primitive に拡張するだけ。新規
mechanism なし、scaling のみ。

* §2 設計

** §2.1 Tier 分類

*** Tier 1 = irreducible primitive (= Rust 残、13 forms)

elisp macro で derive 不可能。理由:
- =eval-form= ループが直接構造を見る必要がある
- Rust 側 control flow (= =EvalError= unwind / =UncaughtThrow=) と直接
  対応する non-local exit
- =Env= push_frame / pop_frame の lexical scope mutation

| form | 役割 | Rust 必須理由 |
|-+-+-|
| =quote= | reader-evaluated literal | =(car args)= をそのまま return、eval 阻止 |
| =function= | =#'FORM= closure capture | =sf_lambda= 経路の同義 |
| =if= | conditional branch | =eval(then)= or =eval(else)= の選択を Rust 側で実行 |
| =let= | parallel lexical bind | =push_frame= / =bind_local= / =pop_frame= の env mutation |
| =let*= | sequential lexical bind | 同上 |
| =lambda= | closure 作成 | =capture_lexical= で env Rc clone を closure に bake |
| =setq= | symbol 値設定 | =set_value= の env mutation |
| =while= | loop control | conditional jump の host iteration |
| =progn= | body sequencing | each form を順 eval |
| =condition-case= | error catch | Rust =EvalError= unwind の handler dispatch |
| =unwind-protect= | cleanup 保証 | error path / success path 両方で cleanup を eval |
| =catch= | non-local exit catch | =EvalError::UncaughtThrow= unwind |
| =throw= | non-local exit raise | =EvalError::UncaughtThrow= raise |

LOC 推定: 1,146 LOC のうち Tier 1 + helper (= =parse_let_binding= /
=eval_body= / =sexp_eq= 等) ~600 LOC が残存。

*** Tier 2 = elisp macro derivable (= 22 forms)

Tier 1 primitive の組み合わせで書ける:

| form | derive shape (= macroexpand-1 結果) | dependency |
|-+-+-|
| =cond= | =if= の chain | =if= |
| =when= | =(if COND (progn BODY...) nil)= | =if= / =progn= |
| =unless= | =(if COND nil (progn BODY...))= | 同上 |
| =and= | nested =if= short-circuit | =if= |
| =or= | =let= + =if= short-circuit | =let= / =if= |
| =prog1= | =(let ((tmp BODY1)) BODY-rest tmp)= | =let= |
| =prog2= | =(progn BODY1 (prog1 BODY2 BODY-rest))= | =prog1= / =progn= |
| =dolist= | =let= + =while= + =car= / =cdr= drive | =let= / =while= / =setq= |
| =dotimes= | =let= + =while= + =1+= drive | 同上 |
| =defun= | =(fset 'NAME (lambda ARGS BODY...))= | =fset= primitive |
| =cl-defun= | =defun= + =&key= 解析 macro (= 既存 sf_cl_defun の elisp port) | =defun= / =memq= / =or= / =car= / =cdr= |
| =defmacro= | =(fset 'NAME (cons 'macro (lambda ARGS BODY...)))= | =fset= |

| =defcustom= | =defvar= 簡化版 | 同上 |
| =defgroup= | symbol entry stub (= no-op に近い) | =put= |
| =setq-default= | =setq= alias | =setq= |
| =save-excursion= | =progn= 直 (= 既存も no-op stub) | =progn= |
| =save-restriction= | =progn= 直 (= 既存も no-op stub) | =progn= |
| =push= | =(setq PLACE (cons X PLACE))= for symbol place | =setq= / =cons= |
| =pop= | =(prog1 (car PLACE) (setq PLACE (cdr PLACE)))= for symbol place | =prog1= / =setq= / =car= / =cdr= |

LOC 推定: 22 macro * ~15 LOC = ~330 elisp LOC、+ unit tests ~70 LOC =
~400 LOC。Rust 側削減 ~500 LOC (= Tier 2 sf_* + dolist/dotimes 等の
比較的大きいハンドラ)。

** §2.2 Tier 1-only macro body 制約

Stage 7.2 の bootstrap dependency cycle 教訓を application:

- *制約*: 本 Doc で書く Tier 2 macro の body は **Tier 1 primitive
  + reader-installed builtin (= cons / car / cdr / eq / null) のみ**
  を使う
- *理由*: macro install timing が *他 stdlib より早い* (= §2.3) ので、
  body から呼ぶ helper は「絶対 install 済」のものに限る必要がある
- *検証*: 各 macro の body から呼ぶ symbol を grep し、Tier 1 / Rust
  builtin 集合に含まれるか確認 (= ERT で =macroexpand-1= 後の form
  を walk して symbol 参照を抽出する gate を追加)

例:
#+BEGIN_SRC emacs-lisp
;; OK — body は if + progn (= Tier 1)
(defmacro when (cond &rest body)
  `(if ,cond (progn ,@body) nil))

;; OK — body は let* + while + setq + car + cdr (= Tier 1 + Rust builtin)
(defmacro dolist (spec &rest body)
  (let ((var (car spec))
        (list-form (cadr spec))
        (result-form (caddr spec))
        (--list (gensym))
        (--result (gensym)))
    `(let* ((,--list ,list-form)
            (,var nil)
            (,--result nil))
       (while ,--list
         (setq ,var (car ,--list))
         ,@body
         (setq ,--list (cdr ,--list)))
       ,--result)))
#+END_SRC

NG (= 制約違反) 例: macro body 中に =mapcar= / =cl-loop= / =seq-each=
等を呼ぶ。これらは別 stdlib (= =nelisp-stdlib-hof.el= / =nelisp-cl-
macros.el=) で defun されており、本 Doc の install order より後に
load される可能性がある。Tier 1 + 既知 Rust builtin だけに自制する。

** §2.4 Rust 側 dispatcher 縮退

Stage 7.3.b で =eval/special_forms.rs= の =apply_special= の match arm

#+BEGIN_SRC rust
pub fn apply_special(name: &str, args: &Sexp, env: &mut Env)
    -> Result<Option<Sexp>, EvalError>
{
    let result = match name {
        // Tier 1 primitive (= elisp macro で derive 不可)
        "quote" => sf_quote(args)?,
        "function" => sf_function(args, env)?,
        "if" => sf_if(args, env)?,
        "let" => sf_let(args, env)?,
        "let*" => sf_let_star(args, env)?,
        "lambda" => sf_lambda(args, env)?,
        "setq" => sf_setq(args, env)?,
        "while" => sf_while(args, env)?,
        "progn" => sf_progn(args, env)?,
        "condition-case" => sf_condition_case(args, env)?,
        "unwind-protect" => sf_unwind_protect(args, env)?,
        "catch" => sf_catch(args, env)?,
        "throw" => sf_throw(args, env)?,
        // Stage 7.3 で削除 (= Tier 2、elisp macro 経由)
        // "cond" / "when" / "unless" / "and" / "or" / "prog1" / "prog2"
        // "dolist" / "dotimes" / "defun" / "cl-defun" / "defmacro"
        // "defvar" / "defvar-local" / "defconst" / "defcustom" / "defgroup"
        // "setq-default" / "save-excursion" / "save-restriction"
        // "push" / "pop"
        _ => return Ok(None),
    };
    Ok(Some(result))
}
#+END_SRC

** §2.6 cl-defun の elisp port

=sf_cl_defun= は 110+ LOC の比較的大きい special form (= =&key= 引数
展開 + body wrap)。これも Tier 2 として elisp macro 化:

#+BEGIN_SRC emacs-lisp
(defmacro cl-defun (name formals &rest body)
  (let ((parsed (nelisp--parse-cl-formals formals)))
    (let ((positional (plist-get parsed :positional))
          (optionals (plist-get parsed :optional))
          (rest-sym (or (plist-get parsed :rest) '--cl-keys))
          (keys (plist-get parsed :keys)))
      (if (null keys)
          ;; &key なし: ただの defun と同じ
          `(defun ,name ,formals ,@body)
        ;; &key あり: formals を書き換え、body を let* で wrap
        (let ((new-formals (append positional
                                   (and optionals (cons '&optional optionals))
                                   (list '&rest rest-sym)))
              (bindings (mapcar (lambda (k)
                                  (let ((kw (car k)) (param (cadr k)) (default (caddr k)))
                                    `(,param (or (car (cdr (memq ',kw ,rest-sym)))
                                                 ,default))))
                                keys)))
          `(defun ,name ,new-formals
             (let* ,bindings ,@body)))))))
#+END_SRC

helper =nelisp--parse-cl-formals= は別 defun (= =nelisp-stdlib-eval-special.el=
内、~50 LOC)。既存 =sf_cl_defun= Rust 実装と完全 1:1 対応。

* §3 段階 / sub-stages

Stage 7.3 を 4 sub-stage に分割、各 sub-stage で independent ship 可能。

** §3.0 Dispatch order の制約 (= 2026-05-07 Stage 7.3.a 着手時に判明)

Doc 67 初版 (= §3.1〜§3.4) は「parallel path」表現で「Rust special form
arm と elisp macro が同居しつつ後者が runtime 経路で使われる」を想定
していたが、実装着手で =build-tool/src/eval/mod.rs:191= の
=apply_combiner= dispatch order が以下と判明:

#+BEGIN_SRC rust
fn apply_combiner(head: &Sexp, tail: &Sexp, env: &mut Env)
    -> Result<Sexp, EvalError> {
    match head {
        Sexp::Symbol(name) => {
            // 1) Special forms — argument evaluation order is form-specific.
            if let Some(result) = special_forms::apply_special(name, tail, env)? {
                return Ok(result);
            }
            // 2) Macro? — expand once, recurse on expansion.
            if let Ok(func) = env.lookup_function(name) {
                if is_macro(&func) { ... }
                ...
            }
        }
        ...
    }
}
#+END_SRC

つまり =apply_special= が match arm hit する間、fcell に install された
elisp macro は **dormant** (= unreachable from runtime path)。よって
Stage 7.3.a〜c で elisp =defmacro= を install しても =(when X Y)= 等の
runtime evaluation は依然 Rust =sf_when= 直行で、elisp macro path は
ERT (= =macroexpand-1= 経由) でしか駆動できない。

対応:
- *Stage 7.3.a〜c は ERT-only parallel install* — elisp macros は install
  + ERT で expansion shape gate のみ、runtime swap は発生しない
- *Runtime swap は Stage 7.3.d で一気発生* — Rust 22 arm 削除と同 commit
  で全 Tier 2 macros が active 化、これ以前の段階では Rust path と elisp
  path の cross-rust 等価性 test が *等価動作* を保証するわけではない
  (= elisp expansion が Rust 直行 evaluation と semantic 同一と確認できる
  のみ、実際の runtime はずっと Rust 経路)

これは Stage 7.2 reader (= =read-from-string= takeover via fcell) と
非対称: reader entry は Rust =bi_read_from_string= で fcell shadowing
可能だったが、special form は match arm = override 不可能。

設計判断: dispatch order を逆転 (= macro fcell → special form) させる
代替案は採らない (= bootstrap 中の =(if ...)= 等 Tier 1 まで影響、リス
ク過大)。Rust 22 arm 撤去 = elisp swap を **同 commit で原子的に行う**
方が安全。

** §3.4 Stage 7.3.d — Rust sf_* 22 件撤去 + runtime swap (SHIPPED)

実装での発見:

- *Bootstrap chicken-and-egg*: =sf_defmacro= 削除と同時に file 先頭の
  =(defmacro X ...)= も dispatch 不能化。解決策として
  =lisp/nelisp-stdlib-eval-special.el= の最先頭に raw =(fset 'defmacro
  (cons 'macro (cons (lambda ...) nil)))= を置き、Tier 1 special form
  + Rust builtin function (=fset= / =cons=) のみで defmacro を installl
  する bootstrap shim を入れた。Doc 67 §2.3 で「defmacro entry 全て
  Rust sf_defmacro 経由」と書かれていた前提が削除と矛盾、shim が解
- *=set= builtin 追加*: =sf_defvar= / =sf_defconst= が直接呼んでいた
  =env.set_value= は elisp macro からは呼べない。=(set SYMBOL VALUE)=
  builtin (=bi_set=, ~15 LOC) を =builtins.rs= に追加、defvar/defconst

7.3.a → 7.3.b → 7.3.c → 7.3.d。各 sub-stage で全 ERT PASS かつ

* §4 Test plan

** §4.1 macroexpand-1 inspection ERT

各 Tier 2 macro について:
- =(macroexpand-1 '(when t 1 2))= = =(if t (progn 1 2) nil)= の equal 比較
- =(macroexpand-1 '(dolist (x '(1 2 3)) (princ x)))= が =let*= +
  =while= の正しい shape を返すか
- =(macroexpand-1 '(push x lst))= = =(setq lst (cons x lst))=

22 macros * ~3 case = ~70 ERT 案件。

** §4.2 cross-rust 等価性 ERT (Stage 7.3.d でのみ有効)

§3.0 通り、Stage 7.3.a〜c では runtime path が依然 Rust 直行のため、
「elisp macro expand 後 evaluate した結果」は実際には *呼び出されな
い*。よって 7.3.a〜c の "parallel path 等価性 test" は意味を持たず、
shape inspection (= §4.1) のみが gate として機能。

cross-rust 等価性 ERT は **Stage 7.3.d で runtime swap が起きた直後**
に初めて意味を持つ:
- 7.3.d 適用前 commit (= Rust 22 arm 残存) で =(when t 1 2)= 等を
  evaluate した結果を保存
- 7.3.d 適用後 (= elisp macros active) で同 form を evaluate
- 両者が =equal= かを git history を跨いだ regression test として確認

差異が出た場合は macro 定義のバグ確定 — Stage 7.3.d revert + macro
shape を 7.3.a〜c に戻して修正。

** §4.3 self-host gate

- 全 stdlib =.el= ファイル + user code が elisp macro 経由で load
  可能 (= Stage 7.3.d 直前で確認)
- 失敗パターン: macro install order の漏れ / Tier 1-only 制約違反
  (= macro body から =seq-= 系を呼んだ等) / =gensym= 衝突

** §4.4 wallclock gate

- Doc 63 §5.3: elisp macro 切替前後で =make test= wallclock < 3x
  degradation
- bench: =(let ((s 0)) (dotimes (i 10000) (setq s (+ s i))) s)= で
  Rust =sf_dotimes= path / elisp macro 展開後 =while= path の比較

** §4.5 Tier 1-only 制約 gate

- 全 Tier 2 macro の =macroexpand-all= 結果を walk、symbol 参照を
  collect、Tier 1 (= 13 forms) + Rust builtin (= =cons= / =car= /
  =cdr= / =eq= / =null= / =not= / =memq= / =fset= / =set= / =put= /
  =boundp= / =gensym= / =1+= / =1-= / =+= / =-= / =<= / =>= /
  =error= 等の最小集合) のみ含むかを ERT で確認
- これにより future な macro 追加でも Tier 1-only 制約が gate される

* §6 共通原則

- *Tier 1-only macro body 制約*: §2.2 通り、Tier 2 macro の body は
  Tier 1 primitive + 既知 Rust builtin のみ使用。bootstrap order に
  関わらず install 直後から動作可能を保証
- *parallel path 期間 minimal*: Stage 7.3.a → 7.3.d を素早く回す。
  Rust + elisp 両 special form が動く期間が長引くと test infra 冗長化

  min 原則)
- *macroexpand-1 inspection を gate 化*: §4.1 で各 macro の expansion
  shape を ERT で固定、future な refactor で expansion が壊れたら
  即検出
- *Stage 7.2 教訓の継承*: chicken-and-egg 回避のため Layer A の install
  order を =STDLIB_SOURCES= 1 番目に固定。reader 同様に bake-in 必須
