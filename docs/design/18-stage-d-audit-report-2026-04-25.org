* §3 ホットスポット分析

buffer-ops が 82% を占めるが、これは anvil.el が *Emacs を計算機として
使う* 設計なので自然。Phase 6.1 headless 化の本質は:

** 3.1 Phase 6.1 MVP scope 内の buffer-ops
MCP tool handler のうち headless 化が現実的なもの:
- =file-read= / =file-outline= (Phase 5-E 実装済、既に NeLisp 上で動作)
- =http-get= (string ベース、buffer 依存弱い)
- =git-log= / =git-status= (subprocess output 解析、buffer 不要)
- =data-get-path= / =data-set-path= / =state-*= (Phase 5-F.1 で追加予定)

これらは *既に buffer-ops 依存を回避した状態で NeLisp 上で動作している*。
Phase 6.1 の launcher は既存 Phase 5-E resources をそのまま expose するだけ。

** 3.2 Phase 6.2 anvil-http port scope
anvil-http Doc 09 には =with-temp-buffer= + =url-retrieve= パターンが
あり、NeLisp 側は Phase 5-E で =insert-file-contents= + =buffer-string=
primitive は揃っている。url-retrieve の host 依存をどう handle するかが
Phase 6.2 の中心課題。

** 3.3 buffer-ops 7903 の伸縮性
buffer-ops は anvil-browser / anvil-org-index / anvil-memory-engine が
濃厚に使う。これらは Phase 6.1 headless MVP *scope 外*、Phase 6.2+ で
個別に port 戦略判断。Phase 6.0 で全部を headless 化する必要なし。
