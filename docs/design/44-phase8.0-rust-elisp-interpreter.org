binary* path は確立 (cdylib + staticlib + 4-stage cold-init coordinator)。

本 doc は user 2026-04-27 vision = *Emacs を一切 spawn しない単一 Rust
binary* を実現するための最終 sub-phase = *Phase 7.5.4 Rust-side minimal

- character literal (=?a=, =?\n=, =?\C-a=、後者は Phase 7.5.4.2 後送り可)
- comment skip (=;= 行末)

*** ERT smoke (+5)

- empty string → error
- =42= / =3.14= / =foo= / ="hello"= / =nil= / =t=
- =(1 2 3)= → 3-element list
- =(quote foo)= = ='foo=

** 3.3 Phase 7.5.4.2 — Rust Lisp evaluator core (~1500 LOC、3-4 週間)

*** scope

- *heap allocator*: =Box<Cons>= / =Vec<u8>= の Rust naive owner、Phase
  7.3 NeLisp GC が後で takeover (= 本 phase は =Rc= or =Arena= で OK、
  Phase 7.5.4.3 で bridge)
- *symbol table*: =HashMap<String, SymbolEntry>=、=intern=, =boundp=,
  =fboundp=、=value-cell= + =function-cell= (= elisp 2-cell model)
- *eval core*:
  - =eval(form, env) -> Result<LispValue, LispError>=
  - special forms: =quote=, =if=, =cond=, =progn=, =and=, =or=,
    =setq=, =let=, =let*=, =lambda=, =defun=, =defvar=, =defconst=,
    =while=, =catch=, =throw=, =condition-case=, =unwind-protect=,
    =save-excursion= (no-op stub)
  - dynamic binding (= elisp default、let 時 =push/pop= で stack)
  - =funcall= / =apply=
- *builtin* (~30-50 = bootstrap で必須):
  - 算術: =+=, =-=, =*=, =/=, =<=, =<==, =>=, =>==, ===, =/==, =mod=
  - list: =cons=, =car=, =cdr=, =list=, =length=, =append=, =reverse=,
    =nth=, =nthcdr=, =memq=, =assoc=
  - string: =concat=, =substring=, =string-length=, =string=, =format=
    (basic =%s=/=%d= のみ)
  - symbol: =intern=, =symbol-name=, =symbolp=, =null=, =eq=, =equal=
  - control: =error=, =message=, =princ=, =prin1-to-string=
  - file: =load=, =require= (=load=path= scan)、=getenv=, =setenv=
  - sequence: =mapcar=, =mapc=, =dolist=, =dotimes= (= macro 展開でなく
    special form として実装)
- *error handling*: =condition-case= の Rust 側実装 (= =Result=
  propagate + =LispError= variant per error type)

*** ERT smoke (+12)

- 算術 closed: =(+ 1 2 3)= → 6
- =let= shadow: =(let ((x 1)) (let ((x 2)) x))= → 2
- =lambda= apply: =((lambda (x y) (* x y)) 3 4)= → 12
- =defun= + recursive: =(defun fib (n) (if (< n 2) n (+ (fib (- n 1)) (fib (- n 2)))))= → fib(10) = 55
- =mapcar=: =(mapcar '1+ '(1 2 3))= → (2 3 4)
- =condition-case= catch: =(condition-case e (error "x") (error 'caught))= → caught
- =catch= / =throw= unwind: 3-level nest
- =setq= dynamic: =(progn (setq x 1) (let ((x 2)) (incf x)) x)= → 1
- =load= + =require=: temp file load
- =format= basic: =(format "%s %d" "n=" 42)= → "n= 42"

*** risks

- *マクロ展開*: elisp =defmacro= の eager expand を Rust 側で実装すると
  ~500 LOC 追加。本 phase は =defmacro= は *NeLisp 自体の bootstrap*
  で必要なら最小実装、不要なら bootstrap source から除外 (= macroexpand
  済 source を embed) で回避
- *byte-code*: =#[...]= byte-code literal は本 phase は *unsupported*
  (= source 側 byte-compile 禁止、=load-prefer-newer= で =.el= 強制)
- *advice*: =advice-add= 等は不要 (= Phase 7.1 native compiler が takeover
  した後で扱う)
- *file scope*: =defvar-local= / =buffer-local-variables= 等は no-op stub、
  bootstrap が要求すれば追加実装

** 3.4 Phase 7.5.4.3 — Self-host bridge (~500 LOC、1 週間、7.5.4.4 と並列可)

*** scope

- =bridge::bootstrap_load(src_dir)= = =src/nelisp.el= → =src/nelisp-eval.el=
  → =src/nelisp-cc.el= の *Phase 7.0/7.1 stage0 順序* で Rust eval が load
- 各 file load 後、=fboundp 'nelisp-eval= で takeover 可能性 check
- takeover 成功時、=control transfer= = 以降の eval は =nelisp-eval=
  経由 (= Rust 側 eval は subgraph reachable から外れる)
- takeover 失敗時、Rust 側 eval で続行 (= "minimal interpreter only" mode)
- =NELISP_FORCE_RUST_EVAL=1= env で takeover 抑止 (= debug + soak test 用)

*** ERT smoke (+5)

- bootstrap load 全 file success (=src/nelisp*.el= 全 file が Rust eval で
  syntactically valid)
- =fboundp 'nelisp-eval= = t after =src/nelisp-eval.el= load
- takeover 後 =(nelisp-eval-string "(+ 1 2)")= → 3
- =NELISP_FORCE_RUST_EVAL= で Rust eval 続行 (= takeover skipped)
- 失敗時 fallback (= 中途 file が syntax error なら =:status fail=
  return、bin/anvil layer で =--strict-no-emacs= 時 exit 1)

*** risks

- *symbol table merge*: Rust 側 intern と =nelisp-eval= 側 intern が
  別 table になり symbol identity 崩壊 risk → mitigation: takeover 時
  symbol table を *渡す* (= =nelisp-eval-set-symbol-table=) bridge API
- *value identity*: Rust 側 =Cons= と NeLisp =Cons= が異 representation
  → mitigation: 本 phase は *takeover 後 Rust 側 value は到達不能* と
  contract、cross-boundary value passing 禁止
- *bootstrap source の Rust eval-only subset 整備*: =src/nelisp-eval.el=
  自身が要求する builtin が Rust 側に揃わなければ chicken-and-egg →
  mitigation: bootstrap 用 =src/nelisp-bootstrap-min.el= を別途用意、
  Rust eval で確実に load できる minimum subset を定義

* §4 Risks + open questions

** 4.1 GC interaction (Phase 7.3 GC は Elisp 側、Rust 側 reader/eval allocate object をどう trace?)

*問題*: Phase 7.5.4.2 で Rust 側 =Box<Cons>= / =Vec<u8>= を allocate するが、
Phase 7.3 NeLisp GC (=src/nelisp-gc.el=) は NeLisp heap span のみ scan する。
Rust 側 object は GC root から外れ、takeover 後に *uncollectable + leak* 化
する risk。

*mitigation 案*:
1. *bridge 時に full copy* — takeover 時に Rust 側 reachable な全 object を
   NeLisp heap に copy (= =Cons= → =nelisp-cons-allocate=)。bootstrap が
   ~MB 級なら現実的、~GB 級だと NG
2. *Rust 側 own arena drop* — takeover 後 Rust 側 arena 全 drop、symbol
   table のみ NeLisp 側に移送。bootstrap source が takeover 時点で
   reachable な runtime value を持たない契約 (= =defun= / =defvar= の
   identity は symbol-cell 経由なので OK、value side は drop)
3. *NeLisp GC に Rust root scan を export* — Phase 7.3 GC の =gc-roots=
   list に Rust 側 stack を含める。Rust 側 ABI 露出が複雑

*推奨*: 案 2 (=arena drop=)。bootstrap 完了後 Rust 側 value は不要、symbol
identity のみ NeLisp 側に持たせれば済む (= 本 phase 7.5.4.3 contract)。

*open question*: bootstrap 中の =defvar= 値 (= top-level value cell) が
Rust 側 =Cons= / =String= を指す場合の取扱。user signoff で確定。

** 4.2 Symbol table (host Emacs symbol vs Rust 側 symbol)

*問題*: Rust 側 =HashMap<String, SymbolEntry>= の symbol identity と、
NeLisp 側 =obarray= の symbol identity を *同一に保つ* 必要がある (=
=(eq 'foo 'foo)= が cross-boundary で =t= を返す保証)。

*mitigation 案*:
1. *takeover 時に Rust symbol table を NeLisp obarray に inject* — Rust
   側 =SymbolEntry= 全 entry を NeLisp =intern= 経由で再 register、Rust
   側 =SymbolEntry*= → NeLisp =Lisp_Object= の forwarding table を保持
2. *NeLisp obarray を Rust 側に export* — Phase 7.5.4.2 reader 開始時点で
   NeLisp obarray が空なら採用可、bootstrap 完成後の =intern= は全 cross-call

*推奨*: 案 1 (=inject=)。bootstrap 中の symbol identity は Rust 内で閉じ、
takeover 時に bulk transfer。

** 4.3 Error handling (=condition-case= の Rust 側実装)

*問題*: Rust =Result<T, E>= で =LispError= を propagate する設計は自然だが、
=condition-case= は dynamic catch (= unwind n level) を要求する。

*mitigation*:
- Rust panic を使う案 = =panic = abort= (Cargo.toml SHIPPED) と矛盾、NG
- *=Result= + dedicated handler stack* = =eval= 内部で =catch_marker:
  Vec<HandlerFrame>= を保持、=condition-case= は marker push、=signal=
  は =Err(SignalThrow { tag, data })= を return、=eval= が marker まで
  unwind

*推奨*: 後者。SBCL =RESTART-CASE= の素朴版、SICP metacircular の継続
渡しを Rust =Result= で elision。

** 4.4 Performance baseline (単純 form の eval が Emacs --batch より遅くないか?)

*問題*: Phase 7.5.4.2 evaluator は AST tree-walking interpreter (= byte-code
VM ではない)、単純 form でも 5-10x 遅い見込み。Doc 32 §5 cold-init 500ms
gate を violation するか?

*mitigation*:
- Rust eval は *bootstrap stage 0 のみ active*、通常 MCP serve は Phase
  7.1 native compiled =nelisp-eval= が処理する (= Rust eval は subgraph
  外)
- bootstrap 自身は ~500-1000 form 程度なので 5-10x 遅くても 50-100ms
  overhead、500ms gate 内
- post-takeover のホットパスは Phase 7.1 native code 側、Rust eval は
  invoke されない

*open question*: bootstrap source 行数の予算 (= =src/nelisp-bootstrap-min.el=
は何 LOC まで膨らむか) を user signoff で確認。~2000 LOC 想定、超過時は
bootstrap subset 削減。

** 4.5 binary size 影響

Doc 32 §6.2 binary size budget: target 10MB / 許容 15MB。Rust Lisp
interpreter ~3800 LOC + dependency 0 増 (=serde_json= 既 dep) で見積
=+1.5-2.0 MB= (LTO + strip 後)。

15MB gate 余裕 (現状 6.2MB stage-d-v2.0) なので OK 範囲、ただし *cdylib
+ binary 二重 link 時 +3-4MB* に膨らむ可能性 → mitigation: =Cargo.toml=
=[features] minimal-interp= を新設、=cdylib= には含めず =bin= のみに link。

** 4.6 host Emacs fallback path 維持の必然性

Doc 32 §2.5 = dual-mode 維持 (Stage D 経路 + =--no-emacs= 経路)。本 phase
完遂後も =bin/anvil mcp serve= (no flag) は host Emacs spawn 経路で動作
するべきか?

*推奨*: *維持*。理由:
- 既存 user (Emacs power user / Doom Emacs etc) は host Emacs path で
  完全機能性 (= 全 anvil-* MCP tool ~138 個) を享受している
- =--no-emacs= path は bootstrap-min subset で起動するため tool 数 28
  程度に縮退 (= Doc 32 headless profile)
- dual-mode = 異なる user persona 対応、3 年程度は co-exist させ、v3.0
  で Emacs path deprecate 判断

* §5 Alternatives considered

** 5.1 vendor crate 採用

*** rust-lisp / rustyline / lisp-rs / lispy / emacs-rs (調査時点で agent
knowledge ベース)

| crate                | scope                              | NeLisp での適合性                  |
|----------------------+------------------------------------+------------------------------------|
| =rustyline=          | readline / line editor             | Lisp interpreter ではない、不適合  |
| =steel= (= Scheme)   | embeddable Scheme interpreter      | elisp 方言 (dynamic binding etc) と差大、port 困難 |
| =rust_lisp=          | toy Scheme tutorial                | API 不安定、依存 risk              |
| =lisp-in-rust=       | tutorial blog series               | crate 化されていない                |
| =rust_emacs= (rsx)   | =emacs=lisp= 互換 attempt            | 開発停滞、incompletetop-level     |

*結論*: vendor 採用は不適合。理由:
- elisp 方言 (=defvar= dynamic + =setq-default= + =defcustom= +
  =condition-case= + =unwind-protect= + buffer-local 等) を覆う既存
  crate なし
- Scheme crate (=steel=) は static binding + lexical-only、bridge 層が
  本実装より大きくなる
- =rust_emacs= 系は GNU Emacs C 全 port を目指す方向、本 doc の
  bootstrap-only path と思想が合わない (= Phase 5 nelisp-emacs と同じ
  問題、user 2026-04-27 vision で却下)

** 5.2 SBCL precedent (=sbcl-mini=)

SBCL の cross-build path は host Lisp で =sbcl-mini= (~3500 LOC Common
Lisp) を build → genesis で target image 構築 → cold-init で hand-off。
NeLisp Phase 7.5.4 は *Rust 側に sbcl-mini equivalent を立てる* 設計、
host Lisp なし bootstrap で SBCL pattern を踏襲。

*差分*: SBCL は host Common Lisp が必要 (=ECL/CCL/Clisp 等)、NeLisp は
*host = Rust = ubiquitous*。bootstrap host の install 摩擦 0 が SBCL
を超える点。

** 5.3 Phase 5 nelisp-emacs (= GNU Emacs C 全 port)

memory =project_nelisp.md= L1 = "Phase 5 nelisp-emacs (= years)"。GNU Emacs
C source ~30 万 LOC を Rust に全 port する path。User 2026-04-27 で却下、
本 doc の minimal-bootstrap path で代替。

*差分*: Phase 5 = ~3-5 年、本 doc = 6-12 週間。trade-off は機能性 (= GUI /
buffer / window / display engine 等は本 path では unsupported、headless
MCP serve のみ)。

** 5.4 何もしない (= Doc 32 §7.5.1-3 で完了)

Doc 32 SHIPPED 状態 = bundled-Emacs tarball (18MB) が =$ANVIL_HOME/emacs/=
に同梱、=bin/anvil= が bundled emacs を spawn する design。これでも "system
Emacs install 不要" は達成。

*差分*: bundled tarball は 18MB (vs 本 doc 7-8MB target)、起動時
fork+exec overhead (~50-100ms)、ps 上 emacs プロセス可視。User vision =
"emacs プロセス一切 spawn しない" は満たさない。
