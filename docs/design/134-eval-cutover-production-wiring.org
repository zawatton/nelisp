* TL;DR — the situation we actually found

The interpreter's *logic* is already Elisp =.o= (Doc 133 §Non-goal): the
elisp archive =libnelisp_elisp_spike.a= already defines =nl_eval_inner=, the
11 special forms (=nl_sf_quote/if/setq/progn/while/let/let_star/lambda/
function/condition_case/unwind_protect=), =nl_apply_lambda_inner=, and
=nl_eval_inner_cons=.

The build is RED for a narrow, fixable reason: commit *fa8932eb*
(2026-05-29 08:59) deleted ~30 Rust =#[no_mangle]= helpers believing the
elisp =.o= already provided them.  For *23* of them that was wrong — the
elisp =.o= eval core *calls* these helpers but does not define them
(=nm= shows them =U=, never =T= — see =project_doc133_cargo_broken_by_fa8932eb=).
So the eval =.o= is *not yet self-sufficient*: it still needs these 23
symbols to exist.  This doc plans re-providing all 23 as nelisp-sys =.o=
(zero Rust), which makes the elisp eval core self-sufficient and turns the
build GREEN — completing the eval-core cutover.

* The self-host evaluator (this session) — what it is and is not

Across 18 loop iterations this session a *complete functional Lisp
evaluator* was built in nelisp-sys and native-verified (37 ERT tests in
=packages/nelisp-sys/test/nelisp-sys-eval-kernel-test.el=): quote / if /
progn / let / setq / while / cond / lambda (immediate + closure-as-value) /
arithmetic (+,-,*,<) / env lookup+mutation / a bump allocator, all over the
*real* cons-form Sexp layout (=NlConsBox= car@0/cdr@32; tags from
=lisp/nelisp-sexp-layout.el=).

*Its role*: a PARALLEL proof-of-concept + a *pattern library*, NOT a
drop-in replacement.  It uses a simplified value model (i64 values +
ad-hoc closure boxes) and =(first-byte,length)= symbol dispatch, and the
production =Env= is an *opaque Rust struct* (passed =*mut c_void=, accessed
only via C-ABI =nelisp_env_*= functions) — incompatible with the PoC's
Sexp-alist env.  So the PoC does not directly satisfy the production
contract.  BUT its validated techniques (real-cons walking, env-alist
lookup, special-form dispatch, bump-allocated frames) are exactly what the
*bind-formals* group (D below) needs, and it is the seed of a fallback
standalone kernel (path C).

* The architectural fork (the decision this doc exists to settle)

Groups A, B, B' are *unconditionally* zero-Rust and can ship immediately
(same proven pattern as the 7 data-op symbols already re-provided).  They
drop the undefined count 23 → ~16.

The crux is Groups C and D (the env + binding core, 11 symbols).  Each of
these *needs the Env*.  Three ways to satisfy that, in preference order:

1. *Call existing C-ABI env accessors* (preferred, zero-Rust).  The
   production Env is reached through C-ABI functions
   (=nelisp_env_lookup_value=, =_set_value=, =_bind_local=, frame
   push/pop).  IF those are defined and callable from a nelisp-sys =.o=
   (via =sys:call-ptr= / extern-call), then =nl_env_*= and =nl_bf_*=
   become THIN nelisp-sys wrappers that marshal arguments and call them —
   zero new Rust.  *Open item*: confirm each accessor exists (defined, not
   also-deleted) and its exact signature.  This is the make-or-break
   verification before any code.

2. *Expose Env as a walkable Sexp alist* (medium Rust refactor).  If the
   accessors are missing, change the env representation so elisp/nelisp-sys
   can walk it directly (the PoC's nl_env_get works on a Sexp alist).  This
   is a real Rust change but is "convert opaque→transparent", not "grow the
   evaluator".  Weigh against the hard rule.

3. *Standalone nelisp-sys kernel* (largest).  Promote the self-host
   evaluator to the production eval, replacing =nl_eval_inner= itself and
   the whole env subsystem.  This deletes the most Rust but requires the
   real value model + symbol interning the PoC currently fakes.  Reserve as
   the long-horizon path / fallback.
