* TL;DR

The real Windows bottleneck collapses to *two* coupled items — the *OS
interface* and the *executable-format writer* — plus a thread/process model
rewrite.  Unlike the macOS port, the *instruction set is free* (the existing
x86_64 backend is reused) and *code signing is not required*.  The single
Windows-specific weight that macOS never had: *raw syscalls are
structurally impossible on Windows*, so every OS call must route through a
PE import table (=ntdll.dll= / =kernel32.dll=), which is inseparable from a
real PE32+ *executable* writer.

* Baseline — what exists today (confirmed from source + worklog)

** Two distinct meanings of "Windows support"

1. *AOT cross-emit* — emitting Windows PE/COFF *objects* from the elisp
   compiler.  =README.org= advertises "5 targets (incl. windows/x86_64)";
   this is object/code emission capability, *not* standalone-run-on-Windows.
2. *Standalone self-host binary running natively ON Windows* (zero-Emacs) —
   the actual goal.  Currently *Linux x86_64 ONLY*.

This doc is about (2).  Confirmed via the 2026-06-02 macOS feasibility
worklog: the standalone runtime is Linux x86_64 specific (raw syscall
=mmap=9= / =clone=56= / =fork=57= / =exit_group=231= + ELF + x86_64).

* Tier 1 — structural, architecture-incompatible (new implementation)

** B1. No stable raw-syscall ABI on Windows  [BIGGEST BLOCKER]

- Today the standalone binary talks to the OS via the raw =syscall=
  instruction + Linux numbers.  =lisp/nelisp-syscall-table.el= is
  build.rs-auto-generated *Linux values* (=read=0 write=1 mmap=9 fork=57
  exit_group=231=).
- Windows does *not* publish stable syscall numbers; they are ntdll-internal
  and change per build.  Raw syscall is structurally unusable.
- *Required*: Windows standalone must call =ntdll.dll= / =kernel32.dll=
  functions through a *PE import table*.  This is the "OS layer moves from
  raw syscall to DLL import" design fork.  Doc 132 already states it (:307-310).
- macOS hit a softer version (no-dyld impossible → dyld + libSystem
  required); Windows is stricter — the raw path is fully closed.
- *Coupling*: inseparable from B2 (the import directory lives in the PE EXE).

** B2. No PE32+ *executable* writer (only a .obj writer exists)

- ELF: =nelisp-elf-write.el= emits *ET_EXEC* (PT_LOAD + entry).  Mach-O:
  =nelisp-mach-o-write.el= emits *MH_EXECUTE* (proven exit=42 on M1).
- PE: =nelisp-pe-write.el= emits *relocatable object (.obj) only* and is
  explicitly "accepted by link.exe / lld-link" + "not yet wired into
  production STDLIB load".  External-linker dependent → violates the
  zero-toolchain standalone policy.
- *Required*: a PE32+ *EXE* writer — DOS stub + PE signature + Optional
  Header (ImageBase / AddressOfEntryPoint / Subsystem) + section table +
  *import directory (IAT / ILT)* + base relocations.  The import directory
  is the core that realizes B1.

** B3. Memory: mmap → VirtualAlloc (incl. arena base)

- The standalone arena is a raw =mmap= SeqCst-atomic bump allocator
  (=README.org:164=).
- Windows has no mmap → =kernel32!VirtualAlloc= / =VirtualProtect= via
  import.  Preferred-base / ASLR constraints differ (analogous to the macOS
  =__PAGEZERO=4GB= rebase issue).
- *Stdlib memory progress 2026-06-02*: =nelisp-os-mmap= now routes anonymous
  Windows mappings through =VirtualAlloc(MEM_COMMIT|MEM_RESERVE)=, protection
  changes through =VirtualProtect=, and release through
  =VirtualFree(MEM_RELEASE)=.  Non-anonymous / fd-backed mappings are rejected
  explicitly until Windows file mapping support is designed.

** B4. Parallel build's fork(2)/clone(2) absent on Windows

- Stage 4 N-way parallel compile uses *fork(2)* workers; the MT eval (Step 2
  Option A) uses *clone(2)* + SeqCst atomics (=README.org:145-146=).  These
  are the most recent flagship features.
- Windows has no fork → full replacement with =CreateProcess= (process
  parallelism) / =CreateThread= (threads).  The "parallel build" and "true
  multi-threaded eval" features are rewritten wholesale for Windows.
- *Minimal process progress 2026-06-02*: =nelisp-os-exit= now uses
  =ExitProcess= and =nelisp-os-getpid= now uses =GetCurrentProcessId= on
  Windows.  =fork/execve/wait/kill/getppid= still require the broader
  =CreateProcess= / HANDLE-based process model.

* Tier 2 — scaffolded but incomplete (Doc 101 / Doc 132 already started)

** B5. Win64 calling convention (Microsoft x64 ABI) emit completion

- *Scaffold present*: =nelisp-asm-x86_64.el= ABI slot =('sysv | 'win64)=;
  =nelisp-phase47-compiler--abi= dynvar.
- Remaining diff: integer args *RCX/RDX/R8/R9* (SysV: RDI/RSI/RDX/RCX/R8/R9),
  *32-byte shadow space*, different callee-saved set, xmm save rules.
- *Why Windows is easier than macOS here*: same x86_64 ISA → backend
  instructions are reused.  macOS arm64 needed *75 new ops* (7/82 implemented);
  Windows only needs the win64 ABI emit wired through every call
  site / prologue.

** B6. PE entry / CRT startup

- =nelisp-crt0.el= / =nelisp-crt0-arm64.el= assume ELF =_start=.  Doc 132
  §"mainCRTStartup / custom PE entry on Windows" (:278).

* Tier 3 — relatively easy / non-blockers (good news)

- *No code signing*: macOS required ad-hoc codesign (CodeDirectory +

* One-line summary

macOS was a triple burden — new ISA (arm64, 75 ops) + exec format + signing.
Windows trades that for *free ISA + no signing*, but carries one weight
macOS never had: *raw syscalls are impossible*, so all OS calls must route
through PE imports — which is inseparable from building a real PE32+ EXE
writer (import directory).  That single coupling (B1+B2) is the whole game.
