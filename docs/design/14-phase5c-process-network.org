** Non-goals

- *Full HTTP server* (RFC 9110 準拠) は本 Phase 範囲外。
  HTTP client (GET/HEAD + 最小 POST) のみ。server は Phase 5-D
  以降で判断。
- *WebSocket* は範囲外。Phase 6+。
- *TTY emulation / terminal renderer* (eat / vterm 相当) は
  範囲外。本 Phase は "PTY を spawn し byte 流を yield" まで。
  escape 解釈は anvil 側 or Phase 6 で。
- *Raw syscall FFI* (host C を貫通する素 syscall) は範囲外。
  host =make-process= / =make-network-process= / =file-notify-
  add-watch= を SBCL-4 核境界の内側として信頼し wrap する。
  真の syscall 化は Phase 7+。
- *TLS stack の自前実装* は範囲外。TLS は host GnuTLS / NSS を
  委譲 (=:type 'tls= wrap)。
- *package.el 互換 installer* / *HTTP proxy* は範囲外。
- *file-notify 全 backend 均質化* (inotify / kqueue / Win32) は
  範囲外。host backend を信頼し event 正規化層のみ提供。

** §2.1 Process spawn 実装

*Q*: subprocess をどう起動するか。Emacs C は =make-process= +
host OS の =fork+exec= / =CreateProcess= を使う。

選択肢:
A. Host =make-process= を wrap — =nelisp-make-process= は内部
   で host =make-process= を呼び、返った host process を
   NeLisp-side process struct で包む
B. Native syscall 化 — SBCL-4 核境界を広げて fork+exec を
   NeLisp から直接呼ぶ (Phase 7+ の FFI 前提)
C. Pool pattern — spawn は常駐 worker へ request、spawn overhead
   を削る

*推奨: A (host wrap)*。SBCL 4 核方針 (syscall は core) と
整合、anvil の実装経験で host =make-process= は信頼できる。
B は Phase 7+、C は Phase 5-C で sub-feature として必要なら
=nelisp-process-pool= を後続 §3.1b で追加可 (optional)。

** §2.2 Pipe vs PTY

*Q*: subprocess との I/O は pipe か PTY か。

選択肢:
A. Pipe-only — stdin/stdout/stderr pipe、ほぼ全 subprocess で
   可。MCP / batch compilation / worker で十分
B. A + PTY on demand — TTY app (REPL, less, vim) 向け
   =:connection-type 'pty= 対応
C. PTY-only — stream 抽象が 1 本化、pipe は廃止

*推奨: B (pipe + PTY on demand)*。anvil の MCP / worker は
pipe 前提で動く一方、eat / claude-code-ide は PTY 必須。両方
対応しないと AI-critical path の TTY broker を NeLisp 化でき
ない。anvil-pty-broker (Doc 25 Phase 1 SHIPPED) は node-pty に
逃げたが、Phase 5-C では host =make-process= =:connection-type=
経由で native 対応。

** §2.3 Process I/O と event loop の統合

*Q*: process の sentinel (state 変化) / filter (stdout data) を
NeLisp event loop (Phase 5-B.4) とどう連動させるか。

選択肢:
A. Host sentinel/filter を thin wrapper で受け、NeLisp actor
   mailbox に =nelisp-event= として post
B. 別の I/O actor を spawn し、host からの callback を直接
   受けず、NeLisp 内で select/poll 相当を回す
C. Hybrid — sentinel は wrapper、filter は pool worker

*推奨: A (thin wrapper + actor post)*。host filter は現状
=inhibit-quit= で starvation 原因 (anvil-eat feedback 参照)
だが、wrapper で *最小* に抑え、重い処理は NeLisp actor 側に
queue すれば starvation は防げる。B は event loop 2 系統化で
複雑。

** §2.4 Network client (TCP / TLS / HTTP)

*Q*: network 通信層をどう組むか。

選択肢:
A. Host =make-network-process= を wrap (TLS も host 委譲) —
   GnuTLS / NSS を信頼
B. Native TCP (NeLisp socket fd + select/poll) + TLS は host
   委譲 — fd 管理は NeLisp
C. Full native (TCP + TLS) — PKI / cert は範囲外

*推奨: A (host wrap)*。TLS stack を自前実装する余力は本 Phase
に無く、anvil-http で実証済の host =url-retrieve= /
=make-network-process= pattern は安定。Phase 7+ で syscall 化
検討。

** §2.5 HTTP client 粒度

*Q*: HTTP level の抽象化をどこまで本 Phase に含めるか。

選択肢:
A. =nelisp-http-get= / =-head= のみ (anvil-http Phase 1a 相当)
   + ETag/TTL cache
B. A + POST + multipart + streaming download
C. A + B + cookie jar + proxy

*推奨: A (最小 GET/HEAD + cache)*。anvil-http の実装
(~500 LOC Phase 1a) を NeLisp port するだけで AI-critical path
の大半はカバー。POST は MCP 用途で既に pipe 経由 (stdio JSON-

*Q*: file change notification をどう提供するか。

選択肢:

B. Poll-based 置換 — =run-at-time= + =file-attributes= の diff

** §2.8 Concurrency model

*Q*: process / connection の並行アクセスをどう制御するか。

選択肢:
A. Per-process actor — 1 process = 1 actor、sentinel/filter
   callback を自 actor mailbox へ
B. Shared I/O actor — 1 actor が全 process poll、per-process
   は受信 only
C. No actor — 従来 Elisp 流 (sentinel/filter + dynamic state)

*推奨: A (per-process actor)*。Phase 4 actor の isolation
semantics で cross-process 干渉を structural に防げる。Phase
5-B.4 event loop の延長で自然。B は poll loop 自作不要な場面
では overkill、C は anvil-eat 相当の starvation 問題を再導入
するため却下。
