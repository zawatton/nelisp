*motivation*: 現状 claude session 内で「こういう操作を MCP tool として
欲しい」と思った時の muscle memory は次:

1. session 内で Bash / Edit を多数発火、anvil-XXX.el を手で edit
2. session 跨ぎで unload-feature / require / byte-compile を手で叩く
3. =anvil-server-register-tool= 直書き、ERT も手書き
4. 動かなければ stale .elc 罠 (=feedback_canonical_byte_compile_stale_elc.md=) を手で踏む
5. session 終わると tool 仕様の rationale が memory に残らず消える
6. 1 tool 追加に *~10-30 min*、session 内 self-modify cycle は実質 ~1-2 回

= boilerplate / safety / persistence の三重コストで「自己拡張=高コスト
イベント」になっており、claude が自分の弱点を即埋める *micro-extension
loop* が回らない。

*Doc 38 SDK の存在意義*: 既存 anvil.el 資産 (server / memory / session /
orchestrator / efficiency-bundle / =anvil-server-register-tools= 関数) の
上に *claude self-modify を first-class workflow にする 6 sub-system* を
1 layer 重ねる:

- §3.A *scaffold* — 1 call で elisp + register + ERT + MCP schema 雛型
- §3.B *hot-reload* — byte-compile + load + register、失敗時 rollback、
  stale .elc 自動掃除
- §3.C *sandbox (二段階)* — *Phase v0* = =anvil-offload= isolated
  subprocess + static AST policy scan (best-effort、現実装相当)、*Phase
  v1* = real syscall sandbox (seccomp / sandbox-exec / Job Object) — 後者
  は Phase E or 9c 後送り
- §3.D *memory integration* — rationale / use-case / version を anvil-memory
  に自動 record (secret pattern scan 経由 redact 込み)
- §3.E *NeLisp execution path* — Phase 9a (=nelisp-emacs-compat= MVP) 着地
  後 dispatch、Phase 9b は coverage expansion として別節。fallback は
  **dev-mode only**、release/CI では =:nelisp t= failure は hard error
- §3.F *review / promotion gate (二分割)* — *pre-execution safety gate*
  (ephemeral 実行前 diff preview + explicit consent) と *post-execution
  promotion gate* (ephemeral → permanent registry 昇格、ERT + signoff)

** 1.3 既存 muscle memory の問題と SDK 解 (3 系統)

*問題 1: 編集 boilerplate*。tool 追加には:

mismatch (memory =feedback_anvil_macro_require.md= /

** 1.4 NeLisp Phase 7+ との関係 (= sandbox + 性能 + 永続性)

Doc 33 v2 完成後 NeLisp は *pure Lisp runtime substrate*、Doc 32 v2 で
=bin/anvil mcp serve --no-emacs= が standalone 動作。Doc 38 §3.E は本
substrate を *self-extension の sandbox host* として活用:

- *sandbox*: NeLisp runtime は host Emacs と process 分離可能、claude の
  hack tool が暴走しても main daemon には波及しない
- *性能*: Phase 7.1 native compiler で sandbox 内 elisp は AOT compile
  され、host Emacs interpreter より速い (Doc 28 v2 perf claims)
- *永続性*: =stage-d-v2.0= 単一 binary に同梱可能、claude 拡張 tool が
  永続 substrate を持つ

ただし Phase 7+ は 2026-04-25 段階で未完成 (Doc 27 §3 sub-phase 7.1-7.5
SHIPPED-but-design-only state per Doc 33 v2 §0)。本 doc §3.E は *Phase 9a
nelisp-emacs-compat 着地後* の dispatch、それまで Phase A-D は host Emacs
で完結。

** 2.7 hot-reload + rollback strategy [推奨 = A]

*** 3.C policy field

*sandbox policy* (Phase v0 + v1 共通):
- *file write*: scaffold artifact area + =/tmp/anvil-extend-*/= のみ許可、
  それ以外は =anvil-disk-warn= 経由 hard-fail (= AST scan で best-effort、
  Phase v1 で OS-level enforce)
- *shell-command*: default-deny、=:policy 'shell-allowed= で explicit
  opt-in 強制
- *network access*: **default-deny + allowlist** (= D8 反映、元 v0 の
  default-allow を反転)。=:policy 'network-allowed= で explicit opt-in、
  allowlist 形式で URL prefix 指定 (例: =:network-allowlist '("https://
  api.github.com/" "https://anthropic.com/")=)。
  - *Phase v0 enforcement layer (= spec layer のみ)*: AST scan の
    best-effort + warning。subprocess REPL 内で eval される動的 network
    access (=url-retrieve= / =make-network-process= / =gnutls-process=
    の dynamic dispatch、=(funcall (intern ...))= 経由 escape 等) を
    AST scan で完全に止めるのは **静的解析の本質的限界** で不可能
  - *Phase v1 enforcement layer (= 真の OS-level)*: iptables / pf /
    Windows Filtering Platform 経由で subprocess の outgoing socket(2)
    を OS-level enforce、これが **完全 enforcement の唯一 path**
  - = 本 Doc 38 v2 SDK の network deny は **spec layer + AST scan の
    best-effort layer**、real enforcement は OS sandbox v1 が必須
    (T89 LOCKED v2 で D-6 反映)
- *require / package-install*: default-deny + permanent promote 時
  =:requires= dependency declaration **必須化** (= D7 supply chain risk
  反映)、未宣言は scaffold-validate で hard-fail

- =anvil-extend-isolated-eval= 動作 (= future chain で control)

** 3.F 二分割 gate (= pre-execution safety + post-execution promotion)

(D6+D9 反映。元 v0 の "3-stage gate" は permanent 化 gate のみで ephemeral
実行自体の safety = 0 だった。ephemeral hack tool が user signoff 前に host
file/network/shell に触れる構造的問題を本節で解消。)

*registry model*:
- *ephemeral registry* (= 旧 ephemeral pool): in-memory hash-table、
  session-end まで in-memory。cross-session 持越しは =anvil-extend-stash
  NAME= で明示的、二系統分離 (D13)
- *permanent registry* (= 旧 permanent pool): =extend/= 配下 file + git
  管理 + module 単位 spec-list、cross-session 永続。promotion gate 通過
  した tool のみ

*** 3.F.1 pre-execution safety gate (= ephemeral 実行前)

claude が ephemeral tool を first run する前に通過必須:

1. *safety 1 (= 自動 scaffold-validate, §3.A 末尾)*: AST scan で =:body=
   の高権限 sexp 検出、:policy 宣言が無ければ hard-fail
2. *safety 2 (= 高権限 policy 必須 explicit consent)*: =:policy=
   ∈ {=shell-allowed=, =network-allowed=, =unrestricted=} の tool は
   first run 前に diff preview + explicit consent
   - default UI: CLI prompt =anvil-extend-consent-prompt= (= y/n + diff
     表示)
   - Claude Code 内 dispatch 時は permission flow 経由、claude session
     から直接 bypass 不可
3. *safety 3 (= rate-limit)*: =:policy 'unrestricted= tool は session
   あたり実行回数 hard-cap (= default 10 回 / session、defcustom 上書き可)

入口関数:
- =anvil-extend-pre-run-check NAME= = first run 前に safety 1-3 を順次
  実行、pass で =t=、fail で reason cell return
- =anvil-extend-run NAME ARGS= は内部で =pre-run-check= を呼び、未 pass
  なら error (= consent 取らずに実行しない)
- consent 与済 (= same session 内 consent 取得済 +ssame :body hash) は
  =anvil-extend--consent-cache= で短期 cache、session-end で消える

*** 3.F.2 post-execution promotion gate (= ephemeral → permanent registry)

ephemeral run + 評価が満足した後、permanent registry に昇格:

入口関数:
- =anvil-extend-promote NAME &key REVIEWER CODEX-REVIEW=
- =anvil-extend-promote-confirm NAME= (= 内部、CLI 限定で叩かれる)
- =anvil-extend-promote-confirm-cli= (= M-x command、CLI shell からの
  user signoff 専用入口)

*3-stage gate*:
1. *promotion 1 (= 自動)*: =anvil-extend-test NAME= で smoke ERT pass +
   change-log 記述必須 (= scaffold 時 :rationale または promote 時
   =:change-log STR= 必須)
2. *promotion 2 (= optional, high-risk default-on)*: =:codex-review t=
   で =anvil-orchestrator-submit= 経由 codex (or claude haiku) に review
   依頼、duplicate / risk / quality 3 軸評価。:policy ∈ {=shell-allowed=,
   =network-allowed=, =unrestricted=} の tool では default-on
3. *promotion 3 (= 必須, CLI 限定)*: user signoff、=anvil-extend-promote-
   confirm-cli= で確認

*MCP 露出 default off (D6 反映)*:
- =anvil-extend-promote-confirm-via-mcp= = nil defcustom
- = MCP transport 経由で =extend-promote-confirm= を叩く path は default
  で無効、claude session が自分で叩いて auto-promote する技術的可能性を
  **不可能** に限定保証
- user が opt-in で =via-mcp= = t に設定する場合は user 自身の判断、Claude
  Code permission flow を経由するよう spec 強制

gate 1 fail で再 scaffold 推奨、gate 2 で MAJOR REVISIONS 付き戻れば
claude が修正 reload、gate 3 deny で ephemeral 残置 (削除しない、claude が
後で再考可能)。

*demote* = =anvil-extend-demote NAME= で permanent → ephemeral 戻し可能、
delete = =anvil-extend-delete NAME &optional REGISTRY-KIND= で完全削除。

*Phase F 完遂 gate*:
- pre-execution safety gate 動作 (= safety 1-3 各 1 ケース)
- post-execution promotion gate 動作 (= smoke / codex-review / user
  signoff 各 1 ケース)
- =anvil-extend-promote-confirm-via-mcp= = nil で MCP 経由 auto-promote
  が hard-fail
- =anvil-extend-promote-confirm-cli= 経由 CLI 限定 signoff 動作
- ephemeral / permanent 状態遷移動作
- demote / delete 動作

** 6.1 isolation escape (= Phase v0 isolated subprocess limit)

(A2 反映で全面 revise。"sandbox" 用語は doc 全体で "isolated subprocess"
+ "policy enforcement" に二分、本節は Phase v0 の現実評価。)

*** 6.1.v0 Phase v0 (= 現実装で組める best-effort)

- *リスク (= Phase v0 で技術的に防げない事)*:
  - claude が =shell-command-to-string "rm -rf $HOME"= を書くと host の
    HOME が消える (= subprocess は host filesystem を share、syscall
    filter なし)
  - claude tool が =/etc/passwd= 等 sensitive file を read/write
  - main daemon を fork bomb で落とす (= RLIMIT_NPROC 設定なし)
  - 無限 loop / huge alloc は =anvil-future-kill= 経由 user 介入 or
    手動 cancel が必要 (= 自動 timeout 未実装、=:timeout= は plist key
    のみで Reserved for Phase 3 preemption)
- *Phase v0 で提供できる対策 (= best-effort)*:
  - *process isolation*: subprocess REPL で eval、暴走しても main daemon
    に直接波及しない (= anvil-offload baseline、anvil-offload.el:443)
  - *static AST policy scan (scaffold-validate)*: =:body= の高権限 sexp
    検出、未宣言 hard-fail
  - *high-risk policy explicit opt-in*: =:policy 'shell-allowed= /
    =network-allowed= / =unrestricted= は scaffold 時 explicit 宣言必須
  - *pre-execution safety gate (§3.F.1)*: high-risk policy tool は first
    run 前に diff preview + explicit consent
  - *=anvil-future-kill= による hard-kill*: 暴走中 subprocess を強制終了
  - *監査*: anvil-memory に rationale + policy が record される、後 audit
    可能
- *Phase v0 で提供できない対策 (= 誇大広告しない)*:
  - file write の OS-level deny (= filesystem syscall filter shipped 上に無し)
  - network deny の OS-level enforce
  - syscall filter (= seccomp BPF profile shipped 上に無し)
  - RLIMIT_AS / RLIMIT_NPROC 自動設定 (= grep 0 hit、未実装)
  - 自動 timeout enforce (= Reserved for Phase 3 preemption、shipped で
    self-enforce はせず)
  - 別 UID 起動 (= optional Phase E のみ)

*** 6.1.v1 Phase v1 (= real syscall sandbox、新規実装、Phase E or 9c に後送り)

- *対策 (新規 +400-600 LOC 見積)*:
  - *Linux*: seccomp BPF profile (file write deny + network deny + ptrace
    deny + setuid 系 deny) + RLIMIT_AS / RLIMIT_NPROC / RLIMIT_CPU
  - *macOS*: =sandbox-exec -p PROFILE_FILE= (= Apple Sandbox SBPL)
  - *Windows*: Job Object + restricted token + Memory limit
  - timeout の OS-level enforce
- *Phase v1 完遂後の保証 level*: claude が =rm -rf $HOME= を書いても OS
  が deny する level までは sandbox profile で抑止可能 (= profile が穴だ
  らけでなければ)。但し SBPL / seccomp profile design 自体に bug がある
  場合は当然抜ける。
- *Phase v1 着手 timing*: Phase E (NeLisp runtime path) と並列 or Phase
  9c で別 doc 化検討。

** 6.2 ephemeral registry memory leak + GC visibility

(A5 反映: =on-stop= は anvil-session に存在しない、Stop event + extend
cleanup 新規実装。D10 反映: 1h timeout 前に 5 min warning + =:expires-at=
表示。)

- *リスク*: ephemeral registry は in-memory hash-table、session 終了で
  cleanup されないと長時間 daemon 内で蓄積。1h timeout gc も user 不可視
  なら気付けない
- *対策*:
  - *Stop event 経由 cleanup (新規実装)*: =anvil-session= lifecycle event
    =stop= に =anvil-extend--ephemeral-cleanup= を追加実装。anvil-session
    に generic =on-stop= callback registry は **存在しない** (anvil-
    session.el:437-443) ので、Doc 38 SDK 側で Claude hook =stop= event +
    =anvil-compact-on-stop= の延長線で cleanup hook を新規実装 (= ~30 LOC
    程度の追加)
  - 念押し: 1h timeout で stale ephemeral を gc (= last-access >= 1h ago
    で auto-unload)
  - *5 min warning (D10)*: 1h timeout 前 5 min で warning event を anvil-
    session log に記録、user が気付ける
  - *=:expires-at= 表示*: =extend-list= MCP tool 出力に =:expires-at=
    timestamp 表示
- *監査*: =extend-list 'ephemeral= で常時 visible、過剰蓄積 + 直近 expire
  予定を user が気付く

** 6.3 promotion gate manipulation (= claude による auto-promote 抑止)

(D6 反映: 元 v0 の "claude が auto-promote 不可" 主張は技術的に enforced
されていなかった = MCP 経由 =extend-promote-confirm= を claude が叩ける
構造的問題。本 revise で MCP 露出 default off + CLI 限定で技術的不可能化。)

- *リスク*: claude session が gate 1 (smoke ERT) を 1 件 trivial test で
  通して promote、または =extend-promote-confirm= MCP tool を自分で叩いて
  user signoff を **bypass**、permanent registry に低品質 tool が蓄積
- *対策 (= 技術的に不可能化)*:
  - *smoke ERT 必須項目*: scaffold が出力する skeleton は =:body= に応じた
    input/output 型 + 範囲 check + error path 1 件を **必須化**、trivial
    test で通せない
  - *codex review (promotion 2) は high-risk policy default-on*: =:policy
    'shell-allowed= / =network-allowed= / =unrestricted= tool では default-on、
    user opt-out
  - *user signoff (promotion 3) は CLI 限定*: =anvil-extend-promote-
    confirm-via-mcp= = nil defcustom で MCP 露出 default off、claude
    session が =extend-promote-confirm= を MCP 経由で叩く path を技術的に
    **不可能** にする
  - =anvil-extend-promote-confirm-cli= は M-x command + interactive
    confirm + diff preview、CLI shell 経由 user 自身でのみ叩ける
  - user opt-in で =via-mcp= = t にする場合は Claude Code permission flow
    を経由するよう spec 強制 (= permission prompt 経由ならば user 介入が
    必要、claude 単独 promotion は不可)
- *監査*: permanent registry は git 管理、PR review = user の追加 line of
  defense

** 6.4 NeLisp 実行 path の maturity 依存 + network default-deny enforce 不在 (Phase E gate)

(D2/D3 反映: hard dependency narrow to 9a only。D8 反映: network default-

- *リスク*:
  - Phase E hard dependency は **Phase 9a only** (=nelisp-elisp-compat=

  - 元 v0 の network default-allow は self-modify 文脈で **危険** = supply
    chain / exfiltration の主経路。現実装に network deny enforce 機構
    なし (anvil-offload subprocess は host network namespace 共有)
- *対策*:
  - *Phase 9a only narrow*: Doc 38 §3.E hard dependency = Phase 9a only、
    Phase 9b は coverage expansion として別節 (§3.E.coverage)
  - *fallback dev-mode only (D3)*: release/CI で =:nelisp t= failure は
    hard error (=anvil-extend-release-mode= = t)、=:nelisp-required t=
    flag で fallback 拒否
  - *availability check 3 段階 (D4)*: =featurep= + cheap probe + version
    check
  - *network default-deny + allowlist (D8)*: §3.C policy field で
    default-deny に反転、=:policy 'network-allowed= explicit opt-in、

  - Phase E 着地は 2026-Q4-2027-Q1 想定 (Doc 32 v2 / Doc 34 想定)、Doc 38

** 6.7 supply chain risk (= 新設、claude が外部 code を fetch)

(D7 + D8 統合反映、新設。元 v0 §6 に列挙されておらず、self-modify 文脈で
網羅的に扱う必要あり。)

- *リスク*:
  - claude が =shell-command-to-string "curl X | sh"= で任意 binary を
    実行
  - claude が =package-install= / =package-vc-install= で任意 ELPA
    package を導入
  - claude が =require 'foo= で未宣言 dependency を pull (= host Emacs
    の =load-path= 経由 stale module)
  - permanent promote 時に dependency 宣言が無いと再現性破綻 + 後 session
    で「何をどこから入れたか」不明
- *対策*:
  - *AST scan で =shell-command= URL allowlist 強制*: =:body= 内
    =shell-command= 系 sexp を AST 走査、URL pattern (=https?://[^|]+= の
    pipe-to-shell 形) を allowlist 照合、未許可 URL は hard-fail
  - *=:requires PKGS= dependency declaration 必須化 (permanent promote 時)*:
    permanent registry 移動時に =:requires= plist 必須、未宣言 =require=
    / =package-install= は scaffold-validate で hard-fail
  - *=:requires= 拡張 PR は user signoff 二段階*: 既存 permanent tool に
    新規 =:requires= 追加 (= dependency surface 拡大) する場合は通常
    promotion gate に加えて user 二段階 signoff (= "新規 dependency 受入"
    confirm + "promotion confirm")
  - *network default-deny + allowlist (D8)*: §3.C 参照、URL prefix
    allowlist で exfiltration 主経路を抑止
- *監査*: permanent registry の git diff で =:requires= 変更が tracking、
  PR review 経路で user が新規 dependency を必ず認識

*** 6.7.1 implementation sketch (= T89 LOCKED v2 で D-5 反映、二段階分離)

AST URL allowlist + =:requires= dep declaration の **実装具体性** を二段階
で分離 (= LOC 見積 swing を抑える):

- *v0 = 簡易 grep base (Phase A-D 着手範囲、~50-80 LOC)*:
  - =:body= sexp を =cl-prettyprint= で text 化、=regex
    "\\bcurl[^\\n]*https?://[^|\\n]+"= で =curl https?://X= 形を grep
  - 検出 URL を =anvil-extend-url-allowlist= defcustom に照合、未許可で
    hard-fail
  - =require= / =package-install= sexp は =funcall (intern ...)= 形で
    obfuscation 可能だが、簡易 grep で literal sym arg のみ検出 (=
    best-effort)
  - 実装 effort: ~50-80 LOC、Phase A AST scan の延長で着地
- *v1 = data-flow analysis base (将来別 phase、Phase E or 9c 後送り、
  ~200-400 LOC)*:
  - =semantic.el= 等 AST analyzer base、=shell-command= call の引数 sexp
    を data-flow trace
  - dynamic dispatch (=funcall (intern (concat "shell-" "command"))= 等)
    の追跡を試みる、ただし完全捕捉は **静的解析の本質的限界** で不可能
  - OS-level enforcement (= Phase v1 syscall sandbox + iptables / pf /
    WFP) と組み合わせて **runtime layer での enforce が必須**
  - 実装 effort: ~200-400 LOC、Phase C v1 と同 timing で並行検討
- *境界*: 簡易 grep base で **大半の素直なケース** は捕捉可能、obfuscated
  / dynamic dispatch ケースは v0 で漏れる前提で OS-level layer に依存。
  v0 単独で「supply chain risk = 0」は主張しない、AST 静的解析 + runtime
  OS sandbox + user signoff の **三段防御** で総合的に抑止

*** 8.1.a ephemeral scaffold + load + smoke (= happy path、LLM body 1 発生成 + reload)

- *current* (SDK 前): claude session 内 1 tool 追加 = ~10-30 min
  - breakdown: handler defun ~3 min + register call ~1 min + ERT skeleton
    ~3 min + module init/unload 経路 ~2 min + reload trial-and-error ~5
    min (stale .elc 罠等) + memory rationale 手書き ~3 min + 余白 ~5 min
- *target* (SDK 後 Phase A-D、= happy path): ~30-60s
  - breakdown: =extend-rationale-search= ~3s + =extend-scaffold= 1 call
    ~5s + LLM =:body= 生成 ~10-20s (= LLM token 生成 reality) +
    =extend-load= ~5-10s (byte-compile + register + tools/list_changed
    push 込み) + =extend-test= ~5s + 余白 ~5-10s
- *speedup*: **5-10x** (= 元 v0 の 20-30x は LLM 生成 + retry cycle +
  設計コストを無視した楽観値、現実の happy path は ~3-5 min/tool に
  着地する可能性あるが、SDK で boilerplate 抑止により 5-10x が現実値)

*** 8.1.b permanent promote (= signoff 待ち + git PR 込み、3 段階 KPI 細分)

- *current* (SDK 前): permanent 化 = anvil 本体 PR 経路、~1日 - 数日
  (= review iteration + merge wait)
- *target* (SDK 後 Phase A-F.2、T89 LOCKED v2 で D-7 反映で 3 段階 KPI に
  細分):
  - *"happy"* = signoff + git commit が ~10 min 以内 (= same-day promotion、
    smoke ERT 自動 ~30s + codex review skip or default-off + user CLI
    signoff 即対応 + git commit ~30s)
  - *"long"* = ~1 hour 以内 (= reviewer iteration 1 round 込み、
    =anvil-orchestrator-submit-and-collect= の 30s-数分 想定 anvil-
    orchestrator.el:3229 と整合 + user signoff iteration 1 回 + body
    rewrite 1 回)
  - *"failed"* = >1 hour で gate 残置 → ephemeral 継続 fall back (= user
    signoff 待ち長期化 / codex review MAJOR REVISIONS 連続 / body rewrite
    失敗等)
- *speedup*: "happy" 体感 ~10-50x (= anvil 本体 PR review より軽量)、
  "long" は ~5-10x、"failed" は SDK 後でも anvil 本体 PR と同等
- *KPI 観測 method*: =anvil-extend-promote= 実行から =anvil-extend-promote-
  confirm-cli= 通過 (or 1 hour timeout) までの elapsed を =anvil-state=
  SQLite に記録、=extend-list 'permanent= に happy/long/failed ratio を
  表示
- 注: retry / failed signoff / body rewrite 込みで現実値は更に長く、楽観
  値ではない。"failed" ratio が >30% で SDK の promotion gate 設計を見直し

- *発生条件*: claude が「あれば便利」と気付くたびに即 scaffold、捨てる前提

** 8.3 ephemeral / permanent ratio

- *target*: ephemeral : permanent = 5:1 - 10:1
  - 多くは「session 内一回使って捨てる」 = ephemeral
  - 真に有用なものだけ promotion gate 通過 = permanent
- ratio が低い (= permanent 過多) なら gate 緩め、高い (= permanent 少)
  なら scaffold 過多/品質低、で運用 1-3 ヶ月 calibrate
