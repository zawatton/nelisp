** §2.2 dispatch arm 67 件の categorization

source-grep で発見した 67 個 (= =^\s*"nelisp--= count) の dispatch
arm を Tier 別に分類:

*** §2.2.1 Tier 1: pure elisp 移管可 (= 推定 ~30 arm)

新 ABI mode 不要、既存 =nl-jit-call-out-*= bridge で対応可能、または
elisp で直接実装可能:

- 算術系 (= div / sxhash / equal / ref-eq) ← Doc 84 §84.1 同 pattern
- predicate (= integerp / floatp / stringp / consp / etc.)
- type 系 (= type-of)
- record 系 (= make-record / record-ref / record-set / record-length /
  record-type / recordp)
- intern / make-symbol / symbol-name
- 単純な string 系 (= string-bytes)

*** §2.2.2 Tier 2: 新 ABI mode + bridge primitive 必要 (= 推定 ~25 arm)

Doc 81 trampoline ABI を拡張する必要、または特殊 calling convention:

- =bi_make_mut_string= (= 可変長 byte array)
- =bi_concat_ints= (= variadic byte stream)
- =bi_string_match_p= (= regex match、外部 backend 必要)
- =bi_syscall_canonicalize= (= path normalize、syscall 連携)
- ffi 系 (= libc bridge 経由)

*** §2.2.3 Tier 3: 環境への副作用が深い (= 推定 ~12 arm)

=Env= 内部 state を直接読み書き、§86.3 (= env.rs elisp 化) と統合する
必要:

- special form 関連 (= setq / set / boundp / unbound)
- scope chain manipulation
- dynamic binding stack push/pop

** §2.3 helper fns 削減先

- Tier 3 の bi_* = §86.3 完了まで残置

** §3.2 elisp signal contract への移管

elisp 側には既に =signal= / =error= / =wrong-type-argument= /
=void-variable= 等の condition system が存在 (= Doc 80 pre-stdlib
substrate で Stage 80.1〜80.5 で establish 済)。

移管 surface:
- Rust =EvalError::WrongType= → elisp =(signal 'wrong-type-argument
  args)=
- Rust =EvalError::ArithError= → elisp =(signal 'arith-error args)=
- Rust =ReadError::Lex= → elisp =(signal 'invalid-read-syntax args)=
- etc.

eval loop 内部で =Result<Sexp, EvalError>= を扱う Rust 部分を、elisp の
=condition-case= / =unwind-protect= に切り替える。

Rust 側に残るのは =EvalError::Internal= 1 variant (= elisp で recover
不能な panic 等価)。残 ~30-50 LOC。

* §4 env.rs 解剖 (= 564 LOC)

** §4.1 構造

- =SymbolEntry= struct (line 47, ~40 LOC) - per-symbol value + plist +
  function slot
- =Env= struct (line 92, ~60 LOC) - global symbol table + lexical
  scope stack + special variable bindings
- impl Env (= 推定 line 150-560) - lookup / set / push-scope /
  pop-scope / bind-special / etc.

** §4.2 elisp 化の工法

Env / SymbolEntry を elisp 側で表現するには:

*** §4.2.1 option A: alist + hash table

- 全 global symbol = =nelisp--global-env= hash-table
- lexical scope stack = elisp =let= form の native stack 利用
- special binding = elisp =defvar= / =setq-default= 利用

trade-off: elisp の native scoping を全活用 = elegant だが、Rust 側で
保持していた C-level efficient lookup を失う

*** §4.2.2 option B: payload pointer 経由 (= Doc 81 ABI mode 拡張)

- Env / SymbolEntry の memory layout を pin
- elisp が payload pointer 経由で直接アクセス
- Tier 1/2 と互換性高い

trade-off: Rust 側に layout pin が残る、純 elisp 度が落ちる

*** §4.2.3 推奨

option A を base、Tier 2 builtin が hot path として hit する symbol
lookup のみ option B (= cache 構造)。

* §7 §86.3 env.rs scope chain elisp 化

** §7.1 scope

- =Env= struct を elisp =nelisp--global-env= hash-table に置換
- =SymbolEntry= struct を elisp plist (= =(:value V :function F :plist
  P)=) に置換
- lexical scope stack を elisp native =let= で実現
- special binding stack を elisp =defvar= で実現
- Tier 3 dispatch arm (= 12 個) を §86.1 完了後に統合

** §7.2 elisp implementation strategy

3-step implementation:

1. *§86.3.a precursor*: =nelisp--global-env= hash-table 作成、空 stub
   (Rust の Env と coexist)
2. *§86.3.b shadow path*: elisp 側で env-lookup を実装、Rust の Env と
   同期 (= dual-path、performance 検証用)
3. *§86.3.c switch-over*: Rust Env 削除、全 lookup を elisp に switch、
   連動 file の env reference を elisp 化

** §7.3 連動削減

- =eval/mod.rs= で =Env= field reference を elisp 経由に書き換え =
  -200 LOC
- =eval/special_forms.rs= で setq / let / lambda の env 操作を elisp
  に委譲 = -300 LOC
- =eval/ffi.rs= で env-aware marshalling を elisp に移す = -100 LOC
- =eval/builtins.rs= 残 Tier 3 arm 消滅 = -200 LOC

** §7.4 工数

- §86.3.a precursor: 3 day
- §86.3.b shadow path: 5 day
- §86.3.c switch-over: 5 day
- 合計: ~13 day

** §7.5 risk (= self-reference / infinite loop)

elisp が自身の env を実装すると、=defvar= macro 自体の expand 中に env
lookup が必要 = chicken-and-egg。

解決策:
- Tier 0 minimal env を Rust に残置 (= ~50 LOC)、defvar / setq-default
  / let の最低限の dispatch のみ提供
- elisp が自身を bootstrap してから extended env を起動

これが最大の risk。precursor stage で実機検証必要。

* §9 Risk / unknowns

** §9.1 perf regression

builtin dispatch の hot path が elisp に移ると、interpreter overhead +
ABI marshalling cost が積もる。Tier 1 移管前後で micro-bench 必要。

対策:
- nelisp-cc の AOT compile で Tier 1 elisp wrapper も native code 化
- =:trampoline-*= ABI mode 経由で marshalling overhead を最小化
- bench gate (= fib 30x perf) に Tier 1 wrapper を含む

** §9.2 type marshalling overhead

Sexp の box accessor を elisp 経由で叩くと、Rust 直接 access に対して
~10x overhead の可能性。Doc 84 §84.3 で box accessor を elisp 化した
pattern を流用。

** §9.3 Tier 3 → §86.3 cycle

builtins Tier 3 (= 12 arm、env 操作系) を §86.1 で elisp 化しようと
すると env.rs の elisp 化が precondition だが、env.rs elisp 化には
Tier 1 の minimal elisp infrastructure が precondition = chicken-and-egg。

解決策:
- §86.3.a precursor で minimal env を elisp 側に確立
- §86.1 Tier 3 は §86.3.a 完了後に着手 (= 直列依存)

** §9.4 Doc 83 §6.2 試算との乖離

Doc 83 が示した「Phase 2 着地 ~6,000-8,000」は Phase 3 連動削減を含む
値。本 Doc 86 は Phase 2 単独 = ~17,000-19,000 着地。差分 ~10,000 は
Phase 3 (= 究極 D-B / Doc 77c) で別途削減する旨を Doc 83 §6.2 に追記
する update が必要。

** §9.5 ERT coverage

builtins.rs 3,804 LOC の全 arm が elisp 化されると、現 test suite (=
=test/= 配下 ~2,700 ERT) が引き続き green を保つこと、新 elisp wrapper
の round-trip ERT を追加することの両方が必要。
