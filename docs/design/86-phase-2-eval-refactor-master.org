* §4 env.rs 解剖 (= 564 LOC)

** §4.1 構造

- =SymbolEntry= struct (line 47, ~40 LOC) - per-symbol value + plist +
  function slot
- =Env= struct (line 92, ~60 LOC) - global symbol table + lexical
  scope stack + special variable bindings
- impl Env (= 推定 line 150-560) - lookup / set / push-scope /
  pop-scope / bind-special / etc.

** §4.2 elisp 化の工法

Env / SymbolEntry を elisp 側で表現するには:

*** §4.2.1 option A: alist + hash table

- 全 global symbol = =nelisp--global-env= hash-table
- lexical scope stack = elisp =let= form の native stack 利用
- special binding = elisp =defvar= / =setq-default= 利用

trade-off: elisp の native scoping を全活用 = elegant だが、Rust 側で
保持していた C-level efficient lookup を失う

*** §4.2.2 option B: payload pointer 経由 (= Doc 81 ABI mode 拡張)

- Env / SymbolEntry の memory layout を pin
- elisp が payload pointer 経由で直接アクセス
- Tier 1/2 と互換性高い

trade-off: Rust 側に layout pin が残る、純 elisp 度が落ちる

*** §4.2.3 推奨

option A を base、Tier 2 builtin が hot path として hit する symbol
lookup のみ option B (= cache 構造)。

** §7.5 risk (= self-reference / infinite loop) — v2 reconciled

* §9 Risk / unknowns

** §9.1 perf regression

builtin dispatch の hot path が elisp に移ると、interpreter overhead +
ABI marshalling cost が積もる。Tier 1 移管前後で micro-bench 必要。

対策:
- nelisp-cc の AOT compile で Tier 1 elisp wrapper も native code 化
- =:trampoline-*= ABI mode 経由で marshalling overhead を最小化
- bench gate (= fib 30x perf) に Tier 1 wrapper を含む

** §9.2 type marshalling overhead

Sexp の box accessor を elisp 経由で叩くと、Rust 直接 access に対して
~10x overhead の可能性。Doc 84 §84.3 で box accessor を elisp 化した
pattern を流用。

すると env.rs の elisp 化が precondition だが、env.rs elisp 化には
Tier 1 の minimal elisp infrastructure が precondition = chicken-and-egg。

解決策:
- §86.3.a precursor で minimal env を elisp 側に確立
- §86.1 Tier 3 は §86.3.a 完了後に着手 (= 直列依存)
