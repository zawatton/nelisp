* §0 要旨 (TL;DR)

Doc 34 (Phase 9b、LOCKED-2026-04-25-v2) は text-property *MVP* を §4.6 に
12 件の API contract で凍結したが、Doc 34 §2.6 と §8.7 は以下を *明示
scope 外* として後送りしている:

- overlay semantics 全般
- overlay-aware =get-char-property=
- text-property keymap
- text-property =display= property の actual render
- text-property =face= property の actual render
- =modification-hooks= / =insert-in-front-hooks= / =insert-behind-hooks=
  / =point-entered= / =point-left= の text-property hook 系
- overlay change hook (=overlay-change-functions=)

本 doc (= Doc 41) はこれら *Phase 9b text-property advanced features* を
Layer 2 (=nelisp-elisp-compat=) で完成させる *Phase 9c* の design draft
である。Phase 9b と同じ "interval tree storage 上に拡張機能を積む"
方針を取り、render 自体は Phase 11 display backend に委譲する。

*担当 scope*:
- 9c.1 face property + face inheritance + face spec resolver (storage / query side)
- 9c.2 display property (string / image / margin / nested spec) (storage / query side)
- 9c.3 text-property keymap dispatch (keymap chain への inject 経路)
- 9c.4 overlay full (interval-overlay tree + priority + after-string / before-string)
- 9c.5 text-property change hook + point-entered / point-left + overlay change hook

*scope 外*:
- actual render (= bitmap / TUI cell 配置 / image decode / glyph layout)
  → Phase 11 display backend
- font / face attribute の OS 依存 fallback
- overlay の display-engine integration ( Phase 11)
- tooltip / mouse-face の event dispatch ( Phase 11 と co-design)
- composition / bidi / stretch glyph (= Phase 11+ で扱う)
- 詳細は §8 で列挙

*v0 で固定したいこと*:
- overlay は interval-tree based、text-property tree とは別物として
  per-buffer に独立保持する (Emacs precedent と整合)
- =get-char-property= の precedence を Doc 34 §2.6 から拡張し
  *overlay > text-property > default* の 3 段に固定
- text-property keymap は Doc 34 §2.3 の keymap precedence chain に
  *overlay keymap と同列で挿入* する (= 9c で precedence table を拡張)
- face / display property の値 schema は *opaque payload + accessor
  helper* に分離し、render は backend へ委譲する pure-data design に
  する

*ship gate 要約*:
- 9c.1 〜 9c.5 を sub-phase ごとに ERT + integration + scenario gate
  通過
- 9c.4 overlay は stub render (= no-op render backend) で暫定 SHIPPED、
  Phase 11 actual render 接続後 full SHIPPED
- 9c.3 keymap inject は Doc 34 §2.3 precedence の *差分 errata* を
  本 doc 側で扱う option B 方式 (Doc 34 LOCKED は不変)、ただし
  *contract version 並走 model* (= 既存 Doc 34 chain は v=1 で pin、
  本 doc が拡張する 9 段 chain は =KEYMAP_CHAIN_INJECT_CONTRACT_VERSION
  = 1= かつ Doc 34 §2.3 chain が =KEYMAP_CONTRACT_VERSION = 1 のまま)
  で互換性確保。詳細は §2.5 + §6.2
- =get-char-property= は contract v1 → v2 増分、ただし *de facto 非
  breaking* (= overlay 0 件時 v1 と byte-identical)、Emacs precedent
  と整合する正方向 fix。詳細は §4.5 + §6.3 + §7.5
- *parallel set escape hatch*: {9c.1, 9c.2, 9c.3} は dependency 上
  独立、anvil leverage で同時 dispatch 可、最良 case で 6-8 ヶ月 →
  4-6 ヶ月圧縮可能 (詳細は §3 table)
- *LOC budget* (POLISH-1 反映): §2.9 で *production code* base 4200

** 2.1 face property storage [推奨 = A interval tree value layer reuse]

| 候補 | 戦略                                                  | 利点                              |
|------+-------------------------------------------------------+-----------------------------------|
| A    | Doc 34 §2.6 interval tree の payload に face value 載せ | storage layer 1 本化、cleanup 共通 |
| B    | face 専用 secondary tree                              | query 高速化期待、複雑性増         |
| C    | per-char vector                                       | memory 過大                       |

*推奨 A 理由*:
- Doc 34 MVP interval tree がすでに gap-buffer adjust contract を持つ
- face 値は opaque payload で良いので tree shape を変える必要なし
- range query / boundary query の semantics を Doc 34 §4.6 と統一できる

** 2.2 face spec value schema [推奨 = A nested face spec list]

| 候補 | 戦略                                                | 利点                             |
|------+----------------------------------------------------+----------------------------------|
| A    | =face-symbol= / =(face-symbol ...)= / =(plist)= 3 形式受入 | Emacs precedent 互換             |
| B    | symbol 単独のみ                                     | 互換性不足                        |
| C    | 専用 record type                                    | Emacs 既存 spec から conversion 必須 |

*推奨 A 理由*:
- =put-text-property= で受け取れる face value の Emacs 既存形は 3 種ある
- 3 形式 normalize は pure data 処理で副作用なし
- =face-attribute-merge= の resolver が単一 in-memory model に集約できる

*schema (DRAFT v0)*:

#+begin_src text
face-spec ::= face-symbol
            | (face-symbol ...)               ;; cascade list
            | (:foreground STR :background STR :weight SYM ...)  ;; raw plist
            | nil
#+end_src

** 2.3 face inheritance / merge resolver [推奨 = A pure left-to-right merge]

| 候補 | 戦略                                                  | 利点                            |
|------+-------------------------------------------------------+---------------------------------|
| A    | left-to-right plist merge + =:inherit= chain expansion | Emacs precedent 互換             |
| B    | right-to-left merge                                   | 既存 Emacs Lisp が逆解釈する risk |
| C    | merge 不要 (= 最終 face のみ)                          | inherit chain が機能しない       |

*推奨 A 理由*:
- Emacs =face-attribute-merge= と semantic 一致
- inherit chain (=:inherit some-other-face=) を pure に展開できる
- resolver は backend 不要、本 doc 単体で freezable

*resolver contract*:
- =nelisp-face-resolve (face-spec) -> normalized-attribute-plist=
- =:inherit= は depth ≤ 16 で truncate (= Emacs と同様 cycle 回避)
- =unspecified= 値はそのまま残し、backend が解釈

| C    | display 専用 record type                               | conversion 必須                       |

** 2.5 text-property keymap chain inject [推奨 = A contract version 並走 model]

| 候補 | 戦略                                                  | 利点                            |
|------+-------------------------------------------------------+---------------------------------|
| A    | Doc 34 §2.3 chain は LOCKED 不変、本 doc が新 contract =KEYMAP_CHAIN_INJECT_CONTRACT_VERSION = 1= で *9 段 chain provider* を提供。Doc 34 SHIPPED bin caller は v=1 chain (= 7 段) のまま、本 doc 適用後 caller は v=2 chain (= 9 段) を opt-in (=nelisp-keymap-chain-with-textprop= デフォ nil) | Doc 34 LOCKED 不変、option B 整合、breaking change なし |
| B    | Doc 34 LOCKED amend (= option A errata 別 task)        | scope creep、Doc 34 LOCKED v3 化必要 |
| C    | precedence chain 全面再設計                            | 互換性破綻                       |

*v1 推奨 A 理由 (MAJOR-A.1 反映)*:
- v0 の "Doc 34 LOCKED 文書を編集しない = option B 整合" だけでは、
  Doc 34 §4.3 lookup-key API の binary semantics が変わる (= 7 段 →
  9 段)。Doc 34 §2.3 explicit exclusions (= text-property keymap /
  overlay keymap / transient maps beyond above precedence) を本 doc
  が runtime semantics で覆す行為で、option B の本質と本質 conflict
- v1 では *contract version 並走 model* に切替え、以下の方法で互換維持:
  - Doc 34 §2.3 chain provider (= 7 段) は =KEYMAP_CONTRACT_VERSION = 1=
    のまま不変、Doc 34 SHIPPED bin の caller はそのまま使用継続
  - 本 doc は新 contract =KEYMAP_CHAIN_INJECT_CONTRACT_VERSION = 1=
    で *9 段 chain provider* を提供、call site が opt-in flag
    =nelisp-keymap-chain-with-textprop= = t で v=2 (= 9 段) を選択
  - =lookup-key= / =command-execute= は Doc 34 §4.3 既存 API 不変、
    本 doc 9c.3 が *新 chain provider helper* =nelisp-keymap-chain-at=
    を追加、Doc 34 既存 helper は不変
- *option A 切替の余地* (= Doc 34 §2.3 errata 別 task で v3 化) は §7.4
  に LOCKED 前 user signoff の選択肢として残置

*precedence (本 doc 拡張 LOCKED v1、9 段、=nelisp-keymap-chain-with-textprop= = t 時のみ)*:
1. =overriding-terminal-local-map=
2. =overriding-local-map=
3. =minor-mode-overriding-map-alist=
4. =minor-mode-map-alist=
5. =emulation-mode-map-alists=
6. *overlay keymap at point* (本 doc 9c.4、=keymap= prop > =local-map= prop)
7. *text-property keymap at point* (本 doc 9c.3)
8. =current-local-map=
9. =global-map=

*sub-precedence detail (MAJOR-A.3 反映、decision 格上げ)*:
- *overlay slot 内* (= slot 6):
  - 第一 key: overlay =priority= property 降順
  - =priority= = nil の場合は -∞ 扱い (= 全 numeric priority より下)、
    Emacs precedent (=keymap.c=) と一致
  - 同 priority の二次 key: overlay insertion 順 (= 後から作った overlay
    が後)、§9.4 を decision に格上げ
  - 同 overlay 内: =keymap= property > =local-map= property (= Emacs
    precedent と一致、=keymap= が先 lookup される)
- *text-property slot 内* (= slot 7):
  - 第一 key: =keymap= property を持つ range の最短 (= 最内側 range)
  - 同 length range の二次 key: range 開始点 後優先 (= start position
    が大きい方が勝ち、Emacs precedent と一致)
- *cross-slot*: slot 6 (overlay) unconditionally > slot 7 (text-property)

*v1 → v2 並走 invariant (MAJOR-A.1 反映)*:

** 2.8 modification hook trigger [推奨 = A range-overlap fan-out]

| 候補 | 戦略                                                  | 利点                                  |
|------+-------------------------------------------------------+---------------------------------------|
| A    | mutation の range と重なる全 hook を fan-out           | Emacs precedent 互換                   |
| B    | mutation point だけで判定                              | range hook が漏れる                    |
| C    | hook 廃止                                             | package 互換性破綻                     |

*推奨 A 理由*:
- =modification-hooks= property を持つ range と重なる mutation で trigger
- =insert-in-front-hooks= は range start = mutation start で trigger
- =insert-behind-hooks= は range end = mutation end で trigger
- overlay hook も同様の fan-out で trigger

*re-entrancy semantics (LOCKED v1、MAJOR-B.3 反映)*:
- hook 内 mutation は *secondary trigger を抑制* する depth=1 ガードを持つ
- ガード切替は Emacs と同様 =inhibit-modification-hooks= 変数で行う
- 本 doc 9c.5 で導入する変数:
  - =inhibit-modification-hooks=
  - =inhibit-point-motion-hooks= (point-entered / left 用)

*guard implementation contract (LOCKED v1、9c.5 freeze 必須)*:
- *counter scope*: =modification-hook-depth= は *per-buffer counter*
  (= buffer-local cell)、Emacs precedent (=textprop.c= /
  =modify_text= の =inhibit_modification_hooks=) と一致
- *cross-buffer mutation*: hook 内で *別 buffer mutation* した場合は
  別 buffer の counter (= cross-buffer 抑制なし)。これにより
  hook 内 minibuffer 編集や別 buffer ログ書き込みが secondary
  trigger される (= Emacs precedent と一致)
- *guard variable semantics*:
  - =inhibit-modification-hooks= は *dynamic binding (let-bind) で復活*
    可能、unwind 時に元の値に戻る
  - =setq= 直書きは *undefined behavior* (= Doc 41 sub-doc 内では
    挙動を未定義扱い、ERT は let-bind のみで gate)
- *hook 種別ごと独立 counter*:
  - point-motion-hook (=point-entered= / =point-left=) と
    modification-hook (= =modification-hooks= / =insert-in-front-hooks=
    / =insert-behind-hooks=) は *独立 counter*
  - overlay-change-hook (= =overlay-change-functions=) も独立 counter
  - これにより一方の hook 内で他方を trigger 可能 (= Emacs precedent)
- *signal-safe unwind*:
  - hook が C-level signal (= =keyboard-quit=、Emacs Lisp =signal=、
    Rust panic) で abort した場合、counter は *unwind-protect で reset*
  - 9c.5 内部 helper =nelisp-with-modification-hook-guard= マクロで
    全 hook 起動を wrap、=unwind-protect= による counter decrement
    を保証
- *fail-safe*:
  - counter が想定値を超えた場合 (= 内部 bug) は warning 出力 + counter
    強制 reset、user 操作不能化を回避

*ERT gate に追加する 5 corner cases (MAJOR-B.3 反映、§3.5.5 で +5)*:
1. hook 内 mutation で secondary trigger 抑制
2. hook 内別 buffer mutation で secondary trigger 発火 (= cross-buffer
   非抑制)
3. =inhibit-modification-hooks= let-bind で復活
4. signal abort 時 counter reset (= 後続 mutation で hook 再 fire)
5. point-motion-hook と modification-hook の独立 counter (= 一方 trigger
   中に他方が fire)

** 6.10 overlay storage memory growth + GC pressure (MAJOR-D.2)

  数MB に到達する case あり (= interval tree node ~64 bytes × 10K +

- flycheck core gate (= 1000 件 / 100ms) では memory + GC pressure が
  立ち上がる
- Phase 7.3 GC との interaction で leak が発生し得る (= overlay handle
  cleanup 遅延)

mitigation:
- §3.4.5 ERT に "1M buffer × 10K overlay × payload で memory footprint
  計測 (= upper bound 50MB 想定、超過は 9c.4 close blocker)" 追加

  1M 回 simulate、leak なし)" 追加

** 6.11 interval tree split/merge の amortized cost (MAJOR-D.2)

  close blocker

  実装切替 (= 9c.4 内部 implementation の自由度、API 不変)

** 6.12 hook 内 overlay delete recursion (MAJOR-D.2)

  hook 内 delete が secondary trigger を fire させ得る

mitigation:

  contract)
- §3.4.5 ERT に "hook 内 =delete-overlay= recursion で safe 完了
