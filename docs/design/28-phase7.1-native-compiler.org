*** 1.2.1 既存 =nelisp-jit.el= との関係

既存 =nelisp-jit.el= は bytecode VM 内の inline-primitive optimizer
(~30 primitive を直接 host call に置換、330 LOC)。これは "JIT" を
名乗るが実態は peephole optimizer であり、native code 生成は行わない
= Phase 7.1 で置換されるものではなく、Phase 7.1 native compiler の
*input optimization stage* として再利用可能。

| 既存 =nelisp-jit=                  | Phase 7.1 native compiler          |
|------------------------------------+------------------------------------|
| input: lambda S-expr                | input: lambda S-expr (同じ)         |
| stage: bytecode VM 内 inline 展開   | stage: SSA IR → machine code       |
| output: 最適化済 bcl                | output: mmap PROT_EXEC page + jump |
| target: 30 primitives               | target: 全 NeLisp form              |

Phase 7.1.1 (SSA IR) で =nelisp-jit--inline-primitives= の判定 logic
を SSA constant-folding に取り込む。

** 1.3 SBCL =sb-c= + Emacs native-comp dual precedent reference

NeLisp の native compiler は SBCL =sb-c= と Emacs native-comp
(=Andrea Corallo, 2019-, Emacs 28.1+=) の dual precedent を踏まえ、
*runtime purity 軸で両者を超え、compiler capability 軸では両者の
MVP subset* を狙う (codex review 観点 1 反映)。

*** 1.3.1 SBCL =sb-c= 参照点

SBCL の native compiler module =src/compiler/= 全体 ~30000 LOC CL、
25 年維持。

| 削減方針     | SBCL =sb-c=                       | NeLisp =nelisp-cc=                          |
|--------------+-----------------------------------+---------------------------------------------|
| 多段最適化   | 多段 IR + register coloring 等    | 1-pass codegen MVP、phase 7.1.6 以降で SSA + GVN 後付け |
| inline       | 高度 inline expansion             | =defsubst= / =declare (inline t)= のみ      |
| target       | x86, x86-64, arm64, ppc, ...      | x86_64 + arm64 のみ                          |
| FFI          | sb-alien 完備                     | syscall stub 経由のみ                       |
| LOC          | ~30000 LOC CL                      | 5000-10000 LOC NeLisp                       |
| 期間         | 25 年維持 (初版 1999-2003)         | 12-20 週間 (anvil leverage 前提)            |
| bootstrap    | Rhodes 5-stage (genesis-2/target-2) | 4-stage (本 doc §3.5、SBCL 直接 transfer 後縮小) |

*** 1.3.2 Emacs native-comp 参照点

Emacs 28.1+ の =native-compile= (Andrea Corallo "Implementing a JIT
Compiled Lisp" 2019、Free Software Magazine + EmacsConf 各回) は
*Emacs Lisp 限定の native compiler* で、本 doc の最も近い precedent。

| 削減方針     | Emacs native-comp                       | NeLisp =nelisp-cc=                          |
|--------------+-----------------------------------------+---------------------------------------------|
| backend      | libgccjit (GCC JIT API、~30 年蓄積資産)   | self-impl bytes emitter (purity max path)   |
| IR           | LIMPLE IR (3-tier: SPILL / LIMPLE / libgccjit) | 1-pass MVP + 後で SSA 拡張 (§2.4)         |
| output       | =.eln= ELF object file (AOT cache)       | mmap PROT_EXEC page (in-process JIT)        |
| 性能         | bytecode 比 *2-5x typical* (heavy loop でも 5-10x) | bytecode 比 30-20x 必達 (3-軸 bench、§7) |
| inline       | call-site profile + type 駆動            | =declare (inline t)= 静的 + nelisp-jit 30 primitive |
| 互換性       | Emacs Lisp 全体                          | NeLisp subset (Phase 5-B 確認済)             |
| LOC          | C ~25000 + elisp ~5000 = ~30000          | NeLisp 5000-10000                            |

*Emacs native-comp との差別化軸*:
1. *Purity*: NeLisp は libgccjit 経路を *永久不採用* (Doc 27 §8.4
   Phase 7+ out of scope)。NeLisp source → NeLisp bytes emitter →
   mmap exec の pure NeLisp pipeline を確立する
2. *Bootstrap simplicity*: =.eln= ELF cache は host filesystem 依存、
   NeLisp は Rhodes 5-stage の 4-stage 簡略版で in-memory bootstrap
3. *Bench target*: Emacs native-comp 典型 2-5x over bytecode は
   *libgccjit 経由でも* この水準。NeLisp が "30x" を狙うのは fib /
   fact-iter のような call/branch heavy benchmark に限定し、
   alloc-heavy では Emacs native-comp 並 (2-5x) を許容する (§7 完遂 gate)

*** 1.3.3 NeLisp の差別化軸 (purity vs capability)

| 軸           | SBCL =sb-c=        | Emacs native-comp | NeLisp =nelisp-cc= |
|--------------+--------------------+-------------------+--------------------|
| capability   | 多段最適化 + 全 ISA | libgccjit 経由 | *MVP subset of both* |
| purity       | C runtime あり      | C + libgccjit     | *pure NeLisp (Doc 27)* |
| target lang  | Common Lisp 全体    | Emacs Lisp 全体   | NeLisp subset (Phase 5-B) |
| LOC budget   | ~30000              | ~30000            | 5000-10000           |

*超える軸 = purity*: Doc 27 §0 の =NeLisp purity max path= 方針通り、
runtime の Lisp 化率と self-host purity で SBCL/Emacs native-comp を
超える。compiler capability は両者の MVP subset で managed。

Doc 27 §2 schema (#1-12 LOCKED items) に揃える。default 案を提示し

** 2.1 target ISA [推奨 = A x86_64 + arm64 第一級]

| 候補 | 戦略                                              | 利点                              |
|------+---------------------------------------------------+-----------------------------------|
| A    | x86_64 + arm64 両方第一級 (default)                | Linux x86 + macOS Apple Silicon 両対応 |
| B    | x86_64 のみ MVP、arm64 後回し                     | 期間短縮 (~1-2 週間)              |
| C    | RISC-V 第一級追加                                  | scope 拡大、不採用                |

*推奨 A 理由*:
- Doc 27 §2.12 で "x86_64 + ARM64 のみ MVP" 明記
- macOS user (M1/M2/M3) priority 想定
- arm64 backend は ABI 単純 (AAPCS64 fixed register convention) で
  実装コスト x86_64 比 70-80%、後回しによる節約効果 < 1 月

** 2.2 output format [推奨 = A in-process JIT (mmap PROT_EXEC)]

| 候補 | 戦略                                            | 利点                          |
|------+-------------------------------------------------+-------------------------------|
| A    | mmap PROT_EXEC page (in-process JIT、default)    | ELF/Mach-O 不要、syscall stub と一致 |
| B    | ELF object file 出力 + dlopen                    | LD/dlopen 必要、Phase 7.0 scope 拡大 |
| C    | AOT executable                                   | bootstrap 完結後、Phase 7.5 で検討 |

*推奨 A 理由*:
- Doc 27 §2.12 "mmap 経由 (syscall stub) で executable page に書込 + jump"
- Phase 7.0 syscall stub に =mmap= / =mprotect= 提供済
- ELF/Mach-O linker 不要 = LOC 1500-2500 LOC 削減

** 2.3 calling convention [推奨 = A System V AMD64 + AAPCS64]

| 候補 | 戦略                                              |
|------+---------------------------------------------------|
| A    | System V AMD64 ABI (Linux/macOS) + AAPCS64 (default) |
| B    | 独自 calling convention (SBCL 同様 Lisp-internal)  |
| C    | Win64 (Windows MSVC ABI) も第一級                  |

*推奨 A 理由*:
- syscall stub (Phase 7.0) と calling conv 一致 → FFI cost 0
- Windows native は Stage A (anvil 単体) 経路で別解 (=Doc 18 Stage D=)、
  Phase 7.1 では opt-in (Win64 ABI 対応は phase 7.1.5 以降に繰越)

** 2.4 IR の階層 [推奨 = A 1-pass MVP + 後で SSA 拡張]

| 候補 | 戦略                                              |
|------+---------------------------------------------------|
| A    | 1-pass codegen MVP (default、phase 7.1.5 で多段化検討) |
| B    | 初期から SSA + 多段最適化 (SBCL 同 pattern)        |
| C    | LLVM IR backend (NeLisp purity 破綻、不採用)       |

*推奨 A 理由*:
- 5000 LOC budget で多段 IR は overrun risk 高
- bench gate 50x speedup は 1-pass + linear-scan で十分達成可
- Phase 7.1.5 (= self-host) 完了後、phase 7.1.6 として SSA + dead-code-elim
  / GVN 等を後付け増設

ただし sub-phase 7.1.1 で "*SSA を data structure として持ち、
optimization pass は MVP では空*" の構造にしておくこと (= 後付け
増設の余地を残す)。

** 2.5 register allocation [推奨 = A linear-scan]

| 候補 | 戦略                                              |
|------+---------------------------------------------------|
| A    | linear-scan (default、Poletto-Sarkar 1999)        |
| B    | graph-coloring (SBCL = Chaitin-Briggs)            |
| C    | greedy / register-window (RISC-V スタイル)         |

*推奨 A 理由*:
- LOC ~400 で実装可能、graph-coloring は 1500-2500 LOC
- hot loop SBCL 比 30-50% 遅い risk 受容 (§6 risk 1)
- Phase 7.1.5 で graph-coloring 後付け可

** 2.6 tail call [推奨 = A proper tail-call 必須]

Phase 5-B selfhost で tail recursion 多用 (e.g. =nelisp-defs-index=
の tree walker) のため proper tail-call 必須。Phase 7.1.4 で実装、
=jmp= (jump-to-tail-target) 形式で stack frame 再利用。

** 2.7 inline expansion [推奨 = A defsubst + declare のみ]

| 候補 | 戦略                                              |
|------+---------------------------------------------------|
| A    | =defsubst= / =declare (inline t)= のみ (default) |
| B    | call-site profile 駆動 inline (SBCL pattern)      |
| C    | 完全 inline 不対応                                 |

*推奨 A 理由*:
- =nelisp-jit--inline-primitives= の 30 primitive を =declare (inline t)=
  注釈付きで提供、Phase 7.1.1 で SSA constant-fold に取り込む
- profile 駆動 inline は phase 7.1.5 以降

** 2.8 symbol table [推奨 = A nelisp-defs-index 経由]

Phase 6.5 SHIPPED の =nelisp-defs-index= (793 LOC + SQLite backing)
を symbol table として再利用。重複実装回避、bootstrap 時の defun /
defmacro / defvar 解決を 1 query で済ませる。

** 2.9 GC integration [推奨 = A safe-point 機構 + GC metadata contract を Phase 7.1 in-scope]

codex review 観点 3.7 反映: "GC integration risk が insertion point に
偏り、stack map/root map contract 欠落" → Phase 7.3 で compiler ABI
を壊さないため、metadata contract を *Phase 7.1 in-scope* に追加する。

Phase 7.1 in-scope (compiler 側):
- safe-point insertion point (loop back-edge + function entry/exit)
- *GC metadata contract* (Phase 7.3 root scanner が要求する format 固定):
  - safe-point ID (uint32 sequential)
  - live root bitmap (each safe-point で live な register / stack slot
    に Lisp object がいる位置を bit map 表現)
  - frame layout descriptor (function prologue/epilogue で stack
    layout を decoder に伝える metadata table、ELF =.debug_frame=
    類比)
- =call gc-poll= stub emit (Phase 7.1.4、stub 中身は no-op)

Phase 7.3 out of scope (collector 側):
- mark-sweep / generational GC algorithm 本体
- root scanning 実行 (上記 metadata を消費する側)
- stop-the-world / concurrent / incremental の GC scheduling

= compiler 側 contract だけ Phase 7.1 で固定、collector 実装本体は
Phase 7.3 へ。これで Phase 7.3 着手時に compiler ABI 改訂が不要。

** 2.10 bootstrap [推奨 = A NeLisp source → byte-code → mmap exec]

| 候補 | 戦略                                                |
|------+-----------------------------------------------------|
| A    | NeLisp source → Emacs byte-code → mmap exec (default) |
| B    | 直接 NeLisp source → mmap exec (bootstrap でも host Emacs 不要) |

*推奨 A 理由*:
- Stage D Phase 6.1 で host Emacs 経由 byte-compile path 確立済
- B は Phase 7.5 (integration) で改めて検討、Phase 7.1 内 scope 外

** 2.11 profiling [推奨 = A nelisp-runtime --profile flag]

=cargo bench= harness + =--profile= flag で hot path 計測。
Phase 7.1.4 までは bench harness のみ整備、actual profile UI は
phase 7.1.5 で。

** 2.12 LOC budget [推奨 = A 5000-10000 LOC NeLisp + ~200 LOC Rust mmap helper]

Doc 27 §2.12 LOCKED の "5000-10000 LOC NeLisp" 踏襲。Rust 側は
Phase 7.0 で =mmap= / =mprotect= 既に提供済、追加は ~200 LOC

* §3 Sub-phase 7.1.1〜7.1.5 詳細

Doc 27 §6.7 mitigation の 4 sub-sub-phase 構成を baseline、self-host
bootstrap を sub-phase 7.1.5 として追加した 5 sub-phase 構成。各
sub-phase は独立 SHIPPED 可能、main へ feat branch =--no-ff= merge。

v2 改訂で期間 base case を *12-20 週間 (3-5 ヶ月)* に re-baseline
(codex review 観点 2.3 = 期間自己矛盾 + dual-ISA optimistic 反映)。

| sub-phase | scope                                                | LOC          | 期間 (single ISA first-class) | ERT delta |
|-----------+------------------------------------------------------+--------------+-------------------------------+-----------|
| 7.1.1     | SSA IR + linear-scan register allocation              | ~1500-3000   | 4-5 週間                      | +10〜+15  |
| 7.1.2     | *x86_64 backend (first-class、System V AMD64)*        | ~1500-2500   | 3-4 週間                      | +10       |
| 7.1.3     | *arm64 backend (provisional、AAPCS64、escape hatch 対象)* | ~1500-2500 | 2-3 週間                      | +8        |
| 7.1.4     | mmap exec page + tail-call + safe-point + GC metadata | ~500-1000    | 2 週間                        | +6        |
| 7.1.5     | self-host bootstrap (4-stage) + bench harness         | ~500-1000    | 2 週間                        | +10       |
|-----------+------------------------------------------------------+--------------+-------------------------------+-----------|
|           | 合計 (full)                                          | ~5000-10000  | *13-16 週間*                  | +44〜+49  |
|           | 合計 (escape hatch: arm64 後送り)                     | ~3500-7500   | *11-13 週間*                  | +36〜+41  |

期間 anvil leverage 込み 12-20 週間 base、escape hatch 発動で 11-13 週間

** 3.5 Phase 7.1.5 — self-host bootstrap (4-stage protocol) + 3-軸 bench harness

v2 改訂 (codex 観点 2.4 反映): bootstrap protocol を *4-stage 詳細
化*、各 stage の failback 条件と "native authoritative 昇格" 条件を
明文化。

*** bootstrap protocol (4-stage)

| stage  | artifact 関係 (§1.4 参照)         | 内容                                                              | failback 条件 |
|--------+-----------------------------------+-------------------------------------------------------------------+---------------|
| stage0 | A1 producing bytecode for A2 input | host Emacs (Stage D Phase 6.1 path) で =nelisp-cc.el= を byte-compile | host Emacs 環境不能 → Phase 7.1.5 中断 |
| stage1 | A2 が =nelisp-cc.el= source を native compile → A3 候補 | A2 (bytecode 上で動く seed compiler) で source を mmap exec 化 | A3 候補生成失敗 → bytecode 経路 fallback (A2 維持) |
| stage2 | A2 vs A3 候補の semantic diff      | golden hash: 同 input AST に対し A2 と A3 候補の 出力 mmap page が semantic-equivalent (=同 bytes for fixed seed=) | hash mismatch → A3 候補 reject、A2 で運用継続、bug 修正後 stage1 再試行 |
| stage3 | A3 を authoritative に昇格          | =nelisp-cc.el= 自身を A3 が再 native compile (= self-recompile)、結果が A3 と semantic equivalent | self-recompile 結果が A3 と diff → A3 reject、A2 fallback |

各 stage で *bytecode 経路 (= A2) を fallback として維持*。stage3
完了で初めて A3 が "authoritative" 化、Phase 7.2 (allocator) 以降が
A3 を信頼して built される。

*** scope

- bootstrap 4-stage protocol 実装 (=src/nelisp-cc-bootstrap.el=、~200-400 LOC)
- =./nelisp-runtime --self-compile= で stage1 + stage2 + stage3 を invoke
- =cargo bench= harness で 3-軸 bench (=fib(30)= 30x / =fact-iter= 20x / =alloc-heavy= 5x、§7 完遂 gate)
- 多段 IR 検討 (phase 7.1.6 への bridge): SSA 上の dead-code-elim /
  GVN / instruction selection の prototype (出荷必須ではない)
- profile UI (=nelisp-runtime --profile=) integration

*** ERT smoke (+10)

- self-host stage0 → stage1 → stage2 → stage3 の通し test
- semantic diff PASS test (A2 vs A3 candidate golden hash)
- failback path test (stage2 hash mismatch で A3 reject、A2 維持)
- bench: =fib(30)= bytecode 比 30x 以上 (call/branch heavy)
- bench: =fact-iter= bytecode 比 20x 以上 (loop hot path)
- bench: =alloc-heavy (cons stress 1M)= bytecode 比 5x 以上 (Phase 7.2 budget)
- regression: 既存 1086/1091 ERT 全 PASS

*** risks

- self-host bootstrap chicken-egg: §6.3 で 4-stage protocol で再現可能化
- A3 self-recompile が A2 と diff する case (compiler bug、recursive
  self-bug など) → mitigation: stage2 reject で A2 fallback、不変条件
  (=fixed seed → fixed bytes=) を ERT で固定

* §4 Architecture (5-stage pipeline)

#+begin_src text
[NeLisp source]
    │ nelisp-read.el (Phase 3a SHIPPED)
    ▼
[AST (S-expr tree)]
    │ nelisp-cc.el §3.1 (Phase 7.1.1)
    │   ├ macro-expand (既存 nelisp-macro.el)
    │   ├ AST → SSA IR
    │   └ optimization pass (constant fold / dead-code-elim)
    ▼
[SSA IR + register allocation]
    │ nelisp-cc-x86_64.el §3.2 (Phase 7.1.2)
    │ nelisp-cc-arm64.el §3.3 (Phase 7.1.3)
    │   ├ instruction selection
    │   ├ linear-scan register allocation
    │   └ codegen (machine bytes)
    ▼
[Machine code bytes]
    │ nelisp-cc.el §3.4 (Phase 7.1.4)
    │   ├ mmap PROT_EXEC page (Phase 7.0 syscall stub)
    │   ├ relocation (call/jmp targets fixup)
    │   └ safe-point + tail-call emit
    ▼
[Executable mmap page]
    │ jump (= function pointer call)
    ▼
[runtime execution]
#+end_src

** 4.1 入力境界 (AST)

=nelisp-read.el= (Phase 3a SHIPPED) の AST 形式を踏襲、変更しない。
quote / backquote / sharp-quote 等の reader form は AST 段階で展開
済 = native compiler は plain S-expr のみ受け取る。

** 4.2 出力境界 (mmap exec page)

Phase 7.0 syscall stub の =mmap=/=mprotect= に依存。Linux/macOS
=MAP_ANONYMOUS | MAP_PRIVATE= + =PROT_READ|PROT_WRITE|PROT_EXEC=
(macOS Apple Silicon は =MAP_JIT= flag 追加必須、§9 open question)。

** 4.3 symbol table 連携

=nelisp-defs-index= (Phase 6.5 SHIPPED) を経由して defun/defmacro/
defvar の name → AST or native code address mapping を 1 SQL query
で解決。Phase 7.1.5 self-host 時、=nelisp-cc.el= 自体の defun が
=nelisp-defs-index= に登録され、自身を native compile する循環が
成立する。

* §6 Risks (8 項目)

** 6.1 register allocation 性能

linear-scan で SBCL の graph-coloring に対し hot loop 30-50%
遅い可能性。

*影響*: bench gate 50x speedup 未達 risk。

*mitigation*:
- Phase 7.1.5 で graph-coloring 後付け可能な構造を §3.1 で確保
- bench miss 時は scope 削減 (gate 30x speedup に下方修正、user
  signoff 要)

** 6.2 safe-point 設計

GC との co-design で deadlock or pause spike risk。

*影響*: Phase 7.3 (GC NeLisp 化) で発覚すると 7.1 の retrofit が必要。

*mitigation*:
- §3.4 で safe-point insertion point だけ確保、actual root scanning
  は Phase 7.3 と共同 design
- Phase 7.1.4 完了時点で Phase 7.3 担当 (session-X) と
  orchestrator consensus review 1 回

** 6.3 self-host bootstrap chicken-egg

Phase 7.1.5 で NeLisp が NeLisp compile する瞬間の状態管理
(Rhodes target-2 cold-init 類比)。

*影響*: bootstrap 失敗で Phase 7.1 全停止。

*mitigation*:
- 2-stage bootstrap: 第1 stage = host Emacs byte-code 経由
  =nelisp-cc.el= ロード、第2 stage = ロード済 =nelisp-cc-compile= で
  =nelisp-cc.el= 自己 native compile
- chicken-egg は第1→第2 移行点だけに集中、bytecode interpret 経路を
  fallback として維持

** 6.4 Emacs byte-code FFI 維持コスト

Phase 7.0 完了まで Emacs byte-code 経由 dispatch、native code との
dual path 期間が長期化。

*影響*: Phase 7.0 + 7.1 並走期間 (~3 ヶ月) で dual path bug 発生 risk。

*mitigation*:
- Phase 7.0 SHIPPED gate に "syscall stub 全 ERT 緑" を必達化済
  (Doc 27 §3 7.0)
- Phase 7.1.1〜7.1.4 は bytecode VM の output と semantic equivalence
  test を 100% maintain (semantic regression なし)

** 6.5 x86_64 / arm64 binary divergence

両 ISA 第一級維持のテスト負荷増、CI matrix 倍加。

*影響*: CI 時間 + メンテコスト。

*mitigation*:
- §5 CI matrix で Linux x86_64 / macOS arm64 / Linux arm64 の 3 環境
  全部緑を強制
- arm64 backend は AAPCS64 fixed register convention で実装 simple、
  divergence は =nelisp-cc-arm64-test= で吸収

** 6.6 anvil leverage 過大評価 (v2 改訂: week 4/8 replan gate + kill criteria)

Doc 27 で SBCL 25 年 → 1-2 ヶ月の見積。本 v2 で 12-20 週間に re-baseline
したが、なお下振れ risk あり。codex 観点 3.8 = "mitigation が管理手法
として粗い" 反映で *事前 kill criteria + 2 replan gate* を導入。

*影響*: Phase 7.2-7.4 の着手遅延 = Doc 27 全体 STATE 遅延。

*mitigation (v2 = 事前 replan gate + kill criteria)*:

| gate    | 時期    | 評価対象            | kill criteria (= 達成不能なら scope 切り) |
|---------+---------+---------------------+-------------------------------------------|
| gate-W4 | week 4  | 7.1.1 SSA IR + linear-scan 完成度 | 7.1.1 未完了 → *second ISA (arm64) defer*: Phase 7.1 完遂 gate から arm64 を外し、Phase 7.5 integration で完成 |
| gate-W8 | week 8  | 7.1.2 x86_64 backend + 7.1.3 arm64 進捗 | 7.1.2 未完了 → *self-host 以外を freeze*: 7.1.4/7.1.5 を一時停止、x86_64 first-class 完成に集中、bench gate 30x 達成で suspension 解除 |
| gate-W12 | week 12 | 7.1.4 + 7.1.5 stage1 進捗 | 7.1.4 未完了 → *bench gate 下方修正*: =fib(30) 30x → 20x= に下方修正、user signoff 要 |

- Phase 7.1.1 (SSA IR) 完成時点で Phase 7.2 (allocator) スケッチ着手、
  parallel に walk (Doc 27 §6.7 mitigation 踏襲)

** 6.9 arm64 JIT I-cache coherence + W^X/MAP_JIT 運用 (v2 新設)

codex review 観点 3.6 反映: §3.3 は MAP_JIT に触れるが、書き込み後の
instruction cache invalidate と write/exec transition protocol が明記
されていない。arm64 を第一級 target にする以上、ここは単なる実装
detail ではなく成立条件。

*影響*: arm64 backend 完了判定が甘くなる。書き込み直後 native code
を実行すると stale I-cache で旧コード or undefined が走る、debug 困難。

*mitigation*:
- Phase 7.0 syscall stub に追加要求 primitive (本 doc §9 open question 5):
  - =MAP_JIT= flag 対応 (macOS Apple Silicon、=mmap= の =flags= 引数)
  - =__clear_cache 相当= (instruction cache invalidate、=__builtin___clear_cache=
    GCC builtin / =sys_icache_invalidate= macOS / =FlushInstructionCache= Win)
  - =write/exec transition protocol= (=pthread_jit_write_protect_np=
    macOS / =mprotect PROT_READ|PROT_EXEC= Linux)
- Phase 7.1.3 exit criteria に追加:
  - "*generated code executes after cache sync on macOS arm64 / Linux arm64*"
- ERT に =arm64-icache-coherence-test= 必須 (write → clear-cache → exec →
  正しい命令が走ること verify、CI matrix で macOS arm64 + Linux arm64 で実行)

** 6.10 GC metadata contract Phase 7.3 ABI 連動 (v2 新設)

codex review 観点 3.7 反映: §2.9 で =GC metadata contract= を Phase 7.1
in-scope に追加したため、Phase 7.3 着手時に metadata format を改訂
すると compiler ABI を壊す risk。

*影響*: Phase 7.3 で root scanner 設計時に metadata 変更要望出る場合、
Phase 7.1 全体の retrofit (全 backend の emitter 修正)。

*mitigation*:
- Phase 7.1.4 完了時点で Phase 7.3 担当 (session-X) と =metadata
  contract review= を 1 回必須 (orchestrator-consensus-submit、
  reviewer = claude + codex、judge = claude)
- contract format を versioned (=GC_METADATA_VERSION= constant) で
  encode、Phase 7.3 で incompatible 改訂が必要なら version bump +
  migration path 設計

** 6.7 ABI compliance bug 検出困難

System V/AAPCS の corner case (struct passing / red zone / SIMD
register save) は table-driven test 必須。

*影響*: silent miscompile で Phase 7.2 以降の hot path に潜伏 bug。

*mitigation*:
- =test/nelisp-cc-abi-test.el= を Phase 7.1.2/.3 で table-driven 整備
- C 製 reference implementation (musl libc / Apple libsystem) と
  bytes-level diff を CI で取る

** 6.8 NeLisp purity max path 全体の自爆 risk

Phase 7.1 で詰まれば Doc 27 全体 (3 核 NeLisp 化) 連鎖遅延 = critical
path 全停止 (Doc 27 §6.7)。

*影響*: NeLisp v1.0 完遂条件 #4 達成不能 → 公開告知保留 (
=feedback_nelisp_announcement_standalone_gate.md=) 解除されず。

*mitigation*:
- 各 sub-phase 7.1.X SHIPPED で本 risk を継続 audit (=anvil-dev=
  =release-audit= scope=)
- Phase 7.1.3 (arm64) 後に "scope 圧縮 emergency procedure" を
  invoke 可能化:
  - 7.1.3 ARM64 を Phase 7.5 へ後送り
  - 7.1.4 の safe-point 簡素化 (loop back-edge のみ)
  - 7.1.5 bench gate 30x へ下方修正
- 上記 emergency 発動時 user signoff + Doc 27 STATUS 改訂
