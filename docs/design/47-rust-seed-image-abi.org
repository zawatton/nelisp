* §0 要旨 (TL;DR)

Doc 46 architecture α Wave 3 で =anvil-runtime/= 切出しが完了し、
=nelisp-runtime/= の責務は「NeLisp 言語 runtime」に純化された。本 doc
は次の極小化軸 = =nelisp-runtime/= 内 Rust LOC を **「Elisp 処理系」から
「seed (image loader + syscall + executable memory + entrypoint)」へ
降格** することの設計仕様。

  *Before* (= main 現状、post Wave 3、2026-04-30 計測)
    Rust LOC = 10,325 (eval 4,018 + reader 1,816 + syscall 2,042
                     + sqlite 623 + bridge 563 + main/lib/bin 1,263)
    起動経路 = =main()= → reader → eval → 30〜50 builtin で
              =src/*.el= を bootstrap

  *After* (= Doc 47 ship 後、target)
    Rust LOC = 2,500〜4,000
    起動経路 = =main()= → mmap(=nelisp.image=) → entry に jump
    runtime 中 Rust に戻るのは syscall trap / signal / GC barrier のみ

key insight: Rust が =.el= を読むから reader が要る。Rust が
=.image= (= 事前 mint された heap snapshot) を読むなら **reader も
evaluator も builtin も runtime には不要**。SBCL =sbcl-mini= →
=core.lisp-obj=、Squeak Smalltalk VM、Open Genera と同 pattern。
build-time の Rust evaluator (= 現 Doc 44 で LOCKED) は **image を mint
する build tool として残す**、ship バイナリからは消える。

*ship 条件*:
- =bin/anvil-runtime --no-emacs= が =nelisp-runtime= から evaluator /
  reader / builtins を *削除した状態で* MCP server として起動する
- runtime crate Rust LOC ≤ 4,000
- =nelisp.image= 作成は =build-tool/= crate (新設) または既存
  =nelisp-runtime= の =--mint-image= フラグで実行
- cold-init 時間 ≤ 500ms (Doc 32 §5 維持)
- Phase 7 native cc で AOT 済 native code arena が image に同梱される

*non-goals*:
- Rust 0 LOC は目指さない (OS syscall / signal handler / mmap は Rust 必須)
- Phase 5 =nelisp-emacs= 全 port (= Open Genera 級) ではない
- 既存 Emacs path (= =bin/anvil mcp serve= default) の変更は最小

* §1 Motivation

** 1.1 Doc 44 との関係 (重要)

Doc 44 (Phase 8.0、LOCKED 2026-04-27) は「Rust 側に bootstrap-grade
minimal Elisp interpreter (~2500-3500 LOC) を実装、cold-init で
=src/= NeLisp source を自己 load → =nelisp-eval= に control を渡す」
方針。これは Doc 44 の言葉では「load bearing が cold-init の一瞬のみ」
とされているが、**ship バイナリには interpreter が残る**。

Doc 47 は Doc 44 の自然な拡張で、interpreter を **build-time のみ**
に降格する。違いは:

| 項目                    | Doc 44 (LOCKED)                  | Doc 47 (本 doc)                              |
|-------------------------+----------------------------------+----------------------------------------------|
| Rust runtime interpreter | 残る (~2500-3500 LOC、cold-init で takeover) | 削除 (= image loader のみ)                   |
| =.el= source の location | runtime が読む (=src/*.el=)          | build-time が読み image に焼く                |
| reader の location       | runtime にあり                    | build-tool にあり                             |
| ship binary Rust LOC    | ~14,000 (= 現状 + Doc 44 増分)    | ~2,500〜4,000                                  |
| Phase 7 native cc       | runtime takeover 後 active       | build-time に AOT、image に native arena 同梱  |
| user vision との整合     | 部分的                            | 完全 (= "Rust = seed, NeLisp = everything")    |

Doc 44 を取り消すのではなく、**Doc 44 を本 doc の precondition として位置づける**:
Doc 44 で reader+eval+builtin が Rust 内に自己完結すれば、それを
build-time に切り出せる boundary が初めて見える。Doc 44 完了 → Doc 47
着手という直列 dependency。

** 1.2 ChatGPT 議論 (2026-04-30 zawatton scoping session) の合意点

zawatton + ChatGPT scoping (2026-04-30) で合意した core 主張:

1. *Rust core should not be an Elisp interpreter*
2. *Rust core should be: image loader + syscall shim + executable
   memory provider + process entry point*
3. *Everything else should be NeLisp*

これを Doc 47 の北極星として固定する。

** 2.2 Image format (NlImage v1 — DRAFT)

#+begin_src rust :tangle no
// nelisp-runtime/src/image/format.rs
#[repr(C)]
pub struct NlImageHeader {
    /// magic = b"NLIMAGE\0" (8 bytes)
    pub magic: [u8; 8],

    /// ABI version (= seed が認識する image format 世代).
    /// seed と image の version 不一致 → load 失敗 + abort.
    pub abi_version: u32,

    /// 0 = none, 1 = zstd, 2 = lz4 (extension point)
    pub compression: u32,

    /// image 本体のバイト長 (header 直後から)
    pub payload_len: u64,

    /// heap segment offset / size (= mmap PROT_READ|PROT_WRITE)
    pub heap_offset: u64,
    pub heap_size: u64,

    /// native code arena offset / size (= mmap PROT_READ|PROT_EXEC).
    /// Phase 7 cc が build-time に AOT 出力した native bytes.
    /// 空 (size = 0) なら native arena 無し = pure bytecode image.
    pub code_offset: u64,
    pub code_size: u64,

    /// relocation table offset / size (= image 内 absolute pointer の
    /// fixup table。ASLR / image base 変動対策)
    pub reloc_offset: u64,
    pub reloc_count: u64,

    /// entry point (= image 内 native code arena 内 offset).
    /// seed は heap_base = mmap(heap_offset)、code_base = mmap(code_offset)
    /// を確定後、(code_base + entry_offset) に jump する.
    pub entry_offset: u64,

    /// signal vector (= NeLisp が signal handler から呼んでほしい
    /// elisp 関数の image 内 entry table、今は最小: GC barrier 1 件のみ).
    pub signal_vector_offset: u64,
    pub signal_vector_count: u32,

    /// minimum syscall ABI version that the image expects.
    /// seed の syscall set がこれ未満なら refuse to load.
    pub required_syscall_abi: u32,

    /// reserved (8 bytes、将来用、0 fill)
    pub _reserved: [u8; 8],
}

const NL_IMAGE_MAGIC: [u8; 8] = *b"NLIMAGE\0";
const NL_IMAGE_ABI_VERSION: u32 = 1;
#+end_src

レイアウト不変条件:
- =heap_offset= / =code_offset= / =reloc_offset= は **page-aligned** (4 KiB)
- relocation table のエントリは =u64 image_offset= (= heap 内 absolute
  pointer が書かれた位置、seed が =heap_base + image_offset= に actual
  address を書き戻す)
- entry function ABI は =extern "C" fn(argc: i32, argv: *const *const u8) -> i32=
- 全 multi-byte integer は **little-endian** (cross-arch portability は
  initial scope 外、x86_64 + arm64 LE 前提)

** 2.3 Boot sequence (cold-init 詳細)

#+begin_src rust :tangle no
// nelisp-runtime/src/main.rs (target shape)
fn main() -> ! {
    // 1. arg parse (極小、--image PATH のみ)
    let cfg = parse_args();

    // 2. install signal handlers FIRST (= image load 中の SEGV を捕まえる)
    signal::install_handlers();

    // 3. read image header
    let header = image::read_header(&cfg.image_path)
        .expect("image header read failed");
    image::verify_magic_and_version(&header)
        .expect("image abi mismatch");

    // 4. mmap heap segment (RW)
    let heap_base = exec_mem::mmap_rw(header.heap_size);
    image::load_segment(&cfg.image_path, header.heap_offset,
                        header.heap_size, heap_base);

    // 5. mmap native code arena (RX after write)
    let code_base = exec_mem::mmap_rwx_then_rx(
        &cfg.image_path, header.code_offset, header.code_size);

    // 6. relocate
    image::apply_relocations(heap_base, code_base, &header,
                             &cfg.image_path);

    // 7. jump to entry
    let entry: extern "C" fn(i32, *const *const u8) -> i32 = unsafe {
        std::mem::transmute(code_base.add(header.entry_offset as usize))
    };
    let argv_ptrs: Vec<*const u8> = std::env::args_os()
        .map(|s| s.as_bytes().as_ptr())
        .collect();
    let rc = entry(argv_ptrs.len() as i32, argv_ptrs.as_ptr());
    std::process::exit(rc);
}
#+end_src

この main.rs 全体で **~150 LOC**。これが極小化の到達点。

** 5.2 Q2 — image format versioning policy

NlImage v1 を ABI として固定したあと、v2 への移行 path をどう設計するか。

  *推奨*: seed が認識する =NL_IMAGE_ABI_VERSION_MAX= を build-time const
  にし、それ未満を accept、より上は reject (= forward-compat なし、
  backward-compat あり)。image 側に v2-only 機能を入れたら v2 seed が
  必要、v1 image は v2 seed でも load 可。SBCL の =SLOT-ACCESS-VERSION=
  と同 pattern。

** 5.3 Q3 — relocation 戦略 (PIC vs reloc table)

native code arena を position-independent code (PIC) で生成すれば
relocation table は heap の絶対 pointer 分だけで済む。=nelisp-cc= が
PIC を出せるか未確認。

  *推奨*: Phase 7 cc PIC 対応の有無を Phase 1 (image format spec freeze)
  で確認、出せなければ reloc table を native arena 内 absolute call
  / load にも拡張。worst case 数 KB 程度の table size なので問題ない
  はず。

** 5.4 Q4 — sqlite を runtime に残すか optional crate に降格するか

現 =sqlite.rs= 623 LOC は runtime crate 内。極小主義の貫徹なら
=nelisp-sqlite-rs= dyn module に分離。ただし Doc 27/28 で sqlite を
=anvil-state= の primary backing store にした経緯があり、image boot
直後から sqlite が無いと state restore が動かない可能性あり。

  *推奨*: phase 9 (= 最後) で計測、cold-init path で sqlite 必須なら
  runtime に残し LOC budget を 4,500 に上方修正。dynamic load で
  間に合うなら optional 化。

** 5.5 Q5 — Windows path

Doc 47 は Linux + macOS 前提で書いた。Windows ConPTY / mmap セマンティクス /
=VirtualAlloc= の違いを image format level でどう吸収するか。

  *推奨*: Stage D Phase 6.4 (= Doc 18 §10.5) と合流、Windows 対応は
  =image/format.rs= の =target_os= field を追加して image 自体を
  platform-specific binary とする。クロスプラットフォーム image は
  non-goal。

** 5.6 Q6 — debug experience の劣化

Rust evaluator を消すと =RUST_LOG= / =cargo test= で eval を inspect
できなくなる。代替手段は?

  *推奨*: Elisp 側 =nelisp-debug-eval= MCP tool を Phase 7 で expose、
  build-tool crate 側にも =nelisp-mint-image --eval-trace= で image 焼き
  時の eval を log できるようにする。ship runtime には debug hook を
  入れない (= 極小化方針優先)。
