**本 Stage で elisp 化する layer**:
- =apply_function= の 4-way dispatch (builtin / closure / lambda / macro)

** §1.3 既存 parallel install pattern の流用

Stage 7.2.b (reader) / Stage 7.3.a-d (specials) で確立した手順:

1. **install (= dormant)**: 新 elisp 実装を =Env::new_global= の
   =STDLIB_SOURCES= に追加。fcell に install されるが、Rust dispatch が
   *先勝* なので runtime には呼ばれない。ERT のみで等価性を確認。
2. **takeover hook (flag-gated)**: =--use-elisp-apply= 等の runtime flag
   で Rust dispatch を short-circuit、elisp side を呼び出す path を有効化。
3. **default flip**: 全 ERT が両 mode で PASS、wallclock < 3x で gate
   通過後、default flag を ON。Rust 側は =cfg= 等で死蔵。
4. **delete (Stage 7.5)**: 1〜2 release 観察 期間後、Rust 死蔵コードを
   完全削除。

本 Stage 7.4 では (1) と (2) と (3) までを実行 (= Stage 7.4.a〜d)。
delete は Stage 7.5 に委譲。

** §1.4 Phase 5 native cc lowering との相補関係

=apply_function= dispatch は構造上 =cond= chain (= "head が builtin か
closure か lambda か macro か") に展開される。Phase 5 SHIPPED 後の
=cond=/=eq=/=car= 系 lower 経路 (= Stage 5.5 / 5.6) で cond chain は
host 分岐命令に直 emit。よって elisp dispatch overhead の主要部 (= 4-way
判定) は <100ns / call の見込み。

bench gate 例:
- =(let ((f (lambda (x) (* x x)))) (dotimes (i 10000) (funcall f i)))=
  = =funcall= 経由 = =apply-fn= 1 回 + body eval = Stage 7.4.d 後の
  実 wallclock を Stage 7.3.d 直前と比較、gate < 1.8x

** §2.2 Tier 1-only body 制約 (= Doc 67 §2.2 の再適用)

elisp 側 =nelisp--apply-fn= 等の body は **Tier 1 special form + Rust
builtin + Tier 2 macros (Stage 7.3 SHIPPED 済)** でのみ書く。具体的に:

許容:
- =if= / =progn= / =let= / =let*= / =setq= / =while= / =catch= / =throw=
  / =condition-case= / =unwind-protect= / =quote= / =function= /
  =lambda= (= Tier 1 13 forms)
- =cond= / =when= / =unless= / =or= / =and= / =defun= / =defvar= /
  =defconst= / =dolist= / =dotimes= / =push= / =pop= / =prog1= / =prog2=
  (= Stage 7.3 SHIPPED elisp macros)
- Rust builtin: =cons= / =car= / =cdr= / =eq= / =equal= / =null= / =not=
  / =symbolp= / =consp= / =listp= / =length= / =nth= / =nthcdr= /
  =append= / =list= / =error= / =signal= / =funcall= / =apply= 等
- =nelisp--*= 系 stdlib helper (= 既存)

不許可:
- =loop= / =cl-loop= / =cl-block= 等 cl-extras (= 自身が =apply-fn=
  経由で展開、bootstrap cycle 発生のリスク)
- =seq-*= / =cl-*-fn= 等 HOF (= recursive funcall に依存、本 Stage が
  まさにそれを実装している)

理由: =nelisp--apply-fn= 自身が「funcall を実装する関数」なので、
*その body の中で funcall を要求する高階構文は使えない*。これは Doc
67 の Tier 1-only macro body 制約と同じ chicken-and-egg。

** §2.3 closure / lambda 表現の不変

closure 表現は Stage 7.4 でも変えない:

#+begin_src text
(closure CAPTURED-ENV ARGS BODY...)
#+end_src

CAPTURED-ENV は alist of =((NAME . CELL) ...)=、CELL は =Sexp::Cell=
(= Rust =Rc<RefCell<Sexp>>= の wrapper、setq write-through 用)。

bare lambda:
#+begin_src text
(lambda ARGS BODY...)   ; defun が fcell に置く正規形
#+end_src

elisp 側からこの cons 構造を =car= / =cdr= で walk するだけ。新規
discriminator や tag は導入しない。

** §2.5 elisp 側 =nelisp-stdlib-eval-core.el= 概要

#+begin_src elisp
;;; nelisp-stdlib-eval-core.el — Phase 7 Stage 7.4 apply / closure / lambda

;; --- predicate -----------------------------------------------------------

(defun nelisp--builtinp (fn)
  "T if FN is a (builtin NAME) sentinel."
  (and (consp fn) (eq (car fn) 'builtin)))

(defun nelisp--closurep (fn)
  "T if FN is a (closure ENV ARGS BODY...) shape."
  (and (consp fn) (eq (car fn) 'closure)))

(defun nelisp--lambdap (fn)
  "T if FN is a (lambda ARGS BODY...) shape."
  (and (consp fn) (eq (car fn) 'lambda)))

(defun nelisp--macrop (fn)
  "T if FN is a (macro . LAMBDA) shape — same as Rust is_macro."
  (and (consp fn) (eq (car fn) 'macro)))

;; --- formal-parameter binding -------------------------------------------

(defun nelisp--bind-formals (formals args)
  "Bind FORMALS to ARGS in the topmost lexical frame.
Supports &optional / &rest per Elisp manual.  Errors mirror the
Rust bind_formals shape (WrongNumberOfArguments / WrongType)."
  (let ((mode 'required)
        (idx 0)
        (saw-rest nil)
        (consumed-rest nil)
        (required-count 0)
        (rest-formals formals))
    ;; first pass: count required arity for diagnostics
    (let ((p formals))
      (while (and (consp p)
                  (not (memq (car p) '(&optional &rest))))
        (setq required-count (1+ required-count))
        (setq p (cdr p))))
    (while (consp rest-formals)
      (let ((name (car rest-formals)))
        (cond
         ((eq name '&optional)
          (when (eq mode 'rest)
            (signal 'wrong-type-argument
                    (list 'formal-after-rest name)))
          (setq mode 'optional))
         ((eq name '&rest)
          (when saw-rest
            (signal 'wrong-type-argument (list 'multiple-rest name)))
          (setq mode 'rest)
          (setq saw-rest t))
         ((symbolp name)
          (cond
           ((eq mode 'required)
            (when (>= idx (length args))
              (signal 'wrong-number-of-arguments
                      (list 'lambda required-count (length args))))
            (nelisp--bind-local name (nth idx args))
            (setq idx (1+ idx)))
           ((eq mode 'optional)
            (let ((v (if (< idx (length args))
                         (prog1 (nth idx args) (setq idx (1+ idx)))
                       nil)))
              (nelisp--bind-local name v)))
           ((eq mode 'rest)
            (when consumed-rest
              (signal 'wrong-type-argument
                      (list 'symbol-after-rest name)))
            (let ((tail (nthcdr idx args)))
              (nelisp--bind-local name tail)
              (setq idx (length args))
              (setq consumed-rest t)))))
         (t (signal 'wrong-type-argument (list 'symbol name)))))
      (setq rest-formals (cdr rest-formals)))
    (when (and (< idx (length args)) (not consumed-rest))
      (signal 'wrong-number-of-arguments
              (list 'lambda (format "at most %d" idx) (length args))))
    (when (and saw-rest (not consumed-rest))
      (signal 'wrong-type-argument (list 'symbol-after-rest '&rest)))
    t))

;; --- closure / lambda body apply ----------------------------------------

(defun nelisp--apply-lambda-inner (captured formals body args)
  "Push CAPTURED env + a fresh frame, bind FORMALS to ARGS, eval BODY."
  (nelisp--push-captured captured)
  (nelisp--push-frame)
  (unwind-protect
      (progn
        (nelisp--bind-formals formals args)
        ;; eval body forms in order, return last
        (let ((last nil))
          (dolist (form body)
            (setq last (eval form)))   ; eval = Rust entry, runs Tier 1 + macros
          last))
    (nelisp--pop-frame)
    (nelisp--pop-frame)))

(defun nelisp--apply-closure (closure args)
  "Apply (closure CAPTURED ARGS-FORMAL BODY...) to ARGS."
  (let* ((parts (cdr closure))           ; (CAPTURED FORMALS BODY...)
         (captured (car parts))
         (formals (cadr parts))
         (body (cddr parts)))
    (nelisp--apply-lambda-inner captured formals body args)))

(defun nelisp--apply-lambda (lambda args)
  "Apply (lambda FORMALS BODY...) to ARGS — bare lambda, empty captured env."
  (let* ((parts (cdr lambda))
         (formals (car parts))
         (body (cdr parts)))
    (nelisp--apply-lambda-inner nil formals body args)))

;; --- top-level dispatch -------------------------------------------------

(defun nelisp--apply-fn (fn args)
  "Dispatch FN to ARGS.  Mirrors Rust eval/mod.rs apply_function."
  (cond
   ((nelisp--builtinp fn)
    ;; (builtin NAME) → registry dispatch via Rust helper
    (nelisp--apply-builtin-dispatch (cadr fn) args))
   ((nelisp--closurep fn)
    (nelisp--apply-closure fn args))
   ((nelisp--lambdap fn)
    (nelisp--apply-lambda fn args))
   ((nelisp--macrop fn)
    (signal 'wrong-type-argument
            (list 'function-not-macro fn)))
   (t
    (signal 'wrong-type-argument (list 'function fn)))))

(defun nelisp--expand-macro (macro-form arg-forms)
  "Expand (macro . LAMBDA) by calling LAMBDA on un-evaluated ARG-FORMS."
  (let ((inner (cdr macro-form)))
    (cond
     ((nelisp--lambdap inner)
      (nelisp--apply-lambda inner arg-forms))
     ((nelisp--closurep inner)
      (nelisp--apply-closure inner arg-forms))
     (t
      (signal 'wrong-type-argument (list 'malformed-macro macro-form))))))

(provide 'nelisp-stdlib-eval-core)
#+end_src

** §2.7 Takeover hook 設計

Rust =apply_combiner= (= mod.rs:186) の 3) plain function path に *flag
分岐* を入れる:

#+begin_src rust
// 3) Plain function — evaluate args, apply.
let args = eval_arg_list(tail, env)?;
if env.use_elisp_apply {
    // Stage 7.4.b takeover: delegate to elisp `nelisp--apply-fn'.
    // Build the form `(nelisp--apply-fn 'FUNC '(ARG...))' and eval
    // it through the regular Rust eval entry.  The elisp side then
    // walks FUNC's cons head and calls back into Rust frame
    // primitives (`nelisp--push-frame' / `nelisp--bind-local' /
    // `nelisp--apply-builtin-dispatch') as needed.
    let dispatch_form = Sexp::list_from(&[
        Sexp::Symbol("nelisp--apply-fn".into()),
        Sexp::list_from(&[Sexp::Symbol("quote".into()), func.clone()]),
        Sexp::list_from(&[
            Sexp::Symbol("quote".into()),
            Sexp::list_from(&args),
        ]),
    ]);
    return eval(&dispatch_form, env);
}
// Default (= Stage 7.4.a〜c の dormant 期間) は Rust path
return apply_function(&func, &args, env);
#+end_src

flag は =Env::use_elisp_apply: bool= field、default =false=。CLI で
=--use-elisp-apply= 指定時、または =(setq nelisp--use-elisp-apply t)=
時に Rust eval の C-level mutex 越しに ON にする。

注意点: macro 経路 (= step 2 の =is_macro(func)=) は **本 flag 経路を
通らない** — macro expansion driver 自身は Stage 7.4.b で elisp 化
するが、Rust =apply_combiner= の dispatch 順序は維持。理由: macro 展開は
=eval(expansion, env)= で再帰、再帰 entry が flag を尊重するので結果
として全 path が elisp 経由になる。

当初設計では =nelisp--bind-formals= を独立 defun として実装するつもり
だったが、smoke test で frame-stack 整合性 bug を発見:
- =defun nelisp--bind-formals= を呼ぶと、apply_function が *bind-formals
  自身の call-frame* を push する
- bind-formals の body 内 =(nelisp--bind-local NAME VAL)= は topmost
  (= bind-formals の call-frame) に bind してしまう
- bind-formals が return すると call-frame と共に bindings が消える

inline 実行する。これにより F_formals (= apply-lambda-inner が push した
frame) が topmost のまま保たれ、bind-local が正しい frame に刺さる。

=(let ((--nl-...))= を生む macro」(= =or= / =and= / 一部の =cond=) を

** §3.3 Stage 7.4.c — =--use-elisp-apply= flag 経路 + cross-equivalence ERT (SHIPPED)

実装:
- =build-tool/src/eval/env.rs=:
  - =Env::use_elisp_apply: bool= field 追加.  default false、env var
    =NELISP_USE_ELISP_APPLY= 非空で true 初期化
  - =Env::delegation_depth: u32= field 追加 (= 後述 cycle 解決の核)
  - =max_recursion= を 512 → 1024 に bump (= elisp dispatch overhead を
    吸収しつつ cargo test 2MB thread stack に収まる、4096 は overflow した)
- =build-tool/src/eval/mod.rs=:
  - =delegate_to_elisp_apply= helper: =(nelisp--apply-fn 'FUNC '(ARGS))=
    の dispatch form を組んで =eval=、その間 =delegation_depth= を ++/--
  - =is_builtin_value= helper: =(builtin NAME)= 形状判定で builtin を
    delegation gate から除外
  - =apply_combiner= step 3 の plain function path + lambda head path に
    flag 分岐:
    =if env.use_elisp_apply && env.delegation_depth == 0 && !is_builtin_value(&func) → delegate=
- =build-tool/src/eval/builtins.rs=:
  - =bi_set_use_elisp_apply= / =bi_get_use_elisp_apply= 2 builtin 追加
  - install list / dispatch arm に追加

CLI integration:
- =NELISP_USE_ELISP_APPLY=1 target/release/nelisp eval EXPR= (= 起動時
  flag-on)
- runtime toggle: =(nelisp--set-use-elisp-apply t)= / =(... nil)=、
  状態確認は =(nelisp--get-use-elisp-apply)=
- CLI flag (=--use-elisp-apply=) は採用せず (= env var + builtin で十分、
  CLI parser を肥大化させない)

設計上の発見 — *outermost-only delegation strategy*:

当初は「ALL 関数 call が elisp dispatch を経由」を目指したが、実装中に
深刻な cycle を発見:

  =consp= (defun) → delegate → =nelisp--apply-fn= body の =(cond ...)=
  expansion で =(null clauses)= 呼び出し → null は defun → delegate
  → cond expansion で再び null → 無限再帰

helper exemption list で逃げようとしたが、helpers が依存する defun
(=consp= / =null= / =nth= / =memq= / =<= / =1+= 等) を全て exempt 必要
で保守不可.  解決として =Env::delegation_depth= counter を導入:

- =delegate_to_elisp_apply= 進入時に depth++、退出時に --
- =apply_combiner= の delegate gate に =delegation_depth == 0= 条件追加
- 結果: **outermost user-level call のみ elisp dispatch、内部 (= helper
  + macro expansion + 内部 defun) は全て Rust apply_function**

trade-off: cross-equivalence は最外 1 layer のみ.  深い coverage は
Stage 7.4.d の default flip 後に「全 user entry point が elisp dispatch」
で達成する.  Stage 7.4.c の ERT は最終値の同一性で検証するので outermost
で十分.

ERT (= test/nelisp-stdlib-eval-core-test.el に追加、stage 7.4.b の 16
件 + 7.4.c の 6 件 = 22 件):
- =flag-recursive-defun-dispatch= = =(fact 10)= = 3628800
- =flag-closure-write-through= = =(let ((x 0)) (let ((f (lambda () (setq x (1+ x))))) (funcall f) (funcall f) x))= = 2
- =flag-rest-arg-defun= = =(defun count-args (&rest xs) (length xs))= +
  =(count-args 'a 'b 'c)= = 3
- =flag-funcall-builtin-passthrough= = =(funcall #'cons 1 2)= = (1 . 2)
- =flag-runtime-toggle-roundtrip= = nil → t → nil の toggle
- =flag-recursion-depth-still-bounded= = =(defun loop () (loop)) (loop)=
  が =max-lisp-eval-depth= で停止 (= delegation_depth + 1024 budget が
  Rust stack overflow を防ぐ)

cross-equivalence: 既存 ERT 全 suite を =NELISP_USE_ELISP_APPLY=1= で
再走するのは Stage 7.4.d 移行と一緒に実施 (= bench gate と並走).

確認:
- =cargo build --release=: PASS
- runtime smoke (=fact 10= / closure write-through / =princ=): PASS
- ERT (eval-core-test.el 単独): 22/22 PASS
- =cargo test --release --lib -- --test-threads=1=: 431/431 PASS (= bump

** §3.5 sub-stage 順序

a → b → c → d を **直列** で進める:
- a (Rust primitive 追加) は b の elisp body が呼ぶので先必須
- b (elisp install) は c の dispatch 経路の dispatch 先なので先必須
- c (flag 経路) は d の default flip の前提条件
- d (default flip) は Stage 7.5 の前提条件

各 sub-stage 終了時に commit + worklog 1 件。

* §6 共通原則

1. **Tier 1-only body 制約**: =eval-core.el= 内 elisp body は Stage 7.3
   までに install 済 macro + Tier 1 primitive + Rust builtin のみで書く。
   cl-loop / seq-* / 高階 HOF を呼ばない (= bootstrap cycle 防止)。
2. **parallel install + flag-gated takeover**: Stage 7.2 / 7.3 と同形。
   default 切り替えは bench gate + ERT 両 mode PASS 後のみ。
3. **Rust frame primitives は不変**: =Env::push_frame= / =push_captured=
   / =bind_local= 等は Stage 7.4 で *使われ方が増える* が、shape は変えない。
   Stage 7.5 / 7.6 の前後で Rust eval 全削除しても、これら primitive
   は **永続的に Rust 残**。
4. **observable contract preservation**: 全 Sexp 結果 / signal /
   recursion-depth エラーが Rust dispatch と elisp dispatch で同一。
   違いは wallclock のみ。
5. **escape hatch**: =--use-rust-apply= flag を Stage 7.5 までは残す。
   bench regression 発見時に default を rollback できる退避路を確保。
