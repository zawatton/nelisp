* §1 67 dispatch arm full inventory

** §1.1 Inventory 方法論

=build-tool/src/eval/builtins.rs= の dispatch 関数 (line 458-808) の
match arm を = 全 132 件 enumerate。各 arm に以下属性を付与:

- *line*: dispatch 関数内行
- *arity*: =require_arity()= 第 3-4 引数 (= min, max)
- *signature*: 引数型と Env 依存
- *Tier*:
  - *Tier 1* = pure elisp 移管可、新 ABI 不要
  - *Tier 2* = 新 ABI mode + bridge primitive 必要
  - *Tier 3* = =Env= state 直接操作、§86.3 hand-off
  - *KEEP* = FFI / system / bridge primitive、elisp 移管不可
- *target sub-stage*: §86.1.a-g 配置

** §1.2 Tier 1 — pure elisp 移管 (= 21 arm)

新 ABI 不要、既存 =nl-jit-call-out-*= bridge or 既存 elisp builtin で
表現可。

*** §1.2.1 算術 / equality (= 4 arm、§86.1.b)

| arm                  | line | arity | bi_* fn        | sig        | target |
|----------------------+------+-------+----------------+------------+--------|
| =/=                   | 466  | 1+    | =bi_div=       | Sexp[]     | §86.1.b |
| =equal=              | 480  | 2     | =bi_equal=     | Sexp[]     | §86.1.b |
| =nelisp--ref-eq=     | 481  | 2     | =bi_ref_eq=    | Sexp[]     | §86.1.b |
| =sxhash=             | 531  | 1     | =bi_sxhash=    | Sexp[]     | §86.1.b |

migration note:
- =/= は 1-arg (reciprocal) と n-arg (左折り) の両方を扱う。Doc 84
  の Float family 同様、Int / Float promote → =nl-jit-call-float-float
  "nl_jit_float_div"= or 既存 =nelisp--*-float= delegation
- =equal= は recursive structural equality + Doc 50 stage 5a の cycle-
  safe path (= visited hash via =nelisp--ref-eq=)。pure elisp で書ける
  pattern が Doc 80 substrate に存在
- =nelisp--ref-eq= は =Rc::ptr_eq= のため Rust 必須に見えるが、JIT
  trampoline (=nelisp_jit_eq_inline=) が tag + payload 比較を既に提供
  しており elisp 側で薄く包めば ref-eq に等しい。ただし *残置候補* (=
  既存 =nl-jit-call-ptr-ptr "nelisp_jit_eq_inline"= が ref-identity
  の代替として機能する) で詳細は §7 risk
- =sxhash= は visit set + recursive hash で elisp ports 可能、ただし
  precision (= sxhash-equal collision above 2^53) に注意

| arm                  | line | arity | bi_* fn          | sig    | target |

migration note:
- =vector= は raw =Sexp::vector(args.to_vec())= の inline 1 行、=make-

* §7 Risk / unknowns

** §7.1 perf regression — 現実的 hot path 計測

elisp 移管が hot path に hit する arm は限定的:
- =type-of= = predicate 連鎖の起点で頻繁に呼ばれる (= =stringp= /
  =integerp= / =consp= 全部の依存)。elisp 化で 2-3 layer の
  interpreter overhead 増 = perf 退行 5-15% 想定
- =equal= = hash-table key 比較 + ERT assertion で多用、cycle-safe
  port は visited alist (= O(n)) で巨大データ比較に 100x 退行 risk
- =sxhash= = hash-table 全操作の起点、collision avoidance 維持必要

対策:
- =type-of= は §86.1.a 完了直後に bench、5% 超なら =nelisp-cc=
  (= AOT compiler) で elisp wrapper を native code 化 = §86.1.g 内で
  config 化 (= =nelisp-cc-aot-port-bin t=)
- =equal= は §86.1.b の Tier 1 として ship した後、退行が観測されたら
  Tier 2 への migrate (= =nl-jit-call-out-*= 経由 visited hash) を §86.1.f に追加
- =sxhash= は ERT snapshot で hash 値 invariance を fix、collision 改
  善は scope 外 (= 既存 Rust impl の collision rate を維持)

** §7.2 dispatch table overhead

=match name { ... }= の arm 数が 132 → ~80 に減ると、Rust match の
linear scan が早くなる方向 = 退行 risk なし。逆に =install_builtins= の
sentinel 設置で 96 entry × 一意 string allocation = startup cost
あり、§86.1.g の cleanup でこれも削減。

** §7.3 Tier 3 deferral 順序問題

Doc 86 §9.3 で articulated chicken-and-egg:
- Tier 3 (= =defalias= / =fset= / =set= / =boundp= / =fboundp= 等) は
  Doc 89 §86.3 (= env.rs elisp 化) と統合する必要
- ただし elisp wrapper 自身が =fset= を使うため、Tier 3 を完全 elisp
  化できない (= =defalias= が elisp で書かれていると =defalias= 自身
  の expand に env access が必要)
- 解決策: Tier 0 minimal env (= ~50 LOC Rust) を残置し、=defalias= /
  =fset= / =set= の最低限 dispatch のみ提供、その上に上層 Tier 3
  arm を elisp 化

本 Doc 87 では Tier 3 を全て §86.3 hand-off 扱いとし、§86.1 では
touch しない。Doc 89 で chicken-and-egg を解決する design を articulate。

** §7.4 string-match-p (= regex backend) の依存

=string-match-p= は Rust regex crate に依存、elisp 純実装は不可。
§86.1.f で primitive 1 件 (= =nl_jit_regex_match=) として残置 + thin
wrapper で elisp lift。Phase 47 (= ultimate 0 Rust) では regex も
elisp で書く必要があり、Doc 85 §6 で audit 予定 = 本 Doc OOS。

** §7.5 Doc 86 §2.2 67 arm 表記との整合

=nelisp--*= prefix は 26 件、=nl-*= prefix は 35 件、その他は 71 件。
本 Doc では 132 全 arm を base に Tier 1/2/3/KEEP に分類、Doc 86 表記
は維持しつつ実数で運用。

count、実 dispatch arm 132、Tier 移管対象 56") を §86.1.g 完了後に
