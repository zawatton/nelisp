* §1 Methodology

A (static intersection) 方式を採用。

*** なぜ A 方式
| 方式 | 精度 | コスト | 判定 |
|------+------+--------+------|
| A. 静的 intersection (host subrs ∩ NeLisp src 参照) | 中 | 1-2h | ◎ 採用 |
| B. 動的 trace (hook で C subr 呼出を log) | 高 | 半日+ | △ 補助で後日 |
| C. bytecode 走査 | 高 | 3h+ | ✗ 投資対効果低 |

*** 手順
1. 現 Emacs で =mapatoms + subrp= により host C 定義 symbol set S を enumerate
2. =src/*.el= + =bench/*.el= 13 ファイルの sexp walk で function 位置 symbol 収集
   + =funcall= / =apply= / =mapcar= 等の高階関数の第 1 引数 =(function SYM)= / =(quote SYM)= も抽出
3. 交差 D = NeLisp が C に依存している symbol 集合
4. 各 symbol に (arity, ref-count, first-3-sites) 付与、ref-count 降順で sort

* §4 Package-authoring 自立の gap

"NeLisp 単独で =(require 'my-pkg)= が成立する" ための最小要件:

*** 必要な施設
1. *Reader* — =src/nelisp-read.el= 既存、=read= / =read-from-string= 相当
2. *Evaluator* — =src/nelisp-eval.el= 既存、T0 特殊形式 + T1 呼出を NeLisp
   内で完結して dispatch できる (現状: bootstrap 後は成立、起動時のみ host eval)
3. *Defining forms* — =defun= / =defmacro= / =defvar= / =defconst=: T0 subset
4. *Loader* — =src/nelisp-load.el= 既存、file IO 3 subr 経由で .el を読む
5. *Provide/require 機構* — T1 subset、=provide= / =require=

*** bootstrap gap
"純 NeLisp で loop を閉じる" ために不足している要素:

- =load-file= 的 API で NeLisp .el を NeLisp reader にかけ、NeLisp evaluator で
  forms を実行する経路が *未公開* (現状: =require= は host load-path 経由)
- =load-path= 相当の search mechanism が NeLisp 側に無い (host 側のものを借用)
- =defmacro= 展開時に host =macroexpand= を呼んでいないか未確認
  (Check フェーズで要検証)

*** Phase 5 pre-req として扱うべき subset
Phase 5 本体 (core 置換) に入る前にまず閉じるべき gap:

| gap | 現状 | 必要な作業 |
|-----+------+------------|
| =nelisp-load-file= (NeLisp reader + eval で .el を load) | partial | reader/eval の glue + cycle handling |
| =nelisp-load-path= | 未実装 | defcustom + search order |
| macroexpand が NeLisp-native か | 未検証 | =macroexpand-all-1= 相当を NeLisp 内に確認 |
| =nelisp-require= (nelisp-provide-ed feature lookup) | 未実装 | feature registry + circular detection |

これらが揃えば "NeLisp だけで package を読み込む" bar に到達する。核となる
T1/T2 subr は既に揃っているので、新規 C 依存は追加発生しない見込み。

* §5 Phase 5 scope 案

audit の発見を踏まえ、Phase 5 は 2 軸に分ける:

*** Phase 5-A: NeLisp 単独 package 作成自立 (先行)
- §4 で列挙した bootstrap gap を埋める
- 成功条件: 純 NeLisp で書かれた toy package (=hello-world.nel.el= 等) を
  NeLisp 自身の loader で require → NeLisp eval で動作する
- 期待スコープ: ~2-4 週。新規 C 依存の追加はなし、既存 T1/T2 の組み換えのみ

*** Phase 5-B: Emacs core 置換 (後続)
- T3 (buffer/window/frame/redisplay) の substrate API を NeLisp side に建設
- *current NeLisp src は T3 subr を参照していない* ので、これは "既存コードの
  置き換え" ではなく "substrate 新規提供"
- 対象 module (memory =project_nelisp.md= の Phase 0 lock 済 decision より):
  1. buffer (テキストバッファ層)
  2. redisplay (描画層)
  3. event-loop (入力処理層)
- 各 module に対し "NeLisp-hosted package がそれを触れる最小 API" を定義する
  設計 doc が必要 (Doc 12 / 13 / 14 相当)

*** 優先順位の根拠
- T3=0 は "NeLisp core is ready, substrate はまだ" を意味する
- T3 substrate を作っても、それを使う package が NeLisp 単独で書けないと
  dog-fooding できない → Phase 5-A を先に閉じるべき
- Phase 5-A が閉じれば "NeLisp で書いた package が NeLisp で動く" loop が
  成立、Phase 5-B の dog-food 環境が整う

* §8 Check-1: T1 VM dispatch coverage (2026-04-24 追補)

Plan の §6 #1 "T1 classification の VM dispatch table 照合" を実施。

** 発見 (§3 の再解釈)

§3 Tier summary で "T1 89 件 ― 既に NeLisp VM 実装済のはず (要 cross check)"
と書いたが、cross check の結果この前提は *誤り*。NeLisp は T1 subr を VM 内
で再実装していない。代わりに 2 経路で *host C に委譲* している:

1. =nelisp--primitive-symbols= (=src/nelisp-eval.el:604=) に列挙された symbol は
   load 時に =(puthash sym (symbol-function sym) nelisp--functions)= で
   host C の =symbol-function= をそのまま NeLisp 関数表に登録。
   runtime に user コードが当該 symbol を call すると =nelisp--apply=
   経由で host C subr を実行。
2. 列挙漏れの symbol は =nelisp--apply= の fallthrough 節
   =((functionp fn) (apply fn args))= (=nelisp-eval.el:562=) で直接 host C dispatch。

どちらの経路でも NeLisp VM は単なる *薄い router* として機能し、実行自体は
host C subr で行う。SBCL 系の pragmatic delegation 設計。

** bytecode VM の T1 inline 化

=nelisp-bytecode.el= の opcode table に *inline 化された T1 subr* は以下のみ:
=1+=, =1-=, =not=, =+=, =-= (2-arg), =<=, =>=, =eq= (2-arg), =car=, =cdr=,
=cons=, =list= (1-4 + N)。これら以外は VM が =CALL= opcode を吐いて
=nelisp--apply= に dispatch する。

** NeLisp-aware wrapper を持つ T1 (高階関数 + require 系)

以下は closure 可視性や NeLisp state 参照のため =nelisp--builtin-*= wrapper
経由で intercept:
- =funcall=, =apply= (NeLisp closure を受ける)
- =mapcar=, =mapc=, =mapconcat= (NeLisp closure を受ける)
- =boundp=, =fboundp=, =symbol-value=, =symbol-function= (NeLisp の
  variable / function table を先に見る)
- =require=, =provide= (NeLisp 側 feature registry)

** =nelisp--primitive-symbols= に *未登録* の gap 13 件

audit §2 Raw data で NeLisp src が参照している T1 subr のうち、=nelisp--
primitive-symbols= に列挙されていないもの。これらは runtime 時に
fallthrough で host C に落ちる:

| symbol | refs | 性質 | 対応優先 |
|--------+------+------+----------|
| =maphash= | 6 | *HOF* — NeLisp closure を受け得るので wrapper 必須 | ★ 高 |
| =plist-get= | 32 | pure data | 中 (intercept 不要だが primitive 登録が clean) |
| =plist-put= | 3 | 同上 | 中 |
| =make-vector= | 4 | pure data | 低 |
| =vector= | 4 | pure data | 低 |
| =caddr= | 3 | pure data | 低 |
| =cdddr= | 2 | pure data | 低 |
| =ash= | 2 | pure data | 低 |
| =logand= | 2 | pure data | 低 |
| =logior= | 1 | pure data | 低 |
| =car-safe= | 1 | pure data | 低 |
| =float= | 1 | pure data | 低 |
| =message= | 18 | side-effect | 低 (fallthrough OK) |

*semantic gap は =maphash= のみ*。NeLisp closure を user が maphash に渡した
場合、host C の =maphash= が host flavor の =funcall= で呼び出そうとして
失敗する経路がある。Phase 5-A 着手前に wrapper 追加が妥当。他は
"primitive 表に並べて一貫性を取る" 整備案件。

** Phase 5-A 先行案への影響

- *変わらず先行妥当*: §4 で挙げた "NeLisp 単独 package 作成自立" の gap
  (=nelisp-load-file= / =nelisp-load-path= / =nelisp-require= /
  macroexpand 非依存化) は上記委譲モデルと両立する。user package が
  =plist-get= 等を call しても host C で動く
- *追加 pre-req*: =maphash= wrapper を =nelisp--install-primitives= に追加
  (1-2h)。残り 12 件も primitive 表に列挙して "どの T1 が NeLisp 認識されて
  いるか" の境界を明確化 (半日)
- *不変な前提*: NeLisp は T1 を再実装しない方針と cross check により判明。
  "Emacs core 置換" は T1 まで全て Lisp 化する意味ではなく、T3 substrate
  を Lisp で *新設* する意味

** Phase 5 scope の再定義

本 Check の結果、Phase 5 scope は以下のように精緻化できる:

- *Phase 5-A (package-authoring 自立、先行)*:
  - gap: load-file / load-path / require / macroexpand
