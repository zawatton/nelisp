#+TITLE: Doc 126 — image.rs retirement (NELIMG v3 → pure-elisp eval boot)
#+DATE: 2026-05-18
#+STATUS: DRAFT

* §126.1 Premise

=build-tool/src/image.rs= is 1636 LOC.  It implements NELIMG v3, a
binary frozen-heap format used to ship STDLIB pre-evaluated into the
production binary.  Decode bypasses the reader + eval-loop.

After Doc 116 (= elisp reader shipped) + Doc 122 (AOT alloc /
string / vector / record grammar), the eval-loop can decode source
forms at startup nearly as fast as binary blob deserialization, while
deleting *all of image.rs* (= −1636 LOC) plus its baker glue (=
=bin/nelisp-baker.rs= −299 LOC, =STDLIB_IMAGES= in env.rs and friends).

Endpoint expectation: image.rs deleted entirely.

* §126.2 What image.rs does today

| Surface          | LOC | Role                                                       |
|------------------+-----+------------------------------------------------------------|
| encode_v3*       | 280 | Walk Env→graph, serialize NlBox graph + globals + fallback |
| decode_v3*       | 320 | Parse byte stream → reconstruct NlBox graph → install Env  |
| NodeTable        | 100 | Dedup + idx assignment during encode                       |
| write_v3_*       | 250 | Per-variant emitters (Cons, Vector, Record, CharTable, …)  |
| read_v3_* (in Reader) | 250 | Per-variant readers                                  |
| iterative_bake_one | 70 | Driver: load one .el → eval → encode_v3 → write .image    |
| char_table body  | 200 | Special-cased CharTable serialization                     |
| misc helpers     | 166 | Magic / version / cursor / error types                    |

Used by:
- =Env::new_global= → =decode_v3_into= for every STDLIB_IMAGES entry
- =bin/nelisp-baker.rs= → =iterative_bake_one= for `make bake-images`
- Image-baker integration tests in eval/tests.rs

* §126.3 Replacement strategy

** §126.A — boot path: =.el= source → reader → eval (Phase 1)

Swap =STDLIB_IMAGES= from `include_bytes!("*.image")` to
`include_str!("*.el")`.  Replace `decode_v3_into` with a 5-line
loader:

#+begin_src rust
for (name, source) in STDLIB_SOURCES {
    let forms = elisp_reader_read_all(source)?;
    for form in forms { env.eval(&form)?; }
}
#+end_src

The elisp reader is already AOT-compiled (Doc 116.C); the
eval-loop is the production hot path.  Startup time becomes
(parse + eval) instead of (decode_v3); empirically the difference is
in the 100ms-range for ~10K STDLIB lines (= acceptable for a build-
tool binary).

** §126.B — delete encode_v3 + NodeTable + write_v3_* (Phase 2)

Once §126.A boots, =encode_v3= has no callers from the production
binary (only from baker + tests).  Move encode_v3 + supporting
helpers to =bin/nelisp-baker.rs= as cfg-gated functions, then…

** §126.C — delete the baker (Phase 3)

=iterative_bake_one= becomes dead once `make bake-images` is replaced
by `make stdlib-test` (= just run the eval-loop on the source and
verify correctness).  Delete =bin/nelisp-baker.rs= entirely.

** §126.D — delete image.rs (Phase 4)

After §A+B+C, image.rs has no remaining callers.  Delete the file.

Estimated Rust削減 across Phases 1-4: −1636 (image.rs) + −299
(nelisp-baker.rs) + ~50 (extern decls and module wiring in lib.rs +
mod.rs) ≈ **−1985 LOC**.

* §126.4 Risks + mitigations

** Risk: startup time regression

Decode binary is ~5x faster than parse-and-eval for the same source.
For a build-tool / repl binary this is acceptable.  Mitigation:
benchmark before/after; if >500ms regression, gate via
`NELISP_IMAGE_FAST_BOOT=1` to keep the binary path for power users.

** Risk: refcount graph isomorphism

NELIMG dedupes shared substructure across STDLIB; re-evaluating
produces fresh allocations.  Memory usage may grow.  Mitigation:
benchmark RSS at idle; if >20% growth, add a one-shot
`(garbage-collect-cycles)` after STDLIB load.

** Risk: image-baker tests in =eval/tests.rs= (~80 tests)

These exercise encode_v3/decode_v3 round-trip.  After §126.B these
move into baker.rs's #[cfg(test)] mod.  Net 0 LOC削減 from the
tests themselves; just relocation.

** Risk: =--verify-elisp-fixtures= gate (Doc 95 §95.e)

Cross-impl byte-identity gate uses encode_v3 to compare elisp Sexp
DSL output against Rust output.  This gate is preserved in §126.B
(baker-cfg-gated encode_v3 stays).

* §126.A.diag — eval-boot blocker (2026-05-18)

** Symptom

#+begin_src
NELISP_EVAL_BOOT=1 ./target/release/nelisp eval '(+ 1 2)'
=> panic: nelisp-stdlib.el eval-boot eval failed at form #7:
   wrong-type-argument: (symbolp 1)
   form: (defun 1 (x) (nelisp--add2 x 1))
#+end_src

The source form is ~(defun 1+ (x) (nelisp--add2 x 1))~ but the
Rust-side ~reader::read_all~ (= elisp §116.A/B lexer pipeline via
~elisp_cc_spike::reader_lex_one~) parses the symbol ~1+~ as the
integer ~1~ followed by no other token; the trailing ~+~ is silently
consumed.

Repro (= isolated repro through ~reader::read_all~):

#+begin_src
"(defun 1+ (x) (nelisp--add2 x 1))"  =>  (defun 1 (x) (nelisp--add2 x 1))
"1+"                                  =>  1   (Sexp::Int(1))
"(a 1+)"                              =>  (a 1)
"(a 1+ b)"                            =>  (a 1 b)
#+end_src

** Root cause

~lisp/nelisp-cc-reader-lexer.el::nelisp_reader_classify_step~ classifies
an atom by walking its bytes and updating a 4-bit class field
(has-dot / has-e / sym-bit / saw-digit).  Sign bytes (`+'/`-') are
"tolerated, no class change" — they neither set the sym-bit nor
terminate the atom.  Given ~1+~:

1. byte ~1~ → ~saw-digit~ bit set, class = 8
2. byte ~+~ → "tolerated", class unchanged
3. end-of-atom (whitespace/paren) → final classifier:
   - sym-bit (bit 2) not set
   - saw-digit (bit 3) set
   - dot/e (bit 0|1) not set
   - → returns 20 (kind Int)
4. Parser converts the captured bytes ~"1+"~ via ~i64::from_str~,
   which... actually accepts ~"1+"~?  No — ~i64::from_str("1+")~
   fails in Rust, but the elisp side's ~nelisp_atoi~ in
   ~nelisp-cc-reader-parser.el~ stops at the non-digit byte and
   returns 1.

The classifier was designed for ~+1~ / ~-1~ / ~1.5e-3~ where signs
appear at boundary positions (= start of atom or just after ~e~).
Post-digit non-exponent signs (~1+~, ~1-~) are not legal numeric
syntax in any language, but the lexer accepts them as the integer
prefix and the parser silently drops the suffix.  Standard Elisp
treats ~1+~ as a single symbol (= the unary increment function name).

** Why ~.image~ baker on disk works

The 28 ~lisp/*.image~ files committed to the repo were baked at
commit ~a8b322ad~ (Doc 98 §98.3 cutover, 2026-05-11) with a then-
current reader.  Several reader-evolving commits landed after:
~60e5c25b~ §116.B+, ~28f8faaa~ §122.G, ~48d2c020~ §116.D delete
lexer.rs+parser.rs.  Running ~nelisp-baker --check~ on current
~main~ reproduces the same ~(symbolp 1)~ panic — the on-disk
.image files predate the lexer regression.

** Impact map

Files containing ~1+~ / ~1-~ as code symbols (= not in comments):

| File                          | Load pos | Has ~1+~/~1-~ as code |
|-------------------------------+----------+------------------------|
| nelisp-jit-substrate.el       |        1 | comments only          |
| nelisp-stdlib-eval-special.el |        5 | comments only          |
| nelisp-stdlib.el              |        7 | YES (= defun 1+/1-)    |
| nelisp-stdlib-reader.el       |       21 | YES                    |
| (rest)                        |    8..28 | varies                 |

The chicken-and-egg: eval-boot needs to load ~nelisp-stdlib.el~
(position 7) using the Rust reader, but stdlib.el is the first file
that uses ~1+~ as code, and the post-bootstrap elisp reader
(~nelisp--read-all-from-string-impl~) lives at position 21
(~nelisp-stdlib-reader.el~).

** Resolution options

A. **Fix the lexer** (recommended): patch
   ~nelisp_reader_classify_step~ to treat post-digit ~+~/~-~ as
   sym-forcing unless preceded by ~e~/~E~ (= exponent context).
   This is a 4-line patch but requires regenerating the
   elisp-CC-compiled object files via
   ~scripts/compile-elisp-objects.el~ (= emacs --batch build
   step).  Once fixed, eval-boot proceeds; baker also unblocks.

B. **Hot-switch reader mid-boot**: load the first 6 STDLIB files
   with the Rust reader (= they don't use ~1+~ as code), then
   re-load ~nelisp-stdlib.el~ + later via a path that doesn't
   need the post-bootstrap elisp reader.  Doesn't actually work
   because the post-bootstrap reader requires ~stdlib.el~ for
   arithmetic.

C. **Rewrite ~1+~/~1-~ in STDLIB sources**: replace every
   ~(1+ x)~ with ~(nelisp--add2 x 1)~.  Mechanical but invasive
   across ~20 files.

Recommendation: option A.  Doc 126.A re-engagement after the
lexer patch lands.

** Scope of §126.A as shipped

The parallel boot path is wired and gated:

- ~build-tool/src/lib.rs~ — un-gates ~pub mod reader~ so the
  production binary can ~use crate::reader::read_all~ (= the
  reader was previously ~#[cfg(any(test, feature = "image-baker"))]~).
- ~build-tool/src/eval/env.rs::new_global~ — augments
  ~STDLIB_IMAGES~ → ~STDLIB_FILES~ tuple (name, include_str!.el,
  include_bytes!.image), dispatches on ~NELISP_EVAL_BOOT~ env-var
  to choose ~reader::read_all~ + ~eval~ per form vs the legacy
  ~decode_v3_into~ path.  Failure mode includes the failing form
  + index in the panic message.  Optional ~NELISP_EVAL_BOOT_TRACE=1~
  prints per-form load progress to stderr.

Verification:

- Legacy ~decode_v3_into~ path: ~nelisp eval '(+ 1 2)'~ → 3 ✓
- Eval-boot path: ~NELISP_EVAL_BOOT=1 nelisp eval '(+ 1 2)'~ →
  panic on form #7 of nelisp-stdlib.el (see Symptom).

Rust LOC delta: +59 (env.rs STDLIB_FILES tuple expansion + boot
dispatch), -7 (lib.rs reader feature gate removed).  Net +52 LOC,
consistent with §126.A being parallel-path scaffolding.  Doc 126.B-D
will retire image.rs (~1567 LOC) + nelisp-baker.rs (~299 LOC) once
the lexer patch lands and eval-boot is validated as default.
