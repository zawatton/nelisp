** 1.3 SBCL gencgc + Emacs alloc.c GC dual precedent reference

NeLisp の GC inner は SBCL =gencgc.c= (~10000 LOC C + asm、25 年維持)
と Emacs =alloc.c= GC 部分 (~6000 LOC C、Emacs 1985-) の dual
precedent を踏まえ、*runtime purity 軸で両者を超え、GC capability
軸では両者の MVP subset* を狙う (Doc 28 / Doc 29 と同 pattern)。

| 軸           | SBCL gencgc              | Emacs alloc.c GC         | NeLisp Phase 7.3          |
|--------------+--------------------------+--------------------------+---------------------------|
| 実装 lang    | C + asm                  | C                        | NeLisp                    |
| GC type      | generational mark-region | mark-sweep (single gen)  | generational mark-sweep   |
| pause type   | stop-the-world (主)      | stop-the-world           | stop-the-world MVP        |
| LOC          | ~10000                   | ~6000                    | 2000-4000                 |
| concurrent   | partially (SBCL 2.x)     | no                       | no MVP (Phase 7.5+ 検討)  |
| write barrier | card-marking            | none (single gen)        | card-marking 4KB page     |
| nursery copy | semispace (Cheney)       | n/a                      | semispace (Cheney)        |
| safe-point   | code-patching            | yield point              | Doc 28 §2.9 metadata 経由 |
| 期間         | 25 年維持 (1999-)        | 40 年維持 (1985-)        | 6-10 週間 (anvil leverage 前提) |

*SBCL gencgc precedent との差別化軸*:
1. *Purity*: NeLisp は host C runtime を Phase 7.3 で永久放棄、pure
   NeLisp + Phase 7.2 mmap heap span の pipeline を確立する
2. *Bootstrap simplicity*: SBCL は Rhodes target-2 cold-init で初の
   GC pause が発生 (Doc 28 §1.4 3 artifact 連携)、NeLisp は Phase 7.2
   region table = GC metadata contract provider が確立済のため cold-init
   時 GC は no-op で起動可能 (= Phase 7.3 GC は warm phase 以降のみ走行)
3. *concurrent GC は Phase 7.5+ で検討*: 本 phase は stop-the-world MVP

*Emacs alloc.c precedent との差別化軸*:
1. mark-sweep の発想は踏襲、ただし 2-generation 化 (Doc 29 §2.2) で
   nursery copy + tenured mark-sweep に分割
2. host C dependency を完全除去、=malloc= / =free= は Phase 7.0 syscall
   stub 経由 =mmap= に置換 (Phase 7.2 で実施済)

** 1.4 Doc 28 §2.9 GC metadata contract / Doc 29 §2.6 region table との関係

** 1.5 SBCL Rhodes 5-stage との対応

Doc 27 §1.4 / Doc 28 §1.4 Rhodes 5-stage methodology に対し、Phase 7.3
は *target-2 cold-init で初の GC pause が発生する位置* に対応。Doc 28
§1.4 の 3 artifact (A1 = host bytecode compiler / A2 = seed native
compiler / A3 = self-compiled native compiler) 連携で:

| Rhodes stage | Phase 7.3 における GC 関与                                          |
|--------------+---------------------------------------------------------------------|
| host-1       | host Emacs 上では host GC 経由、Phase 7.3 GC は inactive            |
| target-1     | Phase 7.0 syscall stub + Phase 7.1 backend は GC 走行不要             |
| host-2       | A2 (seed native compiler) が =nelisp-cc.el= を再 compile、まだ host GC |
| genesis-2    | A3 候補 image を mmap heap 上に構築、region table 登録 (Phase 7.2)    |
| target-2     | *cold-init で初の GC pause = Phase 7.3 mark/sweep 初走行*             |

target-2 cold-init で初の GC pause が発生する点は SBCL 25 年で
"genesis-2 to target-2 transition" の hardest part として記録されてきた
(Rhodes 2008 §3.2 "most fragile")。NeLisp は Phase 3c で reachability
walker を確立済 + Phase 7.2 region table が provider 側を実装済の
ため、SBCL より transition risk が低い (Doc 27 §1.4 既存資産)。

** 2.1 GC algorithm [推奨 = A mark-sweep + generational]

| 候補 | 戦略                                                  | 利点                                       |
|------+-------------------------------------------------------+--------------------------------------------|
| A    | mark-sweep + generational (nursery copy / tenured mark-sweep) (default) | SBCL/Emacs dual precedent middle path、Phase 7.5+ 拡張余地 |
| B    | 単純 mark-sweep (single generation)                    | 実装最簡だが Phase 5-B selfhost で性能不足 |
| C    | mark-region (= G1GC 風)                                | scope 過大、Phase 8+ で検討                |

*推奨 A 理由*:
- Doc 29 §2.1 / §2.2 LOCKED で 2-generation heap layout が確定済、
  Phase 7.3 は collector 本体を heap layout に合わせる
- short-lived object 95%+ という generational hypothesis (Wilson 1992)
  は Lisp で実証済、nursery copy で minor GC の pause time を圧縮可能
- tenured mark-sweep は Emacs precedent + Phase 3c walker の延長で
  実装容易
- Phase 7.5+ で incremental / concurrent / mark-compact 拡張可

** 2.2 generation 管理 [推奨 = A 2 世代 (nursery + tenured)]

| 候補 | 戦略                                              | 利点                                       |
|------+---------------------------------------------------+--------------------------------------------|
| A    | 2 世代 (nursery + tenured) (default、Doc 29 §2.2 整合) | generational hypothesis 活用、SBCL/Emacs middle path |
| B    | 単一世代                                           | Phase 7.3 内 scope 縮小だが Doc 29 §2.2 と整合せず |
| C    | 3+ 世代                                            | scope 過大、Phase 8+ で検討                |

*推奨 A 理由*:
- Doc 29 §2.2 LOCKED 2 世代 (nursery 4MB / tenured 64MB) と完全整合
- 単一世代に縮小すると Doc 29 retrofit 必須、ABI 破綻
- 3+ 世代は SBCL でも 25 年導入されず、generational hypothesis の
  marginal return が小さい

** 2.3 mark phase [推奨 = A tri-color marking + work-list]

| 候補 | 戦略                                              | 利点                                       |
|------+---------------------------------------------------+--------------------------------------------|
| A    | tri-color marking (white/grey/black) + work-list (default) | 標準的、incremental 拡張容易               |
| B    | 単純 DFS (Phase 3c 既存)                          | Phase 3c walker 流用最簡、stack overflow risk |
| C    | parallel mark (multi-thread)                      | scope 過大、Phase 7.5+ で検討              |

*推奨 A 理由*:
- tri-color invariant は Dijkstra 1978 / Steele 1975 で確立、
  incremental / concurrent 拡張時の前提
- work-list (queue) で stack overflow 回避、Phase 3c DFS 比 deep graph
  に強い
- Phase 7.1 backend safe-point + GC metadata contract で root を取得、
  heap object grey 化 → black 化の sweep loop が Phase 7.1 native JIT で
  C 同等速度

mark phase pseudo code (例示、最小限):
#+begin_src elisp
(defun nelisp-gc--mark-roots ()
  (dolist (root (nelisp-gc--collect-roots))
    (nelisp-gc--mark-grey root)))

(defun nelisp-gc--mark-loop ()
  (while-let ((obj (nelisp-gc--worklist-pop)))
    (dolist (child (nelisp-gc--children obj))
      (when (nelisp-gc--white-p child)
        (nelisp-gc--mark-grey child)))
    (nelisp-gc--mark-black obj)))
#+end_src

** 2.4 sweep phase [推奨 = A per-pool free-list rebuild]

| 候補 | 戦略                                                  | 利点                                       |
|------+-------------------------------------------------------+--------------------------------------------|
| A    | per-pool free-list rebuild (default、Doc 29 §2.5 整合) | Phase 7.2 per-type pool と整合、type-aware |
| B    | 単一 free-list                                         | scope 縮小だが Phase 7.2 retrofit          |
| C    | mark-compact (full compaction)                         | fragmentation 解消だが scope 過大          |

*推奨 A 理由*:
- Doc 29 §2.5 per-type pool (cons / string / vector / closure) と完全
  整合、各 pool の free-list を rebuild する形
- type-aware sweep で finalizer hook (Phase 3c) が type 別に呼び出せる
- mark-compact は Phase 7.5+ で defragmentation 単独 phase として検討

** 2.5 promotion [推奨 = A nursery 2 回 survive で tenured 昇格 (Cheney semispace)]

| 候補 | 戦略                                                                    | 利点                                       |
|------+-------------------------------------------------------------------------+--------------------------------------------|
| A    | nursery 2 回 survive で tenured 昇格、Cheney semispace で copy (default) | SBCL gencgc precedent、generational 標準   |
| B    | 1 回 survive で即昇格                                                   | tenured 急増、long-lived young object で 過剰 |
| C    | adaptive (age threshold 動的調整)                                       | scope 過大、Phase 8+ で検討                |

*推奨 A 理由*:
- Cheney semispace (1970) は nursery copy GC の標準、minor GC の pause
  time を圧縮 (live object 数 << total nursery)
- 2 回 survive 閾値は SBCL gencgc default (=GENCGC_RAISE_AGE= = 2) を
  直接 transfer
- adaptive threshold は SBCL でも 25 年で marginal return が小さく
  pinned (Phase 7.5 で実測 tuning task 化)

Phase 7.3.3 (§3.3) で Cheney from-space / to-space の 2 領域を nursery
4MB の半分ずつ (2MB + 2MB) に分割。

** 2.6 write barrier [推奨 = A card-marking 4KB page granularity]

| 候補 | 戦略                                                  | 利点                                       |
|------+-------------------------------------------------------+--------------------------------------------|
| A    | card-marking 4KB page granularity (default)            | SBCL precedent、低 overhead                |
| B    | object-level write barrier                            | precision 高だが overhead 大 (~2-5x)        |
| C    | snapshot-at-the-beginning (SATB)                      | concurrent GC 必須、Phase 7.5+ で検討      |

*推奨 A 理由*:
- Doc 29 §2.3 page granularity (4KB + 16-page chunk) と完全整合
- card-marking は SBCL gencgc / V8 / OpenJDK で標準、cross-generation
  reference 検出に十分な precision
- write barrier overhead は Phase 7.1 native JIT で 1-2 命令 (card
  table store)、allocator hot path への影響 < 5%
- Phase 7.5+ で SATB / Brooks barrier 等 concurrent GC barrier に拡張可

card table layout (例示、最小限):
#+begin_src elisp
;; tenured 64MB / 4KB page = 16384 cards
;; 各 card 1 byte (dirty bit + age) で 16KB の card table
(defvar nelisp-gc--card-table)  ; vector of 16384 bytes
(defun nelisp-gc--card-mark (addr)
  (aset nelisp-gc--card-table
        (/ (- addr nelisp-gc--tenured-base) 4096)
        1))
#+end_src

* §6 Risks (8 項目)

** 6.1 write barrier overhead が allocator hot path で alloc-heavy bench を破壊

§2.6 推奨 A で card-marking 4KB page granularity、Phase 7.1 native JIT
で 1-2 命令 inline と想定。実装後の overhead 実測で alloc-heavy 5x
budget (Phase 7.2 §5.1) を切る risk。

*影響*: Phase 7.2 §5.1 5x budget 違反 = Doc 29 完遂 gate 後退、
allocator hot path 改修必須。

*mitigation*:
- Phase 7.3.4 着手時に Phase 7.2 担当 (session-X) と orchestrator
  consensus review 1 回 (Doc 28 §6.2 / Doc 29 §6.1 と同 pattern)
- write barrier inline は Phase 7.1 §2.7 =declare (inline t)= で
  必須化、bench monitoring 継続
- bench gate fail 時は scope 削減 (write barrier を =log-write= flag
  で disable 可能化、major GC 時 full scan で代替、user signoff 要)

** 6.2 promotion bug (live root mis-classification) で silent memory corruption

§3.3 Phase 7.3.3 promotion path で live root を mis-classify、Cheney
copy で live object を to-space に copy 漏れ → 後続 sweep で誤回収 =
silent memory corruption。post-mortem debug 困難 (=heap snapshot=
取らないと再現不可)。

*影響*: Phase 7.5 long-running soak で intermittent crash、原因特定
週単位。

*mitigation*:
- §5.2 promotion correctness gate を完遂 gate 必達条件 (live root
  mis-classification 0 件)
- ERT で 100K cycle table-driven test、cross-check は Phase 3c
  =nelisp-gc--reachable= walker (= host Emacs heap 全 walk) と本 phase
  mark phase の black set が等価条件
- Phase 7.3.3 着手時 =heap snapshot= API (debug 用、=gc-stats= 拡張)
  を実装、再現条件取得を可能化

** 6.3 safe-point pause spike (mark phase が長い、interactive UX 悪化)

§3.1 mark phase で work-list BFS が長い (例: 64MB tenured 全 walk)
場合、stop-the-world pause が 50ms 超 = interactive UX 悪化 risk。

*影響*: =./nelisp-runtime mcp serve= 経由 anvil tool 呼び出しで pause
spike、user 視点で応答遅延。

*mitigation*:
- §5.1 mark throughput bench で 50MB/sec 必達 (= 64MB scan 1.3sec)、
  major GC 頻度を tenured 80% utilization で限定すれば pause 1 cycle
  50ms 以下に収まる
- bench fail 時は incremental mark への拡張余地を Phase 7.5+ で
  reserve (§8.X で out of scope 宣言、本 phase は stop-the-world MVP)
- worst case 対応で =gc-budget-ms= defcustom 経由 user tunable

** 6.4 Doc 28 §6.10 GC metadata version drift (Phase 7.1 と co-update 必要)

§2.8 推奨 A で =GC_METADATA_VERSION= = 1 を consume、breaking change
時 Doc 28 と co-update。本 phase 実装中に v2 必要性が判明した場合、
Phase 7.1 backend (3 ISA) 全 retrofit 必須、Doc 29 region table format
も再検討。

*影響*: Phase 7.1 backend 全 retrofit + Doc 28 / Doc 29 / Doc 30 整合性
audit、Phase 7.3 SHIPPED 遅延。

*mitigation*:
- Phase 7.3.1 完了時点で Doc 28 担当 (session-X) と =metadata contract
  re-review= を 1 回必須 (orchestrator consensus、reviewer = claude +
  codex、judge = claude)
- v2 必要性が判明したら Doc 28 / Doc 29 / Doc 30 同時 update (=GC_METADATA_V2_MIGRATION_PATH=
  別 design doc 起こし)
- Phase 7.3.1 着手前に Doc 28 §2.9 contract format を再読込 + cross-check

** 6.5 fragmentation 蓄積 (Phase 7.5 で defragmentation 検討)

§3.2 sweep phase は per-pool free-list rebuild で coalescing するが、
long-running anvil daemon (24h 以上) で fragmentation 蓄積 =
allocator failure or OOM risk。Doc 29 §6.5 で同 risk が allocator
側として既に挙げられている。

*影響*: long-running anvil daemon で free-list utilization 低下、
=mmap= chunk 拡張頻度増、最終 OOM。

*mitigation*:
- §5.2 fragmentation-soak gate (10M alloc/free で 80%+ utilization 維持)
  を完遂 gate 必達条件
- mark-compact (full compaction) は Phase 7.5+ で defragmentation
  単独 phase として検討 (本 phase §8 で out of scope)
- =--gc-profile= で fragmentation rate を継続 monitor、threshold 超過時
  warning log

** 6.6 generational hypothesis miss (long-lived young object で promotion 過剰)

§2.5 推奨 A で nursery 2 回 survive で tenured 昇格、Cheney copy。
Phase 5-B selfhost workload で long-lived young object (例: anvil-state
session-scoped value) が多い場合、promotion 過剰 = tenured 急増 = major
GC 頻度増 = 全体 throughput 低下 risk。

*影響*: Phase 7.3 完遂 gate (50MB/sec mark / 50ms pause) は達成しても、
long-running anvil daemon の throughput が C 比 50% 切る。

*mitigation*:
- Phase 7.5 integration で =--gc-profile= 経由 promotion rate 採取、
  threshold tuning task 化
- adaptive threshold (§2.5 候補 C) は Phase 8+ で検討、SBCL 25 年で
  marginal return が小さく pinned した precedent あり
- =defcustom nelisp-gc-promotion-age= で user tunable (default 2、
  long-lived workload で 3-5 に上げる)

** 6.7 anvil leverage 過大評価 (Doc 28 §6.6 / Doc 29 §6.6 同 pattern)

Doc 27 で SBCL 25 年 → 1-2 ヶ月の見積。本 doc 7.3 で 6-10 週間に
re-baseline したが、なお下振れ risk あり。Doc 28 §6.6 / Doc 29 §6.6
の week 4/8/12 / week 2/3 replan gate 同 pattern で *3 replan gate* を
導入。

*影響*: Phase 7.4 (coding) / Phase 7.5 (integration) の着手遅延 =
Doc 27 全体 STATE 遅延 = NeLisp v1.0 完遂条件 #4 達成遅延 = 公開告知
保留延長。

** 6.8 "最難 phase" 宣言の reasoning (本 doc §0 主張の根拠)

Doc 27 §3 sub-phase table で "1-2 ヶ月、最難" と位置付けられた本
phase の "最難" 宣言の根拠を §0 で明示済、ここで詳述。

*難所 5 点*:
1. *write barrier overhead が allocator hot path に直接効く* (§6.1):
   Phase 7.2 への retrofit risk、bench gate fail で allocator design
   再検討要
2. *promotion bug = silent memory corruption* (§6.2): post-mortem
   debug 困難、ERT で 100K cycle gate でも漏れる可能性
3. *safe-point pause spike* (§6.3): Phase 7.1.4 で contract 固定済とは
   言え mark phase の実時間は本 phase 着手まで未測定
4. *Doc 28 §6.10 GC metadata version drift* (§6.4): Phase 7.1 backend
   retrofit risk、Phase 7.3 中の v2 必要性で Phase 7.1 全 ISA 改修
5. *tri-color invariant breach* (§3.1): stop-the-world でも write
   barrier bug で発生可能、white-to-black direct edge は静的検出困難

これらは Phase 7.0 (syscall stub) / Phase 7.1 (native compiler) /
Phase 7.2 (allocator) では発生しない (= Phase 7.3 固有の難所)。
Phase 7.4 (coding) は bit twisting で SBCL/Emacs precedent 直接 transfer
可能 = Phase 7.3 より易しい。

*mitigation 統合*:
- 全 5 難所に対し ERT gate (§5.1 / §5.2) + Phase 7.X 担当 session との
  orchestrator consensus review (§6.4) + replan gate 3 段階 (§6.7) で
  対応
- 本 phase の SHIPPED は Doc 27 全体 critical path の最難山場、SHIPPED
  以後は Phase 7.4 (coding) / Phase 7.5 (integration) の easier downhill

* §8 Out of scope

本 Phase 7.3 では以下を扱わない:

** 8.1 incremental GC / concurrent GC

§2.7 で stop-the-world MVP、Phase 7.5+ で incremental (mutator と並列
mark) / concurrent (GC thread 分離) を検討。本 phase は work-list 分割
+ tri-color invariant の構造を保ち、後付け増設の余地のみ確保。

** 8.2 mark-compact / defragmentation

§2.4 で per-pool free-list rebuild、mark-compact (full compaction) は
Phase 7.5+ で defragmentation 単独 phase として検討。fragmentation
蓄積は §6.5 で coalescing で対応、worst case で =--gc-compact= manual
trigger flag のみ提供 (実装は Phase 7.5+)。

** 8.3 multi-thread GC (per-thread arena + parallel mark)

§2.7 で single-thread MVP、Phase 7.5+ で Phase 4 actor との co-design
時に per-thread arena + parallel mark を検討。Doc 29 §8.2 と同様、
SBCL も 1999-2003 で single-thread MVP 開始、multi 化は段階的 transition。

** 8.4 weak reference / phantom reference

Phase 3c (Doc 09 §1.3) で host Emacs =make-weak-ref= /
=weak-hash-table= に委譲する方針、本 phase でも踏襲。NeLisp 自前 weak
reference は Phase 8+ で検討。

** 8.5 Compaction / re-organization (object layout 変更)

object header / 3-bit tag layout は Phase 7.2 §2.4 / §6.3 freeze、本
phase で変更しない (= ABI freeze 維持)。Cheney semispace の forwarding
pointer は header 内 1 bit 共用 (§6.7)。

** 8.6 Phase 7.4-7.5 の実装

本 doc は Phase 7.3 の design 専念。Phase 7.4 (coding) / Phase 7.5
(integration) は Doc 31-32 (DRAFT 予定) で詳述。

* §9 Open questions

** 9.1 mark phase の root set 拡張範囲

Phase 3c (Doc 09 §1.2) で確立した root set 6 種 (globals / functions /
macros / specpdl / bytecode VM stack / active JIT closures) に、本
phase で以下を追加:
- allocator region table (Doc 29 §2.6)
- safe-point bitmap (Doc 28 §2.9)
- actor mailbox (Phase 4)

具体 enumeration 順序 / weight (= mark 効率に影響) は本 phase 着手時
の orchestrator consensus review で確定。

*default*: Phase 3c 既存 6 種 + allocator + safe-point + actor の 9 種
を §3.1 ERT で table-driven 固定、Phase 7.5 integration で実測 tuning。

** 9.2 forwarding pointer の object header bit 配置

§6.7 で forwarding bit を age bit と共用 (forwarding 中は age 不要) と
方針提示したが、具体 bit 位置 (3-bit tag の上位 / object header
metadata word の特定 bit) は未確定。

*default*: object header metadata word の bit 31 (最上位) を forwarding
bit、bit 30 を 1 bit age (0/1) に縮小。Phase 7.3.3 着手時 user signoff
で確定。

** 9.3 card table の granularity 変更可能性

§2.6 推奨 A で 4KB page granularity (16384 cards / tenured 64MB)、
SBCL gencgc precedent。Phase 5-B selfhost workload で cross-generation
reference rate が低い場合、64KB granularity (= 16-page chunk、Doc 29
§2.3) で十分な可能性。

*default*: 4KB page granularity で初版実装、Phase 7.5 で実測 tuning。
=defcustom nelisp-gc-card-size= で user tunable 化検討。

** 9.4 long-running soak gate の duration

§5.2 で =long-running-soak= 24h 連続稼働を gate に設定したが、CI 環境
での 24h 稼働は実用的でない (GitHub Actions runner 制限)。

*default*: CI では 1h soak、weekly cron で 24h soak (self-hosted
runner、Phase 7.1.5 で整備済)。CI 1h soak が完遂 gate、24h soak は
post-SHIPPED audit task。

** 9.5 GC pause time の 50ms 設定根拠

§5.1 で =major-pause= 50ms 以下を target。SBCL gencgc は 20ms 程度、
Emacs alloc.c GC は 100ms 超のことあり。NeLisp は MVP で SBCL の 2.5x
許容。

*default*: 50ms gate を初版、Phase 7.5 integration で =--gc-profile=
経由実測。interactive UX 観点で 16ms (60fps frame) を将来 target、
本 phase scope 外。
