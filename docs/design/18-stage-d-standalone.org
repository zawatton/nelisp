現行 Stage A (NeLisp substrate) / B (AI-native Emacs) / C (AI/Emacs

**非目標:**
- Emacs editor の代替 (Remacs 型 scope-explosion 回避)
- iOS/Android native app (別 project scope)
- WebAssembly target (将来の option、本 doc では defer)

* §1 motivation

** 1.1 anvil.el の普及壁

anvil.el は現状 *Emacs user 限定* tool。Claude Code 経由 MCP
server は非 Emacs user にも価値があるが、導入には:
1. Emacs 28+ install
2. =~/.emacs.d/= 設定
3. =(require 'anvil)= 等 Elisp 起動設定
4. MCP server spawn config

の 4 段が必要。非 Emacs user (VSCode 中心 / JetBrains 中心 / 純
CLI 中心) にはこの barrier が事実上無理ゲー。Homebrew / apt 経由
1 行インストールできれば対象ユーザーが数桁拡大する。

** 1.2 現 Phase 0 v1.0 条件との関係

v1.0 条件 #4 "残 C = 4 核 (syscall / allocator / GC inner /
coding)" は *そのまま standalone 化の前提条件* になっている:
- 4 核が特定できれば → 4 核を Emacs から剥がして別 host に載せ替え可能
- = standalone runtime の最小 C base

つまり Stage D は v1.0 の自然な延長であり、scope creep ではなく
*配布 surface の追加*。

** 1.3 先例 — Lisp 系 standalone 配布手法

| project | 手法 | size | status |
|---------+------+------+--------|
| SBCL save-lisp-and-die | image dump + small runtime | ~30MB | 商用実績 |
| Clozure CL save-application | 同上 | ~15MB | 成熟 |
| Racket raco exe | image embed + runtime | ~20MB | 実用 |
| LispWorks deliver | image + GC tuning | ~10MB | 商用 |
| mruby | C lib として組込 | ~1MB | IoT/組込 定着 |
| Emacs pdump (27+) | portable dumper | ~40MB (Emacs自身) | 現 Emacs 起動技法 |
| PyInstaller | Python+lib bundle | ~30-100MB | Python 主流配布 |

→ "save image + small runtime = standalone binary" は Lisp 系で
確立した手法。NeLisp も同じ路線に乗せられる。

** 1.4 anvil.el の依存分析 (Emacs-bound な機能)

standalone 化に伴い、anvil.el 側で影響を受ける API を先行特定:

| 依存 | anvil 内使用例 | standalone 代替 |
|------+----------------+-----------------|
| =with-current-buffer= | 全 file ops | NeLisp string buffer で代替 |
| =find-file= / =save-buffer= | anvil-file-* | direct read/write |
| =make-process= sentinel | anvil-browser / worker | POSIX process + select |
| =run-with-idle-timer= | anvil-worker pool | event loop schedule |
| =file-notify-add-watch= | anvil-org-index | inotify / kqueue direct |
| =org-mode= primitives | anvil-org-index | org-parser NeLisp port (Doc 17 候補 2-A) |
| =transient= / TUI | user cmd | CLI flag 化 (headless profile) |
| =tabulated-list= | dashboard | CLI table printer |

*結論:* 7-8 割は NeLisp 側 Elisp re-impl で吸収可、UI 層は CLI
代替必要。anvil-manifest に 5 つ目 profile "headless" として
新設する (Doc 26 の延長)。

** §2.1 Primary approach 選択
- A. Emacs 埋め込み (bundle 型、1-1.5 年)
- B. Native-comp + 4 核 C runtime 自前 (本丸、3-5 年)
- C. 別 host 移植 (Rust/Zig runtime、community 必須)
- *D. Hybrid CLI launcher (現実解、6-12 ヶ月) ← RECOMMENDED as first milestone*

推奨理由: D は *今日の NeLisp + Emacs* で配布可能、sunk cost が
小さく、出せば非 Emacs user 反応が取れる。D 成功後 A、v1.0 到達後 B。

** §2.2 段階シーケンス
- A. D のみ (打ち切り可能性あり)
- *B. D (MVP) → A (成熟) → B (究極) 段階計画 ← RECOMMENDED*
- C. B 直行 (D/A skip)

推奨理由: 各段で user 反応を見て次段を判断できる。B 直行は
community / 工数的に非現実。

** §2.3 Distribution target platform (MVP)
- A. Linux のみ
- *B. Linux + macOS ← RECOMMENDED*
- C. Linux + macOS + Windows (Windows は Doc 27 anvil-pty-broker 経由の既存資産あり)
- D. Linux + Windows (macOS skip)

推奨理由: non-Emacs user の多い VSCode / JetBrains 層は
Linux/macOS 比率が高い。Windows は既存 anvil-pty-broker 資産で
Stage D Phase 6.3 以降に追加可能。

** §2.4 Binary size target (Option D MVP)
- A. <5MB launcher のみ (Emacs は別途 user install)
- *B. <10MB launcher + minimal Emacs assets bundle ← RECOMMENDED*
- C. <50MB launcher + full Emacs bundle
- D. <100MB 全部入り (full Emacs + 全 anvil module)

推奨理由: B は初回起動で小さく、重い module は on-demand load。
C/D は Electron 的肥大化で user 体験劣化。

** §2.5 Emacs dependency in Option D
- A. Emacs を binary に bundle (初回起動快適、size 大)
- *B. Emacs は OS package manager 経由 user install、launcher が検出 ← RECOMMENDED*
- C. Emacs 見つからなければ自動 download (一度だけ、cache)
- D. C + B hybrid

推奨理由: B は GPL 伝播問題を回避、user の既存 Emacs install も
再利用できる (apt/brew 経由で Emacs install するのは非 Emacs user
でも 1 行で済む)。B で体験が悪ければ C に昇格。

** §2.6 License
- A. GPLv3+ (Emacs 互換、埋め込む限り必須)
- *B. GPLv3+ (strict、Emacs 依存を考慮) ← RECOMMENDED*
- C. 二重 license (ambiguous、non-Emacs user 用の明示)
- D. Apache 2.0 / MIT (non-Emacs user 親和性重視、Emacs 埋め込み不可)

推奨理由: Option A (Emacs 埋め込み) を将来 option に残す以上、
GPLv3+ 以外選べない。Option D 現行 (Hybrid CLI) 段階でも NeLisp 自体
GPLv3+ で既に公開予定、license は変える理由なし。

** §2.7 Distribution channel (MVP)
- *A. GitHub Release (tar.gz + install.sh) ← RECOMMENDED first*
- B. Homebrew tap
- C. apt/dnf repo
- D. curl|bash installer (anvil.sh)

推奨理由: A が最低工数で全 platform 同時配布可能。D は A の

推奨理由: A で MVP、Claude Code の .mcp.json に 1 行書けば終わる。

* §5 先例 audit

** 5.1 mruby — 最も近い模範

mruby = Matz が C に組込みやすい形で Ruby VM を再実装した project。
- VM size ~1MB、C static lib として linkable
- Ruby との互換性は core subset (ISO Ruby 準拠)
- 用途: IoT / ゲームエンジン / 組込アプリ

NeLisp Stage D 類推:
- anvil launcher が C/Go/Rust で書かれ、NeLisp VM を static lib
  として link する (将来 Option B)
- 現 Phase 6 MVP (Option D) は launcher から Emacs subprocess
  spawn で始め、VM 同梱は Phase 7 相当に後倒し

** 5.2 SBCL save-lisp-and-die

SBCL core image は実行時 =*.core= として dump、=sbcl= runtime
binary が load して起動する。
- app 配布手法として商用実績 (ITA Software 等)
- ~30MB binary + ~20MB core image
- GC / thread / FFI 全部 lib 側に持つ

NeLisp Stage D 類推:
- Phase 8 相当で =nelisp save-image= / =nelisp-runtime= 分離
- まず Option D は save-image 無しで MVP 出す

** 5.3 Racket raco exe

Racket = Scheme 系 IDE/言語 runtime、=raco exe= で standalone
exe 化。
- image embed + runtime、~20MB
- user は =./myapp= 1 個で配布可能
- 実用実績 (DrRacket 自身)

NeLisp Stage D 類推:
- Phase 9 相当で =anvil exe= / =nelisp build= cmd 追加
- 現 Phase 6 では scope 外

** 5.4 Emacs pdump (27+)

Emacs 自身が pdumper で起動する (=/usr/bin/emacs= は実は
pdump 復元)。
- portable dumper、~40MB
- 起動時間 <0.1s (cold)
- Emacs 27 以降 default

NeLisp Stage D 類推:
- Option A (Emacs 埋め込み) が実際には「Emacs pdump + NeLisp
  loader を同梱」することになる
- 既に確立した手法なので技術的障壁は低い

** 5.5 PyInstaller

Python app 標準配布。Python interpreter + site-packages +
user script を 1 binary に bundle。
- ~30-100MB、起動 ~0.5-2s
- venv 的 install 代替として普及
- 非 Python user が Python app 触る時の entry point

NeLisp Stage D 類推:
- Option A (Emacs 埋め込み) の UX 参考
- Option D MVP はここまで大がかりでない launcher で開始

* §9 想定 FAQ (RFC / Reddit 告知準備)

** Q1. なぜ Emacs を捨てない?
A. Emacs の Elisp ecosystem (org-mode / magit / Lisp 開発環境) は
捨てる気がない。Stage D は *Emacs を知らない人にも anvil の価値を
届ける* ための追加配布 surface であって、Emacs-first 路線自体は
Stage A 本線で継続。

** Q2. SBCL みたいに NeLisp を実行可能 Lisp にする選択肢は?
A. それが Option B (native + 4 核自前) の本丸。ただし 3-5 年かかる
ので、Option D で MVP を先に出して adoption を測る。

** Q3. Electron / Tauri みたいな frontend は?
A. Stage D は *Claude Code 等の MCP client 経由* の CLI 層
(headless) が MVP scope。GUI は Stage B/C の anvil.el successor で
別途検討。

** Q4. NeLisp が standalone 化されたら anvil 以外の Elisp app も?
A. Yes。Phase 6.0 audit で確立する "Emacs 依存 API 代替層" は
anvil.el 特化でなく汎用に使える。magit 等も将来 headless 化可能。
ただし現 Stage D scope は anvil.el のみに絞る (dogfood 対象)。
