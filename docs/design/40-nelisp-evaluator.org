** 2.1 reader scope: =read= / =read-from-string= 完全 Elisp 互換 [推奨 = A 完全互換]

A. *完全互換 (推奨)* — =#@N= byte-skip / =#,FORM= eval-at-read /
   =##= empty-symbol / =#s(...)= record / =#x= hex / =#o= octal /
   =#b= binary / =?\C-a= control char / =?\M-x= meta char / =あ=
   unicode escape を *全件 round-trip*、Emacs Lisp file を unchanged で読める

B. subset MVP — 上記から =#@N= / =#,FORM= の 2 件を out-of-scope (=
   anvil-* corpus で実使用 0 件と確認済の前提、後続 Phase で補完)

C. 既存 Phase 3a level 維持 — 本 Doc 40 では reader は触らない (=
   Phase 3a の不足 syntax は補完しない、anvil-* corpus を pre-process
   で = =#@= 行を strip = で逃げる)

LOCKED 推奨 = A、根拠: anvil-* docstring に =#@= byte-skip が頻出 (=
docstring の position cache、Emacs FNS)、A 以外は anvil 全 module
load 不可能 = M3 milestone 達成不能。

** 2.2 evaluator core: tree-walking interpreter [推奨 = A、Phase 7.1 と階層]

A. *tree-walking interpreter (推奨)* — Phase 3 SHIPPED の
   =nelisp-eval-form= 系列を *拡張・分離* (= special-form を別ファイル
   =src/nelisp-special-forms.el= に切り出し)、Phase 7.1 native compile
   が通る hot path はそのまま VM / native、interpret path は fallback /
   debug / cold-init で活用

B. CPS transform + trampoline — tail-call MVP の代替案、Phase 7.1.5
   既 SHIPPED の TCO と機能重複、本 doc では実装しない

C. 完全 Phase 7.1 native compile 強制 — interpret path 廃止、anvil-*
   module も全件 native compile 経由のみ → cold-init time 爆発 (= 4-stage
   bootstrap 経由で Phase 7.1.5 stage1 が anvil-* 全 module 通過 必要)

LOCKED 推奨 = A、根拠: Phase 3 SHIPPED 資産再利用 + Phase 7.1 と
peaceful coexist (§6.6)、cold-init cost 抑制。

** 2.3 lexical / dynamic binding policy [推奨 = A、Emacs Lisp 既定踏襲]

A. *Emacs Lisp 既定 (推奨)* — file-level =;; -*- lexical-binding: t;
   -*-= file variable で lexical / dynamic 切替、=defvar= / =defconst=
   宣言された symbol は dynamic、それ以外 lexical (Phase 3 SHIPPED
   挙動と同)

B. lexical-only — file variable 無視、全 symbol lexical = legacy
   Elisp 互換崩れ (anvil-* も一部 dynamic 前提あり、grep で audit 必要)

C. dynamic-only — Common Lisp / SBCL 互換寄り、Emacs Lisp 通常 pkg
   ロードで escaped binding bug 多発、scope 外

LOCKED 推奨 = A、根拠: anvil-* 全 module は =;; -*- lexical-binding:
t; -*-= header 統一 (= anvil dev convention、=memory:
project_anvil_package=)、A 以外は corpus level で互換崩れ。

** 2.4 macro expansion timing [推奨 = A、eval-time + memo cache]

A. *eval-time + memo cache (推奨)* — =defmacro= で macro 登録、
   =eval-form= 中の =cons head= ↔ =nelisp--macros= lookup hit で
   =macroexpand-1= → expanded form を再 eval、expand 結果を form
   key に hash memoize (= 同 form 二度目以降は cache hit)

B. read-time eager expansion — =read-from-string= 直後に expand 全
   完了、eval-time は walker 不要 → 但し =defmacro= 自体が読まれる
   前に macro call が現れると fail = anvil-server.el 順序依存

C. eval-time pure (no memo) — A から memo 削除、CPU cost 増 (=
   anvil-server tool dispatch が macro 展開重ね掛けで 2-3x slowdown)

LOCKED 推奨 = A、根拠: B は anvil-orchestrator のような top-level
=defmacro= 後置 module で詰まる、C は perf 不足 (Phase 7.1 native
compile path が無い場合)。

** 2.5 closure capture model [推奨 = A、現 Phase 3 model 維持]

** 6.1 lexical binding closure capture correctness

Phase 3 SHIPPED の =nelisp-closure ENV PARAMS BODY= alist env が
over-captured (= 同 sym の lexical → dynamic 昇格、=defvar= 後置)

全 corpus を =defvar= 後置 grep audit、bug 候補があれば fix scope を
phase 7+C 内に組み込む。

mitigation:
- ERT pin 4 形式 (self-rec / mutual-rec / captured-mutate /
  over-captured) を必須化
- audit script = =scripts/audit-defvar-postpose.sh= 新規 (= 1 file
  内で =defvar SYM= が =setq SYM= や =SYM= reference より後置の箇所
  検出)
- corpus level の interpret path round-trip = M3 smoke で検出可能

** 6.2 dynamic binding fallback の Emacs 互換境界

Emacs Lisp の dynamic binding は file-level =;; -*- lexical-binding:
nil; -*-= で activate、anvil-* は全 lexical-binding: t、ただし
=cl-loop= 等 macro が内部で dynamic 風 idiom を使う場合あり。Phase
3 SHIPPED で動いている事実が baseline、本 doc では新規破壊しない。

mitigation:
- file-level lexical-binding header を Phase 7+A reader が確実に
  parse + propagate
- =defvar= / =defconst= 後の dynamic 昇格は Phase 3 既挙動 (=
  =nelisp--specials= 経由) を維持
- ERT regression suite で全 Phase 3 既存 ERT 緑保持

** 6.3 tail-call optimization (Phase 7.1.5 既 SHIPPED と整合)

interpret path での TCO は scope 外 (= Phase 7.1.5 native path で
SHIPPED 済、Doc 28 v2 §11.5 errata 反映)。interpret path では deep
recursion で stack 食う = stack overflow risk。

mitigation:
- =max-lisp-eval-depth= / =max-specpdl-size= の Emacs 互換
  =defcustom= を提供、user 側で増設可
- anvil-* 内 deep recursion 用法は Phase 7.1 native compile
  routing で実用上問題なし (= interpret path は cold-init / debug
  限定)
- 完全 TCO 求める user は Phase 7.1.5 native compile 経由を推奨

** 6.4 macro hygiene (Emacs Lisp pseudo-hygienic 互換)

完全 hygienic macro は scope 外 (§1.6)、Emacs Lisp 流 =cl-gensym=
ベースの自前 hygiene を user 側責任で実装してもらう方針。

  前必達)

  使い hygiene 確保 (= NeLisp 実装記述時の dev convention) — *T114
  C3 反映 v1: =src/nelisp-macro.el= 278 LOC を audit、core macros
  各々が =cl-gensym= 使用 hygiene 確保しているか source review、不足

  binding 視覚化 (Phase 7+D)
