* §0 TL;DR

** §0.1 Sexp DSL とは何か (= 1-line)

elisp が =build-tool/src/eval/sexp.rs= の =Sexp= enum と *byte-
identical な memory layout を持つ値を生成し serialize できる* DSL =
=nelisp-sexp-dsl.el=.  「elisp の中で Rust の =Sexp= を作る」の elisp
版。

** §0.2 なぜ elisp で実装できるのか

=Sexp= は **pure data** = foreign behavior を持たない (Rust 側で =impl=
される method = =fmt_sexp=, =cons_car= 等は data に対する pure
function、Rust runtime invariant に依存しない)。layout は =#[repr(C,
u8)]= で fixed = elisp 側で同 byte pattern を組めば Rust 側 =*const
Sexp= deref と区別不能。pointer 経由の =Arc<NlConsBox>= 等の
internal handle も「pointer = u64 = 8 byte」が canonical bit pattern
なので elisp の =make-bool-vector / make-string / make-vector= で
byte-buffer を組めば代替可能。

** §0.3 なぜ Phase 47 が本 DSL を要するか

JIT trampoline (= =build-tool/src/jit/=) は現状以下のように呼び出される:

#+begin_src text
caller (Rust)         JIT trampoline (asm)        callee (Rust)
    |                          |                          |
    |---- *const Sexp -------->|                          |
    |                          |---- *const Sexp -------->|
    |                          |                          |
    |<---- *const Sexp --------|<---- *const Sexp --------|
#+end_src

すべての Sexp 値が「Rust 側で alloc された box」前提。Phase 47 で
elisp が trampoline を「emit する側」になる時、elisp が Sexp 値を
alloc する経路がないと caller arm が組めない。本 doc の DSL がその
経路を提供する。

** §2.2 internal repr 選択

elisp 側で Sexp を表す choice 3 候補:

| repr            | pros                                  | cons                               |
|-----------------+---------------------------------------+------------------------------------|
| (a) elisp value | 簡潔 (= int → integer, str → string)  | tag mapping を都度推測必要、box variant repr 不在 |
| (b) tagged vec  | =[TAG PAYLOAD]= explicit              | constructor/accessor が wrap/unwrap |
| (c) raw bytes   | serialize 不要 = byte-buffer 直で持つ | accessor 重い (= byte parse 毎回)  |

採用: *(b) tagged vec* — DSL は internal で =[TAG PAYLOAD]= を持ち、
=nelisp-sexp-write-bytes= の時のみ raw bytes に変換.  これは Doc 47
seed image 路線と整合 (= elisp 側は high-level、serialize は専用 fn).

box variant の場 payload は **=(:box-ptr POINTER-AS-INT :box-kind
KIND :inner INNER-VEC)= の plist** とする.  =POINTER-AS-INT= は ELF
emit 時の relocation target で、生成段階では placeholder (= 0)、linker
が patch (Doc 93).  =:inner= は box 中身を elisp 値として hold (= box
allocation を Rust に push する場合の hint、Phase 47 で実体 box も
elisp 化する path への forward-compat).

** §2.3 constructors 一覧 (= 13 variant ctor)

各 ctor は ~5 LOC = ~70 LOC subtotal.  簡素 variant (= Nil/T/Int/

** §3.1 enum 全体 layout (Sexp = 32 byte)

Rust =#[repr(C, u8)]=:

注:
- *all* variants padded to 32 byte total (= =size_of::<Sexp>()=).  Rust

  本 DSL は **=Symbol/Str= variant の serialize は heap pointer ではなく
  別途 raw bytes を rodata section に emit する** path を採る (= §3.5).

** §3.3 box variant の pointer 値 (= Phase 47 relocation target)

box variant 7 個 (MutStr/Cons/Vector/CharTable/BoolVector/Cell/Record)
は payload @ offset 8 が =NonNull<NlXxx>= = u64 pointer.  この 8-byte
slot が Phase 47 で linker (Doc 93) が patch する target:

#+begin_src text
generation time              link time              run time
[elisp emit]                 [linker patches]       [JIT trampoline reads]
| Sexp::Cons {              | (resolves           | mov rsi, [rdi+8]
|   tag:  7,                |  symbol             | call NlConsBoxRef::car
|   _pad: 0..0,             |  "_box_X" to        |
|   payload: 0xDEADBEEF }   |  actual address)    |
#+end_src

elisp DSL は =:box-ptr 0= の placeholder を埋め、Doc 93 linker が ELF
=R_X86_64_64= relocation で patch.  alignment 制約 = 8-byte aligned
target / 8-byte aligned source = automatic (= payload offset 8 は 8-
byte aligned by §3.1 layout).

** §3.4 =Symbol= / =Str= の文字列 payload

Rust =String= layout は **rustc 内部実装依存** = elisp 側で同 byte
pattern を emit するのは not safe.  解決 choice: (a) =NonNull<u8>=
(rodata pointer) + =u64 len= の 16 byte 新 layout に migrate、(b)
MutStr と同じく =NonNull<NlStr>= refcounted box 統一案 (= layout
breaking change).  本 doc 暫定 assumption: *(b) box variant 統一* で
全 13 variant payload を 16 byte に揃える.  Phase 47 master spec 側で
spec freeze 時 final 選択.

** §3.5 sanity table (= round-trip property per variant)

| variant       | Rust size  | round-trip property                                |
|---------------+------------+-----------------------------------------------------|
| Nil/T         | 32 byte    | =deser . ser . make = make=                        |
| Int(i64)      | 32         | bit-identical for any i64                          |
| Float(f64)    | 32         | bit-identical for finite + NaN preservation        |
| Symbol/Str    | 16 or 32 * | structural — name/text string equal                |
| MutStr        | 16         | inner string equal (= structural; pointer differs) |
| Cons/Vector/Record/CharTable | 16 | recursive structural                       |
| BoolVector    | 16         | bit-vector equality                                |
| Cell          | 16         | inner value equal                                  |

(* §3.4 layout choice 依存.  ctor / serializer 名は §3.2 表参照.)

** §3.6 multi-arch 注釈

x86_64 LE assumption.  aarch64 も LE (Linux+macOS は default LE)
なので =Int= / =Float= / pointer の byte order は同じ.  big-endian
(= s390x, ppc64be) は scope 外 — Phase 47 master が x86_64 + aarch64
LE only と articulate 済 (Doc 85 §7.1 multi-arch contingency).

* §4 mapping to JIT trampoline ABI

** §4.1 現状 (Rust → Rust)

JIT trampoline (= Doc 81 articulated) は現状:

#+begin_src text
caller (Rust) builds Sexp::Int(42)              → *const Sexp
   passes via System V AMD64 ABI rdi register
trampoline reads tag byte at [rdi+0], dispatches
   reads payload at [rdi+8] (= 8-byte slot)
callee (Rust) receives Sexp                     ← *const Sexp
#+end_src

trampoline は Sexp の **layout** (tag at 0, payload at 8) のみ
信頼、ownership / lifetime は Rust 側に任せる (= raw pointer).

** §4.2 Phase 47 (elisp emits, Rust JIT trampoline reads)

elisp DSL の生成物が JIT trampoline に流れる経路:

#+begin_src text
elisp builds                       byte buffer                      JIT trampoline
nelisp-sexp-make-int 42            (32 byte = SEXP_TAG_INT          mov al, [rdi+0]   ;; al = 2 (= INT)
   |                               + 7 byte pad + i64 LE 42)        cmp al, 2
   v                                                                jne .other
nelisp-sexp-write-bytes              ↓ inserted in .data section    mov rax, [rdi+8]  ;; rax = 42
   into a (make-string 32 ?\0)        of ELF binary by Doc 91         ret
   buffer                              writer (relocation _box_42)
   |
   v
the buffer's address               =*const Sexp= layout-identical
   (= Doc 93 linker patches)       to Rust-built Sexp::Int(42)
#+end_src

** §4.3 elisp / Rust boundary 越え不要

key insight: elisp は **byte buffer を作るだけ** = Rust の Sexp 値を
直接 alloc する経路は不要.  byte buffer = =(make-string 32 ?\0)=
or =(make-bool-vector 256 nil)= は elisp の primitive、その raw bytes
を ELF =.data= section に emit すれば Rust JIT trampoline が同 byte
を Sexp として deref できる.

これは Doc 47 seed image と全く同じ pattern (= elisp が NELIMG v3
heap snapshot を bake、Rust runtime が mmap して直 deref).

** §4.4 box pointer の relocation (= Doc 93 dependency)

box variant の payload =NonNull<NlConsBox>= は generation 時には
*どこに alloc されるか不明*.  Doc 93 static linker が:

1. elisp が生成した byte buffer に =SEXP_TAG_CONS= + =0xDEADBEEF=
   placeholder pointer を埋めた状態
2. linker が =.data= section の box 実体 alloc 位置を確定
3. relocation entry =R_X86_64_64= で payload @ offset 8 を実 address に
   patch

この workflow は GCC / clang static link と完全同じ = standard ELF
relocation で済み新 mechanism 不要.

** §4.5 hot path / cold path

| path        | byte operations                                        | speed                           |
|-------------+--------------------------------------------------------+---------------------------------|
| cold path   | elisp DSL ctor → tagged vec → write-bytes → buffer    | elisp byte manipulation = slow  |
| hot path    | JIT trampoline reads buffer's raw bytes via =mov=     | as fast as native Rust deref   |

「cold path = elisp 経由」が Phase 47 perf concern だが、これは bake
time にのみ走り、run time の hot path は trampoline から見える raw
bytes = native speed.

** §4.6 Sexp lifecycle through elisp DSL

#+begin_src text
[bake time (= Doc 91-95 chain)]
  elisp source
    → nelisp-sexp-make-* (= DSL ctor)
    → byte buffer (= 32-byte aligned)
    → ELF .data section (= Doc 91 writer)
    → relocation entries (= Doc 93 linker)

[boot time]
  ./nelisp loads bake image
  .data section mmap'd, pointers resolved by static link
  Sexp values now live in process memory at fixed addresses

[run time (= JIT executes)]
  trampoline gets =rdi= = pointer into .data
  reads tag/payload as raw bytes (= no marshalling)
#+end_src

* §7 Risk / unknowns

** §7.1 elisp byte manipulation の性能

elisp の =(aset (make-string 32 ?\0) 0 ...)= は cold path で speed
issue:

- 32-byte buffer × N variant × M Sexp value at bake time
- =STDLIB_FILES= の Sexp 値数は ~10,000 オーダー (= 推定、stdlib AST
  全体)
- 1 Sexp ~ 1 ms (= elisp byte access)、合計 ~10s = bake time +10s

緩和策:
- elisp byte primitive を C-implemented =unibyte-string= 経由で
  optimize
- =cl-loop= unroll
- 並列化は elisp single-thread のため不可

判断: bake time +10s は許容、Phase 47 acceptance に影響しない.  hot
path (= JIT) は影響なし (§4.5).

** §7.2 Rust =Sexp= layout drift

risk: =rustc= 版で =#[repr(C, u8)]= の挙動が変わる:

- =#[repr(C, u8)]= 自体は **stable since rustc 1.66 (2022)** = 当面
  stable
- =String= 内部 layout は **NOT stable** = §3.4 で box 化統一案を
  採用すれば回避
- =NonNull<T>= layout は =#[repr(transparent)]= = stable

緩和策:
- CI に =cargo expand= で =#[repr(C, u8)]= の expand 結果を snapshot
  test、変化検出
- =sexp_layout_alignment_and_size_sane= test を release gate
- =MSRV (minimum supported rustc version)= を articulate

判断: 緩和策あれば low risk.

** §7.3 bootstrap chicken-and-egg

問題: 本 DSL は Phase 47 で稼働するが、**本 DSL を生成する elisp
code 自体** は既存 Rust nelisp で実行される.

= 既存 Rust nelisp が elisp code を実行 → 本 DSL (= elisp 値 of
Sexp) を bake → output bytes を Doc 91 writer に渡す → ELF binary 生成
→ 新 elisp-only nelisp が boot.

Phase 47 master spec 側 (= Doc 85 §7.4 articulated と整合) で
chicken-and-egg は handled = 既存 Rust nelisp は **archaeology** とし
て残し、新 binary を生成するための bootstrap stage 0 として永久 pin.

判断: bootstrap chain 健全、本 DSL は他 component と同じ条件.

** §7.4 Symbol/Str layout choice (§3.4 不確定)

current Rust = =Symbol(String)= / =Str(String)= は =String= 内部 24 byte
payload + tag + pad = 32 byte.  =String= 内部 layout が不安定.

choice:
1. *box 化統一* (= NlSymbolRef / NlStrRef) = 16 byte payload に縮小、
   layout-breaking
2. *raw rodata pointer + length* = =(u64 ptr, u64 len)= 16 byte
3. *current 維持* = String layout を assume、rustc 版 pin

本 doc 暫定 = (1).  Phase 47 master (= Doc 85 update) で確定要.

判断: medium risk.  Phase 47 master 側で choice すれば本 DSL は
追従するだけ.

** §7.5 box 実体 alloc の locus

§1.4 で articulated = 本 DSL は box の "pointer" を emit するに
留まり、box 実体は Rust 側に残置.  Phase 47 完遂後も =NlConsBox= /
=NlVector= 等の box impl Rust = ~2,880 LOC が残る.

= "Phase 47 完遂 = 0 Rust" の宣伝は **enum Sexp の 305 LOC のみ
削減**、box 実体は別 phase で elisp 化要 (= future Phase 48? 別 doc).

判断: Phase 47 scope は本 doc 範囲で正しく、Doc 85 §0 TL;DR table の
"300 Rust 削減 / Sexp DSL row" は本 doc で正確に articulated.  box
実体 elisp 化は scope 外と明示する.

** §7.6 elisp byte serializer の big-int 制約

=Sexp::Int(i64)= は 64-bit 整数.  elisp の integer は **fixnum (=
typically 62-bit on 64-bit platform)** で full i64 範囲を持たない.

緩和策:
- 64-bit 値は 2 つの 32-bit half-word (= =high * 2^32 + low=) で encode
- elisp の =bignum= (= Emacs 27+ で fboundp) を assume すれば direct
- =(format "%x"...)= → bytes へ char-encode の workaround

判断: low risk.  Emacs 27+ には bignum あり、subset target = nelisp
の elisp interpreter は full bignum support 前提 (= Phase 7.4 NeLisp
coding).

** §7.7 char-table の =Vec<(i64, Sexp)>= recurse 表現

=CharTableInner.entries: Vec<(i64, Sexp)>= は Sexp を再帰 hold する.
elisp DSL で表現する場合 entry list の serializer は recurse 必須.

→ =-write-char-table= 実装が他 variant よりやや複雑 (~25 LOC
vs typical 10 LOC).  §2.6 LOC 計算済.

判断: low risk.

** §7.8 multi-arch (aarch64) layout

Doc 85 §7.1 multi-arch contingency = "Linux x86_64 のみで Phase 47
完遂、他 platform は後続".

aarch64 への extend:
- byte order = LE (= 同 x86_64)
- align_of(Sexp) = 8 (= 同)
- =NonNull<T>= layout = 同
- 違い: 関数 calling convention (= AAPCS64 vs SysV AMD64)、Sexp
  layout 自体は arch-independent

= 本 DSL は arch-independent.  aarch64 対応は trampoline ABI (Doc 81
/ Doc 92) 側の concern.

判断: 本 doc 影響なし.
