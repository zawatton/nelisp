Doc 53〜60 で積み上げた 40 の specialized =bi_syscall_*= primitive
(= sockaddr_in / sockaddr_un / sockaddr_in6 / pollfd / inotify_event /
signalfd_siginfo / msghdr+SCM_RIGHTS / itimerspec / sigset_t / ucred 等

+ 新規 buffer helper 群 (= =nl-ffi-write-bytes= 等) で組み立てる方式

|---------------------------+---------------------------------------------------+--------------------------------------|
| Rust =bi_syscall_*= 専用     | 40 (Linux active + 40 non-Linux stub macro 化)     | 0 (生成 / dispatcher 共に削除)       |

LOC 削減推定 (= Rust):
- 40 specialized primitive 本体 (Linux 実装): ~-30 × 40 = -1,200
- 同 non-Linux stub (= 1-line macro 呼び 40 件): -40 (= macro 呼び自体の line)

** §3.2 nl-ffi-call 拡張 (= 必須 prerequisite Phase 0)

現 =nl-ffi-call= (=ffi.rs= 359 LOC) は call / malloc / read-bytes / free
の 4 primitive のみ. Doc 76 では以下 2 primitive を追加:

*** §3.2.1 =nl-ffi-write-bytes PTR STR=

PTR (= integer raw pointer) に STR (= elisp string、UTF-8 / binary)
の bytes を書き込む。戻り値 t。これで struct field を offset 単位で
poke できるようになり、=sockaddr_in= 等を elisp 側で組み立て可能。

*** §3.2.2 =nl-ffi-errno=

直近 syscall の errno を返す。glibc =__errno_location()= / Darwin
=__error()= / Windows GetLastError の OS 別 thin shim. =nl-ffi-call=
と同 thread / 同 call-site で使う前提。

(libc crate が cross-OS で同名 cfg expose、cross-OS 1 関数で済む。)

** §3.3 struct marshaling 設計

(具体的な =sockaddr_in= / =sockaddr_un= / =sockaddr_in6= / =pollfd[]= /
=inotify_event[]= / =signalfd_siginfo[]= / =itimerspec= / =sigset_t= /
=msghdr+SCM_RIGHTS cmsg= の encode/decode 設計は Stage 0 着手時に確定。
本 Doc は scope articulation のみ。各 struct の OS 別 layout const は
=lisp/nelisp-stdlib-ffi-marshal.el= で管理、Linux x86_64 / aarch64 /
Darwin arm64 / Windows x64 の差は per-OS defconst で吸収。)

** §3.4 errno propagation 設計

specialized =bi_*= の =-errno= 規約は単純で elisp 側 =nelisp-os--check-errno=
が `(< r 0)` で signal する. nl-ffi-call libc 経由では:

1. libc 関数の戻り値は POSIX 規約 (= 失敗時 -1)
2. =nl-ffi-errno= で errno を取得
3. elisp wrapper が判定:

#+BEGIN_SRC emacs-lisp
(let ((r (nl-ffi-call "libc" "open" [:sint32 :string :sint32 :sint32]
                      path flags mode)))
  (if (= r -1)
      (signal 'nelisp-os-error (list (nl-ffi-errno)))
    r))
#+END_SRC

注: =nl-ffi-errno= は thread-local だが elisp 側 single-thread eval
前提なので race は無し. ただし GC が間に挟まると errno が上書きされ
得る (= libffi trampoline 内で再度 syscall 走る可能性) — §5 で gate
test を書く.

| poll loop 10k      | 1 fd POLLIN poll(timeout=0) × 10,000  | × 1.3〜2.0                        |
| sendmsg-recvmsg roundtrip 1k | 1 fd 1 byte payload × 1,000 | × 1.4〜2.2 (cmsg encode 重い)    |

許容上限: 全 bench で × 3.0 以内. これを超えた wrapper は §6 future
work で "JIT lower で specialized 再導入" を検討 (= Cranelift で
=mov rax,NR; syscall= 直 emit、Doc 62 Stage 5.1 の延長).

** §5.5 errno propagation 検証

各 stage で 3 種の errno path を確認:

1. open(/nonexistent) → ENOENT (2)
2. close(99999) → EBADF (9)
3. socket() in container 制限 → EAFNOSUPPORT 等

旧 (=-errno=規約) と新 (=nl-ffi-errno=規約) で同一 errno が elisp 側
=nelisp-os-error= に届くこと.

* §6 Out of scope (= 別 Phase / 別 Doc)

** §6.1 現 Doc では扱わないもの

- *generic =nelisp--syscall=*: SYS_NR ベースの 6-arg generic は残す。
  Doc 62 Stage 5.1 で JIT lower 済 (= mov rax,NR; syscall) で hot path 維持.
- *=nelisp--syscall-canonicalize= / =-stat= / =-readdir= / =-read-file=*:
  Doc 49 で elisp 化済の上位 API (= =file-exists-p= / =directory-files= /
  =load=) が依存している thin primitive、性能 critical. nl-ffi-call で
  realpath/stat/getdents64/loop-read を組み立てるのは可能だが
  hot-path で × 1.5〜3.0 latency penalty が effective、移行は別 Doc.
- *io_uring*: Doc 60 §4 で deferred、別 Doc 候補.
- *kqueue (Darwin) / IOCP (Windows) / FSEvents*: Linux signalfd /
  timerfd / inotify と同等の event surface だが API 全く別。Doc 76 で
  Linux 限定 surface (= signalfd 等) は migration 後も Linux 専用、
  Darwin/Windows は別途 wrapper を別 Doc で追加する.
- *Cranelift JIT で specialized 再導入*: bench で × 3.0 超えた wrapper
  があれば、=mov rax,NR; syscall= 直 emit する SyscallIR を拡張する形で
  再導入候補. 別 Doc.

* §7 Risks

** §7.1 ABI drift

OS / glibc バージョン違いで struct layout が変わる可能性. 緩和策:

- =sizeof= / =offsetof= は Rust =std::mem::size_of=/ =offset_of= の
  既知値を Stage 0 で =build.rs= から JSON 生成 → elisp 側 const に取り込む
- 上記 const を =nelisp-os-ffi--layout.el= (auto-generated) に rev-pin
- Doc 60 / Doc 59 の既存 specialized 実装を **削除前に layout 比較 ERT**
  を書いて Linux/Darwin で gate.

** §7.2 errno race

GC が間に挟まると errno が上書きされ得る. 緩和:

- =nl-ffi-call= 直後に **第 2 alloc を起こさない** wrapper pattern
  (= 戻り値が < 0 の場合だけ =nl-ffi-errno= を呼ぶ、成功時は呼ばない)
- bench で sense check (= 1k 回の errno consistency)

** §7.3 性能 regression

§5.4 bench で許容上限 × 3.0 を超えるものが出た場合の対応:

1. 該当 wrapper を =Doc 76.x.A= として specialized 再 specialize 候補に登録
2. Cranelift JIT で SyscallIR 拡張、=mov rax,NR; syscall= で hot path 維持
3. それでも回復しない場合は specialized を Doc 76 から exception 扱いで保持

** §7.4 SCM_RIGHTS の elisp marshaling 複雑度

Stage G (= SCM_RIGHTS) の cmsg 計算は POSIX マクロ依存で、glibc / musl /
Darwin libc の internal padding が微妙に違う. 緩和:

- Stage G を最終 stage に置く (= 他の simpler stage で nl-ffi marshaling
  pattern 確立後に取り組む)
- =nl-ffi-call libc CMSG_SPACE [:size_t :size_t]= は POSIX が =extern=
  symbol を提供しない (= マクロ only). 苦肉の策として Rust 側に
  =nl_cmsg_space=/=nl_cmsg_len= helper を =cdylib= の =libnelisp_runtime=
  に export する. これは "specialized primitive を 1 つ減らすために
  数行 helper を残す" trade-off で許容.

** §7.5 Stage 中の partial 状態でのリリース

各 stage commit は independent build/test pass を要件にする. user が
途中 stage で release を取ったとしても挙動が一貫している (= signature
不変、内部実装のみ切替). この invariant は Doc 50 stage 7a〜7i で確立済.
