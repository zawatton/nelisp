** 2.1 Actor 境界は何単位か

- A: =(nelisp-spawn THUNK)= で明示生成する単位 (Erlang / Go model)
- B: buffer 単位 (buffer-local state = actor state、NeLisp novelty
  最大化)
- C: 両方 — spawn は primary、buffer-as-actor は syntactic sugar
  (=with-buffer-actor= 的 macro)

*推奨*: A。理由:
- B は魅力的だが既存 Emacs buffer semantics との整合で決めるべき設計論点多
  (buffer-local の overlap / window-local と actor の関係等)、Phase 4 scope
  に入れると MVP 延びる
- B は Phase 4c (v0.4.x minor) として spawn 安定後に追加するのが筋
- C は両方揃ってから判断可、A を先に

** 2.2 Mailbox semantics

- A: unbounded FIFO (Erlang BEAM default)
- B: bounded with back-pressure — =(nelisp-send)= block when full
  (Go channel-style、flow control)
- C: priority queue — message に priority tag
- D: A を default、B を opt-in flag で bounded 指定可

*推奨*: D。MVP は A 単純、bounded は actor 生成時 =(nelisp-spawn thunk
:mailbox-capacity 64)= で opt-in。C は YAGNI。

** 2.3 Message passing semantics

- A: *shared nothing* — =send= 時に deep copy、受信側は独自 heap
- B: *shared immutable* — cons/vector/string の immutable 検出で copy
  スキップ、mutable は copy
- C: *ownership transfer* — sender が reference 失い receiver に移る
  (Rust borrow-check 的)

*推奨*: B。理由:
- A は safety 最大だが fib(30) 結果 list を毎回 copy は辛い。anvil
  痛み解消 test case (tool handler 並列) でも大きな result を copy は
  overhead
- C は novelty 高いが NeLisp 既存コードへの migration burden 高。
  "send したら使えない" を user が納得する API は長期議論必要
- B は Emacs 既存 Lisp の pragma (=pure= 宣言) と親和、Phase 3c.5
  =nelisp-gc-actor-boundary= で mutable cell のみ ledger に入れる方針
  と一貫
- *Note*: "immutable" の決定は =(symbolp) or (numberp) or interned string=
  等の conservative classifier で start、精度向上は v0.4.x 以降

** 2.4 Scheduler model

- A: *cooperative* — actor が =(nelisp-receive)= / =(nelisp-yield)= で
  譲る、host Emacs 1 thread 内 coroutine
- B: *preemptive* — timer-driven quantum でスケジューラが強制切替
- C: *work-stealing* — M:N (goroutine 式)、true thread 前提

*推奨*: A。理由:
- B は host Emacs が single-threaded event loop であり timer 割込を
  "preemption" として使えない (=post-command-hook= では粒度粗すぎ)
- C は Phase 5 の真 thread 前提、Phase 4 scope 超過
- cooperative なら fiber 実装相当を actor scheduler 内部で賄える (§2.7
  fiber 吸収論点)
- BEAM も cooperative で実用十分

** 2.5 yield point の設計

cooperative scheduler の前提として、actor が "自然に" yield する箇所:

- A: =nelisp-receive= block 時のみ yield (message 待ち = 計算終了)
- B: A + 明示 =(nelisp-yield)= call (長い計算中に手動挿入)
- C: A + B + bytecode VM dispatch 内で N instruction おきに自動 yield

*推奨*: B。A は CPU-bound actor が scheduler 独占する問題があり、
明示 yield はあれば十分 (user が "ここ長い" を知っている)。C は
Phase 3b.8 JIT で VM を迂回する経路があり、全 code path 網羅できない。
B で start、C は v0.4.x 以降 opt-in flag として追加検討。

** 2.6 Supervision

- A: supervisor なし (crash したら actor 消える、user が sense)
- B: flat — 1 root supervisor、child crash で自動再起動 (Erlang
  =one_for_one= 相当のみ)
- C: OTP full — hierarchical tree、=one_for_all= / =rest_for_one= /
  =simple_one_for_one= 全部

*推奨*: B。A は anvil 痛み解消に不十分 (tool handler actor が死んだら
user 通知できない)、C は Phase 4 scope 超過。B で crash → log → restart
が出来れば v0.4.0 novelty 条件を満たす。

** 2.7 Fiber との関係

=04-concurrency.org= §5 候補 D (fiber / stackful coroutine) は元々
Phase 3 scope だったが、Phase 3b.8 JIT 実装で bytecode VM dispatch を
skip する経路 (fib 25x) が main path になり、「bytecode VM 内 yield point」
という integration layer の意味が弱くなった。

- A: *fiber を skip、actor scheduler の内部実装として吸収*
  (actor = fiber with mailbox)
- B: *独立 =nelisp-yield= を fiber primitive として ship*、actor は
  その上に build
- C: *VM dispatch 内 auto-yield も別枠で ship* (§2.5 C と同等)

*推奨*: A。=nelisp-yield= は §2.5 B で actor context 限定の yield
として expose、fiber-as-separate-primitive は v0.5+ (真 thread 後) に
再検討。roadmap §7 の "fiber" line item は Phase 4 に吸収宣言する。

* 5. Actor runtime 構造 (草案、§2 推奨 前提)

** 5.1 Actor struct

#+begin_src emacs-lisp
  (cl-defstruct (nelisp-actor (:constructor nelisp-actor--make))
    id             ; unique symbol (generated)
    thunk          ; closure to run as actor body
    mailbox        ; FIFO (list or queue struct)
    mailbox-cap    ; nil = unbounded, integer = bounded
    status         ; :runnable :blocked-receive :dead :crashed
    owned-set      ; hash-table from `nelisp-gc-actor-boundary'
    supervisor     ; parent actor-id or nil
    restart-policy ; :permanent :temporary :transient
    iterator       ; live generator returned by (funcall thunk) (Phase 4.2+;
                   ; originally drafted as `k' / continuation)
    last-error)    ; condition-case err from last crash
#+end_src

** 5.2 Run queue

=nelisp-actor--queue= dynamic list。scheduler は:

1. pop 先頭 actor
2. actor.k を resume (=funcall cont=)
3. thunk が =nelisp-yield= or =nelisp-receive= で block するまで run
4. block 理由で status 更新、次 actor へ
5. queue 空になるまで繰り返し、=run-until-idle= で停止

** 5.3 yield / resume 実装

host Emacs に真 coroutine (=call/cc= 相当) がないため、cooperative yield
は以下のいずれか:

- A: =generator.el= (Emacs 25.1+ の CPS-style generator) を活用
- B: =catch= / =throw= で caller に制御返し、mailbox-待ちフラグ立て
  て次 run 時 re-entry
- C: state machine を手書き、actor thunk を segment に分割

*決定*: A を試す。=generator.el= は =iter-defun= で CPS 変換するため
yield point を透過的に扱える。Elisp 既存機能の活用 = maintenance 負担
最小。driver integration だけ書けば足りる想定。

** 5.4 Ownership boundary enforcement

- spawn 時: =(nelisp-gc-actor-boundary thunk)= で owned-set 計算
- send 時: message を walk して mutable cell を deep-copy (§4.3)、
  receiver owned-set に追加
- crash 時: owned-set は drop (host GC 回収)、finalizer が発火
  (Phase 3c.3 の =nelisp-gc-collect=)

** 5.5 host Emacs event loop との共存

actor scheduler は host Emacs event loop を blocking しない:
- =nelisp-actor-start=: =run-with-idle-timer 0.01 nil driver= で
  queue drive を idle 時に発火
- user command 中 (input pending) は driver 自粛、input idle 時再開
- =nelisp-actor-run-until-idle=: ERT / PoC 用 synchronous runner

* 6. anvil 痛み解消 PoC (§4.6 詳細)

=04-concurrency.org= §4.5 test #1 の MVP 実装イメージ:

#+begin_src emacs-lisp
  ;; 4 worker actor が同 Emacs session で並列に "tool call" を処理
  (defvar *workers*
    (cl-loop for i below 4
             collect
             (nelisp-spawn
              (lambda ()
                (while t
                  (pcase (nelisp-receive)
                    (`(:task ,id ,fn ,args ,reply-to)
                     (let ((result
                            (condition-case e (apply fn args)
                              (error (list :error e)))))
                       (nelisp-send reply-to `(:result ,id ,result))))
                    (`:shutdown (throw 'actor-done nil)))))
              :restart :permanent)))

  (defun dispatch-parallel (fn-args-alist)
    "Send 4 tasks to workers, collect results."
    (let ((me (nelisp-self))
          (pending (length fn-args-alist)))
      (cl-loop for (fn . args) in fn-args-alist
               for w in *workers*
               for i from 0
               do (nelisp-send w `(:task ,i ,fn ,args ,me)))
      (let (results)
        (while (> pending 0)
          (pcase (nelisp-receive)
            (`(:result ,id ,r) (push (cons id r) results)
                               (cl-decf pending))))
        (nreverse results))))
#+end_src
